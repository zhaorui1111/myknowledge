### 1.2 场景化选型建议

| 业务场景 | 推荐方案 | 理由 |
|---------|---------|------|
| 高性能 C 端核心页面 | 原生 / Flutter | 性能要求高，交互复杂 |
| 中等交互 C 端页面 | React Native | 开发效率高，社区成熟 |
| 低频内容页/活动页 | WebView | 快速上线，可热更新 |
| 鸿蒙生态应用 | ArkTS/ArkUI | 官方支持，原生性能 |
| 跨端逻辑共享 | KMP | 逻辑复用，UI 原生 |
| 快速验证 MVP | Flutter/RN | 一套代码多端运行 |
| 大型电商 App | 混合架构（原生+RN+WebView） | 按页面特性选择技术 |
| 企业内部工具 | Flutter | 一套代码覆盖全平台 |

```

└── 否 → 是否需要 Web 端？
    ├── 是 → 是否需要高度统一的 UI？
    │   ├── 是 → Flutter（Web 支持较好）
    │   └── 否 → React Native（需 react-native-web）
    └── 否 → 性能要求是否极高？

        └── 否 → 团队技术栈？
            ├── React/TS → React Native
            ├── Kotlin → KMP + Compose Multiplatform
            └── 无偏好 → Flutter（学习曲线平滑）
```

## 二、React Native 面试要点

### 2.1 架构原理

**Q: RN 新旧架构的区别？**

旧架构（Bridge）：
- JS 和 Native 通过异步 Bridge 通信
- 数据通过 JSON 序列化/反序列化
- UI 操作必须经过 Bridge，存在延迟
- 三线程：JS Thread、Native Thread、Shadow Thread

新架构（JSI + Fabric + TurboModules）：
- JSI：C++ 直接调用，无需序列化
- Fabric：同步 UI 操作，新渲染器
- TurboModules：同步原生模块调用
- CodeGen：类型安全的 JS↔Native 接口

**Q: JSI 的工作原理？**

JSI（JavaScript Interface）是 C++ 层的接口，允许 JS 直接持有 C++ 对象引用。通过 JSI，JS 代码可以同步调用原生方法，无需通过异步 Bridge。Hermes 引擎内置 JSI 支持，JSI 不依赖特定的 JS 引擎，但 Hermes 对其做了优化。

```cpp
// JSI 核心概念
// JS 可以直接调用 C++ 对象的方法
jsi::Object createNativeModule(jsi::Runtime &rt) {
    auto module = jsi::Object(rt);
    module.setProperty(rt, "multiply", jsi::Function::createFromHostFunction(
        rt,
        jsi::PropNameID::forAscii(rt, "multiply"),
        2,
        [](jsi::Runtime &rt, const jsi::Value &thisVal, const jsi::Value *args, size_t count) {
            double a = args[0].asNumber();
            double b = args[1].asNumber();
            return jsi::Value(a * b);  // 同步返回
        }
    ));
    return module;
}
```

### 2.2 性能优化

**Q: RN 常见性能问题及优化手段？**

1. **列表性能**：使用 FlashList 替代 FlatList，支持回收复用
2. **渲染优化**：React.memo、useCallback、useMemo 避免不必要渲染
3. **图片优化**：使用 FastImage（原生图片库），指定 cachePolicy
4. **启动优化**：Hermes 引擎预编译字节码，延迟加载非关键模块
5. **动画优化**：使用 Animated.useNativeDriver 或 Reanimated 在 UI 线程执行动画
6. **Bridge 瓶颈**：新架构下使用 JSI 同步调用，减少 Bridge 通信

**Q: Hermes 引擎的优势？**

- 预编译字节码（AOT），启动时无需解析 JS
- 更低的内存占用
- 更快的启动速度
- 内置 JSI 支持
- 更好的调试体验

### 2.3 状态管理

**Q: React Native 中如何选择状态管理方案？**

- **组件内状态**：useState / useReducer
- **跨组件共享**：Context + useReducer（简单场景）
- **全局状态**：Zustand（轻量）、Redux Toolkit（大型应用）
- **服务端状态**：React Query / SWR（数据获取和缓存）
- **派生状态**：useMemo / selector 模式

## 三、Flutter 面试要点

### 3.1 三棵树

**Q: Flutter 的三棵树是什么？各自职责？**

1. **Widget Tree**：不可变的配置描述，轻量级，频繁创建销毁
2. **Element Tree**：Widget 的实例化，管理生命周期，持有 State，作为 Widget 和 RenderObject 的中间层
3. **RenderObject Tree**：负责实际的布局和绘制，重量级对象

**关系**：
- Widget → Element：`mount` 时创建 Element，`canUpdate` 判断是否复用
- Element → RenderObject：Element 创建对应的 RenderObject
- State 挂载在 Element 上，Widget 重建时 Element 和 RenderObject 可以复用

**Q: 为什么 Flutter 使用三棵树而不是两棵？**

分离关注点：Widget 只描述"是什么"（不可变配置），Element 管理"怎么管理"（生命周期、状态），RenderObject 负责"怎么画"（布局、绘制）。这样 Widget 可以频繁重建而不影响性能，因为 Element 和 RenderObject 通过 diff 算法复用。

### 3.2 渲染机制

**Q: Flutter 的渲染流程？**

1. `setState()` → `markNeedsBuild()` → 将 Element 标记为 dirty
2. 下一帧 vsync 信号到来时，执行 dirty Element 的 `rebuild()`
3. Build Phase：创建/更新 Widget Tree
4. Layout Phase：RenderObject 执行 `performLayout()`，Constraints 向下传递，Size 向上返回
5. Paint Phase：RenderObject 执行 `paint()`，生成 Layer
6. Composite Phase：Layer Tree 交给 Skia/Impeller 进行 GPU 合成
7. GPU 渲染到屏幕

**Q: Skia 和 Impeller 的区别？**

| 维度 | Skia | Impeller |
|------|------|---------|
| 着色器编译 | 运行时即时编译（JIT），首次绘制卡顿 | 预编译，无运行时编译 |
| 渲染后端 | OpenGL/Metal/Vulkan | Metal (iOS) / Vulkan (Android) |
| 并发 | 单线程光栅化 | 多线程渲染 |
| 性能 | 稳定但首次有 Shader Jank | 消除 Shader Jank |
| 状态 | Flutter 3.7 前默认 | Flutter 3.7+ iOS 默认，3.10+ Android 可选 |

### 3.3 状态管理

**Q: Provider、Riverpod、Bloc 的区别？**

**Provider**：
- 基于 InheritedWidget
- 简单易用，适合中小项目
- 缺点：全局 Provider 需要在 Widget 树顶部声明

**Riverpod**：
- Provider 的改进版，编译时安全
- 不依赖 BuildContext
- 支持 AutoDispose、Family 等高级特性
- 适合中大型项目

**Bloc/Cubit**：
- 基于 Stream/Event-Driven
- 强制单向数据流：Event → Bloc → State
- 适合复杂业务逻辑和团队协作
- 学习曲线较高

### 3.4 Platform Channel

**Q: Platform Channel 的三种类型及应用场景？**

1. **MethodChannel**：方法调用（请求-响应），最常用
2. **EventChannel**：事件流（Native → Flutter），如传感器数据
3. **BasicMessageChannel**：双向消息传递，支持自定义编解码

**Q: Platform Channel 的数据如何序列化？**

使用 StandardMessageCodec，支持的基本类型：
- null、bool、int、double、String
- Uint8List、Int32List、Int64List、Float64List
- List、Map

自定义对象需要转换为 Map 传递。

## 四、混合开发面试要点

**Q: JSBridge 的实现原理？**

三种主要方式：
1. **URL Scheme 拦截**：JS 通过修改 `iframe.src` 或 `window.location` 触发原生拦截
2. **注入 JS 对象**：iOS `WKScriptMessageHandler`、Android `@JavascriptInterface`
3. **prompt/console 拦截**：重写 `window.prompt`，原生拦截返回值

**Q: 离线包方案的核心是什么？**

1. **构建发布**：将 H5 资源打包成 zip，上传 CDN
2. **预加载**：App 启动时检查版本，后台下载
3. **请求拦截**：WebView 拦截资源请求，命中本地文件直接返回
4. **版本管理**：增量更新、回滚机制
5. **降级策略**：本地未命中时降级到网络请求

**Q: WebView 混合开发如何优化首屏？**

1. **WebView 预初始化**：应用启动时创建 WebView 池
2. **离线包**：预加载 H5 资源到本地
3. **导航预加载**：在用户点击前预加载页面
4. **服务端渲染（SSR）**：首屏 HTML 服务端直出
5. **骨架屏**：减少白屏感知

## 五、综合面试题

### Q1: 如果让你从零搭建一个电商 App 的跨端架构，你会怎么设计？

**回答框架**：

1. **技术选型**：核心交易链路用原生（性能和稳定性），商品详情/活动页用 Flutter（开发效率+体验），帮助/协议页用 WebView（快速迭代）
2. **架构分层**：
   - 基础层：网络、存储、日志、监控（可用 KMP 共享）
   - 业务层：各业务模块独立开发
   - UI 层：按页面特性选择技术方案
3. **路由设计**：统一路由表，支持原生/Flutter/WebView 页面跳转
4. **状态管理**：全局状态（登录态、购物车）通过统一的状态管理方案
5. **热更新**：非核心页面通过 CodePush / Shorebird 热更新
6. **监控**：Sentry 崩溃监控 + 自研性能监控

### Q2: RN 和 Flutter 在实际项目中如何选择？

从以下维度对比：

1. **团队技术栈**：React/TS 团队选 RN，愿意学习新语言的团队选 Flutter
2. **性能要求**：Flutter 自绘引擎性能更优，RN 通过新架构大幅改善
3. **原生交互**：RN 使用原生组件更自然，Flutter 需要 Platform View 嵌入原生组件
4. **包体积**：Flutter 增加约 4-5MB，RN 增加约 7-10MB（含 Hermes）
5. **生态**：RN 依赖 NPM 生态更成熟，Flutter 的 pub.dev 增长迅速
6. **Web 支持**：Flutter Web 更成熟，RN 需 react-native-web 转换
7. **热更新**：RN 有 CodePush，Flutter 有 Shorebird（较新）

### Q3: 跨端开发中如何处理平台差异？

1. **条件编译**：`Platform.isIOS` / `Platform.isAndroid` 判断
2. **抽象接口**：定义通用接口，各平台提供实现
3. **设计模式**：Strategy 模式，运行时选择平台策略
4. **能力降级**：不支持的平台提供替代方案
5. **响应式设计**：不同屏幕尺寸/安全区域的适配

### Q4: 如何监控和优化跨端应用的性能？

**监控**：
- 启动时间：冷启动/热启动耗时
- 帧率：FPS、掉帧率
- 内存：峰值内存、内存泄漏
- 网络：API 响应时间、失败率
- 崩溃：崩溃率、ANR 率

**优化**：
- 启动：延迟初始化、引擎预热、按需加载
- 渲染：减少重建、RepaintBoundary、列表优化
- 内存：及时释放、图片压缩、大对象用 Isolate/Worker
- 网络：请求合并、缓存、预加载、HTTP/2

### Q5: 鸿蒙系统对跨端开发的影响？

1. **新平台**：需要额外适配 ArkTS/ArkUI
2. **生态早期**：第三方库和工具链尚不成熟
3. **机会**：华为推广力度大，市场占有率增长
4. **策略**：核心功能优先适配鸿蒙原生，非核心功能可考虑 WebView 降级
5. **长期**：Flutter/RN 可能通过社区支持鸿蒙，但性能和体验不如原生 ArkUI

## 八、小结

跨端技术选型没有银弹，需要根据业务场景、团队能力、性能要求、平台覆盖等多维度综合决策。面试中考察的核心是：对各种跨端方案原理的深入理解、实际项目中的选型思考、性能优化的实践经验。建议在准备面试时，不仅要掌握理论知识，还要结合实际项目经验进行回答，展示解决实际问题的能力。
