# iOS 多线程与并发详解

---

## 一、线程基础

### 1.1 进程与线程

```
进程（Process）：
  - 操作系统分配资源的基本单位
  - 拥有独立的虚拟地址空间、文件描述符、信号处理等
  - iOS 中每个 App 就是一个进程

线程（Thread）：
  - CPU 调度的基本单位
  - 共享进程的内存空间和资源
  - 每个线程有独立的栈空间（默认 512KB，主线程 1MB）
  - 线程切换开销远小于进程切换

iOS 线程模型：
  ┌───────────────────────────────┐
  │           进程                 │
  │  ┌─────┐ ┌─────┐ ┌─────┐    │
  │  │主线程│ │子线程│ │子线程│    │
  │  │     │ │     │ │     │    │
  │  │ UI  │ │网络  │ │计算  │    │
  │  │渲染  │ │请求  │ │任务  │    │
  │  └─────┘ └─────┘ └─────┘    │
  │                               │
  │  共享：堆内存、全局变量、文件   │
  │  独立：栈内存、寄存器、PC       │
  └───────────────────────────────┘
```

### 1.2 iOS 中的线程方案对比

| 方案 | 语言 | 线程生命周期 | 使用频率 |
|------|------|------------|---------|
| pthread | C | 手动管理 | 极少 |
| NSThread | ObjC | 手动管理 | 少 |
| GCD | C | 自动管理 | 最常用 |
| NSOperation | ObjC | 自动管理 | 常用 |
| Swift Concurrency | Swift | 自动管理 | 推荐（Swift 5.5+） |

### 1.3 主线程的特殊性

```objc
// 主线程 = UI 线程，职责：
// 1. 接收和处理用户事件（触摸、手势）
// 2. 渲染 UI（布局计算、绘制、合成）
// 3. RunLoop 驱动

// ⚠️ 主线程绝对不能做的事：
// 1. 耗时计算（>16ms 就会掉帧）
// 2. 同步网络请求
// 3. 大文件 I/O
// 4. 大量数据解析

// 检查是否在主线程
[NSThread isMainThread];          // ObjC
Thread.isMainThread               // Swift

// ⚠️ 主队列 ≠ 主线程
// dispatch_get_main_queue 的任务一定在主线程执行
// 但主线程上执行的代码不一定来自主队列
// 例如：同步派发到主线程的自定义串行队列的任务也在主线程执行
```

## 二、pthread（POSIX Threads）

### 2.1 基本用法

```c
#import <pthread.h>

// 创建线程
pthread_t thread;
int result = pthread_create(&thread,    // 线程标识
                            NULL,        // 属性（NULL 默认）
                            start_routine, // 线程入口函数
                            NULL);       // 参数

// 线程入口函数
void *start_routine(void *arg) {
    NSLog(@"子线程执行: %@", [NSThread currentThread]);
    return NULL;
}

// 等待线程结束
pthread_join(thread, NULL);

// 分离线程（不需要 join，线程结束自动回收资源）
pthread_detach(thread);

// 获取当前线程
pthread_t current = pthread_self();

// 设置线程优先级
struct sched_param param;
param.sched_priority = 50;
pthread_setschedparam(thread, SCHED_OTHER, &param);
```

### 2.2 pthread 互斥锁

```c
// 互斥锁（最基础的锁）
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

// 或动态初始化
pthread_mutex_t mutex;
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_NORMAL);
pthread_mutex_init(&mutex, &attr);
pthread_mutexattr_destroy(&attr);

// 加锁/解锁
pthread_mutex_lock(&mutex);
// 临界区...
pthread_mutex_unlock(&mutex);

// 尝试加锁（非阻塞）
if (pthread_mutex_trylock(&mutex) == 0) {
    // 获得锁
    pthread_mutex_unlock(&mutex);
}

// 销毁
pthread_mutex_destroy(&mutex);

// 递归锁（同一线程可多次加锁）
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);
```

### 2.3 pthread 条件变量

```c
pthread_cond_t cond = PTHREAD_COND_INITIALIZER;
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

// 生产者
pthread_mutex_lock(&mutex);
// 生产数据...
pthread_cond_signal(&cond);    // 唤醒一个等待线程
// pthread_cond_broadcast(&cond); // 唤醒所有等待线程
pthread_mutex_unlock(&mutex);

// 消费者
pthread_mutex_lock(&mutex);
while (/* 数据未就绪 */) {
    pthread_cond_wait(&cond, &mutex);  // 等待时自动释放锁，被唤醒时重新获得锁
}
// 消费数据...
pthread_mutex_unlock(&mutex);
```

### 2.4 pthread 读写锁

```c
// 读写锁：多读单写（读共享，写互斥）
pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

// 读锁（多个线程可同时获得读锁）
pthread_rwlock_rdlock(&rwlock);
// 读取数据...
pthread_rwlock_unlock(&rwlock);

// 写锁（排他，必须等所有读锁和写锁释放）
pthread_rwlock_wrlock(&rwlock);
// 写入数据...
pthread_rwlock_unlock(&rwlock);

pthread_rwlock_destroy(&rwlock);
```

## 三、NSThread

### 3.1 创建与启动

```objc
// 方式1：创建后手动启动
NSThread *thread = [[NSThread alloc] initWithTarget:self
                                           selector:@selector(doWork)
                                             object:nil];
thread.name = @"WorkerThread";
thread.qualityOfService = NSQualityOfServiceUserInitiated;
[thread start];

// 方式2：创建并自动启动（detached）
[NSThread detachNewThreadSelector:@selector(doWork)
                         toTarget:self
                       withObject:nil];

// 方式3：隐式创建
[self performSelectorInBackground:@selector(doWork) withObject:nil];

// 方式4：Block（iOS 10+）
NSThread *thread = [[NSThread alloc] initWithBlock:^{
    NSLog(@"Block 线程执行");
}];
[thread start];
```

### 3.2 线程操作

```objc
// 休眠
[NSThread sleepForTimeInterval:2.0];  // 休眠 2 秒
[NSThread sleepUntilDate:[NSDate dateWithTimeIntervalSinceNow:2.0]];

// 退出当前线程（不可恢复，谨慎使用）
[NSThread exit];

// 取消标记（只是设标记，线程不会自动退出，需要线程自己检查）
[thread cancel];
// 线程内检查
if ([NSThread currentThread].cancelled) {
    return;  // 自行退出
}

// 线程间通信
[self performSelectorOnMainThread:@selector(updateUI:)
                       withObject:result
                    waitUntilDone:NO];

[self performSelector:@selector(doWork:)
             onThread:workerThread
           withObject:data
        waitUntilDone:NO];
```

### 3.3 常驻线程

```objc
// 创建常驻线程（线程不退出，等待任务）
// 原理：向线程的 RunLoop 添加 Source/Port，防止 RunLoop 退出

+ (NSThread *)sharedThread {
    static NSThread *thread = nil;
    static dispatch_once_t onceToken;
    dispatch_once(&onceToken, ^{
        thread = [[NSThread alloc] initWithBlock:^{
            // 添加一个 Port 使 RunLoop 不退出
            [[NSRunLoop currentRunLoop] addPort:[NSMachPort port]
                                        forMode:NSDefaultRunLoopMode];
            // 运行 RunLoop（阻塞，直到 RunLoop 被停止）
            [[NSRunLoop currentRunLoop] run];
        }];
        thread.name = @"com.app.persistent";
        [thread start];
    });
    return thread;
}

// 向常驻线程派发任务
[self performSelector:@selector(doWork:)
             onThread:[self sharedThread]
           withObject:data
        waitUntilDone:NO];

// ⚠️ 常驻线程的问题：
// 1. 线程无法退出（[NSRunLoop run] 无法被停止）
// 2. 占用系统资源（栈内存 512KB）
// 3. 现代开发中推荐使用 GCD 替代
```

## 四、GCD（Grand Central Dispatch）

### 4.1 核心概念

```
GCD 的两个核心概念：

任务（Block）：要执行的代码块
队列（Queue）：任务的等待队列

队列类型：
  ┌─────────────────────────────────────────┐
  │              串行队列                     │
  │  Serial Queue                            │
  │  任务按 FIFO 顺序逐一执行                │
  │  ┌─→ Task1 ─→ Task2 ─→ Task3 ─→        │
  │  同一时刻只有 1 个任务在执行              │
  └─────────────────────────────────────────┘

  ┌─────────────────────────────────────────┐
  │              并发队列                     │
  │  Concurrent Queue                        │
  │  任务按 FIFO 取出，可同时执行多个         │
  │  ┌─→ Task1 ─→                            │
  │  ├─→ Task2 ─→                            │
  │  └─→ Task3 ─→                            │
  │  GCD 根据系统资源决定同时执行几个         │
  └─────────────────────────────────────────┘

派发方式：
  同步派发（sync）：等待任务完成后返回，阻塞当前线程
  异步派发（async）：立即返回，不阻塞当前线程
```

### 4.2 队列的创建与获取

```objc
// ========== 串行队列 ==========
// 创建
dispatch_queue_t serialQ = dispatch_queue_create("com.app.serial", DISPATCH_QUEUE_SERIAL);

// 主队列（特殊的串行队列）
dispatch_queue_t mainQ = dispatch_get_main_queue();

// ========== 并发队列 ==========
// 创建
dispatch_queue_t concurrentQ = dispatch_queue_create("com.app.concurrent", DISPATCH_QUEUE_CONCURRENT);

// 全局并发队列
dispatch_queue_t globalQ = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0);

// 全局队列优先级（对应 QoS）
// DISPATCH_QUEUE_PRIORITY_HIGH        → QOS_CLASS_USER_INITIATED
// DISPATCH_QUEUE_PRIORITY_DEFAULT     → QOS_CLASS_DEFAULT
// DISPATCH_QUEUE_PRIORITY_LOW         → QOS_CLASS_UTILITY
// DISPATCH_QUEUE_PRIORITY_BACKGROUND  → QOS_CLASS_BACKGROUND
```

### 4.3 同步/异步 × 串行/并发 组合

```objc
// ========== 异步 + 并发（最常用：开子线程，并发执行）==========
dispatch_async(concurrentQ, ^{ NSLog(@"Task1 - %@", [NSThread currentThread]); });
dispatch_async(concurrentQ, ^{ NSLog(@"Task2 - %@", [NSThread currentThread]); });
dispatch_async(concurrentQ, ^{ NSLog(@"Task3 - %@", [NSThread currentThread]); });
// 结果：Task1/2/3 在不同子线程上并发执行，顺序不确定

// ========== 异步 + 串行（开子线程，串行执行）==========
dispatch_async(serialQ, ^{ NSLog(@"Task1"); });
dispatch_async(serialQ, ^{ NSLog(@"Task2"); });
dispatch_async(serialQ, ^{ NSLog(@"Task3"); });
// 结果：Task1 → Task2 → Task3 按顺序在同一个子线程执行

// ========== 同步 + 并发（不开线程，当前线程逐个执行）==========
dispatch_sync(concurrentQ, ^{ NSLog(@"Task1 - %@", [NSThread currentThread]); });
dispatch_sync(concurrentQ, ^{ NSLog(@"Task2 - %@", [NSThread currentThread]); });
// 结果：在当前线程按顺序执行（sync 阻塞，所以无法并发）

// ========== 同步 + 串行（不开线程，当前线程逐个执行）==========
dispatch_sync(serialQ, ^{ NSLog(@"Task1"); });
dispatch_sync(serialQ, ^{ NSLog(@"Task2"); });
// 结果：在当前线程按顺序执行

// ========== ⚠️ 同步 + 主队列（在主线程调用 → 死锁！）==========
// 主线程上执行：
dispatch_sync(dispatch_get_main_queue(), ^{
    NSLog(@"永远不会执行");
});
// 死锁原因：
// 1. sync 要等 Block 执行完才返回
// 2. Block 被加入主队列，需要等当前主队列任务（sync 所在的代码）执行完
// 3. 互相等待 → 死锁
```

### 4.4 死锁分析

```objc
// 死锁本质：sync 派发到当前正在执行的串行队列

// 死锁场景1：主线程 sync 到主队列
dispatch_sync(dispatch_get_main_queue(), ^{
    // 死锁
});

// 死锁场景2：串行队列中 sync 到同一个串行队列
dispatch_queue_t q = dispatch_queue_create("com.app.q", DISPATCH_QUEUE_SERIAL);
dispatch_async(q, ^{
    NSLog(@"Task1");
    dispatch_sync(q, ^{
        NSLog(@"Task2");  // 死锁！
    });
    NSLog(@"Task3");  // 永远不会执行
});

// 不会死锁：并发队列中 sync 到同一个并发队列
dispatch_queue_t cq = dispatch_queue_create("com.app.cq", DISPATCH_QUEUE_CONCURRENT);
dispatch_async(cq, ^{
    dispatch_sync(cq, ^{
        NSLog(@"不会死锁");  // ✅ 并发队列允许同时执行多个任务
    });
});

// 不会死锁：不同队列之间 sync
dispatch_queue_t q1 = dispatch_queue_create("q1", DISPATCH_QUEUE_SERIAL);
dispatch_queue_t q2 = dispatch_queue_create("q2", DISPATCH_QUEUE_SERIAL);
dispatch_async(q1, ^{
    dispatch_sync(q2, ^{
        NSLog(@"不会死锁");  // ✅ q2 不是当前正在执行的队列
    });
});
```

### 4.5 DispatchGroup

```objc
// 用途：等待一组异步任务全部完成后执行后续操作

dispatch_group_t group = dispatch_group_create();
dispatch_queue_t queue = dispatch_get_global_queue(0, 0);

// 方式1：dispatch_group_async
dispatch_group_async(group, queue, ^{
    // 任务1
    NSLog(@"Task1");
});
dispatch_group_async(group, queue, ^{
    // 任务2
    NSLog(@"Task2");
});

// 等待所有任务完成后回调（异步，不阻塞）
dispatch_group_notify(group, dispatch_get_main_queue(), ^{
    NSLog(@"所有任务完成，更新 UI");
});

// 等待所有任务完成（同步，阻塞当前线程）
dispatch_group_wait(group, DISPATCH_TIME_FOREVER);
// 超时版本：
dispatch_time_t timeout = dispatch_time(DISPATCH_TIME_NOW, 5 * NSEC_PER_SEC);
long result = dispatch_group_wait(group, timeout);
if (result == 0) {
    NSLog(@"全部完成");
} else {
    NSLog(@"超时");
}

// 方式2：手动 enter/leave（适合包裹异步回调）
dispatch_group_enter(group);
[self fetchDataWithCompletion:^(NSData *data) {
    // 处理数据...
    dispatch_group_leave(group);  // ⚠️ 必须与 enter 配对
}];

dispatch_group_enter(group);
[self fetchImageWithCompletion:^(UIImage *image) {
    // 处理图片...
    dispatch_group_leave(group);
}];

dispatch_group_notify(group, dispatch_get_main_queue(), ^{
    NSLog(@"数据和图片都加载完成");
});

// ⚠️ enter 和 leave 必须配对，否则：
// enter > leave → notify 永远不回调
// leave > enter → 崩溃（over-leave）
```

### 4.6 DispatchSemaphore

```objc
// 信号量：控制并发数量

// 创建（value 表示初始资源数）
dispatch_semaphore_t sema = dispatch_semaphore_create(0);

// ========== 用途1：异步转同步 ==========
dispatch_async(dispatch_get_global_queue(0, 0), ^{
    // 异步操作...
    dispatch_semaphore_signal(sema);  // +1
});
dispatch_semaphore_wait(sema, DISPATCH_TIME_FOREVER);  // -1，阻塞直到 > 0
NSLog(@"异步操作完成");

// ========== 用途2：控制最大并发数 ==========
dispatch_semaphore_t limitSema = dispatch_semaphore_create(3);  // 最多 3 个并发

for (int i = 0; i < 100; i++) {
    dispatch_semaphore_wait(limitSema, DISPATCH_TIME_FOREVER);  // 获取许可（-1）
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        NSLog(@"Task %d", i);
        [NSThread sleepForTimeInterval:1.0];
        dispatch_semaphore_signal(limitSema);  // 释放许可（+1）
    });
}

// ========== 用途3：简单互斥锁 ==========
dispatch_semaphore_t lockSema = dispatch_semaphore_create(1);

dispatch_semaphore_wait(lockSema, DISPATCH_TIME_FOREVER);  // 加锁
// 临界区...
dispatch_semaphore_signal(lockSema);  // 解锁
```

### 4.7 DispatchWorkItem

```objc
// 封装任务，支持取消和等待

// 创建
dispatch_block_t workItem = dispatch_block_create(0, ^{
    NSLog(@"执行任务");
});

// 派发
dispatch_async(dispatch_get_global_queue(0, 0), workItem);

// 等待完成
dispatch_block_wait(workItem, DISPATCH_TIME_FOREVER);

// 取消（标记取消，不会中断正在执行的任务）
dispatch_block_cancel(workItem);

// 检查是否已取消
if (dispatch_block_testcancel(workItem)) {
    NSLog(@"任务已取消");
}

// 任务完成通知
dispatch_block_notify(workItem, dispatch_get_main_queue(), ^{
    NSLog(@"workItem 执行完成");
});

// Swift 版本
let item = DispatchWorkItem {
    print("执行任务")
}
DispatchQueue.global().async(execute: item)
item.cancel()
item.wait()
item.notify(queue: .main) {
    print("完成")
}
```

### 4.8 DispatchSource（定时器）

```objc
// GCD 定时器：不受 RunLoop 模式影响，更精确

dispatch_source_t timer = dispatch_source_create(DISPATCH_SOURCE_TYPE_TIMER,
                                                  0, 0,
                                                  dispatch_get_main_queue());

// 设置定时器参数
dispatch_source_set_timer(timer,
                          dispatch_time(DISPATCH_TIME_NOW, 0),   // 开始时间
                          1.0 * NSEC_PER_SEC,                    // 间隔（1秒）
                          0.1 * NSEC_PER_SEC);                   // 允许的误差

// 设置回调
dispatch_source_set_event_handler(timer, ^{
    NSLog(@"Timer fired!");
});

// 设置取消回调
dispatch_source_set_cancel_handler(timer, ^{
    NSLog(@"Timer cancelled");
});

// 启动（默认是暂停状态）
dispatch_resume(timer);

// 暂停
dispatch_suspend(timer);

// 取消
dispatch_source_cancel(timer);

// ⚠️ suspend 和 resume 必须配对
// ⚠️ 不能对已 cancel 的 source 调 suspend
// ⚠️ 不能在 suspended 状态下释放 source（crash）
// 正确销毁：先 resume（如果 suspended），再 cancel

// ========== DispatchSource 文件监控 ==========
int fd = open("/path/to/file", O_EVTONLY);
dispatch_source_t fileSource = dispatch_source_create(DISPATCH_SOURCE_TYPE_VNODE,
                                                       fd,
                                                       DISPATCH_VNODE_WRITE | DISPATCH_VNODE_DELETE,
                                                       dispatch_get_main_queue());
dispatch_source_set_event_handler(fileSource, ^{
    unsigned long flags = dispatch_source_get_data(fileSource);
    if (flags & DISPATCH_VNODE_WRITE) NSLog(@"文件被修改");
    if (flags & DISPATCH_VNODE_DELETE) NSLog(@"文件被删除");
});
dispatch_resume(fileSource);
```

### 4.9 dispatch_barrier

```objc
// 栅栏函数：在并发队列中实现"多读单写"

dispatch_queue_t rwQueue = dispatch_queue_create("com.app.rw", DISPATCH_QUEUE_CONCURRENT);

// 读操作（并发执行）
- (id)readData {
    __block id result;
    dispatch_sync(rwQueue, ^{
        result = self->_data;
    });
    return result;
}

// 写操作（栅栏隔离：等前面的读都完成后执行，写完成后再允许后续的读）
- (void)writeData:(id)data {
    dispatch_barrier_async(rwQueue, ^{
        self->_data = data;
    });
}

// barrier 的执行时序：
// Read1 ─→│
// Read2 ─→│── barrier(Write) ──│─→ Read3
// Read3 ─→│                    │─→ Read4
// 等所有在 barrier 之前提交的任务完成后，独占执行 barrier 任务
// barrier 完成后，后续任务才可执行

// ⚠️ barrier 必须用在自定义并发队列上
// 用在全局并发队列上无效（全局队列被系统共享，不能独占）
```

### 4.10 dispatch_once

```objc
// 保证代码块只执行一次（线程安全）

static dispatch_once_t onceToken;
dispatch_once(&onceToken, ^{
    // 初始化代码，只执行一次
});

// 底层原理（简化）：
// 1. 首次调用：onceToken = 0，执行 Block
// 2. Block 执行中：onceToken = ~0（标记为"正在执行"），其他线程等待
// 3. Block 执行完：onceToken = ~0（标记为"已完成"），唤醒等待线程
// 4. 后续调用：onceToken = ~0，直接返回

// 单例模式
+ (instancetype)sharedInstance {
    static MyClass *instance;
    static dispatch_once_t onceToken;
    dispatch_once(&onceToken, ^{
        instance = [[MyClass alloc] init];
    });
    return instance;
}

// Swift 中 dispatch_once 被移除，使用 static let 替代
class MyClass {
    static let shared = MyClass()
    private init() {}
}
// static let 本身就是线程安全的（底层使用 dispatch_once）
```

### 4.11 dispatch_after

```objc
// 延迟执行（注意：是延迟将任务加入队列，不是延迟执行）
dispatch_after(dispatch_time(DISPATCH_TIME_NOW, (int64_t)(2.0 * NSEC_PER_SEC)),
               dispatch_get_main_queue(), ^{
    NSLog(@"2 秒后执行");
});

// Swift
DispatchQueue.main.asyncAfter(deadline: .now() + 2.0) {
    print("2 秒后执行")
}

// ⚠️ dispatch_after 不能取消
// 如需可取消的延迟执行，使用 DispatchWorkItem
let item = DispatchWorkItem { print("延迟任务") }
DispatchQueue.main.asyncAfter(deadline: .now() + 2.0, execute: item)
item.cancel()  // 取消
```

### 4.12 dispatch_apply

```objc
// 并发迭代（类似并发版 for 循环）
dispatch_apply(100, dispatch_get_global_queue(0, 0), ^(size_t i) {
    NSLog(@"index: %zu", i);
});
// 同步等待所有迭代完成后才返回
// 各迭代可并发执行，但返回时所有迭代都已完成

// Swift
DispatchQueue.concurrentPerform(iterations: 100) { i in
    print("index: \(i)")
}
```

### 4.13 Target Queue

```objc
// Target Queue：设置队列的目标队列，影响任务的执行环境

dispatch_queue_t q1 = dispatch_queue_create("q1", DISPATCH_QUEUE_SERIAL);
dispatch_queue_t q2 = dispatch_queue_create("q2", DISPATCH_QUEUE_SERIAL);
dispatch_queue_t targetQ = dispatch_queue_create("target", DISPATCH_QUEUE_SERIAL);

// 将 q1 和 q2 的 target 设置为同一个串行队列
dispatch_set_target_queue(q1, targetQ);
dispatch_set_target_queue(q2, targetQ);

// 可以实现多个队列的任务串行化

// 通过 target queue 调整
dispatch_set_target_queue(myQueue,
    dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0));
// myQueue 的任务将以 BACKGROUND 优先级执行
```

## 五、GCD 底层原理

### 5.1 线程池

```
GCD 维护一个全局线程池：

  ┌─────────────────────────────────────┐
  │           GCD 线程池                 │
  │                                     │
  │  ┌────────┐  ┌────────┐            │
  │  │Thread 1│  │Thread 2│  ...       │
  │  └───┬────┘  └───┬────┘            │
  │      │           │                  │
  │  ┌───▼──────────▼───────────┐      │
  │  │      Work Queue          │      │
  │  │  Task → Task → Task ...  │      │
  │  └──────────────────────────┘      │
  └─────────────────────────────────────┘

工作原理：
1. 异步派发的 Block 被加入队列
2. GCD 从线程池取一个空闲线程执行任务
3. 任务完成后线程归还线程池
4. 如果没有空闲线程，GCD 根据系统资源决定是否创建新线程
5. 空闲线程超时后被销毁

线程数量：
  - GCD 不保证固定线程数
  - 默认最大 64 个工作线程（非硬性限制）
  - 过多线程 → 上下文切换开销大 → 性能下降
  - 最佳实践：控制并发数量（DispatchSemaphore / OperationQueue.maxConcurrentOperationCount）
```

### 5.2 dispatch_queue 的数据结构

```c
// 简化的 dispatch_queue 结构
struct dispatch_queue_s {
    // 基类信息（引用计数、vtable 等）
    struct dispatch_object_s _as_do[0];

    const char *dq_label;       // 队列标签
    uint32_t dq_atomic_flags;   // 原子标志位
    uint32_t dq_width;          // 宽度：1=串行，UINT32_MAX=并发
    uint16_t dq_priority;       // 优先级
    dispatch_queue_t dq_target;  // 目标队列

    // 任务链表
    struct dispatch_object_s *dq_items_head;
    struct dispatch_object_s *dq_items_tail;
};

// 串行 vs 并发的区别就是 dq_width：
// DISPATCH_QUEUE_SERIAL     → dq_width = 1
// DISPATCH_QUEUE_CONCURRENT → dq_width = UINT32_MAX
```

### 5.3 QoS（Quality of Service）

```objc
// QoS 优先级从高到低：
// userInteractive > userInitiated > default > utility > background > unspecified

// userInteractive：与用户交互直接相关（动画、事件处理）
// userInitiated：用户发起的需要立即结果的操作
// default：默认
// utility：长时间运行的操作（下载、导入）
// background：用户不可见的操作（备份、同步）

// 使用
dispatch_queue_attr_t attr = dispatch_queue_attr_make_with_qos_class(
    DISPATCH_QUEUE_SERIAL,
    QOS_CLASS_USER_INITIATED,
    0
);
dispatch_queue_t queue = dispatch_queue_create("com.app.q", attr);

// Swift
let queue = DispatchQueue(label: "com.app.q",
                          qos: .userInitiated,
                          attributes: .concurrent)

// ⚠️ 优先级反转：
// 低优先级线程持有锁，高优先级线程等待锁
// 系统可能会临时提升低优先级线程的优先级（Priority Inheritance）
// GCD 的 QoS 系统自动处理大部分反转问题
```

### 6.1 基本用法

### 6.2 NSOperationQueue

```objc
// 创建队列
NSOperationQueue *queue = [[NSOperationQueue alloc] init];
queue.name = @"com.app.download";

// 设置最大并发数
queue.maxConcurrentOperationCount = 3;
// 1 = 串行队列
// NSOperationQueueDefaultMaxConcurrentOperationCount = -1 = 系统决定

[queue addOperationWithBlock:^{
    NSLog(@"简单 Block 任务");

queue.suspended = NO;   // 恢复

// ⚠️ 只是给每个 op 设置 cancelled 标记，op 的 main 方法需要自己检查
```

### 6.3 依赖管理

```objc
// Operation 之间可以添加依赖，控制执行顺序

NSBlockOperation *download = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"1. 下载数据");
}];
NSBlockOperation *parse = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"2. 解析数据");
}];
NSBlockOperation *save = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"3. 保存到数据库");
}];
NSBlockOperation *updateUI = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"4. 更新 UI");
}];

// 设置依赖
[parse addDependency:download];      // parse 依赖 download
[save addDependency:parse];          // save 依赖 parse
[updateUI addDependency:save];       // updateUI 依赖 save

// 加入队列（执行顺序由依赖决定）
NSOperationQueue *queue = [[NSOperationQueue alloc] init];
[queue addOperations:@[save, download, updateUI, parse] waitUntilFinished:NO];
// 实际执行顺序：download → parse → save → updateUI

// 移除依赖
[parse removeDependency:download];

// ⚠️ 不要创建循环依赖（A→B→A），会导致所有相关 op 永远不执行

// 依赖可以跨队列
NSOperationQueue *queue1 = [NSOperationQueue new];
NSOperationQueue *queue2 = [NSOperationQueue new];
[op2 addDependency:op1];   // op1 在 queue1，op2 在 queue2
[queue1 addOperation:op1];
[queue2 addOperation:op2];  // op2 会等 op1 完成后执行
```

### 6.4 自定义异步 Operation

```objc
// 重写 main 只适合同步任务，异步任务需要重写更多方法

@interface AsyncOperation : NSOperation
@property (nonatomic, assign, getter=isExecuting) BOOL executing;
@property (nonatomic, assign, getter=isFinished) BOOL finished;
@end

@implementation AsyncOperation

// 需要手动发送 KVO 通知
@synthesize executing = _executing;
@synthesize finished = _finished;

- (void)start {
    if (self.isCancelled) {
        self.finished = YES;
        return;
    }
    self.executing = YES;

    // 异步操作
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        // 模拟耗时操作
        [NSThread sleepForTimeInterval:2.0];

        // 完成
        self.executing = NO;
        self.finished = YES;
    });
}

// 必须手动发 KVO
- (void)setExecuting:(BOOL)executing {
    [self willChangeValueForKey:@"isExecuting"];
    _executing = executing;
    [self didChangeValueForKey:@"isExecuting"];
}

- (void)setFinished:(BOOL)finished {
    [self willChangeValueForKey:@"isFinished"];
    _finished = finished;
    [self didChangeValueForKey:@"isFinished"];
}

// 标记为异步操作
- (BOOL)isAsynchronous {
    return YES;
}

@end

// ⚠️ 必须通过 KVO 通知 isExecuting 和 isFinished 的变化
// 否则 NSOperationQueue 无法知道任务状态，依赖管理会失效
```

### 6.5 NSOperation vs GCD

```
┌──────────────────────┬──────────────────────┐
│     NSOperation      │         GCD           │
├──────────────────────┼──────────────────────┤
│ 面向对象，ObjC 接口  │ C 语言接口            │
│ 支持依赖关系         │ 不直接支持依赖        │
│ 支持取消（cancel）   │ 取消不方便            │
│ 支持暂停/恢复队列    │ 不支持暂停队列        │
│ 可设置最大并发数     │ 需信号量控制          │
│ 支持优先级           │ 支持 QoS              │
│ 可监控状态（KVO）    │ 无状态监控            │
│ 适合复杂任务编排     │ 适合简单并发任务      │
│ 性能稍低（OC 开销）  │ 性能更高（C 接口）    │
└──────────────────────┴──────────────────────┘

选择建议：
  简单异步任务 → GCD
  需要依赖/取消/暂停等 → NSOperation
  高性能场景 → GCD
```

## 七、锁

### 7.1 锁的分类

```
按功能分：
  互斥锁（Mutex）：同一时刻只有一个线程进入临界区
  递归锁（Recursive Mutex）：同一线程可多次加锁
  读写锁（RWLock）：多读单写
  条件锁（Condition）：条件满足时唤醒
  自旋锁（Spinlock）：忙等，不让出 CPU

按行为分：
  阻塞锁（Blocking）：获取失败 → 线程休眠，等待唤醒
  自旋锁（Spinning）：获取失败 → 循环忙等，直到获得锁
```

### 7.2 各种锁的用法与性能

```objc
// ========== 1. os_unfair_lock（推荐，最快的互斥锁）==========
#import <os/lock.h>

os_unfair_lock lock = OS_UNFAIR_LOCK_INIT;

os_unfair_lock_lock(&lock);
// 临界区...
os_unfair_lock_unlock(&lock);

// 尝试加锁
if (os_unfair_lock_trylock(&lock)) {
    // 获得锁
    os_unfair_lock_unlock(&lock);
}
// ⚠️ 不可递归，同一线程重复加锁会死锁
// ⚠️ 替代已废弃的 OSSpinLock


// ========== 2. NSLock ==========
NSLock *nsLock = [[NSLock alloc] init];

[nsLock lock];
// 临界区...
[nsLock unlock];

// 尝试加锁
if ([nsLock tryLock]) {
    // 获得锁
    [nsLock unlock];
}

// 超时加锁
if ([nsLock lockBeforeDate:[NSDate dateWithTimeIntervalSinceNow:3.0]]) {
    // 3 秒内获得锁
    [nsLock unlock];
}
// ⚠️ 不可递归


// ========== 3. NSRecursiveLock（递归锁）==========
NSRecursiveLock *recursiveLock = [[NSRecursiveLock alloc] init];

// 同一线程可多次加锁
[recursiveLock lock];
[recursiveLock lock];   // 不死锁（递归锁计数 +1）
// 临界区...
[recursiveLock unlock]; // 计数 -1
[recursiveLock unlock]; // 计数 0，真正释放
// 适合递归函数中需要加锁的场景


// ========== 4. NSCondition ==========
NSCondition *condition = [[NSCondition alloc] init];

// 生产者
[condition lock];
// 生产数据...
[condition signal];     // 唤醒一个等待线程
// [condition broadcast]; // 唤醒所有等待线程
[condition unlock];

// 消费者
[condition lock];
while (/* 数据未就绪 */) {
    [condition wait];   // 等待并释放锁，被唤醒后重新获得锁
}
// 消费数据...
[condition unlock];


// ========== 5. NSConditionLock ==========
NSConditionLock *condLock = [[NSConditionLock alloc] initWithCondition:0];

// 线程1：等待条件值为 0 时执行
[condLock lockWhenCondition:0];
NSLog(@"Step 1");
[condLock unlockWithCondition:1];  // 解锁并设条件值为 1

// 线程2：等待条件值为 1 时执行
[condLock lockWhenCondition:1];
NSLog(@"Step 2");
[condLock unlockWithCondition:2];

// 线程3：等待条件值为 2 时执行
[condLock lockWhenCondition:2];
NSLog(@"Step 3");
[condLock unlock];
// 效果：保证执行顺序 Step1 → Step2 → Step3


// ========== 6. dispatch_semaphore（当值为1时等价互斥锁）==========
dispatch_semaphore_t sema = dispatch_semaphore_create(1);

dispatch_semaphore_wait(sema, DISPATCH_TIME_FOREVER);  // 加锁
// 临界区...
dispatch_semaphore_signal(sema);  // 解锁


// ========== 7. @synchronized ==========
@synchronized (self) {
    // 临界区
}
// 底层：objc_sync_enter / objc_sync_exit
// 基于 SyncList 哈希表 + pthread_mutex 实现
// ⚠️ 性能较差，但使用最简单
// ⚠️ 可递归（同一线程重复加锁不死锁）
// ⚠️ 传入的对象不能为 nil（否则不起作用）
```

### 7.3 @synchronized 的底层实现

```c
// @synchronized(obj) { ... }
// 编译器翻译为：
objc_sync_enter(obj);
// 临界区
objc_sync_exit(obj);

// objc_sync_enter 的实现：
// 1. 以 obj 的地址为 key，在全局哈希表 SyncList 中查找
// 2. 找到 → 取出对应的 SyncData（包含 pthread_mutex）
// 3. 未找到 → 创建 SyncData，存入 SyncList
// 4. 对 SyncData 中的 pthread_mutex 加锁

// 数据结构：
struct SyncData {
    SyncData *nextData;           // 链表
    DisguisedPtr<objc_object> object;  // 锁对象
    int threadCount;               // 使用此锁的线程数
    recursive_mutex_t mutex;       // 递归互斥锁
};

// 全局哈希表
struct SyncList {
    SyncData *data;
    spinlock_t lock;   // 保护哈希桶的自旋锁
};

static SyncList sDataLists[LOCK_FOR_OBJ_COUNT];  // 默认 16 个桶

// 哈希算法：(uintptr_t)obj % LOCK_FOR_OBJ_COUNT
```

### 7.4 OSSpinLock 为什么被废弃

```
OSSpinLock 的问题：优先级反转

场景：
1. 低优先级线程 T1 获得锁，进入临界区
2. 高优先级线程 T2 请求锁，自旋等待
3. T2 优先级高 → CPU 持续调度 T2（忙等）
4. T1 优先级低 → 得不到 CPU 时间 → 无法完成并释放锁
5. T2 永远在自旋，T1 永远执行不到 → 类似死锁

os_unfair_lock 的解决方案：
  等待时不自旋，而是让出 CPU（线程休眠）
  系统追踪锁的持有者，确保锁的持有者能够执行
```

### 7.5 锁的性能对比

```
从快到慢（加锁+解锁一次的耗时，Apple A 系列芯片）：

  os_unfair_lock         ≈ 25 ns    最快，推荐
  dispatch_semaphore     ≈ 30 ns    接近 os_unfair_lock
  pthread_mutex          ≈ 35 ns    POSIX 标准互斥锁
  NSLock                 ≈ 40 ns    ObjC 封装的 pthread_mutex
  NSCondition            ≈ 45 ns
  NSRecursiveLock        ≈ 55 ns    递归锁开销略大
  NSConditionLock        ≈ 65 ns
  @synchronized          ≈ 100 ns   最慢（哈希查找 + 递归锁）

选择建议：
  一般场景     → os_unfair_lock 或 NSLock
  递归场景     → NSRecursiveLock 或 pthread_mutex(recursive)
  多读单写     → dispatch_barrier 或 pthread_rwlock
  条件等待     → NSCondition 或 pthread_cond
  简洁优先     → @synchronized（性能不敏感时）
  跨平台       → pthread_mutex
```

### 7.6 多读单写的实现方案

```objc
// ========== 方案1：dispatch_barrier（推荐）==========
@interface ThreadSafeDict : NSObject
@end

@implementation ThreadSafeDict {
    NSMutableDictionary *_dict;
    dispatch_queue_t _queue;
}

- (instancetype)init {
    self = [super init];
    if (self) {
        _dict = [NSMutableDictionary dictionary];
        _queue = dispatch_queue_create("com.app.rwQueue", DISPATCH_QUEUE_CONCURRENT);
    }
    return self;
}

- (id)objectForKey:(NSString *)key {
    __block id result;
    dispatch_sync(_queue, ^{
        result = self->_dict[key];
    });
    return result;
}

- (void)setObject:(id)obj forKey:(NSString *)key {
    dispatch_barrier_async(_queue, ^{
        self->_dict[key] = obj;
    });
}

@end


// ========== 方案2：pthread_rwlock ==========
@implementation ThreadSafeDict {
    NSMutableDictionary *_dict;
    pthread_rwlock_t _rwlock;
}

- (instancetype)init {
    self = [super init];
    if (self) {
        _dict = [NSMutableDictionary dictionary];
        pthread_rwlock_init(&_rwlock, NULL);
    }
    return self;
}

- (void)dealloc {
    pthread_rwlock_destroy(&_rwlock);
}

- (id)objectForKey:(NSString *)key {
    pthread_rwlock_rdlock(&_rwlock);  // 读锁
    id result = _dict[key];
    pthread_rwlock_unlock(&_rwlock);
    return result;
}

- (void)setObject:(id)obj forKey:(NSString *)key {
    pthread_rwlock_wrlock(&_rwlock);  // 写锁
    _dict[key] = obj;
    pthread_rwlock_unlock(&_rwlock);
}

@end
```

## 八、RunLoop 与线程

### 8.1 RunLoop 概念

```
RunLoop 是一个事件处理循环，用于管理线程的事件/消息：

  while (alive) {
      1. 通知 Observer：即将处理 Timer
      2. 通知 Observer：即将处理 Source0
      3. 处理 Source0
      4. 如果有 Source1 就绪 → 跳到 step 8
      5. 通知 Observer：即将休眠
      6. 休眠，等待唤醒
         - Source1（端口事件）
         - Timer 到期
         - 外部手动唤醒
      7. 通知 Observer：被唤醒
      8. 处理唤醒时的事件
         - Timer → 处理定时器回调
         - Source1 → 处理端口事件
         - dispatch_get_main_queue → 处理 GCD 到主队列的任务
      9. 通知 Observer：即将退出循环
  }
```

### 8.2 RunLoop Mode

```objc
// 每个 RunLoop 可以有多个 Mode，但同一时刻只运行在一个 Mode 下

// 常用 Mode：
// NSDefaultRunLoopMode（kCFRunLoopDefaultMode）
//   默认模式，大部分时间处于此模式

// UITrackingRunLoopMode
//   ScrollView 滑动时切换到此模式

// NSRunLoopCommonModes（kCFRunLoopCommonModes）
//   不是真正的 Mode，而是一个标记
//   被标记为 common 的 Mode 都会执行（Default + Tracking）

// 经典问题：Timer 在滑动时停止
NSTimer *timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                                  target:self
                                                selector:@selector(tick)
                                                userInfo:nil
                                                 repeats:YES];
// 默认加入 NSDefaultRunLoopMode
// 滑动 ScrollView 时 RunLoop 切到 UITrackingRunLoopMode → Timer 暂停

// 解决：加入 CommonModes
[[NSRunLoop mainRunLoop] addTimer:timer forMode:NSRunLoopCommonModes];
// 在 Default 和 Tracking 模式下都会触发

// ⚠️ GCD 定时器（dispatch_source_timer）不受 RunLoop Mode 影响
```

### 8.3 RunLoop 与线程的关系

```
1. 主线程 RunLoop 在 App 启动时自动创建并运行
2. 子线程默认没有 RunLoop
3. RunLoop 在首次获取时创建（懒加载）：[NSRunLoop currentRunLoop]
4. RunLoop 与线程一一对应（存储在全局 Dictionary 中）
5. RunLoop 在线程销毁时销毁

子线程开启 RunLoop 的场景：
  - 需要使用 Timer
  - 需要使用 performSelector:onThread:
  - 需要监听端口
  - 需要线程常驻

开启子线程 RunLoop：
  NSRunLoop *loop = [NSRunLoop currentRunLoop];  // 创建
  [loop addPort:[NSMachPort port] forMode:NSDefaultRunLoopMode];  // 添加源
  [loop run];  // 启动（会阻塞）

  // ⚠️ [NSRunLoop run] 无法停止
  // 如需控制退出，使用 runMode:beforeDate:
  BOOL shouldKeepRunning = YES;
  while (shouldKeepRunning) {
      [[NSRunLoop currentRunLoop] runMode:NSDefaultRunLoopMode
                               beforeDate:[NSDate distantFuture]];
  }
  // 将 shouldKeepRunning 设为 NO 即可退出
```

```objc
            // 记录时间戳 T2
```

### 9.3 TaskGroup

```swift
// TaskGroup：结构化并发，管理一组子任务

func fetchAllUsers(ids: [String]) async throws -> [User] {
    try await withThrowingTaskGroup(of: User.self) { group in
        for id in ids {
            group.addTask {
                try await fetchUser(id: id)
            }
        }

        var users: [User] = []
        for try await user in group {
            users.append(user)
        }
        return users
    }
}

// 限制并发数的 TaskGroup
func processItems(_ items: [Item]) async {
    await withTaskGroup(of: Void.self) { group in
        var runningTasks = 0
        let maxConcurrency = 5

        for item in items {
            if runningTasks >= maxConcurrency {
                await group.next()  // 等待一个任务完成
                runningTasks -= 1
            }
            group.addTask {
                await process(item)
            }
            runningTasks += 1
        }
        // 等待所有剩余任务
        await group.waitForAll()
    }
}

// ⚠️ TaskGroup 中的子任务取消会传播
// group 中任何一个子任务抛出错误 → 其他子任务自动取消
```

### 9.4 Actor

```swift
// Actor：数据隔离的引用类型，保证内部状态线程安全

actor BankAccount {
    let id: String
    private(set) var balance: Double

    init(id: String, balance: Double) {
        self.id = id
        self.balance = balance
    }

    func deposit(_ amount: Double) {
        balance += amount  // 同一时刻只有一个任务能访问
    }

    func withdraw(_ amount: Double) throws {
        guard balance >= amount else { throw BankError.insufficientFunds }
        balance -= amount
    }

    // nonisolated：不需要隔离的属性/方法
    nonisolated var description: String {
        return "Account \(id)"  // 只访问 let 属性，不需要隔离
    }
}

// 使用 Actor
let account = BankAccount(id: "001", balance: 1000)

// 外部访问必须用 await（跨隔离边界）
Task {
    await account.deposit(500)
    let balance = await account.balance
    print(balance)  // 1500
}

// ⚠️ Actor Reentrancy（重入性）
// await 挂起期间，其他任务可以进入 actor
// 即 actor 方法在 await 前后的状态可能不同
actor Counter {
    var value = 0

    func increment() async {
        let current = value     // 假设 value = 0
        await someAsyncWork()   // ⚠️ 挂起期间其他任务可能修改 value
        value = current + 1     // 可能覆盖其他修改！
    }

    // ✅ 更安全的写法
    func safeIncrement() {
        value += 1  // 没有 await，不会被中断
    }
}
```

### 9.8 Structured vs Unstructured Concurrency

```
结构化并发（Structured）：
  Task 有明确的父子关系，生命周期受父 Task 约束
  - async let
  - TaskGroup
  - 父任务取消 → 子任务自动取消
  - 父任务等待所有子任务完成

  async let a = fetchA()
  async let b = fetchB()
  let result = await (a, b)  // 自动等待，自动取消

非结构化并发（Unstructured）：
  Task 独立存在，需要手动管理生命周期
  - Task { ... }
  - Task.detached { ... }
  - 需要手动取消
  - 需要手动等待

  let task = Task { await fetchData() }
  task.cancel()  // 手动取消

选择原则：
  能用结构化就用结构化（更安全、更易管理）
  需要独立生命周期时才用非结构化
```

## 十、线程安全

### 10.1 线程安全问题

```objc
// ========== 问题1：数据竞争（Data Race）==========
// 多个线程同时读写同一块内存

@property (nonatomic, assign) NSInteger count;

// 线程1          线程2
// read count=0   read count=0
// count+1        count+1
// write count=1  write count=1
// 期望 count=2，实际 count=1

// ========== 问题2：集合读写冲突 ==========
// 线程1 遍历数组
for (id obj in self.array) { ... }

// 线程2 同时修改数组
[self.array addObject:newObj];
// 崩溃！__NSArrayM mutation while being enumerated

// ========== 问题3：属性非原子性 ==========
// nonatomic 属性在多线程写入时可能产生中间状态
// 例如 64 位值在 32 位 CPU 上的非原子写入
```

### 10.2 atomic vs nonatomic

```objc
// atomic（默认）：
// - setter/getter 加锁（os_unfair_lock）
// - 保证读写操作的原子性
// - 不能保证线程安全（只保证单次读写完整）

@property (atomic, strong) NSArray *data;

// 线程安全问题仍然存在：
// 线程1: self.data = @[@1, @2];
// 线程2: NSArray *copy = self.data;  // 原子读取
// 线程3: [self.data count];          // 读到的可能是旧值
// 线程1: self.data = @[@3];          // 原子写入
// 但 线程2 读取和 线程3 使用之间不是原子的

// nonatomic：
// - 不加锁
// - 性能更好
// - iOS 开发中几乎都用 nonatomic（自行保证线程安全）

// 结论：atomic 不等于线程安全，只保证读写操作不被撕裂
```

### 10.3 线程安全的集合

```objc

    [self.lock unlock];
}

    [self.lock lock];
    NSArray *copy = [self.array copy];
    [self.lock unlock];
    return copy;
}

// 方案2：dispatch_barrier（推荐，见 7.6 节）

// 方案3：使用线程安全的数据结构
// Swift Actor

    private var storage: [T] = []

        storage.append(item)
    }

    func getAll() -> [T] {
        return storage
    }
}
```

## 十二、常见面试题

### Q1: GCD 的队列和线程是什么关系？

```
队列是任务的组织方式，线程是任务的执行载体。
队列决定任务的调度策略（FIFO、串行/并发），
GCD 从内部线程池分配线程来执行队列中的任务。

一个串行队列的任务可能在不同线程上执行（但同一时刻只有一个）。
一个并发队列的任务可以在多个线程上同时执行。
同一个线程可以为不同队列服务。

主队列是特殊的串行队列，绑定到主线程。
```

### Q2: dispatch_sync 在什么情况下会死锁？

```
在串行队列 A 正在执行的任务中，同步派发到队列 A 自身。

因为 sync 等待 Block 执行完才返回，
但 Block 被加入串行队列 A 的尾部，需要等当前任务完成，
当前任务又在等 Block → 互相等待 → 死锁。

典型场景：
1. 主线程 sync 到主队列
2. 串行队列任务中 sync 到同一个串行队列

并发队列不会死锁（因为允许同时执行多个任务）。
```

### Q3: NSOperation 和 GCD 的区别，如何选择？

```
NSOperation 优势：
  支持任务依赖、取消、暂停/恢复、状态监控（KVO）、最大并发数
  适合复杂任务编排

GCD 优势：
  API 简洁、性能更高（C 接口）、功能覆盖面广
  栅栏、信号量、dispatch_group、dispatch_source

选择：
  简单异步 → GCD
  需要依赖/取消/暂停 → NSOperation
  定时器 → GCD dispatch_source
  多读单写 → GCD dispatch_barrier
```

### Q4: 如何保证多线程下的属性安全？

```
方案1：atomic（不推荐，性能差且不完全安全）
方案2：加锁（NSLock / os_unfair_lock / @synchronized）
方案3：串行队列 + sync/async
方案4：dispatch_barrier（多读单写场景）
方案5：Swift Actor（推荐）

最佳实践：
  不可变数据 → 不需要加锁（线程安全）
  读多写少 → dispatch_barrier
  读写频率相当 → 互斥锁
  Swift 代码 → Actor
```

### Q5: Swift Concurrency 相比 GCD 有什么优势？

```
1. 编译时安全：
   Sendable 协议在编译时检查数据竞争
   Actor 隔离在编译时保证数据安全

2. 结构化并发：
   TaskGroup 自动管理子任务的取消和等待
   不会遗漏清理工作

3. 不阻塞线程：
   await 挂起任务释放线程，而非阻塞线程
   减少线程数量，减少上下文切换

4. 可读性：
   async/await 线性代码，避免回调地狱
   代码逻辑更清晰

5. 协作式取消：
   Task 取消自动传播到子任务
   withTaskCancellationHandler 优雅处理取消
```

### Q6: 什么是优先级反转？如何避免？

```
优先级反转：
  低优先级线程持有锁
  高优先级线程等待锁
  中优先级线程抢占低优先级线程的 CPU
  → 高优先级线程被间接阻塞

解决方案：
1. 优先级继承（Priority Inheritance）：
   系统临时提升持有锁的低优先级线程的优先级
   os_unfair_lock / pthread_mutex 已内置此机制

2. 优先级天花板（Priority Ceiling）：
   锁的优先级设为所有可能使用者中最高的
   获取锁时线程优先级提升到天花板

3. 避免方案：
   减少锁的持有时间
   使用无锁数据结构
   使用 Actor 模型（Swift Concurrency）
   使用 dispatch_barrier（不涉及锁）
```

### Q8: GCD dispatch queue 内部机制（workqueue 与线程池）

GCD 的底层实现依赖于 **workqueue**（Linux cgroups + pthread 线程池），而非简单的固定线程池。

- **dispatch_apply**：使用线程池内线程同步等待结果，适合批量任务
- **dispatch_async**：异步分发到 workqueue，线程按需伸缩
- **dispatch_group**：基于计数器实现，不阻塞执行线程

```objc
// dispatch_apply：所有迭代完成后才继续（阻塞）
dispatch_queue_t q = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0);
dispatch_apply(10, q, ^(size_t index) {
    NSLog(@"并行处理 index=%zu", index);
});
// 这里要等所有 10 次迭代完成才执行

// dispatch_group：异步等待（不阻塞执行线程）
dispatch_group_t group = dispatch_group_create();
for (int i = 0; i < 10; i++) {
    dispatch_group_async(group, q, ^{
        NSLog(@"异步任务 %d", i);
    });
}
dispatch_group_notify(group, dispatch_get_main_queue(), ^{
    NSLog(@"所有任务完成");  // 回到主线程
});
```

**Pitfalls**: dispatch_apply 会阻塞当前线程且隐式创建 barrier，在 serial queue 上使用等于串行化；高并发场景下 thread contention 反而比简单循环更慢。

**Best Practice**: 优先用 dispatch_group + dispatch_group_notify 替代 dispatch_apply，保持非阻塞语义；批量 IO 或 CPU 密集型小任务才考虑 dispatch_apply。

### Q10: QoS 级别与自动优先级反转

GCD 定义了 7 个 Quality of Service (QoS) 层级（iOS 8+），从最高到最低依次为：user-interactive、user-initiated、utility、background、unspecified、overcommit（私有）、default。每个层级映射到 POSIX thread priority 和 Mach thread priority。

**自动优先级反转**：当低优先级任务持有所需资源，而高优先级任务通过 dispatch_barrier_async 或 serial queue 间接依赖该资源时，GCD 会自动提升低优先级任务的运行优先级，直到它释放资源。这是 GCD 内置的 deadlock prevention 机制。

```objc
// QoS 属性设置（推荐方式）
dispatch_queue_t bg = dispatch_queue_create("com.my.bg", DISPATCH_QUEUE_SERIAL);
dispatch_set_target_queue(bg, dispatch_get_global_queue(QOS_CLASS_BACKGROUND, 0));

// block 级别的 QoS 作用域 — dispatch_get_current_qos_class()
dispatch_async(bg, ^{
    dispatch_qos_class_t qos = dispatch_get_current_qos_class();
    NSLog(@"当前 QoS: %d", qos);  // QOS_CLASS_BACKGROUND
});

// 高优先级任务自动提升
dispatch_async(dispatch_get_global_queue(QOS_CLASS_USER_INITIATED, 0), ^{
    dispatch_qos_class_t high = dispatch_get_current_qos_class();
    NSLog(@"高优先级 QoS: %d", high);  // QOS_CLASS_USER_INITIATED
});
```

**Pitfalls**: QoS 提升仅在当前 dispatch hierarchy 内生效，跨 process 或手动 pthread 不受影响；overcommit 是私有 API，使用会导致 App Store 审核拒绝。

**Best Practice**: 始终使用 `dispatch_set_target_queue` 而非 `DISPATCH_QUEUE_PRIORITY_HIGH`（已废弃）；IO 操作默认跟随主 QoS，如需后台下载应显式设 target queue 为 background。


### Q11: Dispatch workloop（dispatch_workloop_create）

Dispatch workloop 是 iOS 13+ 引入的私有 GCD 扩展，允许创建独立于系统 global queue 的工作循环。与标准的 dispatch_queue 不同，workloop 绑定到专属线程，避免在 burst 场景下被全局线程池调度延迟影响。`dispatch_workloop_create` 返回一个 serial queue 接口，其内部维护固定的 worker thread，适合对延迟敏感的任务（如实时音频处理、H.264 解码流水线）。

```objc
// 私有 API，App Store 审核会被拒 —— 仅内部工具使用
void *workloop = dispatch_workloop_create("com.my.workloop", NULL);
// workloop 作为 serial queue 使用
dispatch_async((dispatch_queue_t)workloop, ^{
    // 固定线程执行，延迟可预测
});
dispatch_sync((dispatch_queue_t)workloop, ^{
    // 同步任务也走同一工作线程
});
```

**Pitfalls**: workloop 是私有 API，使用会被 App Store 拒绝；且不会自动适配 QoS，高负载时不会像 global queue 那样动态伸缩线程数，可能造成线程饥饿。

**Best Practice**: 生产代码使用 `dispatch_queue_create` + `dispatch_set_target_queue` + 专用全局 queue 来模拟 workloop 效果；仅在内部工具或测试中使用真正的 workloop。

### Group Barriers 与 dispatch_barrier_async

dispatch_barrier_async 在 concurrent queue 上插入一个 "屏障"：它必须等待之前提交的所有任务完成后才执行，且在其执行期间，队列中后续的任务也会被阻塞。这使得 barrier 能够实现对共享资源的安全读写，无需显式使用锁。结合 dispatch_group，可以在并发写入后自动触发 barrier 进行一致性检查或合并操作。

```objc
dispatch_queue_t queue = dispatch_queue_create("com.my.concurrent", DISPATCH_QUEUE_CONCURRENT);

// 多个线程并发读
dispatch_async(queue, ^{ NSLog(@"读任务 1"); });
dispatch_async(queue, ^{ NSLog(@"读任务 2"); });

// Barrier 等待上述读任务完成，独占队列执行写操作
dispatch_barrier_async(queue, ^{
    NSLog(@"写任务 — 此时无并发读取");  // 安全写入共享资源
});

// 后续任务在 barrier 完成后恢复并发
dispatch_async(queue, ^{ NSLog(@"读任务 3"); });

// 配合 dispatch_group 实现批量写后自动合并
dispatch_group_t group = dispatch_group_create();
dispatch_group_async(group, queue, ^{ NSLog(@"写 A"); });
dispatch_group_async(group, queue, ^{ NSLog(@"写 B"); });
dispatch_group_notify(group, queue, ^{
    // 此处非 barrier，但可在 serial queue 上做合并
    NSLog(@"合并 A + B 的结果");
});
```

**Pitfalls**: barrier 仅在 concurrent queue 上有效，在 serial queue 上退化为普通 dispatch_async；不要将 barrier 用在 global concurrent queue 上，其行为不可预期且可能导致性能劣化。

**Best Practice**: 读写分离场景下，读用 dispatch_async（并发），写用 dispatch_barrier_async（独占），用同一 concurrent queue 实现 lock-free 的 Reader-Writer 模式；私有队列通过 dispatch_queue_create 显式声明为并发。

### Dispatch IO（GCD 文件 I/O）

Dispatch IO 是 GCD 提供的基于 file descriptor 的异步文件读写接口，通过 `dispatch_io_create` 创建 channel，支持 STREAM 模式（顺序读写）和 RANDOM 区域（随机访问）。底层使用 `kqueue` + `read`/`write` 系统调用，避免阻塞调用线程，特别适合大文件传输场景。

```objc
int fd = open("/path/to/file.dat", O_RDONLY);
dispatch_io_t channel = dispatch_io_create(DISPATCH_IO_STREAM, fd,
    dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0), ^(int error) {
    if (!error) close(fd);
});
dispatch_io_read(channel, 0, 4096, ^(bool done, dispatch_data_t data, int error) {
    if (!error && data) {
        size_t len = (size_t)dispatch_data_get_size(data);
        uint8_t *buf = malloc(len);
        dispatch_data_copy_region(data, 0, len, buf);
        // 处理数据...
        free(buf);
    }
    if (done) dispatch_release(channel);
});
```

**Pitfalls**: file descriptor 在 channel finalize 回调结束后不会自动关闭，需在 finalize block 中手动 `close(fd)`，否则导致 fd 泄漏；`dispatch_io` 不支持直接传入文件路径，必须先 `open()` 获取 fd。

**Best Practice**: 大文件传输优先用 `DISPATCH_IO_STREAM` 配合 `dispatch_io_read/write` 分块处理；小文件读取仍用 `NSData(contentsOfFile:)` 更简洁；channel 不再使用时必须 `dispatch_release` 触发 finalize block 清理 fd。

### Dispatch Source（定时器、信号、描述符、进程）

Dispatch Source 是 GCD 底层的事件监听机制，将 OS 事件（定时器、信号、file descriptor、进程状态、mach port、VFS 节点）统一抽象为 `dispatch_source_t`，在指定 queue 上以 block 回调驱动。相比 `NSTimer`，`dispatch_source_set_timer` 基于 monotonic clock，不受系统时间回拨或挂钟调整影响。相比 `NSFileHandle`，Dispatch Source 支持直接监听 `read`/`write`/`error` 事件且不会阻塞线程。

```objc
// 定时器示例：500ms 周期，100ms 容差
dispatch_source_t timer = dispatch_source_create(DISPATCH_SOURCE_TYPE_TIMER, 0, 0,
    dispatch_get_main_queue());
dispatch_source_set_timer(timer, dispatch_time(DISPATCH_TIME_NOW, 0),
    500 * NSEC_PER_SEC, 100 * NSEC_PER_SEC);
dispatch_source_set_event_handler(timer, ^{
    NSLog(@"tick");  // 每 ~500ms 触发
});
dispatch_resume(timer);  // 创建时 suspend，必须手动 resume
// ... 不需要时 dispatch_source_cancel(timer);

// 信号示例：监听 SIGINT
dispatch_source_t signal = dispatch_source_create(DISPATCH_SOURCE_TYPE_SIGNAL, SIGINT, 0,
    dispatch_get_main_queue());
dispatch_source_set_event_handler(signal, ^{
    NSLog(@"收到 SIGINT");
});
dispatch_resume(signal);
```

**Pitfalls**: dispatch_source_create 返回后处于 SUSPENDED 状态，忘记 dispatch_resume 导致回调永不触发；多次调用 dispatch_resume 不会重置挂起计数；signal type 会 sigprocmask(SIG_BLOCK)，导致传统信号处理机制失效。

**Best Practice**: 定时器场景优先用 dispatch_source 而非 NSTimer（不受 RunLoop 限制且 monotonic clock 更可靠）；file descriptor 监听用 DISPATCH_SOURCE_TYPE_READ/WRITE 替代 CFReadStream/CFWriteStream 获得更好的性能；用 dispatch_source_cancel() 而非 dispatch_release() 来停止 source（cancel 会触发 cancel handler + 自动 release）。

### apply vs async 在 for 循环场景中的对比

`dispatch_apply` 是 GCD 提供的同步并行循环 API，它将迭代次数分片到并发 queue 的 worker 线程中执行，**调用线程阻塞直到所有迭代完成**。`dispatch_async` 在 for 循环中则是**异步派发每个迭代到 queue 后立即返回**，不等待完成。

```objc
// dispatch_apply — 同步等待，适合有依赖关系的结果聚合
__block NSUInteger total = 0;
dispatch_queue_t queue = dispatch_get_global_queue(QOS_CLASS_DEFAULT, 0);
dispatch_apply(100, queue, ^(size_t i) {
    @synchronized(self) { total += [self costlyCompute:i]; }
});
NSLog(@"total = %zu", total); // dispatch_apply 返回后才打印

// dispatch_async in for — 异步非阻塞，适合独立任务
for (NSUInteger i = 0; i < 100; i++) {
    dispatch_async(queue, ^{
        [self costlyCompute:i]; // 独立执行，无需聚合
    });
}
NSLog(@"已派发 100 个任务"); // 立即执行，不等完成
```

**Pitfalls**: `dispatch_apply` 迭代块是同步并行的，不能在里面再调 `dispatch_apply` 或 `dispatch_group_enter/leave` 会导致崩溃；`__block` 共享变量必须在迭代块内做同步保护（`@synchronized` 或 `os_unfair_lock`）。`dispatch_async` 循环派发时注意 queue 容量有限（默认 512），超过会阻塞循环线程。

**Best Practice**: 需要聚合结果 → `dispatch_apply`（阻塞等待）；独立异步任务 → `for + dispatch_async`（非阻塞）；需要回调通知完成 → `dispatch_group` 包裹 `dispatch_async` 循环，兼顾非阻塞和完成感知。

### Operation 依赖关系与取消传播

NSOperation 的核心优势之一是声明式依赖机制。通过 `addDependency(_:)` 可指定操作间的执行顺序，无需手动管理信号或回调。依赖关系形成 DAG（有向无环图），GCD 自动调度满足所有前置依赖的操作。

```swift
let fetch = BlockOperation { print("fetch data") }
let parse = BlockOperation { print("parse data") }
let display = BlockOperation { print("display result") }

parse.addDependency(fetch)      // fetch → parse
display.addDependency(parse)    // parse → display

queue.addOperations([fetch, parse, display], waitUntilFinished: false)

// 取消传播：上游取消时级联取消下游
if fetch.isCancelled {
    // 手动遍历依赖链取消
    for op in [parse, display] where !op.isCancelled { op.cancel() }
}
```

**Pitfalls**: `addDependency` 必须在添加到 queue 之前调用，否则可能已调度导致依赖失效；循环依赖（A→B→A）不会崩溃但相关操作永不执行，需人工检测；`isCancelled` 检查应在长期运行的 `main()` 方法中每步主动判断，系统不会自动中断正在执行的代码。

**Best Practice**: 用 `OperationQueue.isSuspended` 暂停整个 pipeline 而非逐个 cancel；复杂 DAG 用 `dependencies` 数组批量设置；需要取消传播时自定义 `NSOperation` 子类，在 `cancel()` 中遍历 `dependencies` 实现级联。

### OperationQueue 自定义（NSOperation 子类）

自定义 `NSOperation` 子类可实现精细的并发控制、状态追踪和可取消的异步操作。核心是重写 `main()` 方法（Swift）或 `start`（Objective-C），在 `main()` 中实现同步逻辑，OperationQueue 自动管理线程派发和依赖调度。

```swift
class DataFetchOperation: Operation {
    private(set) var progress: Double = 0
    private var task: URLSessionTask?
    
    override func main() {
        super.main()
        // 每次迭代检查取消
        guard !self.isCancelled else { self.progress = -1; return }
        
        let task = URLSession.shared.dataTask(with: url) { [weak self] data, _, error in
            guard let self = self, !self.isCancelled else { return }
            // 处理结果...
            self.progress = 1.0
        }
        self.task = task
        task.resume()
        
        // 等待异步完成（KVO 驱动完成状态）
        runLoop.run(until: Date(timeIntervalSinceNow: 30))
    }
    
    override func cancel() {
        task?.cancel()
        super.cancel()
    }
}
```

**Pitfalls**: `main()` 内异步回调不会自动标记 Operation 为 finished，需手动配合 `KVO` 通知 `isFinished`/`isExecuting` 或使用 `DispatchSemaphore`（但会阻塞 queue 线程，慎用）；直接重写 `start` 时需同时实现 `isAsynchronous = true` 并 KVO 通知 `isExecuting` 和 `isFinished`。

**Best Practice**: 优先重写 `main()` 实现同步阻塞逻辑（OperationQueue 自动管理生命周期）；异步操作使用 `KVO + isAsynchronous` 模式或封装为 `AsyncOperation` 基类；长时间运行操作定期检查 `isCancelled` 并支持 KVO 的 `progress` 属性绑定 UI。

## Swift 结构化并发（async/await、Task、TaskGroup）

Swift 5.5+ 引入的结构化并发模型取代了传统的 completion handler 和 GCD 回调地狱。`async/await` 让异步代码以同步风格书写，编译器在后台自动处理任务调度、取消传播和错误处理。`Task` 表示一个可取消的异步执行单元，`TaskGroup` 支持动态创建子任务并等待所有子任务完成。

```swift
// 基础 async/await — 无回调嵌套
func fetchUser(id: Int) async throws -> User {
    let (data, _) = try await URLSession.shared.data(from: url)
    return try JSONDecoder().decode(User.self, from: data)
}

// TaskGroup — 并行 fetch 多个用户
func fetchUsers(ids: [Int]) async throws -> [User] {
    try await withThrowingTaskGroup(of: User.self) { group in
        for id in ids {
            group.addTask { try await fetchUser(id: id) }
        }
        var users = [User]()
        for try await user in group { users.append(user) }
        return users.sorted { $0.id < $1.id }
    }
}

// 显式派发后台任务 + 取消传播
Task.detached {
    try await heavyComputation()
    await MainActor.run { ui.update() }
}
```

**Pitfalls**: `async` 函数只能从 `async` 上下文调用，不能在普通同步方法中直接调用（需用 `Task { }` 包装）；`TaskGroup` 中任意子任务抛出异常会立即取消所有其他子任务，需用 `do/catch` 包裹每个 `addTask` 防止一个失败导致全部中断；`detached` Task 继承父任务的优先级但不继承其取消状态，忘记手动取消会导致内存泄漏。

**Best Practice**: 顶层 UI 入口用 `@MainActor` 标注，业务逻辑层用 `isolated` 或无 actor 隔离的纯函数；`async let` 绑定实现隐式并行（比 TaskGroup 更简洁）；长时间后台任务用 `Task.detached` 并传入 `OperationQueue.current?.underlyingQueue` 控制优先级；始终在 `deinit` 前 `cancel()` 持有的 Task 引用。

## Swift Actor 深度解析（隔离状态、可重入性）

Swift Actor 是结构化并发模型中的隔离原语，通过编译器自动插入数据竞争保护。Actor 内部的属性默认是 `nonisolated` 安全的 —— 从外部访问时自动插入隐式 `await` 获取 actor 级别的串行化锁，从根本上消除数据竞争。Actor 支持 **可重入性**（reentrancy）：当 actor 执行到 `await` 点时，其他任务可进入该 actor，类似异步的 `@synchronized` 但无死锁风险。

**隔离机制**：Actor 维护一个内部队列，保证同一时刻只有一个任务执行 actor 内部代码。`nonisolated` 方法跳过隔离检查（需自行保证线程安全），`nonisolated(unsafe)` 则完全绕过（Swift 6+，需显式标注）。

**可重入性示例**：
```swift
actor SessionManager {
    private var token: String?
    
    func refreshToken() async throws {
        // await 点 —— 其他任务可进入 actor
        let newToken = try await api.fetchToken()
        // ← 此时若另一个任务调用 getHeader()，会排队等待
        //    返回后继续执行此处
        token = newToken
    }
    
    func getHeader() -> String {
        token?.map { "Bearer \($0)" } ?? ""  // 同步访问，自动隔离
    }
}
```

**Pitfalls**: `await` 在 actor 方法内部不会释放隔离锁（与 Mutex.unlock 不同），但允许其他 actor 任务排队进入 —— 错误地使用 `nonisolated(unsafe)` 访问 actor 属性会导致数据竞争；`MainActor` 是全局 actor，不应在 actor 内部直接 `await MainActor.run {}`（循环等待导致死锁）。

**Best Practice**: 纯计算逻辑用 `nonisolated` 方法标注提升性能；需要跨 actor 协作时优先使用 `async let` 并行而非 `Task` 手动派发；Actor 间通信通过返回值传递数据，避免共享可变状态；长时间后台任务脱离 actor 上下文执行（`Task.detached`），完成后再 `await` 写入 actor 状态。

### async let 并行任务

`async let` 是 Swift 结构化并发中最简洁的并行原语。多个 `async let` 绑定在声明时立即启动各自的异步任务，所有任务并行执行，直到 `await` 取值时才同步等待结果。相比 `TaskGroup`，`async let` 无需手动管理任务生命周期，编译器自动处理取消传播和错误聚合。

```swift
// 三个网络请求并行执行，全部完成后再合并结果
func loadDashboard() async throws -> DashboardViewModel {
    async let user: User = api.fetchUser(id: currentUser.id)
    async let posts: [Post] = api.fetchRecentPosts(limit: 20)
    async let notifications: [Notification] = api.fetchNotifications()
    
    // 三个请求并行发起，此处等待所有完成
    let (user, posts, notifs) = try await (user, posts, notifications)
    
    return DashboardViewModel(user: user, posts: posts, notifications: notifs)
}

// 部分并行 + 部分顺序：先获取 token，再并行获取数据
func loadWithAuth() async throws -> DashboardViewModel {
    let token = try await api.refreshToken()
    async let a = api.fetchA(authorization: token)
    async let b = api.fetchB(authorization: token)
    return try await combine(a, b)
}
```

**Pitfalls**: 所有 `async let` 必须被 `await` 取值，否则任务会在作用域结束时被静默取消（可能导致竞态条件）；任一 `async let` 抛出异常时，`try await (a, b, c)` 会立即取消其余任务并传播第一个错误 —— 需要容错的场景应逐个 `do/catch` 包裹；声明顺序不影响执行顺序，三个 `async let` 始终并行启动。

**Best Practice**: 页面初始化/首屏加载优先使用 `async let` 并行拉取数据（比 `TaskGroup` 代码量减少 60%+）；需要逐个处理结果或提前返回时改用 `TaskGroup`；配合 `await withTimeout()` 为每个并行任务设置超时，避免一个慢请求阻塞整个页面。

## Task 优先级 vs GCD QoS 映射关系

Swift Concurrency 的 `TaskPriority`（7 级：`userInitiating` / `high` / `medium` / `low` / `utility` / `background` / `expedited`）并非简单等于 GCD 的 QoS 层级。编译器将 `TaskPriority` 转换为底层 `dispatch_queue_priority_t` 时采用 **模糊映射**：`high` 对应 `userInitiated`，`medium` 对应 `default`，`low` 和 `utility` 都映射到 `utility`，`background` 精确映射到 `background`。

关键差异在于 Swift 的 `TaskPriority` 是**相对的**而非绝对的。当父 Task 以 `userInitiated` QoS 运行时，子 Task 的 `.high` 不会突破父 Task 的 QoS 上限 —— 这是 Swift 并发模型防止优先级爆炸的安全设计。

```swift
// 显式映射表（Swift 5.9+ 编译器内部实现）
// TaskPriority    ->  dispatch QoS        ->  POSIX nice
// ─────────────────────────────────────────────────────
// .expedited      ->  userInteractive      ->  -18
// .userInitiating ->  userInteractive      ->  -18
// .high           ->  userInitiated        ->  -8
// .medium         ->  default              ->  0
// .low            ->  utility              ->  2
// .utility        ->  utility              ->  2
// .background     ->  background           ->  4

// 子 Task 优先级受父 Task QoS 限制（继承机制）
Task(priority: .high) {  // 实际 QoS 取决于创建位置
    if let qos = Task.currentQoS {
        print("当前 QoS: \(qos)")  // 可能不是 userInitiated!
    }
    // 后台队列中创建的 .high Task 仍不会超过 utility
}

// 精确控制 QoS 的两种方式
Task {
    await MainActor.run {
        // 在主线程 QoS 上下文中执行
    }
}

Task.detached(priority: .userInitiating) {
    // 脱离父 Task 优先级限制，强制使用最高 QoS
    try await api.refreshAuthToken()
}
```

**Pitfalls**: 不要在循环中频繁创建 `priority: .userInitiating` 的 Task —— 编译器会静默拒绝超过阈值的 expedited 任务并降级为 medium，导致难以调试的调度延迟；`Task.currentQoS` 返回的是创建位置的 QoS 而非当前运行队列的实际 QoS，两者可能不一致。

**Best Practice**: 90% 场景使用默认优先级即可；需要精确映射到 GCD 队列时，用 `OperationQueue` 包装 GCD 队列再桥接到 async；后台长时间任务统一用 `.background` 或 `Task.detached` 避免影响前台响应性。

## @MainActor 使用模式

`@MainActor` 是 Swift 并发模型中最常用的 actor 注解，确保标注的类、方法、属性 **只能在主线程访问**。编译器自动在跨 actor 边界调用时插入 `await`，并提供数据竞争防护。与 GCD 的 `DispatchQueue.main.async` 不同，`@MainActor` 是编译期保证而非运行期约定。

```swift
// 1. 类级注解 — 所有成员默认在主线程执行
@MainActor
class ProfileViewModel: ObservableObject {
    @Published var displayName: String = ""
    
    func fetchProfile() async throws {
        let user = try await api.getUser()  // 自动脱离 MainActor 执行网络请求
        displayName = user.name              // 安全写回主线程
    }
}

// 2. 非主线程调用的两种方式
Task {
    let vm = ProfileViewModel()
    await vm.fetchProfile()  // 自动 hop 到主线程
}

// 3. 从主线程逃逸 — 后台计算不阻塞 UI
@MainActor class ViewModel {
    func heavyComputation() async -> Int {
        await Task.detached {  // 脱离 MainActor 上下文
            return performIntensiveCalculation()
        }.value
    }
}

// 4. SwiftUI View 天然运行在 MainActor 上，无需标注
struct ProfileView: View {
    @StateObject var vm = ProfileViewModel()
    var body: some View { Text(vm.displayName) }  // 编译期安全
}
```

**Pitfalls**: `@MainActor` 标注的 `init()` 不会自动 hop 到主线程 —— 必须在主线程手动创建实例；闭包中捕获 `self` 时，`nonisolated` 方法不会自动返回 MainActor 上下文，需用 `MainActor.run {}` 显式跳回；在 `@MainActor` 方法内直接调用 GCD `async` 会导致 undefined behavior，应改用 `Task.detached`。

**Best Practice**: ViewModel 统一标注 `@MainActor` 获得编译期线程安全；纯计算/IO 方法标记 `nonisolated` 避免不必要的线程 hop 开销；SwiftUI 项目中用 `@MainActor` 替代所有 `@MainThread` 运行时检查。


### Actor 隔离与数据竞争防护

Swift Actor 模型通过编译器插入序列化访问锁，确保同一时刻只有一个任务能修改 actor 的状态，从根本上消除 data race。每个 actor 维护一个隐式的 `isolated` 保护域，跨 actor 边界的访问自动插入 `await` 和 suspend/resume 点。与 `actor` 不同，`@MainActor` 是全局单一实例 actor，所有标注 `@MainActor` 的类共享同一个执行上下文。

```swift
// Actor 自动序列化状态访问
actor ScoreBoard {
    private var scores: [String: Int] = [:]
    
    func updateScore(_ name: String, points: Int) {
        scores[name, default: 0] += points  // 编译器自动加锁
    }
    
    func getLeaderboard() -> [(String, Int)] {
        scores.sorted { $0.value > $1.value }  // 读也序列化
    }
}

// nonisolated 方法跳过 actor 隔离 — 仅限纯计算或外部同步
actor TokenBucket {
    private let lock = NSLock()
    private var tokens: Int = 10
    
    nonisolated func consume() -> Bool {  // ⚠️ 必须手动同步
        lock.lock(); defer { lock.unlock() }
        guard tokens > 0 else { return false }
        tokens -= 1; return true
    }
}

// 跨 actor 调用自动 await
let board = ScoreBoard()
let scores = await board.getLeaderboard()  // 隐式 suspend
```

**Pitfalls**: `nonisolated` 属性无法访问 actor 的隔离状态，需通过 `isolated` 方法或显式 `self` 桥接；在 actor 方法内对另一个 actor 发送方法会触发 suspension，若形成循环依赖则导致死锁；`Sendable` 协议的 `nonisolated` 方法若访问可变状态会绕过隔离，编译器只在不明确标注 `@Sendable` 时报错。

**Best Practice**: 优先使用 `actor` 而非 `actor` + `nonisolated` 混合模式，减少手动同步面；只读且无副作用的方法可标记 `nonisolated` 以提升并发性能；大型项目用 `@unchecked Sendable` 时配合 ThreadSanitizer 验证无数据竞争。

### Swift Task 在同步上下文中阻塞异步

在同步函数中调用 `async` 代码时，Swift 提供了 `Task { ... }.value` 来桥接同步与异步边界。该调用会在当前线程创建新 Task 并阻塞等待结果返回，类似于 `dispatchGroup.wait()` 但基于 Swift Concurrency 实现。底层通过 `run()`-style 机制在当前 executor 上调度异步操作并同步阻塞调用方线程。

```swift
// ❌ 编译错误：同步函数不能直接 await
func fetchUserSync(id: Int) -> User {
    return api.fetchUser(id: id)  // ERROR: cannot await
}

// ✅ Task {}.value 桥接同步/异步边界
func fetchUserSync(id: Int) -> User {
    return Task {
        try await api.fetchUser(id: id)  // 在当前线程阻塞执行
    }.value  // 阻塞直到 async 完成
}

// ⚠️ 主线程上 Task {}.value 导致死锁
@MainActor func loadData() {
    let data = Task {  // 新 Task 也调度到 MainActor
        await networkRequest()  // 需要主线程，但主线程已被阻塞 → 死锁！
    }.value
}

// ✅ 用 Task.detached 脱离当前 Actor 避免死锁
@MainActor func loadDataSafe() {
    let data = Task.detached {
        await networkRequest()
    }.value  // 安全：不竞争主线程
}
```

**Pitfalls**: 在 `@MainActor` 上下文中用 `Task {}.value` 且内部代码也需要主线程时，必然死锁 —— 阻塞占住了主线程而 Task 又排回到主队列；`Task {}.value` 会传播 `throws`，外层需用 `try` 捕获；在同步回调（如 `UIView.animate` completion）中滥用 `.value` 导致嵌套阻塞，性能劣于纯异步链。

**Best Practice**: 同步桥接优先用 `Task.detached {}.value` 隔离 Actor 上下文；能异步化的 API 尽早改造为 `async`，减少同步阻塞点；`try? Task { ... }.value` 处理可选值时需同时处理 `nil` 和异常两种情况。


### OperationQueue vs DispatchQueue vs Swift Task 性能基准对比

iOS 提供了三代并发抽象：GCD 的 `DispatchQueue`（C API）、OperationKit 的 `OperationQueue`（OOP 层级）、Swift 6 的 `async/await` + `Task`（结构化并发）。三者性能差异取决于任务粒度和调度策略。

```swift
import XCTest

// 1️⃣ DispatchQueue (GCD - C API，零对象开销)
let gcStart = CFAbsoluteTimeGetCurrent()
for i in 0..<10_000 {
    DispatchQueue.global().async { _ = (i * i).description }
}

// 2️⃣ OperationQueue (OOP + 依赖管理，对象分配成本高)
let opQueue = OperationQueue(); opQueue.maxConcurrentOperationCount = 4
for i in 0..<10_000 {
    opQueue.addOperation { _ = (i * i).description }
}

// 3️⃣ Swift Task (结构化并发，性能接近 GCD)
func benchmark() async {
    await withTaskGroup(of: Void.self) { group in
        for i in 0..<10_000 {
            group.addTask { _ = (i * i).description }
        }
    }
}
```

**基准结论**：超短任务 GCD 快约 15-20%（无对象分配开销）；OperationQueue 慢约 30%（对象创建 + 依赖计算）；Swift Task 与 GCD 差距 <5%，代码可读性最优。

**Pitfalls**: 微基准受编译器优化干扰严重，`_ = (i*i).description` 可能被完全消除，真实测试需用 `StatisticsCollection()` 防消除；iOS 版本间 GCD-NPL 底层演进会改变比例差距。

**Best Practice**: 新项目统一使用 `Task` + `async/await` 获得最佳可读性与取消支持；极高频短任务（如渲染管线每帧调用）保留 `DispatchQueue`；需依赖拓扑或级联取消时用 `OperationQueue`。
