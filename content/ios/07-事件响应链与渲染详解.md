# iOS 事件响应链与渲染详解

---

## 一、UIResponder 与响应者链

### 1.1 UIResponder 是什么

```
UIResponder 是所有能处理事件的对象的基类。

继承关系：
  UIView       → UIResponder → NSObject
  UIViewController → UIResponder → NSObject
  UIApplication    → UIResponder → NSObject
  UIWindow         → UIView → UIResponder → NSObject
  AppDelegate      → UIResponder → NSObject（通常）

UIResponder 定义了事件处理的统一接口：
  - 触摸事件：touchesBegan / touchesMoved / touchesEnded / touchesCancelled
  - 按压事件：pressesBegan / pressesEnded（物理按键、遥控器）
  - 运动事件：motionBegan / motionEnded（摇一摇）
  - 远程控制事件：remoteControlReceived（耳机线控）
```

### 1.2 响应者链（Responder Chain）

```
响应者链是一条由 nextResponder 属性串联起来的链表。

链的构成规则：
  UIView → 父视图（superview）
  UIView（VC 的 view）→ UIViewController
  UIViewController → 父 VC 或 Window
  UIWindow → UIApplication → AppDelegate

示例：一个按钮被点击后的响应者链

  Button
    ↓ nextResponder
  ContentView
    ↓ nextResponder
  UIViewController.view
    ↓ nextResponder
  UIViewController
    ↓ nextResponder
  UIWindow
    ↓ nextResponder
  UIApplication
    ↓ nextResponder
  AppDelegate

事件沿着链向上传递：
  如果当前响应者不处理（没有重写 touchesBegan 等方法）
  → 自动传给 nextResponder
  → 直到有人处理或到达链末端（丢弃）
```

### 1.3 代码验证响应者链

```objc
// 打印完整的响应者链
- (void)printResponderChain {
    UIResponder *responder = self;
    while (responder) {
        NSLog(@"%@", [responder class]);
        responder = responder.nextResponder;
    }
}

// 输出示例：
// MyButton
// UIView
// MyViewController
// UIDropShadowView
// UITransitionView
// UIWindow
// UIWindowScene
// UIApplication
// AppDelegate
```

## 二、事件传递：Hit-Testing

### 2.1 触摸事件的产生与传递

```
用户手指触摸屏幕后的完整流程：

1. 硬件层：
   触摸屏产生电信号 → IOKit 驱动封装为 IOHIDEvent

2. 系统进程：
   SpringBoard（桌面进程）接收 IOHIDEvent
   → 通过 Mach Port 转发给当前前台 App 进程

3. App 进程接收：
   主线程 RunLoop 的 Source1 回调被触发
   → __IOHIDEventSystemClientQueueCallback()
   → Source0 触发内部事件处理
   → UIApplication 将 IOHIDEvent 包装为 UIEvent

4. Hit-Testing：
   UIApplication → UIWindow → 递归遍历子视图
   → 找到最合适的响应视图（Hit-Test View）

5. 事件处理：
   UIEvent 被发送给 Hit-Test View
   → 响应者链开始工作

时序：
  硬件 → IOKit → SpringBoard → Mach Port → App RunLoop
  → Source1 → Source0 → UIApplication → Hit-Testing → 响应
```

### 2.2 hitTest:withEvent: 与 pointInside:withEvent:

```objc
// UIView 的两个核心方法：

// hitTest: 找到最合适的响应视图
// 返回值就是最终接收事件的视图
- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event;

// pointInside: 判断触摸点是否在自己的范围内
- (BOOL)pointInside:(CGPoint)point withEvent:(UIEvent *)event;
```

### 2.3 hitTest 的默认实现

```objc
// hitTest 的伪代码实现（系统默认逻辑）
- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event {

    // 1. 不接收事件的三种情况，直接返回 nil
    if (!self.userInteractionEnabled ||   // 交互关闭
        self.hidden ||                     // 被隐藏
        self.alpha <= 0.01) {              // 几乎透明
        return nil;
    }

    // 2. 触摸点不在自己范围内
    if (![self pointInside:point withEvent:event]) {
        return nil;
    }

    // 3. 倒序遍历子视图（后添加的视图在最上面，优先检测）
    for (NSInteger i = self.subviews.count - 1; i >= 0; i--) {
        UIView *subview = self.subviews[i];
        // 将坐标转换到子视图的坐标系
        CGPoint convertedPoint = [subview convertPoint:point fromView:self];
        // 递归调用子视图的 hitTest
        UIView *hitView = [subview hitTest:convertedPoint withEvent:event];
        if (hitView) {
            return hitView;  // 找到了，直接返回
        }
    }

    // 4. 子视图都没命中，自己就是最合适的
    return self;
}
```

### 2.4 Hit-Testing 的遍历顺序

```
Hit-Testing 是从 UIWindow 开始，自上而下递归的过程。
遍历时子视图按照 从后往前 的顺序（视觉上从最上面开始）。

示例视图层级：
  Window
    ├─ ViewA
    │    ├─ ViewA1
    │    └─ ViewA2
    └─ ViewB
         ├─ ViewB1
         └─ ViewB2

触摸点在 ViewB1 上时的 hitTest 调用顺序：

  Window.hitTest     → pointInside YES
    ViewB.hitTest    → pointInside YES  （先检查 B，因为 B 后添加）
      ViewB2.hitTest → pointInside NO   （先检查 B2，后添加的先查）
      ViewB1.hitTest → pointInside YES  → 返回 ViewB1 ✓
    （ViewA 不再检查）

关键点：
  1. 倒序遍历：后添加的子视图先检查（视觉上在最前面）
  2. 找到即返回：命中后不继续遍历兄弟视图
  3. 父视图先判断 pointInside：如果 NO 则子视图都不检查
```

### 2.5 常见的 hitTest 应用

```objc
// ========== 场景1：扩大按钮点击区域 ==========
@implementation ExpandButton

- (BOOL)pointInside:(CGPoint)point withEvent:(UIEvent *)event {
    // 各方向扩大 20pt
    CGRect expandedBounds = CGRectInset(self.bounds, -20, -20);
    return CGRectContainsPoint(expandedBounds, point);
}

@end


// ========== 场景2：让超出父视图的子视图也能响应 ==========
// 子视图超出了父视图的 bounds 时，默认不能响应事件
// 因为父视图的 pointInside 返回 NO，子视图根本不会被检查

@implementation ContainerView

- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event {
    // 先走默认逻辑
    UIView *hitView = [super hitTest:point withEvent:event];
    if (hitView) return hitView;

    // 默认没命中，检查超出 bounds 的子视图
    for (UIView *subview in self.subviews) {
        CGPoint subPoint = [subview convertPoint:point fromView:self];
        hitView = [subview hitTest:subPoint withEvent:event];
        if (hitView) return hitView;
    }
    return nil;
}

@end


// ========== 场景3：穿透视图（让事件传递给下面的视图）==========
@implementation PassthroughView

- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event {
    UIView *hitView = [super hitTest:point withEvent:event];
    // 如果命中的是自己（没有命中子视图），返回 nil 让事件穿透
    if (hitView == self) {
        return nil;
    }
    return hitView;
}

@end


// ========== 场景4：ScrollView 中嵌套的小按钮响应问题 ==========
// ScrollView 子视图中的小按钮有时不好点
// 可以在按钮的父视图中把事件转发给按钮

@implementation CardView

- (UIView *)hitTest:(CGPoint)point withEvent:(UIEvent *)event {
    UIView *hitView = [super hitTest:point withEvent:event];
    if (hitView == self) {
        // 点到了卡片空白区域，检查是否靠近按钮
        CGPoint btnPoint = [self.actionButton convertPoint:point fromView:self];
        CGRect expandedRect = CGRectInset(self.actionButton.bounds, -15, -15);
        if (CGRectContainsPoint(expandedRect, btnPoint)) {
            return self.actionButton;
        }
    }
    return hitView;
}

@end
```

## 三、触摸事件处理

### 3.1 UITouch 与 UIEvent

```objc
// UITouch：代表一根手指的触摸
// 一根手指从按下到抬起，是同一个 UITouch 对象

@interface UITouch : NSObject
@property (readonly) NSTimeInterval timestamp;    // 时间戳
@property (readonly) UITouchPhase phase;          // 触摸阶段
@property (readonly) NSUInteger tapCount;         // 短时间内的点击次数
@property (readonly) UITouchType type;            // 类型（直接/间接/pencil）
@property (readonly) CGFloat majorRadius;         // 触摸半径
@property (readonly) CGFloat force;               // 3D Touch 力度
@property (readonly) CGFloat maximumPossibleForce;

- (CGPoint)locationInView:(UIView *)view;         // 当前位置
- (CGPoint)previousLocationInView:(UIView *)view; // 上一次位置
@end

// UITouchPhase 阶段：
typedef NS_ENUM(NSInteger, UITouchPhase) {
    UITouchPhaseBegan,        // 手指按下
    UITouchPhaseMoved,        // 手指移动
    UITouchPhaseStationary,   // 手指静止（多指时其他手指在动）
    UITouchPhaseEnded,        // 手指抬起
    UITouchPhaseCancelled,    // 被系统取消（来电等）
};


// UIEvent：代表一个事件（可能包含多个 UITouch）
@interface UIEvent : NSObject
@property (readonly) UIEventType type;            // 触摸/运动/遥控/按压
@property (readonly) UIEventSubtype subtype;
@property (readonly) NSTimeInterval timestamp;

- (NSSet<UITouch *> *)allTouches;                 // 所有触摸
- (NSSet<UITouch *> *)touchesForView:(UIView *)view; // 某个视图的触摸
@end
```

### 3.2 触摸回调方法

```objc
// UIResponder 的四个触摸回调

- (void)touchesBegan:(NSSet<UITouch *> *)touches withEvent:(UIEvent *)event {
    // 手指按下
    UITouch *touch = [touches anyObject];
    CGPoint location = [touch locationInView:self];
    NSLog(@"按下位置：%@", NSStringFromCGPoint(location));
}

- (void)touchesMoved:(NSSet<UITouch *> *)touches withEvent:(UIEvent *)event {
    // 手指移动（可能每秒调用 60-120 次）
    UITouch *touch = [touches anyObject];
    CGPoint current = [touch locationInView:self];
    CGPoint previous = [touch previousLocationInView:self];
    CGFloat dx = current.x - previous.x;
    CGFloat dy = current.y - previous.y;
    // 利用偏移量做拖拽
    self.center = CGPointMake(self.center.x + dx, self.center.y + dy);
}

- (void)touchesEnded:(NSSet<UITouch *> *)touches withEvent:(UIEvent *)event {
    // 手指抬起
}

- (void)touchesCancelled:(NSSet<UITouch *> *)touches withEvent:(UIEvent *)event {
    // 被系统中断（来电、手势识别器接管等）
    // ⚠️ 必须处理，否则可能状态不一致（比如高亮没有取消）
}

// ⚠️ 如果重写了这些方法但还想让事件继续传递：
- (void)touchesBegan:(NSSet<UITouch *> *)touches withEvent:(UIEvent *)event {
    // 自己处理...
    [super touchesBegan:touches withEvent:event];  // 传给 nextResponder
}
```

## 四、手势识别器（UIGestureRecognizer）

### 4.1 手势识别器的种类

```
系统提供的 7 种手势识别器：

UITapGestureRecognizer        点击（单击/双击/多击）
UILongPressGestureRecognizer  长按
UIPanGestureRecognizer        拖拽
UISwipeGestureRecognizer      轻扫（上下左右）
UIPinchGestureRecognizer      捏合（缩放）
UIRotationGestureRecognizer   旋转
UIScreenEdgePanGestureRecognizer  屏幕边缘拖拽
```

### 4.2 手势识别器的状态机

```
手势识别器有自己的状态机，分两种类型：

离散型手势（Tap、Swipe）：
  Possible → Recognized（成功）
  Possible → Failed（失败）

连续型手势（Pan、Pinch、Rotation、LongPress）：
  Possible → Began → Changed → Changed → ... → Ended
  Possible → Began → Changed → ... → Cancelled
  Possible → Failed

状态流转图：
                  ┌──────────┐
                  │ Possible │ （初始状态）
                  └────┬─────┘
                       │
           ┌───────────┼───────────┐
           ↓           ↓           ↓
      ┌─────────┐ ┌─────────┐ ┌────────┐
      │Recognized│ │  Began  │ │ Failed │
      │（离散型） │ │（连续型）│ │        │
      └─────────┘ └────┬────┘ └────────┘
                       │
                       ↓
                  ┌─────────┐
                  │ Changed │ ←─┐（可多次）
                  └────┬────┘ ──┘
                       │
               ┌───────┴───────┐
               ↓               ↓
          ┌─────────┐    ┌───────────┐
          │  Ended  │    │ Cancelled │
          └─────────┘    └───────────┘
```

### 4.3 手势识别器的用法

```objc
// ========== 点击手势 ==========
UITapGestureRecognizer *tap = [[UITapGestureRecognizer alloc]
    initWithTarget:self action:@selector(handleTap:)];
tap.numberOfTapsRequired = 1;    // 点击次数
tap.numberOfTouchesRequired = 1; // 手指数
[view addGestureRecognizer:tap];

- (void)handleTap:(UITapGestureRecognizer *)gesture {
    CGPoint location = [gesture locationInView:self.view];
    NSLog(@"点击位置：%@", NSStringFromCGPoint(location));
}


// ========== 拖拽手势 ==========
UIPanGestureRecognizer *pan = [[UIPanGestureRecognizer alloc]
    initWithTarget:self action:@selector(handlePan:)];
[view addGestureRecognizer:pan];

- (void)handlePan:(UIPanGestureRecognizer *)gesture {
    CGPoint translation = [gesture translationInView:self.view];

    switch (gesture.state) {
        case UIGestureRecognizerStateBegan:
            // 拖拽开始
            break;
        case UIGestureRecognizerStateChanged:
            // 拖拽中，移动视图
            gesture.view.center = CGPointMake(
                gesture.view.center.x + translation.x,
                gesture.view.center.y + translation.y
            );
            // ⚠️ 重置 translation，否则是累计偏移
            [gesture setTranslation:CGPointZero inView:self.view];
            break;
        case UIGestureRecognizerStateEnded: {
            // 拖拽结束，可以获取速度做惯性动画
            CGPoint velocity = [gesture velocityInView:self.view];
            NSLog(@"速度：%@", NSStringFromCGPoint(velocity));
            break;
        }
        default:
            break;
    }
}


// ========== 捏合手势 ==========
UIPinchGestureRecognizer *pinch = [[UIPinchGestureRecognizer alloc]
    initWithTarget:self action:@selector(handlePinch:)];
[view addGestureRecognizer:pinch];

- (void)handlePinch:(UIPinchGestureRecognizer *)gesture {
    gesture.view.transform = CGAffineTransformScale(
        gesture.view.transform,
        gesture.scale,
        gesture.scale
    );
    gesture.scale = 1.0;  // 重置，防止累计
}
```

### 4.4 手势识别器与触摸事件的关系

```
手势识别器和 touches 方法的交互时序：

1. 触摸开始时，UIWindow 会同时把事件发给：
   - Hit-Test View 的 touchesBegan
   - 该视图及其祖先视图上的所有手势识别器

2. 手势识别器在识别过程中（Possible 状态）：
   touches 方法正常调用（touchesBegan / touchesMoved）

3. 手势被识别成功（Began 或 Recognized）：
   → UIWindow 对 Hit-Test View 调用 touchesCancelled
   → 后续 touch 事件不再发给 View，只发给手势识别器

4. 手势识别失败（Failed）：
   → touches 方法继续正常接收事件

时序图（手势成功的情况）：
  时间 →
  手势:   Possible ──────────→ Recognized
  View:   touchesBegan → touchesMoved → touchesCancelled
                                              ↑
                                    手势成功，View 被取消

这就是为什么给 UIView 同时添加手势和 touches 方法时：
  手势成功 → touchesCancelled 被调用
  手势失败 → touchesEnded 正常调用
```

### 4.5 手势识别器的三个属性

```objc
// cancelsTouchesInView（默认 YES）
// 手势识别成功后，是否取消 View 的 touches 事件
gesture.cancelsTouchesInView = NO;
// 设为 NO：手势成功后 View 仍然收到 touchesEnded（不会被 cancel）
// 场景：同时需要手势和 touches 回调

// delaysTouchesBegan（默认 NO）
// 手势在 Possible 阶段时，是否延迟向 View 发送 touchesBegan
gesture.delaysTouchesBegan = YES;
// 设为 YES：手势识别结束前 View 不会收到 touchesBegan
// 场景：防止 View 出现短暂的高亮又立即取消

// delaysTouchesEnded（默认 YES）
// 手势在 Possible 阶段手指抬起时，是否延迟向 View 发送 touchesEnded
gesture.delaysTouchesEnded = NO;
// 默认 YES：给手势识别器一个延迟窗口来判断（约 0.15s）
// 场景：双击手势需要等待确认不是单击
```

### 4.6 手势冲突处理

```objc
// ========== require(toFail:) ==========
// 当前手势必须等另一个手势失败后才能识别
// 典型：区分单击和双击

UITapGestureRecognizer *singleTap = [[UITapGestureRecognizer alloc]
    initWithTarget:self action:@selector(handleSingleTap:)];

UITapGestureRecognizer *doubleTap = [[UITapGestureRecognizer alloc]
    initWithTarget:self action:@selector(handleDoubleTap:)];
doubleTap.numberOfTapsRequired = 2;

// 单击必须等双击失败后才触发（避免双击时触发两次单击）
[singleTap requireGestureRecognizerToFail:doubleTap];

[view addGestureRecognizer:singleTap];
[view addGestureRecognizer:doubleTap];


// ========== delegate 精细控制 ==========
@interface MyVC () <UIGestureRecognizerDelegate>
@end

// 是否允许手势开始识别
- (BOOL)gestureRecognizerShouldBegin:(UIGestureRecognizer *)gestureRecognizer {
    // 例：只在向右滑时允许 pan 手势
    if ([gestureRecognizer isKindOfClass:[UIPanGestureRecognizer class]]) {
        CGPoint velocity = [(UIPanGestureRecognizer *)gestureRecognizer
                            velocityInView:self.view];
        return velocity.x > 0;  // 只允许向右
    }
    return YES;
}

// 是否允许两个手势同时识别
- (BOOL)gestureRecognizer:(UIGestureRecognizer *)g1
    shouldRecognizeSimultaneouslyWithGestureRecognizer:(UIGestureRecognizer *)g2 {
    // 允许缩放和旋转同时工作
    return YES;
}

// 当两个手势冲突时，是否让 other 失败
- (BOOL)gestureRecognizer:(UIGestureRecognizer *)gestureRecognizer
    shouldRequireFailureOfGestureRecognizer:(UIGestureRecognizer *)other {
    // 动态版的 require(toFail:)
    return NO;
}
```

## 五、UIView 与 CALayer 的关系

### 5.1 UIView 和 CALayer 的分工

```
UIView：
  负责事件处理（响应者链、手势识别器）
  负责布局管理（Auto Layout、frame）
  是 CALayer 的 delegate
  提供高层 API（背景色、圆角、动画等最终映射到 layer）

CALayer：
  负责实际的视觉内容展示和合成
  管理位图（backing store / contents）
  处理动画的插值计算
  不参与事件处理

关系：
  每个 UIView 都持有一个 CALayer（self.layer）
  UIView 是 layer 的 delegate（响应 display/layout/action 等）

  ┌─────────────────────────────┐
  │ UIView                      │
  │  - 事件处理                  │
  │  - Auto Layout               │
  │  - 作为 layer 的 delegate    │
  │                              │
  │  ┌────────────────────────┐  │
  │  │ CALayer                │  │
  │  │  - 视觉内容（位图）     │  │
  │  │  - 几何属性（位置/大小） │  │
  │  │  - 动画                 │  │
  │  │  - 合成                 │  │
  │  └────────────────────────┘  │
  └─────────────────────────────┘

为什么分两层：
  macOS 上 NSView 没有 layer-backed 的概念是后来加的
  CALayer 是跨平台的（macOS + iOS 共享 Core Animation）
  iOS 的 UIView 是对 CALayer 的轻量封装 + 事件处理
```

### 5.2 CALayer 的常用属性

```objc
// 几何属性
layer.frame           // 外部坐标系中的位置和大小
layer.bounds          // 自身坐标系中的大小
layer.position        // 锚点在父图层中的位置（对应 UIView.center）
layer.anchorPoint     // 锚点（0,0 左上角 ~ 1,1 右下角，默认 0.5,0.5）
layer.transform       // 3D 变换（CATransform3D）
layer.zPosition       // Z 轴位置（影响显示层级，不影响 hitTest 顺序）

// 视觉属性
layer.backgroundColor // 背景色
layer.contents        // 内容（通常是 CGImageRef）
layer.contentsScale   // 内容缩放因子（配合 Retina）
layer.contentsGravity // 内容对齐模式（类似 contentMode）
layer.opacity         // 透明度（对应 UIView.alpha）
layer.cornerRadius    // 圆角
layer.borderWidth     // 边框宽度
layer.borderColor     // 边框颜色
layer.shadowColor     // 阴影颜色
layer.shadowOffset    // 阴影偏移
layer.shadowRadius    // 阴影模糊半径
layer.shadowOpacity   // 阴影透明度
layer.shadowPath      // 阴影路径（⚠️ 不设置会触发离屏渲染）
layer.masksToBounds   // 裁剪子图层（对应 clipsToBounds）
layer.mask            // 遮罩图层

// ⚠️ anchorPoint 的影响
// anchorPoint 改变不影响 frame 的计算，但影响 position 的含义
// position = frame.origin + frame.size * anchorPoint
layer.anchorPoint = CGPointMake(0, 0);  // 左上角旋转
layer.anchorPoint = CGPointMake(1, 1);  // 右下角旋转
```

### 5.3 图层树

```
Core Animation 维护三棵平行的图层树：

1. Model Layer Tree（模型树）
   layer.xxx 访问的就是模型树
   存储动画的最终目标值
   代码中直接操作的对象

2. Presentation Layer Tree（呈现树）
   layer.presentationLayer 访问
   存储动画过程中的当前插值
   用于获取动画中某一刻的实际显示值
   hitTest 默认用模型树，动画中的 hitTest 需要用呈现树

3. Render Tree（渲染树）
   私有的，运行在 Render Server（独立进程）
   GPU 实际绘制用的数据
   开发者无法直接访问

使用场景：
  // 获取动画中间态
  CALayer *presentationLayer = self.layer.presentationLayer;
  CGFloat currentX = presentationLayer.position.x;

  // 动画过程中的 hitTest
  CGPoint hitPoint = [self.layer.presentationLayer
      hitTest:[touch locationInView:self.superview]];
```

### 5.4 从创建 UIView 到屏幕显示像素的完整流程

```
本节把前面所有知识串联起来，跟踪一个 UIView 从代码创建到屏幕亮起像素
的全过程。以下代码：

  UIView *view = [[UIView alloc] initWithFrame:CGRectMake(0, 0, 100, 100)];
  view.backgroundColor = [UIColor redColor];
  [self.window addSubview:view];

经历了这些阶段：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段 1：对象创建（同步，立即完成）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  alloc                          → 分配内存
  initWithFrame:                 → 初始化
    ├─ 创建 CALayer（self.layer）
    │    ├─ bounds = (0,0,100,100)
    │    ├─ position = (50,50)（center，anchorPoint 默认 0.5,0.5）
    │    └─ opacity = 1.0
    ├─ UIView 成为 layer 的 delegate
    └─ view.layer.frame 自动同步

  此时：对象存在于内存中，但还没有进入任何图层树，不会显示。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段 2：加入图层树（addSubview）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  [window addSubview:view]
    ├─ view.layer 被加入 window.layer.sublayers
    ├─ view.superview = window
    ├─ view.layer 的 model tree 已更新（位置关系确定）
    └─ ⚠️ 此时视图树已构建，但屏幕上还没变化

  同时触发：
    ├─ view 的 layoutSubviews 被标记（setNeedsLayout）
    │    因为 addSubview 会触发 layout 标记
    └─ 但不会立即执行，要等 RunLoop 的 BeforeWaiting

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段 3：RunLoop 的下一轮迭代（延迟执行）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  当前 RunLoop 迭代结束 → 进入 BeforeWaiting 观察者回调

  ┌─────────────────────────────────────────────────────────────┐
  │  BeforeWaiting 阶段，Core Animation 依次执行：              │
  │                                                             │
  │  ① Layout（布局）                                          │
  │     遍历有 setNeedsLayout 标记的视图                        │
  │     → 调用 layoutSubviews                                  │
  │     → Auto Layout 引擎求解约束（如果有）                    │
  │     → 计算每个视图的 frame                                 │
  │     → 本例：view.frame = (0,0,100,100)，已确定             │
  │                                                             │
  │  ② Display（绘制）                                         │
  │     遍历有 setNeedsDisplay 标记的视图                       │
  │     → 本例没有重写 drawRect，跳过自定义绘制                 │
  │     → 但 backgroundColor 会被处理：                        │
  │       系统将 redColor 转为 CGColorRef                       │
  │       赋值给 layer.backgroundColor                          │
  │     → 大多数视图到此为止，不需要 CPU 绘制位图               │
  │     → 如果重写了 drawRect，系统会分配 backing store         │
  │       并在此时调用 Core Graphics 绑定上下文执行绘制         │
  │                                                             │
  │  ③ Prepare（准备）                                         │
  │     图片解码、格式转换等                                    │
  │     → 本例无图片，跳过                                     │
  │     → 如果有 UIImageView，这里会解码 PNG/JPEG               │
  │                                                             │
  │  ④ Commit（提交）                                          │
  │     → 递归遍历整棵图层树                                    │
  │     → 序列化每个图层的属性（frame、backgroundColor、opacity  │
  │       transform、contents 等）                              │
  │     → 打包成一个 IPC 消息                                   │
  │     → 通过 Mach Port 发送给 Render Server 进程              │
  └─────────────────────────────────────────────────────────────┘

  ⚠️ 以上 4 步必须在当前 VSync 周期（16.67ms）内完成
     否则这一帧的内容不会被提交，导致掉帧

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段 4：Render Server（独立进程，GPU 渲染）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Render Server（backboardd）收到 IPC 消息后：

  ① Decode（反序列化）
     → 解析图层树数据
     → 重建图层结构（Render Tree，开发者无法访问）

  ② 设置 GPU 绘制参数
     → 每个 layer 的位图作为纹理上传到 GPU（或引用已有纹理）
     → 本例：backgroundColor 不是纹理，而是一个 fill 指令
     → 设置几何参数：顶点坐标（由 frame 计算矩形 4 个顶点）
     → 设置变换矩阵：position、transform、anchorPoint
     → 设置混合模式：opacity、blend mode

  ③ Draw Calls（绘制调用）
     → GPU 执行渲染命令
     → 本例：画一个 100x100 的红色矩形
     → 从图层树最底层开始，逐层合成到帧缓冲区
       window 的背景 → view 的红色矩形 → 其他上层视图

  ④ 合成到帧缓冲区（Frame Buffer）
     → GPU 将最终合成结果写入后帧缓冲区（Back Buffer）
     → 等待下一个 VSync 信号

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
阶段 5：屏幕显示（硬件层）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  VSync 信号到来：
    → 视频控制器交换前/后帧缓冲区（双缓冲机制）
    → 前帧缓冲区（Front Buffer）的新内容被逐行扫描输出
    → LCD/OLED 面板的像素根据信号改变状态
    → 用户看到 100x100 的红色矩形出现在屏幕上

  从代码执行到像素亮起的总延迟：
    ┌──────────────────────────────────────────────────┐
    │ RunLoop 提交（Commit）     ──→  本帧 VSync 前    │
    │ Render Server GPU 渲染    ──→  下一帧 VSync 前  │
    │ 帧缓冲区交换 + 屏幕扫描   ──→  再下一帧 VSync   │
    └──────────────────────────────────────────────────┘
    最坏情况：约 2~3 帧（33~50ms）延迟
    这是为什么修改 UI 后不会立即看到变化

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
完整时序图
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  时间轴 ──────────────────────────────────────────────────→

  App 进程:
    [alloc/init]──[addSubview]──[RunLoop BeforeWaiting]
                                      │
                          ┌───────────┤
                          │ Layout    │ frame 计算
                          │ Display   │ backgroundColor → layer
                          │ Prepare   │ （无图片跳过）
                          │ Commit ───┼───────────────────→ IPC 消息
                          └───────────┘                    │
                                                           ▼
  Render Server:                                    [Decode]──[Draw Calls]
                                                           │
                                                           ▼
                                                  [GPU 合成 → Back Buffer]
                                                           │
  VSync: ──────── V ──────────── V ──────────── V ────────┘
                                                    │
                                                    ▼
                                           [缓冲区交换]
                                                    │
                                                    ▼
                                           [屏幕扫描显示] ← 用户看到红色矩形

  关键点：
    1. 代码执行和屏幕显示是异步的，中间隔了 RunLoop 调度和 IPC
    2. Render Server 是独立进程，App 不会阻塞等待 GPU 渲染
    3. 双缓冲 + VSync 保证不会看到"半渲染"的画面（撕裂）
    4. 如果 Commit 超时，这帧的修改推迟到下下帧显示 → 掉帧

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
为什么理解这个流程很重要（面试/实战要点）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  1. 为什么子视图在 addSubview 后不立即有 frame？
     → Layout 是延迟执行的，在 BeforeWaiting 才跑

  2. 为什么 drawRect 里不能用 self.frame 而要用 self.bounds？
     → Display 阶段在 Layout 之后，frame 已确定
     → 但 bounds 是绘制坐标系，frame 包含 position 位移，绘制时不需要

  3. 为什么大量 addSubview 不会卡？
     → 同一次 RunLoop 的多次 setNeedsLayout 只触发一次 layoutSubviews
     → Commit 也是批量提交的

  4. 为什么修改 UI 要在主线程？
     → RunLoop 的 BeforeWaiting 回调在主线程
     → layoutSubviews / drawRect 在主线程
     → 多线程同时修改图层树会导致数据竞争和渲染异常

  5. 为什么 UIView 动画不卡顿？
     → 动画只修改 layer 的 model tree 目标值
     → 中间帧的插值由 Render Server 的 GPU 计算（不在主线程）
     → 主线程只需要提交起始和终止状态，GPU 做插值
```

## 六、渲染流水线

### 6.1 渲染的整体流程

```
iOS 渲染分为 App 进程内的工作和 Render Server 的工作：

┌─────────── App 进程 ──────────┐   ┌─── Render Server ───┐
│                                │   │                      │
│  Handle Events                 │   │                      │
│       ↓                        │   │                      │
│  Layout (layoutSubviews)       │   │                      │
│       ↓                        │   │                      │
│  Display (drawRect / draw)     │   │                      │
│       ↓                        │   │                      │
│  Prepare (图片解码等)           │   │                      │
│       ↓                        │   │                      │
│  Commit (提交图层树)  ─────────────→  Decode              │
│                                │   │     ↓                │
│                                │   │  Draw Calls          │
│                                │   │     ↓                │
│                                │   │  Render (GPU)        │
│                                │   │     ↓                │
│                                │   │  Display (屏幕)      │
└────────────────────────────────┘   └──────────────────────┘

    ←── 一帧 16.67ms (60fps) / 8.33ms (120fps) ──→

关键约束：
  App 的 Commit 必须在 VSync 信号前完成
  Render Server 的 GPU 渲染在下一帧完成
  → 从修改到显示，至少 2 帧的延迟（双缓冲）
```

### 6.2 App 侧的四个阶段

```
1. Layout 阶段
   计算每个视图/图层的大小和位置
   调用 layoutSubviews / viewDidLayoutSubviews
   Auto Layout 引擎求解约束 → 生成 frame
   ⚠️ 性能瓶颈：约束过多、约束冲突、频繁更新约束

2. Display 阶段
   生成每个图层的位图内容（backing store）
   调用 drawRect: / draw(_:rect:)
   使用 Core Graphics 绑定 CPU 绘制
   ⚠️ 大多数视图不需要自定义绘制（系统默认用 contents/背景色）
   ⚠️ 性能瓶颈：频繁调用 setNeedsDisplay、绘制内容过复杂

3. Prepare 阶段
   图片解码（PNG/JPEG → 位图）
   图片格式转换（如果 GPU 不支持的格式）
   ⚠️ 性能瓶颈：大图解码很耗时，默认在主线程

4. Commit 阶段
   递归打包整棵图层树 → 序列化 → 通过 IPC 发送给 Render Server
   ⚠️ 性能瓶颈：图层数量过多、图层树过深
```

### 6.3 Render Server（GPU 渲染）

```
Render Server 是一个独立进程（backboardd），负责 GPU 合成。

GPU 的工作：
  1. 接收 App 提交的图层树
  2. 将每个图层的位图作为纹理（texture）
  3. 按照图层的几何变换、透明度等参数
  4. 将纹理合成（compositing）到帧缓冲区
  5. 视频控制器从帧缓冲区读取像素，显示到屏幕

纹理合成的过程（简化）：
  从最底层图层开始，逐层叠加
  对每个图层：
    应用变换（位移、旋转、缩放）
    应用透明度
    应用遮罩（mask）
    与已有内容混合（blend）
  最终结果写入帧缓冲区

GPU 擅长的：
  纹理合成、几何变换、透明度混合 → 硬件加速，极快

GPU 不擅长的（需要 CPU 或离屏渲染）：
  复杂的裁剪路径、遮罩、阴影计算、文本渲染
```

### 6.4 VSync 与帧率

```
VSync（垂直同步信号）：
  屏幕硬件每隔固定时间发出一个信号
  60Hz → 每 16.67ms 一次
  120Hz → 每 8.33ms 一次（ProMotion，iPhone 13 Pro+）
  通知系统"我准备好显示下一帧了"

CADisplayLink：
  和 VSync 同步的定时器
  每次 VSync 信号到来时回调

  CADisplayLink *link = [CADisplayLink displayLinkWithTarget:self
                                                   selector:@selector(tick:)];
  [link addToRunLoop:[NSRunLoop mainRunLoop] forMode:NSRunLoopCommonModes];

  - (void)tick:(CADisplayLink *)link {
      NSTimeInterval duration = link.duration;        // 每帧理想时长
      NSTimeInterval actualInterval = link.timestamp - lastTimestamp;
      // 如果 actualInterval >> duration → 掉帧了
  }

掉帧（Jank）：
  如果一帧的工作没在 VSync 前完成 → 显示还是上一帧的内容
  用户感受到卡顿

  正常：V ─── V ─── V ─── V   每帧都在 VSync 前完成
  掉帧：V ─── V ─────── V     第二帧超时，第三个 VSync 才更新
```

## 七、离屏渲染（Offscreen Rendering）

### 7.1 什么是离屏渲染

```
正常渲染（On-screen Rendering）：
  GPU 直接把图层合成到帧缓冲区 → 显示

离屏渲染（Offscreen Rendering）：
  GPU 需要先在一块临时的离屏缓冲区（Offscreen Buffer）中渲染
  完成后再把结果合成到帧缓冲区

为什么需要离屏渲染：
  某些效果无法用简单的"逐层叠加"一次完成
  需要额外的缓冲区来暂存中间结果

离屏渲染的开销：
  1. 创建新的缓冲区（内存开销）
  2. 上下文切换（从帧缓冲区 → 离屏缓冲区 → 帧缓冲区）
  3. 离屏缓冲区有大小限制（约为屏幕像素的 2.5 倍）
  → 比正常渲染慢得多，帧率受影响
```

### 7.2 触发离屏渲染的常见操作

```objc
// 1. 圆角 + masksToBounds（最常见）
view.layer.cornerRadius = 10;
view.layer.masksToBounds = YES;  // 或 view.clipsToBounds = YES
// ⚠️ 只设 cornerRadius 不会离屏渲染
// 加上 masksToBounds 才会，因为要对子图层做裁剪

// 2. 阴影（没有设置 shadowPath）
view.layer.shadowColor = [UIColor blackColor].CGColor;
view.layer.shadowOffset = CGSizeMake(0, 2);
view.layer.shadowOpacity = 0.5;
view.layer.shadowRadius = 4;
// ⚠️ 没有 shadowPath → GPU 要根据图层内容实时计算阴影轮廓

// 3. mask（遮罩）
view.layer.mask = maskLayer;

// 4. group opacity（组透明度）
view.layer.allowsGroupOpacity = YES;
view.alpha = 0.5;
// 需要先把整个子树渲染到离屏缓冲区，再整体设置透明度

// 5. shouldRasterize（光栅化）
view.layer.shouldRasterize = YES;
view.layer.rasterizationScale = [UIScreen mainScreen].scale;
// 主动触发离屏渲染并缓存结果
// 适合内容不频繁变化的复杂图层

// 6. drawRect: / draw(_:)
// 重写了绘制方法 → Core Graphics CPU 绘制 → 也算"离屏"（CPU 离屏）
```

### 7.3 避免离屏渲染的方法

```objc
// ========== 1. 圆角优化 ==========

// 方案A：只设 cornerRadius，不设 masksToBounds（如果不需要裁剪子视图）
view.layer.cornerRadius = 10;
// 只对 backgroundColor 和 border 生效，不会离屏渲染

// 方案B：使用贝塞尔路径绘制圆角
UIBezierPath *path = [UIBezierPath bezierPathWithRoundedRect:imageView.bounds
                                                cornerRadius:10];
CAShapeLayer *maskLayer = [CAShapeLayer layer];
maskLayer.path = path.CGPath;
imageView.layer.mask = maskLayer;
// ⚠️ 这还是离屏渲染（mask），但在某些情况下可控

// 方案C：后台线程用 Core Graphics 绘制圆角图片（推荐）
dispatch_async(dispatch_get_global_queue(0, 0), ^{
    UIGraphicsBeginImageContextWithOptions(size, NO, 0);
    UIBezierPath *path = [UIBezierPath bezierPathWithRoundedRect:rect
                                                    cornerRadius:radius];
    [path addClip];
    [image drawInRect:rect];
    UIImage *roundedImage = UIGraphicsGetImageFromCurrentImageContext();
    UIGraphicsEndImageContext();

    dispatch_async(dispatch_get_main_queue(), ^{
        imageView.image = roundedImage;
    });
});

// 方案D：iOS 13+ cornerCurve
view.layer.cornerRadius = 10;
view.layer.cornerCurve = kCACornerCurveCircular;  // 或 kCACornerCurveContinuous
// iOS 优化了某些场景下的圆角渲染


// ========== 2. 阴影优化 ==========

// ⚠️ 必须设置 shadowPath
view.layer.shadowColor = [UIColor blackColor].CGColor;
view.layer.shadowOffset = CGSizeMake(0, 2);
view.layer.shadowOpacity = 0.5;
view.layer.shadowRadius = 4;
// ✅ 设置 shadowPath 避免离屏渲染
view.layer.shadowPath = [UIBezierPath bezierPathWithRoundedRect:view.bounds
                                                   cornerRadius:10].CGPath;


// ========== 3. 光栅化（shouldRasterize）适当使用 ==========

// 适合：内容不常变化的复杂视图（如带阴影、圆角的卡片）
view.layer.shouldRasterize = YES;
view.layer.rasterizationScale = UIScreen.mainScreen.scale;

// 不适合：内容频繁变化的视图（每次变化都要重新光栅化）
// 光栅化缓存有 100ms 过期时间，超过未复用则被清除
```

## 八、布局与绘制的触发机制

### 8.1 setNeedsLayout / layoutIfNeeded

```objc
// setNeedsLayout：标记"需要重新布局"
// 不会立即执行，等到下一个 RunLoop 周期的 Layout 阶段才执行
[view setNeedsLayout];

// layoutIfNeeded：如果有待处理的布局，立即执行
// 常用于动画中
[UIView animateWithDuration:0.3 animations:^{
    // 修改约束
    self.topConstraint.constant = 100;
    // 强制立即布局
    [self.view layoutIfNeeded];
}];

// layoutSubviews 的调用时机：
// 1. init 初始化不会触发（此时还没有 superview）
// 2. addSubview 时
// 3. frame / bounds 发生变化时
// 4. 滚动 UIScrollView 时
// 5. 旋转屏幕时
// 6. 手动调用 setNeedsLayout 后的下一个 Layout 阶段
```

### 8.2 setNeedsDisplay / drawRect

```objc
// setNeedsDisplay：标记"需要重新绘制"
// 不会立即执行，等到下一个 RunLoop 的 Display 阶段才执行
[view setNeedsDisplay];

// setNeedsDisplayInRect：只重绘指定区域
[view setNeedsDisplayInRect:CGRectMake(0, 0, 100, 100)];

// drawRect / draw(_:rect:) 的调用时机：
// 1. 视图首次显示时
// 2. 调用 setNeedsDisplay 后
// 3. ⚠️ 不要直接调用 drawRect，总是通过 setNeedsDisplay 间接触发

// ⚠️ 性能注意：
// 重写 drawRect 会导致系统为该视图分配一块 backing store（位图）
// 空的 drawRect 也会分配 → 浪费内存
// 如果不需要自定义绘制，不要重写 drawRect
```

### 8.3 CATransaction

```objc
// CATransaction 是 Core Animation 中的事务机制
// 所有图层属性的修改都在事务中提交

// 隐式事务（自动）：
// RunLoop 每次迭代开始时自动创建，结束时自动提交
// 所以同一次 RunLoop 中的多次修改会合并为一次提交

// 显式事务（手动）：
[CATransaction begin];
[CATransaction setDisableActions:YES];  // 禁用隐式动画
layer.position = CGPointMake(100, 100); // 没有动画，直接到位
[CATransaction commit];

// 设置动画时长
[CATransaction begin];
[CATransaction setAnimationDuration:0.5];
layer.opacity = 0;  // 0.5 秒渐隐
[CATransaction commit];

// 完成回调
[CATransaction begin];
[CATransaction setCompletionBlock:^{
    NSLog(@"动画完成");
}];
layer.position = newPosition;
[CATransaction commit];
```

### 8.4 RunLoop 与渲染的关系

```
RunLoop 每次迭代中与渲染相关的流程：

  RunLoop 唤醒
       ↓
  处理 Timer
       ↓
  处理 Source0（触摸事件等）
       ↓
  处理 Source1（端口事件）
       ↓
  BeforeWaiting 观察者回调  ← 关键！
  │
  ├─ 执行 setNeedsLayout 标记的 Layout
  ├─ 执行 setNeedsDisplay 标记的 Display
  ├─ 提交 CATransaction（Commit）
  │
  └─ 图层树被发送给 Render Server
       ↓
  RunLoop 休眠

所以：
  - setNeedsLayout/setNeedsDisplay 只是打标记
  - 真正执行是在 RunLoop 的 BeforeWaiting 阶段
  - 同一次 RunLoop 中多次调用 setNeedsLayout 只会触发一次 layoutSubviews
  - 这就是 iOS 的"延迟提交"机制，避免重复计算
```

## 九、卡顿检测与优化

### 9.1 卡顿的本质

```
卡顿 = 主线程在一帧时间（16.67ms）内没有完成工作

主线程在每帧需要完成：
  1. 事件处理
  2. 布局计算
  3. 绘制
  4. 提交图层树

如果任何一步耗时过长 → 错过 VSync → 掉帧 → 卡顿

常见卡顿原因（按阶段）：

  Layout 阶段：
  - Auto Layout 约束过多或嵌套过深
  - 频繁调用 layoutSubviews
  - CGSize 计算（如 boundingRect 计算文本高度）

  Display 阶段：
  - 复杂的 drawRect 自定义绘制
  - 大量文本排版（CoreText）

  Prepare 阶段：
  - 大图解码（PNG/JPEG → 位图，在主线程）
  - 图片格式不匹配（需要转换）

  Commit 阶段：
  - 图层数量过多（数千个）
  - 图层树过深

  GPU 阶段：
  - 大量离屏渲染
  - 大量透明度混合（blending）
  - 超大纹理
```

### 9.2 RunLoop 卡顿监控

```objc
// 原理：监控 RunLoop 的状态变化耗时
// 如果从 kCFRunLoopBeforeSources → kCFRunLoopBeforeWaiting 耗时太长
// 或 kCFRunLoopAfterWaiting 后太久没进入下一状态
// → 说明主线程卡住了

@interface LagMonitor : NSObject
@property (nonatomic, assign) CFRunLoopObserverRef observer;
@property (nonatomic, assign) CFRunLoopActivity currentActivity;
@property (nonatomic, strong) dispatch_semaphore_t semaphore;
@end

@implementation LagMonitor

- (void)start {
    self.semaphore = dispatch_semaphore_create(0);

    // 监听 RunLoop 状态变化
    CFRunLoopObserverRef observer = CFRunLoopObserverCreateWithHandler(
        NULL, kCFRunLoopAllActivities, YES, 0,
        ^(CFRunLoopObserverRef obs, CFRunLoopActivity activity) {
            self.currentActivity = activity;
            dispatch_semaphore_signal(self.semaphore);
        });

    CFRunLoopAddObserver(CFRunLoopGetMain(), observer, kCFRunLoopCommonModes);
    self.observer = observer;

    // 子线程检测
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        while (YES) {
            // 等待 50ms（可调整阈值）
            long result = dispatch_semaphore_wait(self.semaphore,
                dispatch_time(DISPATCH_TIME_NOW, 50 * NSEC_PER_MSEC));

            if (result != 0) {
                // 超时：主线程在 50ms 内没有状态变化
                if (self.currentActivity == kCFRunLoopBeforeSources ||
                    self.currentActivity == kCFRunLoopAfterWaiting) {
                    // ⚠️ 可能卡顿了，记录主线程堆栈
                    NSLog(@"主线程可能卡顿，当前 activity: %lu",
                          (unsigned long)self.currentActivity);
                }
            }
        }
    });
}

@end
```

## 十一、实战：自定义转场动画

### 11.1 基本结构

```objc
// 自定义转场需要三个角色：

// 1. TransitioningDelegate：告诉系统用谁来做动画
@interface MyVC () <UIViewControllerTransitioningDelegate>
@end

- (void)presentDetail {
    DetailVC *vc = [[DetailVC alloc] init];
    vc.modalPresentationStyle = UIModalPresentationCustom;
    vc.transitioningDelegate = self;
    [self presentViewController:vc animated:YES completion:nil];
}

// 返回动画控制器
- (id<UIViewControllerAnimatedTransitioning>)
    animationControllerForPresentedController:(UIViewController *)presented
                        presentingController:(UIViewController *)presenting
                            sourceController:(UIViewController *)source {
    return [[FadeInAnimator alloc] init];
}

- (id<UIViewControllerAnimatedTransitioning>)
    animationControllerForDismissedController:(UIViewController *)dismissed {
    return [[FadeOutAnimator alloc] init];
}


// 2. AnimatedTransitioning：具体的动画逻辑
@interface FadeInAnimator : NSObject <UIViewControllerAnimatedTransitioning>
@end

@implementation FadeInAnimator

- (NSTimeInterval)transitionDuration:(id<UIViewControllerContextTransitioning>)ctx {
    return 0.3;
}

- (void)animateTransition:(id<UIViewControllerContextTransitioning>)ctx {
    UIView *toView = [ctx viewForKey:UITransitionContextToViewKey];
    UIView *container = ctx.containerView;

    toView.alpha = 0;
    [container addSubview:toView];

    [UIView animateWithDuration:[self transitionDuration:ctx]
                     animations:^{
        toView.alpha = 1;
    } completion:^(BOOL finished) {
        [ctx completeTransition:!ctx.transitionWasCancelled];
    }];
}

@end


// 3. InteractiveTransitioning（可选）：交互式转场
// UIPercentDrivenInteractiveTransition 简化了交互式转场
@property (nonatomic, strong) UIPercentDrivenInteractiveTransition *interactor;

// 手势驱动
- (void)handlePanGesture:(UIPanGestureRecognizer *)pan {
    CGFloat progress = translation.y / self.view.bounds.size.height;
    progress = MIN(MAX(progress, 0), 1);

    switch (pan.state) {
        case UIGestureRecognizerStateBegan:
            self.interactor = [[UIPercentDrivenInteractiveTransition alloc] init];
            [self dismissViewControllerAnimated:YES completion:nil];
            break;
        case UIGestureRecognizerStateChanged:
            [self.interactor updateInteractiveTransition:progress];
            break;
        case UIGestureRecognizerStateEnded:
            if (progress > 0.5) {
                [self.interactor finishInteractiveTransition];
            } else {
                [self.interactor cancelInteractiveTransition];
            }
            self.interactor = nil;
            break;
        default:
            [self.interactor cancelInteractiveTransition];
            self.interactor = nil;
            break;
    }
}
```

## 十三、常见面试题

### Q1: 事件传递和事件响应的区别？

```
事件传递（Hit-Testing）：
  方向：UIApplication → UIWindow → 子视图（自上而下）
  目的：找到最合适的响应视图（Hit-Test View）
  方法：hitTest:withEvent: + pointInside:withEvent:
  顺序：父视图先判断，子视图倒序遍历

事件响应（Responder Chain）：
  方向：Hit-Test View → 父视图 → VC → Window → App（自下而上）
  目的：找到处理事件的响应者
  方法：touchesBegan/Moved/Ended/Cancelled
  传递：nextResponder 链

两者方向相反：
  传递是从上到下找目标，响应是从下到上找处理者。
```

### Q2: hitTest:withEvent: 的执行逻辑？

```
1. 检查三个前提（userInteractionEnabled、hidden、alpha > 0.01）
2. pointInside 判断点是否在自己范围内
3. 倒序遍历子视图，递归调用 hitTest
4. 子视图有命中 → 返回该子视图
5. 子视图都没命中 → 返回 self

核心特点：
  - 倒序遍历（后添加的先检查）
  - 找到即返回（不继续遍历）
  - 父视图 pointInside 为 NO 则子视图不检查
```

### Q3: 手势识别器和 touches 方法的优先级？

```
手势识别器优先级更高。

时序：
1. 触摸开始 → UIWindow 同时发给手势和 View 的 touchesBegan
2. 手势处于 Possible → touches 方法正常接收
3. 手势识别成功 → View 收到 touchesCancelled，后续事件只给手势
4. 手势识别失败 → touches 方法继续正常接收

可通过三个属性调整行为：
  cancelsTouchesInView（默认 YES）
  delaysTouchesBegan（默认 NO）
  delaysTouchesEnded（默认 YES）
```

### Q4: 什么是离屏渲染？哪些操作会触发？

```
正常渲染：GPU 直接合成到帧缓冲区。
离屏渲染：GPU 先渲染到临时缓冲区，再合成到帧缓冲区。

触发操作：
  cornerRadius + masksToBounds
  shadow（没设 shadowPath）
  mask
  group opacity
  shouldRasterize
  重写 drawRect（CPU 离屏）

开销来源：
  创建临时缓冲区 + 上下文切换
  → 比正常渲染慢，容易掉帧

优化方法：
  圆角 → 后台线程绘制圆角图片 / 只设 cornerRadius 不 clip
  阴影 → 设置 shadowPath
  复杂不变的图层 → shouldRasterize 缓存
```

### Q5: iOS 渲染一帧的完整流程？

```
1. VSync 信号到达
2. RunLoop 唤醒主线程
3. 处理触摸事件（Source0/Source1）
4. Layout 阶段：layoutSubviews，Auto Layout 求解
5. Display 阶段：drawRect（如果需要）
6. Prepare 阶段：图片解码
7. Commit 阶段：提交图层树给 Render Server
8. Render Server（GPU）：纹理合成
9. 帧缓冲区 → 屏幕显示

总预算：16.67ms（60fps）或 8.33ms（120fps）
超时 → 掉帧 → 卡顿
```

### Q6: setNeedsLayout 和 layoutIfNeeded 的区别？

```
setNeedsLayout：

layoutIfNeeded：

```

### Q7: UIView 和 CALayer 的关系？为什么要分两层？

```
UIView 持有一个 CALayer，是 layer 的 delegate。
UIView 负责事件处理和布局管理。
CALayer 负责视觉内容展示和动画。

分两层的原因：
  CALayer 是跨平台的（macOS + iOS 共用 Core Animation）
  UIView 是 iOS 特有的（macOS 用 NSView）

UIView 的属性（frame/alpha/backgroundColor 等）
```

### Q8: hitTesting 算法细节（z-order、子视图遍历）

```
hitTest(_:with:) 是 UIView 定位触摸落点的核心方法。
当触摸发生时，系统从窗口开始递归调用 hitTest，层层深入子视图。

算法流程：
  1. 判断触摸点是否在当前视图 bounds 内 → point(inside:with:)
     不通过 → 返回 nil，结束
  2. 遍历子视图数组，逆序（z-order：后 add 的在最上层）
     for subview in subviews.reversed() {
         let convertedPoint = convert(point, to: subview)
         if let hitView = subview.hitTest(convertedPoint, with: event) {
             return hitView  // 找到最顶层响应者，直接返回
         }
     }
  3. 所有子视图都没命中 → 返回 self

关键细节：
  - hidden == true → point(inside:) 返回 false，跳过
  - isUserInteractionEnabled == false → 同上，跳过
  - alpha < 0.01 → 视为不可交互，跳过
  - transform 会影响 bounds 和 point(inside:) 的判断

重写 hitTest 的常见场景：
  - 扩大点击区域（如小按钮增大热区）
  - 透视点击（让透明区域不拦截下方视图）

Pitfalls: hitTest 递归很深时性能差，避免在 hitTest 中做重型计算。
Best practice: 优先用 isUserInteractionEnabled 和 hidden 控制，避免滥用 hitTest 重写。
```

### Q9: UIGestureRecognizer 状态机（Possible → Began → Changed → Ended/Failed/Canceled）

```
UIGestureRecognizer 核心是一个有限状态机，通过触摸点的变化在不同状态间流转。

状态流转：
  Possible（初始）→ 手势开始时进入
  ↓ 触摸点满足手势条件
  Began（触发）→ didSetGestureState(.began) 调用一次
  ↓ 触摸点持续变化
  Changed（持续）→ didSetGestureState(.changed) 多次调用
  ↓ 手势完成
  Ended（结束）→ didSetGestureState(.ended) 调用一次
  ↓ 手势再次被识别（如连续点击）
  Possible → Began → ...（可循环）

失败与取消分支：
  Possible → Failed（条件不满足）→ 不可恢复，回到 Possible
  Began/Changed → Canceled（系统中断，如电话进来）→ 不可恢复

代码示例：
  let pan = UIPanGestureRecognizer(target: self, action: #selector(panned(_:)))
  func panned(_ gesture: UIPanGestureRecognizer) {
      switch gesture.state {
      case .began: print("开始拖动")
      case .changed: view.center.x += gesture.translation(in: view).x
                    gesture.setTranslation(.zero, in: view)
      case .ended: print("拖动结束")
      case .possible, .failed, .canceled: break
      @unknown default: break
      }
  }

Pitfalls: translation 不会自动清零，每次 .changed 后需手动 setTranslation(.zero) 防止累积偏移。
Best practice: 在 .ended 状态中做最终动画或位置修正，利用 velocity(in:) 实现惯性滚动效果。
```

### Q10: UIGestureRecognizer 代理方法（shouldRecognizeSimultaneously、shouldRequireFailureOf）

```
UIGestureRecognizerDelegate 提供了两个关键的代理方法，用于精细控制手势之间的协作与竞争关系。

核心代理方法：
  1. shouldRequireFailure(of:) → 控制互斥关系
     返回 true：被查询的手势必须失败后，当前手势才能触发
     返回 false：不强制互斥（默认行为）

  2. shouldRecognizeSimultaneously(with:) → 控制并行识别
     返回 true：允许两个手势同时触发
     返回 false：禁止同时识别（默认行为）

使用场景：
  - ScrollView 上叠加滑动手势：让 scrollView 的 pan 先失败，自定义 pan 才触发
  - 捏合缩放 + 平移同时工作：返回 true 允许双指同时缩放和平移
  - 避免冲突：长按手势不应与点击手势同时触发

代码示例：
  func gestureRecognizer(_ gesture: UIGestureRecognizer,
                         shouldRequireFailureOf other: UIGestureRecognizer) -> Bool {
      // 自定义 pan 等待 scrollView 原生 pan 失败后触发
      if other is UIScrollView.panGestureRecognizer { return true }
      return false
  }

  func gestureRecognizer(_ gesture: UIGestureRecognizer,
                         shouldRecognizeSimultaneously with other: UIGestureRecognizer) -> Bool {
      // 允许 pinch 和 pan 同时识别（双指缩放+平移）
      if (gesture is UIPinchGestureRecognizer && other is UIPanGestureRecognizer) ||
         (gesture is UIPanGestureRecognizer && other is UIPinchGestureRecognizer) {
          return true
      }
      return false
  }

Pitfalls: shouldRequireFailureOf 和 shouldRecognizeSimultaneously 方向相反 —— 在 A 的代理中返回 A requires B failure ≠ B requires A failure，需在两个手势的代理中都配置。
Best practice: 优先使用 built-in 的 require(toFail:) 方法代替代理，代码更简洁且不易出错。
```

### Q11: UIView touchesBegan/Moved/Ended vs UIGestureRecognizer 交互

当 UIView 同时有 touch 方法和 UIGestureRecognizer 时，系统优先级和行为如下：

核心规则：
  - UIGestureRecognizer 优先于 touches 方法
  - 如果手势成功识别，touchesBegan/Changed/Ended 不会调用
  - 如果手势识别失败或不存在手势，touches 方法正常触发
  - isExclusiveTouch = true 时，视图独占事件，同层其他视图的 touches 被阻塞

触控分发流程：
  1. hitTest 确定接收视图 → 2. touchesBegan 进入视图
  3. 所有 attached UIGestureRecognizer 进入 .possible → 开始识别
  4. 手势识别成功 → touches 方法被取消，手势 action 触发
  5. 手势识别失败 → touchesBegan/Changed/Ended 正常回调
  6. gestureRecognizerShouldBegin 返回 false → 立即走 touches 流程

代码示例：
  override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
      print("touch 被调用 — 手势未识别或不存在")
  }
  override func gestureRecognizerShouldBegin(_ gesture: UIGestureRecognizer) -> Bool {
      // 返回 false 可让 touches 流程接管
      return shouldUseGesture()
  }

Pitfalls: 在 ScrollView 子类中重写 touches 时，系统 pan 手势会吃掉所有 touch 事件，需用 gestureRecognizerShouldBegin 精细控制。
Best practice: 优先使用 UIGestureRecognizer，仅在需要低级别 touch 数据（如 pressure、contactArea）时才重写 touch 方法。

### Q12: UIResponder 完整的下一响应者链

UIResponder 的 next 属性构建了 iOS 事件传递的响应者链。当视图无法处理事件时，通过 next 将事件向上传递，形成完整的传播路径。

响应者链顺序（View → ViewController → ParentView → ... → Window → AppDelegate）：
  1. UIView.next → 如果嵌在 ViewController 中，返回该 ViewController
  2. UIView.next → 如果不在 ViewController 中，返回 superview
  3. UIViewController.next → 如果视图是根视图，返回父 ViewController
  4. UIViewController.next → 如果不是根视图，返回父 ViewController 的视图
  5. UIWindow.next → 返回 UIApplication
  6. UIApplication.next → 返回 AppDelegate（如果继承 UIResponder）

关键规则：
  - next 链是**单向的**，事件不可回传
  - 每个 responder 可拦截或继续传递（调用 super）
  - 重写 next 会破坏默认链，需极度谨慎

代码示例：
  override var next: UIResponder? {
      // 自定义响应者链 — 将事件直接传给特定管理器
      return eventDispatcher ?? super.next
  }

  override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
      if !handleSpecialTouch(touches) {
          super.touchesBegan(touches: touches, with: event) // 向 next 传递
      }
  }

Pitfalls: 在 ContainerViewController 场景中，子 VC 的 next 指向父 VC，但如果手动将子 VC 的 view 添加到另一个 ViewController 的 view 层级，响应者链会断裂 —— view 的 next 指向当前 superview 链而非逻辑上的父 VC。
Best practice: 永远不要重写 UIResponder.next，而是通过目标-动作(target-action)或 delegation 模式实现自定义事件处理。

### Q13: UIResponder canBecomeFirstResponder / canResignFirstResponder

UIResponder 通过 firstResponer 机制声明自己是当前事件的接收者。只有成为 first responder 的 responder 才能接收键盘输入、快捷键、和远程控制事件。

成为 First Responder 的流程：
  1. 调用 becomeFirstResponder() → 系统从当前 responder 开始遍历 canBecomeFirstResponder
  2. canBecomeFirstResponder 返回 true → 该 responder 成为 first responder
  3. 系统通知旧 first responder 调用 resignFirstResponder
  4. canResignFirstResponder 返回 true → 旧 responder 让出 first responder 状态

关键字段与行为：
  - isFirstResponder 属性：只读，表示当前是否为 first responder
  - 同一响应者链上同时只能有一个 first responder
  - 键盘弹出/隐藏与 first responder 状态直接关联
  - UITextField/UITextView 在获得焦点时自动调用 becomeFirstResponder

代码示例：
  class CustomView: UIView {
      override var canBecomeFirstResponder: Bool {
          return true  // 允许此视图成为 first responder
      }
      override var canResignFirstResponder: Bool {
          return true  // 允许让出 first responder 状态
      }
      override var canPerformAction(_ action: Selector, withSender sender: Any?) -> Bool {
          return action == #selector(copyAction)  // 支持复制菜单
      }
      func becomeFirst() {
          self.becomeFirstResponder()
          print("isFirstResponder:", self.isFirstResponder)
      }
  }

Pitfalls: 重写 canBecomeFirstResponder 返回 false 后，becomeFirstResponder() 调用静默失败 —— isFirstResponder 仍为 false，无任何错误或警告日志。调试时需检查 canBecomeFirstResponder 返回值而非 becomeFirstResponder 的返回值。
Best practice: 在自定义视图需要接收键盘/触摸事件时重写 canBecomeFirstResponder，配合 canPerformAction 实现 UIMenu 上下文菜单（如复制、粘贴）。

### 键盘通知与输入视图

iOS 通过 NotificationCenter 广播键盘显示/隐藏事件，开发者监听 `UIKeyboardWillShowNotification`、`UIKeyboardDidShowNotification`、`UIKeyboardWillHideNotification`、`UIKeyboardDidHideNotification` 来调整 UI 布局。`inputView` 和 `inputAccessoryView` 允许 UITextField/UITextView 替换系统键盘或添加自定义工具栏。

代码示例：
```swift
class ChatViewController: UIViewController {
    var keyboardHeightConstraint: NSLayoutConstraint!

    override func viewDidLoad() {
        super.viewDidLoad()
        NotificationCenter.default.addObserver(self, selector: #selector(keyboardWillShow(_:)),
                                                name: .UIKeyboardWillShow, object: nil)
        NotificationCenter.default.addObserver(self, selector: #selector(keyboardWillHide(_:)),
                                                name: .UIKeyboardWillHide, object: nil)
    }

    @objc func keyboardWillShow(_ notif: Notification) {
        let frame = notif.userInfo?[UIKeyboardFrameEndUserInfoKey] as AnyObject
        let height = frame.cgRectValue.height
        keyboardHeightConstraint.constant = height
        UIView.animate(withDuration: 0.25) { self.view.layoutIfNeeded() }
    }

    @objc func keyboardWillHide(_ notif: Notification) {
        keyboardHeightConstraint.constant = 0
        UIView.animate(withDuration: 0.25) { self.view.layoutIfNeeded() }
    }
}
```

Pitfalls: iPad 分屏模式下键盘帧可能小于预期；`UIKeyboardFrameEndUserInfoKey` 返回 window 坐标系，需 `convertRect(_:from:)` 转换。表情键盘↔文字键盘切换不会触发 WillShow/WillHide。
Best practice: 使用 `UIKeyboardVisibleUserInfoKey` (iOS 16+) 判断键盘可见性；自定义 `inputAccessoryView` 返回 `UIToolbar` 以自动适配系统宽度。

### Core Animation 渲染管线（layer tree → render server → GPU）

Core Animation 的渲染管线分为三个阶段：(1) **Layer Tree 构建** — 在主线程更新 CALayer 属性，生成隐式/显式动画描述；(2) **Render Server** — 后台进程接收 layer tree 变化，执行隐式动画计算、光栅化、位图合成；(3) **GPU 呈现** — 通过 IOSurface 将合成结果传递给 GPU，最终由 Display Server 输出到屏幕。Core Animation 的核心优势在于：即使主线程阻塞，Render Server 仍可独立渲染已提交的帧，保证动画流畅。

代码示例：
```swift
func configureLayer() {
    layer.cornerRadius = 12
    layer.masksToBounds = true          // 触发 GPU 层光栅化
    layer.shouldRasterize = true         // 位图缓存，提升复杂层级性能
    layer.rasterizationScale = UIScreen.main.scale  // 避免 Retina 模糊
    layer.drawsAsynchronously = true     // iOS 11+，GPU 异步绘制
}
```

Pitfalls: `shouldRasterize = true` 对动画属性（如 transform、opacity）无效——栅化发生在动画开始前，动画仍由 Render Server 实时合成，额外栅化反而浪费显存。
Best practice: 仅在静态复杂层级（如大量子层、mask、滤镜）启用 rasterization；动画优先使用隐式动画（modifiable properties：position、transform、opacity、backgroundColor）以保证 Render Server 高效合成。

### UIView vs CALayer 动画差异

UIView 的动画本质是 CALayer 动画的封装。`UIView.animate()` 内部创建 `CABasicAnimation` 并提交到层的 transaction。UIView 动画自动更新模型值（如 `view.frame` 变为终点值），而 CALayer 的隐式动画**不更新** `model layer` 属性，仅动画 `presentation layer`，导致动画结束后 layer 状态与预期不符。

核心差异：
- **UIView.animate**: model 值自动更新为终点值，支持 completion 回调
- **CALayer 隐式动画**: presentation layer 动画，model 值不变，无 completion
- **CABasicAnimation + setValue**: 需手动设置 `fromValue/toValue` 并在完成后更新 model

```swift
// UIView 动画 —— 自动更新 frame，completion 可感知
UIView.animate(withDuration: 0.3) {
    view.frame.origin.x = 100  // model 值变为终点
} completion: { _ in print("done") }  // 触发回调

// CALayer 隐式动画 —— 仅 presentation layer 动，model 不变
layer.position = CGPoint(x: 100, y: layer.position.y)  // model 立即改变
// 动画由 CATransaction 自动触发，无 completion
```

Pitfalls: UIView 动画中修改 `layer.bounds` 触发的是隐式动画（不走 completion），而修改 `frame` 走 UIView 动画块 —— 同一视图混合使用两套 API 导致动画冲突和位置跳变。
Best practice: 统一使用 UIView.animate 进行简单动画；需要精细控制（spring、关键帧、层属性）时切换至 `UIViewPropertyAnimator` 或 `CAAnimation`，避免在同一视图中混用。


### CATransaction 隐式动画控制

CATransaction 是 Core Animation 的默认动画容器。每次修改 layer 的可动画属性（position、transform、opacity 等），CATransaction 自动包裹为 `CABasicAnimation` 并提交给 Render Server。通过 `CATransaction.setAnimationDuration()` 或 `disableActions()` 可精细控制行为。隐式动画是 Core Animation 的默认行为，无需显式创建 CAAnimation 对象。

代码示例：
```swift
// 禁用隐式动画 —— layer 属性立即生效，无过渡效果
CATransaction.begin()
CATransaction.setDisableActions(true)
view.layer.position = CGPoint(x: 100, y: 200)
view.layer.opacity = 0.5
CATransaction.commit()

// 自定义隐式动画时长和时序
CATransaction.begin()
CATransaction.setAnimationDuration(0.6)
CATransaction.setAnimationTimingFunction(CAMediaTimingFunction(name: .easeIn))
view.layer.transform = CGAffineTransform(scaleX: 1.2, y: 1.2)
CATransaction.commit()
```

Pitfalls: CATransaction 是线程安全的但嵌套时仅最内层 commit 触发提交，多层 begin/commit 嵌套时 setDisableActions 仅作用于当前层级，外层仍可能触发隐式动画。
Best practice: 批量修改多个 layer 属性时统一包裹在一个 begin/commit 块中，避免中间提交导致分步动画；禁用动画时优先使用 CATransaction.setDisableActions(true) 而非手动创建 CAAnimation。

### displayLink vs timer 用于帧精确动画

CADisplayLink 与 Timer 的核心差异在于触发机制。CADisplayLink 绑定到 display server 的刷新周期（通常 60Hz/120Hz），在每帧合成前回调，天然与屏幕刷新同步。Timer 基于 runloop 调度，受线程负载、优先级和 timer 精度（默认约 50ms 容差）影响，无法保证帧精确性。

```swift
// CADisplayLink —— 帧精确动画，自动适配刷新率
let link = CADisplayLink(target: Proxy(self), selector: #selector(tick))
link.add(to: .main, forMode: .common)
// link.preferredFramesPerSecond = 30  // 限制帧率
// link.duration  → 当前帧预算（如 1/60 ≈ 0.0167s）

// Timer —— 不适合帧同步动画
let timer = Timer.scheduledTimer(withTimeInterval: 1.0/60.0, repeats: true) { _ in
    // 实际间隔受 runloop 负载影响，可能在 16ms ~ 50ms 之间波动
}
```

Pitfalls: CADisplayLink 在锁屏、切换后台、或导航控制器遮罩动画期间自动暂停回调，但不会自动 invalidate —— 需手动在 `deinit` 或 `viewDidDisappear` 中调用 `link.invalidate()` 防止野引用 crash。
Best practice: 动画循环统一使用 CADisplayLink，通过 `link.duration` 计算 delta time 实现帧率无关的运动逻辑；简单定时任务（非视觉）使用 Timer 或 async/await 延迟。

### CADisplayLink 时序与暂停/时间戳

CADisplayLink 提供精确的帧时序信息。每次回调时，`displayLink.timestamp` 返回基于 `mach_absolute_time` 的高精度时间戳（纳秒级），`displayLink.duration` 返回当前帧的实际间隔（非固定值），`displayLink.target` 触发选择器可获取累积偏移。当屏幕刷新率动态变化（如 ProMotion 60Hz↔120Hz）时，`duration` 自动跟随新周期，无需手动适配。

```swift
class AnimationLoop: NSObject {
    private var displayLink: CADisplayLink!
    private var lastTimestamp: TimeInterval = 0
    
    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(tick(_:)))
        displayLink.add(to: .main, forMode: .common)
    }
    
    @objc func tick(_ link: CADisplayLink) {
        let dt = link.duration  // 帧间隔，如 0.0167 (60Hz) 或 0.0083 (120Hz)
        let timestamp = link.timestamp  // 纳秒级绝对时间
        // 使用 dt 做帧率无关计算：velocity * dt
        if lastTimestamp == 0 { lastTimestamp = timestamp; return }
        let elapsed = (timestamp - lastTimestamp) / 1_000_000_000  // 秒
        lastTimestamp = timestamp
        // 根据 elapsed 更新位置，不依赖固定帧率
    }
    
    deinit { displayLink?.invalidate() }
}
```

Pitfalls: 进入后台时 CADisplayLink 自动暂停回调，回到前台后 `timestamp` 会跳跃（非连续递增），直接用 timestamp 差值计算 dt 会导致首次帧出现巨大 delta 值，产生画面跳帧。
Best practice: 用 `link.duration` 代替 timestamp 差值计算 delta time，duration 在每次回调时自动反映真实帧间隔且不会出现跳跃值；在 `viewDidDisappear` 中 `invalidate()` 避免后台野引用。

### UIView 动画完成回调与弹簧动画

UIView 动画的 completion 回调在动画结束时触发，但 `finished` 参数区分正常结束与被打断的场景。弹簧动画通过 damping 和 velocity 参数模拟物理弹性效果，比标准的 easeIn/easeOut 更自然。UIViewPropertyAnimator（iOS 10+）提供可中断、可反转、可手势驱动的动画控制，是交互式动画的首选 API。

```swift
// 1. 弹簧动画 —— 阻尼 < 1.0 产生回弹效果
UIView.animate(withDuration: 0.6, delay: 0,
              usingSpringWithDamping: 0.7,    // 越小弹越猛
              initialSpringVelocity: 2.0,      // 起始速度（归一化系数）
              options: [], animations: {
    button.transform = CGAffineTransform(scaleX: 0.9, y: 0.9)
}, completion: { finished in
    if finished {
        // 正常完成 —— 可链式触发下一步动画
        self.animateNextStep()
    } else {
        // 被中断（推送通知、手势取消等）—— 不依赖目标状态
    }
})

// 2. UIViewPropertyAnimator —— 手势驱动动画
let animator = UIViewPropertyAnimator(duration: 0.35, curve: .easeOut) {
    sheetView.frame = expandedFrame
}
@objc func handlePan(_ g: UIPanGestureRecognizer) {
    let fraction = translation.y / maxDrag
    animator.fractionComplete = max(0, min(1, fraction))
    if g.state == .ended {
        if fraction > 0.5 { animator.finishAnimation(at: .end) }
        else { animator.reverseAnimation() }
    }
}
```

Pitfalls: initialSpringVelocity 是归一化速度系数而非绝对像素值，动画距离越长实际初速度越大；damping 设为 1.0 时退化为临界阻尼，等同于 easeOut。completion 中 finished == false 时继续执行后续逻辑会导致状态不一致。
Best practice: 物理感交互反馈（点赞抖动、下拉刷新回弹）优先用 spring 动画；手势跟随动画（抽屉展开、卡片翻转）统一用 UIViewPropertyAnimator，避免混用多个 animate 块造成冲突。
## UIViewPropertyAnimator / UIDynamicAnimator

`UIViewPropertyAnimator` 是 iOS 10 引入的基于隐式事务的动画API，支持中断、反转、暂停与恢复，通过 `UIServicingAnimator` 协议扩展。相比传统的 `UIView.animate`，它支持交互式手势驱动（如拖拽式关闭）。`UIDynamicAnimator` 则提供基于物理引擎的模拟（重力、弹性碰撞、旋转），适合实现自然运动效果。

```swift
// UIViewPropertyAnimator - 支持中断的手势动画
let animator = UIViewPropertyAnimator(duration: 0.35, dampingRatio: 0.8) {
    view.frame.origin.x = newOrigin
}
animator.startAnimation()

// 手势拖动时动态更新 Fraction Complete
gestureRecognizer.state == .changed { _ in
    animator.fractionComplete = dragRatio  // 随手势实时更新
}
gestureRecognizer.state == .ended { _ in
    animator.finishAnimation(at: .current)  // 暂停在当前帧
}
```

- **Pitfalls**: `fractionComplete` 在 `startAnimation()` 前设置无效；`UIDynamicAnimator` 的碰撞行为需在 `UIDynamicItem` 上实现协议方法，遗漏则物理模拟失效。
- **Best Practice**: 优先使用 `UIViewPropertyAnimator` 替代 `UIView.animate` 做可中断动画；`UIDynamicAnimator` 仅用于物理效果，避免与 Auto Layout 混用。


### 8.5 布局引擎管线：Auto Layout → layoutSubviews → drawRect

iOS 视图渲染的核心管线是三步流水线：Auto Layout 计算约束 → layoutSubviews 安排子视图框架 → drawRect 执行自定义绘制。这个管线由 RunLoop 在每帧的 `CADisplayLink` 时机统一调度，确保布局与绘制同步。关键触发入口是 `setNeedsLayout()` 标记布局脏状态，系统在 `CARunInLoop` 阶段批量执行 `layoutIfNeeded()` 完成约束求解。

```swift
// 手动同步刷新布局 —— 避免下一帧才生效
view.setNeedsLayout()       // 标记脏标记（异步）
view.layoutIfNeeded()       // 强制立即执行 layoutSubviews
// 配合动画实现约束过渡
UIView.animate(withDuration: 0.3) {
    self.topConstraint.constant = 100  // 修改约束
    view.layoutIfNeeded()              // 在动画块内同步布局
}
// layoutSubviews 中读取 frame 进行子视图排列
override func layoutSubviews() {
    super.layoutSubviews()
    childView.frame = CGRect(x: 10, y: 10, width: bounds.width - 20, height: 50)
}
```

Pitfalls: 在 `layoutSubviews` 中调用 `setNeedsLayout` 会形成递归布局循环，导致性能灾难；在 `viewDidLoad` 中读取 frame 值为零，约束尚未求解。
Best practice: 约束动画必须将 `layoutIfNeeded()` 放入 `UIView.animate` 块内；避免在 `layoutSubviews` 中做耗时计算，将预计算结果缓存为类属性。

### CATransaction 完成回调 vs UIView 动画完成回调

`CATransaction` 的 completion block 在 **事务提交时立即调用**（动画开始前），而 `UIView.animate` 的 completion 在 **动画结束后调用**。这是最常见的动画回调陷阱——Core Animation 的事务机制中，completion 是在 render server 收到变更时同步触发的，而非等动画播放完毕。

```swift
// ❌ CATransaction completion — 动画开始前就触发了
CATransaction.begin()
layer.opacity = 0
CATransaction.setCompletionBlock {
    print("立即执行！动画还没结束")  // 在 commit 时同步调用
}
CATransaction.commit()

// ✅ UIView.animate completion — 动画结束后触发
UIView.animate(withDuration: 0.3, animations: {
    view.alpha = 0
}, completion: { finished in
    print("动画真正结束")  // 300ms 后调用
})

// ✅ 需要 layer 级别的结束回调时用 delegate + didMediaStop
layer.delegate = self
// MARK: - CAAnimationDelegate
func animationDidStop(_ anim: CAAnimation, finished flag: Bool) {
    print("layer 动画播放完毕")
}
```

Pitfalls: 在 CATransaction 的 completion 中读取 layer 的 presentationLayer 为 nil，因为动画尚未提交到 render server；混用 CATransaction completion 和 UIView completion 会导致回调时序混乱。
Best practice: 需要精确控制动画结束时机时，统一使用 `UIView.animate` 的 completion 或 `CAAnimationDelegate.animationDidStop`，避免依赖 CATransaction completion 做动画后处理。

### 触发隐式动画的 Core Animation layer 属性

Core Animation 中，直接修改 layer 的特定属性会自动触发 **隐式动画**（Implicit Animation），无需显式创建 `CABasicAnimation`。这是由 `CATransaction` 自动包装变更并生成过渡动画实现的，默认时长 0.25 秒，使用 `kCAMediaTimingFunctionEaseInEaseOut` 缓动曲线。

```swift
// 以下属性修改自动触发隐式动画（无需 UIView.animate）
layer.position = CGPoint(x: 100, y: 200)      // 位移动画
layer.opacity = 0.5                             // 透明度渐变
layer.transform = CATransform3DMakeScale(1.2, 1.2, 1)  // 缩放
layer.backgroundColor = UIColor.red.cgColor     // 颜色过渡
layer.cornerRadius = 15                         // 圆角变化
layer.shadowOffset = CGSize(width: 5, height: 5) // 阴影位移

// ⚠️ 以下属性【不】触发隐式动画
layer.bounds = CGRect(x: 0, y: 0, width: 100, height: 100)  // 无动画！
layer.frame = ...                            // 无动画（bounds 也不触发）
layer.contents = UIImage(named: "new")?.cgImage // 无动画

// 批量修改时禁用隐式动画
CATransaction.begin()
CATransaction.setDisableActions(true)  // 关闭隐式动画
layer.position = CGPoint(x: 50, y: 50)
layer.opacity = 0
CATransaction.commit()
```

- **Pitfalls**: `bounds` 和 `frame` 修改不触发隐式动画，直接跳变；在 `viewWillAppear` 中修改 layer 属性会触发不期望的入场动画，导致与过渡动画冲突。
- **Best Practice**: 需要瞬时变更时用 `CATransaction.setDisableActions(true)` 包裹；批量修改多个属性时始终包裹在同一个事务中，避免逐个属性触发动画闪烁。
