# iOS 性能优化详解

---

## 一、启动优化

### 1.1 冷启动与热启动

```
冷启动（Cold Launch）：
  App 进程不在内存中，从零开始加载
  整个流程：创建进程 → dyld 加载 → Runtime 初始化 → main() → UI 显示

热启动（Warm Launch）：
  App 进程还在内存中（被挂起但没被杀死）
  恢复速度极快，几乎无优化空间

iOS 13+ 引入了 "预热启动"（Pre-warming）：
  系统可能提前创建进程并执行到 main() 之前
  → 不能依赖启动时间做日志上报的起点

性能指标：
  Apple 建议冷启动 ≤ 400ms（首帧可交互）
  用户体验目标：1 秒内看到有意义的内容
```

### 1.2 启动流程

```
冷启动分为 pre-main 和 post-main 两个阶段：

┌──────── pre-main（系统控制）────────┐
│                                      │
│  1. 创建进程                         │
│  2. dyld 加载：                      │
│     - 加载 Mach-O 可执行文件         │
│     - 递归加载所有动态库（dylib）     │
│     - Rebase：修复 ASLR 偏移         │
│     - Bind：绑定外部符号             │
│  3. Runtime 初始化：                 │
│     - +load 方法（所有类和分类）      │
│     - C++ 静态构造函数               │
│     - __attribute__((constructor))   │
│  4. UIApplicationMain()              │
│                                      │
└──────────────────────────────────────┘
              ↓
┌──────── post-main（开发者控制）──────┐
│                                      │
│  5. application:didFinishLaunching...│
│     - SDK 初始化                     │
│     - 配置读取                       │
│     - 数据库初始化                   │
│     - 网络请求                       │
│  6. 首帧渲染                         │
│     - 创建 UIWindow + rootVC         │
│     - viewDidLoad / viewWillAppear   │
│     - 首次 layoutSubviews / drawRect │
│     - CATransaction commit           │
│     - Render Server 渲染             │
│  7. 首帧上屏 → 启动结束              │
│                                      │
└──────────────────────────────────────┘

度量 pre-main 耗时：
  Xcode → Edit Scheme → Arguments
  → 添加环境变量 DYLD_PRINT_STATISTICS = 1
  → 控制台会输出各阶段耗时
```

### 1.3 pre-main 优化

```
1. 减少动态库数量
   每个动态库都需要加载、Rebase、Bind
   Apple 建议不超过 6 个自定义动态库
   方法：合并多个小动态库为一个、改用静态库

2. 减少 +load 方法
   +load 在 main() 之前同步执行，阻塞启动
   ❌ 在 +load 中做大量初始化
   ✅ 改用 +initialize（懒加载，首次收到消息时调用）
   ✅ 改用 dispatch_once 延迟初始化

3. 减少 ObjC 类和分类数量
   Runtime 初始化时要注册所有类和分类
   合并小分类、删除无用类
   → 直接减少 Rebase/Bind 的工作量

4. 减少 C++ 静态构造函数
   __attribute__((constructor)) 和 C++ 全局对象
   都在 main() 之前执行

5. 二进制重排（Page Fault 优化）
   启动时访问的代码分布在不同的内存页
   每次访问新页 → Page Fault → 从磁盘加载（~0.1-0.8ms / 次）
   把启动相关的函数排列到相邻的内存页 → 减少 Page Fault
   
   方法：
   Clang 插桩（-fsanitize-coverage=func）获取启动调用顺序
   生成 order file → Xcode Build Settings → Order File
```

### 1.4 post-main 优化

```objc
// 核心原则：首帧前只做必须的工作，其余延迟或异步

// ========== 1. 分级初始化 ==========
- (BOOL)application:(UIApplication *)application
    didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

    // 必须同步：影响首屏展示
    [self setupWindow];
    [self setupRootViewController];

    // 必须同步但可以精简：只做最小初始化
    [CrashReporter enableWithMinimalSetup];

    // 异步执行：不影响首屏
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        [Analytics setup];
        [PushManager registerForRemotePush];
        [DatabaseManager warmUpCache];
    });

    // 延迟执行：首屏显示后再做
    dispatch_after(dispatch_time(DISPATCH_TIME_NOW, (int64_t)(2 * NSEC_PER_SEC)),
                   dispatch_get_main_queue(), ^{
        [self setupNonCriticalSDKs];
        [self preloadNextScreenData];
    });

    return YES;
}


// ========== 2. 首屏数据优化 ==========
// 缓存上次数据 → 先展示缓存 → 异步请求新数据 → 增量更新

- (void)viewDidLoad {
    [super viewDidLoad];

    // 先展示缓存数据（瞬间显示）
    NSArray *cachedData = [LocalCache loadHomePageData];
    if (cachedData.count > 0) {
        self.dataSource = cachedData;
        [self.tableView reloadData];
    }

    // 异步请求最新数据
    [API fetchHomePageData:^(NSArray *freshData) {
        self.dataSource = freshData;
        [self.tableView reloadData];
        [LocalCache saveHomePageData:freshData];
    }];
}


// ========== 3. 懒加载非首屏模块 ==========
// TabBar 的非首页 VC 不需要在启动时创建

- (UIViewController *)viewControllerForTab:(NSInteger)index {
    // 只在切换到对应 Tab 时才创建
    switch (index) {
        case 0: return self.homeVC ?: (self.homeVC = [HomeVC new]);
        case 1: return self.searchVC ?: (self.searchVC = [SearchVC new]);
        case 2: return self.profileVC ?: (self.profileVC = [ProfileVC new]);
        default: return nil;
    }
}
```

## 二、内存优化

### 2.1 iOS 内存基础

```
iOS 内存结构：
  虚拟内存：每个进程独立的地址空间（64 位 → 最大 4GB 可用）
  物理内存：实际的 RAM（iPhone 根据型号 3-8GB）
  iOS 没有 Swap（不像 macOS 有磁盘交换）
  → 内存紧张时系统直接杀进程

内存类型：
  Clean Memory：可以被系统回收并重新从磁盘加载的内存
    代码段、Framework、内存映射文件、只读数据
  Dirty Memory：无法被回收，只能保留在 RAM 中的内存
    堆上的对象、全局变量、运行时创建的数据
  Compressed Memory：iOS 7+ 的内存压缩机制
    系统把不活跃的 Dirty Memory 压缩存放
    访问时解压 → 比 Swap 快但消耗 CPU

内存警告级别：
  系统根据内存压力分为四个级别
  Normal → Warn → Critical → 被杀
  App 被杀的优先级：后台 App 先杀，前台 App 最后杀

App 内存上限（参考值，因设备而异）：
  前台约 1.5-2GB（高端设备）
  超出 → 被 Jetsam 机制杀掉（OOM - Out of Memory）
```

### 2.3 常见内存泄漏场景与修复

```objc
// ========== 1. Block 循环引用 ==========
// ❌ self → block → self
self.completionBlock = ^{
    [self doSomething];  // block 强引用 self
};

// ✅ weak-strong dance
__weak typeof(self) weakSelf = self;
self.completionBlock = ^{
    __strong typeof(weakSelf) strongSelf = weakSelf;
    if (!strongSelf) return;
    [strongSelf doSomething];
};


// ========== 2. Delegate 循环引用 ==========
// ❌ self → child → delegate → self
child.delegate = self;

// ✅ delegate 用 weak
@property (nonatomic, weak) id<MyDelegate> delegate;


// ========== 3. NSTimer 循环引用 ==========
// ❌ self → timer → target(self) → self 不释放
self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                              target:self
                                            selector:@selector(tick)
                                            userInfo:nil
                                             repeats:YES];
// self 的 dealloc 永远不会被调用 → timer 也永远不会 invalidate

// ✅ 方案A：iOS 10+ block API（block 中 weak 引用）
__weak typeof(self) weakSelf = self;
self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0 repeats:YES block:^(NSTimer *t) {
    [weakSelf tick];
}];

// ✅ 方案B：中间代理对象（Proxy）
// NSProxy 子类弱引用 target，timer 强引用 proxy
@interface WeakProxy : NSProxy
@property (nonatomic, weak) id target;
+ (instancetype)proxyWithTarget:(id)target;
@end

self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
    target:[WeakProxy proxyWithTarget:self]
    selector:@selector(tick) userInfo:nil repeats:YES];

// ✅ 方案C：在合适的时机手动 invalidate
- (void)viewWillDisappear:(BOOL)animated {
    [super viewWillDisappear:animated];
    [self.timer invalidate];
    self.timer = nil;
}


// ========== 4. 通知未移除（iOS 9 以下）==========
// iOS 9+ 系统会在对象释放时自动移除通知观察者
// iOS 9 以下需要手动移除：
- (void)dealloc {
    [[NSNotificationCenter defaultCenter] removeObserver:self];
}


// ========== 5. CADisplayLink / performSelector:afterDelay: ==========
// CADisplayLink 同 NSTimer，强引用 target
// 解决方案同 Timer

// performSelector:afterDelay: 会持有 target 直到执行完毕
// 页面退出时取消：
[NSObject cancelPreviousPerformRequestsWithTarget:self];
```

### 2.4 大图与内存优化

```objc
// 图片内存计算：
// 内存占用 = width × height × 4 bytes（RGBA）
// 例：4000×3000 的照片 = 4000 × 3000 × 4 = 48 MB
// 但屏幕上 imageView 可能只有 200×150pt → 浪费了大量内存


// ========== 1. 图片降采样（Downsampling）==========
// 不把整张大图解码到内存，只解码需要的尺寸

+ (UIImage *)downsampledImageAtURL:(NSURL *)url maxPixelSize:(CGFloat)maxPixelSize {
    NSDictionary *options = @{
        (id)kCGImageSourceShouldCache: @NO  // 不缓存原始数据
    };
    CGImageSourceRef source = CGImageSourceCreateWithURL((__bridge CFURLRef)url, 
                                                         (__bridge CFDictionaryRef)options);
    if (!source) return nil;

    NSDictionary *downsampleOptions = @{
        (id)kCGImageSourceCreateThumbnailFromImageAlways: @YES,
        (id)kCGImageSourceShouldCacheImmediately: @YES,  // 立即解码
        (id)kCGImageSourceCreateThumbnailWithTransform: @YES,
        (id)kCGImageSourceThumbnailMaxPixelSize: @(maxPixelSize)
    };

    CGImageRef downsampledImage = CGImageSourceCreateThumbnailAtIndex(
        source, 0, (__bridge CFDictionaryRef)downsampleOptions);
    CFRelease(source);

    UIImage *result = [UIImage imageWithCGImage:downsampledImage];
    if (downsampledImage) CFRelease(downsampledImage);
    return result;
}

// 使用：只加载 imageView 需要的尺寸
CGFloat maxPixel = self.imageView.bounds.size.width * UIScreen.mainScreen.scale;
UIImage *small = [self downsampledImageAtURL:url maxPixelSize:maxPixel];
self.imageView.image = small;
// 4000×3000 → 200×150@3x → 只占 600×450×4 = 1.08 MB（比 48MB 小 44 倍）


// ========== 2. imageNamed vs imageWithContentsOfFile ==========
// imageNamed：系统缓存，适合频繁使用的小图（icon 等）
// imageWithContentsOfFile：不缓存，适合大图或一次性使用的图
// ⚠️ 大量大图用 imageNamed 会导致内存暴涨

// 频繁使用的小图
UIImage *icon = [UIImage imageNamed:@"tab_home"];

// 大图或一次性图片
NSString *path = [[NSBundle mainBundle] pathForResource:@"background" ofType:@"jpg"];
UIImage *bg = [UIImage imageWithContentsOfFile:path];


// ========== 3. AutoreleasePool 控制 ==========
// 批量创建临时对象时，嵌套 @autoreleasepool 及时释放
for (int i = 0; i < 10000; i++) {
    @autoreleasepool {
        UIImage *image = [self processImage:i];
        [self saveImage:image atIndex:i];
        // 每次循环结束时 image 立即释放，不会累积
    }
}
```

### 2.5 内存警告处理

```objc
// ========== UIViewController ==========
- (void)didReceiveMemoryWarning {
    [super didReceiveMemoryWarning];
    // 释放可重建的资源
    if (self.isViewLoaded && !self.view.window) {
        // 视图不在屏幕上，可以释放
        self.view = nil;
    }
    [self.imageCache removeAllObjects];
}

// ========== AppDelegate ==========
- (void)applicationDidReceiveMemoryWarning:(UIApplication *)application {
    // 全局内存清理
    [[NSURLCache sharedURLCache] removeAllCachedResponses];
    [[SDImageCache sharedImageCache] clearMemory];
}

// ========== 通知方式 ==========
[[NSNotificationCenter defaultCenter]
    addObserverForName:UIApplicationDidReceiveMemoryWarningNotification
    object:nil queue:nil usingBlock:^(NSNotification *note) {
        [self clearCaches];
    }];

// ⚠️ NSCache 的优势：
// NSCache 在内存紧张时自动释放内容（LRU 策略）
// 不需要手动清理，比 NSMutableDictionary 更安全
NSCache *cache = [[NSCache alloc] init];
cache.countLimit = 100;      // 最多 100 个
cache.totalCostLimit = 50 * 1024 * 1024;  // 最多 50MB
```

## 三、渲染优化

### 3.1 卡顿的常见原因

```
回顾渲染流水线的每个阶段，找到卡顿瓶颈：

CPU 瓶颈（App 侧）：
  Layout：Auto Layout 约束过多或过深
  Display：复杂的 drawRect 绘制
  Prepare：大图解码
  Commit：图层数过多

GPU 瓶颈（Render Server 侧）：
  离屏渲染（圆角 + clip、阴影、mask）
  大量半透明混合（blending）
  超大纹理（大于 GPU 纹理上限 4096×4096）
  大量几何变换

判断是 CPU 还是 GPU 瓶颈：
  Instruments → Core Animation
  → 看 FPS 和各种 Debug 选项
  如果 CPU 占用高 → CPU 瓶颈
  如果 GPU 负载高但 CPU 不高 → GPU 瓶颈
```

### 3.2 离屏渲染优化（详见事件响应链与渲染详解）

```objc
// 快速总结：

// 圆角
// ❌ layer.cornerRadius + masksToBounds → 离屏渲染
// ✅ 后台绘制圆角图片 / 只设 cornerRadius 不 clip
// ✅ 用 CAShapeLayer 做 mask（某些场景）

```

### 3.3 Blending 优化

```objc
// Blending = GPU 对半透明像素的混合计算
// 每个像素要根据上下图层的 alpha 做混合 → 开销大

// 检测：Instruments → Core Animation → Color Blended Layers
// 红色区域 = 有混合，绿色 = 无混合

// 优化方法：

// 1. 设置 opaque 和 backgroundColor
view.opaque = YES;
view.backgroundColor = [UIColor whiteColor];
// 告诉 GPU 这个视图不透明 → 跳过混合计算

// 2. UILabel 的背景色
label.backgroundColor = [UIColor whiteColor];
// ⚠️ UILabel 默认 backgroundColor 是 nil → 透明 → 触发混合
// 设置白色背景 → 消除混合

// 3. UIImageView 使用不透明的图片
// 避免加载带 alpha 通道的 PNG（如果不需要透明）
// 用 JPEG（无 alpha）代替不需要透明的 PNG

// 4. drawRect 中指定不透明
UIGraphicsBeginImageContextWithOptions(size, YES, 0);
// 第二个参数 YES = opaque，生成不透明的位图
```

### 3.4 Auto Layout 性能优化

```objc
// Auto Layout 的计算复杂度：
// 简单约束 → O(n)
// 复杂/冲突约束 → 可能 O(n²) 或更差
// Cell 中大量约束时影响滚动流畅度

// 优化方法：

// 1. 减少约束数量
// 用 UIStackView 替代手动约束（内部更高效）

// 2. 避免约束冲突
// 设置合理的优先级，避免系统花时间解决冲突

// 3. 避免频繁更新约束
// ❌ 每帧都 removeConstraints + addConstraints
// ✅ 修改现有约束的 constant 值

// 4. 性能极致场景用手动布局
// 在 layoutSubviews 中直接设置 frame
- (void)layoutSubviews {
    [super layoutSubviews];
    CGFloat width = self.bounds.size.width;
    self.avatarView.frame = CGRectMake(16, 12, 44, 44);
    self.titleLabel.frame = CGRectMake(72, 12, width - 88, 20);
    self.subtitleLabel.frame = CGRectMake(72, 36, width - 88, 16);
}
// 手动布局比 Auto Layout 快 10 倍以上
// 但牺牲了可维护性，只在极端场景使用

// 5. iOS 12+ Auto Layout 性能大幅提升
// 重写了引擎，性能从 O(n²) 降到 O(n)
// 如果最低支持 iOS 12+，大部分场景不需要手动布局
```

### 3.5 TableView / CollectionView 优化

```objc
// ========== 1. Cell 复用 ==========
// 必须使用 dequeueReusableCell，不要每次创建新 Cell
- (UITableViewCell *)tableView:(UITableView *)tableView
         cellForRowAtIndexPath:(NSIndexPath *)indexPath {
    MyCell *cell = [tableView dequeueReusableCellWithIdentifier:@"MyCell"
                                                   forIndexPath:indexPath];
    [cell configureWithModel:self.dataSource[indexPath.row]];
    return cell;
}


// ========== 2. 高度缓存 ==========
// 避免在 heightForRow 中重复计算
@property (nonatomic, strong) NSMutableDictionary<NSIndexPath *, NSNumber *> *heightCache;

- (CGFloat)tableView:(UITableView *)tableView
    heightForRowAtIndexPath:(NSIndexPath *)indexPath {
    NSNumber *cached = self.heightCache[indexPath];
    if (cached) return cached.floatValue;

    CGFloat height = [MyCell heightForModel:self.dataSource[indexPath.row]
                                      width:tableView.bounds.size.width];
    self.heightCache[indexPath] = @(height);
    return height;
}

// 或使用 estimatedRowHeight 让系统延迟计算
tableView.estimatedRowHeight = 80;
tableView.rowHeight = UITableViewAutomaticDimension;


// ========== 3. 预加载 ==========
// iOS 10+ UITableViewDataSourcePrefetching
- (void)tableView:(UITableView *)tableView
    prefetchRowsAtIndexPaths:(NSArray<NSIndexPath *> *)indexPaths {
    for (NSIndexPath *indexPath in indexPaths) {
        [self.imageLoader prefetchImageForModel:self.dataSource[indexPath.row]];
    }
}

- (void)tableView:(UITableView *)tableView
    cancelPrefetchingForRowsAtIndexPaths:(NSArray<NSIndexPath *> *)indexPaths {
    for (NSIndexPath *indexPath in indexPaths) {
        [self.imageLoader cancelPrefetchForModel:self.dataSource[indexPath.row]];
    }
}


// ========== 4. 减少 Cell 内的图层数量 ==========
// 每个子视图 = 一个 CALayer → 更多的合成工作
// 复杂 Cell 考虑用异步绘制把内容画到一张位图上


// ========== 5. 避免在 cellForRow 中做耗时操作 ==========

## 四、网络优化

### 4.1 DNS 优化

```objc
// DNS 解析可能耗时 100ms-2s（弱网更久）
// 且 DNS 劫持是安全风险

// ========== 1. HTTPDNS ==========
// 通过 HTTP 直接向 DNS 服务器查询，绕过运营商 Local DNS
// 优势：避免劫持、更快解析、支持负载均衡

// 接入方式（美团内部有 HTTPDNS SDK）：
// 1. App 启动时预解析常用域名
// 2. 网络请求时用 IP 直连替代域名
// 3. 需要在 HTTP Header 设置 Host 字段

NSMutableURLRequest *request = [NSMutableURLRequest requestWithURL:
    [NSURL URLWithString:@"http://1.2.3.4/api/data"]];
[request setValue:@"api.example.com" forHTTPHeaderField:@"Host"];


// ========== 2. DNS 预解析 ==========
// 启动时提前解析即将使用的域名
// URLSession 内部有 DNS 缓存，预解析后后续请求更快
- (void)prewarmDNS {
    NSArray *hosts = @[@"api.example.com", @"cdn.example.com", @"img.example.com"];
    for (NSString *host in hosts) {
        NSURL *url = [NSURL URLWithString:[NSString stringWithFormat:@"https://%@", host]];
        NSURLSessionDataTask *task = [NSURLSession.sharedSession
            dataTaskWithURL:url completionHandler:^(NSData *data, NSURLResponse *resp, NSError *err) {
            // 不关心结果，只是为了触发 DNS 解析
        }];
        [task resume];
        [task cancel];  // 立即取消，DNS 缓存已建立
    }
}
```

### 4.2 连接优化

```
HTTP/2 的优势：
  多路复用：一个 TCP 连接并行多个请求（HTTP/1.1 每个域名最多 6 个连接）
  头部压缩：HPACK 算法减少请求头大小
  服务器推送：Server Push
  → iOS 9+ URLSession 自动支持 HTTP/2

HTTP/3（QUIC）：
  基于 UDP 的 QUIC 协议
  0-RTT 建连（比 TCP + TLS 快）
  连接迁移（WiFi 切 4G 不断连）
  → iOS 15+ URLSession 开始支持

连接复用：
  URLSession 默认会复用 TCP 连接（Keep-Alive）
  确保同一个 domain 的请求用同一个 URLSession
  ❌ 每次请求都创建新的 URLSession
  ✅ 用单例或少量 URLSession 实例
```

### 4.3 数据优化

```objc
// ========== 1. 数据压缩 ==========
// 请求头设置 Accept-Encoding: gzip
// URLSession 默认支持 gzip 解压
// 服务端响应 Content-Encoding: gzip

// Protobuf vs JSON：
// Protobuf 序列化后体积约为 JSON 的 1/3 - 1/10
// 解析速度约为 JSON 的 3-10 倍
// 适合高频接口和大数据量场景


// ========== 2. 增量更新 ==========
// 不要每次请求完整数据
// 客户端上报 lastModified / version
// 服务端只返回变化的部分（diff）

// HTTP 304 Not Modified：
// 客户端发送 If-None-Match / If-Modified-Since
// 服务端如果没变化 → 返回 304（无 body）→ 客户端用缓存


// ========== 3. 请求合并 ==========
// 多个并行请求合并为一个批量接口
// ❌ 首页同时发 5 个请求：用户信息、推荐列表、banner、活动、消息
// ✅ 一个聚合接口返回所有首页数据


// ========== 4. 缓存策略 ==========
// 不频繁变化的数据（配置、静态资源）做本地缓存
// 分层缓存：内存缓存（NSCache）→ 磁盘缓存 → 网络

@interface DataManager : NSObject
@property (nonatomic, strong) NSCache *memoryCache;
@end

- (void)fetchDataForKey:(NSString *)key completion:(void(^)(id data))completion {
    // 1. 内存缓存
    id cached = [self.memoryCache objectForKey:key];
    if (cached) { completion(cached); return; }

    // 2. 磁盘缓存
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        id diskData = [self readDiskCacheForKey:key];
        if (diskData) {
            [self.memoryCache setObject:diskData forKey:key];
            dispatch_async(dispatch_get_main_queue(), ^{ completion(diskData); });
            return;
        }

        // 3. 网络请求
        [self requestFromNetwork:key completion:^(id networkData) {
            [self.memoryCache setObject:networkData forKey:key];
            [self writeDiskCache:networkData forKey:key];
            dispatch_async(dispatch_get_main_queue(), ^{ completion(networkData); });
        }];
    });
}
```

### 4.4 弱网优化

```
弱网环境的特点：
  高延迟（RTT > 500ms）
  高丢包率（> 5%）
  带宽低（< 100kbps）
  连接不稳定（频繁断连）

优化策略：

1. 超时策略分级
   WiFi：连接 10s / 读取 30s
   4G：连接 15s / 读取 45s
   弱网：连接 20s / 读取 60s

2. 重试策略
   指数退避：1s → 2s → 4s → 8s → 放弃
   幂等接口才重试（GET），非幂等接口（POST）谨慎重试

3. 请求优先级
   弱网时只发关键请求（首页数据、用户操作）
   非关键请求延迟或取消（日志上报、预加载）

4. 离线模式
   核心功能支持离线使用
   用户操作先存本地 → 网络恢复后同步

5. 图片质量适配
   弱网时降低图片质量（服务端参数控制尺寸和压缩率）
   使用渐进式 JPEG（先加载模糊图，逐步清晰）
```

// ========== 1. 批量上报 ==========
// ❌ 每条日志立即上报 → 频繁唤醒网络
// ✅ 攒一批后批量上报

@interface LogManager : NSObject
@property (nonatomic, strong) NSMutableArray *buffer;
@property (nonatomic, strong) NSTimer *flushTimer;
@end

    if (self.buffer.count >= 50) {
        [self flush];  // 达到阈值立即上报
    }
}

        target:self selector:@selector(flush) userInfo:nil repeats:YES];
    // 每 60 秒上报一次（即使不到 50 条）

        payload.memoryMetrics.peakMemoryUsage;

        // 磁盘 I/O
        payload.diskIOMetrics.cumulativeLogicalWrites;

        // 挂起时间（Hang = 主线程卡顿 > 250ms）
        payload.applicationResponsivenessMetrics
              .histogrammedApplicationHangTime;

        // 上报到自己的后端
        [self reportMetrics:payload];
    }
}

// iOS 14+：还能收到诊断数据（崩溃、卡顿堆栈）
- (void)didReceiveDiagnosticPayloads:(NSArray<MXDiagnosticPayload *> *)payloads {
    for (MXDiagnosticPayload *payload in payloads) {
        // 崩溃报告
        NSArray<MXCrashDiagnostic *> *crashes = payload.crashDiagnostics;
        // 卡顿堆栈
        NSArray<MXHangDiagnostic *> *hangs = payload.hangDiagnostics;
        // CPU 异常
        NSArray<MXCPUExceptionDiagnostic *> *cpuExceptions = payload.cpuExceptionDiagnostics;
    }
}
```

### 9.2 自建监控体系

```
一个完整的性能监控体系应该包含：

1. 崩溃监控
   Mach 异常捕获 / UNIX Signal 捕获

## 十、Swift 性能优化

### 10.1 值类型 vs 引用类型

```swift
// 值类型（struct/enum）在栈上分配 → 更快，无引用计数开销
// 引用类型（class）在堆上分配 → 需要 malloc/free + 引用计数

// ✅ 优先使用 struct
struct Point {
    var x: Double
    var y: Double
}

// class 的额外开销：
// 1. 堆内存分配（malloc → 全局锁竞争）
// 2. 引用计数维护（每次赋值/释放都要原子操作）
// 3. 间接寻址（通过指针访问，缓存不友好）

// ⚠️ struct 内含引用类型时也会有引用计数开销
struct User {
    var name: String    // String 内部是引用类型
    var avatar: UIImage // 引用类型
}
// 拷贝 User 时，name 和 avatar 都要增加引用计数
// 如果 struct 包含多个引用类型字段，拷贝开销可能比 class 更大
```

### 10.2 协议与泛型性能

```swift
// ========== Protocol（存在类型）==========
// 使用 any Protocol 时，编译器通过 Existential Container 做间接调用
// 每次方法调用通过 Witness Table 查找 → 额外开销

func process(items: [any Drawable]) {
    for item in items {
        item.draw()  // 间接调用（Protocol Witness Table）
    }
}


// ========== 泛型（some Protocol / 具体类型）==========
// 编译时确定具体类型 → 静态派发 → 可内联

func process<T: Drawable>(items: [T]) {
    for item in items {
        item.draw()  // 静态派发，可被编译器内联优化
    }
}

// ✅ 能用泛型就用泛型，性能比 any Protocol 好
// ✅ 用 some Protocol（不透明类型）让编译器推断具体类型
// any Protocol 适合异构集合（不同类型混在一起）


// ========== final / private ==========
// 标记 final 或 private → 编译器知道不会被子类重写 → 静态派发

final class MyView: UIView {
    // 所有方法都是静态派发
}

class MyClass {
    private func helper() { }  // private → 编译器自动优化为静态派发
}
```

### 10.3 Copy-on-Write

```swift
// Swift 标准库的集合类型（Array/Dictionary/Set/String）
// 使用 Copy-on-Write（写时复制）优化

var array1 = [1, 2, 3, 4, 5]
var array2 = array1           // 只拷贝指针，不拷贝数据（O(1)）
array2.append(6)              // 写入时才真正拷贝数据（O(n)）

// 自定义 struct 实现 CoW：
struct MyData {
    private var storage: MyStorage  // 引用类型

    var value: Int {
        get { storage.value }
        set {
            // 如果有多个引用 → 先复制再修改
            if !isKnownUniquelyReferenced(&storage) {
                storage = storage.copy()
            }
            storage.value = newValue
        }
    }
}

// ⚠️ 性能陷阱：
// 在循环中频繁"写"会触发大量拷贝
// 避免不必要的值类型拷贝（通过 inout 或引用传递）
```

### Q8: Thread Sanitizer / Data Race Sanitizer 使用方法？

```
1. 开启方式：
   Scheme -> Run -> Diagnostics -> Thread Sanitizer

2. 检测原理：
   TSan 基于 compiler instrumentation，在每次读写共享内存时插入检查代码
   通过向量时钟算法检测潜在的数据竞争

3. 典型问题示例：

   // ❌ 数据竞争：两个线程同时访问非原子变量
   class Counter {
       var count = 0
       func increment() { count += 1 }  // 非原子操作
   }

   // ✅ 修复方案 1：使用锁
   private let lock = NSLock()
   func increment() { lock.lock(); count += 1; lock.unlock() }

   // ✅ 修复方案 2：原子操作
   @Atomic var count: Int = 0

   // ✅ 修复方案 3：actor（Swift 5.5+）
   actor Counter { var count = 0; func increment() { count += 1 } }

4. TSan 报告解读：
   - Thread N read while Thread M wrote → 竞态条件
   - 包含完整 call stack，直接定位问题行

5. 性能开销：~2-15x 降速，仅用于 debug 构建

⚠️ 踩坑点：UITextView 等部分 Apple 框架存在已知误报，可配置 suppression list 过滤。
💡 最佳实践：CI 中定期跑 UI 测试 + TSan，而非仅在本地手动触发。


### Q9: 线程分析器采样技术（Instruments + 采样分析）

```
1. 采样原理：
   Instruments Time Profiler 默认每 1ms 采集一次所有线程的调用栈
   采集足够长时间后，统计每个函数出现的采样次数
   采样占比越高 = 该函数消耗 CPU 时间越多

2. 核心操作：
   - Record 后在 App 中复现卡顿场景
   - 展开 Call Tree → Invert Call Tree（从叶子到根）
   - 勾选 "Hide library functions" 过滤系统库
   - 右键函数 → Mark Interesting Event（标记关键时间点）
   - 用 Red/Blue Range 对比优化前后的采样差异

3. 实战分析案例：
   // 主线程采样中发现大量 Sample 集中在：
   // └── CADisplayLink 回调 → heavyLayout() → 52.3% 采样
   // 结论：每帧 layout 太重，需拆分为增量计算

4. 采样 vs 追踪对比：
   - 采样（Sampling）：低开销 ~1-2%，适合长时间 profiling
   - 追踪（DTrace/Instruments Trace）：精确但开销高 ~10-50%

5. 进阶技巧：
   - 用 "Record System Trace" 捕获全局线程活动
   - 结合 Energy Log 查看功耗热点
   - 导出 .traceglad 文件归档分析记录
```

⚠️ 踩坑点：采样有统计偏差，短于 5 秒的 burst 操作容易被漏掉，应使用 DTrace 精确追踪代替。
💡 最佳实践：每次优化后对比采样报告，用 Mark Interesting Event 记录操作时序，形成可量化的优化基线。


### Q10: Instruments Time Profiler 深度使用指南

1. **核心原理**：
   Time Profiler 基于 DTrace USDT probes，每 1ms 对所有活跃线程采样一次调用栈。
   采样结果以 Call Tree 展示，Self % 表示函数自身耗时占比（不含子调用）。

2. **深度分析技巧**：
   - Invert Call Tree + Hide library functions：快速定位业务代码热点
   - Mark Interesting Event：在时间轴上标记用户操作，精确关联代码与交互
   - Record System Trace：捕获主线程卡顿、后台线程饥饿等系统级问题
   - Thread States：分析线程等待、阻塞、竞争状态分布

3. **实战示例**：
   ```swift
   // 问题：主线程每帧 16ms 中，12ms 花在此函数
   func renderFeedCell(_ model: FeedModel) {
       let image = decodeImage(model.data)    // 6ms - 应移到后台
       layoutSubviews()                        // 4ms - Auto Layout 过深
       applyCustomEffects()                    // 2ms - 可缓存
   }
   // 修复：后台解码 + 简化约束 + 结果缓存
   ```

4. **数据导出与对比**：
   - File → Save As 导出 .trace 文件
   - 多次录制后对比 Self % 变化，量化优化效果

⚠️ 踩坑点：采样间隔固定 1ms，极短函数（<1ms）可能被遗漏；Hotspots 工具更适合追踪此类 burst 操作。
💡 最佳实践：建立性能回归基线 —— 每次大版本发布前录制标准操作流的 trace，与历史基线对比。

### Instruments Allocations 分析策略

1. **核心作用**：
   Allocations 追踪每个内存分配/释放事件，精确定位内存增长来源。
   配合 Objects 视图可按类名聚合，快速找出未释放的对象。

2. **关键操作**：
   - Record 后执行测试流程，查看 Permanent Bytes（永久分配=潜在泄漏）
   - Filter 按 Stack Trace 分组，定位具体分配代码路径
   - 右键记录 → "Mark as Root Object" 标记持久化对象
   - All / Permanent / Released 三个视图切换分析生命周期

3. **实战示例**：
   ```swift
   // Allocations 发现大量 UIImage 未释放
   // Stack Trace: UIImage.resize() → calloc() → +12MB Permanent
   // 修复：使用 CGImageSource 零拷贝路径
   func optimizeImage(_ data: Data) -> UIImage? {
       let source = CGImageSourceCreateWithData(data as CFData, nil)
       guard let scaled = CGImageSourceCreateThumbnailAtIndex(source, 0, options) else { return nil }
       return UIImage(cgImage: scaled)  // 避免中间拷贝
   }
   ```

4. **与 Leaks 工具配合**：
   - Leaks 找 who（哪些对象泄漏）
   - Allocations 找 where（从哪里分配的）
   - VM Tracker 找 what（大内存映射类型）

⚠️ 踩坑点：Permanent Bytes 包含 App 正常缓存，需对比操作前后差值而非绝对值判断泄漏。
💡 最佳实践：每次新 feature 前后各录一次 Allocations，用 Permanent Bytes 增量验证无泄漏。

### Instruments Leaks 分析策略

1. **核心原理**：
   Leaks 基于 LeakDetector + Static Block Tracer，每 250ms（默认）扫描堆内存，查找不可达对象。
   配合 malloc stack logging 记录每个分配的调用栈，泄漏时回溯到分配点。

2. **关键操作**：
   - 勾选 Record Alloc stack / Record system alloc stack（需在真机构建前开启）
   - 测试后查看 "Potential Leaks" 和 "Definitely Leaked" 两类报告
   - 双击泄漏条目查看分配栈和对象类型，右键 "Show Nearest Symbol" 定位代码

3. **实战示例**：
   ```swift
   // Leaks 报告：0x7fce... <Block_byref_1> definitely leaked
   // 栈回溯: setupClosure() → GrandCentralQueue.async { [self] in ... }
   // 修复：捕获弱引用，避免循环引用导致 block 不释放
   func setupClosure() {
       Task.detached { [weak self] in
           guard let self = self else { return }
           await self.fetchData()  // weak self 阻断循环
       }
   }
   ```

4. **真机开启 malloc stack logging**：
   模拟器默认开启，真机需在 Scheme → Run → Diagnostics 勾选 Address Sanitizer 或手动设置环境变量。

⚠️ 踩坑点：Leaks 扫描间隔固定 250ms，快速分配+释放的对象可能被漏报；Definitely Leaked 才需修复，Potential Leaks 可能是误报。
💡 最佳实践：每次发版前在真机上跑标准测试流程，对比 Leaks 报告，确保无 Definitely Leaked 对象。

### Instruments Energy Log 分析

1. **核心原理**：
   Energy Log 基于 powerlogd 系统级采集，每 ~33ms 记录一次设备状态快照。
   指标包括：CPU Active Energy（活跃功耗）、Idle Energy（空闲功耗）、Continuous Energy（持续功耗）、Network、Location、Sensors、Disk I/O。

2. **关键操作**：
   - 打开 Instruments → Energy Log，记录用户操作流程
   - 查看 Energy Impact 曲线，对比不同操作段的功耗差异
   - 点击 spike 定位对应时刻的 Activity 和线程
   - 按 Category 分组：CPU / Network / Location / Sensors / Audio

3. **实战示例**：
   ```swift
   // Energy Log 发现：每秒轮询位置 → Continuous Energy 暴增
   // 修复：改用区域监控 + 精准定位按需开启
   func setupLocationEfficiently() {
       locationManager.allowsBackgroundLocationUpdates = false
       // 仅在关键页面使用 precise，日常用 reduced
       locationManager.desiredAccuracy = kCLLocationAccuracyHundredMeters
       // 区域监控替代轮询
       let region = CLCircularRegion(center: coord, radius: 500, identifier: "store")
       locationManager.startMonitoring(for: region)
   }
   ```

4. **与 Other Metrics 工具配合**：
   - Energy Log 看功耗趋势，Other Metrics 看 FPS / GPU / Disk 同步分析
   - 定位"卡顿+高功耗"双重问题根因：如主线程 heavy drawing 导致掉帧+CPU 飙升

⚠️ 踩坑点：Energy Log 仅支持真机，模拟器无 powerlogd 数据；Absolute Energy 值无跨设备可比性，只作同设备前后对比。
💡 最佳实践：性能优化前先录 baseline，每次优化后重录对比，用 Energy Impact delta 量化改进效果。

### Instruments 网络/文件系统活动追踪

1. **核心原理**：
   Network 仪表基于 darwinRecorder 捕获底层网络事件（通过 dtrace + kevent），记录每个 socket 的 connect/read/write/close 及对应的 NSURLConnection/NSURLSession 状态。
   Filesystem Activity 仪表通过 ktrace/fsevents 追踪 open/close/read/write/lseek 等系统调用，精确到文件路径和线程。

2. **关键操作**：
   - Network 仪表：查看 Timeline 面板，搜索 IPv4/IPv6 连接，筛选 host 名定位问题请求
   - Filesystem Activity：按 Process/Thread/Operation 过滤，展开 Call Stack 追溯代码位置
   - 配合 Time Profiler 使用：网络耗时 vs CPU 耗时，找出 I/O 阻塞导致的主线程卡顿

3. **实战示例**：
   ```swift
   // Network 仪表发现：某 API 响应 2.3s，其中 TCP 握手 1.8s
   // 原因：每次请求新建 connection，无 keep-alive
   // 修复：统一使用 shared session，复用连接
   let sharedSession = URLSession(configuration: .default)
   // Filesystem 仪表发现：每次 viewWillAppear 写 200KB plist
   // 修复：延迟写入 + 合并更新，减少 fsync 次数
   ```

4. **与 Other Instruments 联动**：
   - Network + Time Profiler：定位"网络请求阻塞主线程"（同步请求 → 异步化）
   - Filesystem + Allocations：大文件读写触发频繁 malloc/free，考虑 mmap 替代

⚠️ 踩坑点：Network 仪表在模拟器中无法捕获真实 WiFi/蜂窝数据；Filesystem Activity 采样间隔默认 10ms，高频小写可能被漏记。
💡 最佳实践：用 Network 仪表的 "Search" 按域名过滤，快速定位慢接口；Filesystem 仪表按 Thread 分组，优先优化主线程的文件操作。


### Instruments VM Trace 用于大内存映射

1. **核心原理**：
   VM Trace 基于 dtrace 的 machvm provider 拦截所有虚拟内存系统调用（vm_allocate/vm_deallocate/mmap/mprotect），记录每个虚拟地址区域的分配、释放、保护变化和 Backing Store（物理/文件/swap）。与 Allocations 不同，VM Trace 追踪的是**虚拟地址空间级别的映射**，包括 memory-mapped file、shared memory、paged I/O、大页映射（Jumbo Pages），能暴露 Allocations 无法捕获的内存泄漏和碎片化问题。

2. **关键操作**：
   - 打开 Instruments → VM Trace，关注 Virtual Size（VSS）与 Resident Size（RSS）的差异，VSS 远大于 RSS 说明大量映射未被物化
   - 按 Call Stack 追溯大映射来源，识别 mmap 文件路径或 Jumbo Page 分配点
   - 查看 Purgeable 列，非 purgeable 的大映射在内存压力下无法被系统回收
   - 使用 "Protect" 操作追踪 mprotect(MPROT_READWRITE) → mprotect(MPROT_READONLY) 模式，定位代码/数据段优化机会

3. **实战示例**：
   ```swift
   // VM Trace 发现：Core Data SQLite WAL 文件 mmap 占用 2GB VSS
   // 根因：大事务写入导致 WAL 膨胀，未触发 checkpoint
   // 修复：调整 WAL 自动 checkpoint 阈值
   let context = persistentContainer.newNSManagedObjectContext()
   persistentStoreDescription.setOption(
       5000 as NSNumber,
       forKey: NSPersistentStoreWLAutoAllocateThresholdKey
   )
   // 大图场景：解码后 CGImage 通过 mmap 映射原文件，VSS 暴涨但 RSS 低
   // 修复：配置 purgeable + downscale，避免 OOM
   let options: [String: Any] = [
       kCGImageSourceShouldCache as String: false,
       kCGImageSourceCreateThumbnailFromImageAlways as String: true,
       kCGImageSourceThumbnailMaxPixelSize as String: 414 * UIScreen.main.scale
   ]
   let image = CGImageSourceCreateThumbnailAtIndex(source, 0, options as CFDictionary)
   ```

4. **联动分析**：
   - VM Trace + Allocations：对比 VSS vs 堆分配，定位"虚拟膨胀但堆正常"的 mmap 泄漏
   - VM Trace + Activity Monitor：监控 PSS 和 page fault 频率，高频 page fault 表示内存碎片化
   - VM Trace + Time Profiler：page fault 耗时热点 → 优化内存访问模式或预加载关键数据

⚠️ 踩坑点：VSS 不等于实际内存占用，稀疏映射导致 VSS 虚高但 RSS 极低；VM Trace 在模拟器上数据不完整，必须真机测试；iOS 16+ 支持 Jumbo Pages（64KB）减少页表碎片。
💡 最佳实践：大资源优先使用 purgeable mmap + 按需 downscale；批量操作后主动 purge 或释放引用；Core Data 大事务设置合理 WAL checkpoint 阈值避免虚拟内存爆炸。


### Core Data 性能调优（faulting、批量加载、NSBatchUpdateRequest）

1. **核心原理**：
   Core Data 的 faulting 机制是延迟加载 —— 获取的 NSManagedObject 仅持有 fault（占位符），首次访问属性时才触发 SQL 查询获取数据。合理利用 faulting 可避免批量 fetch 时一次性加载全部属性数据。批量操作（NSBatchInsertRequest/NSBatchUpdateRequest/NSBatchDeleteRequest）绕过 managed object context，直接在 SQLite 层执行，不触发 fault、KVO 或 change tracking，性能比常规 CRUD 高 10-100 倍。

2. **关键操作**：
   - 批量更新：使用 NSBatchUpdateRequest 直接修改数据库，避免逐个 fetch + edit + save
   - 控制 faulting：用 willAccessValue(forKey:) / didAccessValueForKey() 延迟属性初始化，避免不必要的 SQL 查询
   - 批量 fetch：NSBatchDeleteRequest 配合 NSBatchMutationErrorPolicy 处理级联删除
   - 预故障排除：NSFaultingReadOnlyRequest 在只读场景下获取已解 fault 的对象，避免后续访问触发隐式 SQL

3. **实战示例**：
   ```swift
   // 批量将已读消息标记为 read = true，绕过 MOC
   let request = NSBatchUpdateRequest(entityName: "Message", 
       predicate: NSPredicate(format: "chatID == %@", chatID))
   request.propertiesToUpdate = ["isRead": true]
   request.resultType = .updatedObjectsIDsResultType
   do {
       let result = try tryBatch(request, with: viewContext) as NSBatchUpdateResult
       let updatedIDs = result.result as! [NSManagedObjectID]
       viewContext.refreshAllObjects() // 同步主 context
   } catch { /* handle */ }
   ```

4. **注意事项**：
   - BatchRequest 不触发 MOC 变更通知，需手动 refreshAllObjects() 或 processPendingChanges()
   - 跨 context 操作：parent context 中执行 batch 后调用 save()，viewContext 调用 processPendingChanges()

⚠️ 踩坑点：在后台 context 执行 batch update 后，主线程 context 仍持有旧数据对象，导致 UI 显示不一致；fetchRequest.propertiesToFetch 滥用导致全部属性被物化，使 faulting 失效。
💡 最佳实践：列表页只 fetch ID + 展示属性，详情页再 lazy-load 完整对象；批量操作统一在 private queue context 中执行，通过 processPendingChanges() 同步到 view context。

### 后台线程图片解码（CGImageSource）

1. **核心原理**：
   在主线程解码大图会导致明显的帧率下降（1-5ms/张），尤其是在滚动加载图片的列表中。`CGImageSourceCreateThumbnailAtIndex` 支持指定最大像素尺寸，在解码的同时完成 downscale，避免先解码原始尺寸再缩放的浪费。将解码移到后台队列可完全消除主线程阻塞。

2. **关键操作**：
   - 使用 `CGImageSourceCreateThumbnailAtIndex` 直接解码为缩略图，跳过全分辨率解码
   - `kCGImageSourceShouldCache: false` 禁止内部缓存，配合 purgeable 减少内存峰值
   - 在 `DispatchQueue.global(qos: .userInitiated)` 执行解码，完成后将结果返回主线程渲染
   - 配合 `kCGImageSourceCreateThumbnailFromImageAlways: true` 强制缩略，避免全图解码

3. **实战示例**：
   ```swift
   func decodeImage(onDisk url: URL, maxPixel: CGFloat) async -> CGImage? {
       guard let source = CGImageSourceCreateWithURL(url as CFURL, nil) else { return nil }
       let options: [CFString: Any] = [
           kCGImageSourceShouldCache: false,
           kCGImageSourceCreateThumbnailFromImageAlways: true,
           kCGImageSourceThumbnailMaxPixelSize: maxPixel * UIScreen.main.scale
       ]
       return await DispatchQueue.global(qos: .userInitiated).async {
           CGImageSourceCreateThumbnailAtIndex(source, 0, options as CFDictionary)
       }
   }
   ```

4. **注意事项**：
   - 解码后 CGImage 是无状态的 C 结构体，线程安全，可直接跨队列传递给主线程
   - 避免在后台线程创建 UIImage（涉及 CoreGraphics 状态），先解码 CGImage 再在主线程封装

⚠️ 踩坑点：未设置 `kCGImageSourceShouldCache: false` 导致原始大图被隐式缓存，内存峰值飙升；后台解码后忘记将结果 dispatch 回主线程更新 UI。
💡 最佳实践：图片加载链 = 后台 CGImageSource 解码 downscale → 主线程封装 UIImage → UIImageView 显示；列表页 maxPixel 控制在展示尺寸的 1-1.2 倍。


### UIImage 缓存策略（NSCache vs 自定义 LRU）

1. **核心原理**：
   NSCache 是 Apple 提供的线程安全缓存，自动响应内存警告和系统压力进行释放，适合图片、字体等资源缓存。自定义 LRU 缓存可精确控制容量上限和淘汰策略，适合需要严格内存预算的场景。两者的选择取决于是否需要系统自动回收 vs 精确的容量控制。

2. **关键操作**：
   - NSCache 使用 setObject(_:forKey:) 和 countLimit/totalCostLimit 控制上限，系统低内存时自动清理
   - 自定义 LRU 基于有序字典实现，每次访问将 key 移到最后，容量超限时淘汰最早 entry
   - 图片缓存应同时实现内存 + 磁盘双层缓存，内存用 NSCache/LRU，磁盘用 FileManager + URL 映射
   - 配合 eviction handler 监控 NSCache 释放行为，在调试阶段检测缓存命中率

3. **实战示例**：
   ```swift
   class ImageCache {
       let memoryCache = NSCache<NSString, UIImage>()
       private var diskCachePath: String

       init() {
           memoryCache.countLimit = 100
           memoryCache.totalCostLimit = 100 * 1024 * 1024 // 100MB
           diskCachePath = cachesDirectory.appendingPathComponent("images")
       }

       func get(_ url: URL) -> UIImage? {
           memoryCache.object(forKey: url.absoluteString as NSString)
       }

       func set(_ image: UIImage, forKey url: URL) {
           memoryCache.setObject(image, forKey: url.absoluteString as NSString, cost: image.imageBytes())
       }
   }
   ```

4. **注意事项**：
   - NSCache 的 costLimit 是 soft limit，系统压力下可能超量；LRU 可硬限制
   - 大图缓存建议按展示尺寸生成多个版本（缩略图、中等、原始），分别缓存

⚠️ 踩坑点：仅使用 NSCache 不设 costLimit 导致内存无上限增长；LRU 实现未处理并发访问导致崩溃；磁盘缓存未做定期清理使沙盒爆满。
💡 最佳实践：生产环境使用 NSCache + 磁盘双层缓存，内存层设 costLimit 作为软保护，磁盘层按访问频率定期清理，配合 cache hit rate 监控调优容量参数。

### UITableView/UICollectionView prefetchingDataSource

1. **核心原理**：
   prefetchingDataSource 是 iOS 10+ 引入的预加载机制，当 cell 即将进入可视区域前提前调用 prefetch 回调，利用滚动间隙异步准备数据（如图片解码、网络请求、CoreData 抓取），显著减少滚动时的卡顿。与 cellForRowAtIndexPath 不同，prefetch 不要求立即返回结果，可安全执行异步操作。

2. **关键操作**：
   - 实现 UICollectionViewPrefetchingDataSource 或 UITableViewPrefetchingDataSource 协议
   - 在 prefetchItemsAt 中启动异步任务（图片加载、数据解码）
   - 在 cancelPrefetchItemsAt 中取消未使用的请求，避免浪费资源
   - 配合 cancellation 机制确保只加载最终可见的内容

3. **实战示例**：
   ```swift
   extension ViewController: UICollectionViewPrefetchingDataSource {
       func collectionView(_ collectionView: UICollectionView,
                           prefetchItemsAt indexPaths: [IndexPath]) {
           for indexPath in indexPaths {
               let item = items[indexPath.row]
               imageLoader.load(url: item.imageUrl,
                                 identifier: "prefetch-\(item.id)",
                                 completion: { [weak self] image in
                 self?.cache.set(image, key: item.id)
               })
           }
       }

       func collectionView(_ collectionView: UICollectionView,
                           cancelPrefetchingForItemsAt indexPaths: [IndexPath]) {
           for indexPath in indexPaths {
               imageLoader.cancel(identifier: "prefetch-\(items[indexPath.row].id)")
           }
       }
   }
   ```

4. **注意事项**：
   - prefetch 调用时机不确定，用户快速滚动时可能 cancel 后立即重新 prefetch
   - 必须实现 cancel 方法，否则后台任务累积导致内存和 CPU 浪费
   - 每个 indexPath 可能触发多次 prefetch，使用 identifier 去重请求

⚠️ 踩坑点：在 prefetch 中同步阻塞等待（如 synchronous URLSession）完全抵消预加载意义；忘记实现 cancel 导致大量无用网络请求和内存泄漏。
💡 最佳实践：prefetch 只用于异步非阻塞操作，配合支持 cancel 的 ImageLoader（如 Kingfisher/Nuke），为每个请求生成唯一 identifier 实现精准取消和去重。

### 减少主线程 pre-main 时间（Time Profiler 分析）

1. **核心原理**：
   pre-main 时间指从 dyld 启动到 main() 函数执行之间的耗时，包含动态库加载、+load 方法调用、Objective-C 运行时初始化（类注册、category 绑定、method swizzling）三个阶段。这是影响 App 冷启动体验的最关键瓶颈，Apple 建议控制在 150ms 以内。

2. **Time Profiler 分析步骤**：
   - 使用 Instruments → Time Profiler，勾选 "Record Blur" 和 "Hide Noise"
   - 从 `dyld` 的 `_dyld_start` 标记到 `main` 函数之间的区域即为 pre-main 区间
   - 按 Call Tree 展开，识别耗时 top 的动态库和 +load 方法
   - 使用 "Invert Call Tree" + "Hide System Libraries" 过滤系统开销

3. **实战示例**：
   ```swift
   // 测量 pre-main 时间
   class func applicationDidFinishLaunching(_ application: UIApplication) {
       let launchTime = ProcessInfo.processInfo.systemUptime - launchTimestamp
       print("Pre-main time: \(launchTime * 1000)ms")
   }

   // 使用 dtrace 精确测量
   // dtrace -n 'pid$target:::function-entry{self->ts=timestamp}' \
   //         '-n 'pid$target:::function-return/self->ts/(timestamp-self->ts)/1000000,arg1)' \
   //         '-c /path/to/app'
   ```

4. **优化策略**：
   - 减少动态库数量：合并小 library，使用 -dead_strip 移除未引用符号
   - 移除或延迟非必要 +load 方法：+load 在主线程同步执行，不可跳过
   - 使用 Link Map 分析符号表大小，定位臃肿的第三方库
   - 启用 Thin LTO 减少初始化开销

⚠️ 踩坑点：在 +load 中执行网络请求、文件 IO 或复杂计算，这些操作在主线程阻塞执行且无法被 Instruments 直接追踪；多个 library 循环依赖导致 +load 顺序不确定引发 crash。
💡 最佳实践：定期用 Time Profiler 审计 pre-main 时间，将非必要初始化移至 main() 之后异步执行；使用 Split Info.plist 将 Info.plist 解析延迟到 main 后；优先优化 top 3 耗时动态库而非全面优化。

### 减少动态库数量与 +load 影响

1. **核心原理**：
   iOS App 冷启动时，dyld 会按链接顺序加载所有动态库（.framework/.dylib），并逐个调用每个库中声明的 `+load` 方法。`+load` 由 dyld 在主线程同步触发，发生在 `main()` 之前，且调用顺序仅依赖库的链接顺序而非依赖关系。每个 `+load` 方法至少增加 1~5ms 开销，库数量从 50 减至 30 可节省 30~80ms pre-main 时间。

2. **实战示例**：
   ```swift
   // ❌ 反例：在 +load 中做重初始化
   @objc extension HeavyLibrary {
       @objc class func load() {
           // 主线程阻塞！每个 +load 都在 main() 前执行
           setupNetworking()   // 禁止网络 IO
           parseConfigFile()   // 禁止文件 IO
           registerAllClasses() // 非必要延迟到 main 后
       }
   }

   // ✅ 正例：+load 仅做轻量 swizzling，其余延迟初始化
   @objc extension UITableView {
       @objc class func load() {
           DispatchQueue.main.async {
               performSwizzling()  // 延迟到 main 后执行
           }
       }
   }
   ```

3. **优化策略**：
   - 使用 `nm -u` 或 Link Map 分析哪些 .framework 只被间接引用，合并为静态库 (.a)
   - 用 `-ObjC` 替换为 `-force_load` 或 `-all_load` 精确控制符号加载
   - Xcode 14+ 启用 Whole Module Optimization + Thin LTO 减少库边界开销
   - 使用 `nm` 或 `size` 命令对比合并前后的符号数量和 binary 体积

⚠️ 踩坑点：Category 的 `+load` 在所属类 `+load` 之后执行，但多个 category 之间的顺序未定义，依赖执行顺序的 swizzling 可能导致随机 crash。
💡 最佳实践：每迭代审计一次 `nm -u` 输出和 Time Profiler pre-main 区间，将非必要初始化和 swizzling 统一延迟至 `application(_:didFinishLaunching)` 中异步执行；小 library 优先合并为静态库。

### 符号化与崩溃堆栈分析（dSYM、atos）

1. **核心原理**：
   App Store 发布的二进制经过编译链接后，函数名和行号信息被剥离到独立的 dSYM 文件中。崩溃日志中的堆栈地址（如 0x0000000102a3f1c4）需通过 dSYM 中的 DWARF 调试信息还原为可读的函数名和行号。`atos` 是 Apple 提供的命令行符号化工具，Xcode 的 DeviceLogs 和 Crashlytics 等第三方服务在后台也调用相同机制。

2. **实战示例**：
   ```bash
   # 方式一：atos 手动符号化（需对应架构的 dSYM）
   atos -o /path/to/App.dSYM/Contents/Resources/DWARF/App         -arch arm64 -l 0x0000000102a3e000 0x0000000102a3f1c4
   # 输出：-[[ViewController viewDidLoad] + 45] (in App + 0x11c4)

   # 方式二：symbolicatecrash（Xcode 内置，自动匹配 dSYM）
   export DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer
   /usr/bin/symbolicatecrash -v CrashLog.crash ~/dSYM/

   # 方式三：crash 工具（支持自动查找 dSYM 和 System dylib）
   crash -symbol_paths ~/dSYM/ -mach_kernel /tmp/Kernel_debug.zip         - zipfile CrashLog.iprc
   ```

3. **关键注意事项**：
   - dSYM UUID 必须与崩溃日志中记录的 UUID 一致，可通过 `dwarfdump --uuid App.dSYM` 查看
   - 每次 build 生成唯一 dSYM，Archive 时 Xcode 自动保存至 `~/Library/Developer/Xcode/Archives`
   - `-l` 参数（加载偏移地址）可从崩溃日志 image 行获取

⚠️ 踩坑点：Release 包启用了 Strip Debug Symbols 但 dSYM 未妥善保存，导致线上崩溃永远无法符号化；Bitcode 提交后 Apple 重新编译生成的 dSYM 与本地不同，需用 Archive 产物中的 dSYM。
💡 最佳实践：CI/CD 流程中自动提取 dSYM 并上传至符号服务器（如 Crashlytics 或自建），配合 `dwarfdump --uuid` 建立 UUID 到 build number 的映射表实现自动匹配。

### 预热技术 / App 启动隐私 API

1. **核心原理**：
   iOS 16+ 引入 `_swiftLazyFunctionPointerIndex` 和 `prewarm` 相关隐私 API，系统会在后台预读取 App 的二进制到页缓存。苹果提供的 `BKSemaphore` 和 `preheat` 接口允许 App 向系统注册启动热点函数，系统在预加载阶段提前 JIT 编译这些函数的代码页。同时 `task_info` 中的 `TASK_PREWARMED` flag 可判断当前是否从预热状态启动，据此跳过冗余初始化。

2. **实战示例**：
   ```swift
   import Foundation

   // 检测预热状态
   var tinfo = task_info()
   var count = MemoryLayout<task_info>.size / MemoryLayout<Int>.size
   withUnsafeMutablePointer(to: &tinfo) { ptr in
       ptr.withMemoryRebound(to: Int.self, capacity: count) {
           sysctl([CTL_KERN, KERN_TASK, getuid(), TASK_INFO],
                  4, $0, &count, nil, 0)
       }
   }
   let isPrewarmed = tinfo.ts_flags & TASK_PREWARMED != 0
   if isPrewarmed { print("从预热状态启动") }

   // 注册启动关键路径
   let hotspot = unsafeBitCast(UIApplication.shared.sendActions(for: .valueChanged),
                               to: UnsafeMutableRawPointer.self)
   // 通过内部 API 通知系统预热该函数指针指向的代码段
   ```

3. **优化策略**：
   - 在 `Info.plist` 中声明 `NSAppRestartStrategy` 提示系统优先预热
   - 将启动路径中的 `initialize()` 函数地址写入 `.plist` 的预加载列表
   - 使用 `os.signpost` 标记预热前后的初始化耗时，量化收益
   - 结合 `-dead_strip` 和 `-map` 减少非必要符号，缩小预热范围

⚠️ 踩坑点：预热线程使用的是后台低优先级队列，**不可**在预热回调中执行任何阻塞操作，否则会导致 App 被 watchdog 杀死。
💡 最佳实践：预热仅用于「读缓存」场景（如预读 Launch Screen 资源），真正的初始化仍放在 `didFinishLaunching` 主流程中，通过 `TASK_PREWARMED` 判断走快速路径。

### os_signpost 用于结构化性能追踪

1. **核心原理**：
   `os_signpost` 是 Apple 在 macOS 10.12 / iOS 10 引入的低开销性能追踪 API，基于 `libdispatch` 的 `os_log` 基础设施。通过 `os_signpost_id_t` 关联同一逻辑区域的不同事件（Begin/End/Event/Interval），Instrument 的 Time Profiler 和 os_signpost 面板可自动聚合区间、排除线程调度噪声，精度达纳秒级且不影响 Release 性能。

2. **实战示例**：
   ```swift
   import os

   // 定义 signpost 日志域
   private let log = OSLog(subsystem: "com.myapp", category: "startup")

   // 方式一：interval（Begin → End，单线程场景）
   func loadImage() {
       let signpostID = os_SIGNPOST_ID_MAKE_PAID(1, 0)
       os_signpost(.begin, log: log, name: "imageDecode", signpostID: signpostID)
       // ... 解码逻辑 ...
       os_signpost(.end, log: log, name: "imageDecode", signpostID: signpostID)
   }

   // 方式二：signpostEmitter（多线程跨线程追踪，推荐）
   let emitter = OSSignpostEmitter(type: OSSignpostEmitterType.payload)
   func startNetworkRequest() {
       let id = OSSignpostID(log: log, signposter: emitter)
       os_signpost(.begin, log: log, name: "networkFetch", signposter: emitter, signpostID: id)
   }
   func finishNetworkRequest(_ id: OSSignpostID) {
       os_signpost(.end, log: log, name: "networkFetch", signposter: emitter, signpostID: id)
   }
   ```

3. **Signpost ID 类型选择**：
   - `os_SIGNPOST_ID_MAKE_PAID`：Process + Array ID，适用于**单线程**场景
   - `OSSignpostID(log:signposter:)`：Process + Object + Array ID，**多线程跨线程**必须使用 Emitter
   - `OSSignpostEmitterType.payload`：支持跨进程追踪（XPC），Instrument 自动跨进程聚合

⚠️ 踩坑点：Interval 的 Begin/End **必须成对出现**且在同一 log + category 下，否则 os_signpost 面板显示为孤立事件；多线程场景未使用 Emitter 导致区间无法匹配。
💡 最佳实践：用 `.event` 记录瞬时状态（如内存警告），用 interval 包裹耗时操作；Release 下开销接近零，通过 `OS_LOG_SIGNPOST_ENABLED` 宏控制编译期裁剪。
