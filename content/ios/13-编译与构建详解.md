# 编译与构建详解

---

## 一、编译流程概述

### 1.1 从源代码到可执行文件

```
iOS 编译的完整流程：

  源代码（.m / .swift）
       │
       ▼
  ┌──────────────┐
  │  1. 预处理     │   （仅 OC）宏展开、头文件导入、条件编译
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │  2. 编译       │   词法分析 → 语法分析 → 语义分析 → 生成 IR
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │  3. 优化       │   LLVM 中间优化 Pass（-O0 / -Os / -O2）
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │  4. 代码生成   │   IR → 机器码（.o 目标文件）
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │  5. 链接       │   合并所有 .o + 库 → Mach-O 可执行文件
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │  6. 签名打包   │   代码签名 + 资源打包 → .app / .ipa
  └──────────────┘
```

### 1.2 Clang 与 LLVM 架构

```
Apple 的编译工具链基于 LLVM：

  前端（Frontend）              优化器（Optimizer）         后端（Backend）
  ┌─────────────┐              ┌──────────────┐           ┌──────────────┐
  │  Clang (OC)  │──┐          │              │           │              │
  └─────────────┘  │  LLVM IR  │  LLVM Pass   │  LLVM IR  │  ARM64 代码   │
  ┌─────────────┐  ├──────────▶│  优化管道     │──────────▶│  生成器       │
  │  Swift 编译器 │──┘          │              │           │              │
  └─────────────┘              └──────────────┘           └──────────────┘

关键概念：
  - 前端（Clang / Swift Compiler）：负责将源码翻译为 LLVM IR
  - LLVM IR：与语言无关的中间表示，所有语言共用同一套优化管道
  - 后端：将优化后的 IR 生成目标平台的机器码（ARM64）
  - 这种三段式设计使得添加新语言或新平台都很方便
```

## 二、Objective-C 编译流程

### 2.1 预处理阶段

```
预处理器处理所有 # 开头的指令：

  1. 头文件导入
     #import <UIKit/UIKit.h>
     → 将头文件内容复制插入到当前文件

  2. 宏展开
     #define MAX(a, b) ((a) > (b) ? (a) : (b))
     int result = MAX(x, y);
     → int result = ((x) > (y) ? (x) : (y));

  3. 条件编译
     #if DEBUG
         NSLog(@"Debug mode");
     #endif
     → Release 模式下整段代码被移除

查看预处理结果：
  clang -E ViewController.m -o ViewController.i
  // 展开后的文件通常有几万行（因为系统头文件很大）
```

### 2.2 词法分析与语法分析

```
词法分析（Lexer / Tokenizer）：
  将源代码拆分为一个个 Token（词法单元）

  int count = 10;
  → [keyword:int] [identifier:count] [punctuation:=] [literal:10] [punctuation:;]

语法分析（Parser）：
  将 Token 序列构建为抽象语法树（AST）

  int count = 10;
  → AST:
    VarDecl
    ├── type: int
    ├── name: count
    └── init: IntegerLiteral(10)

查看 AST：
  clang -Xclang -ast-dump -fsyntax-only ViewController.m

语义分析（Sema）：
  - 类型检查：赋值类型是否匹配
  - 方法调用检查：对象是否响应该 selector
  - 协议一致性检查
  - 产生警告和错误
```

### 2.3 生成 LLVM IR

```
Clang 将 AST 转换为 LLVM IR（中间表示）：

  Objective-C 源码：
  [self.view addSubview:label];

  → 实际上被转换为 objc_msgSend 调用的 IR

查看 IR：
  clang -S -emit-llvm ViewController.m -o ViewController.ll

IR 的三种形式：
  - .ll 文件：人类可读的文本格式
  - .bc 文件：二进制格式（Bitcode）
  - 内存中的数据结构：编译器内部使用
```

## 三、Swift 编译流程

### 3.1 Swift 编译器架构

```
Swift 编译比 OC 多了一层中间表示 SIL：

  Swift 源码
      │
      ▼
  ┌─────────────┐
  │  解析（Parse） │   词法分析 + 语法分析 → AST
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  语义分析      │   类型检查、类型推断、协议一致性检查
  │  (Sema)      │
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  SILGen      │   AST → Raw SIL（Swift 中间语言）
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  SIL 优化     │   确定性降级、ARC 优化、泛型特化、内联等
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  IRGen       │   Canonical SIL → LLVM IR
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │  LLVM        │   IR 优化 + 机器码生成 → .o 文件
  └─────────────┘
```

### 3.2 SIL（Swift Intermediate Language）

```
SIL 是 Swift 特有的中间表示，位于 AST 和 LLVM IR 之间：

为什么需要 SIL：
  - LLVM IR 太底层，无法表达 Swift 的高级特性（如 ARC、值语义、协议见证表）
  - SIL 保留了 Swift 的类型信息和所有权信息
  - 可以在 SIL 层面做 Swift 特有的优化

SIL 的两种形式：
  1. Raw SIL：SILGen 直接生成，未优化
  2. Canonical SIL：经过优化 Pass 处理后的标准化形式

查看 SIL：
  swiftc -emit-sil main.swift     // Canonical SIL
  swiftc -emit-silgen main.swift  // Raw SIL

SIL 层面的关键优化：
  - ARC 优化：合并多余的 retain/release
  - 泛型特化：为具体类型生成特化版本，消除泛型开销
  - 函数内联：将小函数体直接嵌入调用处
  - 确定性降级：将协议方法调用降级为直接调用（可能时）
  - 逃逸分析：判断闭包是否逃逸，优化内存分配
```

### 3.3 WMO（Whole Module Optimization）

```
WMO 是 Swift 的全模块优化模式：

  普通编译（Single File）：
    每个 .swift 文件独立编译 → 优化范围限于单个文件

  WMO 模式：
    整个 Module 的所有文件一起编译 → 优化器能看到所有代码

  WMO 带来的优化机会：
    1. 跨文件函数内联
       - File A 中的小函数可以被内联到 File B 的调用处
    2. 跨文件泛型特化
       - 泛型函数在其他文件被具体类型调用时可以特化
    3. 访问控制推断
       - internal 方法如果没有被其他文件调用，可以优化为直接调用
    4. 去虚拟化
       - 没有子类重写的方法可以从虚函数调用变为直接调用

  开启 WMO：
    Build Settings → Swift Compiler → Optimization Level
    - Debug：-Onone（不优化，不使用 WMO）
    - Release：-O -whole-module-optimization

  WMO 的代价：
    - 编译时间更长（不支持增量编译）
    - 内存占用更大
    - 通常只在 Release 模式使用
```

## 四、Mach-O 文件结构

### 4.1 Mach-O 概述

```
Mach-O（Mach Object）是 macOS / iOS 的可执行文件格式：

  可执行文件类型：
    - MH_EXECUTE：主程序可执行文件
    - MH_DYLIB：动态库（.dylib / .framework）
    - MH_BUNDLE：可加载的 Bundle（.bundle）
    - MH_OBJECT：目标文件（.o）
    - MH_DSYM：调试符号文件

  查看 Mach-O 类型：
    file MyApp
    // MyApp: Mach-O 64-bit executable arm64

  通用二进制（Universal Binary / Fat Binary）：
    一个文件包含多个架构的 Mach-O
    lipo -info MyApp
    // Architectures in the fat file: MyApp are: x86_64 arm64
```

### 4.2 Mach-O 内部结构

```
Mach-O 文件的三大组成部分：

  ┌─────────────────────────┐
  │       Mach-O Header      │   魔数、CPU 类型、文件类型、Load Commands 数量
  ├─────────────────────────┤
  │      Load Commands       │   描述文件布局的"目录"
  │  ┌───────────────────┐  │
  │  │  LC_SEGMENT_64     │  │   段定义（__TEXT, __DATA, __LINKEDIT）
  │  │  LC_DYLD_INFO_ONLY │  │   动态链接信息
  │  │  LC_SYMTAB         │  │   符号表
  │  │  LC_LOAD_DYLIB     │  │   依赖的动态库
  │  │  LC_MAIN           │  │   程序入口点
  │  │  LC_CODE_SIGNATURE  │  │   代码签名
  │  │  ...               │  │
  │  └───────────────────┘  │
  ├─────────────────────────┤
  │         Data             │   实际的段数据
  │  ┌───────────────────┐  │
  │  │  __TEXT Segment     │  │   代码段（只读、可执行）
  │  │    __text           │  │   ← 机器码
  │  │    __stubs          │  │   ← 符号桩（跳板）
  │  │    __stub_helper    │  │   ← 桩辅助代码
  │  │    __cstring        │  │   ← C 字符串
  │  │    __objc_methname  │  │   ← OC 方法名
  │  ├───────────────────┤  │
  │  │  __DATA Segment     │  │   数据段（可读写）
  │  │    __objc_classlist │  │   ← OC 类列表
  │  │    __objc_protolist │  │   ← OC 协议列表
  │  │    __la_symbol_ptr  │  │   ← 懒绑定符号指针
  │  │    __nl_symbol_ptr  │  │   ← 非懒绑定符号指针
  │  │    __got            │  │   ← 全局偏移表
  │  ├───────────────────┤  │
  │  │  __LINKEDIT Segment │  │   链接信息段
  │  │    符号表            │  │
  │  │    字符串表          │  │
  │  │    重定位信息        │  │
  │  └───────────────────┘  │
  └─────────────────────────┘

常用查看工具：
  otool -h MyApp          // 查看 Header
  otool -l MyApp          // 查看 Load Commands
  otool -t MyApp          // 查看 __text 段（反汇编）
  size -m MyApp           // 查看各段大小
  MachOView（GUI 工具）   // 可视化查看 Mach-O 结构
```

### 4.3 常见 Section 详解

```
__TEXT 段中的重要 Section：

  __text          编译后的机器码指令
  __stubs         外部符号的跳板代码（stub），调用动态库函数时先跳到这里
  __stub_helper   首次调用时帮助 dyld 绑定真实地址
  __cstring       C 语言字符串常量
  __objc_methname OC 方法名字符串
  __objc_classname OC 类名字符串
  __objc_methtype  OC 方法类型编码

__DATA 段中的重要 Section：

  __objc_classlist   所有 OC 类的指针列表
  __objc_catlist     所有 Category 的指针列表
  __objc_protolist   所有协议的指针列表
  __objc_classrefs   被引用的类列表（可用于分析未使用的类）
  __objc_selrefs     被引用的 selector 列表
  __la_symbol_ptr    懒绑定符号指针表
  __nl_symbol_ptr    非懒绑定符号指针表
  __got              全局偏移表（Global Offset Table）
```

## 五、链接

### 5.1 静态链接与动态链接

```
静态链接（Static Linking）：
  - 编译时将库的代码直接复制到最终可执行文件中
  - 生成的文件体积更大
  - 不依赖外部库文件，运行时不需要额外加载
  - 多个 App 使用同一个静态库，各自包含一份副本

  .o + .o + libXxx.a → 链接器（ld） → Mach-O 可执行文件

动态链接（Dynamic Linking）：
  - 可执行文件只记录对动态库的引用
  - 运行时由 dyld 加载并链接动态库
  - 多个进程可以共享同一份动态库（减少内存占用）
  - 系统框架（UIKit、Foundation）都是动态库

  .o + .o → 链接器 → Mach-O（引用 UIKit.framework）
                              ↓ 运行时
                         dyld 加载 UIKit.framework

静态库 vs 动态库：

  ┌──────────────┬────────────────────┬────────────────────┐
  │              │     静态库 (.a)     │   动态库 (.dylib)   │
  ├──────────────┼────────────────────┼────────────────────┤
  │  链接时机     │  编译时              │  运行时              │
  │  文件体积     │  增大主程序体积       │  主程序较小           │
  │  更新方式     │  需重新编译 App      │  可独立更新（系统库） │
  │  加载速度     │  无额外加载          │  启动时需加载         │
  │  内存共享     │  各进程独立副本       │  多进程可共享         │
  │  典型场景     │  第三方 SDK          │  系统框架             │
  └──────────────┴────────────────────┴────────────────────┘

注意：iOS App 不允许包含自定义的独立动态库，但可以使用
      Embedded Framework（嵌入式框架），本质是动态库但打包在 App 内部。
```

### 5.2 符号（Symbol）

```
符号是链接器的核心概念，代表函数、变量、类等的名称标识。

符号的分类：
  1. 已定义符号（Defined Symbol）
     - 在当前文件中实现的函数/变量
     - 链接器知道它的地址

  2. 未定义符号（Undefined Symbol）
     - 引用了外部的函数/变量，当前文件没有实现
     - 链接器需要在其他文件或库中找到它

  3. 导出符号（Exported Symbol）
     - 对外可见的符号，其他模块可以引用

  4. 本地符号（Local Symbol）
     - static 修饰的函数/变量，仅文件内部可见

查看符号表：
  nm MyApp                       // 列出所有符号
  nm -u MyApp                    // 只列出未定义符号
  nm -g MyApp                    // 只列出全局（导出）符号
  objdump --syms MyApp           // 另一种查看方式

符号名修饰（Name Mangling）：
  C 语言：_functionName
  OC：    _OBJC_CLASS_$_MyClass
  Swift：  复杂的修饰名，包含模块名、类型、参数类型等

反修饰工具：
  swift demangle 's7MyClass10myFunctionyyF'
  xcrun swift-demangle 's7MyClass10myFunctionyyF'
  c++filt '_Z3addii'
```

### 5.3 符号绑定（Symbol Binding）

```
动态库的符号在运行时由 dyld 绑定，分为两种：

  1. 非懒绑定（Non-Lazy Binding）
     - 在 dyld 加载时就绑定好
     - 存储在 __DATA,__nl_symbol_ptr
     - 通常用于：类引用、全局变量、OC 类
     - 影响启动时间

  2. 懒绑定（Lazy Binding）
     - 首次调用时才绑定
     - 存储在 __DATA,__la_symbol_ptr
     - 大部分函数调用使用懒绑定
     - 首次调用流程：

     调用 printf()
         │
         ▼
     __stubs 段     → 跳转到 __la_symbol_ptr 中的地址
         │                    （初始值指向 __stub_helper）
         ▼
     __stub_helper  → 调用 dyld_stub_binder
         │
         ▼
     dyld 查找真实地址 → 写回 __la_symbol_ptr
         │
         ▼
     后续调用 printf() → 直接从 __la_symbol_ptr 跳转到真实地址

fishhook 原理：
  - fishhook 利用了懒/非懒绑定的机制
  - 通过修改 __la_symbol_ptr / __nl_symbol_ptr 中的指针
  - 将系统函数（如 NSLog）的地址替换为自定义函数的地址
  - 这就是为什么 fishhook 只能 Hook 动态库的函数，无法 Hook 自定义的 C 函数
```

### 5.4 Strip 与符号裁剪

```
Strip 是去除 Mach-O 中调试符号和本地符号的过程：

  为什么要 Strip：
    - 减小二进制文件体积
    - 保护代码不被轻易反编译分析
    - Release 版本不需要调试符号

  Xcode 中的 Strip 配置（Build Settings）：

    Strip Linked Product = YES        // 是否对最终产物 Strip
    Strip Style：
      - All Symbols：移除所有符号（体积最小）
      - Non-Global Symbols：保留导出符号，移除本地符号
      - Debugging Symbols：只移除调试符号（最常用）

    Strip Swift Symbols = YES         // 移除 Swift 符号
    Deployment Postprocessing = YES   // 在安装时才 Strip（而非编译时）

  dSYM 文件：
    - Strip 之前，Xcode 会生成 dSYM 文件保留完整符号信息
    - 用于线上崩溃日志的符号化
    - 路径：Build/Products/Release-iphoneos/MyApp.app.dSYM

  手动 Strip：
    strip -x MyApp                    // 移除本地符号
    strip -S MyApp                    // 移除调试符号
```

## 六、dyld 加载流程

### 6.1 App 启动与 dyld

```
用户点击 App 图标后的加载过程：

  内核 execve()
       │
       ▼
  ┌──────────────────┐
  │  加载主可执行文件    │   内核将 Mach-O Header 映射到进程地址空间
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  启动 dyld         │   内核根据 LC_LOAD_DYLINKER 找到 dyld 并转交控制权
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  加载所有依赖库      │   递归解析 LC_LOAD_DYLIB，加载依赖的动态库
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  Rebase            │   修正 ASLR 偏移后的指针（因为程序基址每次启动随机化）
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  Bind              │   绑定非懒符号（外部函数/变量地址填入 __got 等）
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  Objective-C 运行时初始化 │  注册所有类、Category、Protocol，执行 +load 方法
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  执行 Initializer   │   C++ 静态构造函数、attribute((constructor)) 函数
  └────────┬──────────┘
           │
           ▼
  ┌──────────────────┐
  │  调用 main()        │   进入 main 函数 → UIApplicationMain
  └──────────────────┘
```

### 6.2 Rebase 与 Bind

```
ASLR（地址空间布局随机化）导致 App 每次启动的加载基址都不同：

  Rebase（重定基址）：
    - 修正镜像内部指针（指向自己模块内部的地址）
    - 例如：函数指针表、vtable 中记录的偏移量
    - 只需加上 slide（实际加载地址 - 预期加载地址）即可

  Bind（符号绑定）：
    - 修正指向外部符号的指针（指向其他动态库的函数/变量）
    - 需要先在其他库的符号表里查找到真实地址
    - 分为非懒绑定（启动时完成）和懒绑定（首次调用时完成，见第五章 5.3）

  两者的本质区别：
    Rebase 只做加法（同一镜像内的相对偏移不变，只是基址变了）
    Bind 需要查表（跨镜像引用，地址无法通过简单加法算出）

dyld 处理顺序：Rebase 先于 Bind，因为 Bind 阶段可能需要用到已经修正过的指针
```

### 6.3 dyld 共享缓存（dyld shared cache）

```
问题：iOS 系统自带几百个 Framework（UIKit、Foundation、CoreGraphics...）
      如果每个 App 启动都独立加载解析这些库，会非常慢

解决方案：dyld shared cache
  - 系统在设备启动时，将所有系统库预先链接、重定位好
  - 合并成一个巨大的缓存文件：
    /System/Library/Caches/com.apple.dyld/dyld_shared_cache_arm64

  优势：
    1. 所有 App 共享同一份已经 rebase/bind 好的系统库内存
    2. 省去了对系统库的重复符号查找和重定位
    3. 大幅减少 App 启动时间和内存占用

  查看共享缓存信息：
    dyld_shared_cache_util -list /path/to/dyld_shared_cache_arm64e

  这也是为什么 App 自己的 Framework（非系统库）越多，
  启动越慢：因为这些库不在共享缓存里，每次都要独立走完整的
  加载 / rebase / bind 流程。
```

### 6.4 +load 与 Initializer 的执行时机

```
dyld 完成符号绑定后，会触发一系列初始化回调，执行顺序：

  1. Objective-C +load 方法
     - 所有类和 Category 的 +load 会在 runtime 初始化阶段被调用
     - 顺序：父类先于子类，主类先于 Category，
             按编译顺序（Compile Sources 中的文件顺序）调用
     - +load 执行时机很早，此时其他类可能还未完全就绪，
       不建议在 +load 里访问其他类的资源

  2. C++ 静态构造函数 / __attribute__((constructor)) 函数
     - 在 +load 之后、main() 之前执行
     - 常用于一些底层 SDK 做提前注入或 Hook

  3. main() 函数
     - 至此 dyld 的工作完成，控制权交还给 App 代码
     - 调用 UIApplicationMain 进入正常的 App 生命周期

启动时间优化关键点（详见性能优化相关文章）：
  - 减少 +load 方法数量和内部逻辑复杂度
  - 减少动态库数量（合并 Framework，或改为静态库/Swift Package）
  - 减少非懒绑定符号数量（减少启动时需要 Bind 的符号）
```

## 七、Xcode 工程与构建工具链

### 7.1 DerivedData 管理与清理

DerivedData 是 Xcode 编译产物的默认输出目录（`~/Library/Developer/Xcode/DerivedData/`），包含构建中间文件、Index 数据库、Clang ModuleCache、Swift 依赖图缓存和测试报告。Xcode 的增量编译、代码索引、Build Vision 全部依赖此目录。

```bash
# 查看 DerivedData 占用空间：
du -sh ~/Library/Developer/Xcode/DerivedData/*

# 清理指定 project（推荐方式）：
# rm -rf ~/Library/Developer/Xcode/DerivedData/YourProject-*

# 自定义 DerivedData 路径（推荐多项目并行开发）：
# Xcode → Settings → Locations → DerivedData → Custom
# 建议放在外置 SSD 提升 I/O 性能：~/Volumes/SSD/DerivedData
```

⚠️ 坑：不同 project 共用 DerivedData 时，修改 target name 导致缓存被误清；CI 环境不清理 DerivedData 导致 "clean build" 与增量编译结果不一致。
✅ 最佳实践：CI 中每次构建前清理 DerivedData 或使用 `${PROJECT_NAME}-${CONFIGURATION}` 命名隔离；本地开发定期清理过期缓存。

### 7.2 Build Phases 自定义（Run Script、Shell Tool）

Build Phases 是 Xcode 编译流程中可自定义的扩展点，允许开发者在编译的不同阶段注入自定义脚本。Xcode 的构建顺序严格遵循 Build Phases 面板中从上到下的排列。

```bash
# 常见 Build Phase 类型：
# - Sources Compile：编译源代码
# - Frameworks Link：链接框架
# - Resources Copy：拷贝资源文件
# - Run Script：执行自定义 Shell 脚本（最常用）
# - Shell Tool（Xcode 15+）：替代 Run Script，结构化更强

# Run Script 示例：自动替换版本号
plist="/path/to/Info.plist"
build_num=${CURRENT_PROJECT_VERSION:-1}
/usr/libexec/PlistBuddy -c "Set :CFBundleVersion ${build_num}" "${plist}"

# Shell Tool 示例（Xcode 15+，结构化配置）：
# - 选择 Shell：/bin/bash
# - 脚本路径：指向项目内的 .sh 文件
# - 参数：通过 UI 配置，无需硬编码在脚本中
# - 可配置：是否忽略错误、超时时间
```

⚠️ 坑：Run Script 执行失败会阻塞整个构建，CI 中路径硬编码导致本地与远程环境不一致；脚本中未处理并行构建（${DERIVED_FILE_DIR} 可能冲突）。
✅ 最佳实践：使用 ${DERIVED_FILE_DIR} 和 ${CONFIGURATION} 等自动变量避免硬编码路径；Shell Tool 优先于 Run Script（Xcode 15+）；复杂逻辑抽离为独立 .sh 文件并添加错误处理。

### 7.3 Tuist 作为大型项目的 Xcode 替代方案

Tuist 是用 Swift 编写的开源构建系统，旨在替代 Xcode 的项目生成与管理。它通过声明式的 Project Description API 定义 target、依赖、构建设置，生成轻量级 .xcodeproj 文件。核心理念是「项目定义即代码」，将项目结构版本化、可测试化。对于拥有数百 target 的大型项目，Tuist 的增量构建缓存（Remote Cache）可显著减少 CI 构建时间，避免 Xcode 的 DerivedData 膨胀问题。

```swift
import ProjectDescription

let project = Project(name: "MyApp",
    targets: [
        Target(name: "MyApp",
            platforms: [.iOS],
            product: .app,
            dependencyTypes: [.target(name: "Core")],
            settings: .settings(baseSettings: .application())
        ),
        Target(name: "Core",
            platforms: [.iOS],
            product: .library,
            dependencyTypes: [.external(name: "SwiftLint", requirement: .upToNextMajor("0.50.0"))])
    ],
    schemes: [Schemes.buildScheme(name: "MyApp")]
)
```

⚠️ 坑：Tuist 生成的 .xcodeproj 不包含 UI 设计信息（Interface Builder layout 位置、scheme 颜色等），从 Xcode 迁移时需手动重新配置 UI 相关设置；Tuist Gen 在依赖变更后全量重新生成项目，首次生成耗时可能超过 Xcode 的 index。
✅ 最佳实践：配合 Tuist Remote Cache 使用，将预编译的 artifact 存储在 S3/GCS 上实现团队共享缓存；Project Description 中封装公司内部的 Template 模块，统一 lint、test、code coverage 配置。

### 7.4 Swift Package 条件依赖（`dependency(.when:)` / `.execution`）

Swift 6+ 引入条件依赖语法，允许根据目标平台、部署目标、构建配置等条件动态控制依赖关系。解决了「同一包在不同平台需要不同依赖」的痛点，避免强制引入不兼容的依赖。核心 API 是 `.when()` 修饰符，支持按 platform、deploymentTarget、config 和 execution 维度过滤。

```swift
// Package.swift — 条件依赖示例
let package = Package(
    name: "MyApp",
    dependencies: [
        // 仅在 Linux 上引入依赖
        .package(url: "https://github.com/apple/swift-mocks", from: "1.0.0")
            .when(platform: [.linux]),

        // 仅在 iOS 15+ 可用
        .package(url: "https://github.com/apple/swift-docc-plugin", from: "1.0.0")
            .when(platform: [.iOS], .deploymentTarget("iOS", .v15)),

        // 仅在 debug 配置时引入测试工具
        .package(url: "https://github.com/nicklockwood/SwiftFormat", from: "0.51.0")
            .when(config: [.debug]),

        // execution: 控制依赖参与哪个阶段
        .product(name: "SwiftLint", package: "SwiftLint")
            .when(execution: [.build])  // 仅 build 阶段，不参与 runtime
    ],
    targets: [
        .target(name: "App", dependencies: [.product(name: "Core", package: "MyCore")])
    ]
)
```

⚠️ 坑：条件依赖在 SPM resolve 时求值，错误的 platform 判断导致依赖静默缺失；`.when(config: .debug)` 在 release 构建中完全跳过，可能遗漏必要的 lint 步骤；不同 target 使用不同条件时，SPM 会为每种组合生成独立的 build plan，显著增加首次 resolve 时间。
✅ 最佳实践：优先使用 `.when(platform:)` 处理跨平台差异，而非在源码中用 `#if os()` 条件编译绕过依赖；`.when(config:)` 仅用于开发/测试工具链，不要将 runtime 依赖绑到 debug 配置；定期 `swift package resolve` 清理缓存避免 stale dependency graph。

## 八、二进制分发与打包

### 8.1 XCFramework 创建与分发

XCFramework 是 Apple 于 2019 年引入的二进制分发格式，将多个架构切片（arm64 真机、x86_64/ARM64 模拟器、Mac Catalyst）打包为统一归档，附带 Headers、Modules 和 Resources。相比传统 .framework + lipo 合并 fat binary，XCFramework 在 Xcode 层面原生支持模拟器/真机自动选择，消除了 "simulator architectures" 链接错误。

```bash
# 1. 为不同平台构建 Framework
xcodebuild archive -scheme MyLib -sdk iphoneos -archivePath build/ios.xcarchive \
  SKIP_INSTALL=NO BUILD_LIBRARY_FOR_DISTRIBUTION=YES
xcodebuild archive -scheme MyLib -sdk iphonesimulator -archivePath build/sim.xcarchive \
  SKIP_INSTALL=NO BUILD_LIBRARY_FOR_DISTRIBUTION=YES

# 2. 合并为 XCFramework
xcodebuild -create-xcframework \
  -framework build/ios.xcarchive/Products/Library/Frameworks/MyLib.framework \
  -framework build/sim.xcarchive/Products/Library/Frameworks/MyLib.framework \
  -output MyLib.xcframework

# 3. 验证
xcrun xcodebuild -check-first-party-frameworks MyLib.xcframework
```

⚠️ 坑：BUILD_LIBRARY_FOR_DISTRIBUTION 必须设为 YES，否则 dSYM 和调试符号混入导致体积膨胀且可能泄露内部实现；忘记包含 Mac Catalyst 切片导致跨平台项目链接失败。
✅ 最佳实践：CI 中自动化构建 + 完整性校验；通过 SPM 或 CocoaPods spec 分发，附带 PrivacyInfo.xcprivacy 满足 App Store 审核。

### 8.2 Archive / Distribution / Ad Hoc / Enterprise 签名

iOS 应用分发依赖代码签名机制。Archive 后通过 Xcode 的 Distribute App 流程生成 .ipa 并签名，支持四种分发模式：App Store Connect（自动签名+审核上架）、Ad Hoc（需预注册设备 UDID，限 100 台/年）、Enterprise（内部部署，无需审核，但需定期重新签名）、Development（仅调试设备可用）。所有签名均由 Apple 的证书链验证，证书吊销通过 CRL/OCSP 实时检查。

```swift
// 查看当前 app 的签名信息
import Foundation
let codeSignIdent = Bundle.main.object(forInfoDictionaryKey: "com.apple_CODE_SIGN_IDENTITY") as? String
let teamID = Bundle.main.bundleIdentifier?.prefix(10)
print("CodeSign: \(codeSignIdent ?? "unknown")")
print("TeamID: \(teamID ?? "unknown")")

// Ad Hoc 分发：Xcode -> Product -> Archive -> Distribute App -> Ad Hoc
// 命令行签名工具
// xcrun altool --upload-app --type ios --file ./app.ipa --apiKey KEY_ID --apiIssuer ISSUER_ID
```

⚠️ 坑：Ad Hoc 设备超过 100 台限制后需删除旧设备或切换到 Enterprise；Enterprise 证书泄露导致大量非法 app 在野传播，Apple 会批量吊销证书使用户已安装的 app 失效；Archive 后修改 Info.plist 但重新签名时忘记 update Entitlements 导致安装失败。
✅ 最佳实践：使用 Fastlane match 管理证书与 provisioning profile，团队共享且自动轮换；Ad Hoc 测试优先用 TestFlight（内部测试最多 100 人），减少设备注册管理开销；Enterprise 分发配合 MDM（Mobile Device Management）推送，实现零交互安装。

## 九、CI/CD 自动化

### 9.1 Fastlane CI/CD 自动化（match、snapshot、deliver、sigh）

Fastlane 是 iOS 最成熟的 CI/CD 自动化框架，核心模块包括：`match`（证书与 Profile 统一管理）、`snapshot`（自动化多语言截图）、`deliver`（元数据与二进制上传至 App Store Connect）、`sigh`（自动创建/更新 provisioning profile）。在 CI 环境中，Fastlane 将原本需要人工操作的开发者证书、截图审核、上架流程全部脚本化，实现一键发布。

```ruby
# Fastfile 核心 action 示例
lane :release do
  match(type: "appstore", readonly: true)       # 拉取证书
  sigh(                          # 自动处理 provisioning
    app_identifier: "com.example.app",
    force_for_new_devices: true
  )
  increment_build_number
  build_app(archive: true, skip_codesigning: false)
  upload_to_app_store(skip_metadata: true, skip_screenshot_verification: true)
  slack(message: "🚀 新版本已提交审核")
end

# match 初始化（一次性执行，将证书加密存储到 Git）
# fastlane match init  # 创建 .match 目录
# fastlane match appstore  # 创建 App Store 证书
```

⚠️ 坑：`match` 的加密存储依赖 `FASTLANE_MATCH_PASSWORD`，密码泄露等于暴露所有证书私钥；`deliver` 元数据更新会覆盖 App Store Connect 上手动编辑的内容，团队需严格区分自动化与手动维护的字段；CI 上未配置 `readonly: true` 导致 build 机器也能创建新证书，造成证书混乱。
✅ 最佳实践：证书仓库使用独立的 private Git repo，通过环境变量注入 match password；生产 lane 始终用 `match(readonly: true)`；`deliver` 配合 `force_latest_upload: false` 防止误覆盖已审核版本。
