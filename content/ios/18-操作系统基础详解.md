# 操作系统基础详解

---

## 一、进程与线程

### 1.1 进程（Process）

```
进程是操作系统进行资源分配和保护的基本单位。

进程拥有的资源：
  - 独立的虚拟地址空间（代码段、数据段、堆、栈）
  - 文件描述符表
  - 信号处理函数
  - 进程 ID（PID）
  - 用户 ID 和组 ID

进程的状态：
  ┌──────────┐  创建   ┌──────────┐  调度  ┌──────────┐
  │  新建态   │ ──────→ │  就绪态   │ ─────→ │  运行态   │
  └──────────┘         └──────────┘ ←───── └──────────┘
                              ↑       抢占        │
                              │                  │ 等待 I/O
                              │                  ↓
                              │           ┌──────────┐
                              └────────── │  阻塞态   │
                              I/O 完成    └──────────┘

iOS 中每个 App 就是一个进程：
  - 进程间相互隔离（沙盒机制）
  - 进程间通信（IPC）：URL Scheme、Universal Link、App Group、XPC
  - 系统通过 Jetsam 机制在内存不足时终止后台进程
```

### 1.2 线程（Thread）

```
线程是 CPU 调度的基本单位，是进程中的执行流。

同一进程的线程共享：
  - 代码段、数据段、堆
  - 文件描述符
  - 信号处理

每个线程独有：
  - 程序计数器（PC）
  - 寄存器组
  - 栈（独立的栈空间）
  - 线程 ID

线程 vs 进程：
  ┌──────────────────┬────────────────────┬────────────────────┐
  │                  │       进程           │       线程          │
  ├──────────────────┼────────────────────┼────────────────────┤
  │ 资源             │ 独立地址空间         │ 共享进程资源         │
  │ 切换开销         │ 大（需要切换页表）   │ 小（只切换寄存器）   │
  │ 通信             │ 复杂（IPC）         │ 简单（共享内存）     │
  │ 安全性           │ 互不影响            │ 一个崩溃影响全部     │
  │ 创建开销         │ 大                 │ 小                  │
  └──────────────────┴────────────────────┴────────────────────┘

iOS 的线程模型：
  - 主线程（UI 线程）：处理 UI 事件和渲染
  - 子线程：执行耗时操作（网络、数据库、图片处理）
  - 默认栈大小：主线程 1MB，子线程 512KB
  - 最大线程数：没有硬性限制，但过多会导致性能下降
```

### 1.3 进程调度

```
CPU 调度算法：

1. FCFS（先来先服务）
   - 按到达顺序执行
   - 简单，但长任务会让短任务等待（护航效应）

2. SJF（最短作业优先）
   - 优先执行预计运行时间最短的进程
   - 最优平均等待时间，但可能饥饿长进程

3. 优先级调度
   - 每个进程有优先级，高优先级先执行
   - 可能导致低优先级进程饥饿（老化机制解决）

4. 轮转调度（Round Robin）
   - 每个进程分配一个时间片（time quantum）
   - 时间片用完后切换到下一个进程
   - 公平，响应时间短，适合交互式系统

5. 多级反馈队列（Multilevel Feedback Queue）
   - 多个优先级队列，高优先级时间片短
   - 新进程进入最高优先级队列
   - 用完时间片后降到下一级队列
   - 长时间等待的进程优先级提升（防饥饿）
   - iOS/macOS 使用类似机制（QoS）

iOS 的 QoS（Quality of Service）：
  User Interactive  → 最高优先级，UI 相关（主线程）
  User Initiated    → 用户发起的即时操作
  Default           → 默认
  Utility           → 长时间运行的任务（下载、导入）
  Background        → 后台任务（备份、同步）
  Unspecified       → 未指定
```

## 二、内存管理

### 2.1 虚拟内存

```
虚拟内存使每个进程都认为自己拥有整个地址空间。

虚拟地址空间（64 位系统，iOS）：

  高地址
  ┌─────────────────────┐
  │     内核空间         │  内核代码、数据（进程无法直接访问）
  ├─────────────────────┤ 0xFFFF_FFFF_FFFF_FFFF
  │                     │
  │     Stack（栈）      │  ↓ 向低地址增长
  │                     │  函数调用帧、局部变量
  ├─────────────────────┤
  │                     │
  │   Memory Mapping    │  动态库、mmap 映射文件
  │                     │
  ├─────────────────────┤
  │                     │
  │     Heap（堆）       │  ↑ 向高地址增长
  │                     │  malloc/new 分配的内存
  ├─────────────────────┤
  │     BSS 段           │  未初始化的全局变量（清零）
  ├─────────────────────┤
  │     Data 段          │  已初始化的全局变量、静态变量
  ├─────────────────────┤
  │     Text 段          │  代码（只读、可执行）
  └─────────────────────┘
  低地址 0x0000_0000_0000_0000

虚拟地址 → 物理地址转换：
  MMU（内存管理单元）通过页表完成转换
  页表存储在内存中，TLB 缓存最近的转换结果

iOS 特点：
  - 没有 Swap（不能将内存换出到磁盘）
  - 使用 Compressed Memory（压缩不活跃内存页）
  - 内存不足时 Jetsam 直接杀进程
```

### 2.2 内存分页与缺页中断

```
分页（Paging）：
  - 将虚拟地址空间和物理内存都分成固定大小的页（Page）
  - iOS 页大小：4KB（旧设备）或 16KB（A9 及以后）
  - 虚拟页映射到物理页帧（Frame）

页表（Page Table）：
  虚拟页号（VPN）→ 物理页帧号（PFN）
  每个页表项还包含：
    - 有效位（Present bit）：该页是否在物理内存中
    - 修改位（Dirty bit）：该页是否被修改过
    - 访问位（Access bit）：该页最近是否被访问
    - 保护位：读/写/执行权限

缺页中断（Page Fault）：
  1. CPU 访问某个虚拟地址
  2. MMU 查页表，发现有效位为 0（页不在内存中）
  3. 触发缺页中断，陷入内核
  4. 内核从磁盘（或 mmap 的文件）加载该页到物理内存
  5. 更新页表，设置有效位为 1
  6. 重新执行触发缺页中断的指令

iOS 启动时的 Page Fault：
  - App 启动时，代码页不在内存中
  - 首次访问每个代码页都会触发 Page Fault
  - 每次 Page Fault 约耗时 0.1-0.8ms
  - 二进制重排（Binary Layout Optimization）将启动时用到的代码
    排列到相邻的内存页，减少 Page Fault 次数

TLB（Translation Lookaside Buffer）：
  - 页表的缓存，存储最近的虚拟→物理地址转换
  - 命中 TLB：直接得到物理地址，速度极快
  - 未命中 TLB：查页表（慢），更新 TLB
  - 进程切换时需要刷新 TLB（或使用 ASID 区分不同进程的 TLB 条目）
```

### 2.3 内存分配

```
用户空间内存分配（malloc）：

  iOS 使用 libmalloc（基于 jemalloc 思想的分级分配）：

  ┌──────────────────────────────────────────────┐
  │  Tiny   ≤ 256 bytes   按 16 字节对齐          │
  │  Small  ≤ 8KB         按 512 字节对齐          │
  │  Large  > 8KB         直接 mmap，按页对齐      │
  └──────────────────────────────────────────────┘

  Tiny/Small 分配策略：
    - 预先分配一大块内存（Region/Magazine）
    - 按大小分成 slot（空闲链表）
    - 分配时从空闲链表取，释放时归还
    - 减少系统调用次数

  Large 分配策略：
    - 直接调用 mmap 分配虚拟内存
    - 释放时 munmap 归还
    - 避免内存碎片

ObjC 对象内存分配：
  [[NSObject alloc] init]
    → objc_alloc
    → calloc（零初始化）
    → 返回指针

  对象大小 = instanceSize（isa + 所有 ivar，16 字节对齐）

  最小对象大小 = 16 字节（即使 NSObject 本身只需要 8 字节）
```

### 2.4 内存泄漏与内存溢出

```
内存泄漏（Memory Leak）：
  - 分配的内存不再使用，但没有释放
  - 程序运行时间越长，内存占用越大
  - iOS 中常见原因：循环引用、未移除的通知、Timer 未 invalidate

内存溢出（OOM - Out of Memory）：
  - 申请内存超过系统限制
  - iOS 没有 Swap，内存不足直接被 Jetsam 杀进程
  - OOM 不会产生 Crash 日志（不是 Crash，是被系统杀死）

iOS 内存限制（参考值，因设备而异）：
  前台 App：约 1.5-3GB（高端设备）
  后台 App：更低，系统可随时回收

内存警告处理：
  1. 系统发出 UIApplicationDidReceiveMemoryWarningNotification
  2. 调用 applicationDidReceiveMemoryWarning
  3. 调用所有 ViewController 的 didReceiveMemoryWarning
  4. 如果还不够，系统开始杀后台进程
  5. 最终杀前台进程

Dirty Memory vs Clean Memory：
  Clean Memory（可回收）：
    - 代码段（Text）
    - 只读数据
    - 内存映射文件（mmap）
    - 系统 Framework（共享缓存）

  Dirty Memory（不可回收）：
    - 堆上分配的对象
    - 全局变量
    - 栈
    - 写时复制（CoW）后的页

  优化目标：减少 Dirty Memory
```

## 三、进程同步与互斥

### 3.1 临界区问题

```
临界区（Critical Section）：
  访问共享资源的代码段

临界区问题的三个要求：
  1. 互斥（Mutual Exclusion）：同一时刻只有一个进程在临界区
  2. 前进（Progress）：没有进程在临界区时，等待进入的进程可以选出一个
  3. 有限等待（Bounded Waiting）：等待时间有限，不会无限期阻塞

解决方案：
  - 互斥锁（Mutex）
  - 信号量（Semaphore）
  - 管程（Monitor）
  - 原子操作（Atomic Operation）
```

### 3.2 死锁

```
死锁（Deadlock）：
  多个进程/线程相互等待对方释放资源，导致都无法继续执行。

死锁的四个必要条件（Coffman 条件）：
  1. 互斥（Mutual Exclusion）：资源不能共享，同一时刻只能被一个进程持有
  2. 持有并等待（Hold and Wait）：进程持有资源的同时等待其他资源
  3. 非抢占（No Preemption）：资源不能被强制抢占，只能主动释放
  4. 循环等待（Circular Wait）：进程之间形成等待环

死锁预防（破坏必要条件）：
  - 破坏持有并等待：一次性申请所有资源
  - 破坏非抢占：允许资源被抢占（优先级反转解决方案）
  - 破坏循环等待：对资源编号，按顺序申请

死锁避免（银行家算法）：
  - 在分配资源前检查是否会进入不安全状态
  - 只在安全状态下分配资源

死锁检测与恢复：
  - 允许死锁发生，定期检测
  - 发现死锁后终止进程或抢占资源

iOS 中的死锁：
  - 主线程 sync 到主队列 → 死锁
  - 串行队列中 sync 到同一串行队列 → 死锁
  - 嵌套加锁（不同顺序）→ 死锁
```

### 3.3 原子操作

```
原子操作（Atomic Operation）：
  不可分割的操作，要么完全执行，要么完全不执行。

CPU 提供的原子指令：
  CAS（Compare-And-Swap）：
    if (memory == expected) { memory = new; return true; }
    else { return false; }
    这是一条原子指令，不会被中断

  LDXR/STXR（ARM64 独占加载/存储）：
    iOS 设备（ARM64）使用这对指令实现原子操作

Swift 的原子操作（Swift 5.9+）：
  import Synchronization
  let counter = Atomic<Int>(0)
  counter.wrappingAdd(1, ordering: .relaxed)

ObjC 的原子操作：
  __sync_fetch_and_add(&value, 1)
  OSAtomicAdd32(1, &value)   // 已废弃，用 Atomic 替代

iOS 中的 atomic 属性：
  @property (atomic, assign) NSInteger count;
  - getter/setter 使用 os_unfair_lock 保护
  - 只保证单次读写的原子性
  - 不保证复合操作（count++）的原子性
```

## 四、文件系统

### 4.1 文件系统基础

```
iOS/macOS 使用 APFS（Apple File System，苹果文件系统）：
  - 2017 年发布，替代 HFS+
  - 特性：
    - 写时复制（Copy-on-Write）：修改文件时先写新位置，再更新指针
    - 快照（Snapshots）：瞬间创建文件系统快照
    - 克隆（Clones）：零拷贝文件克隆
    - 加密（Native Encryption）：文件级别加密
    - 空间共享：多个卷共享同一存储池
    - 纳秒时间戳
    - 崩溃保护（原子操作）

文件描述符（File Descriptor）：
  - 整数，代表打开的文件
  - 0 = 标准输入，1 = 标准输出，2 = 标准错误
  - 每个进程有自己的文件描述符表
  - ulimit 限制每个进程的最大文件描述符数量

iOS 沙盒目录：
  Bundle/              App 包（只读）
  Documents/           用户数据（iCloud 备份）
  Library/
    Caches/            可被系统清除的缓存
    Preferences/       UserDefaults 存储
    Application Support/ 支持文件（iCloud 备份）
  tmp/                 临时文件（可被系统清除）
```

### 4.2 I/O 模型

```
五种 I/O 模型：

1. 阻塞 I/O（Blocking I/O）
   进程发起 I/O 请求，等待数据就绪，期间阻塞
   ┌──────┐  read()  ┌──────────┐  数据就绪  ┌──────┐
   │进程   │ ───────→ │  内核     │ ─────────→ │进程   │
   └──────┘  阻塞     └──────────┘  返回数据   └──────┘

2. 非阻塞 I/O（Non-Blocking I/O）
   read() 立即返回，如果数据未就绪返回 EAGAIN
   进程轮询（busy-waiting），浪费 CPU

3. I/O 多路复用（I/O Multiplexing）
   select/poll/epoll 监听多个文件描述符
   任何一个就绪就返回，然后再发起 read()
   iOS 使用 kqueue（BSD 版本的 epoll）

4. 信号驱动 I/O（Signal-Driven I/O）
   数据就绪时内核发信号通知进程
   进程收到信号后发起 read()

5. 异步 I/O（Asynchronous I/O）
   进程发起 I/O 请求后立即返回
   内核完成 I/O 后通知进程
   iOS 的 URLSession 后台下载本质上是异步 I/O

iOS 网络 I/O：
  URLSession 底层使用 kqueue + 线程池
  GCD 的 DispatchSource 封装了 kqueue
```

## 五、CPU 架构基础

### 5.1 ARM64 架构

```
iOS 设备使用 ARM64（AArch64）架构。

通用寄存器（31 个）：
  x0-x7    函数参数/返回值（前 8 个参数）
  x8       间接结果寄存器（结构体返回值的地址）
  x9-x15   临时寄存器（调用者保存）
  x16-x17  过程链接表（PLT）临时寄存器
  x18      平台保留（iOS 中不使用）
  x19-x28  被调用者保存寄存器
  x29      帧指针（FP）
  x30      链接寄存器（LR，保存返回地址）

特殊寄存器：
  sp       栈指针（Stack Pointer）
  pc       程序计数器（Program Counter）
  xzr      零寄存器（读取总是 0，写入丢弃）

函数调用约定（ABI）：
  1. 前 8 个参数放 x0-x7，其余压栈
  2. 返回值放 x0（整数/指针）或 v0（浮点）
  3. 调用前保存调用者保存寄存器（x9-x15）
  4. bl 指令跳转并将返回地址存入 x30（lr）
  5. ret 指令跳转到 lr 中的地址返回

栈帧结构：
  高地址
  ┌─────────────────┐
  │  调用者的栈帧    │
  ├─────────────────┤ ← 调用前的 sp
  │  保存的 lr       │  x30
  │  保存的 fp       │  x29
  ├─────────────────┤ ← 新的 fp（x29）
  │  被调用者保存的  │
  │  寄存器          │
  ├─────────────────┤
  │  局部变量        │
  └─────────────────┘ ← 新的 sp
  低地址
```

### 5.2 CPU 缓存

```
CPU 缓存层次（从快到慢）：

  寄存器    < 1ns    容量极小（几十个）
  L1 缓存   ~1ns     32-64KB，每个核心独有
  L2 缓存   ~4ns     256KB-1MB，每个核心独有
  L3 缓存   ~10ns    几 MB 到几十 MB，多核共享
  内存（RAM）~100ns  GB 级别
  SSD       ~100µs   TB 级别
  HDD       ~10ms    TB 级别

缓存友好的代码：
  - 顺序访问数组（空间局部性）比随机访问快得多
  - 链表遍历比数组遍历慢（节点分散在内存各处）
  - 结构体字段访问比指针间接访问快

缓存行（Cache Line）：
  - 64 字节（ARM64）
  - 每次从内存加载一个缓存行
  - 多线程中的伪共享（False Sharing）：
    两个线程分别修改同一缓存行中的不同变量
    → 频繁的缓存行失效，性能下降
    解决：使用 padding 让两个变量不在同一缓存行
```

## 六、iOS 系统架构

### 6.1 Darwin 内核（XNU）

```
iOS/macOS 的内核是 XNU（X is Not Unix）：

  XNU = Mach 微内核 + BSD 子系统 + I/O Kit

  ┌─────────────────────────────────────────────────┐
  │                  用户空间                         │
  │  App → UIKit → Foundation → libSystem → 系统调用 │
  └───────────────────────┬─────────────────────────┘
                          │ 系统调用（syscall）
  ┌───────────────────────┴─────────────────────────┐
  │                    XNU 内核                       │
  │  ┌─────────────┐  ┌────────────┐  ┌──────────┐  │
  │  │  Mach 微内核  │  │  BSD 子系统 │  │  I/O Kit │  │
  │  │  进程/线程   │  │  文件系统   │  │  驱动框架 │  │
  │  │  虚拟内存    │  │  网络栈    │  │          │  │
  │  │  IPC        │  │  POSIX API │  │          │  │
  │  └─────────────┘  └────────────┘  └──────────┘  │
  └─────────────────────────────────────────────────┘

Mach 微内核的核心概念：
  - Task：类似进程，拥有虚拟地址空间和资源
  - Thread：Mach 线程，CPU 调度单位
  - Port：进程间通信的端点（Mach Port）
  - Message：通过 Port 传递的消息

Mach Port 在 iOS 中的应用：
  - RunLoop 的 Source1 基于 Mach Port
  - XPC 服务通信
  - 系统事件（触摸、推送）的传递
```

### 6.2 iOS 安全机制

```
1. 沙盒（Sandbox）
   - 每个 App 只能访问自己的沙盒目录
   - 不能访问其他 App 的数据
   - 系统通过 MAC（强制访问控制）实现

2. 代码签名（Code Signing）
   - App 必须由苹果信任的证书签名
   - 系统在加载 Mach-O 时验证签名
   - 防止运行未经授权的代码

3. ASLR（地址空间随机化）
   - 每次启动，进程的基地址随机
   - 增加攻击者利用固定地址漏洞的难度

4. DEP/NX（数据执行保护）
   - 标记数据页为不可执行
   - 防止代码注入攻击

5. SIP（系统完整性保护）
   - macOS 特有，iOS 的沙盒更严格
   - 即使 root 用户也不能修改系统文件

6. Secure Enclave
   - 独立的安全处理器（A7 及以后）
   - 存储和处理敏感数据（Face ID、Touch ID、Apple Pay）
   - 即使主处理器被攻破，Secure Enclave 中的数据仍安全

7. 权限系统（Entitlements）
   - App 需要声明使用的系统功能（相机、位置、通讯录等）
   - 用户需要授权
   - 沙盒外的访问都需要权限
```

## 七、系统调用

### 7.1 系统调用概述

```
系统调用（System Call）：
  用户空间程序请求内核服务的接口

系统调用过程：
  1. 用户程序调用系统调用包装函数（如 open()）
  2. 设置系统调用号和参数（寄存器）
  3. 执行 svc #0 指令（ARM64）触发异常
  4. CPU 切换到内核模式（EL1）
  5. 内核根据系统调用号执行对应处理函数
  6. 返回结果，CPU 切回用户模式（EL0）

常见系统调用：
  文件：open, close, read, write, lseek, stat, mmap
  进程：fork, exec, exit, wait, getpid
  内存：brk, mmap, munmap
  网络：socket, bind, listen, accept, connect, send, recv
  线程：pthread_create, pthread_join（底层是 clone）
  时间：gettimeofday, clock_gettime

用户态 vs 内核态切换开销：
  - 保存/恢复寄存器
  - 切换页表（如果需要）
  - TLB 刷新（部分情况）
  - 约需要几百纳秒
  - 频繁系统调用会影响性能（批量 I/O 比逐字节 I/O 快很多）
```

## 八、常见面试题

### Q1：进程和线程的区别？

```
进程是资源分配的基本单位，线程是 CPU 调度的基本单位。

资源：
  进程拥有独立的地址空间、文件描述符等资源
  同一进程的线程共享这些资源，但有各自独立的栈和寄存器

切换开销：
  进程切换需要切换页表、刷新 TLB，开销大
  线程切换只需切换寄存器和栈，开销小

通信：
  进程间通信复杂（管道、消息队列、共享内存、Socket）
  线程间通信简单（共享内存，但需要同步）

安全性：
  进程间相互隔离，一个崩溃不影响其他进程
  同一进程的线程共享地址空间，一个线程崩溃可能导致整个进程崩溃

iOS 中：
  每个 App 是一个进程，进程间通过 URL Scheme、XPC 等通信
  App 内部使用多线程（GCD、NSOperation）处理并发
```

### Q2：什么是虚拟内存？它解决了什么问题？

```
虚拟内存是一种内存管理技术，使每个进程都认为自己拥有完整的连续地址空间。

解决的问题：
  1. 内存不足：程序可以使用比物理内存更多的地址空间（通过 Swap 或按需加载）
  2. 地址隔离：每个进程有独立的地址空间，互不干扰（安全）
  3. 内存共享：多个进程可以映射同一物理内存（如共享库）
  4. 简化编程：程序不需要关心物理内存布局

实现原理：
  - 将虚拟地址空间和物理内存都分成固定大小的页（Page）
  - 通过页表完成虚拟页到物理页的映射
  - MMU 硬件负责地址转换
  - 访问未映射的页触发缺页中断，内核处理

iOS 的特殊性：
  - 没有 Swap（不能换出到磁盘）
  - 内存不足时直接杀进程（Jetsam）
  - 使用 Compressed Memory 压缩不活跃页
```

### Q3：什么是死锁？如何避免？

```
死锁：多个进程/线程相互等待对方持有的资源，导致都无法继续执行。

必要条件（四个都满足才会死锁）：
  1. 互斥：资源同一时刻只能被一个进程持有
  2. 持有并等待：进程持有资源的同时等待其他资源
  3. 非抢占：资源不能被强制抢占
  4. 循环等待：进程之间形成等待环

避免死锁的方法：
  1. 固定加锁顺序：所有线程按相同顺序获取锁
  2. 一次性申请所有资源：要么全部获得，要么全不获得
  3. 超时机制：等待超时后释放已持有的锁，重试
  4. 使用 tryLock：尝试加锁失败就放弃，避免阻塞
  5. 减少锁的粒度：缩短持有锁的时间

iOS 中常见死锁：
  - 主线程 dispatch_sync 到主队列
  - 串行队列中 dispatch_sync 到同一队列
  - 使用 NSRecursiveLock 代替 NSLock 解决递归加锁问题
```

### Q4：缺页中断是什么？对 iOS 启动速度有什么影响？

```
缺页中断（Page Fault）：
  当 CPU 访问的虚拟页不在物理内存中时，触发的异常。
  内核处理：从磁盘加载该页到物理内存，更新页表，重新执行指令。

对 iOS 启动速度的影响：
  App 启动时，代码分散在多个内存页中。
  首次执行每个新代码页时都会触发 Page Fault（约 0.1-0.8ms 每次）。
  如果启动用到的代码分散在 100 个页中，就有 100 次 Page Fault。

解决方案：二进制重排（Binary Layout Optimization）
  1. 通过 Clang 插桩收集启动时的函数调用顺序
  2. 生成 Order File，按调用顺序排列函数
  3. 链接时按 Order File 的顺序排列代码
  4. 启动时用到的代码集中在相邻的少数几个页
  5. Page Fault 次数大幅减少

效果：抖音等 App 通过二进制重排减少约 30% 的启动时间。
```

### Q5：iOS 的内存警告机制是怎样的？

```
iOS 没有 Swap，内存不足时通过 Jetsam 机制处理：

内存压力等级：
  Normal → Warning → Critical → 终止进程

处理流程：
  1. 系统发出内存警告（UIApplicationDidReceiveMemoryWarningNotification）
  2. App 收到通知，应释放可重建的缓存（图片缓存、NSCache 等）
  3. 如果内存仍不足，系统开始杀后台进程（按内存占用和优先级）
  4. 后台进程全杀完还不够，杀前台进程（OOM 崩溃）

开发者应该：
  - 监听 UIApplicationDidReceiveMemoryWarningNotification
  - 在 didReceiveMemoryWarning 中清理可重建的缓存
  - 使用 NSCache（系统可自动回收）而非 NSDictionary
  - 避免持有大量 Dirty Memory（大图、大数据）
  - 图片使用降采样，只加载显示所需尺寸

OOM 检测（排除法）：
  App 被杀但没有 Crash 日志 → 可能是 OOM
  排除：用户主动杀、系统升级重启、Watchdog 超时
  剩余 → OOM（通过 MetricKit 的 memoryMetrics 监控）
```

### XNU 内核架构（Mach 微内核 + BSD 层 + IOKit）

```
XNU = eXtinct Native Unix，苹果自研内核，三层架构：

┌─────────────────────────────────┐
│         BSD Layer               │  ← POSIX API: file, socket, process
│   (pthread, file I/O, socket)   │     信号量, IPC (pipe, socket)
├─────────────────────────────────┤
│      Mach Microkernel           │  ← 核心机制: 进程/线程管理, 虚拟内存
│   (threads, VM, IPC, interrupts)│     中断处理, 消息传递(Mach ports)
├─────────────────────────────────┤
│        IOKit                     │  ← 硬件抽象层: 驱动框架, power mgmt
│   (drivers, power mgmt)         │     设备管理, 低功耗调度
└─────────────────────────────────┘

关键设计决策：
  Mach 负责「内核机制」（机制层），BSD 负责「策略」（策略层）
  这种分离使得 BSD 层可以被替换而不影响内核核心

iOS 开发者最常接触的层：
  BSD层 → 文件系统、网络、pthread、信号量
  Mach层 → 线程调度、虚拟内存、Mach IPC（XPC 底层就是 Mach ports）
  IOKit → CoreMotion、CoreLocation 等系统服务的驱动层

XPC 就是构建在 Mach ports 之上的高级 IPC 机制，提供了序列化、安全域等抽象。

⚠️ Pitfalls: 不理解内核层会导致对系统 API 行为的误解（如为什么 XPC 比 NSOperationQueue 跨进程更快）
💡 Best practice: 面试和架构设计时能用内核层视角解释 API 选择，体现深度理解
```

### iOS 进程调度（优先级类别、能耗效率）

```
iOS 基于 XNU 内核 BSD 层实现进程调度，核心机制：

┌──────────────────────────────────┐
│ QoS 优先级层级                     │
│ user-interactive > user-initiated  │
│ > utility > background > default   │
└──────────────────────────────────┘

每个 QoS 对应内核参数: CPU优先级, 带宽throttle, timer精度

DispatchQueue.global(qos: .userInteractive) // 触摸响应
DispatchQueue.global(qos: .userInitiated)   // 手势处理
DispatchQueue.global(qos: .utility)          // 网络请求
DispatchQueue.global(qos: .background)       // 预加载

// Swift Concurrency QoS 自动传播
Task { @MainActor in /* user-interactive */ }
// 内核层: QoS -> BSD policy -> Mach thread priority
```

⚠️ Pitfalls: background 队列做密集型计算会被内核降频甚至延迟数秒；将网络请求放在 user-interactive 阻塞触摸响应。
💡 Best practice: Instruments System Trace Energy Log 定位 QoS 设置不当；iOS 15+ 用 @MainActor 替代 manual QoS。


### iOS 内存管理（VM 压缩、jetsam、内存级别）

```
iOS 内存回收三级机制，从轻量到强制逐级升级：

┌──────────────────────────────────────────┐
│ Level 1: VM Compression (内核层)          │
│   将未访问页面压缩到内核内存池 (~3:1)      │
│   触发: WWDM (Write-Wire-Dirty-Min)      │
│   效果: 释放物理RAM但保持页面可访问         │
├──────────────────────────────────────────┤
│ Level 2: Jetsam (进程级)                  │
│   按 jetsam priority 杀进程释放内存        │
│   触发: 持续内存压力超过阈值                │
│   效果: 进程 crash，生成 JetsamEvent      │
├──────────────────────────────────────────┤
│ Level 3: Purgeable Memory                 │
│   CGImage/NSCache 等标记为 purgeable      │
│   触发: WWDM 扫描时自动释放                 │
│   效果: 内容丢失但进程存活，下次重新加载     │
└──────────────────────────────────────────┘

// Swift: 释放 purgeable 资源
let image = CGImage(source, options) // CGImageSourceCreateImage
image!.properties // CGImage 默认为 purgeable

// Objective-C: CGImage 释放
[[NSCache shared] setObject:nil forKey:key] // NSCache 自动 purge

// WWDM 触发条件 (内核参数)
// vm_compression=1 (iOS默认开启)
// WWDM=max(write_count, wire_count) + dirty_pages + min_threshold
```

⚠️ Pitfalls: CGImage 被 WWDM 压缩后再次访问会触发 decompression overhead；大图片直接用 CGImage 不做 downsampling 必被 jetsam 杀。
💡 Best practice: 图片处理用 imageIO + CGImageSourceCreateThumbnailAtIndex() 按需解码；关键进程通过 launchd Label 设置 JetsamPriority；用 Instruments Allocations + VM Tracker 定位压缩率异常。

### 缺页中断处理与按需分页

```
iOS 内核(XNU)按需分页机制 — 页面只在首次访问时加载:

┌─────────────────────────────────┐
│ 进程访问虚拟地址                 │
│   ↓                             │
│ Page Fault (trap to kernel)     │
│   ↓                             │
│ VM Map Lookup → 页面不存在?      │
│   ↓                             │
│ Object Type Check:              │
│   fileback → 从磁盘读(Journal)   │
│   copy-on-write → 复制父页面     │
│   zero-fill → 填零页面           │
│   compression → 解压缩           │
│   ↓                             │
│ 分配物理帧 → 更新PTE → 返回用户态 │
└─────────────────────────────────┘

// Swift: 触发 fileback page fault
let url = Bundle.main.url(forResource: "data", withExtension: "bin")!
let mapped = try MemoryMappedFile(url: url) // mmap(MAP_PRIVATE)
_ = mapped.bytes[0] // ← 触发 page fault，内核从APFS加载

// Objective-C: 预触摸页面避免首次延迟
vm_read_overmap(task, address, size, FALSE) // 预加载关键页面

// WWDM 压缩页面解压缩路径
// WWDM scan → 找到未访问页面 → LZ4压缩 → PTE标记compressed
// 再次访问 → decompression fault → LZ4解压缩 → 恢复正常PTE
```

⚠️ Pitfalls: 大文件 mmap 后顺序遍历触发大量 page fault 导致 jank；copy-on-write 页面在子进程写时触发额外延迟；iOS 17+ 对 Swift actor 隔离内存有专门的 zero-fill 优化路径。
💡 Best practice: 关键资源预热用 madvise(MADV_WILLNEED)；图片解码用 CGImageSourceCreateThumbnailAtIndex 避免全量 page fault；用 Instruments VM Tracker 定位 page fault 热点；iOS 15+ Private Remote Memory 减少 GPU 纹理 page fault。
### APFS 快照与写时复制 (Copy-on-Write)

APFS (Apple File System) 是 iOS 默认文件系统，核心特性是 Copy-on-Write (CoW) 和快照机制。APFS 使用 B-Tree 管理元数据和数据块，所有写操作都是 CoW — 原始块保持不变，新内容写入新块后再更新引用。

```swift
// APFS CoW 原理: 写时复制
// 1. 进程修改文件 → 内核分配新块 → 复制原始内容 → 写入新数据 → 更新 B-Tree 指向
// 2. 快照创建 → 记录当前 B-Tree root → 后续写操作通过 CoW 保持快照一致性
// 3. Time Machine / 系统更新利用快照实现原子回滚

// Swift: 模拟 CoW 行为 (Array/Slice)
var original = [1, 2, 3, 4, 5]
let snapshot = original // Swift Array 是 CoW — 此时共享底层存储
var modified = original // 仍然共享
modified[0] = 99 // ← 触发 copy! 此时才分配新内存并复制

// APFS 层: launchd snapshot (iOS 15+)
// launchctl launch --hispid <label> // 带硬件快照启动
// 进程fork时所有内存页面标记为 CoW，子进程写时内核复制页面
```

APFS 快照在 iOS 系统更新中至关重要 — 更新前创建快照，安装失败时原子回滚到快照状态，确保设备不会变砖。CoW 保证了快照创建是 O(1) 操作，不阻塞用户。

⚠️ Pitfalls: APFS 卷空间共享但每个卷有硬限制；CoW 导致写放大 — 频繁小写场景下空间消耗是原始数据的 2-3 倍；sandbox 内无法直接创建 APFS 快照。
💡 Best practice: 大文件写入先分配空间 (ftruncate) 再写内容，减少 CoW 碎片；系统级用 launchd snapshot 做 crash recovery；用 `df -a` 和 `du -sh` 对比查看 CoW 实际占用。

### POSIX 信号与 iOS 中的处理

POSIX 信号是 Unix 进程间异步通知机制。iOS 基于 XNU 内核，完整支持 BSD 信号模型，但 App 沙箱限制了部分信号的使用场景。常见的信号包括 SIGKILL(9)、SIGTERM(15)、SIGSEGV(11)、SIGBUS(10) 等。iOS 中信号处理有独特限制 — App 不能安装 SIGKILL/SIGSTOP handler，系统保留这些用于强制终止。

```swift
import Foundation

// Swift: 安装信号 handler
signal(SIGUSR1) { sig in
    print("收到 SIGUSR1, 信号编号: \(sig)")
    // 注意: signal handler 中只能调用 async-signal-safe 函数
}

// Objective-C: 使用 sigaction (推荐方式)
var action = sigaction()
action.sa_handler = SIG_IGN // 忽略 SIGCHLD
sigaction(SIGCHLD, &action, nil)

// Swift: 现代方式 — pthread_sigmask 阻塞信号
var mask = sigset_t()
sigemptyset(&mask)
sigaddmember(&mask, SIGUSR1)
pthread_sigmask(.block, &mask, nil) // 阻塞当前线程的 SIGUSR1

// 实际场景: Swift 运行时用 SIGUSR1/SIGUSR2 做 deadlock 检测
// Dispatch 用 SIGKILL 做强制取消 (dispatch_cancel)
```

⚠️ Pitfalls: signal handler 中调用 NSLog/print 会导致 deadlock（logger 内部用锁）；多线程环境下 signal 可能投递到任意线程；iOS 17+ Swift concurrency 中信号会中断 actor 隔离边界。
💡 Best practice: 生产代码用 sigaction 而非 signal；handler 内仅设置 volatile/global 标志位，实际处理在主循环；用 pthread_sigmask + sigwait 做确定性信号处理；crash 分析时关注 SIGSEGV/SIGBUS/SIGTRAP 定位内存安全问题。

### Mach 异常与信号转换

XNU 内核中，硬件异常首先由 Mach 层捕获为 Mach exception（如 EXC_CRASH, EXC_BAD_ACCESS, EXC_BREAKPOINT），再由 BSD 层转换为 POSIX signal（如 SIGSEGV, SIGBUS, SIGTRAP）。这一两层模型是理解 iOS crash 机制的基础 — Mach exception 是内核级事件，signal 是用户态通知。

```swift
// Mach exception → BSD signal 转换关系
// EXC_BAD_ACCESS (KERN_PROTECTION_FAILURE) → SIGSEGV (11)
// EXC_BAD_ACCESS (KERN_INVALID_ADDRESS)   → SIGSEGV (11)
// EXC_CRASH (EXC_i386_PANIC)              → SIGILL (4)
// EXC_BREAKPOINT (EXC_ARM_BREAKPOINT)     → SIGTRAP (5)
// EXC_RESOURCE                             → SIGKILL (9) — 系统资源耗尽

// Swift: 模拟 Mach exception handler 注册 (仅 privileged 进程)
// task_set_exception_ports(task, EXC_MASK_BAD_ACCESS, handler, ...)
// BSD layer: Exception Catcher → translate_to_signal() → kill(curproc, SIGSEGV)

// 实际调试: ThreadSanitizer 将数据竞争转为 EXC_BREAKPOINT
// guard malloc 将越址写转为 EXC_BAD_ACCESS → SIGSEGV
```

Mach exception handler 优先级高于 signal handler — GLibc 的 guard pages、ThreadSanitizer、Playground 的 guard crash 都通过安装 Mach-level handler 实现，绕过了 BSD signal 层。iOS 17+ Swift 6 的 actor isolation violation 也通过 EXC_CRASH 路径触发 crash report。

⚠️ Pitfalls: Mach exception handler 安装需要 privileged task port；iOS App 沙箱禁止安装自定义 exception handler；EXC_RESOURCE crash 无 backtrace（系统直接 kill）。
💡 Best practice: crash 分析先看 crash report 中的 Exception Type（Mach 层）再定位具体原因；用 guard malloc / Address Sanitizer 定位内存越界；Playground crash 通常是 guard crash — 检查未初始化变量和 dangling pointer。
### CPU 亲和性与线程优先级

iOS 基于 XNU 内核的 BSD scheduler 负责线程调度。每个线程有调度策略（SCHED_OTHER/SCHED_FIFO/SCHED_RR）、优先级（-55~55）、CPU 亲和性_mask_等属性。iOS App 沙箱限制了部分调度 API — 无法设置 CPU 亲和性或实时调度策略，但可通过 QoS 和 pthread priority 间接影响调度顺序。

```swift
import Foundation

// Swift: QoS 域 — 现代 iOS 推荐方式
DispatchQueue.global(qos: .userInteractive).async {
    // 最高优先级 — UI 响应、触摸处理
}
DispatchQueue.global(qos: .background).async {
    // 最低优先级 — 日志、缓存清理
}

// pthread 层设置优先级 (iOS 沙箱内有效)
pthread_setprio_np(pthread_self(), 31) // 范围: -55 ~ 55

// 查看当前 QoS 类
Thread.current.qualityOfService // .userInteractive

// Swift 5.5+ actor 模型隐式绑定 QoS — @MainActor 绑定 .main
@MainActor func updateUI() { /* runs on main thread QoS */ }
```

XNU 调度器将 QoS 域映射为内核优先级带 — `.userInteractive` → priority 55, `.userInitiated` → 45, `.default` → 31, `.utility` → 15, `.background` → -55。iOS 17+ 引入了 adaptive QoS — 系统根据电池、温度、负载动态调整实际调度权重。

⚠️ Pitfalls: pthread_setprio_np 仅在同一 QoS 域内有效，跨域设置被内核忽略；后台任务设置高优先级会被 watchdog kill；iOS 15+ 后台扩展有独立的 QoS 域。
💡 Best practice: 始终使用 DispatchQueue QoS 而非 pthread API；长耗时任务用 `.background` QoS + Operation dependencies 做优先级拓扑；用 Instruments Energy Log 验证 QoS 设置是否生效。

### 沙盒（com.apple.security.*）权限

iOS App 运行在沙箱环境中，通过 entitlements 机制限制系统资源访问。每个 App 有默认的沙箱权限集，扩展权限需通过 `com.apple.security.*` entitlements 声明并在 App Store 审核时评估。

```swift
// Entitlements.plist 关键权限
// com.apple.security.app-sandbox — 启用沙箱（macOS App 必需）
// com.apple.security.files.user-selected.read-only — 用户选择文件只读
// com.apple.security.network.server — 允许创建监听 socket
// com.apple.security.network.client — 出站网络连接（默认允许）
// com.apple.security.print — 打印服务访问

// Swift: 运行时检查沙箱边界
import Foundation

// 文件系统沙箱 — App 只能访问自己的容器目录
let appDir = FileManager.default.containerURL(forSecurityApplicationGroupIdentifier: nil)
// → file:///var/mobile/Containers/Data/Application/<UUID>/

// 尝试访问沙箱外路径 — macOS 上抛出 NSCocoaErrorDomain code 513 (fileReadNoPermission)
// iOS 上通过方法级沙箱限制（如不能列 /var 目录内容）
do {
    let contents = try FileManager.default.contentsOfDirectory(atPath: "/var")
    // macOS: 沙箱禁止；iOS: API 层限制
} catch {
    print("沙箱拦截: \(error)")
}
```

XNU 内核通过 launchd 在 App 启动时注入沙箱配置文件（sandbox profile），每个 `com.apple.security.*` entitlement 对应一组内核级规则。iOS 17+ 引入了 stricter sandbox — 默认禁止 dyld 加载未签名库、限制 JIT 编译、收紧 Mach 端口访问。

⚠️ Pitfalls: 沙箱权限在打包时固化，运行时无法动态扩展；entitlements 缺失导致的功能降级通常是 silent failure（如通知收不到但无 crash）；TestFlight 和 App Store 可能有不同沙箱策略。
💡 Best practice: 最小权限原则 — 仅声明必要的 entitlement；用 `sandbox-exec -p` 在 macOS 上模拟沙箱调试；iOS 上通过 Xcode Scheme → Options → App Sandbox 验证；敏感权限（如网络服务）需提供审核描述。

### 系统完整性保护（SIP）与代码签名

iOS 上 SIP（System Integrity Protection）以"方法级沙箱"形式存在 — 系统框架 API 对未签名进程返回 NIL/空结果而非 crash。所有上架 App 必须通过 Apple 的代码签名验证，签名包含 entitlements、团队标识和加密哈希。

```swift
// 检查当前进程签名状态
import Foundation

// 代码签名信息（仅 debug 用，生产环境无意义）
let codeSignIdentity = ProcessInfo.processInfo.environment["CODESIGNING_IDENTITY"]
// App Store: Apple Production; Dev: Apple Development

// 运行时验证 dylib 签名（iOS 17+ stricter policy）
do {
    // dyld 自动验证 — 加载未签名 dylib 直接 crash
    // dyld: Library not loaded: @rpath/unsigned.framework/unsigned
    // dyld: code signature invalid in 'unsigned.framework'
} catch {
    // SIP 层拦截 — 未签名代码无法加载
}

// Swift 5.9+: @_unavailable(inline, message: "") 标记 SIP 保护的 API
@_unavailable(internal) func privilegedOperation() { /* system-only */ }
```

```
┌─ dyld 签名验证流程 ──────────────────┐
│ 1. 加载 dylib → 读取 __LINKEDIT 段   │
│ 2. 提取 CodeSignature → SHA-256 验证  │
│ 3. 检查 entitlements → launchd 授权   │
│ 4. 通过 → map 内存；失败 → kill(2)   │
└──────────────────────────────────────┘
```

⚠️ Pitfalls: Jailbreak 环境 SIP 被禁用但代码签名仍强制；bitcode 重签名会改变 entitlements hash；跨域加载.framework 需 `@rpath` + `LC_CODE_SIGNATURE`；iOS 15+ 禁止未签名的 Swift dynamic linker 支持库。
💡 Best practice: 发布前用 `codesign --verify --deep --strict --entitlements :- <app>.app` 验证签名链；用 `codesign --display --entitlements :- -r -- <app>.app` 审计 entitlement；敏感操作通过系统 API 间接完成而非尝试绕过签名。

### Mach 端口与进程间通信

XNU 内核中 Mach 端口是 IPC 的基础设施 — 每个端口是一个内核对象，通过 rights（send/receive/set）控制访问权限。iOS 上所有高级 IPC（XPC、Distributed Notifications、NSPort）最终都建立在 Mach 端口之上。

```swift
import Foundation

// Swift 层通过 NSPort 间接使用 Mach 端口
let port = NSPort()
let listener = NSPortNameServer.systemDefault().portForName("com.myapp.channel")

// XPC — Apple 推荐的现代 IPC 方式（底层仍是 Mach）
// XPC 服务通过 XPCServices target 创建， launchd 自动管理端口绑定
let connection = NSXPCConnection.maintenanceConnection()
connection.remoteObjectProxy!.getServices(for: "com.apple.locationd") { services in
    // 通过 Mach 端口与安全端口向定位服务发送请求
}

// Mach 端口 rights 模型
// SEND right: 可向端口发送消息（可复制）
// RECEIVE right: 可从端口接收消息（唯一，不可复制）
// SEND_ONCE right: 发送一次后失效（用于端口迁移）
```

```
┌─ Mach 端口 IPC 流程 ─────────────────┐
│                                      │
│  Process A              Process B    │
│  ┌──────┐              ┌──────┐     │
│  │ Send │───msg───>───>│Recv  │     │
│  │ right│              │right │     │
│  └──────┘              └──────┘     │
│                                      │
│  launchd 分配端口对 → 通过 entitle   │
│  ments 授权跨进程端口传输             │
└──────────────────────────────────────┘
```

⚠️ Pitfalls: Mach 端口不能在进程间自由传递 — 需 launchd 授权和 entitlement；端口耗尽（每个进程默认 ~10K）导致 launchd crash；XPC 消息过大触发内核 panic；iOS 17+ stricter policy 下未声明 entitle 的端口传输直接被内核拒绝。
💡 Best practice: 优先使用 XPC 而非原始 Mach API；XPC 消息保持 <1MB；用 `xctrace` 和 `dtrace` 追踪端口活动；跨进程通信始终做超时处理和错误恢复；敏感端口通过 entitle 最小化授权。

### iOS 沙盒容器结构

每个 iOS 应用被限制在一个独立的沙盒目录中，由 launchd 在首次安装时创建。沙盒通过 BSD extended attributes、sandbox profiles (.sb) 和 entitle ments 三层机制实现隔离，防止应用访问自身容器以外的资源。

```swift
import Foundation

// 沙盒目录结构 — 每个路径都是硬编码约定的
let paths = [
    "Documents": FileManager.default.urls(for: .documentDirectory, in: .userDomainMask),
    "Library": FileManager.default.urls(for: .libraryDirectory, in: .userDomainMask),
    "Caches": FileManager.default.urls(for: .cachesDirectory, in: .userDomainMask),
    "tmp": FileManager.default.temporaryDirectory,
]

// 容器根路径 — 所有文件操作被限制在此目录下
let containerURL = FileManager.default.containerURL(forSecurityApplicationGroupIdentifier: nil)!
// 跨组共享: forSecurityApplicationGroupIdentifier("group.com.xxx")

// 尝试访问沙盒外路径 — 被内核 sandbox profile 拦截
let outsidePath = URL(fileURLWithPath: "/var/mobile/Media")
_ = try? FileManager.default.contentsOfDirectory(at: outsidePath) // 抛出 NSFileReadNoPermissionError
```

```
┌─ 沙盒容器布局 ───────────────────┐
│ /var/mobile/Containers/          │
│   Application/<UUID>/            │
│   ├── App/.                     │  ← 主程序 bundle
│   ├── Data/                     │  ← 运行时数据
│   │   ├── Documents/            │  ← iCloud 同步, iTunes 备份
│   │   ├── Library/              │  ← Preferences, Caches, ApplicationSupport
│   │   └── tmp/                  │  ← 重启清空, 不备份
│   └── Temporary/                │  ← launchd 临时目录
│                                 │
│ 隔离层: BSD extattr + seccomp   │
│          + sandbox.sb profiles   │
└─────────────────────────────────┘
```

⚠️ Pitfalls: Documents 目录参与 iCloud 和 iTunes 备份 — 大文件放这里会耗尽用户云空间；Library/Caches 不参与备份但无大小限制；iOS 17+ 对临时文件加密要求更严格；跨沙盒访问必须通过 App Group、File Coordinator 或 Share Extension；越狱环境下沙盒完全失效但签名验证仍有效。
💡 Best practice: 用户上传内容放 Documents（需备份）；网络缓存放 Caches（不备份）；敏感数据用 Keychain（跨沙盒但需 entitle）；大媒体文件用 Files App 集成；定期清理 Caches 目录；用 `NSFileProtectionCompleteUntilFirstUserAuthentication` 标记敏感文件加密。
