# iOS 逆向与安全详解

---

## 一、iOS 安全机制概述

### 1.1 iOS 安全架构

```
iOS 安全是多层防御体系：

  ┌─────────────────────────────────────────────────────────┐
  │                      App 层                              │
  │  代码混淆、SSL Pinning、越狱检测、数据加密                 │
  ├─────────────────────────────────────────────────────────┤
  │                    系统框架层                             │
  │  ATS、Keychain、Data Protection、权限系统                 │
  ├─────────────────────────────────────────────────────────┤
  │                     内核层                               │
  │  沙盒（Sandbox）、代码签名、ASLR、DEP/NX                  │
  ├─────────────────────────────────────────────────────────┤
  │                    硬件层                                │
  │  Secure Enclave、Touch ID/Face ID、硬件加密               │
  └─────────────────────────────────────────────────────────┘

主要威胁：
  - 越狱设备上的恶意代码注入
  - 中间人攻击（抓包、篡改数据）
  - 逆向工程（分析业务逻辑、提取密钥）
  - 数据窃取（Keychain、本地文件）
  - 重打包（修改 App 后重新签名发布）
```

### 2.1 常用逆向工具

```
静态分析工具：

## 三、越狱检测

### 3.1 越狱的原理

```
越狱（Jailbreak）：
  利用 iOS 内核漏洞，绕过苹果的安全限制，获得 root 权限。

越狱后的变化：
  - 可以安装未经 App Store 审核的应用（Cydia）
  - 可以访问文件系统任意位置
  - 可以注入代码到其他进程（MobileSubstrate/Substitute）
  - 系统签名验证被绕过
  - 沙盒限制被绕过

越狱的危险：
  - 恶意应用可以访问其他 App 的数据
  - SSL Pinning 可以被 SSLKillSwitch2 绕过
  - 应用的内存可以被随意读写
  - 敏感数据（Keychain、本地文件）可能泄露

常见越狱工具：
  - checkra1n（利用硬件漏洞，支持 A5-A11）
  - unc0ver（软件漏洞，iOS 11-14）
  - Dopamine（iOS 15-16）
  - Palera1n（iOS 15-17）
```

### 3.2 越狱检测方法

```swift
import Foundation
import UIKit

class JailbreakDetector {
    
    static func isJailbroken() -> Bool {
        return checkJailbreakFiles()
            || checkWritePermission()
            || checkSuspiciousApps()
            || checkSymlinks()
            || checkDynamicLibraries()
            || checkFork()
    }
    
    // 1. 检查越狱相关文件
    static func checkJailbreakFiles() -> Bool {
        let jailbreakPaths = [
            "/Applications/Cydia.app",
            "/Applications/blackra1n.app",
            "/Applications/FakeCarrier.app",
            "/Applications/Icy.app",
            "/Applications/IntelliScreen.app",
            "/Applications/MxTube.app",
            "/Applications/RockApp.app",
            "/Applications/SBSettings.app",
            "/Applications/WinterBoard.app",
            "/Library/MobileSubstrate/MobileSubstrate.dylib",
            "/Library/MobileSubstrate/DynamicLibraries/Veency.plist",
            "/Library/MobileSubstrate/DynamicLibraries/LiveClock.plist",
            "/private/var/lib/apt",
            "/private/var/lib/cydia",
            "/private/var/mobile/Library/SBSettings/Themes",
            "/private/var/stash",
            "/private/var/tmp/cydia.log",
            "/System/Library/LaunchDaemons/com.ikey.bbot.plist",
            "/System/Library/LaunchDaemons/com.saurik.Cydia.Startup.plist",
            "/usr/bin/sshd",
            "/usr/libexec/sftp-server",
            "/usr/sbin/sshd",
            "/etc/apt",
            "/bin/bash",
            "/bin/sh"
        ]
        
        for path in jailbreakPaths {
            if FileManager.default.fileExists(atPath: path) {
                return true
            }
        }
        return false
    }
    
    // 2. 检查沙盒外的写权限
    static func checkWritePermission() -> Bool {
        let testPath = "/private/jailbreak_test_\(Int.random(in: 0...9999))"
        do {
            try "jailbreak_test".write(toFile: testPath, atomically: true, encoding: .utf8)
            try FileManager.default.removeItem(atPath: testPath)
            return true  // 能写入沙盒外 = 越狱
        } catch {
            return false
        }
    }
    
    // 3. 检查可疑 App（通过 URL Scheme）
    static func checkSuspiciousApps() -> Bool {
        let suspiciousSchemes = [
            "cydia://",
            "sileo://",
            "zbra://",
            "filza://"
        ]
        
        for scheme in suspiciousSchemes {
            if let url = URL(string: scheme),
               UIApplication.shared.canOpenURL(url) {
                return true
            }
        }
        return false
    }
    
    // 4. 检查符号链接（越狱会创建一些符号链接）
    static func checkSymlinks() -> Bool {
        let symlinks = [
            "/var/lib/undecimus/apt",
            "/Applications",
            "/Library/Ringtones",
            "/Library/Wallpaper",
            "/usr/arm-apple-darwin9",
            "/usr/include",
            "/usr/libexec",
            "/usr/share"
        ]
        
        for path in symlinks {
            if let attributes = try? FileManager.default.attributesOfItem(atPath: path),
               let type = attributes[.type] as? FileAttributeType,
               type == .typeSymbolicLink {
                return true
            }
        }
        return false
    }
    
    // 5. 检查注入的动态库
    static func checkDynamicLibraries() -> Bool {
        let suspiciousLibs = [
            "MobileSubstrate",
            "CydiaSubstrate",
            "Substitute",
            "libhooker",
            "SSLKillSwitch"
        ]
        
        let imageCount = _dyld_image_count()
        for i in 0..<imageCount {
            if let name = _dyld_get_image_name(i) {
                let imageName = String(cString: name)
                for lib in suspiciousLibs {
                    if imageName.contains(lib) {
                        return true
                    }
                }
            }
        }
        return false
    }
    
    // 6. 检查 fork（越狱设备上 fork 会成功，沙盒中 fork 会失败）
    static func checkFork() -> Bool {
        let pid = fork()
        if pid >= 0 {
            if pid > 0 { kill(pid, SIGTERM) }
            return true  // fork 成功 = 越狱
        }
        return false
    }
}
```

### 3.3 越狱检测的对抗

```
攻击者绕过越狱检测的方式：
  1. 使用 Frida Hook 检测函数，直接返回 false
  2. 删除或重命名越狱文件
  3. Hook FileManager 的文件检查方法
  4. 使用 Liberty Lite / A-Bypass 等工具批量绕过

防御加强措施：
  1. 多种检测方法组合，不要只依赖一种
  2. 检测代码混淆，让攻击者难以定位
  3. 不要在固定位置调用检测，分散在各处
  4. 服务端配合（可疑行为检测、设备指纹）
  5. 关键检测逻辑放在 C 层，避免 ObjC 方法被 Hook
  6. 使用 syscall 直接调用系统调用，绕过 Hook

现实考量：
  - 越狱检测永远无法做到 100% 可靠
  - 越狱用户占比很低（< 1%）
  - 过于激进的检测可能误判正常设备
  - 重要是提高攻击成本，而非完全阻止
```

## 四、SSL Pinning 深度

### 4.1 SSL Pinning 实现方式

```swift
// 方式一：证书绑定（Certificate Pinning）
// 将完整的 DER 格式证书打包进 App

class CertificatePinner: NSObject, URLSessionDelegate {
    
    // 从 Bundle 加载本地证书
    private lazy var localCertificates: [SecCertificate] = {
        guard let certURL = Bundle.main.url(forResource: "server", withExtension: "cer"),
              let certData = try? Data(contentsOf: certURL),
              let cert = SecCertificateCreateWithData(nil, certData as CFData) else {
            return []
        }
        return [cert]
    }()
    
    func urlSession(_ session: URLSession,
                    didReceive challenge: URLAuthenticationChallenge,
                    completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) {
        
        guard challenge.protectionSpace.authenticationMethod == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // 获取服务器证书链
        let serverCertificates = (0..<SecTrustGetCertificateCount(serverTrust))
            .compactMap { SecTrustGetCertificateAtIndex(serverTrust, $0) }
        
        // 比较证书数据
        let serverCertData = serverCertificates.map { SecCertificateCopyData($0) as Data }
        let localCertData = localCertificates.map { SecCertificateCopyData($0) as Data }
        
        let isPinned = serverCertData.contains { serverData in
            localCertData.contains(serverData)
        }
        
        if isPinned {
            completionHandler(.useCredential, URLCredential(trust: serverTrust))
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}
```

```swift
// 方式二：公钥绑定（Public Key Pinning）- 推荐
// 只存储公钥的 SHA-256 哈希值

class PublicKeyPinner: NSObject, URLSessionDelegate {
    
    // 预计算的公钥哈希值（base64 编码的 SHA-256）
    private let pinnedPublicKeyHashes: Set<String> = [
        "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=",  // 主证书
        "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB="   // 备用证书
    ]
    
    func urlSession(_ session: URLSession,
                    didReceive challenge: URLAuthenticationChallenge,
                    completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) {
        
        guard challenge.protectionSpace.authenticationMethod == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust,
              let serverCertificate = SecTrustGetCertificateAtIndex(serverTrust, 0) else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // 提取公钥
        guard let publicKey = SecCertificateCopyKey(serverCertificate),
              let publicKeyData = SecKeyCopyExternalRepresentation(publicKey, nil) as Data? else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }
        
        // 计算 SHA-256 哈希
        var hash = Data(count: Int(CC_SHA256_DIGEST_LENGTH))
        let headerData = Data([0x30, 0x59, 0x30, 0x13, 0x06, 0x07, 0x2A, 0x86,
                               0x48, 0xCE, 0x3D, 0x02, 0x01, 0x06, 0x08, 0x2A,
                               0x86, 0x48, 0xCE, 0x3D, 0x03, 0x01, 0x07, 0x03,
                               0x42, 0x00])  // EC P-256 公钥头
        let combinedData = headerData + publicKeyData
        _ = combinedData.withUnsafeBytes {
            CC_SHA256($0.baseAddress, CC_LONG(combinedData.count), &hash)
        }
        let hashString = hash.base64EncodedString()
        
        if pinnedPublicKeyHashes.contains(hashString) {
            completionHandler(.useCredential, URLCredential(trust: serverTrust))
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}
```

### SecTrust 深度评估机制

SecTrust 是 Apple 提供的证书信任评估 API，底层调用 Secure Transport（TLS 层）和 Security.framework 的证书验证引擎。iOS 17+ 引入了 `SecTrustEvaluateAsync` 替代已废弃的 `SecTrustEvaluate`，支持异步评估避免阻塞主线程。

**自定义评估流程** — 完全控制信任链验证逻辑：

```swift
func evaluateTrust(serverTrust: SecTrust) async -> Bool {
    // 1. 设置自定义评估策略（默认为 .default，仅验证签名链）
    SecTrustSetPolicies(serverTrust, SecPolicyCreateSSL(true, "api.example.com" as CFString))
    
    // 2. 可选：替换系统锚点为自定义 CA（仅信任自己的 CA）
    guard let caCertURL = Bundle.main.url(forResource: "MyCA", withExtension: "cer"),
          let caData = try? Data(contentsOf: caCertURL),
          let caCert = SecCertificateCreateWithData(nil, caData as CFData) else {
        return false
    }
    SecTrustSetAnchorCertificates(serverTrust, [caCert] as CFArray)
    SecTrustSetAnchorCertificatesOnly(serverTrust, true) // 禁用系统根证书
    
    // 3. 异步评估（iOS 12.5+ / macOS 10.15+）
    var error: CFError?
    let result = await withUnsafeContinuation { continuation in
        SecTrustEvaluateAsync(serverTrust) { _, status in
            continuation.resume(returning: status)
        }
    }
    
    guard result == errSecSuccess else { return false }
    
    // 4. 额外验证：检查证书是否被吊销（CRL / OCSP）
    // 注意：iOS 默认不做 OCSP/CRL 检查，需手动实现
    return true
}
```

**坑**：`SecTrustSetAnchorCertificatesOnly(true)` 会完全禁用系统信任库，若自定义 CA 过期将导致所有请求失败；`SecPolicyCreateSSL` 的 hostname 参数用于 CN/SAN 匹配，不传或不正确会导致评估通过但后续 TLS handshake 失败；iOS 17 之后 `SecTrustEvaluate` 完全被标记为 deprecated；OCSP stapling 需服务器端支持，iOS 客户端无法强制；评估通过仅表示证书链有效，不代表公钥 hash 匹配（需额外做 pinning 校验）。

**最佳实践**：生产环境组合使用 SecTrust 评估 + 公钥 pinning；始终设置备用 hash（back-up pin）防止证书续期导致服务中断；评估前先用 `SecTrustCopyCertificateChain` 打印完整链条用于 debug；iOS 17+ 结合 `TLSConfiguration` API（Experimental）做更细粒度的 TLS 参数控制。


### 4.3 绕过 SSL Pinning 的方式（防御参考）

```
1. SSLKillSwitch2（越狱设备）
   - 全局禁用 SSL Pinning
   - 通过 MobileSubstrate 注入，Hook 系统 SSL 验证函数
   
2. Frida 脚本
   - Hook URLSession 的 didReceive challenge 回调
   - 直接调用 completionHandler(.useCredential, ...)

3. 修改二进制
   - 找到 Pinning 验证代码
   - 修改跳转指令，跳过验证

  1. 代码混淆（见下一章）
  2. 越狱检测（见上一章）
  3. 运行时完整性检测（检测是否有代码被 Hook）
  4. 多重 Pinning（多个备用公钥）
  5. 证书透明度检测
  6. 服务端异常行为检测
```

## 五、代码混淆

### 5.1 混淆的目标

```
代码混淆的目的：
  1. 增加逆向分析的难度
  2. 保护核心算法和业务逻辑
  3. 防止竞争对手抄袭
  4. 增加破解的时间成本

混淆不能做到的：
  - 完全阻止逆向（只能增加难度）
  - 保护 ObjC 方法名（运行时需要方法名）
  - 防止动态分析（Frida 等工具可以绕过）

混淆的层次：
  1. 符号混淆：重命名类名、方法名、变量名
  2. 控制流混淆：让代码流程难以理解
  3. 字符串加密：加密硬编码的字符串
  4. 反调试：检测调试器并做出反应
  5. 代码虚拟化：将代码转换为虚拟机指令
```

### 5.2 ObjC 符号混淆

```objc
// ObjC 的方法名无法混淆（动态派发需要方法名）
// 但可以混淆类名、实例变量名

// 使用宏定义混淆类名
#define UserManager          MT_a1b2c3
#define NetworkClient        MT_d4e5f6
#define CryptoHelper         MT_g7h8i9

// 或使用脚本批量重命名
// confuse.sh 工具可以自动替换类名和方法名
// 但要注意不能混淆被 KVO、KVC、NSCoding 等依赖名称的方法

// 字符串加密（避免关键字符串被 strings 工具提取）
// 原始代码：
NSString *apiKey = @"sk-1234567890abcdef";

// 混淆后（运行时解密）：
NSString *apiKey = decryptString((unsigned char[]){
    0x73 ^ 0xFF, 0x6B ^ 0xFF, 0x2D ^ 0xFF, ...
});

// 宏实现字符串加密
#define OBFSTR(s) ({                                        \
    static const volatile char _obf[] = {                  \
        OBFUSCATE_CHARS(s)                                  \
    };                                                      \
    [NSString stringWithUTF8String:deobfuscate(_obf)];     \
})
```

### 5.3 Swift 混淆工具

```
SwiftShield：
  - 自动重命名 Swift 中的类名、方法名、属性名
  - 生成随机名称（如 a1b2c3d4）
  - 开源工具：https://github.com/rockbruno/swiftshield

使用步骤：
  1. 安装：brew install swiftshield
  2. 运行：swiftshield obfuscate --project-root . --scheme MyApp
  3. 生成混淆后的源代码

注意事项：
  - @objc 标记的方法不能混淆
  - 被 Storyboard/Xib 引用的类名不能混淆
  - 被 NSCoding 使用的类名不能混淆（归档兼容性）
  - 公开 API（Framework 的 public 接口）不能混淆

LLVM 混淆（Hikari / Obfuscator-LLVM）：
  - 在编译层面做混淆
  - 控制流扁平化（Control Flow Flattening）
  - 虚假控制流（Bogus Control Flow）
  - 指令替换（Instruction Substitution）
  - 字符串加密
  - 需要修改 Xcode 的编译器，配置复杂
```

### 6.1 Keychain 使用

```swift
import Security

class KeychainManager {
    
    static func save(key: String, data: Data, accessibility: CFString = kSecAttrAccessibleWhenUnlockedThisDeviceOnly) -> Bool {
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrAccount as String: key,
            kSecValueData as String: data,
            kSecAttrAccessible as String: accessibility
        ]
        
        // 先删除旧值
        SecItemDelete(query as CFDictionary)
        
        // 添加新值
        let status = SecItemAdd(query as CFDictionary, nil)
        return status == errSecSuccess
    }

        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrAccount as String: key,
            kSecReturnData as String: true,
            kSecMatchLimit as String: kSecMatchLimitOne
        ]
        
        var result: AnyObject?
        let status = SecItemCopyMatching(query as CFDictionary, &result)
        
        if status == errSecSuccess {
            return result as? Data
        }
        return nil
    }
    
    static func delete(key: String) -> Bool {
        let query: [String: Any] = [
            kSecClass as String: kSecClassGenericPassword,
            kSecAttrAccount as String: key
        ]
        let status = SecItemDelete(query as CFDictionary)
        return status == errSecSuccess
    }
}

### 7.1 请求签名

```swift
// 防止请求被篡改和重放攻击

class RequestSigner {
    private let secretKey: String
    
    init(secretKey: String) {
        self.secretKey = secretKey
    }
    
    func sign(request: URLRequest) -> URLRequest {
        var signedRequest = request
        
        let timestamp = String(Int(Date().timeIntervalSince1970))
        let nonce = UUID().uuidString.replacingOccurrences(of: "-", with: "")
        
        // 构建签名字符串
        var params: [String: String] = [
            "timestamp": timestamp,
            "nonce": nonce
        ]
        
        // 加入 body 参数（如果有）
        if let body = request.httpBody,
           let bodyParams = try? JSONSerialization.jsonObject(with: body) as? [String: String] {
            params.merge(bodyParams) { _, new in new }
        }
        
        // 按字母序排列参数
        let sortedParams = params.sorted { $0.key < $1.key }
        let paramString = sortedParams.map { "\($0.key)=\($0.value)" }.joined(separator: "&")
        
        // 计算 HMAC-SHA256 签名
        let signatureString = "\(request.httpMethod ?? "GET")\n\(request.url?.path ?? "")\n\(paramString)\n\(secretKey)"
        let signature = hmacSHA256(message: signatureString, key: secretKey)
        
        // 添加签名到请求头
        signedRequest.setValue(timestamp, forHTTPHeaderField: "X-Timestamp")
        signedRequest.setValue(nonce, forHTTPHeaderField: "X-Nonce")
        signedRequest.setValue(signature, forHTTPHeaderField: "X-Signature")
        
        return signedRequest
    }
    
    private func hmacSHA256(message: String, key: String) -> String {
        let keyData = Data(key.utf8)
        let messageData = Data(message.utf8)
        let symmetricKey = SymmetricKey(data: SHA256.hash(data: keyData))
        let mac = HMAC<SHA256>.authenticationCode(for: messageData, using: symmetricKey)
        return Data(mac).base64EncodedString()
    }
}

## 九、常见面试题

### Q1：什么是越狱？越狱检测有哪些方法？

```
越狱是利用 iOS 内核漏洞获取 root 权限，绕过苹果的安全限制。

越狱检测方法：
  1. 检查越狱文件：/Applications/Cydia.app、/bin/bash 等
  2. 检查沙盒外写权限：尝试写入 /private/ 目录
  3. 检查 URL Scheme：cydia://、sileo:// 等
  4. 检查注入的动态库：MobileSubstrate、Substitute 等
  5. 检查符号链接：越狱会创建一些特殊的符号链接
  6. 检查 fork：越狱设备上 fork() 会成功
  7. 检查环境变量：越狱设备可能有特殊环境变量

注意：
  - 检测无法 100% 可靠，攻击者可以绕过
  - 多种方法组合，提高攻击成本
  - 检测代码需要混淆，防止被 Hook
```

### Q2：SSL Pinning 是什么？有哪两种实现方式？

```
SSL Pinning（证书绑定）是在客户端硬编码服务器证书信息，
在 TLS 握手时验证服务器证书是否与预期一致，
防止中间人攻击（如 Charles 抓包）。

两种方式：
  1. 证书绑定（Certificate Pinning）
     将完整的服务器证书打包进 App
     证书过期时必须更新 App
  
  2. 公钥绑定（Public Key Pinning）- 推荐
     只存储证书中公钥的 SHA-256 哈希值
     证书更新后只要公钥不变，App 无需更新
     更灵活，生命周期更长

绕过方式：
  - SSLKillSwitch2（越狱设备）
  - Frida Hook 回调函数

防御：
  - 结合越狱检测
  - 代码混淆
  - 多重 Pinning（备用公钥）
```

### Q3：如何防止 App 被逆向工程？

```
防御措施（分层次）：

代码层面：
  - 代码混淆（类名、方法名重命名）
  - 字符串加密（关键字符串运行时解密）
  - 控制流混淆（让反汇编代码难以理解）
  - 关键逻辑用 C/C++ 实现（比 ObjC 更难逆向）

运行时防御：
  - 反调试（ptrace、sysctl 检测调试器）
  - 越狱检测
  - 注入检测（检查可疑动态库）
  - 完整性检测（检查 App 是否被修改）

网络层面：
  - SSL Pinning（防止抓包）
  - 请求签名（防止重放攻击）
  - 敏感数据服务端处理（核心逻辑不在客户端）

数据层面：
  - Keychain 存储敏感数据
  - 本地数据加密
  - 不硬编码密钥

现实考量：
  - 逆向防御是提高成本，不是绝对安全
  - 核心业务逻辑和数据尽量放服务端
  - 定期更新防御策略，对抗新的攻击手段
```

### Q4：Keychain 和 UserDefaults 的安全性对比？

```
UserDefaults：
  - 以 plist 文件存储在沙盒中
  - 未加密（明文存储）
  - 越狱设备可以直接读取
  - 备份到 iTunes/iCloud（可能泄露）
  - 适合：非敏感的用户偏好设置

Keychain：
  - 存储在加密的数据库中（SQLite + AES）
  - 系统级加密，与设备绑定
  - 越狱设备上仍有一定风险（keychain-dumper）
  - 可配置访问控制（Face ID/Touch ID）
  - 可配置不备份（ThisDeviceOnly）
  - 适合：密码、Token、证书、密钥

使用原则：
  - 密码、Token 等敏感数据 → Keychain
  - 用户偏好、非敏感配置 → UserDefaults
  - 大量数据 → 加密的文件或 Core Data（设置数据保护）
```

### Q5：什么是代码混淆？ObjC 和 Swift 各有什么限制？

```
代码混淆是将代码转换为功能相同但难以理解的形式，
增加逆向分析的难度。

ObjC 的限制：
  - 方法名无法混淆（ObjC Runtime 动态派发需要方法名）
  - class_getInstanceMethod、performSelector 等需要字符串形式的方法名
  - KVO/KVC 依赖属性名
  - 可以混淆：类名（通过宏）、实例变量名

Swift 的限制：
  - @objc 标记的方法无法混淆
  - Storyboard/Xib 引用的类名无法混淆
  - NSCoding 相关类名无法混淆
  - 可以混淆：大部分 Swift 类名和方法名（使用 SwiftShield）

通用限制：
  - 混淆后调试困难（需要保留映射文件）
  - 可能影响性能（控制流混淆）
  - 增加编译时间
  - 需要排除第三方库（可能破坏功能）
```

### Mach-O 二进制格式深度解析

```
Mach-O (Mach Object) 是 macOS/iOS 平台的可执行文件格式，
替代了传统的 ELF 格式。理解 Mach-O 是逆向工程的基础。

Mach-O 文件结构三大部分：

┌─────────────────────────────────┐
│   Mach Header (32/64位)         │  ← 文件头：CPU架构、文件类型等
├─────────────────────────────────┤
│   Load Commands (变长)           │  ← 加载指令：SEGMENT、SYMTAB等
├─────────────────────────────────┤
│   Segments & Sections           │  ← 实际数据：代码、数据、调试信息
└─────────────────────────────────┘

关键 Segments：
  __TEXT  → 包含 __text(代码)、__const(常量)、__cstring(C字符串)
  __DATA  → 包含 __data(全局变量)、__const、__got(GOT表)
  __DATA_CONST → Swift 5.9+ 新增，包含 Swift Type Metadata
  __LINKEDIT → 包含 Symbol Table、String Table、Dysymtab

关键 Load Commands：
  LC_SEGMENT_64   → 定义 segment 布局
  LC_SYMTAB       → 符号表位置
  LC_DYLD_INFO_ONLY → 导出/导入的动态链接信息
  LC_LOAD_DYLIB   → 依赖的动态库
  LC_MAIN / LC_ENTRYPOINT → 程序入口点
  LC_CODE_SIGNATURE → 代码签名信息

查看 Mach-O 结构：
  otool -hv app              → 查看 header
  otool -l app               → 列出所有 load commands
  nm app                     → 列出符号表
  size -m app                → segment/section 大小

实例：查看一个 IPA 的可执行文件
  otool -l Payload/App.app/App | grep -A5 LC_SEGMENT
  nm Payload/App.app/App | grep viewDidLoad
  dyld_PRINT=1 ./App         → 打印 dyld 加载过程
```

**坑**：bitcode 已废弃但部分旧项目仍启用，导致无法直接用 otool 查看；iOS 17+ 默认为 Swift Native Mode，不再需要 bitcode。

**最佳实践**：逆向分析时先用 `file` 确认架构，用 `otool -l` 定位关键 section，用 `nm` 找方法符号；Xcode 15+ 的 Swift Native Mode 显著改善了调试符号质量。

### dyld 二进制格式与双层命名空间

```
dyld (dynamic linker) 是 macOS/iOS 的动态链接器，负责加载 Mach-O 可执行文件
及其依赖的动态库。iOS 上 dyld 使用双层命名空间机制确保系统安全。

双层命名空间架构：

┌─────────────────────────────────┐
│   Private Namespace (层1)       │  ← 系统框架层
│   /System/Library/Frameworks    │      Foundation, UIKit, CoreFoundation...
│   /System/Library/PrivateFrameworks │  Private API
├─────────────────────────────────┤
│   Public Namespace (层2)        │  ← 应用层
│   App Bundle Libraries          │      第三方动态库 (iOS 12+)
│   @rpath 解析的动态库           │
└─────────────────────────────────┘

关键机制：
  1. Private namespace 中的库只能被系统框架访问
  2. App 层通过 dyld_shared_cache 共享系统库（节省内存）
  3. @rpath 支持：iOS 12+ 允许 App 打包动态库

dyld 加载流程：
  dyld launch → 加载 dyld_shared_cache → 解析 LC_LOAD_DYLIB
  → 绑定符号 (bind) → 重定位 (rebase) → 调用 init routines → main()

查看 dyld 加载过程：
  dyld_PRINT=1 ./App                    → 打印加载日志
  dyld_OSHYPE=1 ./App                   → 打印共享缓存命中
  dyld_OSHYPER=1 ./App                  → 打印 hyper 缓存命中
  dyld_OSHLOG=1 ./App                   → 完整 dyld 日志

dyld_shared_cache 位置：
  /System/Library/Caches/com.apple.dyld/dyld_shared_cache_arm64e
  dyld_shared_cache_dump <cache>        → 列出缓存内容
  dylibs_in_cache                       → 查看缓存中的库
```

**坑**：iOS 上 Private Namespace 隔离导致 App 无法直接加载 PrivateFrameworks，越狱环境通过替换 dyld 绕过；dyld_shared_cache 在 iOS 17+ 进一步压缩，占用约 1.5GB。

**最佳实践**：理解 dyld 双层命名空间有助于定位 dylib 加载失败问题；使用 `DYLD_PRINT_BINDINGS=1` 追踪符号绑定；iOS 12+ 打包 dylib 时注意 @rpath 配置和代码签名 entitlements。


### class-dump / class-dump-dylib 使用

```
class-dump 是逆向分析 Objective-C 二进制文件的核心工具，用于从 Mach-O 中提取
Objective-C 的元数据（类、方法、属性、协议定义），生成头文件形式的输出。
iOS 逆向中用于分析第三方 App、系统框架甚至 Private API 的结构。

class-dump-dylib 是 class-dump 的现代替代品，由 tidki 开发，支持 Swift 元数据提取
和 Swift Native Mode（iOS 17+），解决了原版 class-dump 无法处理 Swift 的问题。

基本使用：
  class-dumpHeaders Payload/App.app/App                       → 提取所有头文件
  class-dumpHeaders -a Payload/App.app/App                    → 包含类别方法
  class-dumpHeaders -H Payload/App.app/App > headers/         → 输出为独立 .h 文件
  class-dumpHeaders -- hide-app-extensions Payload/App.app/App → 隐藏系统扩展

class-dump-dylib 扩展功能：
  class-dump-dylib Payload/App.app/App                        → 基本提取（同 class-dump）
  class-dump-dylib --swift Payload/App.app/App                → 提取 Swift 类元数据
  class-dump-dylib -- hide-app-extensions -- hide-sdk Payload/App.app/App → 只显示 App 自定义类

实例：分析微信 Private API 调用
  class-dumpHeaders WeChat > WeChatHeaders.txt
  grep -n "libWeChat.dylib" WeChatHeaders.txt                 → 定位 Private API
  nm WeChat | grep -i "encrypt"                                → 定位加密相关符号

输出示例：
  @interface MMLoginViewController : UIViewController
  {
      NSString *_accountField;
      BOOL _rememberPassword;
  }
  - (void)loginWithAccount:(id)arg1 password:(id)arg2;
  - (void)onLoginButtonTouched:(id)arg1;
  @end
```

**坑**：Xcode 15+ Swift Native Mode 下 Swift 符号被 strip，class-dump-dylib 只能提取部分 Swift 元数据；bitcode 启发的项目需先 decompile；Release 模式下方法名被混淆时提取结果无意义；iOS 17+ 系统框架使用 Swift Native Mode，传统 class-dump 完全失效。

**最佳实践**：优先使用 class-dump-dylib 替代 class-dump；配合 nm/otool 交叉验证；对 Swift 项目结合 SwiftDemangle 和 swift-demangle 工具；分析 Private API 时注意 App Store 审核风险，仅用于学习和安全审计目的。


### Cydia Substrate / MobileSubstrate Hook (MSHookFunction)

```
MobileSubstrate 是越狱生态中最经典的 Hook 框架，由 Saurik 开发。
核心 API MSHookFunction 通过 inline hook 技术劫持任意函数调用，
将其重定向到自定义实现。iOS 14+ 上随着 amfi_getout 被移除，
原生 MobileSubstrate 已无法工作，被 Substitute/LiquidSeed 等替代方案取代。

工作原理 — inline hook 替换函数入口指令为 branch:

  Original:     bff bff bff bff bff bff bff bff bff bff bff bff bff bff bff bff
                 ↓ MSHookFunction 替换前5条指令为 B #hook_func
  Hooked:       bff bff bff bff bff bff bff bff bff bff bff bff bff bff bff bff
                 ↓                       ↓
  Trampoline:   orig_code... B #back_to_original   ← 保存原始指令
                 ↓
  Hook Func:    自定义逻辑 → trampoline() 调用原始函数

基本使用:

#include <substrate.h>
void (*orig_printf)(const char*, ...) = printf;
void my_printf(const char *fmt, ...) {
    NSLog(@"Hooked: %@", fmt);
    orig_printf(fmt);  // 调用原始函数
}
MSHookFunction((void *)printf, (void *)my_printf, (void **)&orig_printf);

MSHook 宏 (方法级 Hook):
static void (*orig_viewDidLoad)(id, SEL);
void my_viewDidLoad(id self, SEL _cmd) {
    orig_viewDidLoad(self, _cmd);
    NSLog(@"viewDidLoad called");
}
%hook UIViewController
- (void)viewDidLoad { my_viewDidLoad(self, _cmd); }
%end

iOS 14+ 替代方案:
  Substitute (SHSH) — dylib 注入 + method swizzle
  LiquidSeed — 无 amfi 限制的 dylib loader
  hide-vspringd — 隐藏 SpringBoard 注入痕迹
```

**坑**：MSHookFunction 在 arm64e + PAC 签名环境下极不稳定；inline hook 需处理 Thumb-2 指令对齐；iOS 14+ amfi 移除后原生 Substrate dylib 无法加载；Hook 系统关键函数可能导致 crash 或 bootloop；多 dylib 同时 Hook 同一函数时 trampoline 冲突。

**最佳实践**：iOS 14+ 迁移至 Substitute/LiquidSeed；inline hook 前验证指令边界；始终保留原始函数指针并调用；生产环境添加 crash guard；越狱检测场景下注意 hide-scutil/hide-vgoodness 等反检测工具的对抗。


### Frida 动态插桩

```
Frida 是最强大的跨平台动态分析框架，支持 iOS/macOS/Android/Windows/Linux。
通过注入 JavaScript VM 到目标进程，实现方法 Hook、内存读写、系统调用追踪等。
iOS 场景下通过 frida-darwin + frida-inject 实现 dylib 注入，iOS 15+ 需 amfi_getout 或 debugss 绕过 SIP。

工作流程 — SPA (Spawn-Attach) vs attach 模式:

  SPA 模式 (推荐):                       Attach 模式:
  frida-ps -Ua 列出进程                   frida-ps -Ua 列出进程
  ↓                                      ↓
  frida -n launch_args --no-pause        frida -p <pid>
  ↓                                      ↓
  SPAWNED → resume() → 执行              ALREADY RUNNING → Hook
  
JavaScript Hook API:

const { StringBuffer } = require("frida-bootstrap-heap");
const target = Module.findExportByName("libSystem.B.dylib", "getpwuid");
Intercept.replace(target, new NativeCallback((uid) => {
    // 自定义逻辑 — 如绕过越狱检测
    return NativePointer("0xdead");
}, "pointer", ["int"]));

Intercept.attach("getpwuid", {
    onEnter(args) {
        console.log("getpwuid(" + args[0] + ")");
    },
    onLeave(retval) {
        console.log("  → " + retval.readPointer());
    }
});

iOS Swift 方法 Hook (Objective-C bridging):

const viewDidLoad = ObjC.classes.UIViewController["viewDidLoad"];
Intercept.attach(viewLoaded.implementation, {
    onEnter(args) {
        const self = new ObjC(args[0]);
        console.log("viewDidLoad on: " + self.$className);
    }
});

frida-tools 常用命令:
  frida-ps -Ua          列出所有应用进程
  frida-trace -i "get*" 追踪所有 get* 函数
  frida-lighthouse      定位方法偏移
  frida-kill <pid>      终止目标进程
  frida -U <app> -l hook.js 加载自定义脚本


### LLDB 脚本与动态分析

LLDB 是 Xcode 默认的调试器（替代 GDB），支持 Python 扩展、表达式求值、内存检查等高级功能。iOS 逆向中，LLDB 脚本是定位 crash、分析内存布局、动态 Hook 的核心工具。

**常用命令速查**：

```
po <obj>              // 打印对象描述（po = printf object）
p <var>               // 打印变量值
frame variable        // 列出当前栈帧所有变量
register read         // 读取寄存器（x0-x7, sp, lr...）
memory read 0xaddr    // 读取原始内存
thread return -r      // 强制方法提前返回（po 求值后注入结果）
image list -o -f      // 列出所有加载模块及偏移
```

**LLDB Python 脚本扩展** — 在 debugSession 中执行：

```python
import lldb

def show_swift_objects(debugger, command, result, internal_dict):
    target = debugger.GetSelectedTarget()
    for image in target.module_iter():
        if image.GetPlatformFileSpec().path.endswith('.framework'):
            for sym in image.symtab_iter():
                if b'Swift' in sym.name:
                    lldb.SBStreamWrite(result, sym.name.decode() + '\n')

def __lldb_init_script(debugger):
    debugger.HandleCommand('command script add -f show_swift_objects show_swift_objects')
    debugger.HandleCommand('command alias swift_objs show_swift_objects')
```

将上述脚本保存为 `~/.lldbinit`，每次 Xcode debug session 自动加载自定义命令。

**LLDB + Frida 联动** — SPA 模式下用 LLDB attach 定位 crash 现场，用 Frida 做持续 Hook：

```
frida -U --no-pause -l hook.js SpringBoard
# 在 Frida REPL 中:
lldb.cmd("po [[UIApplication sharedApplication] delegate]")
lldb.cmd("p (void)0x1a2b3c")
```

**坑**：LLDB `po` 在 Swift 代码中可能触发 side effects（如 getter 被调用）；`thread return` 仅适用于简单返回类型，复杂 Swift 值会 crash；LLDB 脚本在模拟器上可用但在真机上部分 API 受限；Xcode 15+ 默认启用 Swift debug bridging，po Swift 对象需先 bridging 为 OC 类型；LLDB 表达式求值会暂停目标进程，长耗时脚本导致超时中断。

**最佳实践**：生产 crash 分析用 `po` + `frame variable` + `register read` 组合定位；自定义脚本放入 `~/.lldbinit`；复杂场景用 `lldb` Python API 写独立脚本；LLDB + Private.DIScover（LLDB plugin）定位 Swift crash；iOS 17+ 结合 Xcode 15 Memory Graph Debugger 做内存快照对比。


### Xcode 内存图调试器（Memory Graph Debugger）

Xcode 11+ 引入的 Memory Graph Debugger（MGD）是可视化内存关系的杀手级工具。在 Debug Navigator 中点击 "Memory Graph" 按钮，Xcode 会通过 LLDB 深度遍历堆对象，生成以选中控制器为根的有向图，清晰展示 retain cycle、僵尸对象、以及意外持有的引用。

**核心操作**：

```
1. Debug Navigator → Memory Graph → Capture Sampling（截图式快照）
2. 点击图中节点查看 retainers（谁在持有我）和 referents（我持有了谁）
3. 右键节点 → Show Retain Cycle 高亮循环引用路径
4. Filter 栏搜索类名快速定位可疑对象
5. Resume Sampling 持续追踪内存变化
```

**典型场景** — ViewController 未释放：

```swift
class DetailVC: UIViewController {
    var timer: Timer?  // strong 引用 Timer，Timer target=self → retain cycle!
    
    override func viewDidLoad() {
        super.viewDidLoad()
        timer = Timer.scheduledTimer(withTimeInterval: 1.0, repeats: true) { [weak self] _ in
            self?.updateUI()  // weak self 中断循环
        }
    }
    
    // ❌ 错误做法：deinit 中 invalidate（永远不会被调用！）
    // ✅ 正确做法：用 weak self + 在不需要时手动 invalidate
}
```

MGD 图中会显示 `DetailVC → Timer → DetailVC` 的环形箭头，一目了然。

**坑**：MGD 仅显示堆上对象，C/C++ malloc 内存、CoreData 持久化层、以及通过 file descriptor 持有的资源不在图中；截图耗时与对象图大小成正比，大型 app 可能需 10-30 秒；MGD 在模拟器上效果远好于真机（真机部分系统框架被 strip）；Xcode 15+ 支持 Swift Concurrency 任务追踪但需启用 "Record Execution" 模式；截图时若 target 正在执行动画或网络请求，图中可能出现瞬时对象导致误判。

**最佳实践**：定位 retain cycle 时先 Pause 再截图；配合 Instruments → Allocations 确认具体泄漏量；iOS 17+ 结合 Xcode 15 Memory API（os_signpost +os_trace）定位非堆内存泄漏；截图前调用 `deinit` 触发点（如 pop VC）确保对象应释放；MGD 截图结果可导出为 .graph 文件用于团队复盘。


### 越狱检测方法

现代 iOS 应用中，越狱检测是防止逆向攻击、未授权修改和商业欺诈的第一道防线。攻击者通过 Cydia/Sileo 安装越狱环境后，可随意 Hook 任意方法、篡改二进制、读取沙盒外数据。检测策略应**多层次叠加**，单一方法可被 Cydia Substrate 或 libjailbreak_detection 轻松绕过。

**经典检测矩阵**（从低到高可信度排列）：

```swift
enum JailbreakStatus { case notJailbroken, jailbroken, unknown }

func detectJailbreak() -> JailbreakStatus {
    // 1. 检查越狱工具路径（低可信度，可伪造）
    let paths = [
        "/Applications/Sileo.app", "/Applications/Cydia.app",
        "/bin/bash", "/usr/sbin/sshd", "/etc/apt",
        "/Library/MobileSubstrate/MobileSubstrate.dylib"
    ]
    for path in paths {
        if FileManager.default.fileExists(atPath: path) {
            return .jailbroken
        }
    }
    // 2. 检查写权限（中可信度）
    if (try? "/private/test_jailbreak.txt".write("test")) != nil {
        return .jailbroken
    }
    // 3. 检查 Cydia Substrate 符号（高可信度）
    if NSClassFromString("substrate_init") != nil {
        return .jailbroken
    }
    // 4. 检查容器完整性（最高可信度，iOS 9+）
    guard let url = Bundle.main.bundleURL.deletingLastPathComponent() else { return .unknown }
    guard let flags = try? url.resourceValues(forKeys: [.isSystemWritableKey]) else { return .unknown }
    if flags.isSystemWritable == true { return .jailbroken }
    return .notJailbroken
}
```

**防御绕过策略**：攻击者可 Hook `fileExists(atPath:)` 返回 false，或 patch 检测函数始终返回 notJailbroken。应对方案：① 将检测逻辑放在**未公开的私有方法**中（增加 Hook 难度），② 使用 `dlopen`/`dlsym` 动态调用系统 API 避免静态符号被搜索，③ 结合 Server-Side 验证（设备指纹 + 行为分析），④ iOS 14+ 利用 `processInfo.hostName` 差异（越狱设备通常修改 hostname）。

**坑**：容器完整性检测在 simulator 上始终返回 true（ simulator 本身就是"可写容器"），需加 `#if targetEnvironment(simulator)` 守卫；`/bin/sh` 在 simulator 也存在，不能作为真机检测依据；部分企业签名/测试分发渠道可能触发误报；LLDB attach 后容器权限短暂提升，检测窗口期极短。

**最佳实践**：生产环境采用"检测矩阵 + 服务端校验"双轨制，本地检测用于快速阻断（UI 层），服务端基于设备指纹（IDFV + 安装时间 + 行为模式）做最终判定；iOS 17+ 可结合 `SecKey` 在 Secure Enclave 中做设备绑定签名，越狱设备无法通过签名验证；避免在 Core 业务逻辑中硬编码检测（防止功能被完全破坏），优先用于反作弊/反欺诈场景。


### Keychain 访问与安全属性

iOS Keychain 是系统级加密存储，数据默认持久化且**不随 App 删除而清除**（除非显式调用 SecItemDelete）。每个 Item 通过 `SecRecord` 定义搜索/存储参数，通过安全属性控制访问策略。

**核心安全属性矩阵**：

```swift
import Security

func saveToKeychain(password: String, service: String = "com.app.main") {
    // ❌ 默认：kSecAttrAccessibleAlways（设备解锁后始终可访问，包括锁屏/后台）
    // ✅ 推荐：kSecAttrAccessibleAfterFirstUnlock（首次解锁后始终可访问，适合后台任务）
    // 🔒 最严：kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly（需设备密码 + 仅限本设备）
    let query: [String: Any] = [
        kSecClass: kSecClassGenericPassword,
        kSecAttrService: service,
        kSecAttrAccount: "user_token",
        kSecValueData: password.data(using: .utf8)!,
        kSecAttrAccessible: kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly,
        // iOS 17+: 可结合 Secure Enclave 做设备绑定
        kSecUseDataProtectionKey: true  // 启用 Data Protection 加密层
    ]
    let status = SecItemAdd(query as CFDictionary, nil)
    guard status == errSecSuccess else {
        print("Keychain save failed: \(status)")
        return
    }
}
```

**Pitfalls**：`kSecAttrAccessibleAlways` 在设备被物理攻击（Cold Boot/DFU 提取 flash）时可被恢复；多设备 Share 使用 `kSecAttrAccessibleWhenUnlocked` + Keychain Access Group 但**不同设备间不自动同步**（需显式配置 Keychain Sharing 能力并在同一个 Keychain Group 下通过 iCloud Keychain 手动触发同步）；`SecItemCopyMatching` 返回 `UnsafeMutablePointer` 需手动 `Deallocator.free()` 防止内存泄漏（Swift 5.9+ 可用 `withUnsafeBytes` 避免手动释放）。

**最佳实践**：生产环境优先使用 `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`（后台服务可用）或 `kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly`（高敏感数据）；配合 Keychain Sharing 实现多 App 共享认证令牌；iOS 17+ 结合 Secure Enclave 的 `SecKey` 实现设备绑定的密钥派生，防止 Keychain Item 被导出后在其他设备使用。

### Secure Enclave（飞地密钥派生）

Secure Enclave 是 A7 及以上芯片中独立的协处理器，通过物理隔离实现密钥存储与加密运算，主处理器（Application Processor）**无法直接读取**其中的密钥数据。密钥派生（Key Derivation）是基于主密钥生成子密钥的核心机制，广泛用于设备绑定认证、数据加密和反欺诈场景。

**密钥派生工作流程**：

```swift
import Security

// 1. 在 Secure Enclave 中生成 256 位 ECC 密钥对（私钥永不离开 Enclave）
let params: [String: Any] = [
    kSecAttrKeyType: kSecAttrKeyTypeECSEC-SHA256,
    kSecAttrKeySizeInBits: 256,
    kSecAttrTokenID: kSecAttrTokenIDSecureEnclave,  // 强制存储在 Enclave
    kSecAttrLabel: "com.app.encryption.key" as CFString,
    kSecAttrAccessible: kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly,
    kSecPrivateKeyUsage: kSecUsageSigningVerification  // 限制用途：仅签名/验签
]
var keyRef: AnyObject?
let status = SecKeyCreateRandomKey(params as CFDictionary, &keyRef)
guard status == errSecSuccess, let privateKey = keyRef as! SecKey? else { return }

// 2. 基于 Enclave 密钥派生 session key（用于业务数据加密）
let userInfo = [kSecSecKeyUsage: kSecUsageKeyDerivation] as CFDictionary
let sessionData = "session_nonce_\(Date().timeIntervalSinceNow)".data(using: .utf8)!
if let derived = SecKeyCopyKeyRef(privateKey, userInfo, sessionData as CFData, nil) {
    // derived 密钥可用于业务加密，基于 Enclave 主密钥 + 业务上下文派生
    print("✅ 密钥派生成功，基于设备唯一密钥")
}
```

**Pitfalls**：Secure Enclave 密钥**不可导出**（`SecKeyCopyExternalRepresentation` 返回 nil），只能用于 Enclave 内部的运算（签名/加密/派生）；设备恢复（Restore/OTA 大版本升级）后 Enclave 密钥**自动销毁**，旧密钥签名无效；密钥操作有速率限制（约 20 次/秒），高频调用触发 `errSecPasswordRefused`；`kSecAttrTokenIDSecureEnclave` 仅在真机可用，Simulator 调用直接返回 nil。

**最佳实践**：设备绑定认证流程 = 服务端下发 challenge → 本地用 Enclave 密钥签名 → 服务端用公钥验签（公钥可导出并上传服务端）；密钥派生结合业务上下文（userID + timestamp + nonce），实现一次泄露不影响全局；配合 Keychain 的 `kSecUseKeychainBinding` 限制密钥使用范围；iOS 17+ 可用 `SecKeyCreateWithData` 导入业务密钥到 Enclave 进行混合加密架构。

### 代码签名验证与权限（Entitlements）

iOS 通过代码签名机制确保 App 完整性与来源可信。签名基于 RSA/ECDSA 数字签名，包含 Code Directory（哈希树）、CDHASH 和 CMS 签名块。系统通过 `codesign` 工具在加载时验证签名链：`Apple Root CA → WWDR CA → 开发/发布证书 → App 二进制`。任何字节修改（包括动态生成的代码）都会导致签名校验失败触发 `EXC_BAD_ACCESS`。

**Entitlements 机制**：`*.entitlements` plist 定义 App 的权限边界（如 push notifications、Keychain Sharing、Associated Domains），签名时将 entitlements 哈希嵌入签名块。系统通过 `sandbox` 机制强制检查：未声明的 entitlement 调用直接返回错误。

```swift
// 运行时检查当前 App 的 entitlements
import MobileCoreServices

func currentEntitlements() -> [String: Any]? {
    guard let url = Bundle.main.url(forResource: "Entitlements", withExtension: "plist"),
          let data = try? Data(contentsOf: url),
          let entitlements = try? PropertyListSerialization.propertyList(
              from: data, format: nil) as? [String: Any] else { return nil }
    return entitlements
}

// 通过 task_info 检查是否为调试/未签名状态（反调试检测基础）
import Darwin
func isJailbroken() -> Bool {
    var k = libkern.task_info
    var ksize = MemoryLayout<libkern.task_info>.stride
    guard task_info(mach_task_self_, libkern.task_flavor, &k, &ksize) == KERN_SUCCESS else { return false }
    // effective_user_id != 0 通常表示未通过签名限制（越狱/调试器附加）
    return k.effective_user_id == 0
}
```

**Pitfalls**：`Entitlements.plist` 修改后**必须重新签名**（Xcode 自动处理，手动签名需用 `codesign --entitlements` 参数）；Ad Hoc 签名最多 100 设备限制；TestFlight 签名 7 天后过期；Entitlement 滥用（如未启用 Associated Domains 却调用 `LNURLSession` deep link）不会崩溃但功能静默失效。

**最佳实践**：生产环境使用 Distribution 证书 + App Store Connect 自动签名；通过 `codesign -d --entitlements - <app_path>` 检查已签名的 entitlement 内容；利用 `kSecAttrSynchronizable` 配合 Keychain Sharing entitlement 实现多 App 共享认证态；iOS 17+ 的 `dlopen` 限制进一步收紧动态库加载，第三方 SDK 必须预链接到主二进制中。

### 反逆向技术（混淆、反调试）

生产 App 面临反编译、动态调试、Hook 注入等威胁。反逆向的核心思路：提高攻击者的时间与经济成本，使其放弃逆向。常见技术手段包括代码混淆（Control Flow Flattening/字符串加密）、反调试检测（ptrace/LLDB 端口监听/`is_debugger_attached`）、完整性校验（Code Signature 运行时自检/LLDB 断点触发检测）、以及 Cydia Substrate/Flipper 检测。

```swift
// 反调试检测：多策略组合（单一检测可被 patch 绕过）
import Darwin

func isDebuggerAttached() -> Bool {
    // 策略 1: ptrace 检测（最有效的反调试手段）
    guard ptrace(PTRACE_TRACEME, 0, nil, 0) != -1 else { return true }
    // 策略 2: task_for_pid 限制（iOS 不允许跨进程获取 task port）
    var port = mach_port_t()
    if task_for_pid(mach_task_self_, getuid(), &port) == KERN_SUCCESS {
        return true  // 通常只有调试器/越狱环境能成功
    }
    // 策略 3: 检测 LLDB 默认监听端口
    let sock = socket(AF_INET, SOCK_STREAM, 0)
    var addr = sockaddr_in(sin_family: sa_family_t(AF_INET),
                           sin_port: htons(12345), sin_addr: in_addr(s_addr: inet_addr("127.0.0.1")))
    if connect(sock, &addr, socklen_t(MemoryLayout<sockaddr_in>.stride)) != -1 {
        close(sock); return true  // LLDB 正在监听
    }
    close(sock)
    return false
}
```

**Pitfalls**：单一反调试手段（仅 ptrace 或仅端口检测）可被 Cydia Substrate 轻松 hook 绕过；`PTRACE_TRACEME` 在越狱设备/LLDB 附加后调用会触发 `EPERM` 但返回值不确定；反调试代码本身也存在于二进制中，高级逆向者可静态分析后 patch 掉检测逻辑；LLDB 的 `process attach` 不触发 ptrace 检测。

**最佳实践**：多策略叠加（ptrace + task_for_pid + 端口检测 + 完整性校验 + 时间间隔检测），每 30 秒异步轮询一次（避免阻塞主线程）；配合代码混淆（LLVM obfuscator/Control Flow Flattening）增加静态分析难度；关键业务逻辑使用 Secure Enclave 隔离；检测触发后执行蜜罐逻辑（返回错误数据/静默退出/上报异常），而非直接 crash（避免成为 DoS 攻击面）。

### OTA 更新与代码注入防护

iOS 不允许真正的全量 OTA（像 Android 那样覆盖系统镜像），但 App 层面可实现部分资源的远程更新。常见方案包括：Widget 扩展通过 `today` 机制拉取远程配置、JavaScript 通过 WKWebView 动态加载页面逻辑、LLVM 字节码通过第三方运行时（如 React Native/Flutter 引擎）实现业务逻辑热更。**Apple 严格禁止通过 `dlopen` 动态加载未签名的 Mach-O 二进制**，违反将导致 App Store 审核拒绝（Guideline 2.5.2）。

**Pitfalls**：dlopen 加载未签名 dylib 在 iOS 17+ 上直接触发 `EXC_BAD_ACCESS`（即使是越狱设备也受限制）；OTA 包未做签名校验 = 攻击者可劫持 CDN 注入恶意逻辑；LLVM 字节码方案（JSC/LLVM runtime）在 2023 年后多次被 Apple 以 Guideline 2.5.2 拒绝上架；JS 通过 WebView 执行是最安全的"热更"路径但性能受限。

**最佳实践**：OTA 仅用于配置/资源/JS 逻辑下发，核心业务逻辑始终通过 App Store 全量更新；OTA 包强制签名校验（ECDSA/Ed25519），公钥预埋在 App 二进制中；OTA 拉取后写入 UserDefaults/沙盒前做 schema 版本校验（防止旧版 config 结构破坏新版逻辑）；配合远程开关实现灰度发布（按设备 ID/用户 cohort 分流），配合 Bugsnag/Sentry 实时监控 OTA 后崩溃率，超阈值自动回滚到本地默认配置。
