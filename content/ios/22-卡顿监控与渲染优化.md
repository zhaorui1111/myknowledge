# 卡顿监控与渲染优化工程详解

---

## 一、卡顿的本质

### 1.1 屏幕刷新原理

```
屏幕硬件每隔固定时间发出 VSync 信号（垂直同步）：
  60Hz 屏幕：每 16.67ms 一次
  90Hz 屏幕：每 11.11ms 一次
  120Hz 屏幕（ProMotion）：每 8.33ms 一次

每一帧的工作（以 60fps 为例，预算 16.67ms）：
  ┌────────────────────────────────────────────────────────┐
  │  VSync 信号到达                                         │
  │       ↓                                               │
  │  CPU 工作（主线程，约 10ms 预算）：                      │
  │    - 处理触摸事件                                       │
  │    - 执行业务逻辑                                       │
  │    - Auto Layout 计算                                  │
  │    - 图片解码                                          │
  │    - 提交 CATransaction（图层树）                       │
  │       ↓                                               │
  │  GPU 工作（Render Server，约 6ms 预算）：                │
  │    - 纹理合成                                          │
  │    - 离屏渲染（如有）                                   │
  │    - 写入帧缓冲区                                       │
  │       ↓                                               │
  │  下一个 VSync：屏幕读取帧缓冲区，显示这一帧              │
  └────────────────────────────────────────────────────────┘

卡顿（Jank）：
  某一帧的 CPU+GPU 工作超过 16.67ms，
  下一个 VSync 到来时帧还没渲染好，
  屏幕只能继续显示上一帧 → 用户感知到卡顿

掉帧严重程度：
  掉 1 帧（33ms 才更新）：轻微，大多数用户感知不到
  掉 2 帧（50ms 才更新）：明显，用户会感到"不流畅"
  掉 4+ 帧（> 66ms）：严重卡顿，用户明显感知
```

### 1.2 ProMotion 与自适应帧率

```swift
// iPhone 13 Pro+ 支持 ProMotion（1-120Hz 自适应）
// 系统根据内容动态调整帧率：

//   快速滚动：提升到 120Hz（流畅）
//   动画：根据动画速度选择合适帧率

// 对开发者的影响：

    preferred: 120  // 期望 120fps
)
displayLink.add(to: .main, forMode: .common)

// 2. 固定帧率动画（如 30fps 的视频）

    minimum: 30,
    maximum: 30,
    preferred: 30

// 3. 不设置时系统默认使用 60fps（不会自动升到 120fps）
// 这是很多 App 在 ProMotion 设备上仍然只有 60fps 的原因

// 检查设备是否支持高刷
let maxFPS = UIScreen.main.maximumFramesPerSecond  // 60 或 120
```

## 二、卡顿监控方案

### 2.1 方案一：RunLoop Observer 监控（最主流）

```swift
// 原理：监控主线程 RunLoop 在事件处理阶段的耗时
// 如果从 BeforeSources → BeforeWaiting 超过阈值，说明主线程卡住了

class RunLoopMonitor {
    
    private var observer: CFRunLoopObserver?
    private var semaphore = DispatchSemaphore(value: 0)
    private var activity: CFRunLoopActivity = []
    private var timeoutCount = 0
    
    // 卡顿阈值（毫秒）
    private let threshold: Int = 50  // 超过 50ms 认为卡顿（约 3 帧）
    // 连续超时次数阈值（避免偶发误报）
    private let continuousThreshold = 3
    
    func start() {
        setupObserver()
        startMonitorThread()
    }
    
    private func setupObserver() {
        observer = CFRunLoopObserverCreateWithHandler(
            kCFAllocatorDefault,
            CFRunLoopActivity.allActivities.rawValue,
            true, 0
        ) { [weak self] _, activity in
            self?.activity = activity
            self?.semaphore.signal()  // 通知监控线程：RunLoop 状态变化了
        }
        CFRunLoopAddObserver(CFRunLoopGetMain(), observer!, .commonModes)
    }
    
    private func startMonitorThread() {
        DispatchQueue.global(qos: .utility).async { [weak self] in
            guard let self = self else { return }
            
            while true {
                // 等待 RunLoop 状态变化，超时时间 = 卡顿阈值
                let result = self.semaphore.wait(
                    timeout: .now() + .milliseconds(self.threshold)
                )
                
                if result == .timedOut {
                    // 超时：RunLoop 在 threshold 时间内没有状态变化
                    // 即主线程在某个阶段停留超过 threshold ms
                    
                    // 只关心事件处理阶段的卡顿
                    let isInEventHandling = self.activity == .beforeSources
                                        || self.activity == .afterWaiting
                    
                    if isInEventHandling {
                        self.timeoutCount += 1
                        
                        if self.timeoutCount >= self.continuousThreshold {
                            // 连续超时，确认为卡顿
                            self.reportJank()
                            self.timeoutCount = 0
                        }
                    }
                } else {
                    // 正常：RunLoop 状态变化了，重置计数
                    self.timeoutCount = 0
                }
            }
        }
    }
    
    private func reportJank() {
        // 抓取主线程调用栈
        let callStack = captureMainThreadCallStack()
        
        // 上报到监控平台
        JankReporter.report(
            duration: threshold * continuousThreshold,
            callStack: callStack,
            timestamp: Date()
        )
        
        #if DEBUG
        print("⚠️ 检测到卡顿！调用栈：")
        callStack.forEach { print($0) }
        #endif
    }
    
    // 抓取主线程调用栈
    private func captureMainThreadCallStack() -> [String] {
        // 使用 BSBacktraceLogger 或 PLCrashReporter
        return Thread.callStackSymbols  // 简化版，实际应使用更精确的工具
    }
}
```

### 2.2 方案二：CADisplayLink FPS 监控

```swift
// 原理：通过 CADisplayLink 计算实际帧率
// 优点：实现简单，可以实时显示 FPS
// 缺点：无法定位卡顿原因，只能知道"卡了"，不知道"为什么卡"

class FPSMonitor: NSObject {
    
    private var displayLink: CADisplayLink?
    private var lastTimestamp: CFTimeInterval = 0
    private var frameCount: Int = 0
    
    var onFPSUpdate: ((Double) -> Void)?
    
    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(tick(_:)))
        displayLink?.add(to: .main, forMode: .common)
    }
    
    func stop() {
        displayLink?.invalidate()
        displayLink = nil
    }
    
    @objc private func tick(_ link: CADisplayLink) {
        if lastTimestamp == 0 {
            lastTimestamp = link.timestamp
            return
        }
        
        frameCount += 1
        let elapsed = link.timestamp - lastTimestamp
        
        if elapsed >= 1.0 {  // 每秒统计一次
            let fps = Double(frameCount) / elapsed
            frameCount = 0
            lastTimestamp = link.timestamp
            
            onFPSUpdate?(fps)
            
            if fps < 55 {
                print("⚠️ FPS 下降: \(String(format: "%.1f", fps))fps")
            }
        }
    }
}

// 悬浮窗显示 FPS（Debug 模式）
#if DEBUG
class FPSWindow: UIWindow {
    
    static let shared = FPSWindow()
    private let label = UILabel()
    private let monitor = FPSMonitor()
    
    init() {
        super.init(frame: CGRect(x: UIScreen.main.bounds.width - 60, y: 40, width: 55, height: 20))
        setupUI()
        setupMonitor()
    }
    
    required init?(coder: NSCoder) { fatalError() }
    
    private func setupUI() {
        windowLevel = .statusBar + 1
        backgroundColor = .black.withAlphaComponent(0.7)
        layer.cornerRadius = 4
        
        label.frame = bounds
        label.textColor = .green
        label.font = .systemFont(ofSize: 11, weight: .bold)
        label.textAlignment = .center
        addSubview(label)
        isHidden = false
    }
    
    private func setupMonitor() {
        monitor.onFPSUpdate = { [weak self] fps in
            DispatchQueue.main.async {
                self?.label.text = "\(Int(fps)) FPS"
                self?.label.textColor = fps >= 55 ? .green : (fps >= 40 ? .yellow : .red)
            }
        }
        monitor.start()
    }
}
#endif
```

### 2.3 方案三：主线程 Ping 监控

```swift
// 原理：子线程定期向主线程发送 ping，如果超时没收到 pong，说明主线程卡住了
// 优点：实现简单，误报率低
// 缺点：精度稍低（取决于 ping 间隔）

class MainThreadPingMonitor {
    
    private let pingInterval: TimeInterval = 0.1  // 每 100ms ping 一次
    private let timeout: TimeInterval = 0.4       // 超过 400ms 没响应 = 卡顿
    private var isRunning = false
    
    func start() {
        isRunning = true
        pingLoop()
    }
    
    func stop() {
        isRunning = false
    }
    
    private func pingLoop() {
        guard isRunning else { return }
        
        let pingTime = Date()
        var pongReceived = false
        
        // 向主线程发送 ping
        DispatchQueue.main.async {
            pongReceived = true  // 主线程响应了 pong
        }
        
        // 子线程等待 timeout 时间
        DispatchQueue.global(qos: .utility).asyncAfter(deadline: .now() + timeout) { [weak self] in
            guard let self = self else { return }
            
            if !pongReceived {
                // 超时未收到 pong，主线程卡住了
                let duration = Date().timeIntervalSince(pingTime)
                self.handleJank(duration: duration)
            }
            
            // 继续下一次 ping
            DispatchQueue.global(qos: .utility).asyncAfter(deadline: .now() + self.pingInterval) {
                self.pingLoop()
            }
        }
    }
    
    private func handleJank(duration: TimeInterval) {
        let callStack = captureMainThreadStack()
        print("主线程卡顿 \(Int(duration * 1000))ms")
        JankReporter.report(duration: duration, callStack: callStack)
    }
    
    private func captureMainThreadStack() -> [String] {
        return Thread.callStackSymbols
    }
}
```

### 2.5 方案五：MetricKit 系统级监控（iOS 13+）

```swift
// 苹果官方提供的性能指标收集框架
// 优点：真实用户数据，系统级精度，无性能开销
// 缺点：每 24 小时聚合一次，实时性差；iOS 13+

import MetricKit

class MetricsSubscriber: NSObject, MXMetricManagerSubscriber {
    
    override init() {
        super.init()
        MXMetricManager.shared.add(self)
    }
    
    // 每 24 小时收到一次聚合数据
    func didReceive(_ payloads: [MXMetricPayload]) {
        for payload in payloads {
            analyzeMetrics(payload)
        }
    }
    
    // iOS 14+ 收到诊断数据（含卡顿堆栈）
    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            // 卡顿诊断（Hang）
            for hang in payload.hangDiagnostics ?? [] {
                print("卡顿时长: \(hang.hangDuration)s")
                print("调用栈: \(hang.callStackTree)")
            }
            
            // CPU 异常
            for cpuException in payload.cpuExceptionDiagnostics ?? [] {
                print("CPU 占用: \(cpuException.totalCPUTime)s")
                print("调用栈: \(cpuException.callStackTree)")
            }
        }
    }
    
    private func analyzeMetrics(_ payload: MXMetricPayload) {
        // 卡顿时间分布（Hang）
        let hangTime = payload.applicationResponsivenessMetrics?
            .histogrammedApplicationHangTime
        
        // 帧率统计
        let animationMetrics = payload.animationMetrics
        let scrollHitches = animationMetrics?.scrollHitchTimeRatio  // 滚动卡顿比例
        
        // 上报到自己的服务端
        if let hitches = scrollHitches {
            print("滚动卡顿比例: \(hitches)")
            // 业界标准：< 5ms/s 为优秀，5-10ms/s 为良好，> 10ms/s 需优化
        }
    }
}
```

## 三、主线程卡顿的常见原因与优化

### 3.1 布局计算卡顿

```swift
// 原因：Auto Layout 在复杂视图层级下计算量大
// 实测：1000 个约束的视图，layoutSubviews 耗时约 10-30ms

// ❌ 问题代码：每次滚动都触发大量约束计算
class BadCell: UITableViewCell {
    override func layoutSubviews() {
        super.layoutSubviews()
        // 在 layoutSubviews 中添加/修改约束
        titleLabel.snp.remakeConstraints { make in
            make.top.equalToSuperview().offset(isExpanded ? 20 : 10)
            // ...
        }
    }
}

// ✅ 优化1：缓存计算结果
class GoodCell: UITableViewCell {
    private var cachedHeight: CGFloat = 0
    
    func configure(with model: CellModel) {
        // 只在数据变化时重新计算
        if model.id != currentModelId {
            currentModelId = model.id
            cachedHeight = model.precomputedHeight  // 提前在后台线程计算好
        }
    }
}

// ✅ 优化2：手动布局（极端性能场景）
class HighPerformanceCell: UITableViewCell {
    override func layoutSubviews() {
        super.layoutSubviews()
        let width = contentView.bounds.width
        // 直接设置 frame，比 Auto Layout 快 10 倍以上
        avatarView.frame = CGRect(x: 16, y: 12, width: 44, height: 44)
        titleLabel.frame = CGRect(x: 72, y: 12, width: width - 88, height: 20)
        subtitleLabel.frame = CGRect(x: 72, y: 36, width: width - 88, height: 16)
    }
}

// ✅ 优化3：预计算高度，避免 Self-Sizing 的重复计算
struct CellViewModel {
    let title: String
    let subtitle: String
    
    // 在数据层提前计算，不在主线程 heightForRow 中计算
    lazy var cellHeight: CGFloat = {
        let titleHeight = title.boundingRect(
            with: CGSize(width: UIScreen.main.bounds.width - 88, height: .infinity),
            options: .usesLineFragmentOrigin,
            attributes: [.font: UIFont.systemFont(ofSize: 16)],
            context: nil
        ).height
        return titleHeight + 60  // 加上固定间距
    }()
}
```

### 3.2 图片解码卡顿

```swift
// 原因：PNG/JPEG 图片在首次渲染时在主线程解码（位图解压缩）
// 实测：一张 1000×1000 的 PNG，解码耗时约 5-15ms

// ❌ 问题：在主线程直接设置图片
imageView.image = UIImage(named: "large_image")  // 解码在主线程

// ✅ 优化：后台线程提前解码
class ImagePreloader {
    
    static func preload(image: UIImage, targetSize: CGSize) -> UIImage {
        let format = UIGraphicsImageRendererFormat()
        format.scale = UIScreen.main.scale
        format.opaque = true  // 不透明，减少内存
        
        let renderer = UIGraphicsImageRenderer(size: targetSize, format: format)
        return renderer.image { _ in
            image.draw(in: CGRect(origin: .zero, size: targetSize))
        }
        // 返回的 image 已经是解码后的位图，设置到 imageView 时无需再解码
    }
    
    static func asyncPreload(url: URL, targetSize: CGSize, completion: @escaping (UIImage?) -> Void) {
        DispatchQueue.global(qos: .userInitiated).async {
            guard let data = try? Data(contentsOf: url),
                  let image = UIImage(data: data) else {
                DispatchQueue.main.async { completion(nil) }
                return
            }
            
            let decoded = preload(image: image, targetSize: targetSize)
            DispatchQueue.main.async { completion(decoded) }
        }
    }
}

// ✅ 优化：使用 ImageIO 降采样（内存 + 解码双重优化）
func downsampledImage(at url: URL, to pointSize: CGSize, scale: CGFloat) -> UIImage? {
    let imageSourceOptions = [kCGImageSourceShouldCache: false] as CFDictionary
    guard let imageSource = CGImageSourceCreateWithURL(url as CFURL, imageSourceOptions) else {
        return nil
    }
    
    let maxDimensionInPixels = max(pointSize.width, pointSize.height) * scale
    let downsampleOptions = [
        kCGImageSourceCreateThumbnailFromImageAlways: true,
        kCGImageSourceShouldCacheImmediately: true,  // 立即解码（在当前线程，应在后台调用）
        kCGImageSourceCreateThumbnailWithTransform: true,
        kCGImageSourceThumbnailMaxPixelSize: maxDimensionInPixels
    ] as CFDictionary
    
    guard let downsampledImage = CGImageSourceCreateThumbnailAtIndex(imageSource, 0, downsampleOptions) else {
        return nil
    }
    
    return UIImage(cgImage: downsampledImage)
}
```

### 3.3 主线程 I/O 卡顿

```swift
// 磁盘 I/O 在主线程执行会严重阻塞
// 实测：读取 1MB 文件约 5-20ms，写入约 10-50ms

// ❌ 问题：在主线程同步读写
func loadUserData() -> UserData? {
    let path = documentsPath + "/user.json"
    let data = FileManager.default.contents(atPath: path)  // 主线程 I/O！
    return data.flatMap { try? JSONDecoder().decode(UserData.self, from: $0) }
}

// ✅ 优化：异步 I/O + 内存缓存
class UserDataManager {
    private var memoryCache: UserData?
    private let queue = DispatchQueue(label: "com.app.userdata", qos: .utility)
    
    func loadUserData(completion: @escaping (UserData?) -> Void) {
        // 先返回内存缓存
        if let cached = memoryCache {
            completion(cached)
            return
        }
        
        // 后台线程读取磁盘
        queue.async { [weak self] in
            let path = self?.documentsPath ?? ""
            let data = FileManager.default.contents(atPath: path + "/user.json")
            let userData = data.flatMap { try? JSONDecoder().decode(UserData.self, from: $0) }
            
            DispatchQueue.main.async {
                self?.memoryCache = userData
                completion(userData)
            }
        }
    }
}

// ❌ 问题：在主线程同步访问数据库
func fetchUser(id: String) -> User? {
    return database.query("SELECT * FROM users WHERE id = ?", id)  // 主线程！
}

// ✅ 优化：数据库操作全部异步
func fetchUser(id: String) async -> User? {
    return await withCheckedContinuation { continuation in
        databaseQueue.async {
            let user = database.query("SELECT * FROM users WHERE id = ?", id)
            continuation.resume(returning: user)
        }
    }
}
```

## 四、离屏渲染深度分析

### 4.1 离屏渲染的八种场景

```
触发离屏渲染的操作（按性能影响从大到小）：

1. mask（遮罩）- 影响最大
   layer.mask = maskLayer
   需要先渲染遮罩和内容，再合成
   
2. 阴影（未设置 shadowPath）
   layer.shadowOpacity > 0 && layer.shadowPath == nil
   GPU 需要实时计算阴影轮廓（遍历所有像素）
   
3. cornerRadius + masksToBounds（圆角裁剪）
   layer.cornerRadius > 0 && layer.masksToBounds = true
   需要对子图层做裁剪
   
4. group opacity（组透明度）
   layer.allowsGroupOpacity = true && layer.opacity < 1
   需要先渲染整个子树，再整体设置透明度
   
5. shouldRasterize（主动光栅化）
   layer.shouldRasterize = true
   主动触发离屏渲染，但结果会被缓存
   
6. drawRect / draw(_:)（CPU 离屏渲染）
   重写了绘制方法
   CPU 在离屏 Bitmap 上绘制，再提交给 GPU
   
7. 渐变（CAGradientLayer）
   某些情况下触发（与其他属性组合时）
   
8. UIBlurEffect
   高斯模糊需要多 Pass 渲染
```

## 五、异步渲染

### 5.1 YYAsyncLayer 原理与实现

```swift
// 核心思路：
// 1. 主线程标记需要渲染，不立即渲染
// 2. 在 RunLoop BeforeWaiting 时，开启后台线程绘制
// 3. 后台线程绘制完成后，在主线程设置 layer.contents

class AsyncLayer: CALayer {
    
    weak var asyncDelegate: AsyncLayerDelegate?
    private var sentinel = Sentinel()  // 用于取消过期的渲染任务
    
    override func setNeedsDisplay() {
        // 取消之前的渲染任务
        sentinel.increment()
        super.setNeedsDisplay()
    }
    
    override func display() {
        super.contents = super.contents  // 保持当前内容，避免闪烁
        displayAsync()
    }
    
    private func displayAsync() {
        guard let delegate = asyncDelegate else { return }
        
        let size = bounds.size
        let scale = contentsScale
        let opaque = isOpaque
        let backgroundColor = self.backgroundColor
        let currentSentinel = sentinel.value  // 记录当前版本号
        
        // 在后台线程绘制
        DispatchQueue.global(qos: .userInitiated).async { [weak self] in
            guard let self = self else { return }
            
            // 检查是否已过期（setNeedsDisplay 被再次调用）
            if self.sentinel.value != currentSentinel { return }
            
            // 创建 Bitmap 上下文
            UIGraphicsBeginImageContextWithOptions(size, opaque, scale)
            guard let context = UIGraphicsGetCurrentContext() else { return }
            
            // 绘制背景
            if opaque, let bg = backgroundColor {
                context.setFillColor(bg)
                context.fill(CGRect(origin: .zero, size: size))
            }
            
            // 让 delegate 绘制内容
            delegate.drawInContext(context, size: size, isCancelled: {
                self.sentinel.value != currentSentinel  // 提供取消检查
            })
            
            // 再次检查是否过期
            if self.sentinel.value != currentSentinel {
                UIGraphicsEndImageContext()
                return
            }
            
            let image = UIGraphicsGetImageFromCurrentImageContext()
            UIGraphicsEndImageContext()
            
            // 回到主线程设置内容
            DispatchQueue.main.async {
                if self.sentinel.value == currentSentinel {
                    self.contents = image?.cgImage
                }
            }
        }
    }
}

protocol AsyncLayerDelegate: AnyObject {
    func drawInContext(_ context: CGContext, size: CGSize, isCancelled: () -> Bool)
}

// 使用示例
class AsyncTextView: UIView, AsyncLayerDelegate {
    
    var text: String = "" {
        didSet { layer.setNeedsDisplay() }
    }
    
    override class var layerClass: AnyClass {
        return AsyncLayer.self
    }
    
    override init(frame: CGRect) {
        super.init(frame: frame)
        (layer as? AsyncLayer)?.asyncDelegate = self
    }
    
    required init?(coder: NSCoder) { fatalError() }
    
    func drawInContext(_ context: CGContext, size: CGSize, isCancelled: () -> Bool) {
        guard !isCancelled() else { return }
        
        let attributes: [NSAttributedString.Key: Any] = [
            .font: UIFont.systemFont(ofSize: 14),
            .foregroundColor: UIColor.black
        ]
        
        let attrString = NSAttributedString(string: text, attributes: attributes)
        let framesetter = CTFramesetterCreateWithAttributedString(attrString)
        let path = CGPath(rect: CGRect(origin: .zero, size: size), transform: nil)
        let frame = CTFramesetterCreateFrame(framesetter, CFRangeMake(0, 0), path, nil)
        
        guard !isCancelled() else { return }
        
        CTFrameDraw(frame, context)
    }
}
```

## 七、常见面试题

### Q1：如何监控 App 的卡顿？有哪些方案？

```
主流方案：RunLoop Observer 监控

原理：
  主线程 RunLoop 在处理事件时（BeforeSources/AfterWaiting 状态），
  如果超过阈值时间没有状态变化，说明主线程卡住了。

实现：
  1. 注册 RunLoop Observer 监听所有状态变化
  2. 子线程用信号量等待，设置超时时间（如 50ms）
  3. 超时且处于事件处理阶段 → 确认卡顿
  4. 抓取主线程调用栈上报

其他方案：
  - CADisplayLink FPS 监控（只知道卡了，不知道原因）
  - Ping/Pong 监控（子线程定期 ping 主线程）
  - MetricKit（系统级，24h 延迟，有堆栈）
  - Hook objc_msgSend（精确但开销大，仅测试用）

推荐组合：
  线上：RunLoop Observer + MetricKit
  开发：FPS 悬浮窗 + Instruments Time Profiler
```

### Q2：离屏渲染是什么？哪些操作会触发？如何优化？

```
离屏渲染：
  GPU 需要先在临时缓冲区渲染，再合成到帧缓冲区。
  比正常渲染多一次上下文切换，性能开销约 10-100%。

触发操作（常见）：
  1. cornerRadius + masksToBounds（圆角裁剪）
  2. 阴影（未设置 shadowPath）
  3. mask（遮罩）
  4. group opacity（组透明度）
  5. shouldRasterize（主动光栅化）

优化方案：
  圆角：
    - 后台线程绘制圆角图片（最彻底）
    - 只设 cornerRadius，不设 masksToBounds（无子视图时）
    - 使用 cornerCurve = .continuous（iOS 13+）
  阴影：
    - 必须设置 shadowPath（告知 GPU 阴影形状）
  光栅化：
    - 适合：内容不变的复杂视图（如带阴影的卡片）
    - 不适合：内容频繁变化的视图

检测：
  模拟器 Debug → Color Off-screen Rendered（黄色高亮）
```

### Q3：ProMotion 120fps 适配需要注意什么？

```
ProMotion 设备（iPhone 13 Pro+）支持 1-120Hz 自适应刷新。

默认情况下 App 仍然以 60fps 运行，要获得 120fps 需要：

1. CADisplayLink 设置 preferredFrameRateRange
   displayLink.preferredFrameRateRange = CAFrameRateRange(
       minimum: 60, maximum: 120, preferred: 120
   )

2. 动画时长要考虑帧率
   - 60fps：16.67ms/帧
   - 120fps：8.33ms/帧
   - 动画在 120fps 下会"快一倍"，需要相应调整时长

3. 性能要求更高
   - 每帧预算从 16.67ms 降到 8.33ms
   - CPU 和 GPU 的工作量必须更小
   - 离屏渲染的影响更大

4. 测试要在真实设备上进行
   - 模拟器不支持 120fps
   - 低端设备（不支持 ProMotion）需要降级到 60fps
```


### 基于 CADisplayLink 的主线程卡顿监控实现

```
原理：利用 CADisplayLink 每帧回调的特性，通过比较相邻两帧的时间戳差值
来判断主线程是否卡顿。当差值超过 16.67ms（60fps 预算）时，说明发生了卡顿。

实现方案：

class FrameLagMonitor:
    private var displayLink: CADisplayLink!
    private var lastTimestamp: CFTimeInterval = 0
    private var frameCount: Int = 0
    private var lagFrames: [CFTimeInterval] = []
    private let lagThreshold: CFTimeInterval = 0.033  // 33ms = 2帧

    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(onTick(_:)))
        displayLink.add(to: .main, forMode: .common)
    }

    @objc private func onTick(_ link: CADisplayLink) {
        guard lastTimestamp > 0 else {
            lastTimestamp = link.timestamp; return
        }
        let interval = link.timestamp - lastTimestamp
        lastTimestamp = link.timestamp
        frameCount += 1
        if interval > lagThreshold {
            lagFrames.append(interval)
            // 上报卡顿：interval 表示卡顿持续时间
            reportLag(duration: interval)
        }
    }

    private func reportLag(duration: CFTimeInterval) {
        // 通过异步队列上报，避免影响主线程
        DispatchQueue.global().async {
            // 发送到遥测服务或本地存储
        }
    }
}

工作流程：
  主线程 ──→ CADisplayLink 每帧触发 ──→ 计算时间差 ──→ 超阈值则记录卡顿
  │                                                  │
  └──────────────────────────────────────────────────┘
  卡顿发生在主线程阻塞期间，displayLink 回调延迟触发

Pitfalls: CADisplayLink 在主线程被阻塞时不会触发回调，因此只能检测到"回调触发后"的卡顿间隔，无法捕获主线程完全冻结的场景。对于长时间冻结（>1s），应配合 RunLoop Observer 使用。

Best practice: 生产环境中应限制采样频率（如每 3 秒统计一次），避免高频上报影响性能。结合 MetricKit 的 XMetricUserInfo 进行补充分析。
```


### CADisplayLink 时间戳帧时间测量

原理：`CADisplayLink` 的 `timestamp` 属性基于 `CFAbsoluteTime`（即 `mach_absolute_time` 转换），其精度可达纳秒级。通过连续两帧的时间戳差值，可以精确测量每帧的实际渲染间隔，进而计算出真实 FPS 和卡顿持续时间。

```swift
class FrameTimeAnalyzer {
    private var displayLink: CADisplayLink!
    private var frameTimes: [Double] = []
    private var lastTimestamp: CFTimeInterval = 0

    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(onTick(_:)))
        displayLink.add(to: .main, forMode: .commonModes)
    }

    @objc private func onTick(_ link: CADisplayLink) {
        guard lastTimestamp > 0 else { lastTimestamp = link.timestamp; return }
        let frameTime = link.timestamp - lastTimestamp  // 纳秒级精度
        lastTimestamp = link.timestamp
        frameTimes.append(frameTime)
        if frameTimes.count > 60 { frameTimes.removeFirst() }  // 滑动窗口
        let avg = frameTimes.reduce(0, +) / Double(frameTimes.count)
        let currentFPS = 1.0 / frameTime  // 当前帧 FPS
        print("Frame: \(frameTime*1000)ms, FPS: \(currentFPS), Avg: \(avg*1000)ms")
    }
}
```

工作流程：
  Vsync 信号 ──→ CADisplayLink 触发 ──→ 读取 timestamp ──→ 计算与前帧差值 ──→ 记录/上报
  │                                                       │
  │                                  差值 = 16.67ms 表示 60fps 正常
  │                                  差值 = 33.34ms 表示丢 1 帧（卡顿）
  │                                  差值 = 50ms 表示丢 2 帧（严重卡顿）

Pitfalls: `timestamp` 基于设备时钟而非 RunLoop 调度时钟，主线程阻塞期间 displayLink 不会触发回调，因此只能测量"回调触发后"的间隔。对于主线程完全冻结（>1s），timestamp 差值会暴增，此时应配合 RunLoop Observer 或 DispatchSource 进行补充检测。

Best practice: 生产环境中使用滑动窗口（如最近 60 帧）计算 FPS 均值和中位数，避免瞬时波动。结合 `preferredFrameRate` 和 `frameInterval` 属性适配 ProMotion 设备（120Hz），在 iOS 17+ 中可配合 `WAKit` 或 `MetricKit` 进行补充分析。

### iOS 16+ TaskInspector / ThreadPerformanceMetrics

原理：`ThreadPerformanceMetrics` 是 iOS 16 引入的基于 Instrumentation 框架的线程性能度量 API。通过定义 `ThreadPerformanceRecord` 后调用 `ThreadPerformanceMetrics(records:for:)`，可以获取线程在指定时间间隔内的调度时间（dispatch time）、运行时间（run time）、等待时间（wait time）和未调度时间（undispatched time），无需导入 private API 或依赖 Instrument.app。

```swift
import Instrumentation

class MainThreadMonitor {
    private var record: ThreadPerformanceRecord?

    func start() {
        record = ThreadPerformanceRecord()
        record?.startRecord()
        // 每 2 秒采样一次
        timer = Timer.scheduledTimer(withTimeInterval: 2.0, repeats: true) { [weak self] _ in
            self?.sample()
        }
    }

    func sample() {
        guard let record = record else { return }
        let metrics = ThreadPerformanceMetrics(records: [record], for: .main)
        let dispatch = metrics.dispatchedTimeInterval  // 线程获得调度器的时间
        let run = metrics.runTimeInterval               // 线程实际执行的时间
        let wait = metrics.waitingTimeInterval          // 等待调度器的时间
        let undispatched = metrics.undispatchedTimeInterval  // 未调度时间
        let utilization = run.duration / dispatch.duration  // 利用率 = 运行/调度
        print("主线程: 调度\(dispatch.duration*1000)ms 运行\(run.duration*1000)ms 等待\(wait.duration*1000)ms 利用率\(utilization)")
    }
}
```

工作流程：
  创建 Record ──→ startRecord() ──→ Instrumentation 内核采集 ──→ 定期调用 Metrics(records:for:) ──→ 获得 4 维时间数据
  │                                                                             │
  │                                                        Dispatch = Run + Wait（调度器分配给线程的总时间）
  │                                                        Utilization = Run / Dispatch（越高说明线程越繁忙）
  │                                                        Utilization 接近 1.0 = 主线程持续繁忙 = 卡顿风险

Pitfalls: `ThreadPerformanceMetrics` 只能测量 Record 开始之后的数据，无法回溯历史。多次调用 `metrics` 返回的是自上次调用以来的增量数据（delta），不是累计值。Record 必须在目标线程（如主线程）上调用 `startRecord()` 才能正确关联。

Best practice: 配合 RunLoop Observer 使用，利用 `dispatchedTimeInterval` 与 `runTimeInterval` 的比值判断主线程是否"被占满"。比值 > 0.9 持续超过 3 秒即触发卡顿预警。在 iOS 17+ 中可与 `ProcessInfo.processInfo.thermalState` 结合，在 Low Power Mode 下降低采样频率以节省电量。
### 基于 RunLoop Observer 的卡顿检测

原理：通过 `CFRunLoopObserver` 注册到主线程 RunLoop 的多个阶段（Entry/Before/After/Exit/BeforeWaiting/Exit），在每个阶段打时间戳。通过计算关键阶段的间隔（如 Before→After 或 AfterWaiting→Entry），可以精确检测主线程是否在主任务执行期间发生了长时间阻塞。相比 CADisplayLink 只能检测"帧是否准时"，RunLoop Observer 能定位"卡在哪一段"。

```swift
class RunLoopJankDetector {
    private var observer: CFRunLoopObserver?
    private var lastEntry: DispatchTime = .now()
    private var lastBefore: UInt64 = 0
    private var lastAfter: UInt64 = 0
    private var jankThreshold: Double = 0.016  // 16.67ms = 60fps 一帧

    func start() {
        observer = CFRunLoopObserverCreateWithHandler(kCFAllocatorDefault,
            CFRunLoopActivity.everyActivity.rawValue, true) { [weak self] _, activity in
            guard let self = self else { return }
            let now = UInt64(DispatchTime.now().uptimeNanoseconds)
            switch activity {
            case .entry:
                let delta = Double(now - self.lastEntry) / 1e6  // ms
                if delta > self.jankThreshold * 3 {  // 超过 3 帧 = 卡顿
                    print("[JANK] 主线程阻塞: \(delta)ms 后恢复")
                }
                self.lastEntry = DispatchTime.now()
            case .beforeSources:
                self.lastBefore = now
            case .afterSources:
                self.lastAfter = now
                let taskTime = Double(now - self.lastBefore) / 1e6
                if taskTime > self.jankThreshold {
                    print("[JANK] 主任务耗时: \(taskTime)ms")
                }
            default: break
            }
        }
        CFRunLoopAddObserver(CFRunLoopGetMain(), observer, .commonModes)
    }
}
```

工作流程：
  RunLoop 循环: Entry ──→ Sources/Signals ──→ Timers ──→ Input ──→ AfterWaiting ──→ Exit
  │             │                       │                       │                       │
  │    打时间戳  │    打时间戳            │    打时间戳            │    打时间戳            │
  │    计算间隔  │    计算间隔            │    计算间隔            │    计算间隔            │
  │    = 总阻塞  │    = 源处理时间        │    = 定时器触发间隔    │    = 空闲等待时间      │

Pitfalls: RunLoop 在 `kCFRunLoopCommonModes` 下滚动页面或触发 Drag 时会切换到 `UITrackingRunLoopMode`，Observer 在默认 Mode 下不会触发，必须在 `.commonModes` 注册。另外 RunLoop 卡顿检测**无法捕获主线程被 Dispatch Semaphore 或 @synchronized 死锁**的场景（此时 RunLoop 本身也冻结），需配合 DispatchSource 或子线程心跳检测。

Best practice: 生产环境中将卡顿数据写入环形缓冲区（最多 200 条），触发 Threshold 后通过 `Thread.callStackSymbols` 抓取调用栈快照，结合 `os_unfair_lock` 或 Semaphore 检测定位具体阻塞源。iOS 17+ 可配合 `ThreadPerformanceMetrics` 交叉验证，避免单一检测方式的盲区。

### 隐式动画导致布局卡顿

原理：在 `UIView` 的隐式动画上下文（`UIView.animate` / `beginAnimations`）中，任何 layer 属性的变化（frame、transform、opacity 等）都会被自动包裹进 Core Animation 隐式事务。当你在主线程执行复杂的布局计算（如 `layoutIfNeeded()` 或手动遍历 subviews 设置 frame）时，**每一个 subview 的属性变化都会触发独立的隐式动画**，导致 Core Animation 在主线程合成器队列中堆积大量隐式 transaction，最终造成页面卡顿甚至主线程阻塞。

```swift
// ❌ 反模式：在 animate 块中手动遍历 100 个 subview 设置 frame
UIView.animate(withDuration: 0.3) {
    for subview in self.container.subviews {
        subview.frame = computeLayout(for: subview)  // 每个 subview 触发一次隐式动画
    }
}

// ✅ 方案 1：使用 layoutIfNeeded() 批量触发（只生成一次隐式动画事务）
UIView.animate(withDuration: 0.3) {
    self.container.layoutIfNeeded()  // 所有 Auto Layout 变化合并为一次 transaction
}

// ✅ 方案 2：手动管理 CATransaction，避免隐式动画触发
CATransaction.begin()
CATransaction.setDisableActions(true)  // 禁用隐式动画
for subview in self.container.subviews {
    subview.layer.frame = computeLayerFrame(for: subview)  // 无动画直接跳转
}
CATransaction.commit()

// ✅ 方案 3：dispatchPreventive 预布局 + 显式动画
self.container.layoutIfNeeded()  // 先在主线程外预计算布局
UIView.animate(withDuration: 0.3) {
    self.container.layoutIfNeeded()  // 再触发一次带动画的 layout
}
```

工作流程：
  ❌ 隐式动画卡顿: 遍历 subview → 每改一个 frame → Core Animation 插入隐式 transaction → 合成器排队 100 次 → 主线程卡顿
  ✅ 批量 layoutIfNeeded: 遍历 subview → 标记 layout dirty → layoutIfNeeded() 触发一次 transaction → 合成器只处理一次 → 流畅

Pitfalls: 即使是 `layer.anchorPoint` 或 `layer.bounds` 这类"不起眼"的属性变化，在隐式动画上下文中也会触发 Core Animation 的隐式事务。`isUserInteractionEnabled = false` 的 view 同样会触发隐式动画，禁用交互并不能避免这个问题。

Best practice: 涉及批量 subview 布局变化时，优先使用 Auto Layout + `layoutIfNeeded()` 在 animate 块中触发。手动设置 frame 的场景使用 `CATransaction.setDisableActions(true)` 显式关闭隐式动画。对于页面首次加载（viewDidAppear 之前），始终禁用隐式动画避免白屏时的卡顿。iOS 17+ 可配合 `UIViewPropertyAnimator` 实现可中断的显式动画，避免隐式事务的不可控性。

### 过度绘制检测与优化

原理：过度绘制（Overdraw）指同一像素在单个帧中被多次绘制。每多一次绘制，GPU 就需要多执行一次 fragment shader 运算和 blend 操作。Xcode 提供内置的 Overdraw 调试工具（`Debug Color Blender Overdraw`），通过颜色叠加直观展示绘制次数：蓝色 = 1 次（理想），绿色 = 2 次，黄色 = 3 次，红色 = 4 次（应消除），深红色 = 5 次以上（严重问题）。

```swift
// 程序化检测：通过 Instrument 或自定义 hook 统计 layer 绘制次数
class OverdrawDetector: NSObject {
    private var originalDisplay: @convention(block) () -> Void = {}
    private var overdrawViews: [UIView: Int] = [:]
    
    func enable() {
        let original = UIView.displayIfNeeded
        // 通过 method_swizzling 或 Core Animation delegate 追踪 display 调用
        UIViewLayerDelegate.shared.displayCallback = { layer in
            guard let view = layer.value(forKey: "view") as? UIView else { return }
            self.overdrawViews[view, default: 0] += 1
            if self.overdrawViews[view, default: 0] > 3 {
                print("[OVERDRAW] \(type(of: view)) 绘制次数: \(self.overdrawViews[view]!)")
            }
        }
    }
}

// 运行时启用/禁用（通过环境变量或 build config 控制）
#if DEBUG
    override var debugDescription: String {
        layer.debugOverdraw = true  // iOS 17+ 新增 API
        return super.debugDescription
    }
#endif
```

工作流程：
  页面渲染: BackgroundLayer(black) → ContainerLayer(white) → ContentViewLayer(content) → 叠加 3 次绘制
  优化后: BackgroundLayer(clear) → ContainerLayer(content) → 叠加 2 次绘制（去掉不必要的背景层）

Pitfalls: 页面首次加载时的 Overdraw 通常高于滚动稳态（因为 view 未懒加载完毕），检测时应以稳态滚动数据为准。`isOpaque = true` 配合 `backgroundColor = nil` 会导致 Core Animation 强制多一次合成 pass 来合成背景色，反而增加 Overdraw。

Best practice: 页面背景统一使用 `UIColor.systemBackground`（系统自动处理多态适配），容器 view 按需设置 `isOpaque = true` 并显式指定 `backgroundColor`。对于列表页面，优先使用 UICollectionView 的 DiffableDataSource（iOS 13+）避免手动管理 cell 背景。Xcode 15 的 Frame Capture 可直接标注 Overdraw 热点区域，配合 Core Animation Instrument 定位具体 layer。

### Layer 光栅化（shouldRasterize）优缺点

原理：`shouldRasterize = true` 让 Core Animation 将 layer 及其所有 sublayer 预渲染到一张离屏纹理（bitmap），后续帧直接复用该纹理合成，避免重复执行 sublayer 的绘制逻辑。以空间换时间，适合**静态或低频变化**的复杂 layer 层级。

```swift
// ✅ 正确使用：静态复杂 view（如带 shadow + 圆角 + 多 subview 的卡片）
class CardView: UIView {
    override func layoutSubviews() {
        super.layoutSubviews()
        layer.shadowPath = UIBezierPath(roundedRect: bounds, cornerRadius: 12).cgPath
        layer.shouldRasterize = true  // 预渲染到纹理
        layer.rasterizationScale = UIScreen.main.scale  // 避免 @2x/@3x 模糊
    }
}

// ❌ 错误使用：频繁变化的 view（如滚动列表 cell 或动画 view）
cell.layer.shouldRasterize = true  // 每帧 invalidate 纹理 → 反而触发离屏渲染
```

工作流程：
  ❌ 未光栅化: 每帧合成器遍历 sublayer 树 → 逐层执行 draw/transform/blend → 合成 final frame
  ✅ 光栅化: 首次渲染到纹理 → 后续帧直接 blit 纹理 → sublayer 变化时才 invalidate 纹理

Pitfalls: 光栅化本身触发**离屏渲染**（一次额外的 rendering pass），如果 layer 每帧都变化（如滚动或动画），纹理每帧 invalidate 反而比不光栅化更慢。`rasterizationScale` 默认 1.0，在 Retina 屏上会导致纹理模糊，必须显式设为 `UIScreen.main.scale`。

Best practice: 仅在 layer 内容**页面生命周期内基本不变**时使用（如 TabBar 图标容器、固定的 Header view）。配合 `shouldRasterize = true` 始终设置 `rasterizationScale = UIScreen.main.scale`。对于动态内容，优先使用 Core Animation 显式动画（隐式触发 GPU 加速合成）而非光栅化。iOS 17+ 可配合 `layer.allowsGroupOpacity` 进一步优化合成路径。滚动列表中**绝对不要**对 cell 启用光栅化。

### 离屏渲染（Off-screen Rendering）检测与规避

定义：当 Core Animation 需要在**主 framebuffer 之外**的临时缓冲区中合成 layer 时触发。常见触发条件包括：圆角 + 裁剪（`maskToBounds = true`）、阴影（`shadowPath`）、光栅化（`shouldRasterize`）、Grouped Transparency（多个半透明 sublayer 合成）、`shouldRasterize = true` 本身。

**检测手段**：

```swift
// 方法 1：Xcode 调试选项（开发阶段最快）
// Scheme → Run → Diagnostics → Color Offscreen-Rendered Yellow
// 黄色区域 = 触发离屏渲染的 layer

// 方法 2：Instrument Core Animation（发布前必做）
// 启用 "Color Offscreen-Rendered" 和 "Record Layer Info"
// 查看 DevTools 中每帧的 Offscreen Rendering 时间占比

// 方法 3：运行时检测（可集成到 CI/CD 或灰度监控）
class OffscreenRenderDetector: NSObject {
    private var displayLink: CADisplayLink!
    private var frameCount = 0
    private var offscreenFrames = 0
    
    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(onTick))
        displayLink.add(to: .main, forMode: .common)
    }
    
    @objc func onTick() {
        frameCount += 1
        guard frameCount % 60 == 0 else { return } // 每秒采样一次
        // 通过 Core Animation Instrument 的 PID 接口读取离屏渲染时间
        // 或使用 layer.presentation() 差异检测
    }
}
```

**规避策略**：

| 触发源 | 规避方案 | 性能提升 |
|--------|---------|---------|
| `maskToBounds + 圆角` | 改用 `UIGraphicsImageRenderer` 预渲染圆角图 或 iOS 11+ `layer.shape` | 避免每帧裁剪 |
| `shadowPath` | 预计算 `UIBezierPath` 赋值给 `shadowPath`（而非 `shadowBounds` 触发离屏） | 避免 Alpha 合成 |
| 多 sublayer 叠加 | 合并为单个 layer 或使用 `shouldRasterize`（仅限静态内容） | 减少合成 pass |
| 渐变背景 | 使用预渲染渐变图片 或 iOS 13+ `CAGradientLayer`（已 GPU 加速） | 避免 Alpha 混合 |

工作流程：
  ❌ 触发: BackgroundLayer → Alpha sublayer 合成到临时 buffer → 再合成到主 framebuffer（多一次 pass）
  ✅ 规避: 预渲染静态内容为纹理/图片 → 主 framebuffer 直接 blit（零离屏 pass）

Pitfalls: `maskToBounds = true` 配合 `transform`（如旋转/缩放）必定触发离屏渲染，因为 Core Animation 无法在主 framebuffer 中裁剪变换后的 layer。即使是固定的圆角（`cornerRadius`），只要同时设置了 `maskToBounds = true` 且有 sublayer 超出圆角范围，就会触发。Xcode 的 Yellow 检测在 Release 模式下**不工作**（dwarf 调试信息未包含 Core Animation 标注），必须在 Debug 或 Profile 下测试。

Best practice: 页面开发完成后，用 Core Animation Instrument 录制 3 秒稳态滚动，Off-screen Rendering 时间应 < 每帧 1ms（60fps 预算 16.67ms 的 6%）。对于卡片圆角 + 阴影的场景，优先使用 iOS 13+ 的 `view.clipsToBounds = true` 配合预计算的 `layer.shadowPath`（避免 Alpha 合成触发离屏），或使用 SwiftUI 的 `.clipShape()`（基于 Metal 路径裁剪，不触发 Core Animation 离屏）。iOS 17+ 的 SwiftUI view 默认使用基于 GPU 的合成路径，离屏渲染触发条件与 UIKit 不同，需单独测试。

### Core Animation layer actions 与事务时序

**Layer Actions**: Core Animation 通过 `CALayer.actions` 字典拦截隐式动画触发点。当 layer 属性改变时，CA 查找 actions 字典中对应的 `CAAction` 来决定如何响应（播放动画、直接赋值、或忽略）。

**事务机制**: `CATransaction` 管理隐式动画的批次提交。每个 Run Loop 循环末尾自动提交隐式事务，显式调用 `CATransaction.begin()/commit()` 可手动控制批次。

```swift
// 禁用隐式动画（页面初始化必备）
CATransaction.begin()
CATransaction.setDisableActions(true)  // 全局禁用
layer.bounds = newBounds               // 无动画跳转
layer.position = newPosition
CATransaction.commit()

// 自定义 action 拦截（高级用法）
layer.actions = [
    "opacity": CABasicAnimation(keyPath: "opacity"),  // 显式定义
    "transform": nil,                                    // nil = 忽略（不触发隐式动画）
    "frame": NSNull()                                    // NSNull = 使用默认行为
]

// 事务嵌套与完成回调
CATransaction.begin()
CATransaction.setAnimationDuration(0.3)
CATransaction.setCompletionBlock {
    // 动画结束后执行（在主线程）
    layer.borderColor = UIColor.green.cgColor
}
layer.opacity = 0.0
CATransaction.commit()
```

**事务嵌套模型**:
```
  begin()    ← 外层事务（默认 2s 持续时间）
    begin()  ← 内层事务（覆盖 0.3s）
    commit() ← 内层提交（不执行，等待外层）
  commit()   ← 外层提交（触发所有动画，使用内层 0.3s 设置）
```

Pitfalls: `setDisableActions(true)` 仅作用于**当前事务层级**，内层 begin/commit 会恢复默认行为。页面初始化时未禁用 actions 导致 view 属性变化触发隐式动画是卡顿 top 3 原因。`completionBlock` 在主线程执行，避免在其中进行重型计算。

Best practice: 页面 `viewDidLoad` 或 view 初始化代码中统一包裹 `CATransaction.setDisableActions(true)`，避免 view 布局触发不必要的隐式动画。对于页面过渡动画，使用显式 `UIView.animate()` 或 `CABasicAnimation` 替代隐式动画，获得精确的时序控制和 cancel 能力。iOS 17+ SwiftUI 基于显式事务模型（`withAnimation {}`），不再有隐式动作概念。

### ARKit / Metal 渲染管线基础

**ARKit 渲染架构**: ARKit 基于 Metal 构建，通过 `ARSession` 管理相机帧捕获、特征点追踪和平面检测。每一帧的渲染流程涉及多个阶段的 GPU 调度。

**Metal 渲染管线**:
```
  Camera Frame → ARKit 处理 → SceneKit/RealityKit/自定义 Metal 渲染
  │              │              │
  │         特征点追踪    虚拟内容合成
  │         平面检测      光照估计
  │              │              │
  └──→ ARFrame ──┘──→ MTLLibrary/MTLCommandBuffer ──→ 合成输出
```

**Metal 渲染管线**: 基于显式命令缓冲模型。`MTLCommandQueue` 创建 `MTLCommandBuffer`，通过 `MTLRenderCommandEncoder` 编码绘制命令（vertex/fragment shader），最终 `commit()` 提交给 GPU 执行。

```swift
// Metal 渲染循环（每帧调用）
func render(frame: ARFrame) {
    guard let commandBuffer = commandQueue.makeCommandBuffer() else { return }
    let descriptor = frame.displayLink.currentRenderPassDescriptor!
    guard let encoder = commandBuffer.makeRenderCommandEncoder(descriptor: descriptor) else { return }
    encoder.setRenderPipelineState(pipelineState)
    encoder.setVertexBytes(vertices, length: vertices.count * MemoryLayout<Vertex>.stride, index: 0)
    encoder.setFragmentBytes(fragmentData, length: fragmentData.count * MemoryLayout<Float>.stride, index: 0)
    encoder.setVertexTexture(texture, index: 0)  // 相机纹理
    encoder.drawPrimitives(type: .triangle, vertexStart: 0, vertexCount: vertices.count)
    encoder.endEncoding()
    commandBuffer.present(frame.displayLink.displayLayerDrawable)  // 提交到 Core Animation 合成
    commandBuffer.commit()  // 发送到 GPU
}
```

**ARKit 多相机与传感器融合**: iOS 15+ 支持多相机（前置/后置/超广角/ToF/TrueDepth），通过 `ARFrame.anchor` 提供 6DoF 追踪数据。`ARWorldTrackingConfiguration` 结合 IMU（加速度计+陀螺仪）、视觉惯性里程计（VIO）实现亚毫米级定位。

**Pitfalls**: Metal 渲染中 `commandBuffer.commit()` 后仍可随时通过 `waitUntilCompleted` 阻塞等待 GPU 完成，但在渲染循环中调用会导致帧率暴跌。ARKit 的 `session(_:didUpdate:)` 回调在 Run Loop 早期触发，距离 VSync 通常只剩 8-10ms 用于 Metal 渲染，超出即丢帧。

**Best practice**: AR 场景优先使用 SceneKit/RealityKit（基于 Metal 的高级抽象），避免手动管理 Metal 管线。自定义 Metal 渲染时，预分配 `MTLCommandBuffer` 池避免每帧 alloc 开销。使用 `MTLLibrary` 预编译 shader（`newDefaultLibrary()` 或 `makeLibrary(path:)`），避免运行时编译卡顿。RealityKit 的 `Entity` 系统基于 Scene Graph 自动批处理 draw call，通常比手动 Metal 渲染更高效。iOS 17+ 的 `RealityKit 2` 支持基于物理的实时全局光照（通过预计算光照探针），显著提升 AR 真实感。

### SwiftUI 渲染优化（稳定标识与 diff 算法）

**SwiftUI 基于声明式 UI 模型**: 视图是值类型的 struct，每次 state 变化触发 body 重新计算。SwiftUI 通过 diff 算法比较新旧 view hierarchy，仅更新差异部分，避免全量重绘。

**diff 算法核心**: SwiftUI 使用基于树的 diff 策略（类似 React 的 Fiber 架构简化版）。对于 List/ForEach 等集合视图，**默认通过元素在数组中的位置（index）进行匹配**，导致插入/删除/重排时触发不必要的 view 重建。

**稳定标识（id: 参数）**: 通过显式提供 `id: \.$id` 或自定义 identifier，SwiftUI 能在 diff 阶段通过 identifier 匹配 view 实例，实现 O(n) 级别的精确差异计算。

```swift
// ❌ 未提供 id - 基于 index 匹配，删除中间元素触发全量重建
List(items) { item in
    ItemView(item: item)
}

// ✅ 提供稳定 id - 基于 identifier 匹配，仅删除/新增受影响行
List(items, id: \.$id) { item in
    ItemView(item: item)
}

// ✅ Item 实现 Identifiable 协议（推荐方式）
struct Item: Identifiable {
    let id: UUID  // 或数据库主键
    let name: String
}
List(items) { item in  // 自动使用 .id 进行 diff
    ItemView(item: item)
}
```

**diff 触发时机**: SwiftUI 在 Run Loop 的 display link 回调中执行 diff（通常距离 VSync 约 8-12ms），diff 结果通过 Core Animation 合成层提交。diff 超过 2ms 即可能导致主线程卡顿。

**Pitfalls**: 未实现 Identifiable 或未提供 `id:` 参数时，List 中任意位置的插入/删除/重排都会触发所有 visible row 的 view 重建（包括其子 view 的所有 state 重置），这是 SwiftUI 列表卡顿的 top 2 原因。即使是 `LazyVStack` 也遵循同样的 diff 逻辑。

**Best practice**: 所有集合元素 struct 统一实现 `Identifiable` 协议，使用业务主键（数据库 ID/UUID）作为 identifier 而非数组索引。对于频繁变动的列表（如聊天消息 feed），配合 `@StateObject`/`@Observable(iOS 17+)` 实现细粒度 state 隔离，避免整个 view 树的无效 diff。iOS 17 的 `@Observable` 宏通过 compile-time 生成 change tracking 代码，diff 精度进一步提升到 property 级别。


### 减少合成层数量

**合成层（Layer/Render Layer）**: Core Animation 中每个 `CALayer` 默认可能触发独立的 GPU 合成。合成层越多，GPU 合成阶段的 memory bandwidth 和 overdraw 越高，直接导致掉帧。通过 `debugOptions = .showViewsBounds` 或 Reveal/Instruments 的 Core Animation 模板可直观查看 layer 数量。

**触发合成层隔离的常见属性**: 满足以下任一条件，`CALayer` 会被提升为独立合成层（Layer Promoted）：
- `shouldRasterize = true`（光栅化，牺牲分辨率换合成性能）
- `opacity < 1.0`（半透明 view）
- `transform` 包含 3D 变换（`m34 != 0` 或 `CATransform3D` 非平面）
- `mask` 或 `shadow`（`layer.shadowPath` 触发隔离合成）
- `zPosition` 不同（多深度平面合成）

```swift
// ❌ 触发合成层隔离：每个 Item 都是独立 layer
itemLayer.shadowOpacity = 0.3  // 触发隔离合成
itemLayer.shouldRasterize = true  // 触发隔离合成（但通常不推荐）

// ✅ 避免隔离合成：使用 Core Graphics 手动绘制阴影
override func draw(_ rect: CGRect) {
    guard let ctx = UIGraphicsGetCurrentContext() else { return }
    ctx.setShadow(offset: CGSize(width: 0, height: 2), blur: 4)
    // 绘制内容后清除 shadow 避免 overdraw
    ctx.setShadow(offset: .zero, blur: 0)
}

// ✅ 避免隔离合成：使用 UIView 的 layer.mask 替代 view 嵌套
// ❌ 嵌套 3 层 view 实现圆角 + 渐变背景（3 个合成层）
```

**合成层排查**: Instruments → Core Animation → "Color Layer Hit" 查看 layer 隔离情况。页面合成层超过 50 个通常触发 GPU 瓶颈（低端设备 30 个即可能卡顿）。通过 `layer.debugDescription` 或 Reveal 检查 view 层级深度。

**Pitfalls**: `shouldRasterize = true` 在 @3x 屏幕上以 @1x 分辨率光栅化后缩放，视觉上明显模糊；且 rasterizeScale 不随设备动态变化，牺牲清晰度换取合成加速通常得不偿失。

**Best practice**: 页面合成层目标控制在 20 个以内（低端设备 15 个）。优先通过 view 合并（减少嵌套）、CG 手动绘制替代 layer effect（shadow/mask/opacity）、避免 3D transform 来降低合成层。iOS 17+ 的 SwiftUI 通过 view 合并优化（view merger pass）自动减少不必要的 layer 创建，但显式触发隔离的属性（opacity/transform/mask）仍会突破 merger 优化。
