# 组件化与工程架构详解

---

## 一、组件化的背景与目标

### 1.1 为什么需要组件化

```
大型 App 的工程痛点（以美团外卖为例，代码量 200 万行+）：

1. 编译速度慢
   - 单工程全量编译：20-40 分钟
   - 修改一个文件可能触发大量重编译
   - 严重影响开发效率

2. 团队协作困难
   - 多个团队修改同一代码库，冲突频繁
   - 代码边界不清晰，相互依赖混乱
   - 无法并行开发

3. 代码复用差
   - 同一功能在不同业务线重复实现
   - 基础能力无法跨 App 共享

4. 测试困难
   - 无法单独测试某个模块
   - 改一个地方，不知道影响哪些功能

组件化的目标：
  - 高内聚低耦合：每个组件独立，依赖最小化
  - 独立编译：每个组件可以单独编译和测试
  - 并行开发：多团队可以同时开发不同组件
  - 二进制化：稳定组件预编译为二进制，加快整体编译
```

### 1.2 组件化的层次结构

```
典型的大型 App 组件层次：

  ┌─────────────────────────────────────────────────────────┐
  │                    业务组件层                             │
  │  首页  │  搜索  │  订单  │  支付  │  用户中心  │  营销    │
  ├─────────────────────────────────────────────────────────┤
  │                    业务基础层                             │
  │  登录  │  地图  │  直播  │  IM  │  评价  │  优惠券       │
  ├─────────────────────────────────────────────────────────┤
  │                    平台服务层                             │
  │  路由  │  埋点  │  AB实验  │  推送  │  日志  │  网络      │
  ├─────────────────────────────────────────────────────────┤
  │                    基础组件层                             │
  │  UI组件库  │  网络库  │  图片库  │  数据库  │  工具库     │
  └─────────────────────────────────────────────────────────┘

依赖规则：
  - 上层可以依赖下层
  - 同层之间通过路由/协议通信，不直接依赖
  - 下层不能依赖上层（依赖倒置）
```

## 二、路由方案深度对比

### 2.1 方案一：URL Router

```objc
// 原理：用 URL 字符串表示页面和操作，通过路由表映射到具体实现

// 注册路由
[[MTRouter shared] registerRoute:@"meituanwaimai://order/detail"
                         handler:^(MTRouterContext *ctx) {
    NSString *orderId = ctx.params[@"orderId"];
    OrderDetailViewController *vc = [[OrderDetailViewController alloc] initWithOrderId:orderId];
    [ctx.navigationController pushViewController:vc animated:YES];
}];

// 跳转
[[MTRouter shared] openURL:@"meituanwaimai://order/detail?orderId=12345"];

// 带回调
[[MTRouter shared] openURL:@"meituanwaimai://login"
                 withParams:@{@"source": @"checkout"}
                   callback:^(NSDictionary *result) {
    if ([result[@"success"] boolValue]) {
        [self proceedToCheckout];
    }
}];
```

```objc
// MTRouter 核心实现
@interface MTRouter : NSObject

@property (nonatomic, strong) NSMutableDictionary<NSString *, RouterHandler> *routes;

+ (instancetype)shared;

- (void)registerRoute:(NSString *)pattern handler:(RouterHandler)handler;
- (void)openURL:(NSString *)urlString;
- (void)openURL:(NSString *)urlString withParams:(NSDictionary *)params callback:(RouterCallback)callback;
- (BOOL)canOpenURL:(NSString *)urlString;

@end

@implementation MTRouter

- (void)openURL:(NSString *)urlString withParams:(NSDictionary *)params callback:(RouterCallback)callback {
    NSURL *url = [NSURL URLWithString:urlString];
    if (!url) return;
    
    // 解析 URL 参数
    NSMutableDictionary *allParams = [NSMutableDictionary dictionary];
    NSURLComponents *components = [NSURLComponents componentsWithURL:url resolvingAgainstBaseURL:NO];
    for (NSURLQueryItem *item in components.queryItems) {
        allParams[item.name] = item.value;
    }
    [allParams addEntriesFromDictionary:params];
    
    // 构建 Context
    MTRouterContext *ctx = [[MTRouterContext alloc] init];
    ctx.url = url;
    ctx.params = allParams;
    ctx.callback = callback;
    ctx.navigationController = [self topNavigationController];
    
    // 查找并执行 Handler
    NSString *pattern = [NSString stringWithFormat:@"%@://%@%@", url.scheme, url.host, url.path];
    RouterHandler handler = self.routes[pattern];
    if (handler) {
        handler(ctx);
    } else {
        NSLog(@"⚠️ 未注册的路由: %@", pattern);
    }
}

@end
```

```
URL Router 的优缺点：

优点：
  ✅ 解耦彻底，组件间不需要相互知道
  ✅ 支持 Deep Link 和 Universal Link（URL 格式天然支持）
  ✅ 可以跨 App 通信（URL Scheme）
  ✅ 灵活，运行时动态注册

缺点：
  ❌ 类型不安全（参数都是字符串，编译器无法检查）
  ❌ 没有编译期检查（URL 写错了运行时才发现）
  ❌ 参数传递复杂（复杂对象需要序列化）
  ❌ 维护成本高（URL 文档容易过时）

适用场景：
  - 需要支持 Deep Link 的场景
  - 跨 App 通信
  - H5 与原生通信（JSBridge）
```

### 2.2 方案二：Target-Action（CTMediator）

```objc
// 蘑菇街开源，Casa Taloyum 设计
// 原理：通过 Target 类 + Action 方法的方式调用，利用 ObjC Runtime 动态调用

// 核心思想：
// 每个组件暴露一个 Target_XXX 类，其中包含 Action_XXX 方法
// 调用方通过 CTMediator 传入 Target 名和 Action 名，Runtime 动态调用

// 组件 A 暴露的接口（Target 类）
// 注意：Target 类放在组件 A 内部，不需要暴露头文件
@interface Target_OrderModule : NSObject

// 跳转到订单详情页
- (UIViewController *)Action_nativeOrderDetailViewController:(NSDictionary *)params;

// 获取订单数量（返回非 VC 的数据）
- (NSNumber *)Action_getOrderCount:(NSDictionary *)params;

@end

@implementation Target_OrderModule

- (UIViewController *)Action_nativeOrderDetailViewController:(NSDictionary *)params {
    NSString *orderId = params[@"orderId"];
    OrderDetailViewController *vc = [[OrderDetailViewController alloc] initWithOrderId:orderId];
    return vc;
}

- (NSNumber *)Action_getOrderCount:(NSDictionary *)params {
    return @([OrderManager sharedManager].orderCount);
}

@end
```

```objc
// CTMediator 核心实现（简化版）
@implementation CTMediator

- (id)performTarget:(NSString *)targetName
             action:(NSString *)actionName
             params:(NSDictionary *)params
  shouldCacheTarget:(BOOL)shouldCacheTarget {
    
    // 构建 Target 类名和方法名
    NSString *targetClassString = [NSString stringWithFormat:@"Target_%@", targetName];
    NSString *actionString = [NSString stringWithFormat:@"Action_%@:", actionName];
    
    // 从缓存或 Runtime 获取 Target 实例
    NSObject *target = [self targetFromCache:targetClassString];
    if (!target) {
        Class targetClass = NSClassFromString(targetClassString);
        target = [[targetClass alloc] init];
    }
    
    if (!target) {
        // Target 不存在，调用 notFound 处理
        [self noTargetActionResponseWithTargetString:targetClassString
                                        selectorString:actionString
                                        originalParams:params];
        return nil;
    }
    
    SEL action = NSSelectorFromString(actionString);
    
    if (shouldCacheTarget) {
        self.cachedTarget[targetClassString] = target;
    }
    
    // 安全调用
    if ([target respondsToSelector:action]) {
        return [self safePerformAction:action target:target params:params];
    }
    
    // 方法不存在
    SEL actionNotFound = NSSelectorFromString(@"notFound:");
    if ([target respondsToSelector:actionNotFound]) {
        return [self safePerformAction:actionNotFound target:target params:params];
    }
    
    [self.cachedTarget removeObjectForKey:targetClassString];
    return nil;
}

@end

// 调用方的封装（Category 方式，类型安全）
// CTMediator+OrderModule.h（放在调用方，不在 OrderModule 中）
@interface CTMediator (OrderModule)
- (UIViewController *)orderModule_orderDetailViewControllerWithOrderId:(NSString *)orderId;
- (NSInteger)orderModule_orderCount;
@end

@implementation CTMediator (OrderModule)

- (UIViewController *)orderModule_orderDetailViewControllerWithOrderId:(NSString *)orderId {
    NSDictionary *params = @{@"orderId": orderId};
    id result = [self performTarget:@"OrderModule"
                             action:@"nativeOrderDetailViewController"
                             params:params
                  shouldCacheTarget:NO];
    return [result isKindOfClass:[UIViewController class]] ? result : nil;
}

- (NSInteger)orderModule_orderCount {
    id result = [self performTarget:@"OrderModule"
                             action:@"getOrderCount"
                             params:nil
                  shouldCacheTarget:NO];
    return [result integerValue];
}

@end

// 使用
UIViewController *vc = [[CTMediator sharedInstance] orderModule_orderDetailViewControllerWithOrderId:@"12345"];
[self.navigationController pushViewController:vc animated:YES];
```

```
Target-Action 的优缺点：

优点：
  ✅ 编译安全（通过 Category 封装后有类型检查）
  ✅ 无需头文件依赖（Runtime 动态调用）
  ✅ 参数可以传复杂对象（NSDictionary）
  ✅ 组件解耦彻底

缺点：
  ❌ 硬编码字符串（Target 名、Action 名）
  ❌ 调用方需要维护 Category（额外代码）
  ❌ 无法在编译期检查 Target/Action 是否存在
  ❌ 学习成本较高

适用场景：
  - 纯 ObjC 项目
  - 需要严格解耦的大型项目
```

### 2.3 方案三：Protocol-Class 注册

```swift
// 原理：通过协议定义接口，组件注册自己对协议的实现
// 调用方通过协议获取实现，不直接依赖具体类

// Step 1：在公共层定义协议
// 放在所有组件都依赖的基础层

public protocol OrderModuleProtocol {
    // 创建订单详情页
    func makeOrderDetailViewController(orderId: String) -> UIViewController
    // 获取订单数量
    var orderCount: Int { get }
    // 跳转到订单列表
    func showOrderList(from viewController: UIViewController)
}

// Step 2：订单组件实现协议
// 这个文件在 OrderModule 组件内

class OrderModuleImpl: OrderModuleProtocol {
    
    func makeOrderDetailViewController(orderId: String) -> UIViewController {
        return OrderDetailViewController(orderId: orderId)
    }
    
    var orderCount: Int {
        return OrderManager.shared.count
    }
    
    func showOrderList(from viewController: UIViewController) {
        let listVC = OrderListViewController()
        viewController.navigationController?.pushViewController(listVC, animated: true)
    }
}

// Step 3：注册实现（在 OrderModule 初始化时）
// 通常在 +load 或 App 启动时注册

extension OrderModuleImpl {
    static func register() {
        ModuleManager.shared.register(
            protocol: OrderModuleProtocol.self,
            implementation: OrderModuleImpl()
        )
    }
}

// Step 4：调用方通过协议获取实现
let orderModule = ModuleManager.shared.module(for: OrderModuleProtocol.self)
let vc = orderModule?.makeOrderDetailViewController(orderId: "12345")
```

```swift
// ModuleManager 实现
class ModuleManager {
    static let shared = ModuleManager()
    private var modules: [ObjectIdentifier: Any] = [:]
    
    func register<T>(protocol: T.Type, implementation: T) {
        let key = ObjectIdentifier(`protocol`)
        modules[key] = implementation
    }
    
    func module<T>(for protocol: T.Type) -> T? {
        let key = ObjectIdentifier(`protocol`)
        return modules[key] as? T
    }
}

// 自动注册（利用 +load 或 Swift 的 @_silgen_name）
// 更优雅的方式：使用属性包装器或 Macro 自动注册

@propertyWrapper
struct ModuleProvider<T> {
    let wrappedValue: T?
    
    init() {
        wrappedValue = ModuleManager.shared.module(for: T.self)
    }
}

// 使用
class CheckoutViewController: UIViewController {
    @ModuleProvider<OrderModuleProtocol> var orderModule
    
    func showOrderDetail() {
        guard let vc = orderModule?.makeOrderDetailViewController(orderId: "12345") else { return }
        navigationController?.pushViewController(vc, animated: true)
    }
}
```

```
Protocol-Class 的优缺点：

优点：
  ✅ 类型安全（编译期检查接口）
  ✅ 自然支持 Swift（协议是 Swift 的核心特性）
  ✅ 接口清晰，易于文档化
  ✅ 方便 Mock（测试时注入假实现）

缺点：
  ❌ 需要维护协议定义（接口变更需要同步更新）
  ❌ 注册时机需要管理（启动时注册可能影响启动速度）
  ❌ 不支持 Deep Link（需要配合 URL Router）

适用场景：
  - Swift 项目
  - 需要高类型安全的场景
  - 需要单元测试的模块
```

### 2.4 三种方案综合对比

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│              │  URL Router  │Target-Action │Protocol-Class│
├──────────────┼──────────────┼──────────────┼──────────────┤
│ 类型安全     │ ❌ 无        │ ⚠️ 部分      │ ✅ 完整      │
│ 编译检查     │ ❌ 无        │ ❌ 无        │ ✅ 有        │
│ Deep Link    │ ✅ 原生支持  │ ❌ 需额外    │ ❌ 需额外    │
│ 跨 App 通信  │ ✅ 支持      │ ❌ 不支持    │ ❌ 不支持    │
│ 参数传递     │ ⚠️ 字符串    │ ✅ 任意对象  │ ✅ 强类型    │
│ ObjC 支持    │ ✅ 好        │ ✅ 最好      │ ⚠️ 需桥接    │
│ Swift 支持   │ ✅ 好        │ ⚠️ 一般      │ ✅ 最好      │
│ 学习成本     │ 低           │ 中           │ 低           │
│ 维护成本     │ 中           │ 高           │ 中           │
└──────────────┴──────────────┴──────────────┴──────────────┘

实际选择：
  大型 ObjC 项目：URL Router + CTMediator（两者结合）
  Swift 项目：Protocol-Class + URL Router（处理 Deep Link）
  混编项目：三种方案都用，各司其职
```

## 三、组件间通信

### 3.1 事件总线（EventBus）

```swift
// 用于组件间的松耦合通信（一对多）

// 事件定义（放在公共层）
enum AppEvent {
    case userDidLogin(userId: String)
    case userDidLogout
    case orderStatusChanged(orderId: String, status: OrderStatus)
    case cartItemCountChanged(count: Int)
}

// 事件总线
class EventBus {
    static let shared = EventBus()
    
    typealias Handler = (AppEvent) -> Void
    private var handlers: [String: [WeakBox<AnyObject>]] = [:]
    private let lock = NSLock()
    
    // 订阅事件
    func subscribe<T: AnyObject>(_ eventType: String, observer: T, handler: @escaping (AppEvent) -> Void) {
        lock.lock()
        defer { lock.unlock() }
        
        let box = WeakBox(observer)
        handlers[eventType, default: []].append(box)
        
        // 存储 handler（通过关联对象）
        objc_setAssociatedObject(observer, eventType.hashValue as AnyObject, handler,
                                 .OBJC_ASSOCIATION_COPY_NONATOMIC)
    }
    
    // 发布事件
    func publish(_ event: AppEvent) {
        let key = eventKey(for: event)
        lock.lock()
        let boxes = handlers[key] ?? []
        lock.unlock()
        
        DispatchQueue.main.async {
            for box in boxes {
                guard let observer = box.value else { continue }
                if let handler = objc_getAssociatedObject(observer, key.hashValue as AnyObject) as? Handler {
                    handler(event)
                }
            }
        }
        
        // 清理已释放的观察者
        lock.lock()
        handlers[key]?.removeAll { $0.value == nil }
        lock.unlock()
    }
    
    private func eventKey(for event: AppEvent) -> String {
        switch event {
        case .userDidLogin: return "userDidLogin"
        case .userDidLogout: return "userDidLogout"
        case .orderStatusChanged: return "orderStatusChanged"
        case .cartItemCountChanged: return "cartItemCountChanged"
        }
    }
}

// 使用
class CartViewController: UIViewController {
    
    override func viewDidLoad() {
        super.viewDidLoad()
        
        EventBus.shared.subscribe("cartItemCountChanged", observer: self) { [weak self] event in
            if case .cartItemCountChanged(let count) = event {
                self?.updateCartBadge(count: count)
            }
        }
    }
}

// 订单模块发布事件
EventBus.shared.publish(.cartItemCountChanged(count: 3))
```

// 通过共享的数据存储实现组件间数据共享

// 定义共享数据接口（放在公共层）
protocol SharedDataStore {

class AppSharedDataStore: SharedDataStore {

        get { return _cartCount }
        set {
            _cartCount = newValue

        }
    }

    private var _cartCount: Int = 0
    var selectedAddress: Address?
}

## 四、大规模工程实践

### 4.1 组件化的落地步骤

```
从单体工程迁移到组件化的步骤：

Phase 1：梳理依赖关系（1-2 周）
  - 用工具（如 objc-dependency-visualizer）分析现有代码的依赖图
  - 识别循环依赖（这是最大的障碍）
  - 制定依赖分层规划

Phase 2：抽离基础组件（2-4 周）
  - 先抽离无依赖的工具库（Utils、Category）
  - 再抽离基础 UI 组件（UIKit 封装）
  - 最后抽离网络、数据库等基础服务

Phase 3：建立路由系统（1-2 周）
  - 选择路由方案并搭建基础设施
  - 梳理所有页面跳转关系
  - 逐步将 ViewController 的直接依赖改为路由调用

Phase 4：抽离业务组件（按业务线逐步推进，数月）
  - 每次抽离一个相对独立的业务模块
  - 每个模块抽离后验证功能完整性
  - 建立组件的独立测试工程

Phase 5：二进制化（稳定后）
  - 将稳定的组件预编译为二进制
  - 建立组件版本管理和发布流程
  - 持续优化编译速度

常见问题与解决：

循环依赖：
  A 依赖 B，B 依赖 A
  解决：将共同依赖的部分抽到更底层的组件 C
  或：通过协议/路由解耦，打破直接依赖

资源冲突：
  多个组件有同名资源文件
  解决：使用 resource_bundles（CocoaPods），资源隔离到各自的 Bundle

Category 加载问题：
  静态库中的 Category 在某些情况下不会被加载
  解决：在 Other Linker Flags 中添加 -ObjC 或 -all_load
```

### 4.2 组件版本管理

```ruby
# Podfile 中的组件版本管理

# 开发模式：本地路径（修改代码立即生效）
pod 'MTOrderModule', :path => '../MTOrderModule'

# 测试模式：指定分支
pod 'MTOrderModule', :git => 'https://git.company.com/ios/MTOrderModule.git',
                     :branch => 'feature/new-order-flow'

# 生产模式：指定版本（锁定版本，保证稳定）
pod 'MTOrderModule', '~> 2.3.0'

# 二进制模式（加速编译）
pod 'MTOrderModule', '~> 2.3.0', :binary => true  # 使用 cocoapods-binary 插件
```

```
组件版本发布流程：

1. 开发：在组件仓库的 feature 分支开发
2. 测试：创建集成测试工程，验证与其他组件的兼容性
3. 代码审查：PR → 审查 → 合并到 main
4. 版本标记：git tag 2.3.1，更新 podspec 版本号
5. 发布：pod repo push company-specs MTOrderModule.podspec
6. 通知：在内部渠道通知依赖方可以升级
7. 主工程升级：修改 Podfile，pod update MTOrderModule
```

### 4.3 编译速度优化

```
大型组件化项目的编译优化：

1. 组件二进制化
   - 稳定组件预编译为 .xcframework
   - 只有正在开发的组件保持源码
   - 效果：编译时间从 20 分钟 → 3-5 分钟

2. 增量编译优化
   - 减少头文件的引用（#import 改为 @class 前向声明）
   - 避免在 .h 文件中 import 大量头文件
   - 使用 @import 代替 #import（Module 编译，更快）

3. Swift 编译优化
   - Debug 模式使用增量编译（SWIFT_COMPILATION_MODE = incremental）
   - Release 模式使用 WMO（SWIFT_COMPILATION_MODE = wholemodule）
   - 减少复杂的类型推断（明确标注类型）

4. 并行构建
   - Xcode 默认支持并行构建多个 Target
   - 合理设置 Target 依赖关系，让无依赖的 Target 并行构建
   - 使用 Bazel 等构建系统支持远程缓存和分布式构建

5. 缓存
   - ccache（ObjC/C++ 编译缓存）
   - Xcode 14+ 的 Explicit Module Builds
   - CI 环境中缓存 DerivedData 目录
```

## 五、模块化通信的完整示例

### 5.1 美团外卖式的组件化实践

## 六、常见面试题

### Q1：组件化有哪些路由方案？各有什么优缺点？

```
三种主要方案：

1. URL Router（蘑菇街 MGJRouter 等）
   优点：支持 Deep Link，跨 App 通信，运行时动态注册
   缺点：类型不安全，参数都是字符串，没有编译检查

2. Target-Action（CTMediator）
   优点：解耦彻底，无需头文件依赖，通过 Category 封装后有类型安全
   缺点：字符串硬编码，学习成本高，维护 Category 有额外工作量

3. Protocol-Class 注册
   优点：类型安全，编译检查，Swift 友好，方便 Mock
   缺点：需要维护协议定义，不支持 Deep Link

实际项目中通常混合使用：
  - URL Router 处理 Deep Link 和 H5 跳转
  - Protocol-Class 或 Target-Action 处理组件间服务调用
```

### Q2：如何解决组件化中的循环依赖问题？

```
循环依赖是组件化最常见的障碍：
  A 模块依赖 B 模块，B 模块又依赖 A 模块

解决方案：

1. 下沉公共部分
   将 A 和 B 共同依赖的内容抽到更底层的组件 C
   A → C，B → C，A 和 B 不直接依赖

2. 协议解耦
   A 不直接依赖 B，而是依赖 B 定义的协议
   通过路由或 ModuleManager 获取 B 的实现

3. 事件总线
   A 和 B 都不直接依赖对方
   通过 EventBus 发布和订阅事件通信

4. 中间层
   创建一个中间模块 M，A 和 B 都依赖 M
   M 负责协调 A 和 B 的交互

实际操作：
  先用工具（objc-dependency-visualizer）画出依赖图
  找到循环依赖的节点，分析哪些是真正需要的依赖
  通过上述方法逐一解除循环依赖
```

### Q3：组件化后如何保证编译速度？

```
主要手段：组件二进制化

原理：
  将稳定的组件预编译为 .xcframework（包含 arm64 和 x86_64）
  主工程只需链接二进制，无需重新编译源码

实现：
  1. 识别稳定组件（变更频率低的基础组件）
  2. CI 在组件发版时自动构建 xcframework
  3. 上传到二进制存储（CDN/Artifactory）
  4. 修改 podspec，指向二进制地址

效果：
  100+ 组件的大型项目，编译时间从 20 分钟 → 3-5 分钟

其他优化：
  - 减少头文件引用（前向声明代替 import）
  - Swift 增量编译（Debug 模式）
  - ccache（ObjC/C++ 编译缓存）
  - 合理设置 Target 依赖，允许并行构建
```

### 4.4 SPM 用于模块化 iOS 项目（package.swift 最佳实践）

```
Swift Package Manager 不仅是第三方库管理器，更是 iOS 项目模块化的首选方案。
Xcode 14+ 原生支持 SPM 本地包（File-based），无需手动添加，拖入即集成。

基本项目结构：

  MyModules/
  ├── Package.swift              // 包定义（多模块出口）
  ├── Sources/
  │   ├── Core/                  // 基础模块（网络、UI 组件）
  │   │   ├── Network/
  │   │   │   ├── APIClient.swift
  │   │   │   └── NetworkManager.swift
  │   │   └── UI/
  │   │       └── ButtonStyle.swift
  │   ├── Features/              // 业务功能模块
  │   │   ├── UserProfile/
  │   │   │   └── UserProfileViewModel.swift
  │   │   └── Order/
  │   │       └── OrderListViewController.swift
  │   └── Shared/                // 跨业务共享（模型、工具）
  │       └── Models/
  │           └── User.swift
  └── Tests/
      ├── CoreTests/
      └── FeaturesTests/

多模块 Package.swift 最佳实践：

  // Package.swift
  // swift-tools-version: 5.9
  import PackageDescription

  let package = Package(
      name: "MyModules",
      platforms: [
          .iOS(.v15),             // 统一最低支持版本
      ],
      products: [
          // 业务模块只暴露需要的接口，隐藏实现细节
          .library(name: "Core", targets: ["Core"]),
          .library(name: "UserProfile", targets: ["UserProfile"]),
          .library(name: "Order", targets: ["Order"]),
      ],
      dependencies: [
          // 第三方依赖统一管理，避免 CocoaPods/Carthage 混用
          .package(url: "https://github.com/Alamofire/Alamofire.git", from: "5.8.0"),
      ],
      targets: [
          // 基础模块：不依赖业务
          .target(name: "Core",
                  dependencies: ["Alamofire"]),
          // 业务模块：依赖 Core 和 Shared
          .target(name: "UserProfile",
                  dependencies: ["Core", "Shared"]),
          .target(name: "Order",
                  dependencies: ["Core", "Shared"]),
          // 共享模型：零依赖
          .target(name: "Shared",
                  dependencies: []),
          // 测试目标：自动关联
          .testTarget(name: "CoreTests",
                      dependencies: ["Core"]),
      ]
  )

Pitfalls: 避免在 Package.swift 中使用 @testable 导入业务模块，测试应通过公开的 API 接口进行。
Best practice: 业务模块通过 SPI（@_spi 或正式 SPI 机制）限制内部接口暴露，主工程通过导入具体 library name 按需引用。
```


### 模块可见性（@_spi、访问级别）

Swift 模块系统通过访问控制修饰符（`private`、`fileprivate`、`internal`、`public`、`open`）定义接口边界。在组件化架构中，合理的访问级别是防止模块间紧耦合的第一道防线。

**SPI（Swift Package Internal）机制**：通过 `@_spi(<label>)` 标注，允许包作者将部分 `internal` 接口有条件地暴露给显式声明依赖该 SPI 的下游包。

```swift
// Core 模块定义 SPI 接口
@_spi(TestUtils)
internal func simulateNetworkLatency() { ... }

// Core 的 Package.swift 中声明 SPI 标签
.target(name: "Core",
        dependencies: [],
        exclude: [],
        swiftSettings: [.unsafeFlags(["-enable-experimental-feature SPIEndpoints"])]),

// 下游模块显式导入 SPI
@_spi(TestUtils) import Core
// 现在可以调用 simulateNetworkLatency()
```

**正式 SPI（Swift 5.9+ / Swift 6）**：Apple 正在推进标准化的 SPI 机制，未来将使用 `@_spi` 的正式替代品 `@_exported` 或基于 `Package.swift` 的 `spiStrategies` 配置。

```swift
// 未来 SPI 声明方式（Swift 6 预览）
// Package.swift
.spiProvider.Strategy.spiEndpoints()

// 模块内标注
@_spi(InternalAPI) public struct NetworkCache { ... }
```

| 访问级别 | 同模块 | 同包其他目标 | 导入后 | 继承/子类 |
|---------|--------|------------|--------|----------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| `fileprivate` | ✅（同文件） | ❌ | ❌ | ❌ |
| `internal` | ✅ | ✅（同包） | ❌ | ❌ |
| `public` | ✅ | ✅ | ✅（调用） | ❌ |
| `open` | ✅ | ✅ | ✅（调用+继承） | ✅ |

Pitfalls: 滥用 `public` 或 `open` 会破坏模块隔离，下游可能依赖未稳定的内部实现；`@_spi` 是非标准 API，随时可能变更。
Best practice: 默认 `internal`，业务模块通过 Protocol 定义契约接口（`public`），实现类保持 `internal`；测试/调试工具通过 SPI 按需暴露。


### 基于路由的导航（URL 路由、DeepLinkRouter）

路由机制是组件化的"神经系统"——通过统一的 URL Scheme 或 Deep Link 定义页面跳转契约，调用方无需导入目标模块，仅需触发路由地址即可完成跨模块导航。

**URL 路由 vs DeepLink 路由**：URL Scheme（`myapp://order/detail?id=123`）适合外部唤起（Safari、通知点击），DeepLink（基于 WKWebView 或 ScenePhase 的内部路由表）适合模块间跳转。现代架构通常两者并存，通过 Router 层统一调度。

```swift
// 路由定义：Protocol 约定跳转接口
protocol Routeable {
    var scheme: String { get }
    var path: String { get }
    var params: [String: Any]? { get }
}

// 订单详情页路由
struct OrderDetailRoute: Routeable {
    let orderId: String
    var scheme: String { "myapp" }
    var path: String { "/order/detail" }
    var params: [String: Any]? { ["id": orderId] }
    var url: URL {
        var comps = URLComponents(string: "\(scheme)://\(path)")!
        comps.queryItems = [URLQueryItem(name: "id", value: orderId)]
        return comps.url!
    }
}

// 全局 Router 注册表
final class DeepLinkRouter {
    static let shared = DeepLinkRouter()
    private var handlers: [String: (URL) -> Bool] = [:]
    private let lock = NSRecursiveLock()
    
    func register(path: String, handler: @escaping (URL) -> Bool) {
        lock.lock(); defer { lock.unlock() }
        handlers[path] = handler
    }
    
    func open(_ url: URL) -> Bool {
        lock.lock(); defer { lock.unlock() }
        guard let path = url.path(percentEncoded: false),
              let handler = handlers[path] else { return false }
        return handler(url)
    }
}
```

**注册示例（模块懒加载）**：
```swift
// Order 模块初始化时注册
OrderModule.init() {
    DeepLinkRouter.shared.register(path: "/order/detail") { url in
        guard let id = url.queryItems?.first(where: { $0.name == "id" })?.value else { return false }
        let vc = OrderDetailViewController(orderId: id)
        UIApplication.shared.keyWindow?.rootViewController?.present(vc, animated: true)
        return true
    }
}
```

| 路由类型 | 触发场景 | 优点 | 限制 |
|---------|---------|------|------|
| URL Scheme | 外部唤起（Safari/通知/URL Type） | 系统原生支持 | 参数传递受限，安全性低 |
| Universal Links | 网页跳转/Apple 生态 | 无需跳转 Safari，体验无缝 | 需配置 apple-app-site-association |
| DeepLink（内部表） | 模块间跳转 | 类型安全，可携带复杂参数 | 需自维护注册表 |
| 基于 Protocol 的路由 | 高级抽象 | 编译期检查，IDE 自动补全 | 仍需导入定义模块 |

Pitfalls: 路由注册顺序不确定（多模块并行初始化），未注册的路由静默失败无日志；URL Scheme 可被第三方劫持，敏感页面应配合 Universal Links 或白名单校验。
Best practice: 路由定义集中管理（Shared 模块定义 Routeable 结构），注册使用懒加载（模块首次访问时触发），配合路由中间件实现埋点/登录态检查/灰度发布。


### 基于协议的模块契约

模块间通过 Protocol 定义接口契约，调用方仅依赖协议而不导入实现模块，实现**编译期解耦**。这是 iOS 组件化的核心手法。

```swift
// ModuleInterface/OrderProtocol.swift（共享接口层）
public protocol OrderServiceProvider: AnyObject {
    func fetchOrderList(completion: @escaping ([Order]) -> Void)
    func fetchOrderDetail(id: String, completion: @escaping (Order?) -> Void)
}

// ModuleInterface/Order.swift（共享模型层）
public struct Order: Codable {
    let id: String
    let status: OrderStatus
    let items: [OrderItem]
}

// 调用方（Home 模块，仅导入 ModuleInterface）
import ModuleInterface
final class HomeViewModel {
    private weak var orderService: OrderServiceProvider? // 通过 DI 注入
    func loadOrders() {
        orderService?.fetchOrderList { [weak self] orders in
            self?.orders = orders
        }
    }
}

// 实现方（Order 模块，导入 ModuleInterface + 业务实现）
import ModuleInterface
final class OrderServiceImpl: OrderServiceProvider {
    func fetchOrderList(completion: @escaping ([Order]) -> Void) {
        API.fetchOrders { completion($0) }
    }
    func fetchOrderDetail(id: String, completion: @escaping (Order?) -> Void) {
        API.fetchOrder(id: id) { completion($0) }
    }
}
```

**注册到服务容器**：
```swift
final class ServiceRegistry {
    static let shared = ServiceRegistry()
    private var services: [String: Any] = [:]
    private let lock = NSRecursiveLock()
    @discardableResult func register<T: AnyObject>(_ service: T, forKey key: String = String(describing: T.self)) -> Bool {
        lock.lock(); defer { lock.unlock() }
        services[key] = service
        return true
    }
    func resolve<T: AnyObject>(_ type: T.Type, forKey key: String = String(describing: T.self)) -> T? {
        lock.lock(); defer { lock.unlock() }
        return services[key] as? T
    }
}
// 初始化：ServiceRegistry.shared.register(OrderServiceImpl())
// 调用：HomeViewModel().orderService = ServiceRegistry.shared.resolve(OrderServiceProvider.self)
```

| 契约类型 | 定义位置 | 导入关系 | 适用场景 |
|---------|---------|---------|---------|
| Protocol（接口协议） | ModuleInterface 静态库 | 调用方 + 实现方均导入 | 方法调用、数据获取 |
| Class（共享模型） | ModuleInterface 静态库 | 双方导入 | 跨模块数据传输 |
| Notification/URL | 无需导入 | 零依赖 | 事件通知、页面跳转 |

Pitfalls: 协议定义在共享库中会导致**接口变更需全量重新发布**；Protocol 方法签名变更时调用方可能静默通过（weak optional 方法未调用），应配合静态分析或接口版本号机制。
Best practice: 接口层（ModuleInterface）作为独立 SPM/静态库发布，版本号与业务模块解耦；Protocol 方法默认返回 nil/空数组避免 optional 污染；配合 DI 容器实现多实现切换（Mock/真实/灰度）。


### 服务注册模式（Service Registration）

服务注册是组件化的核心机制：模块启动时将自身能力以 `Protocol -> 实现类` 的形式注册到全局容器，调用方通过容器按需解析，实现**运行时解耦**而非编译时依赖。

```swift
// 1. 定义服务注册表（线程安全）
final class ServiceRegistry {
    static let shared = ServiceRegistry()
    private var services: [String: Any] = [:]
    private let lock = NSRecursiveLock()

    @discardableResult
    func register<T: AnyObject>(_ service: T,
                                 identifier: String = String(describing: T.self)) -> Bool {
        lock.lock(); defer { lock.unlock() }
        guard services[identifier] == nil else { return false } // 防止重复注册
        services[identifier] = service
        return true
    }

    func resolve<T: AnyObject>(_ type: T.Type,
                                identifier: String = String(describing: T.self)) -> T? {
        lock.lock(); defer { lock.unlock() }
        return services[identifier] as? T
    }
}

// 2. 模块注册（通常在 SceneDelegate 或 Plugin 加载时调用）
func loadOrderModule() {
    ServiceRegistry.shared.register(OrderServiceImpl())
    ServiceRegistry.shared.register(PaymentServiceImpl())
}

// 3. 调用方解析（零编译依赖）
final class HomeViewModel {
    private var orderService: OrderServiceProvider? {
        ServiceRegistry.shared.resolve(OrderServiceProvider.self)
    }
    func fetchOrders() { orderService?.fetchList { ... } }
}
```

注册流程：`定义Protocol → 实现类注册到容器 → 调用方通过Protocol类型解析`。

```
调用方 Module A          服务容器              实现方 Module B
    │                       │                       │
    │  resolve(OrderService) │                       │
    │  ──────────────────→  │                       │
    │                       │  lookup "OrderServiceImpl"
    │                       │  ──────────────────→  │
    │                       │                       │  ✓ 已注册
    │                       │  ◄──────────────────  │
    │  ◄──────────────────  │                       │
    │  OrderServiceImpl     │                       │
```

Pitfalls: 注册顺序依赖（后注册覆盖先注册）、循环注册死锁（NSRecursiveLock 避免）、未注册时 resolve 返回 nil 导致静默失败（应加断言或 fallback 机制）。

Best practice: 配合 DI 容器（如 Swinject/Factory）实现生命周期管理（singleton/transient），注册阶段集中检查未注册服务并抛出明确错误；生产/测试环境通过不同注册表切换 Mock 实现。

### 资源隔离（每个模块独立 Bundle）

在组件化架构中，每个业务模块应打包为独立的 `.bundle` 资源包，实现**资源隔离**：图片、Storyboard、本地化字符串、JSON 配置各自归属所在模块，避免主 Target 资源膨胀与命名冲突。

**原理**：Xcode 的 Bundle 是一个只读目录（`.bundle` 扩展名），通过 `Copy Bundle Resources` 将模块产物嵌入主 App 或作为动态 Framework 携带。运行时通过 `Bundle` API 按需加载，未使用的模块资源不占用主包体积。

```swift
// 1. 定义模块 Bundle（通过哨兵文件定位）
final class OrderBundle {
    static let current: Bundle = {
        guard let url = Bundle.main.url(forResource: "OrderModule", withExtension: "bundle"),
              let bundle = Bundle(url: url) else {
            fatalError("OrderModule.bundle not found!")
        }
        return bundle
    }()
}

// 2. 加载模块内资源
let icon = OrderBundle.current.imageNamed("order_icon")  // iOS 15+
let storyboard = UIStoryboard(name: "OrderList", bundle: OrderBundle.current)
let config = OrderBundle.current.path(forResource: "api_endpoints", ofType: "json")!

// 3. 多语言本地化隔离
let localized = Bundle(path: OrderBundle.current.path(forResource: "OrderModule", ofType: "lproj")!)!
let title = NSLocalizedString("order_title", bundle: localized, comment: "")
```

**Bundle 结构示例**：
```
OrderModule.bundle/
├── OrderList.storyboard
├── order_icon.png
├── api_endpoints.json
├── en.lproj/
│   └── OrderModule.strings
└── zh-Hans.lproj/
    └── OrderModule.strings
```

Pitfalls: 哨兵文件（sentinel file）机制在 SPM/手动集成下失效，应改用 `Bundle.module`（SPM 内置）或基于 `#fileID` 的运行时解析（Swift 5.6+ `Bundle.accessing(__file:...)`）。

Best practice: 配合 Dynamic Framework 分发，未安装的模块其 Bundle 不占主包体积；配合按需下载（On-Demand Resources / App Clip），实现真正的零预加载。

### 多模块项目构建时间优化

多模块拆分后，**预编译头（PCH）、模块导入开销、重复编译**成为构建瓶颈。合理的模块划分应基于**变更频率矩阵**：高频变更模块独立以利用增量编译，低频通用模块合并以减少导入开销。

**核心优化策略**：

```swift
// 1. 使用 @_exported 避免调用方重复导入（Swift 5.2+）
// Core/Exports.swift
@_exported import Foundation
@_exported import Alamofire      // 所有依赖 Core 的模块自动获得 Alamofire
@_exported import Kingfisher     // 避免每个业务模块重复导入

// 2. typealias 简化跨模块类型引用（避免 Module.Type 污染调用方代码）
public typealias APIResponse = CoreNetworking.APIResponse<ResultType>
public typealias ImageCache = CoreUI.CachedImageViewCache
```

**Xcode 构建加速**：
| 优化项 | 方法 | 预期提升 |
|--------|------|----------|
| 预编译导入 | `@_exported` + 统一 Core 层 | 30-50% |
| 并行构建 | Project Settings → Parallelize build | 20-40% |
| 预编目标头 | bridging header 最小化（只放必要 ObjC 头） | 10-20% |
| 增量编译隔离 | 高频模块独立 Target | 按需触发 |
| DerivedData 清理 | 定期 `rm -rf ~/Library/Developer/Xcode/DerivedData` | 避免污染 |

**Pitfalls**: 滥用 `@_exported` 会导致调用方获得不需要的 API（污染命名空间），应仅在 Core/基础层使用；模块过多（>20）反而增加链接时间，建议保持在 8-15 个模块。

**Best practice**: 使用 `xcodebuild -dryRun` 分析依赖图，配合 `time xcodebuild` 定位最慢 Target；CI 环境中启用 `Indexing parallel build` 和预编译缓存（如 Turbofive/Clangd 预索引），将全量构建从 8 分钟降至 2 分钟。

### 跨模块依赖管理（DAG、循环依赖检测）

模块间的依赖关系天然构成**有向无环图（DAG）**。循环依赖（A→B→A）会导致编译死锁、运行时初始化顺序不确定、以及业务逻辑耦合。必须在架构层面强制检测并阻断。

**依赖图定义与检测**：

```swift
// 模块依赖声明（JSON/Script 可解析格式）
struct ModuleDependency: Codable {
    let name: String
    let dependsOn: [String]  // 直接依赖
}

// 拓扑排序检测循环依赖（Kahn 算法）
func detectCycle(_ modules: [ModuleDependency]) -> [String]? {
    var graph: [String: [String]] = [:]
    var inDegree: [String: Int] = [:]
    for m in modules {
        graph[m.name, default: []].append(contentsOf: m.dependsOn)
        inDegree[m.name, default: 0]
        for d in m.dependsOn { inDegree[d, default: 0] += 1 }
    }
    var queue = inDegree.filter { $0.value == 0 }.map { $0.key }
    var visited = Set<String>()
    while let node = queue.first {
        queue.removeFirst(); visited.insert(node)
        for neighbor in graph[node, default: []] {
            inDegree[neighbor]! -= 1
            if inDegree[neighbor]! == 0 { queue.append(neighbor) }
        }
    }
    return visited.count != modules.count ? Array(visited.symmetricDifference(modules.map { $0.name })) : nil
}
```

**实战检测脚本**（集成到 CI pre-check）：

```bash
# 解析 Project.pbxproj 或 Package.swift 生成依赖矩阵
# 配合 Graphviz 输出可视化 DAG
python3 scripts/dependency_check.py --project MyApp.xcodeproj --output dag.dot
dot -Tpng dag.dot -o dependency_graph.png  # 生成可视化图
```

**Pitfalls**: 通过 typealias 或 typealias 转发间接创建的"幽灵依赖"不会被静态分析捕获，应配合 `otool -L` 检查 Dynamic Framework 级别的真实链接关系。

**Best practice**: 在 CI 中强制运行 DAG 校验作为 pre-merge gate；配合 SPI（Swift Package Interface）机制（Swift 5.9+/Xcode 14+），在语言层面限制模块只能暴露 `@_spi(Testable)` 标注的内部 API，从根源杜绝非法跨层调用。


### 公共 API 接口管理（导出控制）

在组件化架构中，**导出控制**决定了哪些 API 对外可见、哪些仅限内部使用。缺乏出口管理的模块会暴露实现细节，导致调用方产生隐性依赖，最终使重构成为不可能。

**Swift 访问控制层级**：

```swift
// Module: NetworkKit
public final class APIClient {          // 公共入口 - 调用方可见
    private let session: URLSession     // 实现细节 - 模块外不可见
    internal let config: APIConfig      // 同模块/同 Target 可见（默认）
    @usableFromInline private let cache: Cache  // SPI 优化：同 Target 编译时可访问
}

// SPI (Swift Package Interface) - Swift 5.9+/Xcode 14+
@_spi(Internal) public struct RequestBuilder {  // 仅导入方显式标注 @_spi 后可见
    public func build() -> URLRequest { ... }
}

// 导入方使用 SPI
import NetworkKit
@_spi(Internal)  // 显式声明使用内部接口
let builder = RequestBuilder()
```

**Objective-C 导出控制**（Framework 场景）：

```objc
// NetworkKit-umbrella.h - 控制哪些头文件对外暴露
#import <NetworkKit/APIClient.h>       // ✅ 公开
#import <NetworkKit/Endpoints.h>       // ✅ 公开
// NetworkKit/Cache.h 未导入 → 调用方不可见 ✅

// Project Settings 中配置：
// Build Rules → "Public Headers" 只添加对外头文件
// 或使用 Module Map 精细控制：
// module NetworkKit {
//     header "APIClient.h"
//     export *
//     private header "Cache.h"  // 仅模块内部可用
// }
```

**导出矩阵管理**（多模块项目必备）：

| 层级 | 可见范围 | 修饰符 | 示例 |
|------|---------|--------|------|
| Public API | 所有导入方 | `public` | APIClient.init() |
| SPI | 显式标注的导入方 | `@_spi(Name)` | RequestBuilder |
| Internal | 同 Target/模块 | `internal`（默认） | APIConfig |
| Private | 同文件 | `private` | session 属性 |
| File Private | 同源文件 | `fileprivate` | 辅助函数 |

**Pitfalls**: `public` 不等于"应被调用"——过度暴露的 public API 会形成隐式契约，调用方一旦依赖就无法重构。始终遵循最小暴露原则：默认 `internal`/`private`，仅将稳定接口提升为 `public`。`@_exported` 会转发整个子模块的所有 public API，滥用会导致 API 污染。

**Best practice**: 配合 SPI 机制（Swift 5.9+）建立三级出口：Public（稳定 API）、SPI（受控内部 API）、Internal（模块私有）。在 CI 中使用 `swift accessiblity` 检查或自定义脚本审计各模块的 public 接口数量，超过 20 个 public 符号即触发 review，强制拆分子模块或降级为 SPI。

### 跨模块边界的测试策略

在组件化架构中，**模块边界**是最脆弱的测试盲区。单元测试覆盖模块内部逻辑，UI 测试覆盖最终交互，但模块间的契约 breach 往往在集成阶段才暴露。跨模块测试的核心目标是**在隔离环境中验证模块间接口的正确性**，避免"我的模块没问题"但组合后失败的困境。

**测试金字塔（组件化视角）**：

```
         /        /  \       E2E: 完整用户流程（少量）
       /    \
      /------\
     /        \    Integration: 2-3 模块交互（核心）
    /          \
   /------------\  Unit: 单模块内部逻辑（大量）
  /              \
```

**跨模块集成测试示例**（Swift Testing / XCTest）：

```swift
// 测试 NetworkKit ↔ CoreData 模块边界
@MainActor
func testFetchAndCacheFlow() async {
    // 1. 使用真实 NetworkKit 但 mock 底层 URLSession
    let network = APIClient(session: MockURLSession(fixtures: ["users.json"]))
    // 2. 使用真实 CoreDataManager 但 in-memory 持久层
    let store = CoreDataManager(persistence: InMemoryPersistence())
    // 3. 测试跨模块数据流
    let users = try await network.fetchUsers()
    try store.save(users: users)
    let cached = store.loadCachedUsers()
    #expect(cached.count == 3)  // Swift Testing 语法
    #expect(cached[0].id == "u-001")
}
```

**测试隔离矩阵**：

| 测试类型 | Network 层 | Core 层 | UI 层 | 持久层 |
|---------|-----------|--------|------|-------|
| Network 单元 | 真实 | Mock | ❌ | ❌ |
| Core 单元 | Mock | 真实 | ❌ | Mock |
| 边界测试 | 真实+mock底层 | 真实 | ❌ | in-memory |
| UI 测试 | mock 全部 | 真实 | 真实 | in-memory |

**Pitfalls**: 跨模块测试最常见的错误是"全真实"——所有模块都用真实实现，导致测试慢、不稳定、难以定位失败源。始终只测试**两个真实模块的交互**，其余全部 mock。避免在集成测试中调用 `URLSession.shared`（真实网络请求会使测试 flaky）。

**Best practice**: 为每个模块定义 Protocol 边界接口（如 `APIProvider`, `DataStore`, `CacheBackend`），集成测试中通过 DI 容器组装"部分真实 + 部分 mock"的组合。配合 Swift Testing（Xcode 16+/Swift 6）的 `@Test` 和 `#expect` 语法，测试代码更简洁。在 CI 中设置"跨模块测试"作为独立的 test plan，与 unit 测试并行执行，确保模块契约变更不被合并。

### Tuist 模块化架构

Tuist 是用 Swift 编写的 CLI 工具，用于生成和管理 Xcode 项目。与手动维护 `.xcodeproj` 不同，Tuist 通过声明式的 `ProjectDescription` DSL 定义项目结构，实现**项目即代码（Project as Code）**。在大型组件化项目中，Tuist 解决了手动维护多 target 依赖关系繁琐、团队成员项目配置不一致等痛点。

**核心概念**：

```
Project (项目)
├── Module A (Feature/Profile)  ← 业务模块
├── Module B (Feature/Settings)  ← 业务模块
├── Core/NetworkKit              ← 基础设施模块
├── Core/DesignSystem            ← 共享 UI 组件
└── App/MyApp                    ← App Shell（组装层）
```

**项目定义示例**（`ProjectDescription/Projects/NetworkKit.swift`）：

```swift
import ProjectDescription

let project = Project.makeModuleName(name: "NetworkKit") {
    Project(
        name: "NetworkKit",
        targets: [
            Target.target(
                name: "NetworkKit",
                destinations: .iOS,
                product: .library,
                dependencyFiles: ["Sources"],  // 自动发现源文件
                infoPlist: .default,
                sources: ["NetworkKit/Sources/**"],
                resources: [],
                dependencies: [
                    .external(name: "Alamofire", kind: .gitURL("https://github.com/Alamofire/Alamofire.git", requirement: .upToNextMajorVersion("5.8"))),
                ],
                settings: .settingsBase()  // 共享配置
            )
        ],
        packaging: [
            .package(name: "Alamofire", path: "../../Alamofire")  // 本地 override
        ]
    )
}
```

**Tuist vs 手动项目管理的对比**：

| 维度 | 手动 Xcode 项目 | Tuist |
|------|----------------|-------|
| 依赖管理 | 手动拖拽/SPM | 声明式定义 + 自动生成 |
| 多环境配置 | 手动维护 Scheme | ProjectDescription DSL |
| 团队成员同步 | 手动合并冲突 | `git pull && tuist generate` |
| 跨项目共享 | 复制粘贴/子模块 | Project template + override |
| CI 加速 | 全量 build | 增量 build + 并行 |

**Pitfalls**: Tuist 学习曲线陡峭——`ProjectDescription` DSL 的 API 经常随版本变化，升级 Tuist 后可能需批量修改定义文件。另外，Tuist 生成的项目**不包含手动在 Xcode 中做的修改**（如 Breakpoint 或自定义 Run Script），这些会丢失。始终通过 DSL 定义所有配置。

**Best practice**: 从最小的模块开始迁移（如 NetworkKit 或 DesignSystem），验证 workflow 后再全量切换。配合 Tuist 的 Project Template 功能定义团队标准模板（LLDB 设置、Code Signing、Swift 版本），确保所有新模块从零开始就是标准化的。Xcode 15+/Tuist 3.9+ 支持 Swift 6 并发检查，在 DSL 中通过 `settings: .settingsBase().enablingStrictConcurrency()` 全局启用。

### 每模块 Lint 规则（SwiftLint 配置）

在组件化架构中，不同模块有不同的代码规范要求。基础设施层（如 NetworkKit）通常要求更严格的 API 设计和文档规范，而业务层（如 Feature/Profile）则需要灵活的 UI 代码容忍度。统一的 SwiftLint 配置无法满足这种差异化的需求。

**分层 SwiftLint 策略**：

```
Project/
├── .swiftlint.yml                      ← 全局基础规则（所有模块继承）
├── Core/NetworkKit/
│   └── .swiftlint.yml                  ← 基础设施层补充规则
├── Core/DesignSystem/
│   └── .swiftlint.yml                  ← UI 组件层补充规则
└── Feature/Profile/
    └── .swiftlint.yml                  ← 业务模块补充规则
```

**全局基础配置**（`/Users/zhaorui/iOS Knowledge/24-组件化与工程架构.md` 项目根 `.swiftlint.yml`）：

```yaml
disabled_rules:
  - trailing_whitespace
  - line_length
opt_in_rules:
  - empty_count
  - closure_spacing
  - missing_docs          # 公共 API 必须有文档
included:
  - Core
  - Feature
excluded:
  - Tuist/Generated/
  - Pods/
  - **/Tests/             # 测试代码放宽
line_length: 120
file_length: 500
type_body_length: 200
```

**基础设施层补充**（`Core/NetworkKit/.swiftlint.yml`）：

```yaml
# 继承全局配置后追加
disabled_rules:
  - trailing_whitespace
opt_in_rules:
  - missing_docs           # 强制文档
  - override_require_explicit_pointer_conversion
  - unhandled_task_completion
  - identifier_name        # 命名规范更严格
  - explicit_acl           # 强制显式访问控制
  - explicit_top_level_acl
identifier_name:
  min_length: 2           # 不允许单字符变量（除 i, j 外）
  excluded: ["i", "j", "id"]
type_body_length: 150     # 基础设施类不允许过长
file_length: 400          # 基础设施文件不允许过长
```

**业务模块补充**（`Feature/Profile/.swiftlint.yml`）：

```yaml
disabled_rules:
  - trailing_whitespace
  - missing_docs           # 业务层不强制文档
  - file_length            # 业务页面可能较长
opt_in_rules:
  - closure_spacing
  - empty_count
type_body_length: 300     # 业务页面 VC 通常较长
file_length: 800          # 业务页面容忍更大
```

**SwiftLint 多配置文件搜索机制**：
SwiftLint 默认从当前源文件所在目录开始，逐层向上搜索 `.swiftlint.yml`。通过 `parent_config` 指令显式引用父配置：

```yaml
# Core/NetworkKit/.swiftlint.yml
parent_config: ../../.swiftlint.yml    ← 显式继承全局

disabled_rules:
  - trailing_whitespace
opt_in_rules:
  - missing_docs
  - explicit_acl
```

**Pitfalls**: 不使用 `parent_config` 时，子目录的 `.swiftlint.yml` 会**完全覆盖**全局配置而非合并，导致全局规则在子模块中失效。始终显式声明 `parent_config` 或手动复制全局规则。

**Best practice**: 在 CI 中为每个模块单独运行 SwiftLint（`swiftlint lint --config Core/NetworkKit/.swiftlint.yml --path Core/NetworkKit/Sources/`），配合 `disabled_from_basement: true` 确保只检查当前模块。Xcode 15+/Swift 6 项目中，通过 `disabled_rules: ["nonoverriding_mutable_method_init"]` 避免与严格并发检查冲突。业务模块可进一步禁用 `missing_docs` 以平衡开发效率与代码质量。
