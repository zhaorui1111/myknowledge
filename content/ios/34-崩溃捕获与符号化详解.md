# 崩溃捕获与符号化详解

## 一、崩溃捕获机制

iOS 崩溃主要分为三类：Mach 异常、Signal 信号、Objective-C 异常（NSException）和 C++ 异常。一个完善的崩溃捕获系统需要覆盖所有类型。

### 1.1 Mach 异常

Mach 是 macOS/iOS 的微内核，底层异常通过 Mach 异常机制传递。常见的 Mach 异常类型：

```c
EXC_BAD_ACCESS      // 访问无效内存（野指针、空指针、已释放内存）
EXC_BAD_INSTRUCTION  // 非法指令
EXC_ARITHMETIC       // 算术异常（除零等）
EXC_BREAKPOINT       // 断点（__builtin_trap、fatalError）
EXC_RESOURCE         // 资源限制（CPU、内存）
EXC_GUARD            // 守卫违规（文件描述符保护）
```

### 1.2 Signal 信号

Unix 信号是 POSIX 层的异常通知机制。Mach 异常会被转换为对应的 Unix 信号：

```c
SIGABRT   // abort() 调用，通常来自未捕获的 NSException
SIGSEGV   // 段错误，对应 EXC_BAD_ACCESS
SIGBUS    // 总线错误
SIGFPE    // 浮点异常
SIGILL    // 非法指令
SIGTRAP   // 陷阱
SIGPIPE   // 管道破裂
```

### 1.3 NSException

Objective-C 层的异常，通过 `@try/@catch` 或 `NSSetUncaughtExceptionHandler` 捕获：

```swift
NSSetUncaughtExceptionHandler { exception in
    let stack = exception.callStackSymbols
    let reason = exception.reason ?? "Unknown"
    let name = exception.name.rawValue

    // 保存崩溃信息到文件
    let crashReport = """
    Exception: \(name)
    Reason: \(reason)
    Stack:
    \(stack.joined(separator: "\n"))
    """
    saveCrashReport(crashReport)
}
```

### 1.4 Signal 捕获

```c
#include <signal.h>

void signalHandler(int signal) {
    void *stack[128];
    int count = backtrace(stack, 128);
    char **symbols = backtrace_symbols(stack, count);

    // 保存堆栈到文件
    // ...

    // 重新抛出信号让系统处理
    signal(signal, SIG_DFL);
    raise(signal);
}

void installSignalHandler() {
    signal(SIGABRT, signalHandler);
    signal(SIGSEGV, signalHandler);
    signal(SIGBUS, signalHandler);
    signal(SIGFPE, signalHandler);
    signal(SIGILL, signalHandler);
    signal(SIGPIPE, signalHandler);
    signal(SIGTRAP, signalHandler);
}
```

### 1.5 捕获链的优先级

Mach 异常最先触发，然后转换为 Signal。如果注册了 Mach 异常处理器，Signal 处理器不会收到通知。NSException 通过 `NSSetUncaughtExceptionHandler` 捕获，它内部调用 `abort()` 触发 SIGABRT。

实践中，第三方崩溃收集库（PLCrashReporter、Bugly、Sentry）通常同时注册 Mach 异常处理器和 Signal 处理器，确保覆盖所有情况。但同一进程只能有一个活跃的异常处理器——多个崩溃收集 SDK 会互相冲突。

## 二、崩溃堆栈采集

### 2.1 backtrace

C 标准库提供 `backtrace()` 函数获取调用栈：

```c
#include <execinfo.h>

void *callstack[128];
int frames = backtrace(callstack, 128);
char **symbols = backtrace_symbols(callstack, frames);

for (int i = 0; i < frames; i++) {
    printf("%s\n", symbols[i]);
}
free(symbols);
```

`backtrace_symbols` 返回的符号信息是未符号化的地址，需要后续用 dSYM 符号化。

### 2.2 PLCrashReporter

PLCrashReporter 是 PLCrashReporter 开源库，提供更完整的崩溃采集能力，包括 Mach 异常、Signal、NSException 的全量捕获，以及线程状态的完整保存。

```swift
import PLCrashReporter

let config = PLCrashReporterConfig(signalHandlerType: .BSD, symbolicationStrategy: .all)
let crashReporter = PLCrashReporter(configuration: config)

crashReporter?.enable()

// 应用启动时检查是否有崩溃报告
if crashReporter?.hasPendingCrashReport() == true {
    let crashData = try? crashReporter?.loadPendingCrashReportDataAndReturnError()
    if let data = crashData {
        let report = try? PLCrashReport(data: data)
        let text = PLCrashReportTextFormatter.stringValue(for: report, with: PLCrashReportTextFormatiOS)
        // 上传到服务器
    }
    crashReporter?.purgePendingCrashReportData()
}
```

## 三、符号化

### 3.1 什么是符号化

崩溃报告中只有内存地址（如 `0x1024d8 0x00000001024d8000 + 8765`），不包含函数名和源码行号。符号化是将地址映射到函数名、文件名、行号的过程。

### 3.2 dSYM 文件

dSYM（Debug Symbol）是包含调试符号的文件，由编译器在 Archive 构建时生成。它记录了二进制地址到源码位置的映射关系。

```
MyApp.app.dSYM/
├── Contents/
│   ├── Info.plist
│   └── Resources/
│       └── DWARF/
│           └── MyApp    # DWARF 调试信息
```

dSYM 与二进制文件通过 UUID 关联。每个构建产物有唯一的 UUID，对应的 dSYM 必须有相同的 UUID 才能正确符号化。

### 3.3 atos 符号化

```bash
# atos 将地址转换为符号
atos -arch arm64 -o MyApp.app.dSYM/Contents/Resources/DWARF/MyApp -l 0x1024d8000 0x1024d8234

# 输出
# -[ViewController viewDidLoad] (in MyApp) (ViewController.m:25)
```

参数说明：`-arch` 架构，`-o` dSYM 文件路径，`-l` 加载基址（从崩溃报告中获取），最后的地址是要符号化的指令地址。

### 3.4 symbolicatecrash

Apple 提供的 `symbolicatecrash` 工具可以自动符号化 .crash 文件：

```bash
# 找到 symbolicatecrash
find /Applications/Xcode.app -name symbolicatecrash

# 符号化
export DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer
symbolicatecrash MyApp.crash MyApp.app.dSYM -o MyApp-symbolicated.crash
```

需要将 .crash 文件、.app 文件和 .dSYM 文件放在同一目录。

### 3.5 批量符号化脚本

```bash
#!/bin/bash
# 符号化崩溃报告
CRASH_FILE=$1
DSYM_PATH=$2
APP_NAME=$3

DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer
export DEVELOPER_DIR

# 获取 UUID
DWARF_DSYM="$DSYM_PATH/Contents/Resources/DWARF/$APP_NAME"
UUID=$(dwarfdump --uuid "$DWARF_DSYM" | awk '{print $2}')
echo "dSYM UUID: $UUID"

# 符号化每行地址
while IFS= read -r line; do
    if [[ $line =~ ^[0-9]+\ +0x[0-9a-f]+ ]]; then
        address=$(echo $line | awk '{print $2}')
        load_address=$(echo $line | awk '{print $4}' | sed 's/.*0x/0x/')
        if [ -n "$load_address" ]; then
            symbol=$(atos -arch arm64 -o "$DWARF_DSYM" -l "$load_address" "$address")
            echo "$line  →  $symbol"
        else
            echo "$line"
        fi
    else
        echo "$line"
    fi
done < "$CRASH_FILE"
```

## 四、常见崩溃类型分析

### 4.1 EXC_BAD_ACCESS

最常见也最难排查的崩溃。原因包括：

**野指针**：访问已释放的对象。ObjC 中 `__unsafe_unretained` 或 unsafe 的 delegate 常导致此问题。开启 Zombie Objects（Scheme → Diagnostics → Zombie Objects）可以将已释放对象替换为 Zombie，访问时打印信息而非崩溃。

**空指针**：解包 nil（Swift 的 `!` 强制解包）。在 Swift 中应优先使用 `if let` / `guard let` 安全解包。

**悬垂指针**：`Unmanaged` 或 `withUnsafePointer` 使用不当。确保引用的生命周期覆盖指针的使用范围。

**多线程数据竞争**：一个线程释放对象，另一个线程同时访问。使用 Thread Sanitizer（TSan）检测。

### 4.2 SIGABRT（NSException）

通常由以下原因导致：

**数组越界**：`array[index]` 访问不存在的索引。Swift 数组越界会直接崩溃，需要检查 `indices.contains(index)`。

**KVC 找不到 key**：`setValue:forKey:` 传入了不存在的 key。检查 key 拼写或实现 `setValue:forUndefinedKey:`。

** unrecognized selector**：向对象发送未实现的消息。检查方法名拼写、delegate 协议声明、类别加载。

**NSInternalInconsistencyException**：如修改正在遍历的数组、在后台线程更新 UI。

### 4.3 Watchdog 超时

系统看门狗杀进程，崩溃报告通常显示 `Exception Type: EXC_RESOURCE` 或 `0x8badf00d`（eat bad food）。原因：

**启动超时**：App 启动超过 20 秒（iOS）。优化启动流程，延迟非关键任务。

**主线程卡顿**：主线程长时间不响应（超过数秒）。将耗时操作移到后台线程。

**后台任务超时**：`beginBackgroundTask` 超过系统分配时间。在 `expirationHandler` 中保存状态并结束任务。

### 4.4 OOM（Out of Memory）

内存不足被系统杀死。崩溃报告中没有堆栈信息（因为是系统直接杀进程而非抛出异常）。通过 MetricKit 或 Xcode Memory Graph 诊断内存峰值。

### 4.5 死锁

线程互相等待锁导致永久阻塞。主线程死锁表现为 UI 冻结，最终触发 Watchdog。使用 `dispatch_async` 避免在主线程同步等待。

## 五、崩溃收集最佳实践

### 5.1 启动时检查崩溃报告

```swift
class CrashReporter {
    static func setup() {
        // 1. 注册信号/异常处理器
        installHandlers()

        // 2. 检查上次崩溃报告
        if let crashData = loadPendingCrashReport() {
            uploadCrashReport(crashData)
            clearPendingCrashReport()
        }
    }

    private static func uploadCrashReport(_ data: Data) {
        // 延迟上传，避免影响启动速度
        DispatchQueue.global().asyncAfter(deadline: .now() + 3.0) {
            // 上传到服务器
        }
    }
}
```

### 5.2 信号处理安全

Signal 处理函数运行在受限环境中——不能调用非异步信号安全（async-signal-safe）的函数。`malloc`、`printf`、`NSLog`、Objective-C 方法都不是安全的。只能使用 `write()` 系统调用写入文件，或使用预分配的缓冲区。

PLCrashReporter 等专业库通过在进程内运行一个独立的 Mach 异常处理服务来绕过这些限制。

### 5.3 多 SDK 冲突

多个崩溃收集 SDK（如 Bugly + Sentry）会争夺异常处理器注册权。最后一个注册的 SDK 覆盖之前的。解决方案：只用一个 SDK，或配置链式调用（转发到上一个 handler）。

## 六、线上崩溃分析流程

1. 收集崩溃报告（从第三方平台或自建系统获取）
2. 确定崩溃类型（Mach 异常 / Signal / NSException）
3. 找到对应版本的 dSYM 文件（通过 UUID 匹配）
4. 符号化堆栈地址
5. 分析崩溃路径和触发条件
6. 复现并修复

dSYM 管理建议：每次 Archive 构建后自动上传 dSYM 到服务器或第三方平台。建立版本号-Build号-UUID 的映射表，方便查找。
