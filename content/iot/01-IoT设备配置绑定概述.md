# 01 · IoT 设备配置绑定概述

## 一、为什么配网是 IoT 的第一道门槛

一台刚出厂的 IoT 设备，通常没有键盘、没有屏幕，甚至没有完整的网络栈。如何让它从「通电的硬件」变成「联网的智能设备」，这个过程中的核心环节就是 **设备配网（Device Provisioning / Commissioning）**。

配网的本质是完成两件事：

1. **网络凭证交付**：将 SSID + Password（或等效的网络身份）传递给设备，使其能够加入目标网络。
2. **设备-账号绑定**：将设备的唯一标识（如 MAC、Serial Number、证书公钥）与用户账号关联，建立云端归属关系。

这两步完成后，设备才能上报数据、接收指令、与其他设备联动。

## 二、主流配网方式分类

按照信息传递的物理通道，可以将配网方式分为以下几类：

| 分类 | 典型方式 | 代表技术 | 适用场景 |
|------|---------|---------|---------|
| WiFi 通道 | SmartConfig / SmartConnection | TI SmartConfig、ESP-TOUCH、AirKiss | 纯 WiFi 设备，成本极低 |
| AP 热点通道 | SoftAP 模式 | 设备开启 AP，手机连接后输入 WiFi 密码 | 兼容性最好的 WiFi 配网 |
| 蓝牙通道 | BLE Provisioning | BLE GATT 传递凭证 | BLE+WiFi 双模设备 |
| 近场通道 | NFC / 二维码 / 声波 | NFC 碰一碰、声波配网 | 智能家居、穿戴设备 |
| 有线通道 | Ethernet / USB | 有线直连配置 | 网关类设备 |
| 云端通道 | 云端预置 | 设备出厂前预绑定 | 运营商定制设备 |

下面逐一展开讲解。

## 三、各配网方式详解

### 3.1 SmartConfig（广播/组播配网）

**原理**：手机连接到目标 WiFi 后，通过发送特定编码的 UDP 广播/组播包，将 SSID 和密码编码在数据包的长度、内容或目标地址中。设备端进入混杂模式（Promiscuous Mode），监听空口所有 802.11 帧，解析出 WiFi 凭证。

**编码方式**：
- 长度编码：利用 UDP 包长度的不同值映射为字符（如 ESP-TOUCH）
- 地址编码：利用 UDP 目标 IP 地址的不同值编码数据
- 内容编码：直接在 UDP payload 中放入加密后的凭证

**流程**：
```
手机 (已连WiFi)                    设备 (监听模式)
    |                                  |
    |--- UDP广播包 (编码SSID+密码) --->|
    |                                  |--- 解码得到凭证 ---
    |                                  |--- 连接目标WiFi ---
    |<--------- 连接成功通知 -----------|
```

**优点**：用户体验好，一键配网；设备端实现简单，成本低。

**缺点**：依赖手机和设备在同一 2.4GHz 频段；5GHz WiFi 下手机可能无法发送 2.4G 广播；成功率受路由器、环境干扰影响较大（实测 70%-90%）；不支持隐藏 SSID 的路由器。

**代表实现**：TI SmartConfig、乐鑫 ESP-TOUCH、微信 AirKiss。

### 3.2 SoftAP 配网（AP 模式配网）

**原理**：设备启动后进入 AP 模式，广播一个 WiFi 热点（如 "MyDevice-XXXX"）。手机连接此热点后，通过 HTTP 或自定义协议将家庭 WiFi 的 SSID 和密码发送给设备。设备收到后切换到 STA 模式，连接家庭 WiFi。

**流程**：
```
手机                              设备 (AP模式)
    |                                  |
    |--- 连接设备热点 ----------------->|
    |--- HTTP POST {ssid, password} -->|
    |                                  |--- 切换为STA模式 ---
    |                                  |--- 连接家庭WiFi ---
    |  (手机断开设备热点，连回家庭WiFi)  |
    |<--------- 局域网/云端通知 --------|
```

**优点**：成功率极高（95%+），不依赖路由器频段，兼容性最好；支持隐藏 SSID。

**缺点**：用户操作步骤多（连热点→输密码→切回WiFi），体验稍差；配网过程中手机断网；iOS 对临时连接热点的处理有系统限制，需要引导用户在系统设置中连接。

### 3.3 BLE 辅助配网（BLE Provisioning）

**原理**：设备启动后广播 BLE 广播包。手机扫描到设备后建立 BLE GATT 连接，通过 GATT 特征值（Characteristic）写入 WiFi SSID 和密码。设备收到凭证后连接 WiFi，同时可通过 BLE 完成设备绑定。

**流程**：
```
手机                              设备 (BLE广播)
    |                                  |
    |--- BLE扫描发现设备 ------------->|
    |--- 建立GATT连接 ---------------->|
    |--- Write: SSID ----------------->|
    |--- Write: Password ------------->|
    |--- Write: 触发连接指令 --------->|
    |                                  |--- 连接WiFi ---
    |<--- Notify: 连接结果 ------------|
    |--- 绑定请求(token/certificate)->|
    |<--- 绑定确认 --------------------|
```

**优点**：成功率最高（98%+）；BLE 通道可同时完成设备发现、凭证传递、身份绑定；用户体验好（App 自动扫描连接）；支持隐藏 SSID。

**缺点**：设备需要 BLE+WiFi 双模芯片（或双芯片），硬件成本略高；需要处理 BLE 和 WiFi 共存干扰问题。

**代表实现**：ESP32 BLE Provisioning、Silicon Labs BLE Provisioning、蓝牙 SIG 的 Mesh Provisioning。

### 3.4 声波配网

**原理**：手机播放包含编码信息的声波（通常在 4kHz-10kHz 频段），设备麦克风采集后解码出 WiFi 凭证。

**优点**：不依赖 WiFi 通道，可用于纯音频设备；跨平台兼容性好。

**缺点**：受环境噪声影响大；传输速率极低（几十 bps）；安全性差（声波可被录音重放）；实际应用较少，主要在早期智能音箱中使用。

### 3.5 NFC 碰一碰配网

**原理**：手机 NFC 感应设备上的 NFC Tag，读取其中的配网信息（如设备 ID、配网 URL、BLE MAC 地址等），触发 App 进入配网流程。NFC 本身不直接传输 WiFi 密码，而是作为配网入口和设备发现手段，后续配网走 BLE 或 SoftAP。

**优点**：用户体验极佳，碰一碰即开始；安全性好（近场通信难以截获）。

**缺点**：设备需要 NFC Tag，增加硬件成本；需要手机支持 NFC。

**代表应用**：华为一碰连、小米碰一碰、HomeKit NFC 配对。

### 3.6 云端预绑定

**原理**：设备在出厂时将设备标识（如 SN、MAC）预录入云端数据库。用户购买后，通过扫描设备上的二维码或输入 SN，云端校验后直接建立绑定关系。配网环节由设备自主完成（如预置了运营商 WiFi 信息）或通过其他方式补充。

**适用场景**：运营商定制的智能音箱、机顶盒等。

## 四、设备绑定流程

配网（连上网络）只是第一步，还需要完成 **设备绑定**，将设备与用户账号关联：

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  App     │     │  Cloud   │     │  Device  │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     | 1. 登录获取token|                |
     |--------------->|                |
     |<----- token ---|                |
     |                |                |
     | 2. 发起绑定请求 |                |
     | (deviceId, token)               |
     |--------------->|                |
     |                | 3. 设备认证请求 |
     |                |--------------->|
     |                |<--- 设备签名 ---|
     |                |                |
     |                | 4. 校验设备签名 |
     |                | (用预置公钥验签)|
     |                |                |
     |<-- 绑定成功 ---|                |
     |                |                |
     | 5. 建立通信通道 |                |
     | (MQTT/WebSocket)                |
     |<===============|===============>|
```

**绑定认证方式**：
- **静态密钥**：设备出厂烧录固定密钥，简单但安全性低。
- **一机一密**：每台设备有唯一密钥（如 ECDSA 私钥），出厂时公钥录入云端。安全性高，是主流方案。
- **动态令牌**：配网过程中通过 BLE/NFC 协商临时密钥，再用临时密钥交换固定凭证。

## 五、配网方式选型建议

| 因素 | 推荐 |
|------|------|
| 纯 WiFi 设备，低成本 | SmartConfig（主）+ SoftAP（备） |
| BLE+WiFi 双模设备 | BLE Provisioning（首选） |
| 智能家居，强调体验 | NFC + BLE 组合 |
| 有线网关类设备 | 有线直连配置 |
| 运营商定制设备 | 云端预绑定 |
| 需要兼容旧设备 | SoftAP（兼容性最好） |

## 六、当前行业趋势

1. **Matter 统一配网**：CSA（连接标准联盟）推出的 Matter 协议定义了统一的 commissioning 流程，基于 BLE 进行设备发现和凭证传递，支持 Thread 和 WiFi 双栈。目标是让不同生态（Apple/Google/Amazon）的设备用同一个 App 配网。

2. **零接触配网（ZTP）**：设备开机后自动连接云端，通过预置证书完成认证和注册，无需用户手动配网。适用于运营商和大型部署场景。

3. **Wi-Fi Easy Connect（DPP）**：WiFi 联盟推出的 Device Provisioning Protocol，通过二维码或 NFC 传递引导信息，使用公钥加密交换 WiFi 凭证。Android 10+ 和 iOS 13+ 已支持。

## 七、总结

IoT 设备配网是连接物理世界和数字世界的桥梁。选择配网方案时，需要综合考虑硬件成本、用户体验、成功率和安全性。当前行业正在向 BLE 辅助配网和 Matter 统一配网方向演进，但 SoftAP 和 SmartConfig 仍是大批量低成本设备的主流选择。
