# 22 · 纯蓝牙设备配网流程详解

## 一、概述

前面第 3 篇详细讲解了 BLE 的配对（Pairing）与绑定（Bonding）协议机制，但那只解决了"链路安全"问题。对于一台纯蓝牙 IoT 设备（如蓝牙体温计、蓝牙门锁、蓝牙体重秤），从出厂到用户可用，还需要经历完整的**配网流程**——包括设备发现、配对认证、GATT 服务发现、配置数据交换、云端账号绑定等环节。

纯蓝牙设备的配网与 Wi-Fi 设备配网有本质区别：Wi-Fi 配网的核心是把 SSID+密码交给设备让它接入路由器；而纯蓝牙设备不需要加入任何局域网，它的配网核心是**建立设备与用户账号的绑定关系**，以及**下发初始配置参数**（如用户 ID、时间同步、加密密钥等）。

本文聚焦纯蓝牙设备（非 BLE Provisioning 辅助 Wi-Fi 配网、非蓝牙 Mesh），系统讲解从设备发现到云端绑定的完整流程。

## 二、纯蓝牙设备的通信模型

### 2.1 设备不接入局域网

```
纯蓝牙设备                     手机 App                    云平台
   │                             │                           │
   │◄──── BLE 连接 ─────────────►│                           │
   │                             │◄──── HTTPS ──────────────►│
   │                             │                           │
   │  设备数据通过手机中转上云      │                           │
   │  控制命令通过手机下发          │                           │
   │                             │                           │
   │  (设备没有独立IP,不直连云端)   │                           │
```

纯蓝牙设备的特征：
- 没有 Wi-Fi/以太网模块，无法直连互联网。
- 所有数据交互通过 BLE 连接，由手机 App 或网关中转。
- 设备"在线"状态取决于手机是否在 BLE 范围内并保持连接。
- 离开 BLE 范围后，设备处于"离线"状态，数据可能缓存在设备本地。

### 2.2 纯蓝牙设备的典型品类

| 品类 | 通信模式 | 配网特点 |
|------|---------|---------|
| 体温计/血压计/血糖仪 | 周期性测量→主动连接上报 | 需绑定用户账号，测量数据关联用户 |
| 体重秤/体脂秤 | 测量后广播→App接收 | 绑定后自动同步数据 |
| 智能门锁 | 事件驱动→手机控制开锁 | 安全性要求高，需要双向认证 |
| 蓝牙标签/防丢器 | 持续广播→手机监听 | 绑定后持续监控距离 |
| 蓝牙骑行传感器 | 持续连接→实时数据流 | 低延迟配对，快速恢复连接 |
| 蓝牙智能灯（非Mesh） | 手机直连控制 | 需配对后控制，无网关场景 |

## 三、完整配网流程

### 3.1 流程总览

```
┌─────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 1.设备   │──→│ 2.扫描   │──→│ 3.配对   │──→│ 4.服务   │──→│ 5.配置   │
│   发现   │   │   连接   │   │   认证   │   │   发现   │   │   交换   │
└─────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                                    │
                                                                    ▼
                                                              ┌──────────┐
                                                              │ 6.云端   │
                                                              │   绑定   │
                                                              └──────────┘
```

### 3.2 阶段一：设备发现

纯蓝牙设备出厂后处于**广播状态**，等待手机扫描发现。

**设备端广播设计：**

```
广播包 (Advertising Packet):
┌─────────────────────────────────────────────┐
│ Preamble (1B) │ Access Address (4B)         │
│ PDU Header (2B) │ Payload (0-31B)           │
│ CRC (3B)                                    │
└─────────────────────────────────────────────┘

Payload 内容:
  ├── Flags (3B): LE General Discoverable Mode
  ├── Complete Local Name / Shortened Local Name
  │   如 "Mi_Band_5_A1B2" / "Therm_001"
  ├── Service UUID (2B or 16B)
  │   如 0x1810 (Blood Pressure)
  │   如 0xFE95 (Xiaomi custom)
  ├── Manufacturer Specific Data (厂商自定义)
  │   ├── 厂商 ID (2B)
  │   └── 自定义数据 (设备型号、MAC、配网标识等)
  └── TX Power Level (可选)

扫描响应包 (Scan Response, 可选, 0-31B):
  ├── 补充设备名称
  ├── Service UUID (16-bit/128-bit)
  └── 厂商扩展数据
```

**配网专用广播标识：**

设备通常通过 Manufacturer Specific Data 中的标志位标识自身是否处于"待配网"状态：

```
Manufacturer Data 结构示例:
  厂商ID: 0x5901 (Xiaomi)
  自定义数据:
    ├── Frame Control (1B)
    │   └── Bit 0: 0=已配网, 1=待配网
    ├── Product ID (2B)
    ├── 设备 MAC (6B)
    └── 随机数/认证Token (可选)

App 扫描时过滤: 待配网标志=1 的设备才显示在列表中
```

**广播参数选择：**

| 参数 | 待配网模式 | 已配网模式 |
|------|-----------|-----------|
| 广播间隔 | 20-100ms（快速，便于发现） | 1000-2000ms（省电） |
| 广播类型 | ADV_IND（可连接+可扫描） | ADV_IND 或 ADV_NONCONN_IND |
| 广播通道 | 37/38/39（全开） | 37/38/39（全开） |
| 持续时长 | 30-120秒后超时停止 | 持续或按需开启 |

快速广播期（如前 30 秒用 20ms 间隔）确保用户快速发现设备，之后切换到慢速广播省电。

### 3.3 阶段二：扫描与连接

**手机端扫描：**

```
App 扫描流程:
  1. 启动 BLE 扫描 (Scan)
     - 扫描间隔: 1000ms, 扫描窗口: 500ms (50%占空比)
     - 过滤策略: 可按 Service UUID 或设备名前缀过滤

  2. 解析广播包
     - 提取设备名称、Service UUID、Manufacturer Data
     - 检查"待配网"标志
     - 匹配设备类型 (如只搜索体温计)

  3. 展示设备列表
     - 显示设备名称、信号强度 (RSSI)
     - 用户选择目标设备

  4. 发起 BLE 连接
     - 连接参数: Interval 30-50ms, Latency 0, Timeout 3-6s
     - 连接成功后进入配对流程
```

**连接参数协商：**

连接建立后，Master（手机）和 Slave（设备）协商连接参数：

```
连接参数:
  Interval: 7.5ms - 4000ms (配网阶段用较小值如 30ms, 配网后切换到较大值如 200ms 省电)
  Slave Latency: 0 (配网阶段), 配网后可设 4-10
  Supervision Timeout: 3-6 秒

参数更新流程:
  Slave (设备) → L2CAP Connection Parameter Update Request
  Master (手机) → L2CAP Connection Parameter Update Response
  (手机同意后, 连接参数生效)
```

### 3.4 阶段三：配对认证

配对是建立加密链路的过程，第 3 篇已详细讲解了 BLE 配对协议（Legacy / Secure Connections、Just Works / Passkey / OOB），此处不再重复协议细节，重点讲配网场景下的选择策略。

**配网场景的认证方式选择：**

```
设备 IO 能力 → 认证方式映射:

┌──────────────┬─────────────────────────────────────────┐
│ 设备IO能力    │ 推荐认证方式                             │
├──────────────┼─────────────────────────────────────────┤
│ 无输入无输出  │ Just Works (最低安全)                    │
│ (温度传感器)  │ 或 OOB (NFC/二维码增强)                  │
├──────────────┼─────────────────────────────────────────┤
│ 有LED/显示屏  │ Passkey Entry (设备显示6位码)            │
│ (体温计有屏)  │ 或 Numeric Comparison (双方显示确认)     │
├──────────────┼─────────────────────────────────────────┤
│ 有按键        │ Passkey Entry (按键输入6位码)            │
│ (门锁)        │ 或 确认按钮 (Numeric Comparison确认)    │
├──────────────┼─────────────────────────────────────────┤
│ 有NFC         │ OOB (NFC传递认证数据, 最高安全)          │
│ (高端门锁)    │                                          │
└──────────────┴─────────────────────────────────────────┘
```

**配对安全级别要求：**

不同设备类型对安全级别的要求不同：

| 设备类型 | 最低安全级别 | 说明 |
|---------|------------|------|
| 温度/湿度传感器 | LE Secure Connections + Just Works | 数据不敏感，但需防窃听 |
| 体重秤/体脂秤 | LE Secure Connections + Just Works | 数据含个人健康信息 |
| 智能门锁 | LE Secure Connections + Passkey/OOB | 涉及物理安全，需 MITM 防护 |
| 医疗设备 | LE Secure Connections + Passkey/OOB | 涉及健康数据，需高安全 |
| 蓝牙支付设备 | LE Secure Connections + OOB | 涉及金融，最高安全 |

**配对成功后的绑定（Bonding）：**

配对成功后，双方保存以下密钥用于后续重连：
- **LTK (Long Term Key)**：用于连接加密。
- **EDIV + Rand**：LTK 的标识，用于重连时查找对应 LTK。
- **IRK (Identity Resolving Key)**：用于解析随机地址（如果设备使用 RPA）。
- **CSRK (Connection Signature Resolving Key)**：用于数据签名（可选）。

绑定后，设备下次重连时使用已保存的 LTK 加密，无需重新配对。

### 3.5 阶段四：GATT 服务发现

配对加密完成后，App 需要 discover 设备的 GATT 服务，了解设备提供哪些功能和数据：

```
App (GATT Client)                    设备 (GATT Server)
    │                                    │
    │ 1. Discover All Primary Services   │
    ├───────────────────────────────────→│
    │←───────────────────────────────────┤
    │  Service: 0x180A (Device Info)     │
    │  Service: 0x1810 (Blood Pressure)  │
    │  Service: 0xFF01 (厂商自定义)       │
    │                                    │
    │ 2. Find Characteristics (per svc)  │
    ├───────────────────────────────────→│
    │←───────────────────────────────────┤
    │  Char: 0x2A00 (Device Name)        │
    │  Char: 0x2A24 (Model Number)       │
    │  Char: 0x2A25 (Serial Number)      │
    │  Char: 0x2A19 (Battery Level)      │
    │  Char: 0x2A35 (Blood Pressure Meas)│
    │  Char: 0x2A38 (Body Sensor Location)│
    │  ...                               │
    │                                    │
    │ 3. Find Descriptors                │
    ├───────────────────────────────────→│
    │←───────────────────────────────────┤
    │  CCCD (0x2902) on each notify char │
    │                                    │
    │ 4. Enable Notifications            │
    ├──── Write CCCD 0x0001 ────────────→│
    │  (订阅血压测量通知)                   │
    │                                    │
```

**标准 Service 与厂商自定义 Service：**

```
标准 Service (SIG 定义):
  0x180A - Device Information (设备信息)
    0x2A24 - Model Number
    0x2A25 - Serial Number
    0x2A26 - Firmware Revision
    0x2A27 - Hardware Revision
    0x2A28 - Software Revision
    0x2A29 - Manufacturer Name

  0x180F - Battery Service
    0x2A19 - Battery Level (0-100%)

  0x1810 - Blood Pressure
    0x2A35 - BP Measurement (Notify)
    0x2A38 - Body Sensor Location

  0x1822 - Indoor Positioning
  0x181A - Environmental Sensing
    0x2A6E - Temperature
    0x2A6F - Humidity
  ...

厂商自定义 Service (128-bit UUID):
  如 Xiaomi: 0000FE95-0000-1000-8000-00805F9B34FB
  如 Huawei: 0000FE90-0000-1000-8000-00805F9B34FB
  (厂商通过自定义 Service 实现私有配网协议)
```

### 3.6 阶段五：配置数据交换

GATT 服务发现后，App 通过读写 GATT 特征值与设备交换配置数据。这一步是配网的核心——把设备从"裸机"状态配置为"可工作"状态。

**典型配置数据内容：**

```
配置数据包 (通过厂商自定义 Service 传输):

写设备特征值 (App → 设备):
  ┌──────────────────────────────────────────┐
  │ 配置项                  │ 值              │
  ├────────────────────────┼────────────────┤
  │ 用户 ID                │ "user_123456"   │
  │ 用户 Token             │ "a1b2c3d4..."   │
  │ 时间戳                 │ 1719907200      │
  │ 时区                   │ "Asia/Shanghai" │
  │ 云端服务器地址          │ "api.example.com"│
  │ 数据加密密钥            │ (16B AES Key)   │
  │ 设备名称               │ "客厅温度计"     │
  │ 测量间隔               │ 60 (秒)         │
  └────────────────────────┴────────────────┘

读设备信息 (App ← 设备):
  ┌──────────────────────────────────────────┐
  │ 设备信息项              │ 值              │
  ├────────────────────────┼────────────────┤
  │ 设备唯一标识 (DeviceID) │ "MAC或序列号"   │
  │ 固件版本               │ "1.2.3"        │
  │ 硬件版本               │ "2.0"          │
  │ 配网状态               │ "configured"   │
  │ 电池电量               │ 85 (%)         │
  └────────────────────────┴────────────────┘
```

**配置协议设计模式：**

大多数厂商使用自定义的"配置 Service"来交换配置数据，常见的设计模式：

```
配置 Service 结构:
  Service UUID: (厂商128-bit UUID)

  Characteristic 1: Config Write (Write)
    - App 写入配置指令 (JSON/Protobuf/二进制TLV)
    - 设备解析并执行

  Characteristic 2: Config Response (Notify)
    - 设备通过 Notification 返回执行结果
    - App 监听并解析

  Characteristic 3: Device Info (Read)
    - App 读取设备基本信息

配置指令格式示例 (TLV):
  ┌──────┬──────┬────────────────────┐
  │ Type │ Len  │ Value              │
  │ (1B) │ (1B) │ (Len bytes)        │
  ├──────┼──────┼────────────────────┤
  │ 0x01 │ 12   │ "user_123456"     │ ← 用户ID
  │ 0x02 │ 32   │ "a1b2c3d4..."     │ ← 用户Token
  │ 0x03 │  4   │ 0x668F9A00        │ ← 时间戳(LE)
  │ 0x04 │  8   │ "Asia/Sha"        │ ← 时区(截断)
  │ 0x05 │ 16   │ [AES Key]         │ ← 加密密钥
  │ 0xFF │  0   │                   │ ← 配置完成标志
  └──────┴──────┴────────────────────┘
```

**配置完成确认：**

设备收到所有配置数据后，执行以下操作：
1. 将配置数据写入 Flash 持久化存储。
2. 切换设备状态：待配网 → 已配网。
3. 修改广播内容：清除"待配网"标志，切换到慢速广播。
4. 通过 Notification 向 App 发送"配置完成"确认。

### 3.7 阶段六：云端账号绑定

设备配置完成后，App 需要将设备绑定到用户账号：

```
App                     云平台                      设备
  │                       │                          │
  │ 1. 绑定请求            │                          │
  │  POST /device/bind     │                          │
  │  {                     │                          │
  │    userId: "user_123", │                          │
  │    deviceId: "AA:BB:CC:DD:EE:FF",               │
  │    deviceType: "thermometer",                     │
  │    model: "TH-001",    │                          │
  │    firmware: "1.2.3",  │                          │
  │    bindingToken: "xxx" │ ← 设备提供的绑定凭证       │
  │  }                     │                          │
  ├──────────────────────→│                          │
  │                       │                          │
  │                       │ 2. 验证 bindingToken      │
  │                       │  (检查设备是否已配网)       │
  │                       │                          │
  │                       │ 3. 创建绑定记录            │
  │                       │  device_table:            │
  │                       │  {deviceId, userId, ...}  │
  │                       │                          │
  │ 4. 绑定成功响应         │                          │
  │  {success: true,       │                          │
  │   deviceId: "..."}     │                          │
  │←──────────────────────┤                          │
  │                       │                          │
  │ 5. App 通知设备        │                          │
  │  "云端绑定完成"         │                          │
  ├──────────────────────────────────────────────────→│
  │                       │                          │
  │                       │    设备切换到正常工作模式    │
```

**绑定凭证（bindingToken）的设计：**

为防止他人恶意绑定设备，通常由设备在配网阶段生成一个一次性绑定凭证：

```
bindingToken 生成方式:
  方案A: 设备使用 DevKey 对 (MAC + 时间戳 + 随机数) 签名
         App 将签名传给云端, 云端用预置的设备公钥验证

  方案B: 设备生成随机 Token, 通过 BLE 传给 App
         App 将 Token 传给云端, 云端记录为设备所有者凭证

  方案C: 设备出厂时预置一次性配网码 (印在包装/标签上)
         用户在 App 中输入配网码, 云端验证后绑定
```

**绑定关系的数据结构：**

```
云端绑定记录:
{
  "deviceId": "AA:BB:CC:DD:EE:FF",
  "userId": "user_123456",
  "deviceType": "thermometer",
  "model": "TH-001",
  "firmware": "1.2.3",
  "bindingTime": 1719907200,
  "deviceName": "客厅温度计",
  "sharedUsers": [],
  "lastOnline": 1719908100,
  "status": "online"
}
```

## 四、数据同步机制

配网完成后，纯蓝牙设备的数据同步有两种模式：

### 4.1 实时同步（在线模式）

手机在 BLE 范围内并保持连接时，设备数据实时同步：

```
设备                     手机 App                    云平台
  │                         │                          │
  │  BLE Notification       │                          │
  │  (温度: 25.6°C)         │                          │
  ├────────────────────────→│                          │
  │                         │  HTTPS POST              │
  │                         │  /device/data/upload     │
  │                         ├─────────────────────────→│
  │                         │                          │
  │                         │  200 OK                  │
  │                         │←─────────────────────────┤
  │                         │                          │
  │  (数据已同步到云端)        │                          │
```

### 4.2 历史数据同步（离线缓存模式）

设备在手机不在范围时缓存数据，下次连接后批量同步：

```
设备内部:
  ┌──────────────────────────┐
  │     数据缓存队列 (Flash)    │
  │  ┌───┬───┬───┬───┬───┐   │
  │  │T1 │T2 │T3 │T4 │T5 │   │ ← 离线期间缓存的测量数据
  │  └───┴───┴───┴───┴───┘   │
  │  每条含: 时间戳 + 数据 + 序号│
  └──────────────────────────┘

重连后同步:
  手机 App                    设备
    │                           │
    │ 1. 读取已同步序号          │
    │  Read: LastSyncSeq        │
    ├──────────────────────────→│
    │←──────────────────────────┤
    │  Seq = 10                 │
    │                           │
    │ 2. 请求 Seq 11 之后的数据   │
    │  Write: RequestFrom(11)   │
    ├──────────────────────────→│
    │                           │
    │ 3. 批量 Notification       │
    │←──────────────────────────┤
    │  Seq11: T=25.6, Time=...  │
    │←──────────────────────────┤
    │  Seq12: T=25.8, Time=...  │
    │←──────────────────────────┤
    │  ...                      │
    │←──────────────────────────┤
    │  Seq15: T=26.1, Time=...  │
    │                           │
    │ 4. 确认同步完成             │
    │  Write: AckSeq(15)        │
    ├──────────────────────────→│
    │                           │
    │  设备清除已同步的缓存数据     │
```

缓存设计要点：
- 缓存存储在 Flash 中（断电不丢失）。
- 每条数据带序号和时间戳，支持断点续传。
- 缓存满时按 FIFO 丢弃最早的数据。
- 同步完成后设备清除已确认的缓存。

## 五、重连与自动恢复

配网绑定后，设备每次上电或从覆盖范围外回来时，需要自动恢复连接：

### 5.1 使用已绑定密钥重连

```
设备 (Peripheral)              手机 (Central)
  │                              │
  │ 1. 广播 (使用RPA随机地址)     │
  │  ADV_IND                     │
  ├─────────────────────────────→│
  │                              │
  │                              │ 2. 手机扫描, 通过IRK解析RPA
  │                              │  "这是我的设备 AA:BB:CC..."
  │                              │
  │ 3. 发起连接                   │
  │←─────────────────────────────┤
  │                              │
  │ 4. 加密连接 (使用已保存的LTK)  │
  │←──── encrypted_link ────────→│
  │                              │
  │ 5. 直接进入数据同步            │
  │  (跳过配对和服务发现)           │
```

**RPA（Resolvable Private Address）解析：**

为保护隐私，已配网设备通常使用 RPA 而非真实 MAC 地址广播。手机使用绑定时保存的 IRK 解析 RPA：

```
RPA 生成:
  Random Part (24b) = AES(IRK, Prand) ^ Prand
  RPA = 0x40 || Random Part || Prand (24b)

RPA 解析:
  手机收到 RPA 后:
  1. 提取 Prand
  2. 计算 hash = AES(IRK, Prand)
  3. 比较 hash ^ Prand 是否等于 Random Part
  4. 如果匹配, 则识别出设备身份
```

### 5.2 后台重连策略

iOS 和 Android 的 BLE 后台行为差异很大：

| 平台 | 后台扫描 | 后台连接 | 注意事项 |
|------|---------|---------|---------|
| iOS | 仅 iBeacon 或 State Restoration | 可维持连接 | 后台扫描被系统限制，App 可能被挂起 |
| Android | 可持续后台扫描（需权限） | 可维持连接 | 不同厂商行为差异大，需省电优化 |
| iOS 后台 | 广播可继续 | 连接可能被系统断开 | 需使用 State Restoration 恢复 |
| Android 后台 | 广播可继续 | 连接可维持 | Doze 模式可能限制后台 BLE |

## 六、设备分享

绑定的设备可以分享给家庭其他成员：

```
设备拥有者                    云平台                    被分享者
    │                          │                          │
    │ 1. 创建分享邀请            │                          │
    │  POST /device/share       │                          │
    │  {deviceId, inviteeId}    │                          │
    ├─────────────────────────→│                          │
    │                          │                          │
    │                          │ 2. 生成分享码/通知          │
    │                          │  push通知给被分享者         │
    │                          ├─────────────────────────→│
    │                          │                          │
    │                          │ 3. 被分享者接受分享         │
    │                          │←─────────────────────────┤
    │                          │                          │
    │                          │ 4. 下发设备绑定信息         │
    │                          │  (设备MAC, IRK, LTK等)    │
    │                          ├─────────────────────────→│
    │                          │                          │
    │                          │ 5. 被分享者扫描连接设备     │
    │                          │                          ├──→ BLE连接
```

分享的关键问题：
- **密钥分发**：被分享者需要获得设备的 IRK 和 LTK 才能连接加密设备。这需要拥有者通过云端转发密钥，或设备支持多用户绑定（各自独立配对）。
- **权限控制**：分享可分为"管理员"和"普通用户"，普通用户可能只能读取数据不能修改配置。
- **撤销分享**：拥有者可随时撤销分享，云端删除分享记录。已撤销用户无法再连接设备（需设备端也更新权限）。

## 七、配网安全要点

### 7.1 常见攻击与防护

| 攻击方式 | 说明 | 防护措施 |
|---------|------|---------|
| 配对窃听 | 攻击者监听配对过程 | 使用 LE Secure Connections (ECDH) |
| 中间人攻击 | 攻击者冒充双方 | 使用 Passkey/OOB 认证 |
| 重放攻击 | 录制并重放配网数据 | 配置数据带时间戳和序号 |
| 伪造绑定 | 非授权用户绑定设备 | 设备生成一次性 bindingToken |
| 恶意解绑 | 攻击者解绑他人设备 | 云端验证用户身份后才允许解绑 |
| 密钥泄露 | LTK 被提取 | 使用安全存储（TrustZone/SE） |

### 7.2 设备重置

设备重置（恢复出厂设置）是配网安全的重要环节：

```
重置方式:
  1. 物理重置: 长按设备按钮 10 秒
  2. App 重置: App 发送重置命令 (需已绑定用户权限)
  3. 云端重置: 用户在云端发起远程解绑

重置后设备操作:
  - 清除所有已保存的绑定密钥 (LTK, IRK, CSRK)
  - 清除配置数据 (用户ID, Token等)
  - 清除缓存数据
  - 恢复到"待配网"状态
  - 恢复快速广播模式
  - 通知云端解除绑定关系 (如能联网)
```

## 八、与 Wi-Fi 配网的对比

| 维度 | 纯蓝牙设备配网 | Wi-Fi 设备配网 |
|------|--------------|---------------|
| 核心目标 | 建立设备-账号绑定关系 | 传递 Wi-Fi 凭证 + 账号绑定 |
| 网络依赖 | 不依赖局域网 | 必须接入 Wi-Fi |
| 配网通道 | BLE 连接 | BLE/广播/AP热点等 |
| 在线状态 | 手机在范围内才在线 | 连接路由器即在线 |
| 数据同步 | 手机中转或缓存同步 | 直连云端实时同步 |
| 距离限制 | BLE 范围 (10-50m) | Wi-Fi 范围 (整个家庭) |
| 功耗 | 极低 | 较高 |
| 典型设备 | 体温计、门锁、体重秤 | 摄像头、插座、灯具 |

## 九、总结

纯蓝牙设备的配网流程包含六个阶段：设备发现（待配网广播）→ 扫描连接 → 配对认证（建立加密链路）→ GATT 服务发现 → 配置数据交换（下发用户信息和参数）→ 云端账号绑定（建立归属关系）。与 Wi-Fi 配网不同，纯蓝牙配网的核心不是网络凭证传递，而是建立设备与用户账号的安全绑定关系。配网完成后，设备需要支持历史数据缓存与断点续传同步，以应对手机不在 BLE 范围的场景。在安全方面，应优先使用 LE Secure Connections 配对，通过 bindingToken 防止伪造绑定，并通过 RPA 保护设备隐私。理解纯蓝牙设备的完整配网流程，是开发蓝牙 IoT 产品的基础。
