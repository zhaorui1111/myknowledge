## 1. 概念总览：堆与优先队列是什么

### 1.1 优先队列（Priority Queue）

**优先队列**是一种抽象数据类型（ADT, Abstract Data Type），它和普通队列（Queue）的区别在于：普通队列遵循「先进先出」（FIFO, First-In-First-Out），而优先队列每次出队的是**优先级最高**（或最低）的元素，与入队顺序无关。

优先队列对外暴露的核心操作通常是：

- `insert(x)` / `push(x)`：插入一个带优先级的元素。
- `extractMax()` / `pop()`：取出并删除优先级最高的元素（最大堆）；或 `extractMin()`（最小堆）。
- `peek()` / `top()`：查看优先级最高的元素但不删除。

注意「优先队列」是**接口/ADT 概念**，并没有规定底层用什么数据结构实现。它可以用无序数组、有序数组、有序链表、平衡二叉搜索树、堆等多种方式实现，只是各自复杂度不同。

| 实现方式 | insert | extractMax | peek |
|---|---|---|---|
| 无序数组/链表 | O(1) | O(n) | O(n) |
| 有序数组/链表 | O(n) | O(1) | O(1) |
| 平衡 BST（如红黑树） | O(log n) | O(log n) | O(log n) |
| **二叉堆（Binary Heap）** | **O(log n)** | **O(log n)** | **O(1)** |

可以看到，二叉堆在 insert 与 extractMax 之间取得了优雅的平衡，且 peek 为 O(1)，这正是它成为优先队列首选实现的原因。

### 1.2 堆（Heap）

**堆**是一种满足**堆序性质（heap property）**的**完全二叉树（complete binary tree）**：

- **最大堆（Max-Heap）**：任意节点的值都 ≥ 其子节点的值。因此根节点是全局最大值。
- **最小堆（Min-Heap）**：任意节点的值都 ≤ 其子节点的值。因此根节点是全局最小值。

注意区分两个「堆」：

1. **数据结构的堆（Heap）**：本篇主题，一棵完全二叉树。
2. **内存的堆（Heap memory）**：运行时动态分配内存的区域（与栈内存相对）。两者**毫无关系**，只是恰好同名，面试和阅读时务必根据上下文区分。

> ⚠️ 常见误解：堆**不是**有序的。堆只保证「父 ≥ 子」（最大堆）这一局部关系，**兄弟之间、不同子树之间没有顺序保证**。所以堆的中序遍历不是有序序列，这与二叉搜索树（BST）有本质区别。堆只擅长「快速拿到最值」，不擅长「查找任意元素」或「有序遍历」。

## 2. 为什么需要堆：从「反复找最值」说起

考虑一个典型需求：维护一个动态集合，频繁地「插入新元素」并「取出当前最大值」。

**朴素方案一：无序数组。** 插入 O(1)（直接追加），但每次取最大值要扫描整个数组 O(n)，删除后还要搬移元素。若进行 n 次「插入+取最大」操作，总复杂度 O(n²)。

**朴素方案二：始终保持有序数组。** 取最大值 O(1)，但插入要找到位置并搬移，O(n)。同样退化为 O(n²)。

**方案三：平衡 BST（红黑树）。** 插入与删除最值都是 O(log n)，可行，但实现复杂、常数大，且功能「过剩」（我们并不需要按任意 key 查找、范围查询）。

**方案四：二叉堆。** 插入与取最值都是 O(log n)，peek 为 O(1)，实现简单、常数小、缓存友好（数组连续存储）。n 次操作总复杂度 O(n log n)。

这正是堆诞生的动机：**当你只关心「最值」而不关心「全序」时，堆是性价比最高的结构。** 它牺牲了「任意查找」能力，换来了「维护最值」的高效与简洁。

经典案例对比：

- **任务调度器**：操作系统/语言运行时需要从大量待执行任务中，反复选出「优先级最高」或「截止时间最近」的任务执行 → 优先队列。
- **Dijkstra 最短路**：每次从未确定的节点中选出「距离起点最近」的节点 → 最小堆。
- **Top-K 热搜**：海量数据流中维护「最热门的 K 个」 → 大小为 K 的堆。
- **流式中位数**：实时数据流中随时查询中位数 → 双堆对顶。

## 3. 堆的形态：完全二叉树与数组映射

### 3.1 完全二叉树（Complete Binary Tree）

完全二叉树的定义：除了最后一层，其余层都被节点填满；且最后一层的节点都**靠左连续排列**。

```
        100
       /    \
      19      36
     /  \    /  \
    17   3  25   1
   / \
  2   7
```

这种「层序填满、最后一层靠左」的结构，使得我们可以用**一维数组**来紧凑存储整棵树，无需任何指针。这是堆能高效运行的关键工程优势：

- 内存连续，**缓存命中率高**（CPU cache friendly）。
- 无需为左右孩子指针额外分配内存。
- 父子关系通过下标算术直接计算，O(1)。

### 3.2 数组下标映射公式

将完全二叉树按层序（从上到下、从左到右）编号，存入数组。有两种常见约定：

**约定一：下标从 0 开始（本篇主要采用）**

对于下标为 `i` 的节点：

- 父节点：`parent(i) = (i - 1) / 2`（整数除法）
- 左孩子：`left(i) = 2 * i + 1`
- 右孩子：`right(i) = 2 * i + 2`

**约定二：下标从 1 开始（很多教材采用，数组 0 号位空置或放哨兵）**

- 父节点：`parent(i) = i / 2`
- 左孩子：`left(i) = 2 * i`
- 右孩子：`right(i) = 2 * i + 1`

从 1 开始的公式更简洁优雅（纯位运算 `i << 1`、`i >> 1`），但浪费一个数组位置；从 0 开始更贴合现代语言的数组习惯。**本篇统一采用从 0 开始的约定**，请在面试与工程中明确说明你用的是哪种约定，避免 off-by-one 错误。

上面那棵树的数组表示（从 0 开始）：

```
下标:  0    1    2    3   4   5   6   7  8
值:  [100, 19,  36,  17, 3, 25, 1,  2, 7]
```

验证：下标 1（值 19）的左孩子下标 = 2*1+1 = 3（值 17）✓；右孩子 = 2*1+2 = 4（值 3）✓；父 = (1-1)/2 = 0（值 100）✓。

### 3.3 几个重要事实

- 数组长度为 `n` 时，**最后一个非叶子节点**的下标是 `(n - 1 - 1) / 2 = n/2 - 1`（即最后一个节点 `n-1` 的父节点）。下标 `> n/2 - 1` 的全是叶子节点。这个事实在「自底向上建堆」时至关重要。
- 完全二叉树的高度为 `⌊log₂ n⌋`，因此从某节点上浮/下沉到底的代价不超过树高 O(log n)。

## 4. 堆序性质与两个核心操作：上浮 / 下沉

维护堆序性质依赖两个基础操作（以**最大堆**为例）：

### 4.1 上浮（sift-up / swim / bubble-up）

当某个节点的值**变大**（或新插入到末尾），可能违反「父 ≥ 子」。处理方式：不断与父节点比较，若比父大就交换，直到不再比父大或到达根。

伪代码：

```
siftUp(i):
    while i > 0 and a[i] > a[parent(i)]:
        swap(a[i], a[parent(i)])
        i = parent(i)
```

**复杂度**：最坏沿一条路径浮到根，O(log n)。

### 4.2 下沉（sift-down / sink / bubble-down / heapify）

当某个节点的值**变小**（或根被替换为末尾元素），可能违反堆序。处理方式：不断与**较大的那个孩子**比较，若比孩子小就交换，直到比所有孩子大或成为叶子。

> ⚠️ 关键易错点：下沉时必须和**两个孩子中较大的那个**交换（最大堆），否则交换后仍可能违反堆序。

伪代码：

```
siftDown(i, n):    // n 为堆的有效大小
    while true:
        l = left(i); r = right(i); largest = i
        if l < n and a[l] > a[largest]: largest = l
        if r < n and a[r] > a[largest]: largest = r
        if largest == i: break
        swap(a[i], a[largest])
        i = largest
```

**复杂度**：最坏沉到叶子，O(log n)。

### 4.3 两大操作支撑全部 API

- `push(x)`：把 x 放到数组末尾，对它 `siftUp`。O(log n)
- `pop()`：取出根（a[0]），把末尾元素移到根，长度减一，对根 `siftDown`。O(log n)
- `peek()`：返回 a[0]。O(1)
- `buildHeap()`：从最后一个非叶子节点起向前对每个节点 `siftDown`。O(n)（见第 6 节证明）

## 5. 二叉堆的完整实现（Swift + Objective-C）

下面给出工程级、可运行、带边界处理的二叉堆。先实现一个泛型最大堆/最小堆，再封装为优先队列。

### 5.1 Swift 实现（泛型，比较器驱动，主实现）

```swift
import Foundation

/// 通用二叉堆。通过传入比较器 `areInIncreasingOrder` 决定是最小堆还是最大堆。
/// - 传入 `<`：父在前的元素「更小」，得到最小堆（堆顶最小）。
/// - 传入 `>`：父在前的元素「更大」，得到最大堆（堆顶最大）。
struct BinaryHeap<Element> {
    /// 底层存储，按完全二叉树层序排列。
    private var storage: [Element] = []
    /// 比较器：areInIncreasingOrder(a, b) == true 表示 a 应排在 b 前面（更靠近堆顶）。
    private let areInIncreasingOrder: (Element, Element) -> Bool

    init(_ areInIncreasingOrder: @escaping (Element, Element) -> Bool) {
        self.areInIncreasingOrder = areInIncreasingOrder
    }

    /// 用已有数组建堆，O(n)。
    init(_ elements: [Element], _ areInIncreasingOrder: @escaping (Element, Element) -> Bool) {
        self.storage = elements
        self.areInIncreasingOrder = areInIncreasingOrder
        buildHeap()
    }

    var count: Int { storage.count }
    var isEmpty: Bool { storage.isEmpty }

    /// 查看堆顶，O(1)。空堆返回 nil。
    func peek() -> Element? { storage.first }

    // MARK: - 下标计算
    private func parent(_ i: Int) -> Int { (i - 1) / 2 }
    private func left(_ i: Int) -> Int { 2 * i + 1 }
    private func right(_ i: Int) -> Int { 2 * i + 2 }

    /// 插入元素，O(log n)。
    mutating func push(_ value: Element) {
        storage.append(value)
        siftUp(from: storage.count - 1)
    }

    /// 弹出堆顶，O(log n)。空堆返回 nil。
    mutating func pop() -> Element? {
        guard !storage.isEmpty else { return nil }
        let top = storage[0]
        let last = storage.count - 1
        storage.swapAt(0, last)        // 把末尾换到堆顶
        storage.removeLast()           // 删除原堆顶（现在在末尾）
        if !storage.isEmpty {
            siftDown(from: 0)
        }
        return top
    }

    /// 替换堆顶为新值并重新调整，等价于 pop()+push() 但只需一次下沉，O(log n)。
    mutating func replaceTop(_ value: Element) -> Element? {
        guard !storage.isEmpty else { push(value); return nil }
        let old = storage[0]
        storage[0] = value
        siftDown(from: 0)
        return old
    }

    // MARK: - 上浮
    private mutating func siftUp(from index: Int) {
        var child = index
        var p = parent(child)
        // 当 child 比父更应靠前（最大堆里更大）时，交换上浮
        while child > 0 && areInIncreasingOrder(storage[child], storage[p]) {
            storage.swapAt(child, p)
            child = p
            p = parent(child)
        }
    }

    // MARK: - 下沉
    private mutating func siftDown(from index: Int) {
        var parentIdx = index
        let n = storage.count
        while true {
            let l = left(parentIdx)
            let r = right(parentIdx)
            var candidate = parentIdx
            if l < n && areInIncreasingOrder(storage[l], storage[candidate]) {
                candidate = l
            }
            if r < n && areInIncreasingOrder(storage[r], storage[candidate]) {
                candidate = r
            }
            if candidate == parentIdx { return }
            storage.swapAt(parentIdx, candidate)
            parentIdx = candidate
        }
    }

    // MARK: - 建堆 O(n)
    private mutating func buildHeap() {
        // 从最后一个非叶子节点开始，向前依次下沉
        guard storage.count > 1 else { return }
        for i in stride(from: storage.count / 2 - 1, through: 0, by: -1) {
            siftDown(from: i)
        }
    }
}

// MARK: - 便捷构造（Comparable 类型）
extension BinaryHeap where Element: Comparable {
    /// 最小堆
    static func minHeap() -> BinaryHeap { BinaryHeap(<) }
    /// 最大堆
    static func maxHeap() -> BinaryHeap { BinaryHeap(>) }
}
```

使用示例：

```swift
// 最小堆
var minH = BinaryHeap<Int>.minHeap()
[5, 1, 8, 3, 2].forEach { minH.push($0) }
print(minH.peek()!)        // 1
print(minH.pop()!)         // 1
print(minH.pop()!)         // 2

// 最大堆 + O(n) 建堆
var maxH = BinaryHeap([5, 1, 8, 3, 2], >)
print(maxH.pop()!)         // 8

// 自定义优先级：按任务截止时间最小优先
struct Task { let name: String; let deadline: Int }
var pq = BinaryHeap<Task> { $0.deadline < $1.deadline }
pq.push(Task(name: "A", deadline: 9))
pq.push(Task(name: "B", deadline: 3))
print(pq.peek()!.name)     // B（最早截止）
```

### 5.2 Objective-C 实现（对照，行为一致）

Objective-C 没有泛型语法糖（轻量级泛型仅作编译期提示），这里用 `NSMutableArray` 存 `id`，通过传入 `NSComparator` block 决定堆类型，**行为与上面的 Swift 版严格一致**。

```objectivec
// MTBinaryHeap.h
#import <Foundation/Foundation.h>

NS_ASSUME_NONNULL_BEGIN

/// 通用二叉堆。
/// comparator(a, b) == NSOrderedAscending 表示 a 应排在 b 前面（更靠近堆顶）。
/// 传入升序比较器 → 最小堆；传入降序比较器 → 最大堆。
@interface MTBinaryHeap<ObjectType> : NSObject

- (instancetype)initWithComparator:(NSComparator)comparator;
- (instancetype)initWithArray:(NSArray<ObjectType> *)array
                   comparator:(NSComparator)comparator;

@property (nonatomic, readonly) NSUInteger count;
@property (nonatomic, readonly) BOOL isEmpty;

- (nullable ObjectType)peek;
- (void)push:(ObjectType)value;
- (nullable ObjectType)pop;
- (nullable ObjectType)replaceTop:(ObjectType)value;

@end

NS_ASSUME_NONNULL_END
```

```objectivec
// MTBinaryHeap.m
#import "MTBinaryHeap.h"

@implementation MTBinaryHeap {
    NSMutableArray *_storage;
    NSComparator _comparator;
}

- (instancetype)initWithComparator:(NSComparator)comparator {
    if (self = [super init]) {
        _storage = [NSMutableArray array];
        _comparator = [comparator copy];
    }
    return self;
}

- (instancetype)initWithArray:(NSArray *)array comparator:(NSComparator)comparator {
    if (self = [super init]) {
        _storage = [array mutableCopy];
        _comparator = [comparator copy];
        [self buildHeap];
    }
    return self;
}

- (NSUInteger)count { return _storage.count; }
- (BOOL)isEmpty { return _storage.count == 0; }
- (id)peek { return _storage.count > 0 ? _storage.firstObject : nil; }

#pragma mark - 下标计算
static inline NSInteger parentIndex(NSInteger i) { return (i - 1) / 2; }
static inline NSInteger leftIndex(NSInteger i)   { return 2 * i + 1; }
static inline NSInteger rightIndex(NSInteger i)  { return 2 * i + 2; }

- (void)swap:(NSInteger)i with:(NSInteger)j {
    [_storage exchangeObjectAtIndex:i withObjectAtIndex:j];
}

/// 比较：a 是否应排在 b 前面（更靠近堆顶）
- (BOOL)shouldComeFirst:(id)a than:(id)b {
    return _comparator(a, b) == NSOrderedAscending;
}

#pragma mark - push / pop
- (void)push:(id)value {
    [_storage addObject:value];
    [self siftUpFrom:(NSInteger)_storage.count - 1];
}

- (id)pop {
    if (_storage.count == 0) return nil;
    id top = _storage[0];
    NSInteger last = (NSInteger)_storage.count - 1;
    [self swap:0 with:last];
    [_storage removeLastObject];
    if (_storage.count > 0) {
        [self siftDownFrom:0];
    }
    return top;
}

- (id)replaceTop:(id)value {
    if (_storage.count == 0) { [self push:value]; return nil; }
    id old = _storage[0];
    _storage[0] = value;
    [self siftDownFrom:0];
    return old;
}

#pragma mark - 上浮 / 下沉
- (void)siftUpFrom:(NSInteger)index {
    NSInteger child = index;
    NSInteger p = parentIndex(child);
    while (child > 0 && [self shouldComeFirst:_storage[child] than:_storage[p]]) {
        [self swap:child with:p];
        child = p;
        p = parentIndex(child);
    }
}

- (void)siftDownFrom:(NSInteger)index {
    NSInteger parentIdx = index;
    NSInteger n = (NSInteger)_storage.count;
    while (YES) {
        NSInteger l = leftIndex(parentIdx);
        NSInteger r = rightIndex(parentIdx);
        NSInteger candidate = parentIdx;
        if (l < n && [self shouldComeFirst:_storage[l] than:_storage[candidate]]) {
            candidate = l;
        }
        if (r < n && [self shouldComeFirst:_storage[r] than:_storage[candidate]]) {
            candidate = r;
        }
        if (candidate == parentIdx) break;
        [self swap:parentIdx with:candidate];
        parentIdx = candidate;
    }
}

#pragma mark - 建堆 O(n)
- (void)buildHeap {
    if (_storage.count <= 1) return;
    for (NSInteger i = (NSInteger)_storage.count / 2 - 1; i >= 0; i--) {
        [self siftDownFrom:i];
    }
}

@end
```

使用示例（OC）：

```objectivec
// 最小堆
MTBinaryHeap<NSNumber *> *minH =
    [[MTBinaryHeap alloc] initWithComparator:^NSComparisonResult(NSNumber *a, NSNumber *b) {
        return [a compare:b];          // 升序 → 最小堆
    }];
for (NSNumber *n in @[@5, @1, @8, @3, @2]) { [minH push:n]; }
NSLog(@"%@", [minH peek]);              // 1
NSLog(@"%@", [minH pop]);               // 1

// 最大堆
MTBinaryHeap<NSNumber *> *maxH =
    [[MTBinaryHeap alloc] initWithArray:@[@5, @1, @8, @3, @2]
                             comparator:^NSComparisonResult(NSNumber *a, NSNumber *b) {
        return [b compare:a];          // 降序 → 最大堆
    }];
NSLog(@"%@", [maxH pop]);               // 8
```

> 注意：Swift 版与 OC 版采用了**完全相同的比较语义约定**——「比较器返回『a 应排在 b 前面』即 a 更靠近堆顶」。Swift 用 `Bool`，OC 用 `NSOrderedAscending`，但传升序得最小堆、传降序得最大堆这一行为在两端是一致的。

### 5.3 push 操作逐步图解（最大堆）

设当前最大堆数组为 `[100, 19, 36, 17, 3, 25, 1]`，对应树：

```
        100(0)
       /      \
     19(1)    36(2)
    /   \     /   \
  17(3) 3(4) 25(5) 1(6)
```

现在 `push(50)`：

第 1 步，把 50 追加到数组末尾（下标 7）：

```
数组: [100, 19, 36, 17, 3, 25, 1, 50]
                                   ↑ 新元素 50 在下标 7
```

第 2 步，siftUp(7)。父 = (7-1)/2 = 3，a[3]=17。50 > 17，交换：

```
数组: [100, 19, 36, 50, 3, 25, 1, 17]
              ↑ 50 上浮到下标 3
```

第 3 步，child=3，父 = (3-1)/2 = 1，a[1]=19。50 > 19，交换：

```
数组: [100, 50, 36, 19, 3, 25, 1, 17]
          ↑ 50 上浮到下标 1
```

第 4 步，child=1，父 = (1-1)/2 = 0，a[0]=100。50 < 100，停止。

最终堆：`[100, 50, 36, 19, 3, 25, 1, 17]`，堆序恢复。50 沿路径上浮了 3 步（树高级别），符合 O(log n)。

### 5.4 pop 操作逐步图解（最大堆）

接上，对 `[100, 50, 36, 19, 3, 25, 1, 17]` 执行 `pop()`：

第 1 步，记下堆顶 top = 100（待返回）。把末尾 a[7]=17 换到堆顶，删除末尾：

```
数组: [17, 50, 36, 19, 3, 25, 1]
       ↑ 17 暂居堆顶，堆序被破坏
```

第 2 步，siftDown(0)。左孩子 a[1]=50，右孩子 a[2]=36，较大者是 50（下标 1）。17 < 50，交换：

```
数组: [50, 17, 36, 19, 3, 25, 1]
           ↑ 17 下沉到下标 1
```

第 3 步，parent=1。左孩子 a[3]=19，右孩子 a[4]=3，较大者 19（下标 3）。17 < 19，交换：

```
数组: [50, 19, 36, 17, 3, 25, 1]
                   ↑ 17 下沉到下标 3
```

第 4 步，parent=3。左孩子下标 = 7 ≥ n(=7)，无孩子，停止。返回 top = 100。

最终堆顶变为 50（次大值升为新最大），堆序正确。17 下沉了 2 步，O(log n)。

> 通过这两个 trace 可以清楚看到：siftUp 只与「父」单向比较并上移，siftDown 与「较大孩子」比较并下移。手写堆时，建议像这样在草稿纸上画出数组下标对照，能极大减少 off-by-one 与方向错误。

### 5.5 关于 swift-collections 的 min-max heap 原理补充

第 9 节会提到 swift-collections 的 `Heap` 是 min-max heap（双端堆）。其原理是：完全二叉树按层交替维护两种性质——

- 偶数层（深度 0, 2, 4...）为 min 层：该节点是其子树的最小值。
- 奇数层（深度 1, 3, 5...）为 max 层：该节点是其子树的最大值。

于是根（深度 0）是全局最小，根的两个孩子（深度 1）中较大者是全局最大。这样既能 O(1) 取 min 也能 O(1) 取 max，popMin / popMax 均为 O(log n)。它的 siftDown 需要「跨两层」比较孙子节点，逻辑比普通二叉堆复杂，但思想同源。理解了普通二叉堆，再看 min-max heap 只是「按层分工」的推广。

## 6. 建堆：自顶向下 vs 自底向上，O(n) 证明

把一个无序数组变成堆，有两种思路：

### 6.1 自顶向下（逐个插入）

从空堆开始，依次 `push` 每个元素，每次 `siftUp`。

```swift
var heap = BinaryHeap<Int>.maxHeap()
for x in array { heap.push(x) }   // n 次 push，每次 O(log n)
```

总复杂度上界为 O(n log n)。这是最直观的方式，但**不是最优**。

### 6.2 自底向上（Floyd 建堆）

先把整个数组当成一棵「形状正确但堆序错误」的完全二叉树，然后从**最后一个非叶子节点**开始，向前对每个节点执行 `siftDown`。

```swift
for i in stride(from: n / 2 - 1, through: 0, by: -1) {
    siftDown(from: i)
}
```

这种方式（Floyd 算法，1964 年提出）的复杂度是 **O(n)**，优于逐个插入。

### 6.3 为什么自底向上是 O(n)：严格证明

直觉上「n 个节点，每个 siftDown 是 O(log n)，总和应该是 O(n log n)」——这是**错误的**估计，因为它假设每个节点都要下沉到底。事实是：**绝大多数节点位于底层附近，下沉高度很小。**

设堆高度为 `h = ⌊log₂ n⌋`。在高度为 `k` 的层上，最多有 `⌈n / 2^(k+1)⌉` 个节点，每个节点 `siftDown` 的代价为 O(k)。总工作量：

$$
T(n) = \sum_{k=0}^{h} \left\lceil \frac{n}{2^{k+1}} \right\rceil \cdot O(k) \le O\!\left( n \sum_{k=0}^{h} \frac{k}{2^{k+1}} \right)
$$

利用级数恒等式 $\sum_{k=0}^{\infty} \frac{k}{2^k} = 2$（收敛），可得：

$$
\sum_{k=0}^{h} \frac{k}{2^{k+1}} = \frac{1}{2}\sum_{k=0}^{h} \frac{k}{2^{k}} \le \frac{1}{2} \cdot 2 = 1
$$

因此 $T(n) = O(n)$。

**直观理解**：高度为 0 的叶子约占一半（n/2 个），它们根本不需要下沉；高度为 1 的节点约 n/4 个，最多下沉 1 步；越往上节点越少、下沉步数虽多但总贡献被指数级衰减的节点数压住。求和收敛到常数倍的 n。

> 📌 记忆点：**逐个插入建堆是 O(n log n)，自底向上 Floyd 建堆是 O(n)。** 但要注意，「堆排序」整体仍是 O(n log n)，因为后续 n 次 extract 各 O(log n)，建堆的 O(n) 只是优化了第一阶段。

## 7. 优先队列 API 设计与泛型封装

「优先队列」可以直接对二叉堆做一层语义封装，使其 API 更贴合业务（`enqueue` / `dequeue`）。

### 7.1 Swift 优先队列封装

```swift
/// 优先队列：对 BinaryHeap 的语义封装。
struct PriorityQueue<Element> {
    private var heap: BinaryHeap<Element>

    /// priorityFn 返回更小值表示优先级更高（越先出队）。
    init(sort: @escaping (Element, Element) -> Bool) {
        heap = BinaryHeap(sort)
    }

    var count: Int { heap.count }
    var isEmpty: Bool { heap.isEmpty }

    mutating func enqueue(_ element: Element) { heap.push(element) }
    mutating func dequeue() -> Element? { heap.pop() }
    func front() -> Element? { heap.peek() }
}

// 示例：Dijkstra 风格的 (距离, 节点) 队列，按距离升序出队
struct State { let dist: Int; let node: Int }
var frontier = PriorityQueue<State> { $0.dist < $1.dist }
frontier.enqueue(State(dist: 0, node: 0))
```

### 7.2 设计要点

- **比较器外置**：把「优先级如何定义」交给调用方，堆本身只负责维护堆序，符合「机制与策略分离」原则。
- **稳定性**：二叉堆**不是稳定的**——相同优先级的元素出队顺序不保证与入队顺序一致。若业务要求「同优先级 FIFO」，需在元素中附加一个**单调递增的序号**作为次级比较键：

```swift
struct StableItem<T> { let value: T; let seq: Int }
var pq = PriorityQueue<StableItem<String>> { a, b in
    // 主键：业务优先级（此处省略）；次键：序号小者先出
    a.seq < b.seq
}
```

- **可变优先级**：标准二叉堆不支持高效地「修改某个已入堆元素的优先级」（需要先 O(n) 找到它）。若有此需求，应使用**索引堆**（见第 14 节）。

## 8. 堆排序（Heap Sort）

堆排序是基于堆的原地（in-place）排序算法。

### 8.1 算法步骤（升序，用最大堆）

1. 把数组**自底向上建成最大堆**，O(n)。此时堆顶 a[0] 是全局最大值。
2. 把 a[0] 与当前末尾元素交换，将最大值「沉淀」到数组末尾，堆有效大小减一。
3. 对新的堆顶 `siftDown`，恢复堆序，O(log n)。
4. 重复 2~3，直到堆大小为 1。最终数组升序排列。

### 8.2 Swift 实现

```swift
func heapSort(_ array: inout [Int]) {
    let n = array.count
    guard n > 1 else { return }

    // 内嵌下沉，作用于 array[0..<size]，维护最大堆
    func siftDown(from i: Int, size: Int) {
        var parent = i
        while true {
            let l = 2 * parent + 1, r = 2 * parent + 2
            var largest = parent
            if l < size && array[l] > array[largest] { largest = l }
            if r < size && array[r] > array[largest] { largest = r }
            if largest == parent { return }
            array.swapAt(parent, largest)
            parent = largest
        }
    }

    // 1. 自底向上建最大堆 O(n)
    for i in stride(from: n / 2 - 1, through: 0, by: -1) {
        siftDown(from: i, size: n)
    }
    // 2. 依次把堆顶换到末尾并缩小堆
    for end in stride(from: n - 1, through: 1, by: -1) {
        array.swapAt(0, end)
        siftDown(from: 0, size: end)
    }
}

var arr = [5, 1, 8, 3, 2, 7]
heapSort(&arr)
print(arr)   // [1, 2, 3, 5, 7, 8]
```

### 8.3 Objective-C 实现

```objectivec
static void siftDownC(NSMutableArray<NSNumber *> *a, NSInteger i, NSInteger size) {
    NSInteger parent = i;
    while (YES) {
        NSInteger l = 2 * parent + 1, r = 2 * parent + 2, largest = parent;
        if (l < size && [a[l] compare:a[largest]] == NSOrderedDescending) largest = l;
        if (r < size && [a[r] compare:a[largest]] == NSOrderedDescending) largest = r;
        if (largest == parent) return;
        [a exchangeObjectAtIndex:parent withObjectAtIndex:largest];
        parent = largest;
    }
}

void heapSortC(NSMutableArray<NSNumber *> *a) {
    NSInteger n = (NSInteger)a.count;
    if (n <= 1) return;
    for (NSInteger i = n / 2 - 1; i >= 0; i--) siftDownC(a, i, n);
    for (NSInteger end = n - 1; end >= 1; end--) {
        [a exchangeObjectAtIndex:0 withObjectAtIndex:end];
        siftDownC(a, 0, end);
    }
}
```

### 8.4 堆排序性质分析

| 维度 | 结论 | 说明 |
|---|---|---|
| 时间（最好/平均/最坏） | 均 O(n log n) | 建堆 O(n) + n 次下沉 O(n log n) |
| 空间 | O(1) | 原地排序，仅常数额外空间 |
| 稳定性 | **不稳定** | 交换会打乱相等元素相对位置 |
| 是否需要随机访问 | 是 | 依赖下标算术，不适合链表 |

**堆排序 vs 快速排序**：两者平均都是 O(n log n)，但快排常数更小、缓存更友好，实践中通常更快；堆排序的优势是**最坏情况也是 O(n log n)**（快排最坏 O(n²)）且空间 O(1)。因此堆排序常用于「需要最坏情况保证」的场景，或作为快排的退化兜底（如 C++ STL 的 introsort 在递归过深时切换到堆排序）。

> 易错点：堆排序求**升序**结果要用**最大堆**（把大的不断换到尾部）；求降序用最小堆。初学者常搞反。

### 9.1 Swift 标准库现状

### 9.3 其他语言对照（帮助建立横向认知）

| 语言 | 类型 | 默认堆性质 | 备注 |
|---|---|---|---|
| C++ | `std::priority_queue<T>` | **最大堆**（默认 `less` → top 最大） | 底层 `std::vector` + `make_heap/push_heap/pop_heap` |

## 10. 经典应用一：Top-K 问题

**问题**：从 n 个元素中找出最大的 K 个（或第 K 大）。

### 10.1 方法对比

| 方法 | 时间 | 空间 | 适用 |
|---|---|---|---|
| 全排序后取前 K | O(n log n) | O(1)~O(n) | n 不大、需要前 K 个有序 |
| 大小为 n 的堆，pop K 次 | O(n + K log n) | O(n) | 数据全在内存 |
| **大小为 K 的小顶堆** | **O(n log K)** | **O(K)** | **海量数据/数据流，K ≪ n** |
| 快速选择（Quickselect） | 平均 O(n)，最坏 O(n²) | O(1) | 一次性、数据可变 |

### 10.2 维护大小为 K 的小顶堆（最优内存方案）

核心思想：维护一个容量为 K 的**小顶堆**。堆顶是这 K 个里最小的。遍历每个新元素：若堆未满则入堆；若堆满且新元素 > 堆顶，则用新元素替换堆顶（弹出最小、压入更大）。遍历结束，堆里就是最大的 K 个，堆顶即第 K 大。

> 关键：求「最大 K 个」用**小顶堆**（堆顶是门槛，比它小的直接淘汰）；求「最小 K 个」用**大顶堆**。又是一个易反点。

```swift
/// 返回 nums 中最大的 k 个元素（顺序不保证），堆顶即第 k 大。
func topK(_ nums: [Int], _ k: Int) -> [Int] {
    guard k > 0 else { return [] }
    var minHeap = BinaryHeap<Int>.minHeap()   // 容量为 k 的小顶堆
    for x in nums {
        if minHeap.count < k {
            minHeap.push(x)
        } else if let top = minHeap.peek(), x > top {
            _ = minHeap.replaceTop(x)          // 一次下沉，O(log k)
        }
    }
    var result: [Int] = []
    while let v = minHeap.pop() { result.append(v) }
    return result
}

print(topK([3,2,1,5,6,4], 2))   // [5, 6]（第 2 大是 5）
```

复杂度：遍历 n 个元素，每个最多一次 O(log K) 堆操作，**O(n log K)**，空间 **O(K)**。当 n 极大（如 10 亿日志）而 K 很小（如 100）时，这是唯一可行的内存方案。

### 10.3 LeetCode 215「数组中第 K 个最大元素」

```swift
func findKthLargest(_ nums: [Int], _ k: Int) -> Int {
    var minHeap = BinaryHeap<Int>.minHeap()
    for x in nums {
        minHeap.push(x)
        if minHeap.count > k { _ = minHeap.pop() }  // 超出 k 个就弹掉最小
    }
    return minHeap.peek()!   // 堆顶即第 k 大
}
```

## 11. 经典应用二：数据流的中位数（双堆对顶）

**问题（LeetCode 295）**：设计数据结构，支持 `addNum(x)` 持续加入数字，`findMedian()` 随时返回当前所有数字的中位数。

### 11.1 双堆对顶思想

用两个堆把数据「劈成两半」：

- **大顶堆 `lo`**：存较小的一半，堆顶是这一半的最大值。
- **小顶堆 `hi`**：存较大的一半，堆顶是这一半的最小值。

维护不变式：
1. `lo` 的所有元素 ≤ `hi` 的所有元素（即 `lo.top ≤ hi.top`）。
2. 两堆大小差不超过 1，且约定 `lo.count == hi.count` 或 `lo.count == hi.count + 1`。

则中位数：若总数为奇数 → `lo.top`；若为偶数 → `(lo.top + hi.top) / 2`。

### 11.2 Swift 实现

```swift
struct MedianFinder {
    private var lo = BinaryHeap<Int>.maxHeap()   // 较小一半，大顶
    private var hi = BinaryHeap<Int>.minHeap()   // 较大一半，小顶

    mutating func addNum(_ num: Int) {
        // 先放入 lo，再把 lo 的最大值挪到 hi，保证 lo 全部 ≤ hi
        lo.push(num)
        hi.push(lo.pop()!)
        // 再平衡大小：lo 至多比 hi 多 1 个
        if hi.count > lo.count {
            lo.push(hi.pop()!)
        }
    }

    func findMedian() -> Double {
        if lo.count > hi.count {
            return Double(lo.peek()!)
        } else {
            return (Double(lo.peek()!) + Double(hi.peek()!)) / 2.0
        }
    }
}

var mf = MedianFinder()
mf.addNum(1); mf.addNum(2)
print(mf.findMedian())   // 1.5
mf.addNum(3)
print(mf.findMedian())   // 2.0
```

复杂度：`addNum` 为 O(log n)（几次堆操作），`findMedian` 为 O(1)。这是「实时中位数」的经典最优解。

### 11.3 OC 版本核心逻辑（复用 MTBinaryHeap）

```objectivec
@interface MTMedianFinder : NSObject
- (void)addNum:(NSInteger)num;
- (double)findMedian;
@end

@implementation MTMedianFinder {
    MTBinaryHeap<NSNumber *> *_lo;  // 大顶
    MTBinaryHeap<NSNumber *> *_hi;  // 小顶
}
- (instancetype)init {
    if (self = [super init]) {
        _lo = [[MTBinaryHeap alloc] initWithComparator:^NSComparisonResult(NSNumber *a, NSNumber *b) {
            return [b compare:a];   // 降序 → 大顶堆
        }];
        _hi = [[MTBinaryHeap alloc] initWithComparator:^NSComparisonResult(NSNumber *a, NSNumber *b) {
            return [a compare:b];   // 升序 → 小顶堆
        }];
    }
    return self;
}
- (void)addNum:(NSInteger)num {
    [_lo push:@(num)];
    [_hi push:[_lo pop]];
    if (_hi.count > _lo.count) { [_lo push:[_hi pop]]; }
}
- (double)findMedian {
    if (_lo.count > _hi.count) return [[_lo peek] doubleValue];
    return ([[_lo peek] doubleValue] + [[_hi peek] doubleValue]) / 2.0;
}
@end
```

## 12. 经典应用三：合并 K 个有序链表

**问题（LeetCode 23）**：合并 K 个升序链表为一个升序链表。

### 12.1 最小堆解法

把 K 个链表的头节点放入小顶堆，每次弹出最小的接到结果链表后面，再把它的 `next` 入堆。直到堆空。

```swift
public class ListNode {
    public var val: Int
    public var next: ListNode?
    public init(_ val: Int) { self.val = val; self.next = nil }
}

func mergeKLists(_ lists: [ListNode?]) -> ListNode? {
    // 小顶堆，按节点 val 升序
    var heap = BinaryHeap<ListNode> { $0.val < $1.val }
    for node in lists {
        if let n = node { heap.push(n) }
    }
    let dummy = ListNode(0)
    var tail = dummy
    while let smallest = heap.pop() {
        tail.next = smallest
        tail = smallest
        if let nxt = smallest.next { heap.push(nxt) }
    }
    tail.next = nil
    return dummy.next
}
```

复杂度：设总节点数为 N，堆大小至多 K，每个节点入堆出堆各一次，**O(N log K)**，空间 O(K)。优于「两两合并」的 O(N·K) 朴素法，与「分治两两合并」O(N log K) 同阶但实现更直观。

> 注意：上面让 `BinaryHeap` 存储 `ListNode`（引用类型 class），比较器只比 `val`，工作正常。若用值类型需注意拷贝语义。

## 13. 经典应用四：Dijkstra 与堆优化

**问题**：带非负权边的图，求单源最短路。

### 13.1 朴素 Dijkstra vs 堆优化

朴素 Dijkstra 每轮线性扫描找「当前距离最小的未确定节点」，O(V²)。用**最小堆**维护候选 (距离, 节点)，每次 O(log V) 取最小，总复杂度降到 **O((V + E) log V)**，在稀疏图上优势明显。

### 13.2 Swift 实现（邻接表 + 最小堆）

```swift
/// 返回从 source 到各点的最短距离数组，不可达为 Int.max。
/// graph[u] = [(v, w)] 表示边 u→v 权重 w（w ≥ 0）。
func dijkstra(_ graph: [[(to: Int, weight: Int)]], source: Int) -> [Int] {
    let n = graph.count
    var dist = [Int](repeating: Int.max, count: n)
    dist[source] = 0
    // 最小堆，按 dist 升序
    var pq = BinaryHeap<(node: Int, d: Int)> { $0.d < $1.d }
    pq.push((source, 0))

    while let cur = pq.pop() {
        let (u, d) = cur
        if d > dist[u] { continue }     // 过期条目，跳过（惰性删除）
        for edge in graph[u] {
            let nd = d + edge.weight
            if nd < dist[edge.to] {
                dist[edge.to] = nd
                pq.push((edge.to, nd))  // 不删旧条目，靠上面的过期判断
            }
        }
    }
    return dist
}
```

> 工程要点：标准二叉堆不支持高效「decrease-key」，所以这里采用**惰性删除**——直接压入新的更短距离，弹出时若发现 `d > dist[u]` 就跳过过期条目。这是最常用的工程写法，简单且正确，代价是堆里可能有冗余条目（至多 E 个），不影响渐进复杂度。若追求理论最优（去掉冗余），需用**索引堆**支持 decrease-key（见下节）。

### 10.4 快速选择（Quickselect）对照实现

Top-K / 第 K 大除了堆，还有基于快排分区思想的快速选择，平均 O(n)，适合「数据全在内存、可原地打乱、只问一次」的场景。

```swift
/// 返回第 k 大元素（k 从 1 计）。会原地修改 nums。
func quickSelectKthLargest(_ nums: inout [Int], _ k: Int) -> Int {
    // 第 k 大 == 升序下标 n-k 的元素
    let target = nums.count - k
    var lo = 0, hi = nums.count - 1
    while lo <= hi {
        let p = partition(&nums, lo, hi)
        if p == target { return nums[p] }
        else if p < target { lo = p + 1 }
        else { hi = p - 1 }
    }
    return -1 // 不会到达
}

/// Lomuto 分区，随机选 pivot 防止最坏 O(n^2)
private func partition(_ a: inout [Int], _ lo: Int, _ hi: Int) -> Int {
    let r = Int.random(in: lo...hi)
    a.swapAt(r, hi)
    let pivot = a[hi]
    var i = lo
    for j in lo..<hi {
        if a[j] < pivot { a.swapAt(i, j); i += 1 }
    }
    a.swapAt(i, hi)
    return i
}
```

复杂度：平均 O(n)（每轮分区规模减半的期望），最坏 O(n^2)（极端分区，靠随机 pivot 几乎可避免）。空间 O(1)（迭代版）。

堆法 vs 快速选择对照：

| 维度 | 大小 K 的堆 | 快速选择 |
|---|---|---|
| 平均时间 | O(n log K) | O(n) |
| 最坏时间 | O(n log K) | O(n^2)（随机化后期望 O(n)） |
| 空间 | O(K) | O(1) |
| 是否需全数据在内存 | 否（可流式） | 是（需原地分区） |
| 是否破坏原数组 | 否 | 是 |
| 多次查询不同 K | 每次重扫 | 一次分区难复用 |

经验法则：数据流/海量数据/K 很小 → 堆；一次性、内存可放下、能改原数组 → 快速选择。

## 10.5 堆 vs 计数排序/桶在 Top-K 中的取舍

当元素值域很小（如频次、分数为有限整数）时，Top-K 还可以用「桶 + 计数」做到 O(n)：按值/频次分桶，从高到低收集 K 个。LC347「前 K 高频」既能用堆 O(n log K)，也能用桶排序 O(n)。

取舍：值域小且离散 → 桶法 O(n) 更快；值域大或连续、或需要严格 K 个有序输出 → 堆更通用。这体现了「没有银弹，按数据特征选结构」的工程思想。

## 14. 进阶堆结构

二叉堆是最常用的堆，但在不同需求下有更专门的变体。

### 14.1 d 叉堆（d-ary Heap）

把「每个节点最多 2 个孩子」推广为「最多 d 个孩子」。下标公式（从 0 开始）：

- 第 j 个孩子：`d * i + j + 1`（j = 0..d-1）
- 父节点：`(i - 1) / d`

权衡：d 越大，树越矮（高度约 log_d n），siftUp（插入/decrease-key）越快；但 siftDown 每层要比较 d 个孩子，下沉变慢。对于 Dijkstra 这类 decrease-key 远多于 extract-min 的场景，4 叉堆常比 2 叉堆更快（缓存也更友好）。

```swift
// 4 叉堆下沉核心：在 4 个孩子里找最值
private mutating func siftDown4(from i: Int) {
    var parent = i; let n = storage.count
    while true {
        var best = parent
        for j in 1...4 {
            let child = 4 * parent + j
            if child < n && areInIncreasingOrder(storage[child], storage[best]) { best = child }
        }
        if best == parent { return }
        storage.swapAt(parent, best); parent = best
    }
}
```

### 14.2 索引堆（Indexed Heap）

标准堆的痛点：无法高效「修改/删除堆中某个特定元素」（需 O(n) 先找到它）。索引堆额外维护两个映射：

- `pos[id]`：元素 id 当前在堆数组中的下标。
- `heap[k]`：堆位置 k 上存的是哪个 id。

这样就能 O(1) 定位任意元素，进而 O(log n) 实现 decreaseKey(id, newValue) / changePriority。Dijkstra、Prim 的理论最优实现都依赖它。

```swift
/// 索引最小堆：键为 0..<capacity 的整数 id，值为可比较的优先级。
struct IndexedMinHeap<Priority: Comparable> {
    private var heap: [Int] = []          // heap[k] = id
    private var pos: [Int]                // pos[id] = 在 heap 中的下标，-1 表示不在堆
    private var priority: [Priority?]     // priority[id] = 优先级

    init(capacity: Int) {
        pos = [Int](repeating: -1, count: capacity)
        priority = [Priority?](repeating: nil, count: capacity)
    }

    var isEmpty: Bool { heap.isEmpty }
    func contains(_ id: Int) -> Bool { pos[id] != -1 }

    mutating func insert(_ id: Int, _ p: Priority) {
        priority[id] = p
        heap.append(id)
        pos[id] = heap.count - 1
        siftUp(heap.count - 1)
    }

    mutating func decreaseKey(_ id: Int, _ p: Priority) {
        guard let old = priority[id], p < old else { return }
        priority[id] = p
        siftUp(pos[id])
    }

    mutating func extractMin() -> Int? {
        guard !heap.isEmpty else { return nil }
        let minId = heap[0]
        swapNodes(0, heap.count - 1)
        heap.removeLast()
        pos[minId] = -1
        if !heap.isEmpty { siftDown(0) }
        return minId
    }

    private mutating func swapNodes(_ i: Int, _ j: Int) {
        heap.swapAt(i, j)
        pos[heap[i]] = i
        pos[heap[j]] = j
    }
    private func less(_ i: Int, _ j: Int) -> Bool {
        priority[heap[i]]! < priority[heap[j]]!
    }
    private mutating func siftUp(_ k: Int) {
        var k = k
        while k > 0 && less(k, (k - 1) / 2) { swapNodes(k, (k - 1) / 2); k = (k - 1) / 2 }
    }
    private mutating func siftDown(_ k: Int) {
        var k = k; let n = heap.count
        while 2 * k + 1 < n {
            var child = 2 * k + 1
            if child + 1 < n && less(child + 1, child) { child += 1 }
            if !less(child, k) { break }
            swapNodes(k, child); k = child
        }
    }
}
```

### 14.3 二项堆、斐波那契堆、配对堆（理论与对比）

为了支持高效的合并（meld/union）与 decrease-key，发展出更高级的堆：

| 堆 | insert | extract-min | decrease-key | merge | 备注 |
|---|---|---|---|---|---|
| 二叉堆 | O(log n) | O(log n) | O(log n) | O(n) | 最常用，常数小 |
| 二项堆（Binomial Heap） | O(1) 摊还 | O(log n) | O(log n) | O(log n) | 由二项树森林组成，支持高效合并 |
| 斐波那契堆（Fibonacci Heap） | O(1) | O(log n) 摊还 | O(1) 摊还 | O(1) | 理论最优，Dijkstra 理论复杂度 O(E + V log V) 的来源；常数极大，实践少用 |
| 配对堆（Pairing Heap） | O(1) | O(log n) 摊还 | 摊还很快 | O(1) | 实现简单、常数小，实践中常优于斐波那契堆 |

> 这些高级堆主要出现在算法理论与竞赛中。工程实践 99% 用二叉堆（或四叉堆），因为它常数小、缓存友好、实现简单。斐波那契堆虽理论漂亮，但常数大到在实际数据规模下往往跑不过二叉堆。

## 15. iOS 工程实践：定时器、任务调度、事件队列

堆/优先队列在 iOS 与系统编程中无处不在，理解它有助于看懂底层。

### 15.1 定时器的最小堆（Timer Heap）

无论是 GCD 的 dispatch_source timer、NSTimer/CFRunLoopTimer，还是 Node.js / Go runtime 的定时器，底层都倾向用最小堆按「下次触发时间」组织：堆顶就是「最近要触发」的定时器，O(1) 拿到、O(log n) 增删。runloop/事件循环只需盯着堆顶，到点触发后 pop 并（若是周期定时器）重新计算下次时间再 push。

```swift
/// 简化版定时器调度器：用最小堆按触发时间排序。
struct TimerTask { let id: Int; let fireAt: TimeInterval; let action: () -> Void }

final class TimerScheduler {
    private var heap = BinaryHeap<TimerTask> { $0.fireAt < $1.fireAt }

    func schedule(_ task: TimerTask) { heap.push(task) }

    /// 由 runloop 周期性调用：触发所有已到期任务。
    func fireDue(now: TimeInterval) {
        while let top = heap.peek(), top.fireAt <= now {
            _ = heap.pop()
            top.action()
        }
    }

    /// 下一次需要唤醒的时间，nil 表示无任务。
    func nextFireTime() -> TimeInterval? { heap.peek()?.fireAt }
}
```

### 15.2 优先级任务队列（自定义并发调度）

在自研图片下载、预加载、埋点上报等模块中，常需要「高优先任务插队」。OperationQueue 通过 Operation.queuePriority 提供有限的优先级支持，但若要精细控制，可自建优先队列：可见任务先于预加载、用户主动触发先于后台任务等。

### 15.3 SAKFlexbox 等渲染框架中的应用

在动态布局/渲染引擎里，「脏节点重排」「动画帧调度」「资源加载优先级」都可用优先队列建模——例如把待渲染节点按「优先级 + 进入视口距离」排序，优先渲染用户即将看到的内容。结合本知识库 SAKFlexboxLibrary 的遍历优化思路，节点处理顺序的调度本质上是一次优先级出队过程。

### 15.4 操作系统层面

进程/线程调度器（如 CFS、优先级调度）、网络包的 QoS 队列、内存回收的优先级等，底层都有堆/优先队列的影子。理解堆，是理解这些系统机制的基础。

## 16. 复杂度全表与对比辨析

### 16.1 二叉堆操作复杂度全表

| 操作 | 时间复杂度 | 说明 |
|---|---|---|
| peek / top | O(1) | 直接返回 a[0] |
| push / insert | O(log n) | 末尾插入 + siftUp |
| pop / extract | O(log n) | 换尾到顶 + siftDown |
| replaceTop | O(log n) | 一次 siftDown，优于 pop+push |
| buildHeap（自底向上） | O(n) | Floyd 算法 |
| buildHeap（逐个插入） | O(n log n) | 次优 |
| 查找任意元素 | O(n) | 堆无全序，需线性扫描 |
| 删除任意元素 | O(n) 找 + O(log n) 调整 | 索引堆可优化到 O(log n) |
| heapSort | O(n log n) | 原地、不稳定 |

### 16.2 堆 vs 二叉搜索树（BST）对比辨析

| 维度 | 堆（Heap） | 二叉搜索树（BST） |
|---|---|---|
| 形状 | 完全二叉树（数组紧凑存储） | 一般二叉树（指针存储） |
| 有序性 | 仅父子偏序，无全序 | 中序遍历得全序 |
| 取最值 | O(1) peek | O(log n)（需走到最左/右） |
| 查找任意 key | O(n) | O(log n)（平衡时） |
| 范围查询 | 不支持 | 支持 |
| 插入/删最值 | O(log n) | O(log n) |
| 缓存友好度 | 高（连续数组） | 低（指针跳转） |
| 适用 | 只关心最值的场景 | 需要全序/查找/范围的场景 |

一句话总结：只要最值，用堆；要全序/查找/范围，用平衡 BST。

### 16.3 堆 vs 排序数组对比

排序数组取最值 O(1)、范围/查找 O(log n)，但插入删除 O(n)；堆插入删除 O(log n) 但查找 O(n)。动态变化频繁选堆，静态查询为主选排序数组。

```swift
func topKFrequent(_ nums: [Int], _ k: Int) -> [Int] {
    var freq: [Int: Int] = [:]
    for n in nums { freq[n, default: 0] += 1 }

    var res: [Int] = []
    while let t = heap.pop() { res.append(t.num) }
    return res
}
```

**例 5. LC373 查找和最小的 K 对数字** —— 两个升序数组，找 K 个和最小的对。用最小堆，初始放入 (i=0..min(k,m), j=0)，每次弹出最小对 (i,j) 后压入 (i, j+1)，避免重复。复杂度 O(K log K)。

```swift
func kSmallestPairs(_ nums1: [Int], _ nums2: [Int], _ k: Int) -> [[Int]] {
    guard !nums1.isEmpty, !nums2.isEmpty, k > 0 else { return [] }
    // 堆元素：(和, i, j)
    var heap = BinaryHeap<(sum: Int, i: Int, j: Int)> { $0.sum < $1.sum }
    for i in 0..<min(nums1.count, k) { heap.push((nums1[i] + nums2[0], i, 0)) }
    var res: [[Int]] = []
    while res.count < k, let cur = heap.pop() {
        res.append([nums1[cur.i], nums2[cur.j]])
        if cur.j + 1 < nums2.count {
            heap.push((nums1[cur.i] + nums2[cur.j + 1], cur.i, cur.j + 1))
        }
    }
    return res
}
```

**例 7. LC264 丑数 II** —— 最小堆 + 去重集合，每次弹出最小丑数，乘 2/3/5 后入堆，弹出第 n 个即答案。O(n log n)。

```swift
func nthUglyNumber(_ n: Int) -> Int {
    var heap = BinaryHeap<Int>.minHeap()
    var seen: Set<Int> = [1]
    heap.push(1)
    var ugly = 1
    for _ in 0..<n {
        ugly = heap.pop()!
        for f in [2, 3, 5] {
            let next = ugly * f
            if !seen.contains(next) { seen.insert(next); heap.push(next) }
        }
    }
    return ugly
}
```

```swift
func lastStoneWeight(_ stones: [Int]) -> Int {
    var heap = BinaryHeap(stones, >)   // 大顶堆，O(n) 建堆
    while heap.count > 1 {
        let a = heap.pop()!, b = heap.pop()!
        if a != b { heap.push(a - b) }
    }
    return heap.peek() ?? 0
}
```

## 18. 高频面试题（10 道，附答案要点）

**Q1. 堆和二叉搜索树有什么区别？**
A：堆是完全二叉树、只满足父子偏序（无全序）、用数组存、取最值 O(1)；BST 满足左<根<右的全序、中序遍历有序、支持任意查找与范围查询、取最值 O(log n)。堆查找任意元素 O(n)，BST O(log n)。只要最值用堆，要查找/范围用 BST。

**Q2. 为什么自底向上建堆是 O(n) 而不是 O(n log n)？**
A：大部分节点在底层，下沉高度很小。按高度求和，节点数乘高度的总和等于 n 乘以级数 k/2^(k+1) 之和，该级数收敛到常数，故总和 O(n)。逐个 siftUp 建堆才是 O(n log n)，因为大部分节点要上浮接近树高。

**Q3. push 用 siftUp、pop 用 siftDown，能反过来吗？**
A：不能。push 在末尾插入，只可能破坏与祖先的关系，故向上修复；pop 把末尾换到顶，只可能破坏与子孙的关系，故向下修复。方向由「破坏发生在哪个方向」决定。

**Q4. 求最大的 K 个数为什么用小顶堆而不是大顶堆？**
A：维护容量为 K 的小顶堆，堆顶是这 K 个里最小的，相当于「门槛」。新元素只要比门槛大就替换门槛，最终留下的就是最大的 K 个。用大顶堆无法在 O(log K) 内淘汰最小的。求最小 K 个则相反，用大顶堆。

**Q5. 海量数据（100 亿个）求 Top 100，怎么做？**
A：数据放不进内存，用容量 100 的小顶堆流式扫描，O(n log 100)、空间 O(100)。若分布式，各节点先各自求局部 Top 100，再归并各节点结果求全局 Top 100。

**Q6. 堆排序是稳定的吗？时间空间复杂度？**
A：不稳定（交换会打乱相等元素相对顺序）。时间最好/平均/最坏均 O(n log n)，空间 O(1) 原地。相比快排，堆排序有最坏 O(n log n) 保证但缓存不友好，实践常慢于快排。

**Q7. 如何实现一个支持「修改任意元素优先级」的优先队列？**
A：用索引堆，额外维护 pos[id] 到堆下标 的映射，定位元素 O(1)，再 siftUp/siftDown 调整 O(log n)。Dijkstra 的 decrease-key 即依赖此。工程中也常用「惰性删除」绕过：直接压新值，弹出时跳过过期条目。

**Q8. C++ priority_queue、Java PriorityQueue、Python heapq 默认是大顶还是小顶？**
A：C++ 默认大顶堆；Java 与 Python heapq 默认小顶堆。跨语言刷题极易搞反，务必牢记。

**Q9. 双堆求中位数，怎么保证两堆平衡且左半 ≤ 右半？**
A：新元素先入大顶堆 lo，立刻把 lo 堆顶（当前左半最大）弹出压入小顶堆 hi，保证 lo 全部小于等于 hi；再若 hi 比 lo 多则把 hi 顶回补给 lo。约定 lo.count 等于 hi.count 或 hi.count+1。奇数取 lo 顶，偶数取两顶平均。

**Q10. replaceTop（pop 后立刻 push）为什么比分开做快？**
A：pop 是「换尾到顶 + siftDown」，push 是「末尾插入 + siftUp」，两次各一趟。replaceTop 直接把新值放堆顶只做一次 siftDown，省去一次 siftUp 和数组增删，常数更优，常用于 Top-K 的「替换门槛」。

## 19. 易错点与最佳实践

**易错点**

1. 下标约定混用：0-based 与 1-based 的父子公式不同，混用必然 off-by-one。全程统一一种，并在注释写明。
2. 下沉时没选「较大孩子」：最大堆下沉必须和两个孩子中较大者交换，只比左孩子会破坏堆序。
3. pop 时忘了边界：堆只剩 1 个元素时，换尾到顶后数组已空，不应再 siftDown；空堆 pop 要返回 nil 而非崩溃。
4. Top-K 堆方向搞反：最大 K 个用小顶堆，最小 K 个用大顶堆，记反就全错。
5. 堆排序堆类型搞反：升序结果用最大堆，降序用最小堆。
6. 以为堆是有序的：堆不能用于「有序遍历」或「二分查找」，它只保证堆顶是最值。
7. 跨语言默认堆类型：C++ 大顶、Java/Python 小顶，刷题踩坑高发区。
8. 可变优先级用普通堆：普通堆改某元素优先级需 O(n) 查找；高频修改应用索引堆或惰性删除。

**最佳实践**

1. 工程首选数组实现的二叉堆：缓存友好、常数小；Swift 工程可直接用 swift-collections 的 Heap。
2. 建堆用自底向上：已有数据批量建堆走 Floyd O(n)，别逐个 push。
3. 比较器外置：把优先级策略交给调用方，堆只管机制。
4. 需要同优先级 FIFO：在元素里加单调递增序号作次级比较键，模拟稳定性。
5. Dijkstra 用惰性删除：压新条目、弹出时跳过过期项，简单且渐进最优，避免实现复杂的 decrease-key。
6. Top-K 用 replaceTop：替换门槛比 pop+push 更高效。

## 附录 A · 堆排序完整 trace 演示

对数组 `[5, 1, 8, 3, 2]` 做升序堆排序（最大堆），逐步观察。

阶段一：自底向上建最大堆。n=5，最后一个非叶子下标 = 5/2 - 1 = 1。

siftDown(1)：a[1]=1，孩子 a[3]=3、a[4]=2，较大 3。1<3 交换 → `[5,3,8,1,2]`。
siftDown(0)：a[0]=5，孩子 a[1]=3、a[2]=8，较大 8。5<8 交换 → `[8,3,5,1,2]`；继续在下标2，无孩子，停。

建堆完成：`[8, 3, 5, 1, 2]`，堆顶 8 是最大值。

阶段二：反复「换尾 + 缩堆 + 下沉」。

- end=4：swap(0,4) → `[2,3,5,1,8]`，堆大小=4，siftDown(0)：2 与孩子3、5较大5交换→`[5,3,2,1,8]`，下标2孩子越界停。
- end=3：swap(0,3) → `[1,3,2,5,8]`，堆大小=3，siftDown(0)：1 与孩子3、2较大3交换→`[3,1,2,5,8]`，停。
- end=2：swap(0,2) → `[2,1,3,5,8]`，堆大小=2，siftDown(0)：2 与孩子1较大1，2>1不换，停。
- end=1：swap(0,1) → `[1,2,3,5,8]`，堆大小=1，结束。

最终升序：`[1, 2, 3, 5, 8]`。可见每次把当前最大值「沉淀」到未排序段的末尾，是「选择排序」的高效堆化版本。

## 附录 B · 逐个插入建堆为何是 O(n log n)

第 6 节证明了自底向上建堆是 O(n)。反过来，逐个 siftUp 插入建堆的最坏代价为：第 i 个插入的元素位于深度约 log i，可能一路上浮到根，代价 O(log i)。总和：

总代价 = ∑(i=1..n) log i = log(n!) ，由斯特林公式 log(n!) ≈ n log n - n + O(log n) = Θ(n log n)。

直观差异在于：自底向上时「下沉代价大的节点（靠近根）数量极少」，而逐个插入时「上浮代价大的节点（后插入、在底层）数量极多」，二者的求和收敛性恰好相反，这就是 O(n) 与 O(n log n) 的本质区别。

## 附录 C · 一句话速记卡

- 堆 = 完全二叉树 + 父子偏序，数组存储，下标 0-based：父 (i-1)/2，左 2i+1，右 2i+2。
- push 末尾 + siftUp；pop 换尾到顶 + siftDown；peek = a[0]，O(1)。
- 自底向上建堆 O(n)，逐个插入 O(n log n)。
- 堆排序原地、不稳定、O(n log n)；升序用大顶堆。
- 最大 K 个 → 小顶堆（容量 K）；最小 K 个 → 大顶堆。
- 流式中位数 → 大顶堆装小一半 + 小顶堆装大一半，对顶平衡。
- C++ 默认大顶，Java/Python 默认小顶；Swift 标准库无内置，用 swift-collections 的 Heap。
- 要 decrease-key → 索引堆或惰性删除（Dijkstra）。
