# iOS 内存管理详解

---

## 一、内存管理核心原理

### 1.1 引用计数（Reference Counting）

iOS 内存管理的本质是**引用计数**。每个对象内部维护一个 `retainCount`，表示有多少个指针指向它。

```
对象创建  →  retainCount = 1
retain    →  retainCount += 1
release   →  retainCount -= 1
retainCount == 0  →  对象销毁，调用 dealloc
```

```
    ┌──────────────┐
    │   NSObject    │
    │ retainCount=3 │  ← 指针A持有
    └──────────────┘  ← 指针B持有
         ↑  ↑  ↑      ← 指针C持有
         A  B  C

    指针B release → retainCount=2，对象仍存活
    指针A release → retainCount=1，对象仍存活
    指针C release → retainCount=0，对象销毁
```

> **关键认知**：谁创建（retain/copy/new/mutableCopy），谁释放（release/autorelease）。持有即 +1，不再需要即 -1。

### 1.2 内存管理的两条规则

| 规则 | 说明 |
|------|------|
| 谁持有谁释放 | 通过 alloc/new/copy/mutableCopy/retain 持有的对象，必须对应 release 或 autorelease |
| 非持有不释放 | 通过其他方式（如数组取值、属性访问）获得的指针，不应该 release |

```objc
// 规则1：持有→释放
NSObject *obj = [[NSObject alloc] init];  // 持有，retainCount=1
[obj release];                            // 释放，retainCount=0，销毁

// 规则2：非持有→不释放
NSString *str = [NSArray array][0];  // 非持有，不应 release
```

## 二、MRC（Manual Reference Counting）

### 2.1 基本操作

```objc
// alloc / init — 创建并持有，retainCount=1
NSObject *obj = [[NSObject alloc] init];

// retain — 持有，retainCount+1
[obj retain];

// release — 释放，retainCount-1
[obj release];

// autorelease — 延迟释放（当前 autoreleasepool drain 时 release）
NSObject *obj = [[[NSObject alloc] init] autorelease];
// 等价于：
// NSObject *obj = [[NSObject alloc] init];
// [obj autorelease];
```

### 2.2 autorelease 与自动释放池

```objc
// autorelease 不立即释放，而是注册到最近的 autoreleasepool
// 当 pool drain 时统一 release

// NSAutoreleasePool（旧式）
NSAutoreleasePool *pool = [[NSAutoreleasePool alloc] init];
// ... 大量临时对象 ...
[pool drain];  // 此刻所有注册的对象统一 release

// @autoreleasepool（推荐）
@autoreleasepool {
    // 作用域内创建的 autorelease 对象，离开作用域时释放
    NSString *str = [NSString stringWithFormat:@"Hello %d", i];
}  // str 在此处 release
```

#### autorelease 的本质

```objc
// autorelease 做了什么？
- (id)autorelease {
    // 1. 获取当前线程的 RunLoop 对应的 autoreleasepool
    // 2. 将 self 添加到 pool 中
    // 3. 返回 self
    return self;
}

// autoreleasepool 何时 drain？
// 主线程：每次 RunLoop 迭代结束（即将进入休眠前）
// 子线程：需要手动创建 @autoreleasepool
```

### 2.3 MRC 下的属性修饰符

```objc
// assign — 简单赋值，不改变引用计数（基本类型、非所有权指针）
@property (nonatomic, assign) NSInteger age;
@property (nonatomic, assign) id delegate;  // delegate 用 assign（MRC）

// retain — 持有新值，释放旧值
@property (nonatomic, retain) NSString *title;
// 编译器生成的 setter：
- (void)setTitle:(NSString *)title {
    if (_title != title) {
        [_title release];      // 释放旧值
        _title = [title retain]; // 持有新值
    }
}

// copy — 拷贝新值，释放旧值
@property (nonatomic, copy) NSString *name;
// 编译器生成的 setter：
- (void)setName:(NSString *)name {
    if (_name != name) {
        [_name release];
        _name = [name copy];  // 拷贝新值（NSString 必须用 copy）
    }
}
```

### 2.4 MRC 实战模板

```objc
@implementation MyClass {
    NSObject *_obj;
}

// alloc/init → 必须在 dealloc 中 release
- (instancetype)init {
    self = [super init];
    if (self) {
        _obj = [[NSObject alloc] init];  // 持有
    }
    return self;
}

- (void)dealloc {
    [_obj release];  // 释放
    [super dealloc];
}

// 工厂方法 → autorelease
+ (instancetype)myClass {
    MyClass *obj = [[self alloc] init];
    return [obj autorelease];  // 调用者不持有，pool drain 时释放
}

// 集合持有 → 集合负责 release
- (void)addToArray {
    NSObject *item = [[NSObject alloc] init];  // 持有
    [self.array addObject:item];                // array retain
    [item release];                             // 释放自己的持有
}
```

### 2.5 MRC 常见错误

```objc
// ❌ 过度释放
NSObject *obj = [[NSObject alloc] init];
[obj release];
[obj release];  // 崩溃！EXC_BAD_ACCESS

// ❌ 野指针（悬垂指针）
NSObject *obj = [[NSObject alloc] init];
[obj release];
// obj 仍指向已释放的内存
NSLog(@"%@", obj);  // 可能崩溃，也可能打印出垃圾值
obj = nil;  // ✅ release 后立即置 nil

// ❌ 忘记 release（内存泄漏）
NSObject *obj = [[NSObject alloc] init];
// 没有对应的 release

// ❌ autorelease 后又 release
NSObject *obj = [[[NSObject alloc] init] autorelease];
[obj release];  // pool drain 时会再次 release → 崩溃
```

## 三、ARC（Automatic Reference Counting）

### 3.1 ARC 的本质

ARC **不是垃圾回收**，而是**编译器自动插入 retain/release/autorelease**。编译器在编译期分析代码，在合适的位置插入内存管理调用。

```objc
// MRC 下你需要写的：
- (void)setName:(NSString *)name {
    if (_name != name) {
        [_name release];
        _name = [name retain];
    }
}
- (void)dealloc {
    [_name release];
    [super dealloc];
}

// ARC 下编译器自动生成等价代码，你只需写：
@property (nonatomic, strong) NSString *name;
// dealloc 不需要写（ARC 自动处理），也不允许调用 [super dealloc]
```

### 3.2 ARC 的四个所有权修饰符

```objc
// __strong — 强引用（默认），持有对象
{
    NSObject *obj = [[NSObject alloc] init];  // __strong 是默认的
    // obj 持有对象，retainCount=1
}  // 离开作用域，obj 自动 release，retainCount=0，对象销毁

// __weak — 弱引用，不持有，对象销毁时自动置 nil
{
    NSObject *obj = [[NSObject alloc] init];
    NSObject *__weak weakObj = obj;  // 不持有，retainCount 仍为 1
    // weakObj 可以正常使用
}  // obj 离开作用域释放，weakObj 自动变 nil

// __unsafe_unretained — 弱引用但不自动置 nil（野指针风险）
{
    NSObject *obj = [[NSObject alloc] init];
    NSObject *__unsafe_unretained unsafeObj = obj;
}  // obj 释放后 unsafeObj 变成悬垂指针！访问会崩溃
// 使用场景：性能敏感 + 确定对象生命周期 > 引用生命周期

// __autoreleasing — 用于传给函数的 out/inout 参数
NSError *error = nil;
BOOL success = [data writeToFile:path options:0 error:&error];
// error 实际类型是 NSError *__autoreleasing
// 编译器自动在函数内做 retain/autorelease
```

### 3.3 __autoreleasing 详解

```objc
// 函数参数的隐式转换规则
// NSError *error = nil;
// 等价于 NSError *__strong error = nil;
// &error 传参时，编译器自动插入临时变量：

NSError *__strong error = nil;
NSError *__autoreleasing temp = error;  // 编译器插入
BOOL success = [data writeToFile:path options:0 error:&temp];
error = temp;  // 编译器插入

// 所以以下写法是错误的：
NSError *error;  // ❌ 未初始化，__strong 指针必须初始化
[data writeToFile:path error:&error];

// 正确：
NSError *error = nil;  // ✅
[data writeToFile:path error:&error];
```

### 3.4 ARC 下的属性修饰符

```objc
// strong — 强引用（对象类型默认）
@property (nonatomic, strong) UIView *contentView;
// setter 语义：释放旧值 → 持有新值

// weak — 弱引用，对象销毁自动置 nil
@property (nonatomic, weak) id<MyDelegate> delegate;
@property (nonatomic, weak) IBOutlet UILabel *titleLabel;

// copy — 拷贝新值，持有副本
@property (nonatomic, copy) NSString *title;
@property (nonatomic, copy) NSArray<NSString *> *items;
@property (nonatomic, copy) void(^completion)(void);  // Block 必须用 copy

// assign — 简单赋值（非对象类型）
@property (nonatomic, assign) NSInteger count;
@property (nonatomic, assign) CGPoint origin;

// unsafe_unretained — 弱引用不置 nil（极少使用）
@property (nonatomic, unsafe_unretained) id unsafeRef;
```

### 3.5 为什么 NSString / Block 必须用 copy？

```objc
// NSString 用 copy 的原因：防止 NSMutableString 被外部修改
@property (nonatomic, strong) NSString *name;    // ❌ 危险
@property (nonatomic, copy) NSString *name;      // ✅ 安全

NSMutableString *mutableStr = [NSMutableString stringWithString:@"Hello"];
person.name = mutableStr;       // strong：直接指向 mutableStr
[mutableStr appendString:@"!"]; // person.name 也变成了 "Hello!"（非预期）
// copy：创建不可变副本，不受外部修改影响

// Block 用 copy 的原因：将栈 Block 拷贝到堆
@property (nonatomic, copy) void(^completion)(void);  // ✅
@property (nonatomic, strong) void(^completion)(void); // ⚠️ ARC 下等价，但 copy 更语义化
```

### 3.6 ARC 禁止的操作

```objc
// ❌ 不能调用 retain / release / autorelease
[obj release];         // 编译错误
[obj retain];          // 编译错误

// ❌ 不能调用 [super dealloc]
- (void)dealloc {
    [super dealloc];   // 编译错误
}

// ❌ 不能使用 NSAutoreleasePool
NSAutoreleasePool *pool = [[NSAutoreleasePool alloc] init]; // 编译错误

// ✅ 使用 @autoreleasepool
@autoreleasepool { }

// ❌ 不能给对象发 retainCount 消息（虽然编译通过但值不准）
NSLog(@"%lu", (unsigned long)[obj retainCount]);  // 值不可靠，勿用
```

## 四、自动释放池（Autorelease Pool）深入

### 4.1 主线程 AutoreleasePool 的创建

```objc
// 主线程 RunLoop 的 AutoreleasePool 创建时机：
// 1. App 启动时，RunLoop 注册了两个 Observer
// 2. Observer1（_objc_autoreleasePoolPush）：RunLoop 即将进入时 push
// 3. Observer2（_objc_autoreleasePoolPop）：RunLoop 即将休眠时 pop+push，RunLoop 即将退出时 pop

// 这意味着主线程 autorelease 的对象，会在当前 RunLoop 迭代结束时释放
```

```
RunLoop 迭代：

  Push（Observer1，进入时）
    │
    ├── 处理 Timer/Source0
    ├── 处理 Source1
    ├── UI 事件处理
    │     └── autorelease 对象注册到当前 pool
    │
  Pop（Observer2，休眠前）── 所有 autorelease 对象释放
  Push（Observer2，休眠前）── 新建 pool
    │
    ├── 等待事件...
    │
  Pop（RunLoop 退出时）── 释放
```

### 4.2 嵌套 AutoreleasePool

```objc
@autoreleasepool {          // push
    NSObject *obj1 = [NSObject new];  // 不在 pool 中
    NSObject *obj2 = [[[NSObject alloc] init] autorelease];  // 在外层 pool

    @autoreleasepool {      // push（嵌套）
        NSObject *obj3 = [[[NSObject alloc] init] autorelease];  // 在内层 pool
    }  // pop（内层）→ obj3 释放

}  // pop（外层）→ obj2 释放（obj1 需要手动 release）
```

### 4.3 大量临时对象的内存峰值问题

```objc
// ❌ 内存峰值过高
for (int i = 0; i < 100000; i++) {
    NSString *str = [NSString stringWithFormat:@"item_%d", i];  // autorelease
    [self processData:str];
    // str 直到当前 RunLoop 迭代结束才释放
    // 100000 个字符串同时存在内存中！
}

// ✅ 在循环内加 autoreleasepool
for (int i = 0; i < 100000; i++) {
    @autoreleasepool {
        NSString *str = [NSString stringWithFormat:@"item_%d", i];
        [self processData:str];
    }  // str 在每次迭代结束时释放
}
```

### 4.4 子线程与 AutoreleasePool

```objc
// 子线程默认没有 autoreleasepool
// 如果子线程中使用了 autorelease 对象，必须手动创建

dispatch_async(dispatch_get_global_queue(0, 0), ^{
    // ⚠️ 如果不创建 pool，autorelease 对象会泄漏
    @autoreleasepool {
        NSString *str = [NSString stringWithFormat:@"Hello"];
        // ...
    }
});

// GCD 的 dispatch 自动包裹 autoreleasepool（某些实现）
// 但最佳实践：子线程中始终手动创建 @autoreleasepool
```

## 五、循环引用（Retain Cycle）

### 5.1 循环引用的本质

```
当 A 强引用 B，B 又强引用 A 时：
    A ←strong→ B
两个对象的 retainCount 永远不为 0，都无法释放。

更复杂的环：
    A → B → C → A（三方环）
    A → B → A + A → C → A（多个环）
```

### 5.2 Block 循环引用

```objc
// ❌ 经典循环引用：self → property → Block → self
@interface MyViewController ()
@property (nonatomic, copy) void(^completion)(void);
@end

@implementation MyViewController
- (void)setup {
    self.completion = ^{
        [self doSomething];  // Block 捕获 self（强引用）
    };
    // self → completion(Block) → self  → 循环！
}
@end

// ✅ 解决方案1：__weak + __strong
- (void)setup {
    __weak typeof(self) weakSelf = self;
    self.completion = ^{
        __strong typeof(weakSelf) strongSelf = weakSelf;
        if (!strongSelf) return;
        [strongSelf doSomething];  // Block 执行期间保证 self 不被释放
    };
}

// ✅ 解决方案2：传参方式（Block 不捕获 self）
- (void)setup {
    self.completion = ^{
        [self doSomething];  // ❌ 还是捕获了 self
    };
}

// 改为：
- (void)performCompletionWithSelf:(MyViewController *)controller {
    [controller doSomething];
}
- (void)setup {
    self.completion = ^{
        // 不捕获 self
    };
}
```

### 5.3 为什么需要 weakSelf + strongSelf？

```objc
// 只用 weakSelf 的问题：
__weak typeof(self) weakSelf = self;
self.completion = ^{
    [weakSelf doSomething];   // weakSelf 可能在执行过程中变 nil
    [weakSelf doAnotherThing]; // 此时 weakSelf 可能已经 nil，第二行不执行
    // 业务逻辑被中途打断！
};

// strongSelf 保证 Block 执行期间 self 不释放：
__weak typeof(self) weakSelf = self;
self.completion = ^{
    __strong typeof(weakSelf) strongSelf = weakSelf;
    if (!strongSelf) return;  // 如果已释放，直接退出
    [strongSelf doSomething];    // 保证执行
    [strongSelf doAnotherThing]; // 保证执行
};
// strongSelf 是 __strong 局部变量，Block 执行期间持有 self
// Block 执行完，strongSelf 超出作用域，自动释放
```

### 5.4 Delegate 循环引用

```objc
// ❌ delegate 用 strong → 循环引用
@property (nonatomic, strong) id<TableViewDelegate> delegate;
// view.delegate = viewController;
// viewController → strong → view → strong → delegate → viewController → 循环

// ✅ delegate 必须用 weak
@property (nonatomic, weak) id<TableViewDelegate> delegate;
// viewController → strong → view → weak → delegate → viewController
// weak 不增加引用计数，打破循环
```

### 5.5 NSTimer 循环引用

```objc
// ❌ Timer 强引用 target，target 强引用 timer
@interface MyViewController ()
@property (nonatomic, strong) NSTimer *timer;
@end

- (void)startTimer {
    self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                                 target:self
                                               selector:@selector(tick)
                                               userInfo:nil
                                                repeats:YES];
    // self → timer → target(self) → 循环
    // 即使 self 退出，timer 仍然持有 self，两者都不会释放！
}

- (void)dealloc {
    [self.timer invalidate];  // 永远不会被调用！因为 self 不会释放
}

// ✅ 解决方案1：在合适的时机 invalidate
- (void)viewWillDisappear:(BOOL)animated {
    [super viewWillDisappear:animated];
    [self.timer invalidate];
    self.timer = nil;
}

// ✅ 解决方案2：使用弱引用的 Timer 包装类
// NSProxy + weakTarget 方案
@interface WeakTimerProxy : NSProxy
@property (nonatomic, weak) id target;
@end

@implementation WeakTimerProxy
- (instancetype)initWithTarget:(id)target {
    _target = target;
    return self;
}
- (id)forwardingTargetForSelector:(SEL)sel {
    if ([_target respondsToSelector:sel]) return _target;
    return nil;
}
- (BOOL)respondsToSelector:(SEL)sel {
    return [_target respondsToSelector:sel];
}
@end

// 使用
- (void)startTimer {
    WeakTimerProxy *proxy = [[WeakTimerProxy alloc] initWithTarget:self];
    self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                                 target:proxy
                                               selector:@selector(tick)
                                               userInfo:nil
                                                repeats:YES];
    // self → timer → target(proxy) → weak → self
    // self 释放后 proxy 的 _target 变 nil，消息发到 nil 安全
}

// ✅ 解决方案3：iOS 10+ block-based Timer
__weak typeof(self) weakSelf = self;
self.timer = [NSTimer scheduledTimerWithTimeInterval:1.0
                                            repeats:YES
                                              block:^(NSTimer *timer) {
    [weakSelf tick];
}];
// ⚠️ 仍有循环：self → timer → block → weakSelf(不增加引用)
// 但 self 释放后 weakSelf 变 nil，需要在 dealloc 中 invalidate

// ✅ 解决方案4：GCD Timer（推荐）
- (void)startGCDTimer {
    __weak typeof(self) weakSelf = self;
    self.timerSource = dispatch_source_create(DISPATCH_SOURCE_TYPE_TIMER, 0, 0,
                                              dispatch_get_main_queue());
    dispatch_source_set_timer(self.timerSource,
                              dispatch_time(DISPATCH_TIME_NOW, 0),
                              1.0 * NSEC_PER_SEC, 0);
    dispatch_source_set_event_handler(self.timerSource, ^{
        [weakSelf tick];
    });
    dispatch_resume(self.timerSource);
}
- (void)stopGCDTimer {
    dispatch_source_cancel(self.timerSource);
    self.timerSource = nil;
}
```

### 5.6 闭包属性与懒加载的循环引用

```objc
// ❌ 懒加载闭包捕获 self
@property (nonatomic, strong) UIView *headerView;
// 懒加载在 Objective-C 中不常见，但在 Swift 中常见

// Swift 版本：
class MyViewController: UIViewController {
    lazy var headerView: UIView = {
        let view = UIView()
        view.delegate = self  // ❌ 闭包捕获 self → 强引用
        return view
    }()
}

// ✅ 使用 [weak self] 不行（lazy 不支持）
// ✅ 改为在 viewDidLoad 中初始化
class MyViewController: UIViewController {
    private(set) var headerView: UIView!

    override func viewDidLoad() {
        super.viewDidLoad()
        headerView = UIView()
        headerView.delegate = self  // 外部设置，不在闭包中
    }
}
```

### 5.7 相互引用的对象

## 六、Copy 语义深入

### 6.1 深拷贝 vs 浅拷贝

```
深拷贝（Deep Copy）：创建新对象，内容相同，互不影响
浅拷贝（Shallow Copy）：只拷贝指针，指向同一对象
```

### 6.2 copy vs mutableCopy 规则表

| 源对象类型 | copy | mutableCopy |
|-----------|------|-------------|
| 不可变（NSString） | 浅拷贝（同对象） | 深拷贝（NSMutableString） |
| 可变（NSMutableString） | 深拷贝（NSString） | 深拷贝（NSMutableString） |

```objc
// 不可变对象的 copy → 浅拷贝（优化：拷贝不可变对象无意义）
NSString *str = @"hello";
NSString *copyStr = [str copy];
// copyStr 和 str 指向同一对象（浅拷贝）
str == copyStr  // YES

// 不可变对象的 mutableCopy → 深拷贝
NSMutableString *mutableCopy = [str mutableCopy];
// 新对象，内容相同
str == mutableCopy  // NO

// 可变对象的 copy → 深拷贝
NSMutableString *mStr = [NSMutableString stringWithString:@"hello"];
NSString *copyMStr = [mStr copy];
// 新的不可变对象
mStr == copyMStr  // NO

// 可变对象的 mutableCopy → 深拷贝
NSMutableString *mCopyMStr = [mStr mutableCopy];
mStr == mCopyMStr  // NO
```

### 6.3 集合类的拷贝

```objc
// ⚠️ 集合的 copy/mutableCopy 只拷贝容器本身，不拷贝内容（单层深拷贝）
NSArray *arr = @[[NSMutableString stringWithString:@"hello"]];
NSArray *copyArr = [arr copy];
// copyArr 和 arr 是不同数组，但元素指向同一对象

NSMutableArray *mArr = [arr mutableCopy];
[mArr[0] appendString:@" world"];

```

## 七、引用计数的底层存储

### 7.1 引用计数存放在哪里？

```
ARM64 下引用计数存储在两个位置：
1. isa 的 extra_rc 字段（8 位，存得下的部分）
2. SideTable 的 RefcountMap（isa 放不下时溢出到这里）

判断依据：isa 的 has_sidetable_rc 位
  - has_sidetable_rc = 0 → 引用计数只在 isa.extra_rc
  - has_sidetable_rc = 1 → 部分在 isa，溢出部分在 SideTable
```

### 7.2 isa 中的 extra_rc

```c
// ARM64 isa 位域（非 Tagged Pointer 对象）
// extra_rc 占 8 位，可存储 0~255
// 实际引用计数 = extra_rc + 1（对象创建时 extra_rc=0 表示 RC=1）

// 对象刚创建：extra_rc = 0 → 实际 RC = 1
// 第一次 retain：extra_rc = 1 → 实际 RC = 2
// ...
// extra_rc = 255 → 实际 RC = 256

// 第 257 次 retain 时，extra_rc 溢出
// → has_sidetable_rc = 1
// → 将 extra_rc 的一半（128）转移到 SideTable
// → extra_rc 重置为 128
// → 后续 retain 优先增加 SideTable 中的计数
```

### 7.3 SideTable 结构

```c
// 全局有若干个 SideTable（StripedMap<SideTable>，64 或 8 个）
// 通过对象地址哈希分配到不同的 SideTable（减少锁竞争）

struct SideTable {
    spinlock_t slock;          // 自旋锁（保护并发访问）
    RefcountMap refcnts;       // 引用计数哈希表
    weak_table_t weak_table;   // 弱引用表
};

// RefcountMap：unordered_map<DisguisedPtr<objc_object>, size_t>
// Key：对象地址（伪装，避免 hash 碰撞攻击）
// Value：size_t，低 2 位为标志位
//   bit 0 (SIDE_TABLE_RC_ONE) = 引用计数值的一部分
//   bit 1 (SIDE_TABLE_WEAKLY_REFERENCED) = 是否有 weak 引用
//   bit 2..63 = 引用计数值（实际值 = (value >> SIDE_TABLE_RC_SHIFT)）

// 对象的完整引用计数：
// RC = isa.extra_rc + 1 + SideTable.refcnts[obj] >> SIDE_TABLE_RC_SHIFT
```

```
引用计数存储示意：

对象 A（RC=3，无溢出）：
  isa.extra_rc = 2
  has_sidetable_rc = 0
  完整 RC = extra_rc + 1 = 3

对象 B（RC=300，溢出）：
  isa.extra_rc = 149
  has_sidetable_rc = 1
  SideTable.refcnts[B] = 150 << 2（低2位为标志）
  完整 RC = 149 + 1 + 150 = 300
```

### 7.4 retain / release 底层流程

```c
// retain 的底层实现
objc_object::retain() {
    // 1. Tagged Pointer → 直接返回，不操作
    if (isTaggedPointer()) return this;

    // 2. 尝试在 isa.extra_rc 中 +1
    if (isa.extra_rc < 255) {
        // 快速路径：直接在 isa 中增加
        isa.extra_rc++;
        return this;
    }

    // 3. extra_rc 溢出 → 转移到 SideTable
    SideTable &table = SideTable::tableForPointer(this);
    table.lock();
    // 将 extra_rc 的一半转移到 SideTable
    size_t carry = isa.extra_rc / 2;
    isa.extra_rc -= carry;
    isa.has_sidetable_rc = 1;
    table.refcnts[this] += carry << SIDE_TABLE_RC_SHIFT;
    table.unlock();
    return this;
}

// release 的底层实现
objc_object::release() {
    // 1. Tagged Pointer → 直接返回
    if (isTaggedPointer()) return;

    // 2. isa.extra_rc > 0 → 直接 -1
    if (isa.extra_rc > 0) {
        isa.extra_rc--;
        return;
    }

    // 3. extra_rc 已为 0，需要从 SideTable 借
    SideTable &table = SideTable::tableForPointer(this);
    table.lock();
    size_t &refcnt = table.refcnts[this];
    size_t sideRC = refcnt >> SIDE_TABLE_RC_SHIFT;

    if (sideRC > 0) {
        // SideTable 有余量，转一部分回 isa
        size_t borrow = min(sideRC, (size_t)255/2);
        refcnt -= borrow << SIDE_TABLE_RC_SHIFT;
        isa.extra_rc = borrow - 1;  // -1 因为本次 release
        if (refcnt == 0) {
            table.refcnts.erase(this);
            isa.has_sidetable_rc = 0;
        }
    } else {
        // SideTable 也为 0 → RC=0，对象销毁
        table.refcnts.erase(this);
        table.unlock();
        objc_object::dealloc();  // 触发销毁
        return;
    }
    table.unlock();
}
```

### 7.5 为什么不用 retainCount？

```objc
// retainCount 返回的值不准确，原因：
// 1. Tagged Pointer 返回固定值（如 NSNumber 返回 LONG_MAX）
// 2. 某些系统对象会重写 retainCount（如单例返回 UINTPTR_MAX）
// 3. autorelease 对象的引用计数可能已被 pool 持有但未反映
// 4. 无法获取 SideTable 中的完整计数（runtime 未暴露）
// 5. 多线程下读到的值可能已过时

// 结论：永远不要用 retainCount 调试内存问题
```

## 八、对象 dealloc 的完整流程

### 8.1 dealloc 调用链

```
[obj release] → RC=0
     │
     ▼
objc_object::rootRelease() → 判断需要 dealloc
     │
     ▼
objc_object::rootDealloc()
     │
     ├── isTaggedPointer()? → 直接返回（不需要释放）
     │
     ├── isa.has_cxx_dtor? → object_cxxDestruct()（调用 C++ 析构函数）
     │
     ├── isa.has_assoc? → _object_remove_assocations()（移除关联对象）
     │
     ├── isa.has_sidetable_rc? → table.refcnts.erase()（清理引用计数表）
     │
     ├── isa.weakly_referenced? → weak_clear_no_lock()（弱引用置 nil）
     │
     └── free(obj)（释放内存）
```

### 8.2 C++ 析构函数

```c
// 如果对象有 C++ 成员变量或 .cxx_destruct 方法
// object_cxxDestruct 会调用 .cxx_destruct
// ARC 编译器为每个类生成 .cxx_destruct，负责释放 __strong ivar

void object_cxxDestruct(id obj) {
    // 遍历类层级（从子类到父类）
    Class cls = obj->getIsa();
    while (cls) {
        // 查找 .cxx_destruct 方法
        void (*dtor)(id) = cls->getCxxDestruct();
        if (dtor) {
            dtor(obj);  // 调用析构
        }
        cls = cls->superclass;
    }
}

// .cxx_destruct 内部等价于：
// - 释放所有 __strong 实例变量
// - 即 ARC 下的"自动 dealloc"
```

### 8.3 完整 dealloc 时序图

```
1. RC 减为 0
   │
2. 调用 -dealloc（可重写，做清理工作如移除通知、invalidate timer）
   │   ⚠️ 不要在 dealloc 中调用其他对象的 方法（那些对象可能已释放）
   │   ⚠️ 不要在 dealloc 中 dispatch_async（self 已在释放）
   │
3. [super dealloc]（MRC 必须调用，ARC 自动调用）
   │
4. object_cxxDestruct（释放 __strong ivar）
   │
5. _object_remove_assocations（移除关联对象，触发 OBJC_ASSOCIATION_RETAIN/COPY 的 release）
   │
6. weak_clear_no_lock（遍历 weak_table，将所有 __weak 指针置 nil）
   │
7. 清理 SideTable 中的 refcnts 条目
   │
8. free()（归还内存给系统）
```

## 九、ARC 编译器插入代码分析

### 9.1 ARC 到底插入了什么？

```objc
// 源代码（你写的）
- (void)setName:(NSString *)name {
    _name = name;
}

// ARC 编译后的等价代码（编译器插入 retain/release）
- (void)setName:(NSString *)name {
    objc_retain(name);       // 持有新值
    objc_release(_name);     // 释放旧值
    _name = name;            // 赋值
}

// 优化：objc_storeStrong 一步完成
- (void)setName:(NSString *)name {
    objc_storeStrong(&_name, name);
}
```

### 9.2 objc_storeStrong 实现

```c
void objc_storeStrong(id *location, id obj) {
    id prev = *location;
    if (obj == prev) return;     // 新旧值相同，不操作
    objc_retain(obj);            // 持有新值
    *location = obj;             // 赋值
    objc_release(prev);          // 释放旧值
}
```

### 9.3 局部变量的 ARC 行为

```objc
// 源代码
- (void)process {
    NSString *str = [[NSString alloc] initWithFormat:@"Hello"];
    NSLog(@"%@", str);
}

// ARC 编译后等价代码
- (void)process {
    NSString *str = [[NSString alloc] initWithFormat:@"Hello"];
    NSLog(@"%@", str);
    objc_release(str);  // 编译器在作用域末尾插入 release
}
```

### 9.4 return 的 ARC 优化

```objc
// 源代码
- (NSString *)greeting {
    NSString *str = [[NSString alloc] initWithFormat:@"Hello"];
    return str;
}

// 无优化时：
- (NSString *)greeting {
    NSString *str = [[NSString alloc] initWithFormat:@"Hello"];
    return objc_autoreleaseReturnValue(str);
    // 调用者：objc_retainAutoreleasedReturnValue(returnValue)
    // 结果：str 进入 autoreleasepool，稍后 release
}

// ⚠️ ARC 优化（caller-callee 协作）：
- (NSString *)greeting {
    NSString *str = [[NSString alloc] initWithFormat:@"Hello"];
    return objc_autoreleaseReturnValue(str);
    // 检测调用者是否紧跟着 objc_retainAutoreleasedReturnValue
    // 如果是 → 不入 pool，直接返回（tls 保存标记）
}

// 调用者：
NSString *result = objc_retainAutoreleasedReturnValue([obj greeting]);
// 检测 tls 标记 → 如果有，直接持有，不入 pool
// 省去 autorelease + retain 的开销
```

### 9.5 ARC 调用优化总结

```
正常流程（无优化）：
  callee:  retain → autorelease → return
  caller:  retain
  总开销：2 次 retain + 1 次 autorelease + 1 次 release

优化后（caller-callee 协作）：
  callee:  检测到 caller 会 retain → 直接 return（存 TLS 标记）
  caller:  检测到 TLS 标记 → 直接使用
  总开销：0 次额外操作

适用场景：
  工厂方法返回对象（[[Foo alloc] init] 后 return）
  方法返回 __strong 对象
```

## 十、Block 底层实现

### 10.1 Block 的 C/C++ 结构

```c
// 一个简单的 Block
void (^block)(void) = ^{
    printf("Hello\n");
};

// 编译器生成的 C++ 结构（简化版）
struct Block_literal {
    void *isa;               // _NSConcreteStackBlock / _NSConcreteMallocBlock / _NSConcreteGlobalBlock
    int flags;               // 标志位
    int reserved;
    void (*invoke)(void *, ...);  // 函数指针，指向 Block 的执行体
    struct Block_descriptor *descriptor;  // 描述信息
    // 捕获的变量（按捕获顺序排列）
};

struct Block_descriptor {
    unsigned long reserved;
    unsigned long size;           // Block 大小
    void (*copy)(void *, void *);   // copy 辅助函数（ARC 用）
    void (*dispose)(void *, void *); // dispose 辅助函数
};
```

### 10.2 Block 捕获变量的底层

```objc
// 源代码
int x = 10;
void (^block)(void) = ^{
    printf("%d", x);
};

// 编译器生成
struct Block_literal_0 {
    void *isa;
    int flags;
    int reserved;
    void (*invoke)(struct Block_literal_0 *, ...);
    struct Block_descriptor *descriptor;
    int x;  // ← 捕获的 x 的副本（值捕获）
};

// invoke 函数
void block_invoke_0(struct Block_literal_0 *block) {
    printf("%d", block->x);  // 使用捕获的副本
}

// 创建 Block
struct Block_literal_0 block = {
    .isa = _NSConcreteStackBlock,
    .flags = 0,
    .invoke = block_invoke_0,
    .descriptor = &descriptor,
    .x = x  // 拷贝 x 的当前值
};
```

### 10.3 __block 变量的底层

```objc
// 源代码
__block int x = 10;
void (^block)(void) = ^{
    x = 20;
};

// 编译器将 __block 变量包装为结构体
struct Block_byref_x {
    void *isa;          // 堆上时有值
    struct Block_byref_x *forwarding;  // 指向自己（栈上）或堆副本
    int flags;
    int size;
    int x;              // 实际值
};

struct Block_literal_1 {
    void *isa;
    int flags;
    int reserved;
    void (*invoke)(struct Block_literal_1 *, ...);
    struct Block_descriptor *descriptor;
    struct Block_byref_x *x;  // ← 指向 byref 结构体（引用捕获）
};

// 关键：forwarding 指针
// 栈上时：forwarding 指向自己
// copy 到堆后：栈上的 forwarding 指向堆副本
// 所以 Block 内外都通过 forwarding 访问同一变量
```

```
__block 变量的 forwarding 机制：

Block 在栈上：
  StackBlock.x_byref ──forwarding──→ StackBlock.x_byref
                                        │ x = 10

Block copy 到堆后：
  StackBlock.x_byref ──forwarding──→ HeapBlock.x_byref
                                        │ x = 20（堆上）
  HeapBlock.x_byref  ──forwarding──→ HeapBlock.x_byref（自己）

  无论从栈 Block 还是堆 Block 修改，都通过 forwarding 找到堆上的值
```

### 10.4 Block 的 copy 时机

```c
// ARC 下 Block 自动 copy 的时机：
// 1. 赋值给 __strong 变量
//    void (^block)(void) = ^{ ... };  // 自动 copy 到堆

// 2. 作为函数返回值
//    - (void(^)())makeBlock { return ^{ ... }; }
//    // objc_retainBlock → _Block_copy

// 3. 赋值给 strong 属性
//    self.block = ^{ ... };  // setter 中 objc_storeStrong → copy

// 4. 作为方法参数（Cocoa API 含 usingBlock 的方法）
//    [UIView animateWithDuration:0.3 animations:^{ ... }];
//    // 方法内部 copy

// ⚠️ 不自动 copy 的情况：
// 1. 作为 C 函数参数
//    dispatch_async(queue, ^{ ... });
//    // GCD 内部自己 copy，但如果你传给自定义 C 函数则不会自动 copy

// 2. 类型转换
//    void (^block)(void) = [^{ ... } copy];  // 需要显式 copy
```

### 10.5 三种 Block 类型的判断


## 十一、内存分配底层（malloc）

### 11.1 对象的内存分配

```
[[NSObject alloc] init] 的底层：

1. +alloc → objc_alloc → calloc()
   - calloc 分配零初始化的内存
   - 大小 = 对象的 instanceSize（isa + 所有 ivar 大小，对齐到 8/16 字节）

2. init → 返回 self（NSObject 的 init 不做额外操作）

3. 对象在堆上分配，由 malloc 管理底层内存
```

### 11.2 malloc 的内存分级

```
iOS 使用 jemalloc / malloc 的分级分配策略：

┌──────────────────────────────────────┐
│  Tiny   (≤ 256 bytes)               │  ← 大多数 ObjC 对象
│  按 16 字节对齐，slot 分配           │
├──────────────────────────────────────┤
│  Small  (257 ~ 8KB)                  │
│  按 256 字节对齐                     │
├──────────────────────────────────────┤
│  Large  (> 8KB)                      │
│  直接 mmap，按页对齐（4KB/16KB）     │
└──────────────────────────────────────┘

Tiny/Small 分配：
  - 预分配一大块内存（region/magazine）
  - 按大小分成 slot（空闲链表）
  - 分配时从空闲链表取，释放时归还链表
  - 减少 syscall 开销

Large 分配：
  - 直接调用 mmap 分配虚拟内存
  - 释放时 munmap 归还
  - 带来的开销大，但避免了碎片
```

### 11.3 虚拟内存与内存页

```
iOS 虚拟内存：

  虚拟地址空间（每个进程独立）
  ┌─────────────────────┐
  │  Stack（高地址向下） │
  │  ↓                   │
  │                     │
  │  ↑                   │
  │  Heap（低地址向上）   │
  ├─────────────────────┤
  │  BSS（未初始化全局）  │
  │  Data（已初始化全局） │
  │  Text（代码段）       │
  └─────────────────────┘

  物理内存
  ┌─────────────────────┐
  │  Page 4KB/16KB       │  ← 最小管理单位
  │  Page               │
  │  Page               │
  └─────────────────────┘

  虚拟地址 → 页表 → 物理地址（MMU 硬件转换）

  iOS 特殊：
  - 没有 swap（不能换出到磁盘）
  - 内存压力时直接 Jetsam 杀进程
  - 内存警告 → 释放可回收内存 → 否则被杀
  - Compressed Memory：压缩不活跃页面而非换出
```

### 11.4 内存页与僵尸内存

```
内存页的状态：

1. 未分配（Unallocated）→ 虚拟地址未映射
2. 已分配未使用（Resident but Clean）→ 已映射但未修改（如 mmap 的文件）
3. 已分配已修改（Resident and Dirty）→ 已修改，必须常驻物理内存
4. 已压缩（Compressed）→ 被系统压缩存储

Dirty Memory 是优化的重点：
  - 必须占用物理内存
  - 无法被系统回收
  - 减少 Dirty Memory = 减少内存压力

减少 Dirty Memory 的技巧：
  - 使用纯代码段（Text Segment）： framework 中的代码是 clean 的
  - 避免大量全局/静态变量（Dirty）
  - 使用 mmap 映射文件而非 read 到内存（Clean）
  - 使用 NSCache 代替 NSDictionary（系统可自动回收）
```

## 十二、weak 的底层实现

### 12.1 weak 表

```
Runtime 维护一个全局的 weak 表（SideTable 中的 weak_table_t）：
- Key：被引用对象的地址
- Value：所有指向该对象的 weak 指针地址数组

当对象销毁时：
1. 通过对象地址在 weak 表中查找
2. 将所有 weak 指针置为 nil
3. 从 weak 表中移除该条目
```

```
SideTable（每个对象对应的辅助数据）
├── spinlock_t（自旋锁）
├── RefcountMap（引用计数表）
└── weak_table_t（弱引用表）
    ├── weak_entry_t
    │   ├── referent: 被引用对象地址
    │   └── referrers: [weak 指针地址]
    └── ...
```

### 12.2 weak 指针的注册过程

```objc
NSObject *__weak weakObj = obj;

// 编译器转换为：
id weakObj;
objc_initWeak(&weakObj, obj);
// 等价于：
weakObj = nil;
objc_storeWeak(&weakObj, obj);

// objc_storeWeak 做了什么：
// 1. 获取 obj 对应的 SideTable
// 2. 加锁
// 3. 在 weak_table 中查找或创建 weak_entry_t
// 4. 将 &weakObj 添加到 entry 的 referrers 数组
// 5. 解锁
```

### 12.3 对象销毁时 weak 置 nil 的过程

```objc
// 对象的 dealloc 过程：
// 1. 调用 objc_deallocInstance
// 2. 在 dealloc 内部：
//    a. 调用 objc_clear_deallocating
//       i.   从 SideTable 中获取 weak_table
//       ii.  查找对象对应的 weak_entry_t
//       iii. 遍历 referrers，将每个 weak 指针置 nil
//       iv.  从 weak_table 中移除该 entry
//    b. 释放对象的内存
```

### 12.4 weak 与 ObjC 对象的约束

```objc
// weak 只能修饰 ObjC 对象（有 isa 指针的对象）
// ❌ 以下不能使用 weak：
NSObject *__weak *ptr;     // ❌ 指向对象指针的指针不能用 weak
CFStringRef __weak cf;     // ❌ Core Foundation 对象不支持 weak
int __weak num;            // ❌ 基本类型不支持

// ✅ 可以使用 weak：
NSObject *__weak obj;
NSString *__weak str;
id __weak delegate;
```

### 13.5 捕获列表详解

```swift

    var value = 10

    func example() {
        var x = 5
        var y = 10

        // 默认：引用捕获（与 ObjC 的值捕获不同！）
        let closure = {
            print(x, y)  // 捕获 x, y 的引用
        }

        // 捕获列表：值捕获（在闭包创建时就拷贝）
        let closure2 = { [x] in
            print(x)  // 捕获 x 的副本
        }

        let closure3 = { [weak self, x] in
            print(x)              // 值副本
            print(self?.value)    // weak 引用
        }

        // 捕获列表中用特定名字
        let closure4 = { [weak weakSelf = self] in
            weakSelf?.doSomething()
        }
    }
}
```

> - ObjC Block：值捕获（捕获变量当前值的 const 副本），`__block` 才是引用
> - Swift 闭包：引用捕获（捕获变量本身），捕获列表 `[x]` 才是值捕获

## 十四、AutoreleasePool 底层实现

### 14.1 数据结构

```c
// AutoreleasePoolPage（C++ 类）
struct AutoreleasePoolPage {
    magic_t const magic;
    id *next;              // 下一个存放 autorelease 对象的位置
    pthread_t const thread; // 所属线程
    AutoreleasePoolPage *parent;  // 双向链表
    AutoreleasePoolPage *child;
    uint32_t const depth;
    uint32_t hiwat;        // high water mark

    static size_t const SIZE = 4096;  // 一页 4096 字节
    id *begin();
    id *end();
};

// 多个 Page 组成双向链表
// 每个线程有自己的 Page 链表（通过 TLS 存储 hotPage）
```

```
AutoreleasePoolPage 链表：

Page1 (parent=nil) ←→ Page2 ←→ Page3 (hotPage)
┌────────────┐     ┌────────────┐     ┌────────────┐
│ POOL_SENTINEL│    │  obj4      │     │  obj7      │
│ obj1        │     │  obj5      │     │  ...       │
│ obj2        │     │  POOL_SENTINEL   │            │
│ obj3        │     │            │     │  next →    │
│ next →      │     │  next →    │     └────────────┘
└────────────┘     └────────────┘
```

### 14.2 push / pop

```c
// push：压入哨兵对象 POOL_SENTINEL（nil）
void *objc_autoreleasePoolPush() {
    return AutoreleasePoolPage::push();
    // 1. 获取 hotPage
    // 2. 在 next 位置存放 POOL_SENTINEL
    // 3. 返回哨兵位置（作为 pop 的参数）
}

// pop：从哨兵位置开始，向 next 释放所有对象
void objc_autoreleasePoolPop(void *token) {
    AutoreleasePoolPage::pop(token);
    // 1. 找到 token 所在的 Page
    // 2. 从 next-1 开始，向 token 方向逐个 release
    // 3. 遇到 POOL_SENTINEL 停止
    // 4. 更新 next 指针
}
```

## 十五、Tagged Pointer

### 15.1 什么是 Tagged Pointer

```
传统方式：小数值（如 NSNumber 存 42）也要分配堆对象
    栈上的指针 ──→ 堆上的 NSNumber 对象（retainCount=1, value=42）
    开销：16字节对象 + 8字节指针 = 24字节 存一个整数

Tagged Pointer：将值直接存储在指针中
    指针本身：[标志位|类型|值] = 42
    无需分配堆对象，无需引用计数，直接存在指针里
    开销：8字节
```

### 15.2 Tagged Pointer 的判断

```objc
// ARM64 下 Tagged Pointer 判断
// 最高位（MSB）= 1 → Tagged Pointer
// ARM64 下支持扩展 Tagged Pointer（最高字节低3位非0）

// ObjC 源码
static inline bool
objc_isTaggedPointer(const void *ptr) {
    return ((uintptr_t)ptr & _OBJC_TAG_MASK) == _OBJC_TAG_MASK;
}

// 效果：
NSNumber *n1 = @42;     // Tagged Pointer（值存在指针中）
NSNumber *n2 = @(INT_MAX);  // 普通对象（值太大，指针放不下）
NSLog(@"%p", n1);  // 0x...2b（带标志位的值）
NSLog(@"%p", n2);  // 0x...（普通堆地址）
```

### 15.3 Tagged Pointer 的影响

```objc
// 1. 不参与引用计数
NSNumber *n = @42;
[n retain];   // 无操作
[n release];  // 无操作
// 不走 ARC 流程

// 2. weak 引用无意义
NSNumber *__weak weakN = @42;  // 编译通过但无效果（不会置 nil，因为本就不在堆上）

// 3. 不调用 dealloc
// Tagged Pointer 没有 dealloc

// 4. NSString 的 Tagged Pointer
NSString *shortStr = @"abc";        // 可能是 Tagged Pointer
NSString *longStr = @"很长的字符串..."; // 普通对象
// 判断：指针长度能容纳的短字符串使用 Tagged Pointer
```

## 十八、Swift 独有的内存特性

### 18.1 值类型的 Copy-on-Write

```swift
// Swift 的值类型（Array, Dictionary, Set, String, Struct）
// 赋值时不立即复制，而是在修改时才复制（Copy-on-Write）

var arr1 = [1, 2, 3, 4, 5]
var arr2 = arr1  // 此时共享底层缓冲区，未复制

arr2.append(6)   // 触发 Copy-on-Write，arr2 才真正复制
// arr1 仍是 [1, 2, 3, 4, 5]
// arr2 变为 [1, 2, 3, 4, 5, 6]

// COW 的实现原理：
// 内部使用引用计数的缓冲区
// 修改时检查引用计数：
//   - refCount == 1 → 直接修改（唯一持有者）
//   - refCount > 1  → 复制缓冲区后再修改
```

### 18.2 结构体中的类属性

```swift
// 结构体是值类型，但包含的类属性是引用类型
struct Wrapper {
    var obj: NSObject  // 引用类型
}

var w1 = Wrapper(obj: NSObject())
var w2 = w1  // w2 是 w1 的副本

// ⚠️ 但 w1.obj 和 w2.obj 指向同一个 NSObject！
// 结构体的 Copy-on-Write 只复制结构体本身
// 类属性仍然是共享的（浅拷贝）
```

### 18.3 Actor 的数据隔离

```swift
// Actor 保证内部数据的线程安全
// Actor 内的数据只能通过 actor 的方法访问
// 每次方法调用都是串行执行的

actor Counter {
    private var value = 0

    func increment() -> Int {
        value += 1
        return value
    }

    func getValue() -> Int {
        value  // 隐式 self 是隔离的
    }
}

let counter = Counter()
Task {
    let v = await counter.increment()  // 必须用 await（跨隔离边界）
}
```

### 18.4 Sendable 协议与并发安全

```swift
// Sendable：类型可以安全地跨并发域传递
// 值类型（所有属性都是值类型）→ 自动 Sendable
// 引用类型 → 需要 final class + 无可变状态，或手动标记 @unchecked Sendable

struct Point: Sendable {  // ✅ 自动满足（纯值类型）
    var x: Double, y: Double
}

class SharedCache: @unchecked Sendable {  // ⚠️ 自行保证线程安全
    private let lock = NSLock()
    private var _cache: [String: Any] = [:]

    func get(_ key: String) -> Any? {
        lock.lock()
        defer { lock.unlock() }
        return _cache[key]
    }

    func set(_ key: String, _ value: Any) {
        lock.lock()
        _cache[key] = value
        lock.unlock()
    }
}
```

## 十九、内存相关面试题

### Q1: ARC 是垃圾回收吗？

```
不是。ARC 是编译器在编译期自动插入 retain/release/autorelease 代码。
与垃圾回收（GC）的区别：
- ARC：编译期确定，引用计数为 0 时立即释放，确定性强
- GC：运行时后台线程回收，释放时机不确定，可能暂停程序（Stop-the-world）
- ARC 有循环引用问题，GC 可以处理循环引用
```

### Q2: 为什么 weak 对象销毁时会自动置 nil？

```
1. weak 指针注册时，地址被存入 SideTable 的 weak_table_t 中
2. 对象 dealloc 时，调用 objc_clear_deallocating
3. 遍历 weak_table 中该对象对应的所有 weak 指针，逐一置 nil
4. 从 weak_table 中移除条目
```

### Q3: @autoreleasepool 什么时候使用？

```
1. 大循环中创建大量临时对象 → 降低内存峰值
2. 子线程中使用 autorelease 对象 → 防止泄漏
3. 框架/库的入口方法 → 隔离 autorelease 对象
```

### Q4: assign 和 weak 的区别？

```
相同：都不增加引用计数
不同：
- assign：对象销毁后指针不置 nil，可能野指针崩溃
- weak：对象销毁后自动置 nil，安全

- assign 可用于非对象类型（int, CGFloat 等）
- weak 只能用于对象类型

- ARC 下 delegate 用 weak，MRC 下用 assign
```

### Q5: strong 和 copy 的区别？

```
- strong：持有对象本身（引用计数 +1）
- copy：持有对象的副本（copy 后持有新对象）

对 NSString：
- strong 赋值 NSMutableString → 指向可变对象，外部修改会影响属性
- copy 赋值 NSMutableString → 创建不可变副本，外部修改不影响

对 Block：
- strong（ARC）：Block 自动 copy 到堆，与 copy 等价
- copy（MRC）：必须用 copy 将栈 Block 拷贝到堆
```

### Q6: 如何检测和定位内存泄漏？

```
1. Instruments Leaks：实时检测泄漏
2. Instruments Allocations：记录所有分配/释放
3. Xcode Memory Graph：可视化查看对象引用关系
4. MLeaksFinder：运行时自动检测 VC/View 泄漏
5. FBRetainCycleDetector：运行时检测循环引用
6. 代码层面：deinit/dealloc 日志确认
7. Zombie Objects：检测野指针
```

### Q7: Swift 的 weak 和 unowned 什么时候用？

```
weak：
- 引用的对象可能先于引用方释放
- 使用时需要解包（Optional）
- 典型场景：delegate、闭包中的 self

unowned：
- 引用的对象生命周期 >= 引用方
- 不需要解包（非 Optional）
- 典型场景：主从关系中从→主的引用（如 CreditCard→Customer）
- 访问已释放对象会崩溃
```

### Q8:autoreleasepool 的底层数据结构？

```
- 由 AutoreleasePoolPage 组成的双向链表
- 每个 Page 4096 字节，存储 autorelease 对象的指针
- push 压入 POOL_SENTINEL（哨兵 nil）
- pop 从 next 向哨兵方向逐个 release 对象
- 嵌套 pool 对应多个哨兵
- 每个线程通过 TLS 持有自己的 hotPage
```

### Q9: weak vs unsafeUnretained vs unmanaged 深度对比

```
三者都是 Swift 中处理引用关系的关键字，核心区别在于是否参与引用计数管理：

weak：
- 参与 ARC 管理，对象释放时自动置 nil
- 类型为 Optional，访问时需解包
- 使用场景：可能先于引用方释放的对象（delegate、闭包 capture）

unsafeUnretained：
- 不增加引用计数，不自动置 nil
- 类型为 non-optional，访问已释放对象直接崩溃
- 使用场景：确定生命周期 >= 当前对象，且不想 Optional 的场合

unmanaged：
- 完全手动管理 retain/release，ARC 不介入
- 通过 .takeRetainedValue() 或 .takeUnretainedValue() 手动接管
- 使用场景：桥接 C/ObjC API、手动内存管理的边界场景

// 示例：闭包中三种 capture 对比
class Service {
    var handler: (() -> Void)?
    func fetch(completion: @escaping () -> Void) {
        completion()
    }
}

class ViewController: UIViewController {
    let service = Service()

    func setup() {
        // weak：安全但需解包
        service.handler = { [weak self] in
            self?.updateUI()        // Optional 访问
        }

        // unsafeUnretained：不需解包但风险高
        service.handler = { [unsafeUnretained self] in
            self.updateUI()         // 直接访问，VC 释放后 crash
        }
    }
}
```

**Pitfalls:** unsafeUnretained 在 delegate 模式中极度危险——如果 delegate 对象先于持有者释放，每次调用都会 crash，且没有 weak 的 nil 保护机制。

**Best Practice:** 99% 的场景用 weak + Optional 解包；仅在性能敏感且生命周期可控时用 unsafeUnretained；unmanaged 仅用于与 C/ObjC 手动内存管理 API 桥接时。

### Q10: 僵尸对象调试（NSZombieEnabled、MallocStackLogging、malloc_history）

```
当对象被释放后仍收到消息时，通常表现为神秘的 "message to deallocated instance" crash。
调试这类问题的三板斧：

1. NSZombieEnabled —— 让已释放对象"复活"为僵尸，拦截非法消息
2. MallocStackLogging —— 记录每块内存的分配/释放堆栈
3. malloc_history —— 查询特定地址的完整生命周期

// Step 1: 在 scheme 中启用 NSZombie
// Edit Scheme -> Diagnostics -> Enable Zombie Objects
// 或者在代码中设置 environment variable:
// NSZombieEnabled = YES, NSZombieExtraEnabled = YES

// Step 2: 崩溃后定位分配点
// 终端执行: malloc_history <pid> <crash_address>
// 输出包含 alloc 和 free 的完整调用堆栈
// 示例: malloc_history 1234 0x600003245000

// Step 3: 在 Xcode 中预启用 MallocStackLogging
// scheme -> Run -> Diagnostics -> Malloc Stack Logging
// 配合 Address Sanitizer 使用效果更佳

// 实战案例：NSTimer 导致的僵尸对象
var timer: Timer?  // 强引用 timer，但 timer 又强引用 self
func start() {
    timer = Timer.scheduledTimer(timeInterval: 1.0, target: self,
                                  selector: #selector(tick), userInfo: nil, repeats: true)
}
// 问题：self 被 dealloc 后，timer 仍持有 self 引用 → crash
// NSZombie 输出: *** -[__NSZombie_NSObject tick]: message sent to deallocated instance 0x...
// 解决：invalidate timer in deinit，或使用 weak 包装
deinit {
    timer?.invalidate()
    timer = nil
}
```

**Pitfalls:** NSZombie 会阻止对象真正释放，导致内存持续增长，仅用于 Debug 构建，切勿带入 Release。且它会掩盖真正的 retain cycle 问题——对象看起来被"释放"了，实际只是变成僵尸。

**Best Practice:** 日常调试优先开启 NSZombie + Malloc Stack Logging；生产环境 crash 用 malloc_history 分析符号化 dSYM；结合 Instruments Zombies 模板可自动检测僵尸对象创建源头。


### Q11: Instruments Allocations/Leaks/VM Trace 实战指导

```
三个最常用 Instruments 模板的实战组合拳，覆盖 iOS 内存问题的 90% 场景：

// 1. Allocations —— 追踪对象分配与实例大小
// 录制后按 "Alloc count" 排序，筛选峰值对象
// 关键操作：Record → 执行操作 → Stop → 展开 "Object Summary"
// 查看 "Size" 列定位内存大户，如 UIImage、NSData 等
// 配合 "Mark Operation" (Cmd-6) 标注操作区间，对比前后差异

// 2. Leaks —— 检测 retain cycle 导致的内存泄漏
// 自动扫描堆，标记无法释放的对象
// 关键操作：Record → 执行操作 → 等待 Leaks 弹出 "Found X leaks"
// 点击 leak → 查看 "Call Tree" 追溯分配路径
// 设置 "Record Allocates" 开启后显示分配堆栈
// 注意：Leaks 只检测 Objective-C 对象，Swift 结构体和原生类型不在此列

// 3. VM Trace —— 分析虚拟内存布局，定位隐私和性能问题
// macOS Ventura+ / iOS 17+ 可用
// 查看各内存区域分布：Code、Data、Stack、Heap、Compressed Page Table
// 关键用途：检测"内存碎片化"导致的 VM pressure
// 配合 "Physical Memory" 查看 RSS 和 Compressed 比例
// Compressed > 30% 时，系统频繁压缩/解压，性能显著下降
```

**Pitfalls:** Leaks 模板在 Swift 项目中经常报"假漏"—— Swift 内部桥接对象、系统框架的私有对象可能被误报。务必先确认 leak 是否在自家代码路径中，避免浪费时间在系统框架上。

**Best Practice:** 标准排查流程——先用 Allocations 定位"谁占了最多内存"，再用 Leaks 确认"是否有未释放对象"，最后用 VM Trace 分析"虚拟内存是否碎片化"。三者结合形成完整的内存诊断闭环。每次测试前清空 Instruments 缓存，确保数据干净。

### Q12: autoreleasepool 放置策略（循环中、后台线程中）

```
autoreleasepool 的正确放置位置，决定了临时对象能否及时释放，直接影响内存峰值。

// ✅ 正确：循环内部创建 autoreleasepool
for _ in 0..<10000 {
    autoreleasepool {
        let data = NSData(contentsOfFile: path)!  // 大量临时 NSData
        // 处理 data ...
        // ← 循环结束，autoreleasepool 出作用域，data 立即释放
    }
}
// 内存峰值：仅一个 data 的大小

// ❌ 错误：autoreleasepool 在循环外部
autoreleasepool {
    for _ in 0..<10000 {
        let data = NSData(contentsOfFile: path)!
        // 处理 data ...
    }
    // ← 所有 10000 个 data 才一起释放
}
// 内存峰值：10000 个 data 的累计大小

// ✅ 后台线程必须手动创建 autoreleasepool
DispatchQueue.global().async {
    autoreleasepool {
        // 所有创建的对象在此作用域内释放
        let result = self.processData()
        self.handleResult(result)
    }
}

// ❌ 后台线程忘记 autoreleasepool
DispatchQueue.global().async {
    // 创建的对象无法自动释放，直到线程结束才清理
    let data = NSMutableData()  // ← 泄漏！
    data.append(someBytes)
}
```

**原理：** 主线程每个 RunLoop 周期自动创建 autoreleasepool，但后台线程没有 RunLoop，必须手动管理。循环中每轮创建 autoreleasepool 可将内存峰值从 O(n) 降为 O(1)。\n

**Pitfalls:** 将 autoreleasepool 放在循环外部时，所有临时对象累积到循环结束后才释放，可能导致 Memory Warning 甚至 Killed by system。后台线程忘记包 autoreleasepool 是 Swift 中最常见的"隐性内存泄漏"来源。\n

**Best Practice:** 循环内每轮包一层 autoreleasepool 控制峰值；后台线程入口第一行就是 autoreleasepool {；析构敏感操作（如图片解码、大文件读取）单独嵌套 autoreleasepool。

### Q13: Swift deinit vs ObjC dealloc 区别

Swift 的 `deinit` 与 Objective-C 的 `dealloc` 都是析构函数，但机制差异显著：

```swift
// Swift deinit：编译器生成，直接释放值类型成员，ARC 自动处理引用计数
class SwiftResource {
    var data: Data
    init(data: Data) { self.data = data }
    deinit {
        print("Swift deinit called")
        // 不需要 super.deinit()
        // ARC 自动释放 self 持有的引用
    }
}

// ObjC dealloc：必须通过 [super dealloc] (MRC) 或自动调用 super (ARC)
// dealloc 内部仍可访问实例变量，但不能发消息给 self
- (void)dealloc {
    [[NSNotificationCenter defaultCenter] removeObserver:self];
    // ARC 下不需要手动 release，super dealloc 自动调用
}
```

**核心差异：**
1. Swift `deinit` 无法被 override，ObjC `dealloc` 可被子类 override
2. Swift 值类型（struct/enum）析构由编译器隐式处理，无需手动 deinit
3. ObjC `dealloc` 中仍可访问 ivar 但不能发消息给 self，Swift `deinit` 中实例完全不可变
4. Swift 的 deinit 在引用计数为0时立即执行；ObjC 在最后一个引用释放后调用 dealloc

**Pitfalls:** Swift 中在 deinit 内发送 async 消息或闭包捕获 self 会导致崩溃 —— deinit 执行期间实例已处于销毁状态，任何延迟访问都会访问已释放内存。ObjC 中在 dealloc 内调用 `self.someMethod` 同样崩溃。

**Best Practice:** 释放资源（通知移除、定时器失效、文件句柄关闭）放在 deinit/dealloc 中；避免在析构函数中执行网络请求、disk I/O 或任何可能延迟的操作。

### Swift Unmanaged\<T\> 与桥接内存所有权

在 Swift 与 Objective-C 互操作时，`Unmanaged\<T\>` 是处理 `UnsafeMutableRawPointer` 类型的桥梁，它让开发者手动控制跨语言对象的内存所有权。

```swift
// ObjC 侧：将 Swift 对象转为 opaque pointer
weakCallbacks.context = Unmanaged<SwiftHandler>.toOpaque(handler)

// Swift 侧：从 opaque pointer 恢复对象，手动管理 retain/release
let handler = Unmanaged<SwiftHandler>.fromOpaque(context).takeUnretainedValue()  // 不改变 RC
let handler2 = Unmanaged<SwiftHandler>.fromOpaque(context).takeRetainedValue()    // -1 release
let ptr = Unmanaged<SwiftHandler>.passUnretained(handler)                          // 不 retain
let ptr2 = Unmanaged<SwiftHandler>.passRetained(handler)                           // +1 retain
```

**四种取值/传递方式：**
- `takeUnretainedValue()` / `passUnretained()` — 不改变引用计数，调用者保证对象存活
- `takeRetainedValue()` / `passRetained()` — 转移所有权，take 时自动 release
- 核心规则：pass 和 take 必须配对使用（Retained↔Retained，Unretained↔Unretained）

**Pitfalls:** Retained 和 Unretained 混用是最致命的 bug —— `passRetained` 后用 `takeUnretainedValue` 取值会导致内存泄漏（多 retain 未 release）；`passUnretained` 后用 `takeRetainedValue` 会导致 over-release 崩溃。

**Best Practice:** 优先使用 `takeUnretainedValue()`（如 callback context 场景），确保对象生命周期由创建方管理；仅在需要跨线程/跨模块转移所有权时使用 `passRetained`/`takeRetainedValue` 配对。

### NSMapTable / NSHashTable vs 弱引用/强引用字典性能对比

`NSMapTable` 和 `NSHashTable` 是 Foundation 提供的替代 `NSDictionary` / `NSSet` 的集合类型，核心优势在于**可配置内存管理语义**，而普通字典只能强持有键值。

```objc
// NSMapTable：弱引用 key + 强引用 value（模拟 weak-to-strong 映射）
NSMapTable *weakToStrong = [NSMapTable mapTableWithKeyOptions:NSMapTableWeakMemory
                                                      valueOptions:NSMapTableStrongMemory];
[weakToStrong setObject:myViewController forKey:self]; // key 释放时自动移除 entry

// NSMapTable：双弱引用（observers / delegates 场景）
NSMapTable *weakToWeak = [NSMapTable mapTableWithKeyOptions:NSMapTableWeakMemory
                                                     valueOptions:NSMapTableWeakMemory];
// key 或 value 任一释放，entry 自动清理，无 retain cycle 风险

// NSHashTable：弱引用集合（替代 NSMutableSet 存储观察者）
NSHashTable *weakSet = [NSHashTable hashTableWithOptions:NSHashTableWeakMemory];
[weakSet addObject:observer]; // observer 释放时自动从集合中移除
```

**与弱引用字典的性能对比：**
- **NSMapTable** 在弱引用场景下，entry 自动清理是**即时触发**的（dealloc 路径），延迟远低于弱引用字典的异步清理
- **弱引用字典**（weak-keyed dict）依赖 `NSMapTable` 底层实现，但语义不够精确，且 iOS 13+ 才稳定支持
- **NSHashTable** 比 `NSMutableSet` + `__weak` wrapper 方案更高效，避免了包装对象的额外 alloc/dealloc 开销
- 多线程安全：`NSMapTable` / `NSHashTable` 与 `NSDictionary` 一样**不保证线程安全**，需自行加锁

**Pitfalls:** NSMapTable 的弱引用 entry 在对象 dealloc 时自动移除，但遍历过程中如果有 entry 被释放，会导致 `enumerateKeysAndObjectsUsingBlock:` 行为未定义 —— 必须先拷贝或使用 `weakToWeak` 配合 `__weak` 局部变量。

**Best Practice:** Delegates / observers 缓存统一用 `NSHashTable weakMemory` 或 `NSMapTable weakToWeak`；需要弱 key 映射强 value 时用 `NSMapTable weakToStrong`；普通键值缓存仍用 `NSDictionary` / `NSCache`，仅在需要弱引用语义时才切换到 MapTable/HashTable。

### UIImage / NSData 大内存占用场景优化

iOS 应用中 UIImage 和 NSData 是最常见的内存大户。`UIImagePNGRepresentation` 和 `UIImageJPEGRepresentation` 会在调用时将图片数据解码到内存，导致内存峰值飙升，大图（如 12MP 照片）可占用数百 MB。`NSData` 从磁盘或网络加载时同样会将完整数据驻留内存。

```swift
// ❌ 危险：全分辨率解码，12MP JPEG → ~48MB 未压缩像素 + 原始 data
let image = UIImage(contentsOfFile: path)
let data = UIImagePNGRepresentation(image)! // 再次编码，额外占用

// ✅ 优化1：按需缩放，避免全分辨率解码
let thumbnail = UIImage(contentsOfFile: path, scale: 0.5, capInsets: .zero)

// ✅ 优化2：使用 CGImageSource 指定目标尺寸，直接解码为小图
let url = URL(fileURLWithPath: path)
let source = CGImageSourceCreateWithURL(url as CFURL, nil)!
let options: [CFString: Any] = [
    kCGImageSourceCreateThumbnailFromImageAlways: true,
    kCGImageSourceThumbnailMaxPixelSize: 300  // 最大边 300px
]
if let cgImage = CGImageSourceCreateThumbnailAtIndex(source, 0, options) {
    let optimizedImage = UIImage(cgImage: cgImage)  // ~900KB vs ~48MB
}

// ✅ 优化3：NSData 分块读取，避免全量加载
let fileHandle = FileHandle(forReadingFrom: url)!
while let chunk = fileHandle.readData(ofLength: 4096), !chunk.isEmpty {
    // 逐块处理，内存恒定 ~4KB
}
```

**Pitfalls:** `imageNamed:` 会将图片缓存到系统级缓存中永不释放，大量图片使用 imageNamed 导致内存只增不减；应在展示固定资源时使用 imageNamed，动态图片一律用 `imageWithContentsOfFile:` 或 `UIImage(contentsOfFile:)`。

**Best Practice:** 列表/无限滚动场景统一用 CGImageSource 缩放 + 自定义 LRU 缓存（限 50-100 张）；用户头像等固定资源用 imageNamed；避免对同一图片重复编码（PNG/JPEG Representation）。

### 内存警告（didReceiveMemoryWarning）处理策略

当系统发出低内存警告时，`UIViewController` 的 `didReceiveMemoryWarning` 会被自动调用。父类实现会释放未使用的 view（`isViewLoaded == false` 时），子类必须调用 `super` 以保留此行为。开发者需在此方法中主动释放可重建的资源缓存。

```swift
override func didReceiveMemoryWarning() {
    super.didReceiveMemoryWarning()  // ⚠️ 必须调用，否则 view 不会被正确释放
    
    // 1. 清除图片/数据缓存
    imageCache.removeAll()
    apiResponseCache.removeAll()
    
    // 2. 释放非必要的预处理数据
    sortedItems = nil
    diffResult = nil
    
    // 3. 暂停后台任务
    backgroundProcessor.pause()
    
    // 4. 卸载离屏视图控制器
    if let nav = self as? UINavigationController {
        let keep = nav.visibleViewController
        nav.viewControllers = [keep ?? nav.topViewController!]
    }
}
```

**Pitfalls:** 忘记调用 `super.didReceiveMemoryWarning()` 导致系统级 view 卸载机制失效；在此方法中做耗时操作（如磁盘写入大文件）可能触发 watchdog 超时被杀。

**Best Practice:** didReceiveMemoryWarning 中只做 O(1) 的释放操作（清除缓存引用、置 nil），耗时重建逻辑延迟到下次访问时懒加载；优先释放大对象（图片、数据缓冲区），保留用户可见状态和表单输入。

### CALayer 隐式动画内存开销

CALayer 的隐式动画（Implicit Animation）会在每个 Property 改变时自动触发 Core Animation 的事务动画，但每个动画都会创建中间帧的位图缓冲区。快速连续修改多个 Layer 属性（如在循环中改 frame/position/bounds）会导致大量临时 CGBitmapContext 堆积，每个约 4-16MB，峰值可超 500MB。

```swift
// ❌ 危险：隐式动画为每次 property 变化创建位图缓冲区
for i in 0..<100 {
    layer.position = CGPoint(x: i, y: 0)  // 100 次隐式动画 = ~1GB 峰值
    layer.bounds.size.width = CGFloat(i)
    layer.backgroundColor = UIColor(hue: CGFloat(i)/100, saturation: 1, brightness: 1, alpha: 1).cgColor
}

// ✅ 正确：批量操作包裹在 disableActions 或 CATransaction 中
CATransaction.begin()
CATransaction.setDisableActions(true)  // 关闭隐式动画
for i in 0..<100 {
    layer.position = CGPoint(x: i, y: 0)  // 仅记录最终状态，零额外内存
    layer.bounds.size.width = CGFloat(i)
}
CATransaction.commit()

// ✅ 替代方案：显式使用 UIView.animate 或 CASpringAnimation，可控帧数
```

**Pitfalls:** 快速滚动的 UITableView 中自定义 Layer 动画可能导致内存随滑动距离线性增长； `layer.shadowPath`、`layer.mask`、`layer.shouldRasterize = true` 触发离屏渲染，每个 Layer 额外占用 screen_width × screen_height × 4 bytes 的离屏缓冲区。

**Best Practice:** 批量修改 Layer 属性时一律用 `CATransaction.setDisableActions(true)` 包裹；需要动画时用 `UIView.animate` 或 `UIViewPropertyAnimator` 统一管理；对大量自定义 Layer 用 `shouldRasterize` 前评估光栅化缓存成本（通常不推荐）。

### Core Data 内存管理（NSPersistentStoreCoordinator 缓存）

NSPersistentStoreCoordinator 默认的 `resultType` 是 `TupleTypeResult`，加载所有列数据到内存。对于大数据量表（如万级消息记录），每条记录占用数百字节，上下文捕获集可迅速膨胀至数百 MB。配合 `NSFetchRequest` 的 `includesPendingChanges` 和 `NSManagedObjectContext` 的 undo/redo 栈，内存使用呈指数级增长。

```swift
// ✅ 1. 使用 NSBatchDeleteRequest 批量删除（不加载到 MOC）
let deleteReq = NSBatchDeleteRequest(fetchRequest: fetchReq)
try context.execute(deleteReq)  // 直接操作 SQLite，绕过 MOC 缓存

// ✅ 2. 控制 fetch 结果类型
fetchReq.resultType = .serializedPropertiesResultType  // 返回 NSDictionary，避免 NSManagedObject 开销
// fetchReq.resultType = .countTypeResult  // 仅统计数量，零对象开销

// ✅ 3. 定期 reset 上下文释放捕获集
context.reset()  // 清空所有对象引用，释放内存
```

**Pitfalls:** 使用后台 PersistentStore 后忘记合并变更，导致主线程看到过期数据；undo 栈不限制容量时，一次大事务可吃掉数百 MB（`undoManager.limit` 默认无上限）。

**Best Practice:** 后台上下文处理完立即 `save()` + `reset()`；主线程用 `NSPersistentStoreCoordinator` 的 `notificationKeysForChangesInContext:` 配合 `NSManagedObjectContextObjectsDidChange` 选择性合并，而非全量 merge。


### Swift 6 所有权追踪类型（实验性）

Swift 6 引入实验性的「追踪类型」（Tracking Types），允许结构体在不拥有成员的情况下 "持有" 对其的引用，类似在值类型中嵌入到另一个值的借用引用。这是 Swift 逐步实现 borrowsafe 模型的核心特性之一，目标是消除数据竞争的同时减少不必要的拷贝。

追踪类型通过 `~C` 语法标记一个属性为 "追踪的"（borrowed），编译器保证该属性的生命周期不超过其宿主结构体的生命周期。结合 Swift 6 的严格并发检查（Strict Concurrency Checking），追踪类型可以在并发安全的前提下实现零拷贝传递。

```swift
// Swift 6 实验性追踪类型示例
// 概念性代码：追踪类型目前需开启 -enable-upcoming-feature TrackingTypes

struct ShapeRenderer {
    let shape: ~Shape  // 追踪引用 — shape 被借用，不转移所有权
    let context: CGContext  // 普通拥有属性

    func render() {
        // 可以安全读取 shape 数据，但不拥有它
        drawShape(shape, to: context)
    }
}

// 使用：shape 仍由调用方拥有
let myShape = Shape(type: .circle, radius: 50)
let renderer = ShapeRenderer(shape: &myShape, context: ctx)  // 借入
renderer.render()
// renderer 超出作用域后，借用结束
```

**Pitfalls:** 追踪类型目前为实验性功能（-enable-upcoming-feature），API 可能随时变更；不能将追踪属性赋值给需要所有权的参数（如 `let s: Shape = renderer.shape`）；追踪引用不能在闭包中逃逸。

**Best Practice:** 在高性能渲染管线、游戏引擎场景图等需要频繁传递只读引用但避免拷贝的场景中试验追踪类型；当前阶段建议通过 Xcode 的 Swift Evolution 预览或 Swift 6 实验性 flag 评估可行性，正式代码暂不依赖此特性。
