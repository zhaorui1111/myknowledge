# iOS 锁的底层原理详解

---

## 一、为什么需要锁

### 1.1 数据竞争问题

```objc
// 两个线程同时对 count 执行 +1
@property (nonatomic, assign) NSInteger count; // count = 0

// Thread A                    Thread B
// 读取 count = 0              读取 count = 0
// 计算 0 + 1 = 1              计算 0 + 1 = 1
// 写入 count = 1              写入 count = 1

// 期望 count = 2，实际 count = 1
// 原因："读-改-写" 不是一步完成的，中间可以被其他线程打断
```

### 1.2 自旋锁 vs 互斥锁

```
两种最基本的锁类型：

自旋锁（Spinlock）：
  获取失败 → CPU 忙等（while 循环不断尝试）
  不做线程切换，一直占着 CPU
  适合：临界区非常短，锁很快就能拿到

互斥锁（Mutex）：
  获取失败 → 线程休眠（让出 CPU，等待唤醒）
  涉及用户态 → 内核态的上下文切换
  适合：临界区较长，或者锁竞争激烈

现代锁一般是混合策略：
  先自旋几轮（看能不能快速拿到）
  拿不到再休眠（避免浪费 CPU）
  os_unfair_lock、pthread_mutex 都是这种混合策略
```

## 二、OSSpinLock（已废弃）

### 2.1 原理与用法

```objc
#import <libkern/OSAtomic.h>

OSSpinLock lock = OS_SPINLOCK_INIT;

OSSpinLockLock(&lock);
// 临界区
OSSpinLockUnlock(&lock);

// 底层原理：
// 1. 本质就是一个 int32_t，0 = 未锁，1 = 已锁
// 2. 加锁时用原子 CAS（Compare-And-Swap）把 0 改为 1
// 3. 如果已经是 1（被占用），就在 while 循环里不断重试
// 4. 解锁时原子地把 1 改回 0
// 整个过程都在用户态完成，不涉及系统调用 → 所以极快
```

### 2.2 为什么被废弃：优先级反转

```
场景：
  Thread L（低优先级）获得了锁，正在执行
  Thread H（高优先级）请求锁 → 自旋忙等

  问题来了：
  H 优先级高 → CPU 一直调度 H（但 H 在忙等，什么也做不了）
  L 优先级低 → 得不到 CPU 时间 → 无法完成临界区 → 无法释放锁
  → H 永远拿不到锁，L 永远执行不了 → 类似死锁

  为什么互斥锁没这个问题：
  互斥锁等待时线程休眠，系统知道 "H 在等 L 的锁"
  → 系统临时提升 L 的优先级（优先级继承 Priority Inheritance）
  → L 执行完释放锁 → H 被唤醒

  自旋锁做不到优先级继承：
  自旋是纯用户态行为，内核不知道谁在等谁
  → 无法做优先级继承

  Apple 在 iOS 10 废弃了 OSSpinLock，替代品是 os_unfair_lock
```

## 三、os_unfair_lock（推荐）

### 3.1 用法

```objc
#import <os/lock.h>

os_unfair_lock lock = OS_UNFAIR_LOCK_INIT;

// 加锁
os_unfair_lock_lock(&lock);
// 临界区
os_unfair_lock_unlock(&lock);

// 尝试加锁（非阻塞）
if (os_unfair_lock_trylock(&lock)) {
    // 获得锁
    os_unfair_lock_unlock(&lock);
}
```

### 3.2 原理

```
os_unfair_lock 的底层原理：

1. 本质是一个 uint32_t，记录了锁状态和持有者线程信息

2. 加锁分两步：
   快速路径：原子 CAS 尝试获取 → 如果锁空闲直接拿到（纯用户态，极快）
   慢速路径：锁被占用 → 调用内核 __ulock_wait → 线程休眠等待唤醒

3. 解锁也分两步：
   原子地清除锁状态
   如果有等待者 → 调用内核 __ulock_wake 唤醒一个等待线程

4. 支持优先级继承：
   内核知道"谁持有锁、谁在等锁"
   高优先级线程等待时，系统会临时提升持有者的优先级

5. "unfair" 的含义：
   唤醒时不保证 FIFO 顺序，谁先抢到就是谁的
   这样性能更好（不需要维护有序队列）
```

### 3.3 注意事项

```objc
// ⚠️ 不可递归：同一线程重复加锁会死锁
os_unfair_lock_lock(&lock);
os_unfair_lock_lock(&lock);  // 💥 死锁

// ⚠️ 不能在错误线程解锁（锁记录了持有者线程 ID）

// ⚠️ Swift 中的坑：os_unfair_lock 是值类型，可能被复制
// ❌ 错误
class MyClass {
    var lock = os_unfair_lock()
    func bad() {
        os_unfair_lock_lock(&lock)  // &lock 可能取到临时拷贝的地址
    }
}

// ✅ 正确：使用指针
class MyClass {
    private let lockPtr: UnsafeMutablePointer<os_unfair_lock>
    init() {
        lockPtr = .allocate(capacity: 1)
        lockPtr.initialize(to: os_unfair_lock())
    }
    deinit {
        lockPtr.deinitialize(count: 1)
        lockPtr.deallocate()
    }
    func safe() {
        os_unfair_lock_lock(lockPtr)
        defer { os_unfair_lock_unlock(lockPtr) }
        // 临界区
    }
}

// ✅ 更好：iOS 16+ 使用 OSAllocatedUnfairLock（见第九章）
```

## 四、pthread_mutex

### 4.1 用法

```c
#import <pthread.h>

// 静态初始化
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

// 动态初始化（可配置属性）
pthread_mutex_t mutex;
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_NORMAL);
pthread_mutex_init(&mutex, &attr);
pthread_mutexattr_destroy(&attr);

// 加锁/解锁
pthread_mutex_lock(&mutex);
// 临界区
pthread_mutex_unlock(&mutex);

// 尝试加锁（非阻塞）
if (pthread_mutex_trylock(&mutex) == 0) {
    // 获得锁
    pthread_mutex_unlock(&mutex);
}

// 销毁
pthread_mutex_destroy(&mutex);
```

### 4.2 三种类型

```c
// 1. PTHREAD_MUTEX_NORMAL（默认）
// 同一线程重复加锁 → 死锁
// 性能最好，最常用
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_NORMAL);

// 2. PTHREAD_MUTEX_RECURSIVE（递归锁）
// 同一线程可以多次加锁，内部维护一个计数器
// 加锁 N 次就要解锁 N 次才真正释放
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);

// 3. PTHREAD_MUTEX_ERRORCHECK（检错锁）
// 同一线程重复加锁不会死锁，而是返回错误码 EDEADLK
// 适合调试阶段使用
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_ERRORCHECK);
```

### 4.3 原理

```
pthread_mutex 的底层原理：

1. 内部结构包含：锁状态、持有者线程 ID、类型标识、递归计数

2. 加锁流程：
   快速路径：CAS 尝试获取 → 锁空闲直接拿到
   慢速路径：锁被占用 → 检查类型：
     RECURSIVE 且同一线程 → 计数 +1，直接返回
     ERRORCHECK 且同一线程 → 返回 EDEADLK 错误
     其他 → 调用 __psynch_mutexwait 系统调用，线程休眠

3. 解锁流程：
   递归锁且计数 > 0 → 计数 -1，直接返回
   否则 → 清除持有者，如果有等待者则调用 __psynch_mutexdrop 唤醒

4. 支持优先级继承（可通过属性配置）：
   pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
```

### 4.4 条件变量 pthread_cond

```c
// 条件变量：让线程等待某个条件满足后再继续

pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cond = PTHREAD_COND_INITIALIZER;
BOOL dataReady = NO;

// 生产者
pthread_mutex_lock(&mutex);
// 生产数据...
dataReady = YES;
pthread_cond_signal(&cond);    // 唤醒一个等待线程
// pthread_cond_broadcast(&cond); // 唤醒所有等待线程
pthread_mutex_unlock(&mutex);

// 消费者
pthread_mutex_lock(&mutex);
while (!dataReady) {  // ⚠️ 必须用 while，不能用 if（防止虚假唤醒）
    pthread_cond_wait(&cond, &mutex);
    // wait 做了三件事（原子地）：
    // 1. 释放 mutex
    // 2. 线程休眠，等待 signal
    // 3. 被唤醒后重新获取 mutex
}
// 消费数据...
pthread_mutex_unlock(&mutex);

// ⚠️ 虚假唤醒（Spurious Wakeup）：
// wait 可能在没有 signal 的情况下返回
// 所以必须在 while 循环中检查条件
```

### 4.5 读写锁 pthread_rwlock

```c
// 读写锁：多读单写
// 多个线程可以同时读（共享锁）
// 写时必须独占（排他锁）

pthread_rwlock_t rwlock = PTHREAD_RWLOCK_INITIALIZER;

// 读锁（多个线程可同时持有）
pthread_rwlock_rdlock(&rwlock);
// 读取数据...
pthread_rwlock_unlock(&rwlock);

// 写锁（排他，等所有读锁和写锁释放后才能获取）
pthread_rwlock_wrlock(&rwlock);
// 写入数据...
pthread_rwlock_unlock(&rwlock);

pthread_rwlock_destroy(&rwlock);

// 原理：
// 内部维护：读者计数 + 写者标记 + 等待队列
// 读锁：没有写者时，读者计数 +1 → 直接通过
// 写锁：等所有读者退出 + 没有其他写者 → 独占
// 解锁时：写者优先唤醒（Apple 默认策略，防止写者饿死）
```

## 五、NSLock 家族

### 5.1 NSLock

```objc
// NSLock 是 pthread_mutex 的 ObjC 封装

NSLock *lock = [[NSLock alloc] init];

[lock lock];
// 临界区
[lock unlock];

// 尝试加锁
if ([lock tryLock]) {
    // 获得锁
    [lock unlock];
}

// 超时加锁
if ([lock lockBeforeDate:[NSDate dateWithTimeIntervalSinceNow:3.0]]) {
    // 3 秒内获得锁
    [lock unlock];
}

// 底层：pthread_mutex(ERRORCHECK) + PRIO_INHERIT
// ERRORCHECK 意味着同一线程重复加锁不会死锁，而是 tryLock 返回 NO
// PRIO_INHERIT 支持优先级继承
```

### 5.2 NSRecursiveLock

```objc
// 递归锁：同一线程可以多次加锁
// 底层是 pthread_mutex(RECURSIVE)

NSRecursiveLock *lock = [[NSRecursiveLock alloc] init];

// 场景1：递归函数
- (void)recursiveMethod:(NSInteger)depth {
    [lock lock];
    if (depth > 0) {
        [self recursiveMethod:depth - 1];  // 再次加锁，不会死锁
    }
    [lock unlock];
}

// 场景2：方法互调
- (void)methodA {
    [lock lock];
    [self methodB];  // methodB 也会加锁
    [lock unlock];
}

- (void)methodB {
    [lock lock];
    // ...
    [lock unlock];
}

// 原理：
// 加锁时检查 owner 是否是当前线程
// 是 → 计数 +1，直接返回
// 否 → 正常等待
// 解锁时计数 -1，计数为 0 时才真正释放
```

### 5.3 NSCondition

```objc
// NSCondition = pthread_mutex + pthread_cond 的组合封装
// 既是锁，又是条件变量

NSCondition *condition = [[NSCondition alloc] init];
BOOL dataReady = NO;

// 生产者
[condition lock];
dataReady = YES;
[condition signal];     // 唤醒一个等待线程
// [condition broadcast]; // 唤醒所有等待线程
[condition unlock];

// 消费者
[condition lock];
while (!dataReady) {
    [condition wait];
    // wait 会：
    // 1. 释放锁
    // 2. 线程休眠
    // 3. 被唤醒后重新获取锁
}
// 使用数据...
[condition unlock];

// ⚠️ wait 必须在 while 循环中使用（防止虚假唤醒）
```

### 5.4 NSConditionLock

```objc
// NSConditionLock = NSCondition + 整数条件值
// 可以根据条件值控制线程执行顺序

NSConditionLock *lock = [[NSConditionLock alloc] initWithCondition:0];

// 线程 A：等条件为 0 时执行
dispatch_async(queue, ^{
    [lock lockWhenCondition:0];
    NSLog(@"Step 1");
    [lock unlockWithCondition:1];  // 完成后把条件改为 1
});

// 线程 B：等条件为 1 时执行
dispatch_async(queue, ^{
    [lock lockWhenCondition:1];
    NSLog(@"Step 2");
    [lock unlockWithCondition:2];
});

// 线程 C：等条件为 2 时执行
dispatch_async(queue, ^{
    [lock lockWhenCondition:2];
    NSLog(@"Step 3");
    [lock unlock];
});

// 无论 A/B/C 谁先被调度，最终输出顺序都是 Step 1 → Step 2 → Step 3

// 原理：
// lockWhenCondition:N → lock + while(conditionValue != N) wait
// unlockWithCondition:N → conditionValue = N + broadcast + unlock
```

## 六、@synchronized

### 6.1 用法

```objc
@synchronized (obj) {
    // 临界区
    // 以 obj 为"锁标识"，同一个 obj 的 @synchronized 互斥
}

// 特点：
// ✅ 语法最简洁，自动配对不会忘记 unlock
// ✅ 支持递归加锁（同一线程对同一对象可以嵌套）
// ✅ 异常安全（内部有 @try/@finally 保证解锁）
// ❌ 性能较差（约为 os_unfair_lock 的 4-5 倍慢）
```

### 6.2 原理

```
编译器转换：
  @synchronized(obj) { ... }
  →
  objc_sync_enter(obj);
  @try { ... }
  @finally { objc_sync_exit(obj); }

底层实现（基于 objc4 源码）：
  1. 系统维护一个全局哈希表（64 个桶），key 是对象地址
  2. objc_sync_enter(obj) 时：
     - 用 obj 的地址做哈希，找到对应的桶
     - 在桶的链表中查找/创建一个 SyncData 节点
     - SyncData 内部包含一个递归锁（pthread_mutex RECURSIVE）
     - 对这个递归锁加锁
  3. 每个线程有 TLS 缓存，加速查找（不用每次都查全局表）
  4. 性能开销主要来自哈希表查找和 TLS 缓存管理
```

### 6.3 常见的坑

```objc
// 坑1：传 nil 不加锁！
id obj = nil;
@synchronized (obj) {
    // ⚠️ objc_sync_enter(nil) 直接返回，完全不加锁！
}

// 坑2：锁对象被替换
@property (nonatomic, strong) NSArray *data;

@synchronized (self.data) {  // 锁的是 数组A
    // ...
}
// 另一个线程替换了 data
self.data = @[@"new"];       // data 指向了 数组B
@synchronized (self.data) {  // 锁的是 数组B
    // ⚠️ 和上面不是同一把锁！
}

// 坑3：@synchronized(self) 粒度太大
@synchronized (self) {
    // 所有用 @synchronized(self) 保护的代码都互斥
    // 即使它们操作的是不同的数据
}

// ✅ 正确做法：每个资源用独立的锁对象
@implementation MyClass {
    NSObject *_dataLock;
    NSObject *_cacheLock;
}
- (instancetype)init {
    self = [super init];
    _dataLock = [NSObject new];
    _cacheLock = [NSObject new];
    return self;
}
- (void)updateData {
    @synchronized (_dataLock) { /* ... */ }
}
- (void)updateCache {
    @synchronized (_cacheLock) { /* ... */ }
}
```

## 七、dispatch_semaphore

### 7.1 用法

```objc
// 信号量：控制并发资源的数量

// 创建（value 表示初始可用资源数）
dispatch_semaphore_t sema = dispatch_semaphore_create(0);

// ========== 用途1：异步转同步 ==========
dispatch_async(queue, ^{
    // 异步操作...
    dispatch_semaphore_signal(sema);  // +1（释放资源）
});
dispatch_semaphore_wait(sema, DISPATCH_TIME_FOREVER);  // -1（获取资源，阻塞等待）

// ========== 用途2：当互斥锁使用（value=1）==========
dispatch_semaphore_t lockSema = dispatch_semaphore_create(1);

dispatch_semaphore_wait(lockSema, DISPATCH_TIME_FOREVER);  // 加锁
// 临界区
dispatch_semaphore_signal(lockSema);  // 解锁

// ========== 用途3：控制最大并发数 ==========
dispatch_semaphore_t limitSema = dispatch_semaphore_create(3);  // 最多 3 个并发

for (int i = 0; i < 100; i++) {
    dispatch_semaphore_wait(limitSema, DISPATCH_TIME_FOREVER);  // 获取许可
    dispatch_async(queue, ^{
        // 执行任务（同时最多 3 个）
        dispatch_semaphore_signal(limitSema);  // 释放许可
    });
}
```

### 7.2 原理

```
dispatch_semaphore 的原理：

内部维护一个 long 类型的值（dsema_value）

wait（获取资源）：
  1. 原子减 1
  2. 如果结果 ≥ 0 → 直接返回（快速路径，无系统调用）
  3. 如果结果 < 0 → 调用内核 Mach 信号量等待，线程休眠

signal（释放资源）：
  1. 原子加 1
  2. 如果结果 > 0 → 直接返回（之前没有等待者）
  3. 如果结果 ≤ 0 → 调用内核 Mach 信号量唤醒一个等待线程

值变化示例（value 初始为 1，当互斥锁用）：
  线程A wait:   1 → 0   ≥ 0 直接通过（获得锁）
  线程B wait:   0 → -1  < 0 休眠等待
  线程A signal: -1 → 0  ≤ 0 唤醒线程B
  线程B signal: 0 → 1   > 0 无需唤醒

⚠️ 关键区别：dispatch_semaphore 不支持优先级继承
  因为信号量不记录"谁持有它"，内核无法做优先级提升
  对 QoS 敏感的场景（如 UI 相关）应使用 os_unfair_lock
```

## 八、dispatch_barrier（多读单写）

### 8.1 用法

```objc
// dispatch_barrier 在并发队列中实现"多读单写"

dispatch_queue_t rwQueue = dispatch_queue_create("com.app.rw", DISPATCH_QUEUE_CONCURRENT);

// 读操作（可并发）
- (id)readData {
    __block id result;
    dispatch_sync(rwQueue, ^{
        result = self->_data;
    });
    return result;
}

// 写操作（独占执行）
- (void)writeData:(id)data {
    dispatch_barrier_async(rwQueue, ^{
        self->_data = data;
    });
}

// ⚠️ 必须用自定义并发队列，全局队列无效
```

### 8.2 原理

```
dispatch_barrier 的原理：

barrier 任务被提交到并发队列时带上 BARRIER 标记。

队列调度时：
  1. 普通任务：可并发执行
  2. 遇到 barrier 任务：
     a. 等待当前所有正在执行的任务完成
     b. 独占执行 barrier 任务（此时不调度其他任务）
     c. barrier 完成后，继续调度后续任务

时序图：
  [读1] [读2] [读3] [写(barrier)] [读4] [读5]

  时间 →
  读1 ──┐
  读2 ──┤          写(独占)         读4 ──┐
  读3 ──┘       ←等全部完成→         读5 ──┘

⚠️ 为什么全局队列无效：
  全局队列被整个系统共享
  barrier 无法独占全局队列（会影响其他模块）
  → GCD 忽略全局队列上的 barrier 标记
```

## 九、Swift 中的锁

### 9.1 NSLock（所有版本可用）

```swift
let lock = NSLock()

lock.lock()
// 临界区
lock.unlock()

// 推荐用 withLock 避免忘记 unlock
lock.withLock {
    // 临界区
}
```

### 9.2 OSAllocatedUnfairLock（iOS 16+）

```swift
import os

// Apple 官方为 Swift 提供的 os_unfair_lock 安全封装
// 是引用类型（class），不会被意外复制

let lock = OSAllocatedUnfairLock(initialState: 0)

// 锁和状态绑定在一起
lock.withLock { state in
    state += 1
}

// 带返回值
let value = lock.withLock { state -> Int in
    state += 1
    return state
}

// 尝试加锁
if lock.withLockIfAvailable({ state in
    state += 1
}) != nil {
    // 获得锁并执行了
}

// 无关联状态的简单用法
let simpleLock = OSAllocatedUnfairLock()
simpleLock.lock()
defer { simpleLock.unlock() }
// 临界区

// 原理：
// 底层就是 os_unfair_lock，但用引用类型（class）包装
// 避免了 os_unfair_lock 在 Swift 中作为值类型被复制的问题
```

### 9.3 Mutex（Swift 6.0+）

```swift
import Synchronization

// Swift 标准库提供的锁
let mutex = Mutex(42)

mutex.withLock { value in
    value += 1
}

// 特点：
// ~Copyable — 编译期阻止复制（根本原因解决值类型锁的复制问题）
// Sendable — 可安全跨并发域传递
// 底层使用平台原生锁（Darwin 上就是 os_unfair_lock）
```

### 9.4 Actor

```swift
// Actor 不是"锁"，但解决了同样的问题：数据隔离与线程安全

actor BankAccount {
    private var balance: Double = 0

    func deposit(_ amount: Double) {
        balance += amount  // 同一时刻只有一个任务能访问
    }

    func getBalance() -> Double {
        return balance
    }
}

// 外部访问必须用 await

await account.deposit(100)
let b = await account.getBalance()

// Actor vs 锁：
// Actor：编译时检查隔离，异步（await），不会死锁
// 锁：运行时互斥，同步（阻塞），可能死锁
// Actor 适合粗粒度隔离（整个对象的状态）
// 锁适合细粒度保护（某个属性、某段关键代码）
```

## 十、atomic 属性

### 10.1 atomic vs nonatomic

```objc
// atomic（默认）：getter/setter 加锁，保证单次读写的完整性
// nonatomic：不加锁，性能更好

@property (atomic, strong) NSArray *data;    // 读写加锁
@property (nonatomic, strong) NSArray *data; // 不加锁

// atomic 的底层原理：
// 系统维护一个全局的锁数组（按属性地址做哈希映射到某个锁）
// setter 时：加锁 → 赋值 → 解锁
// getter 时：加锁 → 读取并 retain → 解锁 → 返回
```

### 10.2 为什么 atomic 不等于线程安全

```objc
// atomic 只保证单次 getter/setter 不被撕裂
// 不保证复合操作的原子性

@property (atomic, assign) NSInteger count;

// Thread A
self.count = self.count + 1;
// 展开为：
// 1. [self count]       → 读到 5
// 2. 5 + 1 = 6
// 3. [self setCount:6]  → 写入 6
// 步骤 1 和 3 各自原子，但 1→2→3 整体不是原子操作
// Thread B 可能在步骤 1 和 3 之间修改了 count

// 所以 iOS 中一般用 nonatomic + 手动加锁
// 性能更好，安全性由开发者自己保证
```

### 11.1 性能排行

```
无竞争场景（单线程 lock+unlock，耗时从快到慢）：

  os_unfair_lock          ≈ 22 ns   最快，推荐
  dispatch_semaphore      ≈ 28 ns
  pthread_mutex(NORMAL)   ≈ 32 ns

  NSRecursiveLock         ≈ 48 ns
  NSConditionLock         ≈ 60 ns
  @synchronized           ≈ 100 ns  最慢

为什么 @synchronized 最慢：
  不是锁本身慢（底层也是 pthread_mutex）

  dispatch_semaphore 竞争时下降更快
    → 不支持优先级继承
```

## 十二、如何选择锁

### 12.1 决策指南

```
需要线程安全？
  ├─ 只是简单计数器/标志位？
  │    └─ 原子操作（atomic_fetch_add）
  │
  ├─ 读多写少（缓存）？
  │    └─ dispatch_barrier（推荐）/ pthread_rwlock
  │
  ├─ 需要递归加锁？
  │    └─ NSRecursiveLock
  │
  ├─ 需要条件等待/生产消费？
  │    └─ NSCondition
  │
  ├─ 控制并发数量？
  │    └─ dispatch_semaphore
  │
  ├─ Swift 代码？
  │    ├─ iOS 16+ → OSAllocatedUnfairLock
  │    ├─ Swift 6+ → Mutex
  │    ├─ 粗粒度隔离 → Actor
  │    └─ 老版本 → NSLock
  │
  ├─ 性能极致要求？
  │    └─ os_unfair_lock
  │
  └─ 一般场景？
       └─ NSLock（简洁安全）
```

### 12.2 实战代码

```objc

    os_unfair_lock_unlock(&_lock);
}

    os_unfair_lock_lock(&_lock);
    NSArray *copy = [_items copy];
    os_unfair_lock_unlock(&_lock);
    return copy;
}
@end


// ========== 场景2：缓存多读单写 ==========
@implementation Cache {
    NSMutableDictionary *_storage;
    dispatch_queue_t _queue;
}

- (instancetype)init {
    self = [super init];
    _storage = [NSMutableDictionary dictionary];
    _queue = dispatch_queue_create("cache", DISPATCH_QUEUE_CONCURRENT);
    return self;
}

- (id)objectForKey:(NSString *)key {
    __block id result;
    dispatch_sync(_queue, ^{
        result = self->_storage[key];
    });
    return result;
}

- (void)setObject:(id)obj forKey:(NSString *)key {
    dispatch_barrier_async(_queue, ^{
        self->_storage[key] = obj;
    });
}
@end


// ========== 场景3：生产者-消费者 ==========
NSCondition *condition = [[NSCondition alloc] init];
NSMutableArray *buffer = [NSMutableArray array];

// 生产者
dispatch_async(queue, ^{
    id data = [self produceData];
    [condition lock];
    [buffer addObject:data];
    [condition signal];  // 唤醒消费者
    [condition unlock];
});

// 消费者
dispatch_async(queue, ^{
    [condition lock];
    while (buffer.count == 0) {
        [condition wait];  // 等待数据
    }
    id data = buffer.firstObject;
    [buffer removeObjectAtIndex:0];
    [condition unlock];
    [self consumeData:data];
});
```

## 十三、常见面试题

### Q1: 自旋锁和互斥锁的区别？

```
自旋锁：获取失败时 CPU 忙等（while 循环），不休眠。
  优点：没有线程切换开销，临界区极短时性能好。
  缺点：浪费 CPU，长时间等待性能差。

互斥锁：获取失败时线程休眠，让出 CPU。
  优点：等待时不消耗 CPU。
  缺点：有线程上下文切换开销（~微秒级）。

现代锁（os_unfair_lock、pthread_mutex）是混合策略：
先自旋几轮，获取不到再休眠。
```

### Q2: OSSpinLock 为什么被废弃？

```
优先级反转问题。

低优先级线程持有自旋锁 → 高优先级线程请求锁 → 忙等（占 CPU）
→ 低优先级线程得不到 CPU 时间 → 无法释放锁 → 类似活锁。

自旋锁是纯用户态，内核不知道等待关系，无法做优先级继承。

替代：os_unfair_lock（等待时休眠，内核支持优先级继承）。
```

### Q3: @synchronized 的原理？传 nil 会怎样？

```
编译为 objc_sync_enter(obj) / objc_sync_exit(obj)。

底层维护全局哈希表，以 obj 地址为 key 查找 SyncData，
SyncData 内部是一个递归锁（pthread_mutex RECURSIVE）。
查找顺序：线程 TLS 缓存 → 全局哈希表 → 创建新节点。

传 nil：直接返回，不加锁。多线程下会有数据竞争。
```

### Q4: dispatch_semaphore 和 os_unfair_lock 的区别？

```
os_unfair_lock：
  互斥锁语义，记录持有者线程 ID，支持优先级继承，4 字节，最快。

dispatch_semaphore：
  信号量语义，不记录持有者，不支持优先级继承，开销稍大。
  优势：可以控制并发数（初始值 > 1）。

高 QoS 敏感场景应使用 os_unfair_lock。
需要控制并发数量时使用 dispatch_semaphore。
```

### Q5: 如何避免死锁？

```
死锁四条件：互斥、持有等待、非抢占、循环等待。

实践方法：
1. 固定加锁顺序（破坏循环等待）
2. 使用 tryLock + 回退
3. 超时机制（lockBeforeDate:）
4. 减少锁的持有时间
5. 使用 GCD 串行队列或 Actor 替代锁

⚠️ dispatch_sync 死锁：
在串行队列中 sync 到同一个串行队列 → 死锁。
在主线程 sync 到主队列 → 死锁。
```

### Q6: atomic 属性线程安全吗？

```
不完全安全。只保证单次 getter/setter 不被撕裂。

复合操作（self.count = self.count + 1）仍然不安全：
get 和 set 各自原子，但整体不是原子操作。

iOS 中一般用 nonatomic + 手动加锁，性能更好。
```

### Q7: 什么场景用什么锁？

```
简单属性保护      → os_unfair_lock（最快）
递归调用中加锁    → NSRecursiveLock
读多写少（缓存）  → dispatch_barrier
条件等待/生产消费  → NSCondition
代码简洁优先      → @synchronized
控制并发数        → dispatch_semaphore
Swift 现代代码    → Actor / OSAllocatedUnfairLock / Mutex
```


### 6.4 Monitor 锁与运行时实现深度剖析

```
@synchronized 底层并非简单的 pthread_mutex，而是一个 Monitor 结构体：

struct SyncData {
    DISPATCH_ATOMIC sterling          // 引用计数
    pthread_mutex_t     mutex         // 递归锁 (RECURSIVE)
    pthread_cond_t      cond          // 条件变量
    DisguisedPtr<objc_object> object  // 保护的 obj
    pointer_hash_lock   tablelock     // 内部哈希表锁
    struct SyncData     *nextTab      // 哈希桶链表
    struct SyncData     *nextPtr      // per-thread 链表
    thread_id_t         thread        // 当前持有线程
    uintptr_t           lockCount     // 递归深度计数器
    uintptr_t           syncVMCount   // VM-based sync 计数
};

运行时查找链（objc4-986.1）：
1. 当前线程的 TLS → threadDataCache（DisguisedPtr<SyncData>）
   → 命中则直接进入 trylock，O(1) 最快路径
2. 未命中 → threadData->data → 遍历 per-thread SyncData 链表
   → 线性查找，线程本地数据量小
3. 仍未命中 → globalData()->syncDataLock（pointer_hash_lock）
   → 哈希桶查找 64 桶 → 链表遍历
   → 找到则插入 threadDataCache + 链表，未找到则创建新 SyncData
   → 创建后 increment + insert 到哈希桶和线程链表
4. Lock 逻辑：
   → trylock 失败 → 进入线程饥饿检测
   → __sync_thread_hung() 检查是否可能死锁（递归同线程）
   → 递归来自同线程直接返回成功（递归锁语义）
   → 否则 cond_wait 等待 mutex
5. Unlock 逻辑（objc_sync_exit）：
   → lockCount 减 1，>0 则直接返回（递归未耗尽）
   → 减到 0 则 pthread_mutex_unlock + pthread_cond_broadcast
   → 线程退出时自动清理 per-thread 链表

⚠️ Pitfalls:
- 每把 Monitor 创建都要争抢全局哈希表锁 → 大量不同锁对象时成为瓶颈
- Monitor 没有自动释放机制 → obj dealloc 后 SyncData 仍存于全局表
- 递归加锁每次都要遍历 threadData 链表 → 深递归时性能退化

✅ Best practice:
- 生产环境优先用 os_unfair_lock，仅在需要异常安全或递归锁时选 @synchronized
- 避免 @synchronized(obj) 中 obj 是临时创建的对象
- iOS 13+ VM-based sync 优化：对 x86_64/ARM64 使用 Mach semaphore 替代条件变量
```

### OSSpinLock → os_unfair_lock 迁移（优先级反转问题）

OSSpinLock 在 iOS 10 后被标记为不安全，根本原因是**优先级反转**问题：

优先级反转场景：
1. 低优先级线程持有一个 OSSpinLock（自旋锁）
2. 高优先级线程尝试获取同一把锁 → 开始自旋忙等
3. 中优先级线程被调度抢占 → 低优先级线程无法继续运行
4. 结果：高优先级线程自旋等待，中优先级线程反而先执行
5. 高优先级线程被中优先级线程"反超" → 优先级反转

Apple 的修复方案：
- iOS 10 起用 os_unfair_lock 替代 OSSpinLock
- os_unfair_lock 使用 futex 风格的内核同步原语
- 获取失败时线程进入**内核睡眠**而非 CPU 自旋
- 唤醒顺序严格按照内核优先级调度，杜绝优先级反转
- 直接集成到 XNU 内核的调度器中，无用户态忙等开销

迁移方式：

```swift
// ❌ 旧代码 - 不要再用
var lock = OSSpinLock(0)
OSSpinLockLock(&lock)
// ... critical section ...
OSSpinLockUnlock(&lock)

// ✅ 新代码 - iOS 10+
var lock = OS_UNFAIR_LOCK_INIT  // C 结构体
os_unfair_lock_lock(&lock)
// ... critical section ...
os_unfair_lock_unlock(&lock)

// ✅ Swift 推荐封装
import os
class SpinLock {
    private var value = os_unfair_lock()
    func lock()   { os_unfair_lock_lock(&value) }
    func unlock() { os_unfair_lock_unlock(&value) }
    func synchronized<T>(_ block: () throws -> T) rethrows -> T {
        lock(); defer { unlock() }; return try block()
    }
}
```

⚠️ Pitfalls:
- os_unfair_lock 不支持递归调用，递归使用必死锁 → 需改用 NSRecursiveLock
- os_unfair_lock_lock 是阻塞调用，对极短临界区(约50ns)反而比 @synchronized 慢
- Swift 5.5+ 优先使用并发原语 Actor，os_unfair_lock 仅用于 C 互操作
- 迁移时注意：自旋锁改为阻塞锁后，依赖"忙等可被中断"的逻辑行为改变

✅ Best practice:
- Swift 5.5+ 新项目使用 Actor 隔离状态；锁选 OSAllocatedUnfairLock(Swift 5.10+)
- 旧代码迁移三步走：OSSpinLock → os_unfair_lock → 评估升级 Actor/CheckedContinuation
- 临界区内禁止任何阻塞操作（dispatch_sync、网络、文件IO）

### os_unfair_lock vs pthread_mutex 性能对比

os_unfair_lock 和 pthread_mutex 是 iOS 中最常用的两种互斥锁，底层机制差异决定了它们的性能特征：

| 特性 | os_unfair_lock | pthread_mutex |
|------|---------------|---------------|
| 底层原语 | XNU 内核 fair sleep/wakeup | macOS: OSUnfairLock(iOS) / pthread_cond_t |
| 竞争策略 | 不公平 FIFO，先醒先抢 | 公平 FIFO，按等待顺序 |
| 递归支持 | ❌ | ✅ (PTHREAD_MUTEX_RECURSIVE) |
| 超时支持 | ❌ | ✅ (pthread_mutex_timedlock) |
| 平均延迟 | ~30-50ns | ~80-200ns |
| 适用场景 | 短临界区、高吞吐 | 需要递归/超时/条件变量 |

```swift
import os
import pthread

// os_unfair_lock — 极致性能，无额外功能
var uLock = OS_UNFAIR_LOCK_INIT
os_unfair_lock_lock(&uLock)
// ... 超短临界区 ...
os_unfair_lock_unlock(&uLock)

// pthread_mutex — 功能完整，略慢
var mLock = pthread_mutex_t()
pthread_mutex_init(&mLock, nil)
pthread_mutex_lock(&mLock)
// ... 可能需要递归/超时 ...
pthread_mutex_unlock(&mLock)
```

⚠️ Pitfalls:
- os_unfair_lock 在持续高竞争场景(>10个线程争抢同一锁)下可能产生**饥饿**——某些线程永远拿不到锁，此时改用 pthread_mutex(公平模式)
- 频繁竞争时 os_unfair_lock 虽然快但 CPU 利用率高，pthread_mutex 的 FIFO 更节能
- 单线程无竞争时两者差异可忽略(约2x)，选型应优先考虑功能需求而非微基准

✅ Best practice:
- 默认选 os_unfair_lock，仅在需要递归锁/超时/条件变量配对/公平调度时才用 pthread_mutex
- 对锁做基准测试：用 `mach_absolute_time()` 测量 100 万次加解锁的 P50/P99 延迟
- 全局配置锁尽量用 `@MainActor` 或 `actor` 替代，避免无谓的多线程竞争

### pthread_mutex 四种模式详解

pthread_mutex 通过 attr 配置支持四种类型，底层实现差异决定了它们的适用场景和潜在陷阱：

| 类型 | 宏常量 | 行为 | 适用场景 |
|------|--------|------|----------|
| 快速模式(自适应) | PTHREAD_MUTEX_STOMATIC | 低竞争自旋，高竞争阻塞 | 短临界区默认 |
| 普通模式 | PTHREAD_MUTEX_NORMAL | 快速死锁检测 | 通用场景 |
| 自旋模式 | PTHREAD_MUTEX_SPIN | 自旋后阻塞 | CPU繁忙场景 |
| 错误检查 | PTHREAD_MUTEX_ERRORCHECK | 递归返回 EDEADLK | 调试/安全关键 |

```swift
import pthread

var attr = pthread_mutexattr_t()
pthread_mutexattr_init(&attr)

// 设置自旋模式 — 短临界区 + 高竞争场景
pthread_mutexattr_setkind_np(&attr, PTHREAD_MUTEX_SPIN)

var lock = pthread_mutex_t()
pthread_mutex_init(&lock, &attr)
pthread_mutexattr_destroy(&attr)

// 正常加解锁
pthread_mutex_lock(&lock)
// ... 临界区操作 ...
pthread_mutex_unlock(&lock)
```

⚠️ Pitfalls:
- 自旋锁在单核或后台线程被挂起时**永远阻塞**，iOS后台限制下自旋锁风险极高
- ERRORCHECK 模式下重复 lock 返回 EDEADLK 而非死锁挂起，但性能开销比普通模式高约 3x
- `pthread_mutexattr_setkind_np` 是 Apple 非标准扩展，macOS 13+ 已标记为 deprecated

✅ Best practice:
- iOS 开发中 99% 场景使用默认 NORMAL 模式即可，无需显式配置 attr
- 优先使用 os_unfair_lock 替代 pthread_mutex，仅在有条件变量需求时使用 mutex + cond
- 测试死锁场景时临时切换 ERRORCHECK 模式定位问题，发布前切回 NORMAL

### 5.5 NSLock 家族层级关系总结

NSLock 家族的四个类本质上是 **pthread_mutex** 上的不同封装层，从功能复杂度递增排列：

```
NSLock (基础互斥) ──┬── NSRecursiveLock ( + 递归计数)
                     ├── NSCondition ( + 条件变量 wait/signal)
                     └── NSConditionLock ( + 整数条件值)
```

| 层级 | 类 | 底层实现 | 新增能力 |
|------|----|---------|----------|
| L1 | NSLock | `pthread_mutex` (NORMAL) | 基础 lock/unlock |
| L2 | NSRecursiveLock | `pthread_mutex` (RECURSIVE) | + 同一线程可重入 |
| L3 | NSCondition | `pthread_mutex` + `pthread_cond` | + wait/signal/broadcast |
| L4 | NSConditionLock | `pthread_cond` + 整数状态机 | + 条件值校验 |

```swift
// 选型决策树：
// 1. 是否需要递归调用？
//    ✅ → NSRecursiveLock
//    ❌ ↓ 是否需要等待条件？
//      ✅ → NSCondition (通用条件) / NSConditionLock (整数状态机)
//      ❌ → NSLock (最轻量)
```

⚠️ Pitfalls: NSCondition 的 `wait()` 会**自动释放锁**并在被唤醒后**自动重锁**，中间状态对其他线程可见 — 这是它与普通 lock/unlock 的根本区别，但也是死锁源头。

✅ Best practice: 能不用条件变量就不用，优先用 `async/await` + `actor` 或 GCD 信号量替代 NSCondition/NSConditionLock，只有在维护老 Objective-C 代码库时才选它们。

### 5.6 dispatch_semaphore 作为锁的替代方案

`dispatch_semaphore` 是 GCD 提供的信号量原语，通过 `signal/wait` 控制并发数量。信号量值为 1 时等价于互斥锁，值为 N 时限制最多 N 个并发执行者。

```swift
import Dispatch

let sem = DispatchSemaphore(value: 1)  // 初始值=1 → 互斥锁效果

// 线程 A
sem.wait()          // value: 1→0，获取信号量
// ... 临界区操作 ...
sem.signal()        // value: 0→1，释放信号量

// 线程 B（wait 阻塞直到线程 A 调用 signal）
sem.wait(timeout: .now() + .seconds(5))  // 带超时的 wait 避免死锁
// ... 临界区 ...
sem.signal()
```

信号量限流器模式（最多3个并发 worker）：

```swift
let workerSem = DispatchSemaphore(value: 3)

for task in 0..<100 {
    workerSem.wait()  // 超过3个时阻塞
    DispatchQueue.global().async {
        defer { workerSem.signal() }  // 任务完成释放
        doWork(task)
    }
}
```

⚠️ Pitfalls: `sem.wait()` 是**同步阻塞调用**，主线程上调用导致 UI 冻结；无超时参数的 wait 在 signal 未调用时永久死锁。

✅ Best practice: 信号量适合"限制并发数"场景（如网络请求池），而非普通互斥保护 — 后者优先用 `actor` 或 `os_unfair_lock`；始终使用 `wait(timeout:)` 避免死锁。


### 5.7 Swift Actor 模型作为无锁并发方案

Swift Actor 是 Swift Concurrency 提供的结构化并发原语。Actor 通过编译期的数据竞争检测，自动在 actor 边界插入异步序列化的保护屏障，实现"无锁"线程安全。与 OSAllocatedLock/os_unfair_lock 等传统锁相比，Actor 消除了手动加锁/解锁带来的死锁风险。

```swift
actor ScoreManager {
    private var scores: [String: Int] = [:]

    func updateScore(_ score: Int, for player: String) {
        scores[player] = score  // 自动序列化，无需手动 lock
    }

    func getScore(for player: String) -> Int? {
        return scores[player]  // 读取同样被 actor 隔离
    }
}

// 使用：所有跨 actor 边界的访问必须用 await
let manager = ScoreManager()
await manager.updateScore(100, for: "Alice")
let score = await manager.getScore(for: "Alice")
```

⚠️ Pitfalls: Actor 本质是异步互斥锁，所有方法在内部串行执行 — 高并发场景下可能成为瓶颈；`nonisolated` 方法绕过 actor 隔离但需手动保证线程安全；Actor 不能直接被 Objective-C 代码调用。

✅ Best practice: 优先用 Actor 替代 NSLock/os_unfair_lock 管理共享状态；内部用 `nonisolated` 标记纯计算函数避免不必要的序列化开销；与 Objective-C 互操作时用 `@MainActor` 桥接。

### 5.8 原子操作（OSAtomicCompareAndSwap、atomic_bool）

原子操作是在单条 CPU 指令级别完成的不可分割操作，无需加锁即可保证多线程环境下的数据安全。iOS 提供 C11 `atomic_bool`、`atomic_int` 等类型，以及已废弃但仍可用的 `OSAtomicCompareAndSwap` 系列函数。核心原语 CAS（Compare-And-Swap）是构建无锁数据结构和自旋锁的基石。

```swift
import Darwin

// C11 原子类型（推荐）
@unchecked Sendable
class AtomicInt {
    var value: Int {
        get { Int(OSAtomicLoad32(&storage)) }
        set { OSAtomicStore32(Int32(newValue), &storage) }
    }
    private var storage: Int32 = 0
    
    func compareAndSwap(expected: Int32, newValue: Int32) -> Bool {
        OSAtomicCompareAndSwap(expected, newValue, &storage)
    }
}

// C11 原生 atomic_bool
var isReady: AtomicBool = false
atomic_store(&isReady, true)
if atomic_load(&isReady) { /* ... */ }
```

⚠️ Pitfalls: CAS 存在 ABA 问题 — 线程 A 读取值 X，线程 B 将 X→Y→X，线程 A 的 CAS 成功但语义已错乱，需用版本号或 `atomic_compare_exchange_weak` 的强语义缓解；`OSAtomic*` 系列已在 iOS 11+ 废弃，新代码统一使用 C11 `<stdatomic.h>`。

✅ Best practice: 简单标志位用 `AtomicBool`/`AtomicInt` 替代 `os_unfair_lock` 实现零开销同步；复杂无锁结构用 `atomic_compare_exchange_weak` 配合 monotonic 内存序 + 自旋重试；iOS 11+ 新项目全部迁移至 `stdatomic` 或 Swift `actor`。

#### 读写锁进阶：pthread_rwlock 与无锁读写方案

标准 `pthread_rwlock` 在 Apple 平台默认采用写者优先策略，读多写少场景性能最优。其内部实现维护了 `__rwlock_reader_count`（活跃读者数）、`__rwlock_writer_count`（活跃写者标记）和等待队列。当无写者时，多个读线程直接自增计数进入临界区；写者需等待队列排空。

基于 `OSSpinLock`（已废弃）的手动实现（仅供理解原理）：

```c
// ⚠️ 仅用于理解，生产环境用 os_unfair_lock
// 读写锁基于 CAS 的简化实现（读者计数 + 写者独占）
typedef struct {
    volatile int32_t readers;  // CAS 保护的读者计数
    volatile int32_t writer;   // 0=无写者, 1=写者持有
    OSSpinLock lock;           // 保护状态变更的自旋锁
} SimpleRWLock;

void simple_rdlock(SimpleRWLock *rw) {
    OSSpinLockLock(&rw->lock);
    while (rw->writer) { /* 自旋等待写者退出 */ }
    rw->readers++;
    OSSpinLockUnlock(&rw->lock);
}

void simple_wrlock(SimpleRWLock *rw) {
    OSSpinLockLock(&rw->lock);
    while (rw->readers > 0 || rw->writer) { /* 等所有读者和写者 */ }
    rw->writer = 1;
    OSSpinLockUnlock(&rw->lock);
}

void simple_unlock(SimpleRWLock *rw) {
    OSSpinLockLock(&rw->lock);
    if (rw->writer) rw->writer = 0;        // 写者退出
    else rw->readers--;                     // 读者退出
    OSSpinLockUnlock(&rw->lock);
}
```

真正的 `pthread_rwlock` 内核实现远比上述复杂：使用 `ulock_wait` 系统调用挂起线程而非忙等，集成 NUMA 感知的读者分组（per-group reader optimization），在 iOS 上还会根据运行态动态切换优先级策略。

⚠️ Pitfalls: 写者优先导致读者饥饿 — 持续写入时新读者无限排队；pthread_rwlock 不可递归使用，同一线程重复加 rdlock 会死锁；基于自旋锁的手动实现在 iOS 上会触发 watchdog（忙等超 15ms），必须用内核态挂起方案。

✅ Best practice: 读多写少选 `pthread_rwlock`，读写均衡用 `os_unfair_lock`，纯 Swift 项目优先 `actor` + `@Sendable`；需要读并发 + 写屏障时，GCD 的 `barrierAsync` 队列是最高效的 Swift 原生方案。


#### 无锁数据结构（Lock-free Stack/Queue 与 ABA 问题）

无锁数据结构利用原子操作（CAS）代替互斥锁实现线程安全集合。核心思想是：线程间从不阻塞等待锁，而是通过 CAS 循环重试（backoff）完成操作。典型实现包括 Treiber 无锁栈（CAS 更新 top 指针）和 Michael-Scott 无锁队列（双指针 + CAS 并发入队）。


**ABA 问题**：线程 A CAS 检查 top=X，线程 B 弹出 X→Y→再弹 Y→恢复 X，线程 A 的 CAS 成功但链表内部已断裂。这是无锁结构最大的陷阱。


```swift
// ABA 解决方案：带版本号的 tagged pointer（双宽 CAS）
struct TaggedNode<T> {
    var tag: UInt64          // 版本号，每次操作递增
    var next: UnsafeMutablePointer<Node<T>>?
    var value: T
}

// 无锁栈 pop（tag 随 CAS 变化，ABA 不可能）
@_silgen_name("__compare_and_swap")
func cmpxchg<T>(ptr: UnsafeMutablePointer<T>,
                expected: UnsafePointer<T>,
                desired: UnsafePointer<T>) -> Bool

// 实际生产使用 OSAtomicCompareAndSwapBarrier 或 stdatomic
// CAS + 递增 tag → ABA 被彻底阻止
```


⚠️ Pitfalls: 普通 CAS 无法检测 ABA，必须用 tagged pointer 或 hazard pointer 保证安全释放；ARC 环境下手动管理 `UnsafeMutablePointer` 生命周期极易崩溃，`CFRetain`/`CFRelease` 或 hazard pointers 是唯一可靠方案。


✅ Best practice: 优先使用 `os_unfair_lock` 保护集合而非自造无锁结构；必须无锁时，Swift `actor` + `@Sendable` 在绝大多数场景已足够；Michael-Scott 队列只在内核/极致性能场景使用，普通 App 用 GCD 并发队列 + `barrier` 更简单安全。


#### 死锁检测与预防策略

死锁发生的四个必要条件是**互斥、持有并等待、非抢占、循环等待**。iOS 运行时本身不提供死锁检测机制，因此预防策略完全依赖开发者设计。最有效的预防方式是建立锁的**全局偏序关系**（lock ordering invariant）：所有线程按固定顺序获取锁，从而破坏循环等待条件。

```swift
// 方案1：超时获取锁（pthread_mutex_timedlock）
let ts = timespec(tv_sec: Int(timeIntervalSinceNow + 2), tv_nsec: 0)
let rc = pthread_mutex_timedlock(&mutex, UnsafePointer<timespec>.init(
    UnsafeMutablePointer<timespec>.allocate(capacity: 1)))
// rc == ETIMEDOUT 时立即退出并上报，避免无限阻塞

// 方案2：锁层次检查器（Debug 模式运行时验证）
class LockHierarchyChecker {
    private var acquiredLevels: Set<Int> = []
    func acquire<T>(_ lock: HierarchicalLock, level: Int, _ closure: () -> T) -> T {
        assert(!acquiredLevels.contains(where: { $0 >= level }), "违反锁顺序 → 死锁风险")
        acquiredLevels.insert(level)
        lock.lock(); defer { lock.unlock(); acquiredLevels.remove(level) }
        return closure()
    }
}
```

⚠️ Pitfalls: `pthread_mutex_timedlock` 在 Release 模式下仅能检测、不能自动恢复；`dispatch_semaphore_wait` 无超时回调，永远阻塞时进程直接 HANG。

✅ Best practice: Debug 环境启用锁顺序检查器 + 超时告警；生产环境关键路径用 `DISPATCH_TIMEOUT` 包装 semaphore；长任务优先 async/await 替代锁，从根本上消除死锁可能。

#### Instruments OS Trace（锁竞争分析）

Xcode Instruments 的 **OS Trace** 模板可追踪 `os_unfair_lock`、`pthread_mutex`、`os_lock` 等内核锁的完整生命周期，精确到微秒级 contention 窗口。配合 **Lock Activity** 视图，可视化展示锁持有者、等待者队列、以及锁竞争热点函数调用栈。

```swift
// 在代码中埋点（无需额外 instrumentation）
import Darwin

// os_unfair_lock 自动被 Instruments 追踪
private var resourceLock = OS_UNFAIR_LOCK_INIT

// 方案：os_signpost 标记临界区边界（配合 Energy Log / Time Profiler）
private let log = OSSignpostLog.domain("com.myapp.resources")
func loadResource() -> Data {
    let signpost = OSSignpostLogger(log: log)
    let id = signpost.makeID()
    let interval = signpost.beginInterval("Lock Contention", id: id)
    os_unfair_lock_lock(&resourceLock)
    defer { os_unfair_lock_unlock(&resourceLock) }
    // ... 临界区操作 ...
    signpost.end("Lock Contention", id: id)
    return cachedData
}
```

⚠️ Pitfalls: OS Trace 只在 `os_unfair_lock` 和 `pthread_mutex`（需 `PTHREAD_MUTEX_POLICY` 属性）上有效；`DispatchSemaphore` 和 `NSLock`（底层 pthread）不直接暴露 contention 数据；Release 构建需开启 `ENABLE_USER_SCRIPT_SANDBOXING=NO` 否则 Instruments 无法附加。

✅ Best practice: 性能测试阶段全程开启 OS Trace recording，用 **Heatmap** 视图定位 Top 3 锁竞争热点；配合 `os_log()` 输出临界区上下文，在 Xcode Debug Navigator 的 System Trace 中交叉验证。
