# 线上性能监控体系详解

## 一、监控体系设计原则

### 1.1 为什么需要线上监控

```
开发测试 vs 线上的差距：

  开发测试环境：
    - 高端设备（最新 iPhone）
    - 良好网络（WiFi/5G）
    - 少量数据（测试数据）
    - 少量并发（1 个用户）

  线上真实环境：
    - 各种设备（包括 iPhone 6s 等老设备）
    - 各种网络（2G/3G/弱 4G/地铁隧道）
    - 海量数据（真实用户数据）
    - 海量并发（百万用户同时使用）

  结论：只有线上监控才能发现真实问题

监控体系的目标：
  1. 快速发现问题（分钟级告警）
  2. 精确定位问题（有调用栈、有上下文）
  3. 量化问题影响（影响多少用户、多少比例）
  4. 验证优化效果（对比优化前后的指标）
  5. 建立质量基线（什么是"正常"，什么是"异常"）
```

## 二、崩溃监控

### 2.1 崩溃类型与捕获

```swift
// iOS 崩溃的三种类型

// 类型1：ObjC 异常（NSException）
// 最常见，如数组越界、字典 nil 值、unrecognized selector
NSSetUncaughtExceptionHandler { exception in
    // 捕获未处理的 ObjC 异常
    let info = CrashInfo(
        type: "ObjCException",
        name: exception.name.rawValue,
        reason: exception.reason ?? "",
        callStack: exception.callStackSymbols,
        callStackReturnAddresses: exception.callStackReturnAddresses
    )
    CrashReporter.save(info)
    CrashReporter.flush()  // 立即写入磁盘！
}

// 类型2：Signal 信号（来自内核或进程）
// SIGSEGV（空指针/野指针）、SIGABRT（abort 调用）、SIGBUS（内存对齐）
import Darwin

func setupSignalHandlers() {
    let signals: [Int32] = [SIGABRT, SIGILL, SIGSEGV, SIGFPE, SIGBUS, SIGTRAP]
    signals.forEach { sig in
        signal(sig) { signo in
            let callStack = Thread.callStackSymbols
            let info = CrashInfo(
                type: "Signal",
                name: "Signal \(signo)",
                reason: signalDescription(signo),
                callStack: callStack,
                callStackReturnAddresses: []
            )
            CrashReporter.save(info)
            CrashReporter.flush()
            
            // 重置为默认处理，让系统生成 Crash 日志
            signal(signo, SIG_DFL)
            raise(signo)
        }
    }
}

// 类型3：Swift 运行时错误
// force unwrap nil、数组越界（Swift）、类型转换失败
// Swift 的运行时错误最终也会触发 SIGABRT，被上面的 Signal 处理器捕获

// Watchdog 超时（0x8badf00d）
// 主线程在 launch/suspend/resume/terminate 时卡住超过 20s
// 系统强杀，生成 JetsamEvent 日志，不触发 Signal
// 只能通过 MetricKit 或 Xcode Organizer 查看
```

```swift
}

// 记录面包屑（在关键操作时调用）
class BreadcrumbRecorder {

    private let maxCount = 50
    
    func record(type: String, message: String, data: [String: String]? = nil) {
        let crumb = Breadcrumb(
            type: type,
            message: message,
            timestamp: Date().timeIntervalSince1970,
            data: data
        )
        breadcrumbs.append(crumb)
        if breadcrumbs.count > maxCount {
            breadcrumbs.removeFirst()
        }
    }
    
    func getAll() -> [Breadcrumb] { return breadcrumbs }
}

// 在关键位置记录面包屑
```

### 2.3 崩溃率计算与告警

```
崩溃率的正确计算方式：

  错误方式：崩溃次数 / 启动次数
  问题：同一用户多次崩溃被重复计算

  正确方式：
    崩溃用户数 / 活跃用户数（UV 崩溃率）
    
  更精细的分析：
    按版本：找出哪个版本引入了 Bug
    按设备：找出是否特定设备型号有问题
    按系统版本：找出是否特定 iOS 版本有问题
    按用户属性：找出是否特定用户群体有问题

告警策略：
  P0（立即告警，人工处理）：
    - 崩溃率 > 0.1%（任意 5 分钟窗口）
    - 新版本上线后崩溃率较上版本上涨 > 50%
    - 单个崩溃问题影响用户数 > 1000
  
  P1（1 小时内处理）：
    - 崩溃率 > 0.05%
    - 崩溃率持续上升趋势
  
  P2（工作时间内处理）：
    - 长尾崩溃问题
    - 崩溃率 < 0.01% 的低频崩溃

聚合策略：
  相同调用栈的崩溃聚合为一个 Issue
  聚合 Key = 去掉地址后的调用栈 hash
  避免同一问题产生大量重复告警
```

## 三、OOM 监控

### 3.1 OOM 的识别

```swift
// OOM（Out of Memory）：进程因内存不足被系统杀死
// 特点：没有 Crash 日志，只有 JetsamEvent 日志

// OOM 识别（排除法）：
// App 异常退出但不是以下情况 → 可能是 OOM
//   1. 用户主动退出（applicationWillTerminate）
//   2. 系统崩溃（有 Crash 日志）
//   3. Watchdog 超时（有 JetsamEvent，exitReason 包含 "watchdog"）
//   4. 系统升级重启
//   5. 后台被系统挂起

class OOMDetector {
    
    private static let kLastRunKey = "oom_last_run_state"
    private static let kVersionKey = "oom_app_version"
    
    enum ExitReason: String {
        case normal         // 正常退出
        case crash          // 已知崩溃
        case oom            // OOM
        case unknown        // 未知
    }
    
    static func setup() {
        // 检查上次是否是 OOM
        let lastState = UserDefaults.standard.string(forKey: kLastRunKey)
        let lastVersion = UserDefaults.standard.string(forKey: kVersionKey)
        let currentVersion = Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String
        
        if lastState == "running" && lastVersion == currentVersion {
            // 上次运行时 App 处于运行状态，且版本相同
            // 没有正常退出 → 可能是 OOM
            reportPotentialOOM()
        }
        
        // 标记当前运行状态
        UserDefaults.standard.set("running", forKey: kLastRunKey)
        UserDefaults.standard.set(currentVersion, forKey: kVersionKey)
        
        // 正常退出时清除标记
        NotificationCenter.default.addObserver(
            forName: UIApplication.willTerminateNotification,
            object: nil, queue: .main
        ) { _ in
            UserDefaults.standard.set("terminated", forKey: kLastRunKey)
        }
    }
    
    private static func reportPotentialOOM() {
        // 上报 OOM 事件
        Analytics.track(event: "potential_oom", properties: [
            "device": UIDevice.current.model,
            "os": UIDevice.current.systemVersion,
            "app_version": Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String ?? ""
        ])
    }
}
```

### 3.2 内存水位监控

```swift
// 实时监控内存使用，在接近 OOM 前主动释放

class MemoryMonitor {
    
    // 内存压力等级
    enum MemoryLevel {
        case normal    // < 60% 设备内存
        case warning   // 60-80% 设备内存
        case critical  // > 80% 设备内存
    }
    
    private var timer: DispatchSourceTimer?
    var onMemoryLevelChange: ((MemoryLevel, UInt64) -> Void)?
    
    func start() {
        timer = DispatchSource.makeTimerSource(queue: .global(qos: .utility))
        timer?.schedule(deadline: .now(), repeating: .seconds(5))
        timer?.setEventHandler { [weak self] in
            self?.checkMemory()
        }
        timer?.resume()
    }
    
    private func checkMemory() {
        let usage = currentMemoryUsage()
        let total = totalDeviceMemory()
        let ratio = Double(usage) / Double(total)
        
        let level: MemoryLevel
        if ratio < 0.6 { level = .normal }
        else if ratio < 0.8 { level = .warning }
        else { level = .critical }
        
        DispatchQueue.main.async { [weak self] in
            self?.onMemoryLevelChange?(level, usage)
        }
        
        // 上报内存使用情况（采样上报，不是每次都报）
        if Int.random(in: 0..<100) < 5 {  // 5% 采样率
            PerformanceReporter.reportMemory(usage: usage, total: total)
        }
    }
    
    // 获取当前进程内存使用
    func currentMemoryUsage() -> UInt64 {
        var taskInfo = task_vm_info_data_t()
        var count = mach_msg_type_number_t(MemoryLayout<task_vm_info_data_t>.size / MemoryLayout<integer_t>.size)
        
        let result = withUnsafeMutablePointer(to: &taskInfo) {
            $0.withMemoryRebound(to: integer_t.self, capacity: Int(count)) {
                task_info(mach_task_self_, task_flavor_t(TASK_VM_INFO), $0, &count)
            }
        }
        
        if result == KERN_SUCCESS {
            return taskInfo.phys_footprint  // 物理内存占用（更准确）
        }
        return 0
    }
    
    // 获取设备总内存
    func totalDeviceMemory() -> UInt64 {
        return ProcessInfo.processInfo.physicalMemory
    }
}

// 内存警告时的分级处理
class MemoryWarningHandler {
    
    static func handle(level: MemoryMonitor.MemoryLevel) {
        switch level {
        case .normal:
            break
            
        case .warning:
            // 释放可重建的缓存
            URLCache.shared.removeAllCachedResponses()
            ImageCache.shared.clearMemoryCache()
            // 通知业务层释放非必要数据
            NotificationCenter.default.post(name: .memoryWarning, object: nil)
            
        case .critical:
            // 更激进的释放
            ImageCache.shared.clearAllCache()
            DatabaseCache.shared.clearAll()
            // 通知 VC 释放不可见的视图
            NotificationCenter.default.post(name: .memoryCritical, object: nil)
            // 上报 OOM 风险
            PerformanceReporter.reportOOMRisk(
                memoryUsage: MemoryMonitor().currentMemoryUsage(),
                threshold: UInt64(Double(MemoryMonitor().totalDeviceMemory()) * 0.8)
            )
        }
    }
}
```

## 四、网络监控

### 4.1 URLSession 请求监控

```swift
// 通过 URLProtocol 拦截所有网络请求

class NetworkMonitorProtocol: URLProtocol {
    
    private var dataTask: URLSessionDataTask?
    private var startTime: CFTimeInterval = 0
    private var metrics: NetworkMetrics?
    
    override class func canInit(with request: URLRequest) -> Bool {
        // 避免递归拦截
        if URLProtocol.property(forKey: "NetworkMonitored", in: request) != nil {
            return false
        }
        return true
    }
    
    override class func canonicalRequest(for request: URLRequest) -> URLRequest {
        return request
    }
    
    override func startLoading() {
        startTime = CACurrentMediaTime()
        
        var mutableRequest = request
        URLProtocol.setProperty(true, forKey: "NetworkMonitored", in: &mutableRequest)
        
        let session = URLSession(configuration: .default)
        dataTask = session.dataTask(with: mutableRequest) { [weak self] data, response, error in
            guard let self = self else { return }
            
            let duration = CACurrentMediaTime() - self.startTime
            
            if let error = error {
                self.client?.urlProtocol(self, didFailWithError: error)
                self.reportRequest(response: nil, error: error, duration: duration, dataSize: 0)
            } else if let response = response {
                self.client?.urlProtocol(self, didReceive: response,
                                         cacheStoragePolicy: .notAllowed)
                if let data = data {
                    self.client?.urlProtocol(self, didLoad: data)
                    self.reportRequest(response: response as? HTTPURLResponse,
                                       error: nil, duration: duration, dataSize: data.count)
                }
                self.client?.urlProtocolDidFinishLoading(self)
            }
        }
        dataTask?.resume()
    }
    
    override func stopLoading() {
        dataTask?.cancel()
    }
    
    private func reportRequest(response: HTTPURLResponse?, error: Error?,
                               duration: TimeInterval, dataSize: Int) {
        let metrics = NetworkRequestMetrics(
            url: request.url?.absoluteString ?? "",
            method: request.httpMethod ?? "GET",
            statusCode: response?.statusCode ?? 0,
            duration: duration,
            dataSize: dataSize,
            error: error?.localizedDescription,
            timestamp: Date().timeIntervalSince1970
        )
        
        NetworkMonitor.shared.record(metrics)
    }
}

// 注册 URLProtocol
URLSessionConfiguration.default.protocolClasses?.insert(NetworkMonitorProtocol.self, at: 0)

// 更精确的方案：使用 URLSessionTaskDelegate 的 didFinishCollecting metrics
extension NetworkManager: URLSessionTaskDelegate {
    
    func urlSession(_ session: URLSession, task: URLSessionTask,
                    didFinishCollecting metrics: URLSessionTaskMetrics) {
        
        for transactionMetrics in metrics.transactionMetrics {
            let report = NetworkTransactionReport(
                // DNS 解析时间
                dnsTime: transactionMetrics.domainLookupEndDate?.timeIntervalSince(
                    transactionMetrics.domainLookupStartDate ?? Date()) ?? 0,
                // TCP 建连时间
                connectTime: transactionMetrics.connectEndDate?.timeIntervalSince(
                    transactionMetrics.connectStartDate ?? Date()) ?? 0,
                // TLS 握手时间
                tlsTime: transactionMetrics.secureConnectionEndDate?.timeIntervalSince(
                    transactionMetrics.secureConnectionStartDate ?? Date()) ?? 0,
                // 首字节时间（TTFB）
                ttfb: transactionMetrics.responseStartDate?.timeIntervalSince(
                    transactionMetrics.requestEndDate ?? Date()) ?? 0,
                // 总耗时
                totalTime: transactionMetrics.responseEndDate?.timeIntervalSince(
                    transactionMetrics.fetchStartDate ?? Date()) ?? 0,
                // 是否使用了缓存
                isFromCache: transactionMetrics.resourceFetchType == .localCache,
                // 协议版本
                protocol: transactionMetrics.networkProtocolName ?? "unknown"
            )
            
            PerformanceReporter.reportNetworkTransaction(report)
        }
    }
}
```

### 4.2 HTTPDNS 完整实现

```swift
// HTTPDNS：通过 HTTP 查询 DNS，绕过运营商 Local DNS 的劫持

class HTTPDNSManager {
    static let shared = HTTPDNSManager()
    
    // DNS 缓存
    private var cache: [String: DNSRecord] = [:]
    private let lock = NSLock()
    
    struct DNSRecord {
        let ips: [String]
        let ttl: TimeInterval
        let timestamp: TimeInterval
        
        var isExpired: Bool {
            return Date().timeIntervalSince1970 - timestamp > ttl
        }
    }
    
    // 预解析常用域名（App 启动时调用）
    func prefetch(domains: [String]) {
        for domain in domains {
            resolve(domain: domain) { _ in }
        }
    }
    
    // 解析域名
    func resolve(domain: String, completion: @escaping ([String]?) -> Void) {
        // 1. 检查内存缓存
        lock.lock()
        if let record = cache[domain], !record.isExpired {
            lock.unlock()
            completion(record.ips)
            return
        }
        lock.unlock()
        
        // 2. 向 HTTPDNS 服务器查询
        let url = URL(string: "https://httpdns.example.com/d?host=\(domain)&ttl=1")!
        URLSession.shared.dataTask(with: url) { [weak self] data, _, _ in
            guard let data = data,
                  let response = try? JSONDecoder().decode(HTTPDNSResponse.self, from: data) else {
                completion(nil)
                return
            }
            
            // 3. 缓存结果
            let record = DNSRecord(
                ips: response.ips,
                ttl: TimeInterval(response.ttl),
                timestamp: Date().timeIntervalSince1970
            )
            self?.lock.lock()
            self?.cache[domain] = record
            self?.lock.unlock()
            
            completion(response.ips)
        }.resume()
    }
    
    // 将请求的域名替换为 IP（IP 直连）
    func buildRequest(originalURL: URL) -> URLRequest? {
        guard let host = originalURL.host else { return nil }
        
        // 同步获取 IP（应提前预解析，此处直接取缓存）
        lock.lock()
        let record = cache[host]
        lock.unlock()
        
        guard let ip = record?.ips.first, !record!.isExpired else {
            return URLRequest(url: originalURL)  // 降级到正常 DNS
        }
        
        // 将 URL 中的域名替换为 IP
        var components = URLComponents(url: originalURL, resolvingAgainstBaseURL: false)!
        components.host = ip
        
        guard let ipURL = components.url else { return nil }
        var request = URLRequest(url: ipURL)
        
        // 设置 Host 头（HTTPS SNI 需要）
        request.setValue(host, forHTTPHeaderField: "Host")
        
        return request
    }
}

struct HTTPDNSResponse: Codable {
    let host: String
    let ips: [String]
    let ttl: Int
}

// 处理 HTTPS + IP 直连的 SNI 问题
// 当使用 IP 直连时，TLS 握手中的 SNI 会是 IP 而非域名
// 服务器无法选择正确的证书 → 握手失败
// 解决：自定义 URLSessionDelegate，在 TLS 握手时提供正确的域名

class HTTPDNSSessionDelegate: NSObject, URLSessionDelegate {
    
    private let originalHost: String
    
    init(originalHost: String) {
        self.originalHost = originalHost
    }
    
    func urlSession(_ session: URLSession,
                    didReceive challenge: URLAuthenticationChallenge,
                    completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) {
        
        guard challenge.protectionSpace.authenticationMethod == NSURLAuthenticationMethodServerTrust,
              let serverTrust = challenge.protectionSpace.serverTrust else {
            completionHandler(.performDefaultHandling, nil)
            return
        }
        
        // 用原始域名验证证书（而非 IP）
        SecTrustSetPolicies(serverTrust, SecPolicyCreateSSL(true, originalHost as CFString))
        
        var error: CFError?
        if SecTrustEvaluateWithError(serverTrust, &error) {
            completionHandler(.useCredential, URLCredential(trust: serverTrust))
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}
```

## 五、卡顿与 ANR 监控

### 5.1 ANR（Application Not Responding）监控

```swift
// ANR：主线程在一定时间内无响应
// iOS 中通常定义为主线程卡顿超过 5s（Watchdog 阈值是 20s）

class ANRMonitor {
    
    private let anrThreshold: TimeInterval = 5.0  // 5 秒认为 ANR
    private var lastMainThreadActivityTime: TimeInterval = 0
    private var isMonitoring = false
    
    func start() {
        isMonitoring = true
        
        // 在主线程定期更新活跃时间
        let mainTimer = CADisplayLink(target: self, selector: #selector(mainThreadTick))
        mainTimer.add(to: .main, forMode: .common)
        
        // 在子线程检测
        DispatchQueue.global(qos: .utility).async { [weak self] in
            self?.monitorLoop()
        }
    }
    
    @objc private func mainThreadTick() {
        lastMainThreadActivityTime = CACurrentMediaTime()
    }
    
    private func monitorLoop() {
        while isMonitoring {
            Thread.sleep(forTimeInterval: 1.0)
            
            let now = CACurrentMediaTime()
            let elapsed = now - lastMainThreadActivityTime
            
            if elapsed > anrThreshold {
                // 主线程超过 5s 没有活动
                reportANR(duration: elapsed)
            }
        }
    }
    
    private func reportANR(duration: TimeInterval) {
        // 抓取主线程调用栈
        let callStack = captureMainThreadCallStack()
        
        PerformanceReporter.reportANR(
            duration: duration,

    }
```

## 六、性能数据上报

### 6.1 上报策略设计

```swift
// 性能数据上报的核心挑战：
// 1. 不能影响 App 性能（上报本身不能卡顿）
// 2. 弱网/无网时不丢失数据
// 3. 控制流量（采样上报）
// 4. 数据聚合（减少请求数量）

class PerformanceReporter {
    static let shared = PerformanceReporter()
    
    private var buffer: [PerformanceEvent] = []
    private let bufferLock = NSLock()
    private let maxBufferSize = 100
    private let flushInterval: TimeInterval = 30  // 30 秒批量上报一次
    private var flushTimer: Timer?
    
    // 采样率配置（从服务端动态下发）
    var samplingRates: [String: Double] = [
        "launch_time": 1.0,    // 启动时间 100% 上报
        "crash": 1.0,          // 崩溃 100% 上报
        "anr": 1.0,            // ANR 100% 上报
        "jank": 0.1,           // 卡顿 10% 采样
        "memory": 0.05,        // 内存 5% 采样
        "network": 0.1,        // 网络 10% 采样
    ]
    
    func report(_ event: PerformanceEvent) {
        // 采样
        let rate = samplingRates[event.type] ?? 0.1
        guard Double.random(in: 0...1) < rate else { return }
        
        bufferLock.lock()
        buffer.append(event)
        let shouldFlush = buffer.count >= maxBufferSize
        bufferLock.unlock()
        
        if shouldFlush {
            flush()
        }
    }
    
    func flush() {
        bufferLock.lock()
        guard !buffer.isEmpty else {
            bufferLock.unlock()
            return
        }
        let events = buffer
        buffer.removeAll()
        bufferLock.unlock()
        
        // 序列化并上报
        sendEvents(events)
    }
    
    private func sendEvents(_ events: [PerformanceEvent]) {
        // 后台线程上报，不阻塞主线程
        DispatchQueue.global(qos: .background).async {
            guard let data = try? JSONEncoder().encode(events) else { return }
            
            var request = URLRequest(url: URL(string: "https://perf.example.com/report")!)
            request.httpMethod = "POST"
            request.httpBody = data
            request.setValue("application/json", forHTTPHeaderField: "Content-Type")
            
            // 使用后台 URLSession（App 进入后台后仍可完成上报）
            let config = URLSessionConfiguration.background(withIdentifier: "perf.report")
            let session = URLSession(configuration: config)
            session.dataTask(with: request).resume()
        }
    }
    
    // 本地持久化（弱网时不丢失数据）
    private func persistEvents(_ events: [PerformanceEvent]) {
        let path = NSTemporaryDirectory() + "perf_events.json"
        if let data = try? JSONEncoder().encode(events) {
            try? data.write(to: URL(fileURLWithPath: path))
        }
    }
    
    private func loadPersistedEvents() -> [PerformanceEvent] {
        let path = NSTemporaryDirectory() + "perf_events.json"
        guard let data = try? Data(contentsOf: URL(fileURLWithPath: path)),
              let events = try? JSONDecoder().decode([PerformanceEvent].self, from: data) else {
            return []
        }
        try? FileManager.default.removeItem(atPath: path)
        return events
    }
}
```

### Q1：如何设计一个完整的 iOS 性能监控体系？

```
分四个维度：

1. 稳定性监控
```

### Q2：OOM 如何检测？与普通崩溃有什么区别？

```
区别：
  普通崩溃：有 Crash 日志，有明确的崩溃原因和调用栈
  OOM：没有 Crash 日志，系统静默杀死进程

OOM 检测原理（排除法）：
  App 异常退出，且不是以下情况：
    - 用户主动退出（收到 applicationWillTerminate）
    - 已知崩溃（崩溃监控有记录）
    - Watchdog 超时（JetsamEvent 有记录）
    - 系统重启/升级
  
  排除以上情况后，剩余的异常退出 = 可能是 OOM

实现：
  App 启动时检查上次是否正常退出（UserDefaults 标记）
  正常退出时清除标记，异常退出时标记保留
  下次启动时检查标记，判断是否 OOM

OOM 预防：
  - 内存水位监控（超过阈值主动释放缓存）
  - 大图降采样（避免加载超大图片）
  - 及时释放不可见视图的资源
  - 避免内存泄漏（循环引用等）
```

### Q3：网络监控如何不影响业务代码？

```
最佳方案：URLProtocol 拦截（AOP 思想）

原理：
  注册自定义 URLProtocol，系统在发起请求前先经过我们的 Protocol
  我们记录请求信息，然后转发给系统处理
  响应回来时，我们记录响应信息，再转发给业务方

优点：
  - 业务代码完全不需要修改
  - 覆盖所有使用 URLSession 的请求
  - 可以监控第三方库的请求

注意事项：
  - 避免递归拦截（设置标记位）
  - 不要在 URLProtocol 中做耗时操作
  - 正确处理 cancel 和 redirect

更精确的方案：URLSessionTaskMetrics
  通过 URLSessionTaskDelegate 的 didFinishCollecting metrics 回调
  可以获取 DNS/TCP/TLS/TTFB 各阶段的精确时间
  但需要业务方设置 delegate，侵入性稍高
```

### 崩溃上报（PLCrashReporter、Firebase Crashlytics、Bugsnag）

```
三大主流方案对比：

  PLCrashReporter    ObjC  开源/隐私/自定义           国内/隐私敏感项目
```

### ANR 检测与上报

```
ANR (Application Not Responding) 是 iOS 上主线程长时间阻塞导致的事件。
系统会在主线程卡死 200ms 以上触发看门狗机制，超过 2-3 秒强制 kill 进程。

检测原理：基于主线程 RunLoop 或 Dispatch 的 idle 时间窗口。
当主线程持续繁忙超过阈值（通常 2-5 秒），判定为 ANR。

基于 RunLoop 的检测方案：

  let anrDetector = ANRDetector(timeout: 3.0) { backtrace in
      // 触发 ANR 回调，backtrace 包含主线程调用栈
      reportANR(backtrace: backtrace)
  }
  anrDetector.start()

  // 核心实现：在主线程 RunLoop 的 before/after 观察点打时间戳
  // 如果 after 触发距离 before 超过 timeout，说明主线程被阻塞

基于 Dispatch 的检测方案（推荐）：

  private var tick = DispatchTime.now()
  private let monitor = DispatchSource.makeTimerSource()

  func start() {
      monitor.schedule(deadline: .now(), repeating: 1.0)
      monitor.setEventHandler { [weak self] in
          guard let self = self else { return }
          let latency = DispatchTime.now().uptimeNanoseconds - self.tick.uptimeNanoseconds
          self.tick = DispatchTime.now()
          if latency > Double(self.timeout * 1_000_000_000) {
              // 主线程超过 1 秒未处理 tick，触发 ANR 回调
              self.onANR()
          }
      }
      monitor.resume()
  }

  // 在主线程每次 RunLoop 迭代后调用 tick = DispatchTime.now()
  // 通过 overrideRunloop 或 method_swizzling 实现自动打点

常见 ANR 原因：
  主线程阻塞操作    占比    典型案例
  网络请求          35%    主线程发起同步 HTTP 请求
  磁盘 I/O          25%    主线程读写大文件/数据库
  锁竞争            20%    主线程等待子线程持有的锁
  复杂计算          12%    主线程执行大数组遍历/JSON 解析
  第三方 SDK        8%    广告/统计 SDK 主线程阻塞

ANR 上报内容：
  - 主线程调用栈（通过 NSThread.callStackSymbols 或 libunwind 抓取）
  - 所有线程调用栈（排查死锁/锁竞争）
  - 触发时间、持续时间、设备信息
  - 业务上下文（当前页面、用户操作、网络状态）

Pitfalls: Dispatch 方案在主线程调用 DispatchMain.async 时可能触发误报，需在异步块入口处重置 tick。
Best practice: 结合 RunLoop 观察（低开销）+ Dispatch 检测（高精度），双重确认后再上报，避免污染数据。
```

### 生产环境内存使用监控

生产环境中内存监控是预防 OOM 崩溃的第一道防线。通过持续采集 PID 级别内存数据，结合业务页面维度的打点，定位内存泄漏与异常增长。

**核心指标**:
- **PSS (Proportional Set Size)**: 真实物理内存占用（共享库按进程数均分），最准确的内存指标
- **RSS (Resident Set Size)**: 常驻物理内存（共享库全量计入），偏大但采集成本低
- **Compressed**: 页面压缩后占用的内存（macOS/iOS 内核页面压缩机制）
- **VM Size**: 虚拟地址空间总量（参考价值低，通常数百 GB）

**采集方案**:

```swift
// 基于 libproc 获取 PSS（iOS 推荐方案）
import Darwin

func getCurrentPSS() -> UInt64 {
    var info = task_vm_info_data_t()
    var count = mach_msg_type_number_t(MemoryLayout<task_vm_info_data_t>.size) / 4
    let kr = withUnsafeMutablePointer(to: &info) { ptr in
        ptr.withMemoryRebound(to: IntegerType.self, capacity: Int(count)) {
            task_info(mach_task_self_, task_flavor_t(TASK_VM_INFO), $0, &count)
        }
    }
    guard kr == KERN_SUCCESS else { return 0 }
    return UInt64(info.phys_footprint)  // 字节为单位
}

// 基于 PID 获取更细粒度信息（PSS 通过 /proc 或 task_info 扩展）
func getProcessMemoryInfo() -> (PSS: UInt64, RSS: UInt64) {
    var info = task_info()
    var count = mach_msg_type_number_t(taskInfoMax)
    task_info(mach_task_self_, task_flavor_t(TASK_INFO), &info, &count)
    // info.task.resident_size = RSS
    // info.task.physical_footprint = 物理足迹（近似 PSS）
    return (PSS: info.task.physical_footprint, RSS: UInt64(info.task.resident_size))
}
```

**页面级打点**: 在 `viewDidAppear`/`viewDidDisappear` 采集 PSS 差值，关联页面名称上报。

Pitfalls: `malloc_size()` 只统计 heap 区域，不包含 CoreFoundation/UIImage 预分配缓冲，会低估真实占用。
Best practice: 每 30 秒后台采样一次 PSS，页面切换时打点差值，结合 Trend 分析（而非单点绝对值）判断泄漏。


### 网络性能监控（请求延迟、成功率）

生产环境中网络请求是用户体验的最大变量。通过全局拦截所有 HTTP/HTTPS 请求，采集延迟分布、成功率、错误类型，定位慢接口与不稳定服务。

**核心指标**:
- **P50/P90/P99 延迟**: 请求耗时分布，P99 超过 2 秒即需优化
- **成功率**: 2xx/3xx 占总请求比例，低于 95% 触发告警
- **错误分类**: 4xx（客户端错误）、5xx（服务端错误）、Timeout、DNS 失败、TLS 握手失败
- **DNS 解析时间**: 单独剥离，DNS 慢通常意味着网络环境问题

**基于 URLSession 的全局拦截**:

```swift
class NetworkMonitor: NSObject, URLSessionDelegate {
    private let queue = DispatchQueue(label: "net.monitor")
    private var requests: [(url: String, start: Date)] = []

    func urlSession(_ session: URLSession, task: URLSessionTask, didSendBodyBytes: Int64, totalBytesSent: Int64, totalBytesExpectedToSend: Int64) {}

    func urlSession(_ session: URLSession, task: URLSessionTask, didCompleteWithError error: Error?) {
        let url = task.originalRequest?.url?.absoluteString ?? "<unknown>"
        let latency = Date().timeIntervalSince(Self.startTime)
        let success = error == nil && (task.response as? HTTPURLResponse)?.statusCode ?? 0 < 400
        report(url: url, latency: latency, success: success, error: error)
    }
}

// iOS 17+: 使用 Instrumentation 框架零侵入监控
import Instrumentation
let networkInstrument = Instrument(identifier: Instrument.Identifier(category: "com.app.network", name: "api_latency"))
networkInstrument.measure(interval: .seconds(10)) { interval in
    // 自动聚合每秒请求数、平均延迟
}
```

Pitfalls: 在主线程调用 `URLSession.shared.dataTask` 不会阻塞但会污染主线程 RunLoop 统计，应始终使用自定义 Background 队列的 session。
Best practice: 结合 SWC (System Wi-Fi Configuration) 和 NWPathMonitor 判断当前网络类型（WiFi/蜂窝/无网络），将网络质量作为延迟分析的维度。


### 生产环境 FPS 监控

生产环境中 FPS（Frames Per Second）是衡量 UI 流畅度的核心指标。60fps 是标准目标（每帧 16.67ms），低于 45fps 用户可感知卡顿，低于 30fps 体验严重下降。通过 CATransaction 回调或 CADisplayLink 在主线程打点，计算每帧实际渲染时间。

**基于 CADisplayLink 的 FPS 采集**:

```swift
class FPSMonitor {
    private var displayLink: CADisplayLink?
    private var frameTimes: [CFTimeInterval] = []
    private var lastInterval: CFTimeInterval = 0

    func start() {
        displayLink = CADisplayLink(target: DispatchProxy(self), selector: #selector(DispatchProxy.tick))
        displayLink?.add(to: .main, forMode: .common)
    }

    @objc func tick(_ link: CADisplayLink) {
        let interval = link.duration  // 目标帧间隔 1/60
        let actual = link.targetTimestamp - lastInterval
        lastInterval = link.targetTimestamp
        frameTimes.append(actual)
        if frameTimes.count >= 60 {  // 每 60 帧统计一次
            let avg = frameTimes.reduce(0, +) / Double(frameTimes.count)
            let fps = 1.0 / avg
            report(fps: fps)
            frameTimes.removeAll()
        }
    }
}
```

**iOS 17+ Instrumentation 框架**: 零侵入 FPS 监控，系统自动聚合页面级帧率数据。

Pitfalls: CADisplayLink 在主线程负载高时被系统自动降频（frameInterval 增大），此时采集到的 FPS 已经是"降频后"的值，会低估真实卡顿程度。
Best practice: 结合 Framecad 或 Core Animation Instrument 定期录制页面过渡动画，配合生产端 FPS 采样（每 5 秒一次均值上报），区分"瞬时卡顿"与"持续低帧"两类问题。
### 用户行为分析（事件、漏斗、留存）

用户行为分析是理解 App 使用模式和优化转化路径的核心。通过埋点采集用户事件（页面浏览、按钮点击、搜索、购买等），构建漏斗模型追踪转化链路，结合留存分析评估长期用户价值。现代 iOS 开发中通常集成第三方 SDK（如 Firebase Analytics、AppsFlyer、友盟+）或自建埋点系统。

**自定义事件追踪系统**:

```swift
final class AnalyticsTracker {
    static let shared = AnalyticsTracker()
    private let queue = DispatchQueue(label: "com.app.analytics")
    private var pendingEvents: [AnalyticsEvent] = []

    struct AnalyticsEvent: Codable {
        let name: String          // "screen_view", "button_click"
        let params: [String: Any] // {"screen": "Home", "source": "push"}
        let timestamp: Date
    }

    func track(_ name: String, params: [String: Any] = [:]) {
        let event = AnalyticsEvent(name: name, params: params, timestamp: Date())
        queue.async { [weak self] in
            self?.pendingEvents.append(event)
            if self?.pendingEvents.count ?? 0 >= 20 {
                self?.flush()  // 批量上报，减少网络请求
            }
        }
    }

    private func flush() {
        guard let data = try? JSONEncoder().encode(pendingEvents) else { return }
        // 通过 URLSession 批量上报到后端
        pendingEvents.removeAll()
    }
}

// 使用示例
AnalyticsTracker.shared.track("screen_view", params: ["screen": "ProductDetail", "product_id": "123"])
AnalyticsTracker.shared.track("add_to_cart", params: ["product_id": "123", "quantity": 2])
```

**漏斗分析**: 定义关键转化步骤（首页→搜索→商品详情→加购→下单→支付），计算每步转化率。通过 A/B 测试优化低转化环节。

**留存分析**: 计算 D1/D7/D30 留存率（次日/周/月后仍活跃的用户比例），结合用户分群（新/老/沉默用户）制定差异化运营策略。

Pitfalls: 在主线程同步发送埋点请求会阻塞 UI 渲染，始终使用异步队列 + 批量上报策略；隐私合规方面，GDPR/CCPA 要求用户明确授权后才能采集设备标识（IDFA 已限制，改用 IDFV 或 Server-GAID）。
Best practice: 结合 iOS 17 的 App Attest 机制验证设备真实性，使用 SWCData 限制单个维度的数据粒度以防止隐私泄露，配合 Server-Side GTAG 实现端云协同的事件归因。
### 远程配置 / 功能开关（LaunchDarkly、Optimizely）

远程配置（Remote Config）和功能开关（Feature Flags）是现代 iOS 应用必备的基础设施。通过云端控制中心动态下发配置，实现功能灰度发布、A/B 测试、运行时热修复，无需等待 App Store 审核。LaunchDarkly 和 Optimizely 是两大主流商业方案，Firebase Remote Config 则是 Google 免费替代品。

**基于 UserDefaults + 自定义 SDK 的 Feature Flag 实现**:

```swift
final class FeatureFlagManager {
    static let shared = FeatureFlagManager()
    private var flags: [String: Bool] = [:]
    private var configVersion: String = ""

    func fetchRemoteConfig() async {
        guard let url = URL(string: "https://api.example.com/flags/v2") else { return }
        let (data, _) = try! await URLSession.shared.data(from: url)
        guard let json = try? JSONDecoder().decode([String: Bool].self, from: data) else { return }
        flags = json  // 覆盖本地默认值
    }

    func isEnabled(_ flag: String) -> Bool {
        flags[flag] ?? true  // 默认开启/关闭策略
    }
}

// 使用：页面级功能灰度
if FeatureFlagManager.shared.isEnabled("new_checkout_flow_v2") {
    present(NewCheckoutView())
} else {
    present(LegacyCheckoutView())
}
```

**LaunchDarkly 高级特性**: 基于用户属性的目标规则（targeting rules）、多环境隔离（dev/staging/prod）、实时评估（Stream 协议，无需轮询）、实验分析（自动统计转化差异）、Rollback 一键回滚。

Pitfalls: 功能开关滥用会导致"永不开关"的技术债——定期清理过期 flag（建议超过 3 个月未变更即触发 Code Owner 审查），避免代码库中堆积死分支。
Best practice: 结合 iOS 17 的 `onChange` 配合 @Observable 实现 flag 变更的 UI 热更新，配合本地缓存（URLCache + 过期时间）确保离线场景下功能可用，评估逻辑始终走 SDK 层而非硬编码在业务代码中。

### A/B 测试框架集成（Experiments SDK、Google Optimize 替代品）

A/B 测试是数据驱动产品决策的核心方法。通过将用户随机分配到不同实验组（Control vs Variant），量化功能变更对关键指标（转化率、留存、收入）的影响。现代 iOS 方案包括：LaunchDarkly Experiments（基于 Feature Flags）、Firebase A/B Testing（免费）、Statsig（开源 SDK + 自托管）、以及自研基于 UserDefaults 分桶的轻量方案。

**基于 Statsig 的 A/B 测试实现**:

```swift
import Statsig

// 初始化（通常在 SceneDelegate 或 App 入口）
Statsig.initialize(appKey: "your-ios-sdk-key")

// 定义实验 gate
let gate = Statsig.checkGate("new_homepage_layout_v3")

// 基于 gate 分流 UI
struct HomePage: View {
    var body: some View {
        if Statsig.checkGate("new_homepage_layout_v3") {
            NewDesignHomePage()  // Variant B
        } else {
            ClassicHomePage()    // Control A
        }
    }
}

// 追踪转化事件（用于统计显著性）
Statsig.logEvent(StatsigEvent(name: "completed_purchase", value: 29.99))
```

**自研分桶方案（基于设备哈希）**:

```swift
final class A/BTester {
    private let deviceBucket: Int

    init() {
        // 基于 IDFV 哈希，确保同一设备始终分到同一组
        let idfv = UIDevice.current.identifierForVendor?.uuid.uuidString ?? ""
        deviceBucket = Int(hash: idfv.prefix(8))! % 100
    }

    func variant(experiment: String, groups: [String: Double]) -> String {
        // groups: ["control": 50.0, "variant_a": 25.0, "variant_b": 25.0]
        let offset = deviceBucket ^ Int(experiment.hashValue) % 100
        var cumulative = 0.0
        for (name, weight) in groups {
            cumulative += weight
            if offset < Int(cumulative) { return name }
        }
        return groups.keys.first!
    }
}
```

Pitfalls: 设备哈希分桶在用户多设备切换时会导致跨设备分到不同组，污染实验结果；应优先使用 Server-Side 基于用户 ID 的全局分桶（如 Statsig/LaunchDarkly 的 User ID 维度的 consistent hash）。
Best practice: 结合 iOS 17 的 App Attest 防止伪造设备号刷量，实验数据通过 SWCData 限制粒度后上报，配合 Statsig 或 Mixpanel 的自动显著性检测（p-value < 0.05）决定是否全量发布。

### 性能基线追踪与回归检测

性能基线（Performance Baseline）是在特定设备/系统版本/数据量下测得的核心指标参考值（启动时间、帧率、内存峰值、功耗）。每次发布前通过自动化测试与基线比对，超出阈值即触发回归告警，防止"性能悄无声息地恶化"。

**基于 os_signpost 的自动化基线采集**:

```swift
import os

let signpostLog = OSSignpostLog(domain: "com.myapp.perf")

func measureLaunchTime() -> TimeInterval {
    let signpostID = OSSignpostType.event.id()
    signpostLog.signpost(type: .begin, name: "appLaunch", signpostID: signpostID)
    // ... 模拟启动流程 ...
    signpostLog.signpost(type: .end, name: "appLaunch", signpostID: signpostID)
    // 通过 Instruments 或 os_activity 自动提取区间数据
    return 0.0  // 实际通过 metricd 或 CI 脚本解析
}
```

**CI 集成方案（GitHub Actions + metricd 数据比对）**:

```bash
# 在 CI 中运行性能测试并生成基线报告
xcodebuild test -scheme MyApp -derivedDataPath build/   -only-testing:MyAppTests/PerformanceTests   -resultBundlePath PerformanceResults.xcresult

# 提取指标并与基线比对（Xcode 15+ 内置 metricd 支持）
xcresulttool get-metrics --path PerformanceResults.xcresult   --metrics-group com.apple.dt.MetricsGroup.Performance   --output baseline_metrics.json

# 与上个版本基线比对（JSON diff 触发回归检查）
python3 scripts/compare_baseline.py baseline_metrics.json last_release_metrics.json
```

Pitfalls: 不同设备（iPhone 13 vs 15 Pro）性能差异可达 3 倍，基线必须按设备型号 + iOS 版本隔离存储，否则误报率极高；CI 环境中 simulator 数据与真机差异大，关键指标（特别是 GPU/功耗）必须在真机农场（如 BrowserStack/Device Farm）上采集。

Best practice: 结合 Xcode 15 的 Performance Metrics（基于 metricd 的隐私安全指标采集），在 CI 中自动触发基于真机的测试矩阵（通过 `xcodecloud` 或自建的 Device Farm），设置 10% 容忍阈值（超出即阻断合并），配合 Trending 图表（如 Grafana/InfluxDB 持久化历史数据）观察长期性能走势。

### os_signpost + os_log 结构化日志用于生产环境

`os_log`（OSLog API）是 Apple 在 iOS 10 引入的统一日志框架，替代传统的 `print()` 和 `NSLog()`。配合 `os_signpost` 可实现区间性能追踪。两者通过 Unified Logging 系统写入，零运行时开销（debug 级别日志在 Release 构建中自动丢弃）。

**生产环境结构化日志示例**:

```swift
import os

private let logger = Logger(subsystem: Bundle.main.bundleIdentifier!, category: "API")
private let signpostLog = OSSignpostLog(domain: "com.myapp.perf")

// 结构化日志（自动携带 PID、TID、时间戳、隐私分级）
logger.info("User \(privacy: userID) started checkout flow")
logger.error("Payment failed: \(privacy: error.localizedDescription)")

// 区间追踪（Instruments 自动聚合为区间图表）
func fetchUserData() async -> UserData {
    let id = OSSignpostType.event.id()
    signpostLog.signpost(type: .begin, name: "fetchUser", signpostID: id)
    defer { signpostLog.signpost(type: .end, name: "fetchUser", signpostID: id) }
    // ... network call ...
    return userData
}
```

**隐私分级**: `.private`（详细信息）、`.public`（无隐私）、通过 `\(privacy: variable)` 语法在日志中自动处理隐私数据。

Pitfalls: 滥用 `.debug` 级别会导致日志量暴增（即使是 Release 也通过 console 写入磁盘），应限制到关键路径；`privacy` 占位符在 simulator 上可能不脱敏，以真机表现为准。

Best practice: iOS 17 起配合 Server-Side 日志聚合（如 Datadog/Logtail 通过 SPI 接口抓取 os_log 数据），实现从本地到后端的完整链路追踪。结合 `os_activity` API 跨函数/跨线程追踪请求全生命周期。

### 隐私合规（ATT、数据收集透明度）

iOS 14.5 引入的 App Tracking Transparency (ATT) 框架要求 App 在追踪用户前必须通过 `ATTrackingManager.trackingAuthorizationStatus` 显式请求授权。配合 `IDFA`（`advertisingIdentifier`）限制，未获授权的 App 无法获取设备广告标识符。

**ATT 请求时机与实现**:

```swift
import AppTrackingTransparency
import AdSupport

func requestTrackingPermission() {
    // 必须在主线程调用，距离 App 首次 foreground 至少 5 秒（建议 10-30 秒）
    DispatchQueue.main.asyncAfter(deadline: .now() + 10) {
        UIApplication.shared.windows.first?.makeKeyAndVisible()
        ATTrackingManager.requestTrackingAuthorization { status in
            switch status {
            case .authorized:
                let idfa = ASIdentifierManager.shared().advertisingIdentifier.uuidString
                print("IDFA: \(idfa)")  // 可合法使用
            case .denied, .restricted:
                // 使用 IDFV 作为备选（始终可用，仅限同开发者生态内）
                let idfv = UIDevice.current.identifierForVendor?.uuidString
                print("IDFV: \(idfv)")
            default: break
            }
        }
    }
}
```

**数据收集透明度报告（DCTR）**: iOS 14+ 要求使用 Apple 定义的分类（`NSPrivacyTracking`, `NSPrivacyCollectedDataTypeIdentifier` 等）在 `Info.plist` 或 Xcode 的 Privacy Manifest 中声明数据收集用途。

Pitfalls: 未在首次 foreground 后延迟请求（Apple 审核指南 5.1.1 强制要求），或将 ATT 弹窗放在注册/登录流程中（视为"引导式"触发，大概率被拒）；未正确配置 DCTR 会导致 2024 年起 App Store 隐私标签页面自动生成错误。

Best practice: iOS 17 起配合 `PrivacyInfo.xcprivacy` 隐私清单（Privacy Manifest），声明使用的 API 及其用途（如 `NSPrivacyAccessedAPICategoryFileTimestamp`），未提供隐私清单的 SDK 在 2024 年底后将被 App Store 拒绝上架。所有第三方 SDK 也需提供隐私清单，通过 `privacy.manifests` 数组在 Info.plist 中声明。


### SDK 集成影响评估方法论

在集成第三方 SDK 前，系统性评估其对 App 性能、隐私合规、包体积、启动时间的影响，避免"隐式债"积累。

**评估矩阵**:

| 维度 | 指标 | 阈值 | 测试方法 |
|------|------|------|----------|
| 包体积 | Binary size 增量 | < 2MB | `dSYM` 差异分析 + `dwarfdump` |
| 启动时间 | Main runtime 增量 | < 200ms | Instrument/`os_activity` |
| 内存 | 稳态 RSS 增量 | < 30MB | Allocations/Instruments |
| 隐私 | API 访问面 | 零未声明 SPI | Privacy Manifest 审计 |
| 兼容性 | 最低 iOS 版本 | ≥ 项目 baseline | 真机矩阵测试 |

**自动化评估脚本**:

```swift
// 集成后自动运行：隔离测试 SDK 初始化开销
class SDKImpactEvaluator {
    static func measureInitialization() -> ImpactReport {
        let start = CFAbsoluteTimeGetCurrent()
        let memBefore = ProcessInfo.processInfo.physicalMemoryUsage  // bytes
        
        SomeSDK.initialize()  // 被测 SDK 初始化
        
        let memAfter = ProcessInfo.processInfo.physicalMemoryUsage
        let elapsed = CFAbsoluteTimeGetCurrent() - start
        
        return ImpactReport(
            initTime: elapsed * 1000,  // ms
            memoryImpact: memAfter - memBefore  // bytes
        )
    }
}
```

**包体积拆解**: 使用 `dwarfdump --dry-run` 列出符号表，配合 `nm` 或 `size` 定位大符号。通过 `Thin Binary` 工具（如 `bsdiff` 或 `otool -s __DATA __unwind_info`）量化死代码。

Pitfalls: 仅在 simulator 测试（ simulator 不触发部分 native 初始化逻辑），忽略 bitcode/LLVM 死代码剥离差异，未在 Release 配置下测试（Debug 包含额外符号和断言）。

Best practice: 建立 SDK 准入 CheckList（隐私清单、dSYM 提供、API 文档完整性、最小目标版本），通过 CI 自动化运行 ImpactEvaluator 并在 PR 中报告增量；超阈值的 SDK 强制要求提供 Tree Shaking 配置或按需加载方案。
