# 包体积优化详解

## 一、包体积为什么重要

App 包体积直接影响用户下载意愿和留存率。据 Google 统计，包体积每增加 6MB，安装转化率下降 1%。在移动网络环境下，用户对大体积 App 更抗拒。App Store 对超过 200MB 的 App 限制蜂窝网络下载（iOS 13+ 放宽到 200MB，iOS 15+ 进一步放宽）。

出海场景下包体积更为敏感。东南亚、中东等市场的设备存储空间有限，网络带宽不稳定，用户对包大小容忍度更低。

## 二、包体积构成分析

### 2.1 IPA 结构

```
MyApp.ipa
├── Payload/
│   └── MyApp.app/
│       ├── MyApp                    # 主二进制（Mach-O）
│       ├── Frameworks/              # 嵌入的动态框架
│       ├── Assets.car               # Asset Catalog 编译产物
│       ├── *.png / *.jpg            # 散落图片资源
│       ├── *.json / *.plist         # 配置文件
│       ├── *.bundle                 # 资源 bundle
│       ├── *.strings                # 本地化字符串
│       ├── *.nib / *.storyboardc    # 编译后的界面文件
│       └── embedded.mobileprovision  # 描述文件
```

### 2.2 构建报告分析

在 Xcode 中生成 App Size Report：

```bash
xcodebuild -exportArchive -archivePath MyApp.xcarchive -exportOptionsPlist ExportOptions.plist -exportPath ./Export
```

或使用 App Thinning Size Report：

```xml
<!-- ExportOptions.plist -->
<key>generateAppStoreInformation</key>
<true/>
<key>thinning</key>
<string><all></string>
```

生成的报告会显示各部分的预估下载大小和安装大小。

### 2.3 二进制大小分析

使用 `otool` 和 `size` 命令分析二进制文件：

```bash
# 查看各段大小
size -m MyApp.app/MyApp

# 查看 section 大小
otool -l MyApp.app/MyApp | grep -A5 LC_SEGMENT

# 查看符号表大小
nm -S MyApp.app/MyApp | sort -k2 -n -r | head -20
```

使用 LinkMap 文件分析各模块的代码大小。在 Build Settings 中开启 `Write Link Map File`，生成 `.linkmap` 文件：

```
# LinkMap.txt
# Sections:  各段大小
# Symbols:   每个符号（函数/变量）的大小
# Dead Stripped Symbols: 被剔除的未使用符号
```

## 三、代码体积优化

### 3.1 编译器优化

```swift
// Build Settings
Swift Optimization Level:
- Onone: 无优化（Debug）
- O: 优化速度（Release，默认）
- Osize: 优化代码大小

// 整体优化（Whole Module Optimization）
// Build Settings → Swift Compiler → Compilation Mode → Whole Module
```

`-Osize` 相比 `-O` 略微牺牲性能但减少代码体积，适合对包体积敏感的场景。Whole Module Optimization 允许编译器跨文件优化，消除冗余代码。

### 3.2 Dead Code Stripping

编译器自动剔除未使用的代码。但需要满足条件：

- 未被引用的函数和变量会被剔除
- `@objc` 标记的方法不会被自动剔除（可能被 runtime 调用）
- `public` 符号不会被剔除（可能被其他模块引用）
- 被 `@available` 标记为过时的代码不会自动剔除

检查 Dead Stripped Symbols 部分，确认编译器确实剔除了未使用代码。

### 3.3 减少 ObjC 桥接代码

每个 `@objc` 方法、`dynamic` 属性都会生成 ObjC runtime 元数据，增加二进制体积。检查不必要的 `@objc` 标注：

```swift
// 不必要的 @objc（不需要被 ObjC/runtime 调用）
@objc func helperMethod() { ... }  // 删掉 @objc

// 必要的 @objc（被 @IBAction、@IBOutlet、NSCoding 等需要）
@objc func buttonTapped(_ sender: UIButton) { ... }
```

### 3.4 泛型特化

Swift 泛型在编译时可能为每种类型生成特化版本，导致代码膨胀。使用 `@inlinable` 和 `@_specialize` 控制特化策略：

```swift
// 限制特化，减少代码膨胀
func process<T: Numeric>(_ values: [T]) -> T {
    values.reduce(0, +)
}

// 或在大型库中使用 existential container 而非泛型
func process(_ values: [any Numeric]) -> any Numeric { ... }
```

### 3.5 C/C++ 代码优化

C/C++ 静态库是包体积大户。检查第三方 C/C++ 库是否只包含必要的源文件。使用 strip 剔除调试符号：

```bash
strip -S -x MyApp.app/MyApp  # 剥离调试符号
```

在 Build Settings 中设置 `Strip Style`：
- `All Symbols`：剥离所有符号（Release 推荐）
- `Non-Global Symbols`：剥离局部符号
- `Debugging Symbols`：仅剥离调试符号

## 四、资源优化

### 4.1 Asset Catalog

使用 Asset Catalog 管理图片资源，系统自动处理压缩和格式转换：

- PNG 图片自动压缩优化
- 按设备类型分发不同分辨率（slicing）
- 图片资源支持 App Thinning

确保所有图片都在 Asset Catalog 中管理，而非散落在 bundle 中。

### 4.2 图片格式优化

```bash
# 将 PNG 转换为 HEIC（体积减少 50%+）
# 需要运行时解码支持
sips -s format heic input.png --out output.heic

# 使用 WebP（需要 libwebp）
cwebp -q 80 input.png -o output.webp
```

HEIC 在 iOS 11+ 原生支持，体积比 PNG 小 50% 以上，但解码稍慢。WebP 需要第三方库支持。

对于不需要透明通道的图片，使用 JPEG 而非 PNG。

### 4.3 音频与视频

压缩音频为 AAC 或 Opus 格式。移除未使用的音视频资源。对于大视频文件，考虑在首次启动时从服务器下载而非打包在 App 中。

### 4.4 字体文件

字体文件通常很大（2-10MB/个）。只打包实际使用的字重和字形。使用字体子集化工具移除未使用的字符：

```bash
# 使用 fonttools 创建字体子集
pip install fonttools
pyftsubset SourceHanSans.ttf --text-file=used-chars.txt --output-file=Subset.ttf
```

### 4.5 本地化资源

确保只打包目标市场的语言资源。App Thinning 会自动为不同地区的设备分发对应语言的资源，但前提是资源正确配置了本地化属性。

## 五、App Thinning

### 5.1 Slicing

App Store 根据设备的架构和分辨率，只分发需要的资源切片。开发者不需要做额外工作，确保资源在 Asset Catalog 中正确配置即可。

效果：arm64 设备不会下载 armv7 切片，@3x 设备不会下载 @2x 切片。

### 5.2 Bitcode（已废弃）

Bitcode 允许 App Store 在服务器端重新编译，生成最优架构的包。Xcode 14 起移除了 Bitcode 支持，不再需要配置。

### 5.3 On-Demand Resources

对于非启动必须的资源（如游戏关卡、教程视频），可以使用 On-Demand Resources（ODR），按需从 App Store 下载：

```swift
let request = NSBundleResourceRequest(tags: ["level-5"])
request.beginAccessingResources { error in
    if error == nil {
        // 资源已下载，可以使用
    }
}
```

ODR 适合游戏类 App 按需加载关卡资源。

## 六、动态框架优化

### 6.1 动态 vs 静态

动态框架（Dynamic Framework）会增加 App 启动时间（dyld 加载和链接），但可以减少包体积（多个目标共享一份代码）。静态框架在编译时链接到主二进制，启动快但可能增加体积。

实践中，第三方依赖如果提供了静态库选项，优先选择静态链接以减少启动开销。

### 6.2 Mergeable Libraries（Xcode 15+）

Xcode 15 引入了 Mergeable Libraries，允许将多个动态框架在编译时合并为一个：

```swift
// Package.swift
.target(
    name: "MyLibrary",
    dependencies: [...],
    mergeable: true  // 声明可合并
)
```

在 App target 中设置 `Mergeable Libraries` 为 `Yes`，Xcode 会自动将标记为 mergeable 的依赖静态合并到主二进制中，兼顾启动速度和编译性能。

## 七、持续监控

### 7.1 CI 中集成包体积检查

在 CI 流水线中添加包体积检查步骤：

```bash
# 构建
xcodebuild archive -scheme MyApp -archivePath MyApp.xcarchive

# 导出 IPA
xcodebuild -exportArchive -archivePath MyApp.xcarchive -exportOptionsPlist ExportOptions.plist -exportPath ./Export

# 检查大小
IPA_SIZE=$(stat -f%z MyApp.ipa)
THRESHOLD=104857600  # 100MB

if [ $IPA_SIZE -gt $THRESHOLD ]; then
    echo "Error: IPA size ($IPA_SIZE bytes) exceeds threshold"
    exit 1
fi
```

### 7.2 趋势追踪

记录每次构建的包体积，绘制趋势图。当包体积异常增长时及时告警，定位引入大体积资源的 commit。

### 7.3 第三方工具

**Emerge Tools**：自动分析 IPA 构成，对比每次 PR 的体积变化。**AppThinning**：Apple 官方工具，生成各设备类型的预估下载大小。**Bloaty**：开源二进制分析工具，查看 Mach-O 各 section 大小。

## 八、优化检查清单

代码层面：开启 `-Osize` 和 Whole Module Optimization；检查并移除不必要的 `@objc` 标注；开启 Dead Code Stripping；strip 所有调试符号。

资源层面：所有图片放入 Asset Catalog；大图转 HEIC/WebP；字体子集化；移除未使用的音视频资源。

架构层面：评估第三方库的实际使用范围，移除或替换过重的依赖；优先静态链接减少动态框架数量；使用 Mergeable Libraries 合并小框架。

分发层面：确保 Asset Catalog 正确配置 slicing；考虑 On-Demand Resources 按需加载非必要资源；只打包目标市场语言资源。
