# 调试与性能分析工具链详解

## 一、Instruments

Instruments 是 Xcode 附带的性能分析工具集，包含多个专门的分析模板。每个模板针对一类性能问题，通过 trace（追踪）文件记录时间线数据。

### 1.1 Time Profiler

Time Profiler 采样 CPU 使用情况，显示每个函数的运行时间和调用层级。用于定位 CPU 热点。

使用方法：打开 Instruments → Time Profiler → 选择目标 App → 录制 → 执行操作 → 停止。

分析技巧：

查看 Call Tree 面板，勾选 "Invert Call Tree" 让叶子函数排在前面，快速定位最耗时的函数。勾选 "Hide System Libraries" 只显示自己的代码。勾选 "Hide Missing Symbols" 隐藏无法解析的符号。

Double-click 函数名可以跳转到源码行。

### 1.2 Allocations

Allocations 追踪堆内存分配，用于定位内存泄漏和内存峰值。

关键指标：

**All Heap Allocations**：所有堆分配的总量。
**Persistent**：当前仍在使用（未释放）的内存。
**Transient**：已释放的内存。
**Total**：Persistent + Transient。

使用 Mark Generation 功能：在操作前标记一次，操作后标记一次，对比两次标记之间的内存增量。这可以精确测量某段操作的内存开销。

### 1.3 Leaks

Leaks 自动检测循环引用导致的内存泄漏。周期性扫描堆内存，查找不可达对象。

Leaks 只能检测"泄漏"（完全不可达的对象），不能检测"保留环"（多个对象互相引用但整体不可达）。对于 retain cycle，需要配合 Allocations 或 Xcode Memory Graph 分析。

### 1.4 Animation Hitches

Animation Hitches 专门检测动画卡顿。它记录每一帧的渲染时间，标记超过 16.67ms（60Hz）或 8.33ms（120Hz）的帧。

Hitch 类型：
- **Commit Hitch**：主线程在提交帧时耗时过长
- **Render Hitch**：GPU 渲染耗时过长

分析时关注 hitch 前后的调用栈，定位导致卡顿的代码。

### 1.5 System Trace

System Trace 记录系统级事件：线程调度、锁竞争、系统调用、GPU 活动。适合分析主线程被阻塞的原因（如等待锁、等待 I/O）。

使用场景：UI 卡顿但 Time Profiler 显示 CPU 使用率不高时，用 System Trace 查看主线程是否在等待资源。

### 1.6 Core Animation

Core Animation 模板直接在设备上检测渲染性能问题。通过 Debug Options 可视化：
- Color Offscreen-Rendered（离屏渲染检测）
- Color Blended Layers（混合图层检测）
- Color Misaligned Images（图片缩放检测）

## 二、Xcode Memory Graph

### 2.1 使用方法

运行 App → 点击 Xcode 底部的 Debug Memory Graph 按钮 → Xcode 暂停 App 并展示当前内存图。

内存图显示所有对象及其引用关系。节点是对象，箭头是引用。红色节点表示泄漏对象（只被强引用环持有）。

### 2.2 检测 Retain Cycle

```swift
class ViewController: UIViewController {
    var completion: (() -> Void)?

    override func viewDidLoad() {
        super.viewDidLoad()

        completion = {
            self.doSomething()  // 闭包持有 self，self 持有闭包 → 循环引用
        }
    }
}
```

在 Memory Graph 中搜索 ViewController，如果发现它被一个闭包引用，而闭包又被 ViewController 引用，就是循环引用。

修复方式：使用 `[weak self]` 打断循环。

### 2.3 导出内存图

可以将内存图导出为 `.memgraph` 文件，供后续分析或分享给同事：

```bash
# 命令行分析 memgraph
xcrun malloc_stats MyApp.memgraph    # 内存统计
xcrun heap MyApp.memgraph            # 堆对象列表
xcrun leaks MyApp.memgraph           # 泄漏检测
```

## 三、LLDB 高级技巧

### 3.1 基础命令

```
po object           # 打印对象描述（print object）
p expression        # 打印值（print）
frame variable      # 打印当前栈帧所有变量
bt                  # 打印调用栈
thread backtrace    # 线程调用栈
expression var = 42 # 执行表达式（修改变量）
```

### 3.2 表达式求值

```swift
// 在断点处执行代码
(lldb) expression self.view.backgroundColor = UIColor.red
(lldb) expression let label = self.view.subviews.first as? UILabel
(lldb) expression label?.text = "Modified"

// 调用方法
(lldb) expression self.reloadData()
```

### 3.3 条件断点

在 Xcode 断点编辑器中设置条件：

```
// 条件断点
i == 42
array.count > 0
userName == "admin"
```

条件为 true 时断点才触发，避免在循环中频繁中断。

### 3.4 Symbolic Breakpoint

在 Xcode → Breakpoint Navigator → + → Symbolic Breakpoint：

```
// 在所有调用 viewDidLoad 的地方断点
-[UIViewController viewDidLoad]

// 在所有 stringWithFormat: 调用断点
+[NSString stringWithFormat:]

// 在 C 函数断点
malloc_error_break
```

### 3.5 Watchpoint

Watchpoint 监视变量或内存地址的变化。在 Variables View 中右键变量 → Watch:

```
// 当某属性被修改时断点
(lldb) watchpoint set variable self.count
```

Watchpoint 对排查"谁修改了这个值"的问题非常有用。

### 3.6 chisel

chisel 是 Facebook 开源的 LLDB Python 脚本集合，提供大量便捷命令：

```bash
# 安装
brew install chisel

# 在 ~/.lldbinit 中添加
command script import /path/to/chisel/fbobjclldb.py
```

常用命令：

```
pvc            # 打印当前 view controller 层级
pviews         # 打印 view 层级
fv label       # 查找 view
fvc user       # 查找 view controller
mask view      # 给 view 加边框（可视化定位）
border view    # 给 view 加边框
unmask         # 移除边框
caflush        # 强制重绘界面
```

### 3.7 Python 脚本

LLDB 支持用 Python 编写自定义命令：

```python
# ~/.lldb/count_views.py
import lldb

def count_views(debugger, command, result, internal_dict):
    target = debugger.GetSelectedTarget()
    process = target.GetProcess()
    thread = process.GetSelectedThread()
    frame = thread.GetSelectedFrame()

    # 执行表达式计算子视图数量
    expr = "(NSUInteger)[[[(UIView *)0x{:x}] subviews] count]".format(int(command, 16))
    val = frame.EvaluateExpression(expr)
    result.PutCString("Subviews count: {}".format(val.GetValue()))

def __lldb_init_module(debugger, internal_dict):
    debugger.HandleCommand('command script add -f count_views.count_views cv')
```

## 四、视图调试

### 4.1 3D View Debugger

运行 App → 点击 Xcode 底部的 Debug View Hierarchy 按钮 → 进入 3D 视图调试模式。

功能：
- 3D 可视化展示视图层级和层级深度
- 选中视图查看约束、属性、frame
- 显示/隐藏特定视图
- 检查离屏渲染、混合层级
- 截取当前视图状态

### 4.2 常用调试场景

**布局问题**：用 3D 视图查看约束是否冲突，frame 是否为负数或超出父视图。

**视图层级过深**：3D 视图直观展示层级深度，过深的层级影响渲染性能。

**透明度和混合**：底部面板显示选中视图的 alpha、opaque 属性，辅助定位混合渲染问题。

## 五、网络调试

### 5.1 Charles / Proxyman

HTTP/HTTPS 抓包工具，可以查看 App 的所有网络请求：

- 查看请求/响应的 headers 和 body
- 修改请求参数（Map Local / Rewrite）
- 模拟慢速网络（Throttle Settings）
- 设置断点拦截请求

HTTPS 抓包需要安装并信任证书。在 iOS 设备上：设置 → 通用 → 关于本机 → 证书信任设置 → 启用对根证书的完全信任。

### 5.2 Instruments Network Instrument

Instruments 中的 Network 模板追踪 URL Session 的请求和响应，配合时间线分析网络请求对启动和交互的影响。

### 5.3 NSURLProtocol 拦截

```swift
class LoggingURLProtocol: URLProtocol {
    override class func canInit(with request: URLRequest) -> Bool {
        return true
    }

    override class func canonicalRequest(for request: URLRequest) -> URLRequest {
        return request
    }

    override func startLoading() {
        // 记录请求
        print("Request: \(request.url?.absoluteString ?? "")")
        // 转发给真正的网络层
    }

    override func stopLoading() {
        // 记录响应
    }
}

// 注册
URLProtocol.registerClass(LoggingURLProtocol.self)
```

## 六、Sanitizer

### 6.1 Address Sanitizer (ASan)

ASan 检测内存错误：缓冲区溢出、Use-After-Free、栈溢出。

在 Scheme → Diagnostics → Runtime Sanitization 中勾选 Address Sanitizer。

### 6.2 Thread Sanitizer (TSan)

TSan 检测数据竞争——多线程同时读写同一变量且至少一个是写操作。

```swift
// TSan 会检测到这里的数据竞争
class Counter {
    var count = 0  // 多线程访问非线程安全

    func increment() {
        count += 1  // Race condition!
    }
}
```

### 6.3 Undefined Behavior Sanitizer (UBSan)

UBSan 检测 C/C++ 的未定义行为：整数溢出、空指针解引用、对齐违规等。

### 6.4 使用建议

Sanitizer 只在 Debug 构建中启用，有 2-10 倍的性能开销。在 CI 中运行 Sanitizer 测试可以提前发现问题。三种 Sanitizer 不能同时启用，需要分别运行测试。

## 七、sysdiagnose

sysdiagnose 是系统级诊断报告，包含系统日志、内核状态、进程信息等。当问题难以在本地复现时，可以请用户或 QA 采集 sysdiagnose：

```bash
# 在 Mac 上触发连接的 iOS 设备生成 sysdiagnose
idevicesyslog
# 或在设备上：设置 → 隐私 → 分析与改进 → 分析数据
```

sysdiagnose 包含大量信息，重点关注：crash logs、spin dumps（死锁）、log archive（系统日志）。
