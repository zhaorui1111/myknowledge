# 二进制框架与组件分发详解

## 一、框架类型

iOS 中的 Framework 分为静态框架和动态框架，它们在链接方式、启动性能和分发方式上有本质区别。

**静态框架**（Static Framework）：在编译时链接到目标二进制中，代码被直接拷贝到最终的可执行文件。启动时不需要额外的动态加载，性能最优。缺点是多个模块使用同一个静态框架时，代码会被复制多份（除非使用 Mergeable Libraries）。

**动态框架**（Dynamic Framework）：在运行时由 dyld（动态链接器）加载和链接。多个模块共享同一份动态框架代码。缺点是每个动态框架增加约 1-100ms 的启动时间（取决于依赖数量和符号量）。

**XCFramework**：Apple 推荐的多架构框架分发格式，可以包含不同架构（arm64、x86_64）和不同平台（iOS、iOS Simulator、macOS）的切片。XCFramework 不是一种新的链接方式，而是静态/动态框架的多架构容器。

## 二、创建 Framework

### 2.1 创建 Framework Target

在 Xcode 中 File → New → Project → Framework。设置 Product Name，选择语言和平台。

### 2.2 访问控制

```swift
// Public API（对外暴露）
public class UserService {
    public init() {}

    public func fetchUser(id: String) async throws -> User {
        return try await repository.fetchUser(id: id)
    }

    public struct User: Codable {
        public let id: String
        public let name: String
    }
}

// Internal API（模块内可见）
class UserRepository {
    func fetchUser(id: String) async throws -> UserService.User {
        // 实现细节
    }
}

// 完全隐藏的实现细节
private struct APIResponse {
    let data: Data
}
```

框架的公开 API 应该只暴露 `public` 和 `open` 标记的符号。`internal` 和 `private` 的符号不会出现在公开接口中。

### 2.3 编译为 XCFramework

```bash
# 构建 device 版本
xcodebuild archive \
    -scheme MyFramework \
    -configuration Release \
    -destination "generic/platform=iOS" \
    -archivePath build/device.xcarchive \
    SKIP_INSTALL=NO \
    BUILD_LIBRARY_FOR_DISTRIBUTION=YES

# 构建 simulator 版本
xcodebuild archive \
    -scheme MyFramework \
    -configuration Release \
    -destination "generic/platform=iOS Simulator" \
    -archivePath build/simulator.xcarchive \
    SKIP_INSTALL=NO \
    BUILD_LIBRARY_FOR_DISTRIBUTION=YES

# 合并为 XCFramework
xcodebuild -create-xcframework \
    -archive build/device.xcarchive -framework MyFramework.framework \
    -archive build/simulator.xcarchive -framework MyFramework.framework \
    -output MyFramework.xcframework
```

`BUILD_LIBRARY_FOR_DISTRIBUTION=YES` 启用 Module Stability，确保框架可以被不同 Swift 编译器版本消费。

## 三、Module Stability

### 3.1 为什么需要 Module Stability

Swift 5.1 之前，框架的 Swift module 接口与编译器版本强绑定——用 Swift 5.0 编译的框架无法被 Swift 5.1 的项目使用。Module Stability 通过引入 `.swiftinterface` 文件（纯文本的模块接口描述）解决了这个问题。

### 3.2 启用

设置 `BUILD_LIBRARY_FOR_DISTRIBUTION = YES`。Xcode 会同时生成 `.swiftmodule`（二进制）和 `.swiftinterface`（文本）。当消费方的编译器版本与框架不匹配时，使用 `.swiftinterface` 接口。

### 3.3 限制

启用 Module Stability 后，某些 Swift 特性不可用：`@_inline(__always)` 等编译器内部属性、泛型特化信息不暴露（可能影响性能）、不能使用 `import` 私有模块。

## 四、Binary Framework Distribution

### 4.1 通过 SPM 分发二进制框架

```swift
// Package.swift
let package = Package(
    name: "MyFramework",
    products: [
        .library(name: "MyFramework", targets: ["MyFramework"]),
    ],
    targets: [
        .binaryTarget(
            name: "MyFramework",
            path: "MyFramework.xcframework"
        )
    ]
)
```

### 4.2 远程二进制框架

```swift
.binaryTarget(
    name: "MyFramework",
    url: "https://github.com/org/MyFramework/releases/download/1.0.0/MyFramework.xcframework.zip",
    checksum: "a1b2c3d4e5f6..."
)
```

SPM 下载 zip 后验证 checksum，解压并链接。适合公开发布的框架。

### 4.3 私有分发

私有框架可以通过私有 Git 仓库或私有 Registry 分发：

```swift
// 私有 Git 仓库（需要 SSH key 或 Token）
.package(url: "git@github.com:MyOrg/PrivateFramework.git", from: "1.0.0")
```

## 五、框架中的资源处理

### 5.1 Bundle 资源

框架中的资源（图片、Storyboard、JSON）被打包到独立的 Bundle 中：

```swift
// 访问框架内资源
if let bundleURL = Bundle(for: MyFrameworkClass.self).url(forResource: "MyFramework", withExtension: "bundle"),
   let resourceBundle = Bundle(url: bundleURL) {
    let image = UIImage(named: "logo", in: resourceBundle, compatibleWith: nil)
}
```

### 5.2 SPM 中的资源

```swift
.target(
    name: "MyFramework",
    dependencies: [],
    resources: [
        .process("Images"),       // 处理目录
        .copy("Templates"),       // 原样拷贝
    ]
)
```

使用 `Bundle.module` 访问 SPM 资源。

## 六、依赖管理

### 6.1 框架间依赖

```swift
// Framework A 依赖 Framework B
.target(
    name: "FrameworkA",
    dependencies: ["FrameworkB"]
)
```

静态框架的依赖在编译时被链接。动态框架的依赖在运行时被 dyld 解析。

### 6.2 避免符号冲突

当多个框架包含相同的符号（如第三方库被重复打包）时，会导致 duplicate symbol 错误。解决方案：

- 使用 SPM/CocoaPods 统一管理依赖，避免重复打包
- 将共用依赖提取为独立框架
- 使用 `-fvisibility=hidden` 隐藏内部符号
- 对于 C/C++ 库，使用 `static` 限定符

## 七、Mergeable Libraries

### 7.1 概述

Xcode 15 引入 Mergeable Libraries，将多个动态框架在编译时静态合并到主二进制中。兼顾动态框架的开发体验和静态框架的启动性能。

### 7.2 使用

```swift
// Package.swift 中标记
.target(
    name: "MyLibrary",
    dependencies: [...],
    mergeable: true
)
```

在 App Target 的 Build Settings 中设置 `MERGED_BINARY_TYPE = embedAndSign`，Xcode 自动将 mergeable 的依赖合并到主二进制。

### 7.3 效果

编译时：框架正常编译为动态库，保持增量编译速度。链接时：标记为 mergeable 的框架被静态合并到主二进制，减少动态加载开销。运行时：只有一个主二进制需要加载，启动更快。

## 八、版本管理与兼容性

### 8.1 语义化版本

框架版本遵循 `MAJOR.MINOR.PATCH`：
- MAJOR：不兼容的 API 变更
- MINOR：向后兼容的功能新增
- PATCH：向后兼容的 bug 修复

### 8.2 弃用标注

```swift
@available(*, deprecated, message: "Use fetchUser(id:) instead")
public func getUser(id: String) -> User? { ... }

@available(*, deprecated, renamed: "UserService.User")
public typealias AppUser = UserService.User

// 指定未来版本移除
@available(iOS, deprecated: 16.0, message: "Use new API")
public func oldMethod() { ... }
```

### 8.3 API 兼容性检查

使用 `swift-api-digester` 工具比较不同版本框架的公开 API 差异：

```bash
swift api-digester -dump-sdk -sdk <path> -o old.json
# 修改框架后
swift api-digester -dump-sdk -sdk <path> -o new.json
swift api-digester -diagnose old.json new.json
```

输出报告中标注了新增、移除和变更的公开符号，帮助维护版本兼容性。

## 九、调试框架

### 9.1 源码调试

开发阶段使用源码直接编译的框架（非二进制），Xcode 可以正常断点调试。确保 Debug 构建不使用二进制分发。

### 9.2 二进制框架调试

二进制框架需要 dSYM 文件进行符号化。分发二进制框架时同时提供 dSYM 文件。消费方在 crash 报告中用 dSYM 符号化框架内部的堆栈。

### 9.3 框架内部日志

```swift
// 使用 os.Logger（系统统一日志）
import os

let logger = Logger(subsystem: "com.example.MyFramework", category: "network")

public func fetchUser(id: String) async throws -> User {
    logger.debug("Fetching user \(id, privacy: .public)")
    // ...
    logger.error("Failed to fetch user: \(error.localizedDescription)")
}
```

`os.Logger` 不会暴露框架内部实现，消费方可以通过 Console.app 或 Console 命令查看框架日志。
