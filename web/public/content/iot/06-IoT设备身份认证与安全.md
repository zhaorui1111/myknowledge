# 06 · IoT 设备身份认证与安全

## 一、概述

IoT 设备的安全是整个物联网体系的基石。一台 IoT 设备从出厂到运行，面临多种安全威胁：伪造设备冒充合法身份、中间人窃听通信、固件被篡改、密钥泄露等。设备身份认证与安全机制确保每台设备是真实的、通信是加密的、数据是完整的。

## 二、威胁模型

IoT 设备面临的主要安全威胁：

| 威胁 | 描述 | 后果 |
|------|------|------|
| 设备克隆 | 攻击者复制设备身份信息，制造假冒设备 | 假设备接入网络，伪造数据 |
| 中间人攻击 | 攻击者劫持设备与云端通信 | 窃取数据，注入恶意指令 |
| 重放攻击 | 攻击者捕获并重放合法消息 | 触发非预期操作 |
| 固件篡改 | 攻击者替换设备固件 | 植入后门，控制设备 |
| 密钥泄露 | 设备密钥被提取或泄露 | 通信加密失效 |
| 拒绝服务 | 攻击者发送大量请求 | 设备不可用 |
| 侧信道攻击 | 通过功耗、电磁等旁路信息提取密钥 | 密钥被还原 |

## 三、设备身份标识

### 3.1 设备唯一标识

每台 IoT 设备需要一个全球唯一的标识符：

- **MAC 地址**：网卡硬件地址，48 位。简单但不安全（可修改），不适合作为唯一安全标识。
- **序列号（SN）**：厂商分配的设备序列号，通常印在设备标签上。
- **UUID**：通用唯一标识符，128 位。可由芯片 ID 生成或出厂烧录。
- **设备证书公钥**：基于公钥基础设施（PKI）的设备标识，安全性最高。

### 3.2 一机一密（Per-Device Key）

**概念**：每台设备在出厂时烧录唯一的密钥对（公钥+私钥），私钥存储在安全硬件中，公钥由厂商云端记录。

**流程**：
```
出厂阶段:
  厂商生成 → 密钥对(公钥P, 私钥S)
  私钥S → 烧录到设备安全存储(如 SE/TPM/eFuse)
  公钥P + 设备ID → 录入云端数据库
  设备证书 = 签名(设备ID, 公钥P, 有效期, CA签名)

设备认证阶段:
  设备 → 向云端出示证书
  云端 → 验证CA签名，提取设备ID和公钥P
  云端 → 发送随机挑战 nonce
  设备 → 用私钥S签名 nonce → 签名值
  云端 → 用公钥P验签 → 认证成功
```

**优势**：即使一台设备的密钥泄露，不影响其他设备。攻击者无法从公钥推导私钥。

### 3.3 安全存储

设备密钥的存储方式决定了安全性：

| 存储方式 | 安全性 | 成本 | 适用场景 |
|---------|--------|------|---------|
| Flash 明文存储 | 最低 | 最低 | 低端设备 |
| Flash 加密存储 | 中 | 低 | 中端设备 |
| eFuse（一次性熔丝） | 高 | 低 | 中端设备 |
| Secure Element (SE) | 最高 | 高 | 高安全设备 |
| TPM (可信平台模块) | 最高 | 高 | 网关、工控设备 |
| TEE (可信执行环境) | 高 | 中 | 支持 TrustZone 的芯片 |

## 四、认证协议

### 4.1 挑战-响应认证

最基本的设备认证方式：

```
云端                              设备
  |  1. 发送随机挑战 nonce            |
  |--------------------------------->|
  |                                  | 2. 用私钥签名 nonce
  |                                  |    signature = Sign(SK, nonce)
  |<--- signature -------------------|
  |                                  |
  | 3. 用设备公钥验签                 |
  |    Verify(PK, nonce, signature)  |
  |                                  |
  | 4. 验签成功 → 设备认证通过        |
```

### 4.2 TLS 双向认证

IoT 设备与云端通信时使用 TLS 加密，双向认证（mTLS）确保双方身份：

```
设备                               云端
  |<--- ServerHello + Certificate ---|  (云端出示服务器证书)
  |                                  |
  |--- ClientHello + Certificate --->|  (设备出示设备证书)
  |                                  |
  |  双方验证对方证书                  |
  |  协商TLS会话密钥                   |
  |                                  |
  |<=== 加密TLS通信 =================>|
```

设备证书由厂商 CA 签发，云端信任该 CA 的根证书。连接建立时，双方互相验证证书链。

**TLS 优化**：
- IoT 设备通常使用 TLS 1.2 或 1.3。
- TLS 1.3 减少了握手往返次数（1-RTT 甚至 0-RTT），降低延迟和功耗。
- 使用 ECDHE-ECDSA 套件，兼顾前向安全和非对称加密性能。
- 会话恢复（Session Resumption）避免重复握手。

### 4.3 JWT Token 认证

设备完成首次认证后，云端签发 JWT Token 用于后续通信：

```
JWT Header: {"alg": "ES256", "typ": "JWT"}
JWT Payload: {
  "device_id": "DEVICE_001",
  "issued_at": 1700000000,
  "expires_at": 1700086400,
  "scope": ["telemetry", "command"]
}
JWT Signature: ES256(header + "." + payload, cloud_private_key)
```

设备每次请求携带 JWT，云端验证签名和有效期。Token 过期后需要重新认证。

### 4.4 OAuth 2.0 设备授权流程

对于有用户交互的 IoT 设备（如智能音箱），可使用 OAuth 2.0 Device Flow：

```
1. 设备向授权服务器请求 device_code
2. 授权服务器返回 device_code 和 user_code
3. 设备显示 user_code（或二维码），引导用户在手机/浏览器上输入
4. 用户在手机上登录并授权
5. 设备轮询授权服务器，获取 access_token
6. 设备使用 access_token 访问 API
```

## 五、通信加密

### 5.1 传输层加密

| 协议 | 适用场景 | 特点 |
|------|---------|------|
| TLS | TCP 长连接 | 最常用，基于证书认证 |
| DTLS | UDP 通信 | TLS 的 UDP 版本，用于 CoAP |
| IPSec | VPN/网关 | 网络层加密，透明传输 |

### 5.2 应用层加密

在传输层加密之外，可增加应用层加密实现端到端安全：

**AES-GCM 对称加密**：
- 使用 128 位或 256 位 AES 密钥。
- GCM 模式同时提供加密和完整性保护（AEAD）。
- 密钥通过设备认证时协商或预置。

**混合加密**：
- 用非对称加密（RSA/ECC）交换对称密钥。
- 用对称加密（AES）加密实际数据。
- 兼顾安全性和性能。

## 六、固件安全

### 6.1 安全启动

安全启动确保设备只运行厂商签名的固件：

```
Boot ROM (不可修改)
  → 验证 Bootloader 签名
    → 验证 Kernel/Firmware 签名
      → 验证 Application 签名
        → 启动完成
```

每一级启动前验证下一级的数字签名，验证失败则停止启动。签名验证使用烧录在 eFuse 或 OTP 中的厂商公钥哈希。

### 6.2 固件签名

```
固件发布流程:
  1. 编译固件 → firmware.bin
  2. 计算哈希 → hash = SHA256(firmware.bin)
  3. 签名 → signature = RSA-Sign(firmware_private_key, hash)
  4. 打包 → firmware_package = {firmware.bin, signature, metadata}

设备升级流程:
  1. 下载固件包
  2. 计算固件哈希 → hash = SHA256(firmware.bin)
  3. 验签 → RSA-Verify(firmware_public_key, hash, signature)
  4. 验签通过 → 写入 Flash，重启
  5. 验签失败 → 拒绝升级
```

### 6.3 回滚保护

防止攻击者刷入旧版本固件利用已知漏洞：
- 设备存储一个单调递增的固件版本计数器（防回滚计数器）。
- 每次升级时，新固件版本必须大于等于当前计数器。
- 计数器存储在 eFuse 或受保护的安全存储中，只能增加不能减少。

## 七、常见 IoT 安全框架与标准

### 7.1 Matter 安全

Matter 使用基于 PKI 的设备认证证书（DAC），PASE/SPAKE2+ 配网认证，CASE 会话加密。详见 Matter 配网章节。

### 7.2 Amazon FreeRTOS / AWS IoT Core

- 设备使用 X.509 证书进行 mTLS 认证。
- AWS IoT 策略控制设备对 MQTT Topic 的访问权限。
- 支持 Job 文档进行 OTA 升级。

### 7.3 AliOS Things / 阿里云 IoT

- 使用设备三元组（ProductKey, DeviceName, DeviceSecret）认证。
- 基于 HMAC-SHA256 签名生成 MQTT 连接凭证。
- 支持一机一密和一型一密两种模式。

### 7.4 ARM Platform Security Architecture (PSA)

ARM 提出的 IoT 安全架构框架：
- **PSA Root of Trust (RoT)**：硬件安全根，包括安全启动、密钥存储。
- **隔离机制**：安全世界（Secure World）和非安全世界（Non-secure World）隔离。
- **加密服务**：统一的加密 API，支持硬件加速。

## 八、最佳实践

1. **一机一密**：每台设备烧录唯一密钥，避免一密多机。
2. **硬件安全存储**：私钥存储在 SE/eFuse/TEE 中，防止软件提取。
3. **安全启动**：实现全链路安全启动，防止固件篡改。
4. **TLS 加密通信**：所有设备与云端的通信使用 TLS/DTLS 加密。
5. **最小权限**：设备只能访问必要的资源和 Topic。
6. **固件签名**：所有固件必须签名验证后才能升级。
7. **防回滚**：防止固件降级到有漏洞的版本。
8. **密钥轮换**：定期更新通信密钥，降低密钥泄露风险。
9. **安全审计**：记录设备认证、固件升级等关键事件，便于追溯。
10. **漏洞响应**：建立固件安全更新机制，及时修补漏洞。

## 九、总结

IoT 设备安全是一个系统性工程，涵盖设备身份标识、密钥管理、认证协议、通信加密、固件安全等多个层面。核心原则是：设备有唯一可信的身份（PKI 证书），通信全程加密（TLS/DTLS），固件签名验证（安全启动），密钥安全存储（硬件安全模块）。没有银弹，需要根据设备成本和安全需求选择合适的安全方案。
