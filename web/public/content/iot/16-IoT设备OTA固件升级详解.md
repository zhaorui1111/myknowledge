# 16 · IoT 设备 OTA 固件升级详解

## 一、概述

OTA（Over-The-Air）固件升级是 IoT 设备生命周期管理的关键能力。它允许通过无线方式远程更新设备固件，修复漏洞、添加功能、优化性能。一个可靠的 OTA 机制需要解决多个工程挑战：断电安全、回滚保护、差分升级、多设备同步等。

## 二、OTA 架构

### 2.1 基本架构

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Cloud   │     │  CDN     │     │ Gateway/ │     │  Device  │
│  Server  │     │          │     │  Hub     │     │          │
└────┬─────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
     |                |                |                |
     | 1. 发布固件     |                |                |
     |--------------->|                |                |
     |                |                |                |
     | 2. 通知升级     |                |                |
     | (MQTT/HTTP)    |                |                |
     |--------------------------------->|                |
     |                |                |                |
     |                | 3. 下载固件     |                |
     |                |<---------------|                |
     |                |                |                |
     |                | 4. 分发固件     |                |
     |                |                |--------------->|
     |                |                |                |
     |                |                | 5. 验证+写入    |
     |                |                |                |
     |                |                | 6. 重启+验证    |
     |                |                |                |
     |<---------------------------------| 7. 上报结果    |
```

### 2.2 升级模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 推送式 (Push) | 云端主动通知设备升级 | 大规模批量升级 |
| 拉取式 (Pull) | 设备定期检查是否有新版本 | 低功耗设备 |
| 网关中转 | 云端→网关→设备 | Zigbee/Thread 子设备 |
| 直连下载 | 设备直接从 CDN 下载 | WiFi/Ethernet 设备 |

## 三、固件包结构

一个完整的固件包通常包含：

```
┌───────────────────────────────────┐
│         Firmware Package           │
├───────────────────────────────────┤
│ Header:                            │
│   - Magic Number (0xCAFEBABE)     │
│   - Version (major.minor.patch)   │
│   - Hardware Revision             │
│   - Firmware Size                 │
│   - Firmware Hash (SHA-256)       │
│   - Signature (RSA-2048/ECDSA)    │
│   - Build Timestamp               │
│   - Minimum Version (防回滚)       │
├───────────────────────────────────┤
│ Firmware Image:                    │
│   - Bootloader (可选)              │
│   - Kernel / RTOS                 │
│   - Root Filesystem               │
│   - Application                   │
├───────────────────────────────────┤
│ Metadata:                          │
│   - Release Notes                 │
│   - Rollback Flag                 │
│   - Compatibility List            │
└───────────────────────────────────┘
```

## 四、A/B 分区方案

A/B 分区（双分区）是最可靠的 OTA 方案：

### 4.1 分区布局

```
Flash 布局:
┌─────────────┬──────────────┬──────────────┬──────────┐
│ Bootloader  │ Partition A  │ Partition B  │ Storage  │
│ (只读)      │ (活跃)        │ (备用)       │ (配置)   │
└─────────────┴──────────────┴──────────────┴──────────┘

当前状态: A 活跃, B 备用
```

### 4.2 升级流程

```
1. 当前运行在 Partition A
   Bootloader 标志: A=Active, B=Inactive

2. 下载新固件，写入 Partition B
   下载过程中 A 继续正常运行

3. 验证 Partition B 固件
   - 校验 Hash
   - 验证签名
   - 验证失败 → 丢弃 B，保持 A 运行

4. 切换活跃分区
   Bootloader 标志: A=Inactive, B=Active(Pending)

5. 重启
   Bootloader 检测到 B=Pending
   - 标记 B=Trying
   - 从 B 启动

6. 新固件运行
   - 如果运行正常: 
     标记 B=Active, A=Inactive
     升级成功
   
   - 如果运行异常(看门狗超时/崩溃):
     Bootloader 检测到 B=Trying 但未确认
     → 回滚到 A=Active
     → B 标记为 Bad
```

### 4.3 优势

- **断电安全**：下载过程中断电不影响当前运行的 A 分区。
- **自动回滚**：新固件启动失败自动回滚到旧版本。
- **无缝升级**：下载新固件时设备正常工作，无需停机。

### 4.4 缺点

- 需要 2 倍的 Flash 空间存储固件。
- 对于 Flash 空间受限的低端设备可能不适用。

## 五、单分区方案

对于 Flash 空间受限的设备，使用单分区 + 临时存储：

```
Flash 布局:
┌─────────────┬──────────────┬──────────────┐
│ Bootloader  │ Application  │ Download     │
│ (只读)      │ (活跃)        │ Buffer       │
└─────────────┴──────────────┴──────────────┘

升级流程:
1. 下载新固件到 Download Buffer (需要足够大小)
2. 校验固件完整性
3. 擦除 Application 区域
4. 将 Download Buffer 的固件写入 Application
5. 校验写入后的固件
6. 重启

风险:
- 步骤 3-4 期间断电 → 设备变砖
- 需要额外的 Download Buffer 空间
- 通常通过 Bootloader 提供恢复机制
```

## 六、差分升级（Delta Update）

### 6.1 原理

差分升级只传输新旧固件之间的差异部分，大幅减少下载量：

```
完整升级:
  传输 512KB 完整固件

差分升级:
  传输 20KB diff patch
  设备端: 旧固件 + patch → 新固件
  传输量减少 96%
```

### 6.2 差分算法

| 算法 | 原理 | 压缩率 | 适用场景 |
|------|------|--------|---------|
| bsdiff | 二进制差分 + bzip2 压缩 | 高 | 通用 |
| xdelta3 | Rsync 算法 + VCDIFF | 中高 | 大文件 |
| courgette | Chromium 的差分工具 | 极高 | 含大量地址偏移的二进制 |
| detools | Python 差分工具 | 高 | 嵌入式 |

### 6.3 差分升级流程

```
云端:
  1. 比较 v1.0 和 v1.1 固件
  2. 生成 delta patch (v1.0 → v1.1)
  3. 发布 patch 包

设备端:
  1. 检查当前版本 (v1.0)
  2. 下载对应的 delta patch
  3. 从 Flash 读取当前固件
  4. 应用 patch: 新固件 = ApplyPatch(旧固件, patch)
  5. 校验新固件 Hash
  6. 写入 Flash
```

### 6.4 注意事项

- 设备端需要足够的 RAM 来执行 patch 操作。
- 需要确保设备当前版本与 patch 的源版本匹配。
- Patch 应用失败需要回退到完整升级。

## 七、固件签名与验证

### 7.1 签名流程

```
固件发布:
  1. 编译固件 → firmware.bin
  2. 计算哈希 → hash = SHA-256(firmware.bin)
  3. 用私钥签名 → sig = ECDSA-Sign(private_key, hash)
  4. 打包 → {firmware.bin, sig, metadata, public_key_cert}

设备验证:
  1. 下载固件包
  2. 计算固件哈希 → hash = SHA-256(firmware.bin)
  3. 验证证书链 → 检查 public_key_cert 由信任的 CA 签发
  4. 验证签名 → ECDSA-Verify(public_key, hash, sig)
  5. 验证通过 → 写入 Flash
  6. 验证失败 → 拒绝升级，上报错误
```

### 7.2 防回滚

- 固件元数据中包含 `min_version` 字段。
- 设备存储当前固件版本号（防回滚计数器，通常在 eFuse 或受保护存储中）。
- 升级时检查：新版本 >= 当前版本。
- 防回滚计数器只能递增，不能递减。

## 八、低功耗设备 OTA

### 8.1 分块下载

对于电池供电的低功耗设备（如 Zigbee/Thread 传感器），固件下载需要分块进行：

```
Gateway                          Device (Sleeping)
  |                                 |
  | 1. 通知有新固件 (MQTT/通知)      |
  |                                 |
  | 2. 设备唤醒，请求固件块          |
  |<-- Request Block 0 (4KB) -------|
  |--- Block 0 ------------------->|
  |                                 |
  | (设备睡眠，下次唤醒继续)          |
  |                                 |
  |<-- Request Block 1 (4KB) -------|
  |--- Block 1 ------------------->|
  |                                 |
  | ... 重复直到所有块下载完成 ...    |
  |                                 |
  | 3. 所有块下载完成               |
  |    校验+写入+重启               |
```

### 8.2 优化策略

- **压缩固件**：使用 LZMA/Gzip 压缩固件，减少传输量。
- **差分升级**：只传输差异部分。
- **分块大小**：根据设备 RAM 和网络 MTU 选择合适的块大小（通常 256B - 4KB）。
- **断点续传**：记录已下载的块，重连后从断点继续。
- **多设备共享**：网关缓存固件，多个设备从网关下载（减少云端带宽）。

## 九、OTA 管理策略

### 9.1 灰度发布

```
灰度发布流程:
  Phase 1: 1% 设备升级 → 观察 24h
  Phase 2: 10% 设备升级 → 观察 24h
  Phase 3: 50% 设备升级 → 观察 24h
  Phase 4: 100% 设备升级

每阶段检查:
  - 升级成功率
  - 设备在线率
  - 崩溃率
  - 功能异常报告
  
异常处理:
  - 成功率 < 95% → 暂停发布
  - 崩溃率 > 1% → 暂停发布 + 回滚已升级设备
```

### 9.2 设备分组

- 按地域分组：不同区域分批发送。
- 按设备型号分组：不同硬件版本使用不同固件。
- 按固件版本分组：从特定版本升级时使用差分包。
- 按网络条件分组：网络好的设备先升级。

### 9.3 升级窗口

- 避免在业务高峰期升级。
- 智能家居设备在夜间升级（减少用户感知）。
- 工业设备在维护窗口升级。
- 设置升级超时时间，超时后自动取消。

## 十、OTA 协议与框架

### 10.1 LwM2M FOTA

OMA LwM2M 定义了标准化的固件升级对象（Object 5）：

```
LwM2M Firmware Update Object (OID: 5):
  - Package (资源0): 固件包 (Write)
  - Package URI (资源1): 固件下载URL (Write)
  - Update (资源2): 触发升级 (Execute)
  - State (资源3): 升级状态 (Read)
    0=Idle, 1=Downloading, 2=Downloaded, 3=Updating
  - Update Result (资源5): 升级结果 (Read)
    0=Default, 1=Success, 2=NoSpace, 3=OOM, ...
  - PkgName (资源6): 固件名称
  - PkgVersion (资源7): 固件版本
  - Cancel (资源8): 取消升级 (Execute)
```

### 10.2 平台方案

| 平台 | OTA 方案 | 特点 |
|------|---------|------|
| AWS IoT Jobs | Job Document + MQTT | 与 AWS 生态集成 |
| Azure Device Update | ADU Agent + IoT Hub | 支持 A/B 和 Image 方案 |
| Google Android Things | OTA Service | 已停止维护 |
| 阿里云 IoT | OTA API + MQTT | 差分+完整升级 |
| 自建 | MQTT/HTTP + 自定义协议 | 完全可控 |

## 十一、常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 升级失败变砖 | 写入中断/断电 | A/B 分区 + 自动回滚 |
| 升级后设备离线 | 新固件 bug | 看门狗 + 自动回滚 |
| 下载超时 | 网络不稳定 | 断点续传 + 重试 |
| 内存不足 | 固件过大 | 差分升级 + 流式写入 |
| 版本混乱 | 多版本共存 | 严格版本管理 + 差分包匹配 |
| 安全风险 | 固件被篡改 | 签名验证 + 安全传输 |

## 十二、总结

IoT OTA 固件升级是一个需要极高可靠性的系统工程。A/B 分区方案是最安全的升级策略，提供了断电安全和自动回滚能力。差分升级技术大幅减少了传输量，特别适合低带宽和低功耗设备。固件签名验证是防止固件篡改的安全基石。灰度发布策略是降低大规模升级风险的有效手段。一个设计良好的 OTA 系统应该做到：不丢设备（断电安全）、不砖设备（自动回滚）、不泄密（签名验证）。
