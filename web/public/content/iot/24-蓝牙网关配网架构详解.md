# 24 · 蓝牙网关配网架构详解

## 一、概述

前面三篇分别讲了纯蓝牙设备的配网（第 22 篇）和蓝牙 Mesh 设备的配网（第 23 篇）。这两种模式有一个共同局限：设备必须通过手机 BLE 直连才能配网和控制，离开手机蓝牙范围就"失联"。而**蓝牙网关**正是解决这个问题的关键——它常驻在固定位置，持续与蓝牙设备保持通信，并把数据转发到云端，让用户随时随地远程控制设备。

蓝牙网关在配网体系中扮演双重角色：它自身需要完成**网关配网**（接入 Wi-Fi/以太网并绑定账号），同时它又作为**配网代理**，帮助蓝牙设备完成入网和云端绑定。本文系统讲解蓝牙网关的配网架构、网关自身的配网流程、网关代理蓝牙设备配网的两种模式（纯蓝牙设备代理配网和 Mesh Proxy 配网），以及三方绑定关系的管理。

## 二、蓝牙网关的角色与架构

### 2.1 网关在配网体系中的角色

```
                         ┌─────────┐
                         │  云平台  │
                         └────┬────┘
                              │ HTTPS/MQTT
                    ┌─────────┴─────────┐
                    │   蓝牙网关          │

        │ (已配网)   │   │ (Proxy) │   │ (新设备)   │
```

## 三、网关自身的配网

蓝牙网关本身也是一个 IoT 设备，需要先完成自己的配网——接入 Wi-Fi 并绑定用户账号，然后才能代理其他蓝牙设备。

### 3.1 网关配网流程

```
阶段 1: 网关接入 Wi-Fi
  ┌───────────────────────────────────────┐
  │ 方式A: BLE Provisioning (推荐)         │
  │   手机 App ──BLE──→ 网关 (传Wi-Fi凭证) │
  │   网关连接 Wi-Fi                      │
  ├───────────────────────────────────────┤
  │ 方式B: AP 模式配网                     │
  │   网关开启 AP → 手机连接 → 传 Wi-Fi   │
  │   网关连接 Wi-Fi                      │
  ├───────────────────────────────────────┤
  │ 方式C: 以太网                          │
  │   网关插网线 → DHCP 获取 IP           │
  └───────────────────────────────────────┘

阶段 2: 网关绑定用户账号
  手机 App                     云平台                    网关
    │                           │                         │
    │ 1. 发现网关 (mDNS/局域网)  │                         │
    ├──→ 网关 (局域网通信)       │                         │
    │   获取网关ID/MAC           │                         │
    │                           │                         │
    │ 2. 绑定请求                │                         │
    │  POST /gateway/bind        │                         │
    │  {gatewayId, userId,       │                         │
    │   bindingToken}            │                         │
    ├──────────────────────────→│                         │
    │                           │ 3. 验证 Token            │
    │                           │ 4. 创建绑定记录           │
    │ 5. 绑定成功                │                         │
    │←──────────────────────────┤                         │
    │                           │                         │
    │ 6. 下发云端凭证             │                         │
    │  (MQTT broker 地址,        │                         │
    │   设备证书, Token)          │                         │
    ├──────────────────────────→├────────────────────────→│
    │                           │  网关保存云端凭证         │
    │                           │                         │
    │                           │  7. 网关连接云端 MQTT     │
    │                           │←────────────────────────│
    │                           │  MQTT Connect (TLS)      │
    │                           │  (网关在线)              │
```

### 3.2 网关配网安全

网关配网需要比普通蓝牙设备更高的安全级别，因为网关持有所有子设备的密钥：

- **Wi-Fi 凭证传输加密**：BLE Provisioning 时使用 BLE 加密通道传输 Wi-Fi 密码。
- **绑定认证**：网关生成 bindingToken（类似纯蓝牙设备的机制），防止伪造绑定。
- **云端证书**：网关使用设备证书进行 MQTT 双向 TLS 认证。
- **本地通信加密**：App 与网关的局域网通信使用 TLS。

## 四、模式一：网关代理纯蓝牙设备配网

这是最常见的场景——用户通过网关给蓝牙设备配网，无需手机在设备旁边。

### 4.1 架构

```
手机 App                     蓝牙网关                   蓝牙设备
  │                            │                          │
  │ 1. 远程触发配网              │                          │
  │  (通过云端下发配网命令)       │                          │
  ├─────── 云平台 ─────────────→│                          │
  │                            │                          │
  │                            │ 2. 网关扫描 BLE 设备       │
  │                            │  (扫描待配网广播)          │
  │                            │                          │
  │                            │ 3. 网关连接设备            │
  │                            ├─────────────────────────→│
  │                            │                          │
  │                            │ 4. BLE 配对认证            │
  │                            │←─────────────────────────→│
  │                            │  (LE Secure Connections)  │
  │                            │                          │
  │                            │ 5. GATT 服务发现           │
  │                            │←─────────────────────────→│
  │                            │                          │
  │                            │ 6. 配置数据交换            │
  │                            │  (下发网关凭证, 采集配置)   │
  │                            ├─────────────────────────→│
  │                            │                          │
  │ 7. 云端绑定                 │                          │
  │  (网关代为发起)              │                          │
  │←──── 云平台 ────────────────┤                          │
  │  POST /device/bind          │                          │
  │  {deviceId, gatewayId,      │  8. 设备绑定到网关         │
  │   userId, bindingToken}     │  (设备记录网关为管理者)    │
  │                            │                          │
  │ 9. 绑定成功通知              │                          │
  │←──── 云平台 ────────────────┤                          │
  │                            │                          │
  │    配网完成, 设备通过网关上云  │                          │
```

### 4.2 三方绑定关系

网关代理配网后，形成**用户-网关-设备**三方绑定关系：

```
云端绑定记录:
{
  "userId": "user_123",          // 设备所有者
  "gatewayId": "GW_AABBCC",     // 管理网关
  "deviceId": "BLE_00112233",   // 蓝牙设备
  "bindingType": "gateway_proxy", // 绑定类型: 网关代理
  "deviceKey": {                 // 设备加密密钥 (网关持有)
    "irk": "...",                // IRK (解析RPA)
    "ltk": "...",                // LTK (连接加密)
    "address": "AA:BB:CC:DD:EE:FF"
  },
  "gatewayAuth": true,           // 网关有管理权限
  "status": "online_via_gateway" // 通过网关在线
}
```

三方绑定的权限关系：
- **用户**：设备所有者，可通过 App 远程控制。
- **网关**：设备管理者，持有设备密钥，负责日常通信。
- **设备**：只与网关通信，不知道用户的存在。

### 4.3 网关代理配网的认证问题

网关代理配网时，认证方式受到限制——网关通常没有屏幕和按键，无法进行需要用户交互的认证（如 Passkey Entry、Numeric Comparison）：

| 认证方式 | 网关代理可用性 | 说明 |
|---------|--------------|------|
| Just Works | 可用 | 无需交互，但安全性低 |
| Passkey Entry | 不可用 | 网关无输入能力 |
| Numeric Comparison | 不可用 | 网关无显示能力 |
| OOB (NFC) | 可用 | 网关有 NFC 模块时 |
| OOB (二维码) | 间接可用 | 用户在 App 扫码，传给网关 |

**推荐方案：App 辅助认证**

```
认证流程 (App 辅助 OOB):
  1. 网关扫描发现设备
  2. 网关获取设备的 OOB 信息 (如二维码内容)
  3. 网关通过云端通知 App
  4. App 显示"请扫描设备上的二维码"
  5. 用户扫描二维码, App 获取 OOB 数据
  6. App 通过云端将 OOB 数据传给网关
  7. 网关使用 OOB 数据完成配对认证
```

### 4.4 网关代理数据转发

配网完成后，网关持续与设备通信并转发数据：

```
数据上行 (设备 → 云端):
  蓝牙设备                    网关                     云平台
    │  BLE Notification        │                         │

    │                         ├────────────────────────→│
    │                         │                         │

数据下行 (云端 → 设备):
  云平台                     网关                     蓝牙设备

    │                         │  BLE Write               │

```

## 五、模式二：网关作为 Mesh Proxy

蓝牙 Mesh 设备通过网关的 Proxy 功能与云端通信。

### 5.1 Mesh Proxy 配网流程

```
手机 App                     蓝牙网关                 Mesh 设备
  │                            │                        │
  │ 1. 触发 Mesh 配网           │                        │
  │  (通过云端或局域网)          │                        │
  ├───────────────────────────→│                        │
  │                            │                        │
  │                            │ 2. 扫描 Unprovisioned   │
  │                            │    Device Beacon        │
  │                            │                        │
  │                            │ 3. GATT 连接 + 配网协议 │
  │                            ├───────────────────────→│
  │                            │  (完整的 9 步配网)       │
  │                            │                        │
  │                            │ 4. 节点配置              │
  │                            │  (AppKey, 订阅, 发布)    │
  │                            ├───────────────────────→│
  │                            │                        │
  │ 5. 云端记录                 │                        │
  │←───────────────────────────┤                        │
  │  (网关上报节点信息)          │                        │
  │                            │                        │
  │                            │ 6. 网关作为 Proxy 节点   │
  │                            │  持续与 Mesh 网络通信    │
```

### 5.2 Mesh Proxy 数据转发

网关作为 Mesh Proxy 节点，在 Mesh 网络和云端之间转发消息：

```
Mesh 网络                   网关 (Proxy)               云平台
    │                          │                          │
    │ 1. Mesh 消息              │                          │
    │  (广播承载)               │                          │
    │  如: 温度传感器上报        │                          │
    ├─────────────────────────→│                          │
    │                          │ 2. 解析 Mesh PDU          │
    │                          │  提取应用数据              │
    │                          │                          │
    │                          │ 3. MQTT 上报              │
    │                          ├─────────────────────────→│
    │                          │  /mesh/{netId}/data       │
    │                          │                          │
    │                          │ 4. 云端命令               │
    │                          │←─────────────────────────┤
    │                          │  MQTT: /mesh/{netId}/cmd  │
    │                          │                          │
    │                          │ 5. 封装为 Mesh PDU        │
    │                          │  通过 GATT/广播发送        │
    │←─────────────────────────┤                          │
    │                          │                          │

Proxy 过滤:
  网关可设置 Proxy Filter, 只转发特定地址的消息:
  - 白名单: 只转发指定 Group/Unicast 地址的消息
  - 减少不必要的云端流量
```

### 5.3 Mesh Proxy 的持续在线

与纯 BLE 设备不同，Mesh 设备不维护与网关的持续 GATT 连接。Mesh 设备通过广播发送消息，网关作为 Relay/Proxy 节点接收并转发：

```
Mesh 设备通信方式:
  - 广播承载: 设备发送 Mesh 消息到广播通道
  - 网关监听广播, 接收 Mesh PDU
  - 网关通过 GATT Proxy 或直接解析转发到云端

  vs 纯 BLE 设备:
  - 需要 GATT 连接, 网关主动连接设备
  - 连接数有限 (一般 4-8 个并发连接)
  - Mesh 无连接数限制 (广播方式)
```

## 六、网关的多设备管理

### 6.1 连接管理

纯 BLE 模式下，网关需要管理多个 BLE 连接：

```
网关 BLE 连接池:
  ┌─────────────────────────────────────────┐
  │  连接1: 温度计 (Interval: 200ms)         │
  │  连接2: 门锁 (Interval: 1000ms)          │
  │  连接3: 体重秤 (按需连接)                 │
  │  连接4: 传感器 (Interval: 500ms)         │
  │  ...                                     │
  │  连接N: 空闲 (等待新设备)                 │
  └─────────────────────────────────────────┘

  最大并发连接数: 取决于 BLE 芯片 (通常 4-20 个)
  超过上限时:
  - 按优先级断开低优先级连接
  - 轮询连接 (Round Robin)
  - 按需连接 (事件触发时才连接)
```

**连接策略：**

| 设备类型 | 连接策略 | 说明 |
|---------|---------|------|
| 门锁/安防 | 常连接 | 需要实时响应 |
| 温度/湿度传感器 | 周期连接 | 每 N 分钟连接一次采集数据 |
| 体重秤 | 按需连接 | 测量事件触发连接 |
| 蓝牙标签 | 不连接 | 仅扫描广播包 |

### 6.2 广播扫描管理

对于不需要 GATT 连接的设备（如蓝牙标签、iBeacon），网关持续扫描广播：

```
扫描策略:
  - 扫描窗口: 50% 占空比 (如 500ms 开 / 500ms 关)
  - 过滤: 只解析已绑定设备的广播
  - 上报: 批量上报 (每 30 秒打包一次), 减少 MQTT 流量

  广播数据处理:
  - iBeacon: 提取 UUID/Major/Minor/RSSI → 上报位置/距离
  - Eddystone: 提取 URL/Telemetry → 上报
  - 厂商自定义: 按厂商协议解析 → 上报
```

## 七、网关代理配网的优势与局限

### 7.1 优势

| 优势 | 说明 |
|------|------|
| 远程配网 | 用户不在蓝牙范围内也能配网设备 |
| 持续在线 | 网关常驻，设备始终在线（通过网关） |
| 远程控制 | 通过云端随时随地控制设备 |
| 自动化 | 网关可执行本地自动化（如传感器触发灯控） |
| 多设备管理 | 一个网关管理多个蓝牙设备 |
| Mesh 扩展 | 作为 Mesh Proxy 扩展 Mesh 网络的云连接 |

### 7.2 局限

| 局限 | 说明 |
|------|------|
| 连接数限制 | 纯 BLE 模式下网关并发连接数有限（4-20） |
| 范围限制 | 设备必须在网关 BLE 范围内（10-50m） |
| 认证限制 | 网关通常无法进行需要用户交互的配对认证 |
| 功耗 | 网关需要常电供电 |
| 延迟 | 云端控制增加 100-300ms 延迟 |
| 单点故障 | 网关故障则所有子设备离线 |

## 八、网关配网的典型场景

### 8.1 智能家居蓝牙网关

```
场景: 用户购买了蓝牙门锁和蓝牙温湿度计

配网流程:
  1. 用户先配网网关 (BLE Provisioning 接入 Wi-Fi)
  2. App 绑定网关到用户账号
  3. 用户在 App 中添加门锁
     → App 通过云端通知网关扫描配网
     → 网关发现门锁, 使用 Just Works 配对
     → 网关代为完成云端绑定
     → 用户可在 App 远程开锁
  4. 用户在 App 中添加温湿度计
     → 网关发现温湿度计, 配对连接
     → 网关代为完成云端绑定
     → 温湿度数据自动上报

  日常使用:
  - 门锁事件 → 网关 → 云端 → App 推送
  - App 远程开锁 → 云端 → 网关 → 门锁
  - 温湿度定时上报 → 网关 → 云端 → App 显示
```

### 8.2 Mesh 照明网关

```
场景: 办公楼蓝牙 Mesh 照明系统

部署流程:
  1. 安装网关 (以太网连接, 多个网关覆盖不同楼层)
  2. 网关作为 Provisioner 配网所有灯具
  3. 配置灯具的 Group 订阅 (按楼层/区域分组)
  4. 配置传感器的发布目标 (Group 地址)
  5. 网关作为 Proxy 节点连接 Mesh 网络

  日常使用:
  - 传感器检测到人 → Mesh 消息 → 网关 → 云端记录
  - 云端下发调光命令 → 网关 → Mesh 消息 → 灯具执行
  - 网关上报能耗数据 → 云端分析
  - 本地自动化: 传感器 → 网关规则引擎 → 直接控制灯具 (无需云端)
```

### 8.3 蓝牙资产标签网关

```
场景: 仓库资产追踪

部署:
  1. 仓库各区域安装网关 (Wi-Fi + BLE)
  2. 资产贴蓝牙标签 (iBeacon 广播)
  3. 标签无需配网, 只需在云端注册 UUID

  运行:
  - 标签持续广播 iBeacon
  - 网关扫描广播, 记录 UUID + RSSI
  - 网关上报到云端 (标签在哪些网关附近, RSSI 多少)
  - 云端通过多网关 RSSI 三角定位计算资产位置
  - App 显示资产位置地图
```

## 九、网关代理配网的安全考虑

### 9.1 密钥安全

网关持有所有子设备的密钥，是安全的核心：

```
网关密钥存储:
  ┌──────────────────────────────────────┐
  │ 安全存储区 (Flash 加密 / TrustZone)   │
  │                                      │
  │ 网关自身:                             │
  │   - 云端 MQTT 证书/Token              │
  │   - Wi-Fi 密码                        │
  │                                      │
  │ 子设备密钥:                            │
  │   - 设备1: LTK, IRK, MAC              │
  │   - 设备2: LTK, IRK, MAC              │
  │   - ...                               │
  │                                      │
  │ Mesh 密钥:                            │
  │   - NetKey (可能多个子网)              │
  │   - AppKey (可能多个应用域)            │
  │   - DevKey (所有节点的设备密钥)        │
  └──────────────────────────────────────┘

安全要求:
  - 密钥存储在加密 Flash 或安全芯片中
  - 密钥不通过日志/调试接口暴露
  - 网关重置时清除所有密钥
  - 云端通信使用 TLS, 密钥不在传输中暴露
```

### 9.2 网关与云端的安全通道

```
安全通道设计:
  网关 ←── TLS 1.2+ (双向认证) ──→ 云平台

  认证:
  - 网关使用设备证书进行 TLS 双向认证
  - 证书存储在安全芯片中, 不可提取
  - 证书有有效期, 支持轮换

  数据加密:
  - 所有 MQTT 消息在 TLS 隧道中传输
  - 敏感数据 (如设备密钥) 额外应用层加密
  - 设备控制命令带时间戳和签名, 防重放
```

### 9.3 设备解绑与密钥更新

当设备从网关移除时，需要安全地清除密钥：

```
设备解绑流程:
  1. 用户在 App 中删除设备
  2. 云端通知网关: "解绑设备 X"
  3. 网关断开 BLE 连接
  4. 网关清除设备 X 的所有密钥 (LTK, IRK 等)
  5. 网关清除设备 X 的缓存数据
  6. 网关回复云端: "解绑完成"
  7. 云端删除绑定记录

  安全考虑:
  - 设备解绑后, 设备端也应清除网关的绑定信息
  - 如果设备不在范围, 网关单方面清除密钥
  - 设备恢复出厂设置后才能被新网关配网
```

## 十、与其他配网模式对比

| 维度 | 手机直连配网 (第22篇) | Mesh 配网 (第23篇) | 网关代理配网 (本文) |
|------|---------------------|-------------------|-------------------|
| 配网发起者 | 手机 App | Provisioner (手机/网关) | 手机 App → 网关代理 |
| 配网通道 | BLE 直连 | GATT 配网 Service | 网关 BLE → 设备 |
| 认证能力 | 全面 (Passkey/OOB) | ECDH + OOB | 受限 (Just Works/OOB) |
| 远程配网 | 不支持 | 不支持 (需在 BLE 范围) | 支持 (通过云端) |
| 持续在线 | 手机在范围才在线 | Mesh 网络内在线 | 通过网关始终在线 |
| 远程控制 | 不支持 | 需 Proxy 节点 | 支持 |
| 适用场景 | 个人穿戴/健康 | 大规模照明/楼宇 | 智能家居/远程管理 |
| 成本 | 最低 | 中 (需 Mesh 支持) | 中 (需网关硬件) |

## 十一、总结

蓝牙网关配网架构解决了纯蓝牙设备"离开手机就失联"的核心痛点。网关在配网体系中扮演三重角色：云端桥梁（BLE 设备 ←→ 云平台）、配网代理（帮助新设备完成配网和绑定）、Mesh Proxy（Mesh 网络与外部通信的桥梁）。网关自身的配网包括接入 Wi-Fi 和绑定用户账号两个阶段，完成后即可代理其他蓝牙设备配网。对于纯 BLE 设备，网关通过 GATT 连接完成配网并持续转发数据；对于 Mesh 设备，网关作为 Proxy 节点转发 Mesh 消息。网关代理配网形成了用户-网关-设备的三方绑定关系，网关持有所有子设备的密钥，因此密钥安全存储是核心安全要求。相比手机直连配网，网关代理配网支持远程配网和持续在线，是智能家居和楼宇自动化场景的推荐方案。
