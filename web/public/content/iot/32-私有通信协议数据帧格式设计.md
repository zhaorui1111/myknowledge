# 私有通信协议数据帧格式设计

## 概述

在 BLE、WiFi 等底层传输通道之上，IoT 设备与 App 之间需要一套应用层通信协议来交换结构化数据。协议帧格式的设计直接决定了通信的可靠性、扩展性和解析效率。本文从实际工程角度，系统讲解私有通信协议的数据帧格式设计原则、字段编码方式和工程实现。

## 协议分层模型

```
应用层    — 私有通信协议（本文重点）
  ↓ 封装为帧
传输层    — BLE GATT / TCP / UDP
  ↓ 传输
物理层    — BLE Radio / WiFi Radio
```

私有协议运行在传输层之上，负责定义数据的组织格式、命令体系和交互流程。一个好的协议帧格式需要满足以下要求：

**自描述性**：接收端无需额外上下文即可解析帧内容。

**可扩展性**：新增命令字段不需要破坏已有版本兼容性。

**容错性**：能检测和定位传输错误，支持重同步。

**高效性**：帧头开销小，解析速度快。

## 帧格式总体设计

### 推荐帧结构

```
┌─────────┬──────────┬─────────┬─────────┬───────────┬───────────┬──────────┬───────────┐
│ Preamble│ Magic    │ Version │ CmdID   │ SeqNum    │ PayloadLen│ Payload  │  CRC      │
│ 1 byte  │ 2 bytes  │ 1 byte  │ 1 byte  │ 2 bytes   │ 2 bytes   │ N bytes  │ 2 bytes   │
└─────────┴──────────┴─────────┴─────────┴───────────┴───────────┴──────────┴───────────┘

帧头 (Header): Preamble + Magic + Version + CmdID + SeqNum + PayloadLen = 9 bytes
帧尾 (Footer): CRC = 2 bytes
总开销: 11 bytes
```

### 字段详解

**Preamble（前导码，1 byte）**：帧起始标记，固定值 `0x55`。用于接收端在数据流中快速定位帧起始位置。前导码不是必须的，但有助于流式传输场景的帧同步。

**Magic Number（魔数，2 bytes）**：帧头标识，固定值 `0xAA55`（大端序为 `0xAA 0x55`）。用于区分有效帧和噪声数据。接收端通过搜索魔数来定位帧起始。选择 `0xAA55` 是因为它具有良好的位均衡性（0 和 1 交替），便于线路检测。

**Version（版本号，1 byte）**：协议版本，用于向前兼容。解析时根据版本号选择不同的解析逻辑。推荐使用高 4 位为主版本，低 4 位为次版本：`0x12` 表示 v1.2。

**CmdID（命令字，1 byte）**：标识本帧的命令类型。命令字空间规划：

```
0x00        — 保留
0x01-0x0F   — 系统命令（心跳、握手、版本查询等）
0x10-0x2F   — 设备管理命令（绑定、解绑、状态查询等）
0x30-0x4F   — 数据上报命令（传感器数据、事件通知等）
0x50-0x6F   — 控制命令（开关、调节、模式切换等）
0x70-0x8F   — 配置命令（参数读取、参数设置等）
0x90-0xAF   — OTA 命令（开始、数据、校验、重启等）
0xB0-0xCF   — 文件传输命令
0xD0-0xEF   — 诊断命令
0xF0-0xFE   — 厂商自定义
0xFF        — 保留
```

**SeqNum（序列号，2 bytes）**：帧序列号，用于请求-响应匹配和丢包检测。发送方维护递增计数器，接收方在响应帧中回填对应序列号。序列号回绕到 0。

**PayloadLen（负载长度，2 bytes）**：Payload 的字节数。大端序编码。最大 65535 字节，实际受传输层 MTU 限制。PayloadLen=0 表示无负载帧（如纯命令帧）。

**Payload（负载数据，0-N bytes）**：实际业务数据，格式由 CmdID 决定。详见下文 TLV 编码。

**CRC（循环冗余校验，2 bytes）**：CRC16-CCITT 校验，覆盖 Magic 到 Payload 的所有字节（不含 Preamble）。用于检测传输错误。

### 帧总长度计算

```
FrameLength = 1(Preamble) + 2(Magic) + 1(Version) + 1(CmdID)
            + 2(SeqNum) + 2(PayloadLen) + PayloadLen + 2(CRC)
            = 11 + PayloadLen
```

## Payload 编码：TLV 格式

### TLV 结构

Payload 内部采用 TLV（Type-Length-Value）格式组织字段，提供良好的扩展性：

```
┌────────┬────────┬───────────┐
│ Type   │ Length │ Value     │
│ 1 byte │ 1 byte │ N bytes   │
└────────┴────────┴───────────┘
```

**Type（类型，1 byte）**：字段标识符。不同 CmdID 下的 Type 空间独立。
**Length（长度，1 byte）**：Value 的字节数。0-255。
**Value（值，N bytes）**：字段值。

### TLV 优势

相比固定偏移的结构体编码，TLV 的好处是字段顺序无关、可选字段灵活、新增字段不破坏旧版本解析。旧版本解析器遇到未知的 Type 可以通过 Length 跳过。

### Payload 示例

以"设备状态上报"命令（CmdID=0x30）为例：

```
Payload:
┌────────┬────────┬────────────────┬────────┬────────┬──────────┬────────┬────────┬───────────────┐
│ Type 1 │ Len 1  │ Value: 温度     │ Type 2 │ Len 2  │ Value: 湿度│ Type 3 │ Len 1  │ Value: 电量    │
│ 0x01   │ 0x02   │ 0x00 0xFA (250) │ 0x02   │ 0x02   │ 0x01 0x2C │ 0x03   │ 0x01   │ 0x53 (83%)   │
└────────┴────────┴────────────────┴────────┴────────┴──────────┴────────┴────────┴───────────────┘

解析结果:
  温度 = 250 → 25.0°C (放大10倍)
  湿度 = 300 → 30.0% (放大10倍)
  电量 = 83%
```

## 数据类型编码规范

### 整数类型

统一使用大端序（Big-Endian）编码，确保跨平台一致性：

```swift
// UInt16 大端序编解码
func encodeUInt16BE(_ value: UInt16) -> Data {
    var be = value.bigEndian
    return Data(bytes: &be, count: 2)
}

func decodeUInt16BE(_ data: Data, offset: Int) -> UInt16 {
    data.subdata(in: offset..<offset+2).withUnsafeBytes {
        $0.load(as: UInt16.self).byteSwapped
    }
}

// UInt32 大端序编解码
func encodeUInt32BE(_ value: UInt32) -> Data {
    var be = value.bigEndian
    return Data(bytes: &be, count: 4)
}

func decodeUInt32BE(_ data: Data, offset: Int) -> UInt32 {
    data.subdata(in: offset..<offset+4).withUnsafeBytes {
        $0.load(as: UInt32.self).byteSwapped
    }
}
```

### 定点数编码

避免使用浮点数（浮点数在不同平台可能有精度差异），使用定点数代替：

```swift
// 温度编码：实际值 × 10，用 Int16 存储
// 范围: -3276.8°C ~ +3276.7°C，精度 0.1°C
func encodeTemperature(_ celsius: Double) -> Data {
    let raw = Int16(celsius * 10)
    return encodeInt16BE(raw)
}

func decodeTemperature(_ data: Data, offset: Int) -> Double {
    let raw = decodeInt16BE(data, offset: offset)
    return Double(raw) / 10.0
}
```

### 字符串编码

字符串使用 UTF-8 编码，Length 字段记录字节数（非字符数）：

```swift
func encodeString(_ str: String) -> Data {
    let bytes = Array(str.utf8)
    var data = Data()
    data.append(UInt8(bytes.count))
    data.append(contentsOf: bytes)
    return data
}

func decodeString(_ data: Data, offset: Int) -> (String, Int) {
    let len = Int(data[offset])
    let str = String(data: data.subdata(in: offset+1..<offset+1+len), encoding: .utf8) ?? ""
    return (str, 1 + len)  // 返回字符串和消耗的字节数
}
```

### 时间戳编码

使用 32 位 Unix 时间戳（UTC 秒级），大端序：

```swift
func encodeTimestamp(_ date: Date) -> Data {
    let timestamp = UInt32(date.timeIntervalSince1970)
    return encodeUInt32BE(timestamp)
}
```

## iOS 协议帧编解码实现

### 帧编码器

```swift
struct ProtocolFrame {
    static let preamble: UInt8 = 0x55
    static let magic: [UInt8] = [0xAA, 0x55]
    static let version: UInt8 = 0x10  // v1.0
    static let headerSize = 9
    static let crcSize = 2

    let cmdID: UInt8
    let seqNum: UInt16
    let payload: Data

    func encode() -> Data {
        var frame = Data()
        frame.append(Self.preamble)
        frame.append(contentsOf: Self.magic)
        frame.append(Self.version)
        frame.append(cmdID)

        var seq = seqNum.bigEndian
        frame.append(Data(bytes: &seq, count: 2))

        var payloadLen = UInt16(payload.count).bigEndian
        frame.append(Data(bytes: &payloadLen, count: 2))

        frame.append(payload)

        // CRC16-CCITT 覆盖 Magic 到 Payload
        let crcData = frame.subdata(in: 1..<frame.count)  // 跳过 Preamble
        let crc = CRC16.ccitt(data: crcData)
        var crcBE = crc.bigEndian
        frame.append(Data(bytes: &crcBE, count: 2))

        return frame
    }
}

// CRC16-CCITT 实现
enum CRC16 {
    static func ccitt(data: Data) -> UInt16 {
        var crc: UInt16 = 0xFFFF
        let polynomial: UInt16 = 0x1021

        for byte in data {
            crc ^= UInt16(byte) << 8
            for _ in 0..<8 {
                if crc & 0x8000 != 0 {
                    crc = (crc << 1) ^ polynomial
                } else {
                    crc <<= 1
                }
            }
        }
        return crc
    }
}
```

### 帧解码器

```swift
class FrameDecoder {

    enum DecodeResult {
        case complete(ProtocolFrame)
        case partial    // 数据不完整，等待更多数据
        case error      // 帧格式错误
    }

    private var buffer = Data()

    func append(_ data: Data) -> [DecodeResult] {
        buffer.append(data)
        var results: [DecodeResult] = []

        while !buffer.isEmpty {
            // 1. 寻找帧头（Preamble + Magic）
            guard let frameStart = findFrameStart() else {
                // 没有找到帧头，清空无效数据
                buffer.removeAll()
                break
            }

            // 2. 检查是否有足够的数据读取帧头
            guard buffer.count >= frameStart + ProtocolFrame.headerSize else {
                // 数据不完整
                if frameStart > 0 {
                    buffer.removeFirst(frameStart)
                }
                results.append(.partial)
                break
            }

            // 3. 解析帧头
            let headerData = buffer.subdata(in: frameStart..<frameStart+ProtocolFrame.headerSize)
            let version = headerData[3]
            let cmdID = headerData[4]
            let seqNum = headerData.subdata(in: 5..<7).withUnsafeBytes {
                $0.load(as: UInt16.self).byteSwapped
            }
            let payloadLen = Int(headerData.subdata(in: 7..<9).withUnsafeBytes {
                $0.load(as: UInt16.self).byteSwapped
            })

            // 4. 检查是否有完整帧
            let frameTotalLen = ProtocolFrame.headerSize + payloadLen + ProtocolFrame.crcSize
            guard buffer.count >= frameStart + frameTotalLen else {
                results.append(.partial)
                break
            }

            // 5. CRC 校验
            let crcData = buffer.subdata(in: frameStart+1..<frameStart+ProtocolFrame.headerSize+payloadLen)
            let calculatedCRC = CRC16.ccitt(data: crcData)
            let receivedCRC = buffer.subdata(
                in: frameStart+ProtocolFrame.headerSize+payloadLen..<frameStart+frameTotalLen
            ).withUnsafeBytes { $0.load(as: UInt16.self).byteSwapped }

            if calculatedCRC != receivedCRC {
                // CRC 错误，跳过此帧
                buffer.removeFirst(frameStart + 1)  // 跳过 Preamble，继续搜索下一帧
                results.append(.error)
                continue
            }

            // 6. 提取 Payload
            let payload = buffer.subdata(
                in: frameStart+ProtocolFrame.headerSize..<frameStart+ProtocolFrame.headerSize+payloadLen
            )

            let frame = ProtocolFrame(cmdID: cmdID, seqNum: seqNum, payload: payload)
            results.append(.complete(frame))

            // 7. 移除已解析的帧
            buffer.removeFirst(frameStart + frameTotalLen)
        }

        return results
    }

    private func findFrameStart() -> Int? {
        guard buffer.count >= 3 else { return nil }

        for i in 0...(buffer.count - 3) {
            if buffer[i] == ProtocolFrame.preamble
               && buffer[i+1] == 0xAA
               && buffer[i+2] == 0x55 {
                return i
            }
        }
        return nil
    }
}
```

### TLV 解析器

```swift
struct TLVField {
    let type: UInt8
    let value: Data
}

class TLVParser {
    static func parse(_ payload: Data) -> [TLVField] {
        var fields: [TLVField] = []
        var offset = 0

        while offset + 2 <= payload.count {
            let type = payload[offset]
            let length = Int(payload[offset + 1])

            guard offset + 2 + length <= payload.count else {
                break  // 数据不完整
            }

            let value = payload.subdata(in: offset+2..<offset+2+length)
            fields.append(TLVField(type: type, value: value))
            offset += 2 + length
        }

        return fields
    }

    static func encode(fields: [TLVField]) -> Data {
        var data = Data()
        for field in fields {
            data.append(field.type)
            data.append(UInt8(field.value.count))
            data.append(field.value)
        }
        return data
    }
}
```

## 命令定义与分发

### 命令注册表

```swift
enum CommandType: UInt8 {
    // 系统命令
    case heartbeat = 0x01
    case handshake = 0x02
    case versionQuery = 0x03

    // 设备管理
    case bindRequest = 0x10
    case unbindRequest = 0x11
    case statusQuery = 0x12

    // 数据上报
    case sensorDataReport = 0x30
    case eventNotification = 0x31

    // 控制命令
    case switchControl = 0x50
    case modeControl = 0x51

    // OTA
    case otaStart = 0x90
    case otaData = 0x91
    case otaVerify = 0x92

    // 响应
    case response = 0xFF
}

protocol CommandHandler {
    func handle(frame: ProtocolFrame) -> ProtocolFrame?
}

class CommandDispatcher {
    private var handlers: [CommandType: CommandHandler] = [:]

    func register(_ type: CommandType, handler: CommandHandler) {
        handlers[type] = handler
    }

    func dispatch(_ frame: ProtocolFrame) -> ProtocolFrame? {
        guard let type = CommandType(rawValue: frame.cmdID),
              let handler = handlers[type] else {
            return makeErrorResponse(frame: frame, errorCode: 0x01)
        }
        return handler.handle(frame: frame)
    }

    private func makeErrorResponse(frame: ProtocolFrame, errorCode: UInt8) -> ProtocolFrame {
        var payload = Data()
        payload.append(frame.cmdID)  // 原始命令字
        payload.append(errorCode)
        return ProtocolFrame(cmdID: CommandType.response.rawValue,
                            seqNum: frame.seqNum,
                            payload: payload)
    }
}
```

## 协议版本兼容策略

### 向前兼容设计

新版本协议新增字段时，使用 TLV 格式的 Payload 可以自然兼容：旧版本解析器遇到未知的 Type，通过 Length 跳过即可。关键是**永远不删除已有字段、不改变已有字段的 Type 值**。

### 版本协商

```swift
// 握手帧（CmdID=0x02）Payload
struct HandshakePayload {
    let protocolVersion: UInt8      // 支持的最高版本
    let featureFlags: UInt8         // 功能特性位图
    let deviceCapability: UInt16    // 设备能力位图
}

// 握手响应 Payload
struct HandshakeResponsePayload {
    let negotiatedVersion: UInt8    // 协商后的版本（取双方最小）
    let negotiatedFeatures: UInt8   // 协商后的特性
    let status: UInt8               // 0=成功
}
```

## 设计要点总结

帧格式设计需要平衡几个关键因素：帧头开销（尽量小）、容错能力（魔数+CRC）、扩展性（Version+TLV）、解析效率（定长帧头+变长Payload）。推荐的 11 字节帧头 + TLV Payload 方案在实际项目中验证了良好的可靠性和可维护性。核心原则是：帧格式自描述、字段编码跨平台一致、版本兼容性内置考虑、CRC 覆盖全帧数据。这些设计为上层的分包重组、流控重传和加密策略提供了坚实的数据格式基础。
