# 通信协议流控与重传机制详解

## 概述

在不可靠的传输通道上保证数据可靠送达，需要流控（Flow Control）和重传（Retransmission）两大机制。流控防止发送方灌爆接收方，重传保证丢失的数据最终送达。本文系统讲解停等协议、回退 N 步、选择重传三种 ARQ（Automatic Repeat reQuest）机制的原理和实现，以及滑动窗口流控的设计。

## ARQ 协议体系

### 三种 ARQ 机制对比

| 机制 | 发送窗口 | 接收窗口 | 乱序处理 | 重传范围 | 效率 |
|------|---------|---------|---------|---------|------|
| 停等协议 (Stop-and-Wait) | 1 | 1 | 不支持 | 当前帧 | 最低 |
| 回退 N 步 (GBN) | N | 1 | 不支持 | 从丢失帧起全部 | 中等 |
| 选择重传 (SR) | N | N | 支持 | 仅丢失帧 | 最高 |

### 停等协议（Stop-and-Wait）

最简单的可靠传输方式：发送一帧后等待 ACK，收到 ACK 才发下一帧。

```
发送方                        接收方
  |                              |
  | -------- Frame 0 ----------> |
  |                              |
  | <--------- ACK 0 ----------- |
  |                              |
  | -------- Frame 1 ----------> |
  |                              |
  |     (ACK 1 丢失)             |
  | <------- (无 ACK) ---------- |
  |                              |
  | 超时，重传 Frame 1           |
  | -------- Frame 1 ----------> |
  |                              |
  | <--------- ACK 1 ----------- |
  |                              |
```

优点：实现简单。缺点：效率极低，信道利用率约为 1/(2×RTT/传输时间)。

### 回退 N 步（Go-Back-N）

允许发送方在未收到 ACK 时连续发送多个帧（窗口大小 N），但接收方只按序接收。一旦某帧丢失，从该帧起的所有后续帧都要重传。

```
发送方 (窗口 N=4)              接收方
  |                              |
  | -------- Frame 0 ----------> | 收到，ACK 0
  | -------- Frame 1 ----------> | 收到，ACK 1
  | -------- Frame 2 --X (丢失)  |
  | -------- Frame 3 ----------> | 期望 2，收到 3，丢弃
  | <--------- ACK 0 ----------- |
  | <--------- ACK 1 ----------- |
  |                              |
  | Frame 2 超时                  |
  | 重传 Frame 2                 |
  | -------- Frame 2 ----------> | 收到，ACK 2
  | 重传 Frame 3                 |
  | -------- Frame 3 ----------> | 收到，ACK 3
```

优点：接收端实现简单（只需维护期望序号）。缺点：丢失一帧导致大量重传，浪费带宽。

### 选择重传（Selective Repeat）

发送方和接收方都维护窗口，接收方可以接收乱序帧并缓存。只重传真正丢失的帧。

```
发送方 (窗口 N=4)              接收方 (窗口 N=4)
  |                              |
  | -------- Frame 0 ----------> | 收到，缓存，ACK 0
  | -------- Frame 1 ----------> | 收到，缓存，ACK 1
  | -------- Frame 2 --X (丢失)  |
  | -------- Frame 3 ----------> | 收到，缓存（乱序），ACK 3
  |                              |
  | <--------- ACK 0 ----------- | 窗口滑动
  | <--------- ACK 1 ----------- | 窗口滑动
  | <--------- ACK 3 ----------- | 记录 3 已收到
  |                              |
  | Frame 2 超时，仅重传 Frame 2 |
  | -------- Frame 2 ----------> | 收到，2+3 按序交付，ACK 2
```

优点：效率最高，只重传丢失帧。缺点：接收端需要缓存乱序帧，实现复杂。

## 滑动窗口实现

### 发送端窗口

```swift
class SendWindow {
    private let windowSize: Int
    private var base = 0          // 窗口左边界（最早未确认的序号）
    private var nextSeq = 0       // 下一个待发送序号
    private var unacked: [Int: (data: Data, sentTime: Date, retryCount: Int)] = [:]
    private let lock = NSLock()

    // 定时器参数
    private let rto: TimeInterval = 3.0       // 重传超时
    private let maxRetries = 3
    private var timer: DispatchSourceTimer?

    var onDataSend: ((Int, Data) -> Void)?
    var onAllAcked: (() -> Void)?
    var onMaxRetriesExceeded: ((Int) -> Void)?

    init(windowSize: Int = 8) {
        self.windowSize = windowSize
        startTimer()
    }

    // 尝试发送数据
    func send(_ data: Data) -> Bool {
        lock.lock()
        defer { lock.unlock() }

        guard nextSeq < base + windowSize else {
            return false  // 窗口已满
        }

        let seq = nextSeq
        nextSeq += 1
        unacked[seq] = (data, Date(), 0)

        onDataSend?(seq, data)
        return true
    }

    // 收到 ACK（累积确认）
    func receiveACK(_ ackSeq: Int) {
        lock.lock()
        defer { lock.unlock() }

        // 累积确认：ACK n 表示 n-1 及之前都已收到
        if ackSeq >= base {
            // 移除已确认的条目
            for seq in base...ackSeq {
                unacked.removeValue(forKey: seq)
            }
            base = ackSeq + 1

            if unacked.isEmpty {
                onAllAcked?()
            }
        }
    }

    // 收到 NACK（选择重传）
    func receiveNACK(_ nackSeq: Int) {
        lock.lock()
        defer { lock.unlock() }

        if var entry = unacked[nackSeq] {
            entry.sentTime = Date()
            entry.retryCount += 1
            unacked[nackSeq] = entry
            onDataSend?(nackSeq, entry.data)
        }
    }

    // 定时检查超时重传
    private func startTimer() {
        let timer = DispatchSource.makeTimerSource(queue: .global(qos: .utility))
        timer.schedule(deadline: .now() + 0.5, repeating: 0.5)
        timer.setEventHandler { [weak self] in
            self?.checkTimeouts()
        }
        timer.resume()
        self.timer = timer
    }

    private func checkTimeouts() {
        lock.lock()
        let now = Date()
        var toRetransmit: [(Int, Data)] = []
        var toFail: [Int] = []

        for (seq, entry) in unacked {
            if now.timeIntervalSince(entry.sentTime) > rto {
                if entry.retryCount >= maxRetries {
                    toFail.append(seq)
                } else {
                    toRetransmit.append((seq, entry.data))
                    unacked[seq] = (entry.data, now, entry.retryCount + 1)
                }
            }
        }
        lock.unlock()

        for (seq, data) in toRetransmit {
            onDataSend?(seq, data)
        }

        for seq in toFail {
            onMaxRetriesExceeded?(seq)
        }
    }

    // 自适应 RTO（RTT 估算）
    private var srtt: TimeInterval = 1.0  // 平滑 RTT
    private var rttvar: TimeInterval = 0.5  // RTT 方差

    func updateRTT(sampleRTT: TimeInterval) {
        // Jacobson 算法
        srtt = 0.875 * srtt + 0.125 * sampleRTT
        rttvar = 0.75 * rttvar + 0.25 * abs(sampleRTT - srtt)
        rto = srtt + max(0.2, 4 * rttvar)
        rto = max(0.5, min(rto, 10.0))  // RTO 范围限制
    }

    var windowAvailable: Int {
        lock.lock()
        defer { lock.unlock() }
        return base + windowSize - nextSeq
    }
}
```

### 接收端窗口（选择重传）

```swift
class ReceiveWindow {
    private let windowSize: Int
    private var expectedSeq = 0     // 期望接收的下一个序号
    private var buffer: [Int: Data] = [:]  // 乱序缓存
    private let lock = NSLock()

    var onDataReceived: ((Data) -> Void)?
    var onSendACK: ((Int) -> Void)?
    var onSendNACK: ((Int) -> Void)?

    init(windowSize: Int = 8) {
        self.windowSize = windowSize
    }

    func receive(seq: Int, data: Data) {
        lock.lock()
        defer { lock.unlock() }

        if seq < expectedSeq {
            // 重复包，重新 ACK
            onSendACK?(seq)
            return
        }

        if seq == expectedSeq {
            // 按序到达，直接交付
            onDataReceived?(data)
            expectedSeq += 1

            // 检查缓存中是否有后续连续的包
            while let cached = buffer[expectedSeq] {
                onDataReceived?(cached)
                buffer.removeValue(forKey: expectedSeq)
                expectedSeq += 1
            }

            // ACK 已交付的最高序号
            onSendACK?(expectedSeq - 1)

        } else if seq < expectedSeq + windowSize {
            // 乱序但在窗口内，缓存
            buffer[seq] = data

            // 发送 NACK 请求重传缺失的包
            for missing in expectedSeq..<seq {
                if buffer[missing] == nil {
                    onSendNACK?(missing)
                }
            }
            // ACK 已收到的最高连续序号
            onSendACK?(expectedSeq - 1)

        } else {
            // 超出窗口范围，丢弃
            onSendACK?(expectedSeq - 1)
        }
    }
}
```

## 背压机制

### 接收端驱动的流控

```swift
class BackpressureController {
    private let maxBufferSize: Int
    private var currentBufferSize: Int = 0
    private let lock = NSLock()

    // 接收缓冲区水位线
    private let highWatermark: Int  // 高水位：暂停接收
    private let lowWatermark: Int   // 低水位：恢复接收

    var onPauseReceive: (() -> Void)?
    var onResumeReceive: (() -> Void)?

    init(maxBufferSize: Int = 1024 * 1024) {  // 1MB
        self.maxBufferSize = maxBufferSize
        self.highWatermark = Int(Double(maxBufferSize) * 0.8)
        self.lowWatermark = Int(Double(maxBufferSize) * 0.3)
    }

    func onDataAdded(_ size: Int) {
        lock.lock()
        currentBufferSize += size
        let needPause = currentBufferSize >= highWatermark
        lock.unlock()

        if needPause {
            onPauseReceive?()
        }
    }

    func onDataConsumed(_ size: Int) {
        lock.lock()
        currentBufferSize -= size
        let needResume = currentBufferSize <= lowWatermark
        lock.unlock()

        if needResume {
            onResumeReceive?()
        }
    }
}
```

### 发送端速率控制

```swift
class RateController {
    private var tokens: Int          // 当前令牌数
    private let refillRate: Int      // 每秒补充令牌数
    private let maxTokens: Int       // 最大令牌数（突发容量）
    private var lastRefillTime: Date
    private let lock = NSLock()

    init(rate: Int, burst: Int) {
        self.refillRate = rate
        self.maxTokens = burst
        self.tokens = burst
        self.lastRefillTime = Date()
    }

    func tryAcquire(tokensNeeded: Int = 1) -> Bool {
        lock.lock()
        defer { lock.unlock() }

        refill()

        if tokens >= tokensNeeded {
            tokens -= tokensNeeded
            return true
        }
        return false
    }

    private func refill() {
        let now = Date()
        let elapsed = now.timeIntervalSince(lastRefillTime)
        let newTokens = Int(elapsed * Double(refillRate))
        if newTokens > 0 {
            tokens = min(maxTokens, tokens + newTokens)
            lastRefillTime = now
        }
    }
}
```

## BLE 场景的流控特化

BLE 的 Write Without Response 有系统级流控机制，需要与之配合：

```swift
class BLEFlowControl {
    private let peripheral: CBPeripheral
    private let characteristic: CBCharacteristic
    private let window: SendWindow
    private var pendingData: [Data] = []

    init(peripheral: CBPeripheral, characteristic: CBCharacteristic) {
        self.peripheral = peripheral
        self.characteristic = characteristic
        self.window = SendWindow(windowSize: 4)  // BLE 建议小窗口

        window.onDataSend = { [weak self] _, data in
            guard let self = self else { return }
            // 检查 CoreBluetooth 流控
            if self.peripheral.canSendWriteWithoutResponse(for: self.characteristic) {
                self.peripheral.writeValue(data, for: self.characteristic, type: .withoutResponse)
            } else {
                // CoreBluetooth 队列满，缓存等待回调
                self.pendingData.append(data)
            }
        }
    }

    // CoreBluetooth 队列恢复
    func onReadyToSend() {
        let dataToSend = pendingData
        pendingData.removeAll()

        for data in dataToSend {
            if peripheral.canSendWriteWithoutResponse(for: characteristic) {
                peripheral.writeValue(data, for: characteristic, type: .withoutResponse)
            } else {
                pendingData.append(data)
                break
            }
        }
    }

    // 设备 ACK 回调
    func onDeviceACK(_ ackSeq: Int) {
        window.receiveACK(ackSeq)
    }

    func send(_ data: Data) -> Bool {
        return window.send(data)
    }
}
```

## 重传策略优化

### 快速重传

不等超时，收到 3 个重复 ACK 即触发重传：

```swift
class FastRetransmit {
    private var duplicateACKCount = 0
    private var lastACKSeq = -1
    private let fastRetransmitThreshold = 3

    var onRetransmit: ((Int) -> Void)?

    func receiveACK(_ ackSeq: Int) {
        if ackSeq == lastACKSeq {
            duplicateACKCount += 1
            if duplicateACKCount == fastRetransmitThreshold {
                // 快速重传
                onRetransmit?(ackSeq + 1)
                duplicateACKCount = 0
            }
        } else {
            duplicateACKCount = 0
            lastACKSeq = ackSeq
        }
    }
}
```

### 自适应 RTO 详解

```swift
class AdaptiveRTO {
    private var srtt: Double = 1.0       // Smoothed RTT
    private var rttvar: Double = 0.5     // RTT Variance
    private(set) var rto: Double = 3.0   // Retransmission Timeout

    // RFC 6298 算法
    func updateSample(rtt: Double) {
        if srtt == 0 {
            // 首次测量
            srtt = rtt
            rttvar = rtt / 2
        } else {
            // 后续更新
            let alpha = 0.125
            let beta = 0.25

            rttvar = (1 - beta) * rttvar + beta * abs(rtt - srtt)
            srtt = (1 - alpha) * srtt + alpha * rtt
        }

        rto = srtt + max(0.2, 4 * rttvar)
        rto = max(0.5, min(rto, 10.0))  // RTO 范围 [0.5s, 10s]
    }

    // 指数退避：每次重传 RTO 翻倍
    func backoff() {
        rto = min(rto * 2, 10.0)
    }

    func reset() {
        // 收到新 ACK 后重置 RTO
        rto = srtt + max(0.2, 4 * rttvar)
    }
}
```

## 完整的可靠传输层

```swift
class ReliableTransport {

    private let sendWindow: SendWindow
    private let receiveWindow: ReceiveWindow
    private let rtoManager = AdaptiveRTO()
    private let fastRetransmit = FastRetransmit()
    private let backpressure = BackpressureController()

    private var sendTimestamps: [Int: Date] = [:]  // 用于 RTT 测量

    var onSend: ((Int, Data) -> Void)?       // 实际发送回调
    var onReceived: ((Data) -> Void)?         // 数据交付回调
    var onSendACK: ((Int) -> Void)?           // 发送 ACK
    var onSendNACK: ((Int) -> Void)?          // 发送 NACK

    init(windowSize: Int = 8) {
        sendWindow = SendWindow(windowSize: windowSize)
        receiveWindow = ReceiveWindow(windowSize: windowSize)

        // 绑定回调
        sendWindow.onDataSend = { [weak self] seq, data in
            self?.sendTimestamps[seq] = Date()
            self?.onSend?(seq, data)
        }

        receiveWindow.onDataReceived = { [weak self] data in
            self?.backpressure.onDataAdded(data.count)
            self?.onReceived?(data)
        }

        receiveWindow.onSendACK = { [weak self] ackSeq in
            self?.onSendACK?(ackSeq)
        }

        receiveWindow.onSendNACK = { [weak self] nackSeq in
            self?.onSendNACK?(nackSeq)
        }

        fastRetransmit.onRetransmit = { [weak self] seq in
            self?.sendWindow.receiveNACK(seq)
        }
    }

    // 发送数据
    func send(_ data: Data) -> Bool {
        return sendWindow.send(data)
    }

    // 接收 ACK
    func receiveACK(_ ackSeq: Int) {
        // RTT 测量
        if let sendTime = sendTimestamps[ackSeq] {
            let rtt = Date().timeIntervalSince(sendTime)
            rtoManager.updateSample(rtt: rtt)
            sendTimestamps.removeValue(forKey: ackSeq)
        }

        fastRetransmit.receiveACK(ackSeq)
        sendWindow.receiveACK(ackSeq)
    }

    // 接收数据
    func receive(seq: Int, data: Data) {
        receiveWindow.receive(seq: seq, data: data)
    }
}
```

## 协议参数推荐

针对不同场景的推荐参数：

| 场景 | 窗口大小 | 初始 RTO | 最大重试 | ACK 策略 |
|------|---------|---------|---------|---------|
| BLE OTA 传输 | 4 | 3s | 3 | 每 4 包 ACK |
| WiFi 设备通信 | 16 | 1s | 5 | 累积 ACK |
| 低功耗传感器 | 1（停等）| 5s | 2 | 每包 ACK |
| 实时控制 | 8 | 0.5s | 3 | NACK 为主 |

## 总结

流控与重传是可靠传输协议的核心。选择重传（SR）配合滑动窗口在效率和可靠性之间取得了最佳平衡，适用于大多数 IoT 通信场景。关键实现要点包括：基于 Jacobson 算法的自适应 RTO 调整、快速重传（3 次重复 ACK 触发）减少等待时间、NACK 机制加速丢包恢复、背压机制防止接收端溢出。在 BLE 场景下还需特别注意与 CoreBluetooth 的 `canSendWriteWithoutResponse` 流控机制配合，避免应用层窗口和系统级窗口冲突。
