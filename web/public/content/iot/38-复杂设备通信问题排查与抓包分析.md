# 复杂设备通信问题排查与抓包分析

## 概述

IoT 设备通信涉及 App、蓝牙协议栈、设备固件、网络环境等多个环节，出现问题时定位难度极大。连接不稳定、OTA 中断、数据丢包等问题需要系统化的排查方法和抓包分析能力。本文从问题分类、排查方法论、抓包工具链和实战案例四个方面，讲解复杂通信问题的分析与解决。

## 问题分类与特征

### 连接类问题

**连接超时**：`connect()` 调用后既不触发 `didConnect` 也不触发 `didFailToConnect`。通常原因：设备不在广播范围、设备固件异常、iOS 蓝牙缓存脏数据。

**频繁断连**：连接建立后短时间内断开，`didDisconnectPeripheral` 频繁回调。通常原因：连接参数不合理（Supervision Timeout 过短）、设备电量不足、信号干扰、多端竞争连接。

**重连失败**：断开后自动重连始终失败。通常原因：iOS 蓝牙状态异常（需重启蓝牙）、设备未重新进入广播模式、系统缓存了旧的连接状态。

### 数据类问题

**数据丢失**：发送的数据设备未收到，或设备发送的数据 App 未收到。通常原因：Write Without Response 丢包、Notify 频率过高导致 iOS 回调堆积、MTU 不匹配导致大包被截断。

**数据错乱**：接收到的数据帧解析错误、字段值异常。通常原因：字节序不一致、帧同步丢失（粘包/拆包处理不当）、CRC 校验未生效。

**传输缓慢**：OTA 或数据同步速度远低于预期。通常原因：MTU 未协商到最优值、Write With Response 导致每包等待 ACK、连接间隔过大。

### OTA 类问题

**OTA 中断**：固件传输过程中断开连接。通常原因：传输时间过长导致 iOS 后台限制、设备写入 Flash 耗时导致蓝牙响应超时、信号波动。

**OTA 失败回滚**：升级后设备无法启动，回滚到旧版本。通常原因：固件不兼容、CRC/签名校验失败、Flash 写入异常。

**OTA 进度卡住**：传输进度不前进。通常原因：滑动窗口死锁、ACK 丢失导致重传风暴、设备端缓冲区满。

## 排查方法论

### 分层排查法

```
应用层    — 检查协议帧格式、命令逻辑、状态机
  ↓
CoreBluetooth 层 — 检查 API 调用时序、回调日志、错误码
  ↓
BLE 协议层 — 检查 MTU、连接参数、GATT 服务
  ↓
链路层    — 抓包分析 LL 层交互、信号质量
  ↓
物理层    — 检查信号强度、干扰源、距离
```

### 日志系统

建立完善的日志系统是排查的基础：

```swift
class BLELogger {

    enum LogLevel: Int {
        case debug = 0, info, warning, error
    }

    static let shared = BLELogger()
    private var logs: [(timestamp: Date, level: LogLevel, tag: String, message: String)] = []
    private let lock = NSLock()
    private let maxLogCount = 10000

    // 日志级别控制
    var minLevel: LogLevel = .debug
    var enabledTags: Set<String>? = nil  // nil = 全部启用

    func log(_ level: LogLevel, tag: String, _ message: String) {
        guard level.rawValue >= minLevel.rawValue else { return }
        if let enabled = enabledTags, !enabled.contains(tag) { return }

        lock.lock()
        defer { lock.unlock() }

        logs.append((Date(), level, tag, message))

        // 控制日志总量
        if logs.count > maxLogCount {
            logs.removeFirst(logs.count - maxLogCount)
        }

        // 实时输出
        let levelStr = ["D", "I", "W", "E"][level.rawValue]
        let timeStr = formatTime(Date())
        print("[\(timeStr)][\(levelStr)][\(tag)] \(message)")
    }

    // 导出日志用于分析
    func exportLogs() -> String {
        lock.lock()
        defer { lock.unlock() }

        return logs.map { entry in
            let levelStr = ["D", "I", "W", "E"][entry.level.rawValue]
            let timeStr = formatTime(entry.timestamp)
            return "[\(timeStr)][\(levelStr)][\(entry.tag)] \(entry.message)"
        }.joined(separator: "\n")
    }

    private func formatTime(_ date: Date) -> String {
        let formatter = DateFormatter()
        formatter.dateFormat = "HH:mm:ss.SSS"
        return formatter.string(from: date)
    }
}

// 使用示例
BLELogger.shared.log(.info, tag: "Central", "开始扫描设备")
BLELogger.shared.log(.info, tag: "Central", "发现设备: \(peripheral.name ?? "Unknown"), RSSI: \(rssi)")
BLELogger.shared.log(.info, tag: "Central", "连接设备: \(peripheral.identifier)")
BLELogger.shared.log(.error, tag: "Central", "连接失败: \(error.localizedDescription)")
BLELogger.shared.log(.debug, tag: "GATT", "写入特征值 \(char.uuid): \(data.hexString)")
BLELogger.shared.log(.debug, tag: "GATT", "收到 Notify \(char.uuid): \(data.hexString)")
BLELogger.shared.log(.warning, tag: "OTA", "分片 \(seqNum) 超时，重传")
```

```swift
struct DiagnosticInfo {
    // iOS 端信息
    let iosVersion: String          // iOS 版本
    let appVersion: String          // App 版本
    let deviceModel: String         // iPhone 型号

    // BLE 连接信息
    let peripheralIdentifier: UUID
    let peripheralName: String?
    let connectionState: String
    let mtu: Int                    // 协商后的 MTU

    // 设备信息
    let deviceFirmwareVersion: String?
    let deviceModel: String?
    let deviceManufacturer: String?

    // 信号质量
    let rssi: Int
    let noiseLevel: Int?            // 如果可用

    // 时间线
    let eventLog: [String]          // 关键事件时间线
}

class BLEDiagnosticCollector {

    static func collectDiagnostics(peripheral: CBPeripheral,
                                    centralManager: CBCentralManager) -> DiagnosticInfo {
        return DiagnosticInfo(
            iosVersion: UIDevice.current.systemVersion,
            appVersion: Bundle.main.infoDictionary?["CFBundleShortVersionString"] as? String ?? "",
            deviceModel: UIDevice.current.model,
            peripheralIdentifier: peripheral.identifier,
            peripheralName: peripheral.name,
            connectionState: "\(peripheral.state)",
            mtu: peripheral.maximumWriteValueLength(for: .withoutResponse) + 3,
            deviceFirmwareVersion: nil,  // 从设备读取
            deviceModel: nil,
            deviceManufacturer: nil,
            rssi: 0,  // 从最近一次 didDiscover 获取
            noiseLevel: nil,
            eventLog: BLELogger.shared.exportLogs().components(separatedBy: "\n")
        )
    }
}
```

## 抓包工具链

### Packet Logger（macOS）

Apple 官方的蓝牙抓包工具，可以抓取 iOS 设备的 BLE 通信：

```
使用方法:
1. iPhone 通过 USB 连接到 Mac
2. 打开 Packet Logger
3. 选择连接的 iOS 设备
4. 开始抓包
5. 在 App 中操作 BLE
6. 停止抓包，分析数据包

可以看到的信息:
- GAP 广播包内容
- ATT 请求/响应
- GATT 发现过程
- MTU 协商
- 连接参数更新
- 数据传输内容
- 加密状态
```

### Wireshark + BLE Sniffer

使用专用的 BLE Sniffer 硬件（如 nRF52 Sniffer、TI CC2640 Sniffer）配合 Wireshark 抓取空口数据：

```
使用方法:
1. 将 Sniffer 硬件 USB 连接到 Mac
2. 打开 Wireshark，选择 Sniffer 接口
3. 设置过滤频道（BLE 使用 37/38/39 三个广播频道 + 0-36 数据频道）
4. 开始抓包
5. 在 App 中操作 BLE
6. 分析空口数据包

优势:
- 可以看到物理层信息（信号强度、重传、CRC 错误）
- 可以同时抓取 iOS 和设备端的数据
- 可以分析 iOS 无法直接获取的链路层数据
```

### nRF Connect（移动端）

在 Android 设备上使用 nRF Connect 作为调试工具：

```
功能:
- 扫描发现 BLE 设备
- 连接并浏览 GATT 服务树
- 读写特征值
- 订阅 Notify
- 记录通信日志
- 测量信号强度

用途:
- 验证设备 GATT 结构是否正确
- 测试设备响应是否正常
- 对比 iOS 和 Android 的行为差异
```

## 实战案例分析

### 案例一：连接后 3 秒自动断开

**现象**：App 连接设备后约 3 秒自动断开，错误码为 0x08（Connection Timeout）。

**排查过程**：

```swift
// 日志记录
BLELogger.shared.log(.info, tag: "Connection", "didConnect: \(peripheral.name ?? "")")
// 3 秒后...
BLELogger.shared.log(.error, tag: "Connection", "didDisconnect: error=\(error?.localizedDescription ?? "")")

// CoreBluetooth 错误码
if let error = error as NSError? {
    let code = error.code
    // code = 6 (CBError.connectionTimeout)
    // 表示 Supervision Timeout 内未收到设备响应
}
```

**抓包分析**：

Packet Logger 显示连接建立成功，但连接后没有任何 ATT 请求。设备端在 Supervision Timeout（约 3 秒）内未收到任何数据包，判定连接超时并主动断开。

**根因**：App 在 `didConnect` 中没有及时发起 `discoverServices`，设备端配置的 Supervision Timeout 过短（仅 3 秒）。

**解决方案**：

```swift
func centralManager(_ central: CBCentralManager,
                    didConnect peripheral: CBPeripheral) {
    BLELogger.shared.log(.info, tag: "Connection", "didConnect")

    peripheral.delegate = self

    // 立即发现服务，不要延迟
    peripheral.discoverServices(nil)

    // 同时开启连接保活心跳
    startHeartbeat(peripheral)
}
```

同时要求设备端固件将 Supervision Timeout 调整为 6 秒以上。

### 案例二：OTA 传输到 50% 卡住

**现象**：OTA 固件传输到约 50% 时进度不再前进，App 端持续发送但设备不再 ACK。

**排查过程**：

```swift
// 日志显示
BLELogger.shared.log(.debug, tag: "OTA", "发送分片: offset=\(offset), seq=\(seqNum)")
// ...正常发送...
BLELogger.shared.log(.debug, tag: "OTA", "发送分片: offset=\(offset), seq=\(seqNum)")
// 最后几条:
BLELogger.shared.log(.warning, tag: "OTA", "分片 \(seqNum) 超时，重传")
BLELogger.shared.log(.warning, tag: "OTA", "分片 \(seqNum) 超时，重传")
BLELogger.shared.log(.warning, tag: "OTA", "分片 \(seqNum) 超时，重传")
// 设备不再响应任何请求
```

**抓包分析**：

Wireshark 空口抓包显示：App 在持续发送 BLE 数据包，但设备端在某个点后停止了所有 BLE 响应（包括 Link Layer 的 ACK）。这说明设备固件卡死，无法处理 BLE 事件。

**根因**：设备端在写入 Flash 时关闭了中断，导致 BLE 协议栈无法处理链路层事件。Flash 写入时间过长（单次写入超过连接间隔），触发 Supervision Timeout。

**解决方案**：

设备端固件优化：Flash 写入分块进行，每次写入不超过 5ms，写完后允许 BLE 中断处理。App 端增加流控：在发送一批数据后主动等待设备 ACK，避免持续灌包。

```swift
// App 端流控优化
class OTAFlowControl {
    private let batchSize = 16  // 每批发送 16 个分片
    private var sentInBatch = 0
    private var ackedInBatch = 0

    func onDataSent() {
        sentInBatch += 1
        if sentInBatch >= batchSize {
            // 暂停发送，等待 ACK
            pauseSending()
        }
    }

    func onACKReceived() {
        ackedInBatch += 1
        if ackedInBatch >= batchSize {
            // 一批完成，继续下一批
            sentInBatch = 0
            ackedInBatch = 0
            resumeSending()
        }
    }
}
```

### 案例三：Notify 数据偶发丢失

**现象**：设备通过 Notify 推送传感器数据（100Hz），偶发性丢失部分数据。

**排查过程**：

```swift
// 接收计数
var receivedCount = 0
var expectedCount = 0  // 设备端发送的总数

func peripheral(_ peripheral: CBPeripheral,
                didUpdateValueFor characteristic: CBCharacteristic,
                error: Error?) {
    receivedCount += 1

    // 每秒打印一次统计
    if receivedCount % 100 == 0 {
        BLELogger.shared.log(.info, tag: "Notify",
            "已接收 \(receivedCount), 预期 \(expectedCount), 丢失率 \(lossRate)%")
    }
}
```

**抓包分析**：

Packet Logger 显示设备确实发送了所有数据包，但 iOS 端的回调次数少于发送次数。进一步分析发现，当 iOS 主线程繁忙（如 UI 渲染）时，CoreBluetooth 的回调被延迟或合并。

**根因**：100Hz 的 Notify 频率对 iOS 来说过高。CoreBluetooth 的回调在主线程执行，当主线程被 UI 任务占用时，部分回调被丢弃。

**解决方案**：

```swift
// 1. 降低设备端 Notify 频率到 10Hz，单包聚合 10 次采样
// 2. CoreBluetooth 回调切换到后台队列
class BLEManager {
    init() {
        let bleQueue = DispatchQueue(label: "com.iot.ble", qos: .userInitiated)
        centralManager = CBCentralManager(delegate: self, queue: bleQueue)
        // 回调在 bleQueue 执行，不阻塞主线程
    }
}

// 3. 接收端增加序列号检查和丢包统计
class NotifyReceiver {
    private var lastSeqNum: UInt16 = 0
    private var lostPackets: Int = 0

    func handleData(_ data: Data) {
        guard data.count >= 2 else { return }
        let seqNum = data.subdata(in: 0..<2).withUnsafeBytes {
            $0.load(as: UInt16.self).byteSwapped
        }

        if lastSeqNum > 0 {
            let expected = lastSeqNum + 1
            if seqNum != expected {
                let lost = Int(seqNum) - Int(expected)
                if lost > 0 {
                    lostPackets += lost
                    BLELogger.shared.log(.warning, tag: "Notify",
                        "丢包: 期望 \(expected), 收到 \(seqNum), 丢失 \(lost)")
                }
            }
        }
        lastSeqNum = seqNum
    }
}
```

### 案例四：后台连接频繁断开

**现象**：App 切到后台后几分钟内连接断开，重连后又断开，循环往复。

**排查过程**：

```swift
// 后台断连日志
BLELogger.shared.log(.error, tag: "Background",
    "后台断连: \(error?.localizedDescription ?? "")")

// 错误码分析
// CBError.code = 6 (connectionTimeout) — Supervision Timeout
// 或 CBError.code = 7 (peripheralDisconnected) — 设备主动断开
```

**根因**：

1. App 未开启 bluetooth-central 后台模式，后台时 CoreBluetooth 被系统挂起
2. 设备端连接参数中 Slave Latency 设置为 0，后台时设备保持高频率连接，功耗过大导致电池快速消耗
3. Supervision Timeout 设为 2 秒，iOS 后台调度延迟可能导致超时

**解决方案**：

```swift
// 1. 确保开启 Background Modes: bluetooth-central
// 2. 使用 State Restoration
// 3. 配合设备端调整连接参数:
//    - Connection Interval: 后台时 500ms-1000ms
//    - Slave Latency: 4-10
//    - Supervision Timeout: 6000ms (6秒)
```

## 总结

复杂设备通信问题的排查需要系统化的方法论和工具链支持。核心是分层排查（应用层→CoreBluetooth→BLE协议→链路层→物理层）和完善的日志系统。抓包工具中 Packet Logger 是 iOS BLE 调试的首选，Wireshark + Sniffer 用于链路层问题分析。实际案例中最常见的问题类型是连接超时（Supervision Timeout 配置不当）、OTA 中断（设备端中断处理与 BLE 协议栈冲突）、Notify 丢包（高频推送超出 iOS 处理能力）和后台断连（后台模式未正确配置）。建立日志→抓包→定位→修复→验证的排查闭环，是高效解决通信问题的保障。
